# 16 维审计矩阵 — 详细检查清单

> **用途**: 子代理审计时的逐维度检查指南。每个维度包含：检查项、判定标准、引用要求、置信度指南。
> **使用方式**: 子代理接到指定维度后，按此清单逐条检查，每条产出 finding 或标注 PASS。

---

## A 类：正确性（硬错误 —— 可自动检测）

### A1 — 结构完整性

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| A1.1 | **死 wikilink** | 正则提取所有 `[[...]]`，逐一解析目标文件是否存在 | 目标 .md 不存在 → 死链接 |
| A1.2 | **残缺 wikilink** | 检查 `[[` 和 `]]` 是否配对闭合 | 未闭合的 wikilink → 残缺 |
| A1.3 | **编号重复** | 同模块内所有 `NNN-` 前缀文件名/章节编号去重 | 两个文件/章节同号 → 冲突 |
| A1.4 | **编号跳号** | 同模块内编号序列是否连续（如 001→002→004，缺 003） | 缺号 → 标记为遗漏风险 |
| A1.5 | **README 存在性** | 每个模块根目录是否有 README.md | 缺失 → 硬错误 |
| A1.6 | **README 最低行数** | README.md 行数 >= 60（CHG-048 规定） | 不足 → 硬错误 |
| A1.7 | **接头总览时间戳过期** | 检查每条"最后验证"时间戳，距今 >90 天 → 过期 | 过期条目数 / 总条目数 = 腐化率 |
| A1.8 | **归档文件引用** | 检查是否有文件引用了已归档/删除的文件 | 引用目标在 `archived/` 或不存在 → 过期引用 |
| A1.9 | **frontmatter 完整性** | 每个 .md 文件是否有 `> ` 块引用元数据（开发代号、引擎、开发者） | 缺失 → 不完整 |

**置信度指南**: A1 全部为确定性检测——无歧义，confidence = `certain`。

---

### A2 — 接口契约一致性

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| A2.1 | **出口 vs CLAUDE-INTERFACES 对齐** | 接头总览 `001-接口出口.md` 的提供项 与 CLAUDE-INTERFACES.md 中该模块的 Owner 条目逐一比对 | 出口有但 CLAUDE-INTERFACES 无 → 遗漏；CLAUDE-INTERFACES 有但出口无 → 漏登记 |
| A2.2 | **入口 vs 消费方出口对齐** | 接头总览 `002-接口入口.md` 的消费项，溯源到被消费模块的 `001-接口出口.md`，检查类型签名一致 | 入口引用的概念在源模块出口中不存在或签名不同 → 契约断裂 |
| A2.3 | **trait 索引登记** | `00-全局基础设施/003-通用trait索引.md` 中是否登记了所有跨模块 trait | 源模块定义了 trait 但索引无 → 未登记 |
| A2.4 | **所有权唯一性** | CLAUDE-INTERFACES.md 中同一概念是否只在一个模块标记为 Owner | 多 Owner → 所有权冲突 |
| A2.5 | **变更影响链反查** | `003-变更影响链.md` 列出的消费方列表 vs `002-接口入口.md` 实际消费方列表 | 影响链遗漏消费方 → 影响传播盲区 |
| A2.6 | **CHG 接头段落** | 最近 5 个 CHG 文档是否都包含"接头总览更新"章节 | 缺失 → 保鲜协议违规 |

**置信度指南**: A2 大部分为确定性检测。A2.3 若 trait 名称模糊匹配可能有歧义 → confidence = `likely`。

---

### A3 — 数学正确性

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| A3.1 | **公式求值** | python 执行所有数学表达式，比较文档中给出的结果 | 计算结果 ≠ 文档结果 → 错误 |
| A3.2 | **边界值代入** | 对公式代入极端值（0、负值、极大值、NaN），检查是否有溢出/除以零/未定义 | 无边界处理逻辑 → 遗漏 |
| A3.3 | **量纲一致性** | 检查公式两边的单位是否匹配（如 "秒 + 米"、"概率 × 距离"） | 量纲不匹配 → 错误 |
| A3.4 | **概率范围** | 所有概率值是否在 [0, 1] 范围 | 超出 → 错误 |
| A3.5 | **百分比之和** | 同组百分比是否加总 = 100%（或明确标注允许溢出） | 不加总或未标注 → 遗漏 |
| A3.6 | **浮点精度说明** | 涉及 f32/f64 的计算是否说明了精度取舍策略 | 无说明 → 遗漏风险 |

**置信度指南**: A3.1-A3.5 确定性检测。A3.6 为软判断 → confidence = `likely`。

## B 类：设计质量（软 —— AI 语义理解）

### B1 — 内部一致性

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| B1.1 | **同概念同定义** | 同模块内同一术语的所有出现，提取其定义句，比对语义 | 两个段落给同一概念互斥定义 → 自相矛盾 |
| B1.2 | **同概念跨模块一致** | 搜索同一概念在不同模块中的描述，比对 Owner 定义 vs Consumer 理解 | Consumer 理解偏离 Owner 定义 → 跨模块漂移 |
| B1.3 | **版本号一致性** | 同一文档内引用的版本号（如 "v4.0"、"v2.1"）是否自洽 | 同文档内版本号打架 → 不一致 |
| B1.4 | **参数传递一致性** | 函数/系统 A 输出参数的类型+范围 与 函数/系统 B 的同名输入参数是否匹配 | 类型不匹配或范围不兼容 → 接口断裂 |
| B1.5 | **命名一致性** | 同一概念在不同文件中是否使用相同名称（排除大小写/中英混用） | 同概念多名 → 命名漂移 |

**置信度指南**: 需要语义理解。必须引用原文原句。confidence = `certain` 仅在原文明确且互斥时；否则 `likely`。

---

### B2 — 边界完整性

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| B2.1 | **输入边界** | 每个函数/规则/算法是否声明了输入的有效范围 | 无范围声明 → 遗漏 |
| B2.2 | **极端值行为** | 边界外输入（0、空、负、极大、NaN）是否有定义的处理行为 | 无定义 → 遗漏 |
| B2.3 | **失败模式** | 操作失败/资源不足/权限不够时的行为是否定义 | 无失败路径 → 遗漏 |
| B2.4 | **并发边界** | 涉及并行/多实体操作的规则是否考虑了竞态/冲突 | 无并发处理 → 遗漏 |
| B2.5 | **时间边界** | 涉及时间的规则是否定义了 tick 开始/结束/跨天/跨年行为 | 无时间边界 → 遗漏 |

**置信度指南**: 基于"是否存在"而非"是否合理"。缺失是确定的 → confidence = `certain`。

---

### B3 — 设计原则遵循

| # | 检查项 | 方法 | 判定标准（对照 CLAUDE.md 核心哲学） |
|---|--------|------|---------|
| B3.1 | **故事生成器非故事容器** | 检查是否定义了涌现规则而非硬编码情节 | 硬编码剧情线 → 违背 |
| B3.2 | **NPC=完整的人** | 检查 NPC 系统是否有记忆/情感/欲望/习惯/人际关系维度 | 维度缺失 → 不完整 |
| B3.3 | **玩家=NPC 同套规则** | 检查玩家特殊待遇是否违背"同一套底层系统" | 玩家专用硬编码 → 违背 |
| B3.4 | **涌现>门控** | 检查是否"if skill_lvl >= X then allow"模式 —— 应为物理涌现 | 门控规则 → 违背（战斗三层模型例外） |
| B3.5 | **种子确定性** | 涉及随机生成的模块是否声明种子确定性 + 增量存档 | 无种子说明 → 遗漏 |
| B3.6 | **零硬编码禁止** | 检查是否存在无法通过 TOML/mod 修改的魔法数字 | 硬编码数值 → 违背 |
| B3.7 | **技能是事后记录** | 技能是否定义为"练过多少次"而非"能不能做的门" | 技能做门控 → 违背 |
| B3.8 | **玩家特色平等** | 冒险/生活/政治等路径是否由同一套底层系统支撑 | 某路径有专属硬编码系统 → 违背 |

**置信度指南**: 设计原则的违背判断需要语义理解。必须引用 CLAUDE.md 原句作为对照基准。

---

### B4 — 遗漏检测

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| B4.1 | **关键概念未定义** | 识别文档中使用的但未在模块内或依赖模块中定义的术语 | 使用未定义 → 遗漏 |
| B4.2 | **引用链断裂** | 文档说"详见 XXX"但 XXX 不存在或未写 | 空承诺 → 遗漏 |
| B4.3 | **接口无消费方** | 出口定义中有些接口没有被任何模块消费 | 孤立接口 → 可能是过度设计或遗漏消费方 |
| B4.4 | **消费方无接口** | 入口声明消费某概念，但概念 Owner 未在出口提供 | 依赖无供应 → 遗漏 |
| B4.5 | **示例缺失** | 复杂概念/算法是否提供了数值示例或使用场景 | 抽象概念无示例 → 可理解性遗漏 |
| B4.6 | **与"开发文档待完善之处"对齐** | 检查根目录 `开发文档待完善之处与奇思妙想 20260602.md` 中的待办是否已解决 | 长期未解决 → 标记 |

**置信度指南**: B4.1-B4.4 基于存在性检查，`certain`。B4.5-B4.6 涉及判断，`likely`。

## C 类：架构与解耦（软 —— AI 语义理解）

### C1 — Trait 边界清晰度

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| C1.1 | **trait 暴露行为非数据** | 检查 trait 方法签名——返回的是行为结果还是内部数据结构 | 暴露内部结构体 → 封装泄漏 |
| C1.2 | **pub(crate) 封装** | 模块间只通过 trait 交互，数据字段不 pub | 非 trait 路径的 pub 字段 → 封装泄漏 |
| C1.3 | **trait 单一职责** | 检查每个 trait 的方法数和方法之间的关系 | >10 方法或无关方法混在同一个 trait → 职责过重 |
| C1.4 | **trait 可 mock** | trait 是否足够抽象，消费方可否在测试中用 mock 实现 | 依赖具体类型 → 不可测试 |
| C1.5 | **异步边界** | 涉及 I/O/网络/大计算的方法是否在 trait 签名中体现 async | 同步签名隐藏阻塞操作 → 架构隐患 |

**置信度指南**: C1 判断基于 Rust 解耦最佳实践。需要推理 trait 设计意图 → confidence = `likely` 为主。

---

### C2 — 模块间零循环依赖

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| C2.1 | **直接循环** | 从接头总览构建依赖图，DFS 检测环 | A→B→A 或 A→B→C→A → 循环依赖 |
| C2.2 | **隐式循环** | 通过事件总线的间接循环（A emit Event → B 响应 → B emit Event → A 响应） | 无限事件循环 → 隐式循环 |
| C2.3 | **类型循环** | woworld_core 中的类型 A 引用类型 B，类型 B 又引用类型 A | 类型定义循环 → 编译风险 |
| C2.4 | **文档引用循环** | "...详见 A"，A 说 "...详见 B"，B 说 "...详见 C"，C 说 "...详见 A" | 文档死循环 → 重定向 |

**置信度指南**: C2.1 和 C2.3 可机械检测，`certain`。C2.2 需要推理事件流 → `likely`。

---

### C3 — 数据驱动边界

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| C3.1 | **TOML 可覆盖性** | 检查模块中的数值/参数是否有 TOML 配置入口，还是硬编码在 Rust 源码中 | 硬编码数值 → 不可 mod |
| C3.2 | **Mod 安全边界** | 检查 TOML 可修改的参数范围——哪些改会影响平衡/安全/存档兼容 | 无安全边界说明 → 遗漏 |
| C3.3 | **配方表分离** | 交互配方表（TOML 数据驱动）和 CraftingRecipe 是否独立 | 混在同一表 → 耦合 |
| C3.4 | **扩展点预留** | 是否有注册机制（Registry trait）供 mod 注入自定义类型 | 无 Registry → 不可扩展 |
| C3.5 | **事件订阅** | mod 是否可以订阅系统事件而不修改核心代码 | 无事件总线接口 → 不可扩展 |

**置信度指南**: C3.1-C3.3 基于存在性检查，`certain`。C3.4-C3.5 涉及设计意图 → `likely`。

---

### C4 — 事件总线覆盖

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| C4.1 | **跨模块通知** | 两个模块间有数据流但无事件定义 → 检查是否是直接 import 调用 | 直接 import 对方模块 → 耦合 |
| C4.2 | **事件完整性** | 检查关键生命周期事件（出生/死亡/结婚/战争/交易）是否都有对应事件定义 | 生命周期缺口 → 遗漏 |
| C4.3 | **事件负载** | 事件携带的数据是否足够消费方独立决策（不需要回调查询源模块） | 数据不足 → 消费方需反向依赖 |
| C4.4 | **事件幂等性** | 重复接收同一事件是否会导致错误（如双重扣款） | 无幂等保护 → 风险 |

**置信度指南**: C4.1-C4.2 基于存在性，`certain`。C4.3-C4.4 需推理 → `likely`。

## D 类：体验与涌现（软 —— AI 语义理解）

### D1 — 玩家接触面

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| D1.1 | **首次接触点** | 玩家在游戏中第一次碰到这个系统是什么时刻？设计文档是否描述了？ | 无首次接触描述 → 遗漏 |
| D1.2 | **学习曲线** | 系统机制的复杂度是否分层暴露给玩家（简单→复杂） | 一次性全暴露 → 体验风险 |
| D1.3 | **掌握深度** | 系统是否有浅层可玩 + 深层可精通的梯度 | 无深度梯度 → 浅薄 |
| D1.4 | **反馈清晰度** | 玩家的操作是否能得到可感知的反馈（视觉/音频/数值变化） | 无反馈描述 → 玩家困惑 |
| D1.5 | **无障碍性** | 是否有对新手/非核心玩家的简化入口或辅助 | 无辅助 → 门槛过高 |
| D1.6 | **与其他系统的玩家视角关联** | 玩家理解这个系统时是否自然需要理解另一个系统 | 需要跨系统理解但无引导 → 体验断裂 |

**置信度指南**: 玩家体验判断高度主观。必须结合游戏类型（开放世界 RPG）评估。confidence = `likely` 为主。

---

### D2 — 涌现潜力

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| D2.1 | **门控 vs 涌现** | 检查交互是否通过"if 条件 then 结果"硬编码，还是通过参数概率涌现 | 硬 if-then 过多 → 门控 |
| D2.2 | **参数自由度** | 系统关键结果由多少个独立参数决定（>3 = 高涌现潜力） | <2 参数 → 低涌现 |
| D2.3 | **NPC 自主性** | NPC 使用该系统时是否完全自主决策，还是仅响应玩家 | 纯响应式 → 有限涌现 |
| D2.4 | **连锁反应** | 一个系统的输出是否可以触发其他系统的不确定性反应 | 输出被截断 → 涌现断流 |
| D2.5 | **稀有事件** | 是否有低概率但高影响力的边缘情况 | 无稀有事件 → 缺少惊喜 |

**置信度指南**: 涌现判断依赖设计哲学理解。confidence = `likely` 为主。

---

### D3 — 程序化参数空间

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| D3.1 | **种子确定性** | 给定相同种子，是否保证相同输出 | 无种子声明 → 遗漏 |
| D3.2 | **参数可调范围** | 程序化生成的每个参数是否有 min/max/default 声明 | 无范围 → 不可调 |
| D3.3 | **玩家可感知变化** | 参数变化是否产生玩家能注意到的世界差异 | 参数调了世界不变 → 参数无意义 |
| D3.4 | **参数交互** | 不同参数之间是否存在非线性的交互效应 | 参数孤立 → 变化维度低 |
| D3.5 | **增量可扩展** | 新的程序化内容类型可否通过添加参数集来引入（不修改核心算法） | 新类型需改算法 → 不可扩展 |

**置信度指南**: D3.1-D3.2 基于存在性，`certain`。其余 `likely`。

---

### D4 — 模块独特性

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| D4.1 | **核心差异** | 与同类游戏/同类系统相比，这个模块的不可替代点是什么？文档是否明确？ | 无差异声明 → 模块定位模糊 |
| D4.2 | **与 WoWorld 世界观的绑定** | 模块是否利用了 WoWorld 独有的设定（程序化历史/涌现社会/完整NPC） | 可原样搬到别的游戏 → 不够独特 |
| D4.3 | **最小可行复杂度** | 模块的复杂度是否必要——有没有更简单的设计实现同样的独特体验 | 过度设计 → 维护负担 |
| D4.4 | **玩家会记住的时刻** | 这个模块能否产生玩家愿意分享的"你猜发生了什么"时刻 | 无记忆点 → 平庸 |

**置信度指南**: 高度主观。confidence = `likely` 或 `uncertain`。

---

### D5 — 性能可落地性

| # | 检查项 | 方法 | 判定标准 |
|---|--------|------|---------|
| D5.1 | **预算存在性** | 模块是否有性能预算（CPU/内存/VRAM/磁盘） | 无预算 → 遗漏 |
| D5.2 | **预算推导** | 预算数字是否有推导过程（"为什么是这个数"而非"我们定了这个数"） | 无推导 → 拍脑袋 |
| D5.3 | **预算整体协调** | 本模块预算 + 相关模块预算是否在总硬件预算内（GTX 1660 SUPER 6GB） | 超总预算 → 不可落地 |
| D5.4 | **LOD 策略** | 涉及视觉/音频/计算的模块是否有 LOD 降级方案 | 无 LOD → 不可缩放 |
