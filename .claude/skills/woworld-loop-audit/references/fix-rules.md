# 硬错误自动修复规则表

> **用途**: 当审计发现 A 类（正确性）硬错误时，按此规则表自动修复。
> **核心原则**:
> - **可逆原则**: 每次自动修复生成一个 revert 记录（写入 audit-reports session 目录下的 `auto-fixes.log`）
> - **最小修改原则**: 只改必要的最小范围，不顺手重构
> - **不碰设计意图**: 绝不修改数值、算法、设计决策——只修正结构性错误

---

## A1.1 — 死 wikilink 修复

### 判定
`[[...]]` 中的目标文件路径在仓库中不存在。

### 修复策略（按优先级尝试）

```
1. 模糊匹配搜索
   - 提取目标文件名（去掉路径前缀）
   - Glob 搜索仓库中同名或相似文件名
   - Levenshtein 距离排序候选
   - 距离 = 0（精确匹配，路径不同）→ 自动修正路径
   - 距离 ≤ 3（拼写错误）→ 自动修正
   - 距离 > 3 → 不自动修，标记为 "待人工裁决"

2. 文档重命名/移动检测
   - 检查 git log --diff-filter=R -1 -- <target>
   - 如果文件被 git mv 重命名 → 自动更新链接

3. 无法匹配
   - 标记为 "存疑死链接" → 写入待人工裁决清单
   - 不删除链接，不添加 `<!-- TODO -->` 注释
   - 原始链接保持不动
```

### 修复日志格式
```
[A1.1] 死 wikilink 修复
  文件: <源文件>:<行号>
  原链接: [[旧路径]]
  新链接: [[新路径]]
  匹配方法: 模糊匹配 (距离=2) | git-rename | 未修复
```

## A1.2 — 编号重复修复

### 判定
同模块内两个 .md 文件或章节使用相同的前缀编号（如两个 `003-xxx.md` 或两个 `§三`）。

### 修复策略

```
1. 同目录文件编号冲突
   - 列出冲突文件及其 git 创建时间
   - 更早创建的文件保留原编号
   - 更晚创建的文件编号追加后缀（如 003→003b）
   - 更新所有引用该编号的 wikilink

2. 同文档内章节编号冲突
   - 保留首次出现的章节
   - 后续重复编号章节追加后缀
   - 但 §一~§十 等中文序号——检查是否真的是章节号还是普通文本
```

### 修复日志格式
```
[A1.2] 编号重复修复
  模块: <模块名>
  冲突编号: 003
  文件A: <路径> (保留)
  文件B: <路径> → 重编号为 003b
  受影响 wikilink: 3 处已更新
```

## A1.3 — README 缺失修复

### 判定
模块根目录无 README.md 或 README.md < 60 行（CHG-048 规定）。

### 修复策略

```
1. 完全缺失 → 自动生成 README.md 骨架
   模板内容：
   - 模块名称 + "> " frontmatter
   - 文档索引（列出本模块所有 .md 文件）
   - 架构概述（从 001 总纲提取核心概念摘要）
   - 关键参数表（若有）
   - 标注"⚠️ 自动生成 —— 请人工丰富"

2. 行数不足 → 不自动修改
   - 标记为 "README 过短 (<60行)"，写入报告
   - 不清空已有内容强行填满
```

### 修复日志格式
```
[A1.3] README 自动生成
  模块: <模块名>
  模板: 文档索引 + 架构概述（从 001 提取）
  行数: 初始 ~40 行（请人工补充至 60+）
```

## A1.7 — 接头总览时间戳过期修复

### 判定
接头总览条目"最后验证"时间戳距今 > 90 天。

### 修复策略

```
不自动修改时间戳 —— 因为"验证"意味着人工确认过。
自动操作：
  - 统计过期条目数和总条目数
  - 计算腐化率 = 过期 / 总数
  - 腐化率 > 30% → 在报告中醒目标注，建议运行 loop-audit
  - 腐化率 ≤ 30% → 在报告中列出过期条目清单
```

## A1.9 — Frontmatter 缺失修复

### 判定
.md 文件开头缺少 `> ` 块引用元数据。

### 修复策略

```
从同模块 README.md 或 001 文档提取元数据模板，自动补到文件开头：
  > **开发代号**: WoWorld (Wonder World)
  > **引擎**: Godot 4.7
  > **开发者**: 独立游戏开发者（Solo）
  > **所属模块**: <从路径推断>
```

### 修复日志格式
```
[A1.9] Frontmatter 自动补全
  文件: <路径>
  模板来源: <模块 README>
```

## A2.2 — 入口 vs 源出口签名不一致

### 判定
模块 B 的入口声明消费模块 A 的概念 X，但模块 A 的出口中 X 的签名类型与 B 记录的不一致。

### 修复策略

```
以 CLAUDE-INTERFACES.md 为权威源：
  1. 如果 CLAUDE-INTERFACES.md 有明确契约 → 以它为准，修正入口和出口
  2. 如果 CLAUDE-INTERFACES.md 无此契约 → 以 Owner 模块的出口为准，修正消费方入口
  3. 如果两边签名语义等价但表述不同 → 统一为 Owner 的表述
  4. 如果两边确实矛盾且无法判断哪个对 → 不自动修，写入待人工裁决

不自动修改模块设计文档正文，只修正接头总览的入口/出口文件。
```

### 修复日志格式
```
[A2.2] 契约签名修正
  概念: <概念名>
  Owner: <模块A>
  消费方: <模块B>
  权威源: CLAUDE-INTERFACES.md / Owner出口
  修正: 模块B 入口条目更新为 <新签名>
```

## A3.1 — 数学公式错误修复

### 判定
python 执行公式得到的结果 ≠ 文档中给出的结果。

### 修复策略

```
1. 重新推导——公式对但数字错 → 自动修正数字
2. 公式本身错误 → 不自动修公式（设计意图），标记"待人工裁决"
3. 浮点精度差异（<0.1%）→ 忽略（不是错误）
4. 单位换算错误（如 km→m 的倍数）→ 自动修正，注明换算
```

### 修复日志格式
```
[A3.1] 数学修正
  文件: <路径>:<行号>
  原值: <文档中的数字>
  新值: <python 计算出的数字>
  公式: <已验证的公式>
  原因: 计算错误（非公式错误）
```

## A3.4 — 概率范围错误修复

### 判定
概率值超出 [0, 1]。

### 修复策略

```
1. 值为百分比（如 50）且上下文暗示是百分比 → 自动除以 100 + 标注
2. 值为负数 → 标记"待人工裁决"（可能是公式错误，不能机械取绝对值）
3. 值 > 1 且上下文中概率都用 [0,1] → 标记"待人工裁决"
```

## 不可自动修复项（必须人工裁决）

以下情况一律不自动修，写入 `pending_human_decisions`：

- wikilink 模糊匹配距离 > 3
- 两个模块对同一概念有矛盾定义且无法判断哪个是权威
- 数学公式本身错误（非计算错误）
- 涉及设计意图的任何修改
- 概率值异常且上下文不明
- 跨模块契约断裂且 CLAUDE-INTERFACES.md 无明确裁决依据
- 性能预算数字的修改（即使看起来"明显不合理"）
- 任何会导致业务流程或游戏机制改变的修改
