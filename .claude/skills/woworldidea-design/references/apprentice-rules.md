# 铁匠学徒规则详解

## 三层权限（被动 / 手动纸条 / 绝不碰）

### 被动区（直接执行，不询问用户）

以下文件属于本模块自己的管理文件，纯机械同步，不涉及跨模块设计判断：

1. `开发阶段/模块接头总览/XX-模块名/001-接口出口.md` — 出口条目与模块文档定义同步
2. `开发阶段/模块接头总览/XX-模块名/002-接口入口.md` — 入口条目与消费事实同步
3. `开发阶段/模块接头总览/XX-模块名/003-变更影响链.md` — 影响链从出口派生
4. `开发阶段/模块接头总览/XX-模块名/000-变更日志.md` — 日志是机械记录
5. `开发阶段/模块接头总览/变更追踪/001-待处理变更队列.md` — 队列条目从改动派生
6. `Change/CHG-XXX-...md` — 生成 CHG 初稿，用户审核后定稿
7. `CLAUDE.md` — CHG 条目和接口契约摘要机械添加

**执行时机**: sync 模式的 S4 + S8-S10

### 手动纸条（准备好修改，用户逐条 y/n/e 确认）

以下涉及跨模块边界，需要用户确认：

1. **消费方模块文档** — 任何其他模块中引用了本模块概念的文件
2. **CLAUDE-INTERFACES.md 契约表** — "宪法"级文档，改动必须用户签字

**纸条格式**：
```
📍 文件: <消费方文档路径>
📝 当前: "<原文引用>"
⚠️ <变更影响描述>
🔧 建议: <具体修改内容>
📊 <Hard/Soft/Ref> | 置信度: <高/中/低> (<纯技术同步/可能涉及设计意图>)
[y]应用 [n]跳过 [e]编辑
```

**执行时机**: sync 模式的 S7

### 绝不碰

以下内容**完全不动、不评论、不标注**：

- 消费方自己提出的设计理念、哲学、架构决策
- 包含因果推理词的句子：因为 / 所以 / 因此 / 选择 / 设计为 / 理由 / 我们假定 / 核心哲学
- 消费方对概念的独特用法或解释（与 owner 定义不完全一致但有理由的）

**核心判断法**：一段话去掉技术引用后，**它仍然在表达一个设计看法** → 不动。一段话只有技术引用本身 → 可同步。

## Hard / Soft / Ref 分级标准

### Hard（类型级硬依赖）

**判断标准**（满足任一条即为 Hard）：
1. 消费方文档中明确写了 Rust 类型路径（如 `woworld_life::Vitals`）
2. 消费方文档中引用了具体字段名（如 `Vitals.hp`、`Vitals.stamina`）
3. 消费方文档中写了具体数值范围或约束（如 `hp: f32, 0-1`）
4. 消费方文档中的代码块/伪代码直接使用了此类型
5. 消费方的接口入口条目（002-接口入口.md）中列出了此概念的具体参数

**同步策略**: 类型签名变了必须同步更新。纯技术同步，置信度高。

**示例**:
- `woworld_life::Vitals → woworld_economy::NpcEconomicState` → Hard
- `Vitals.hp 做归一化除法 hp/max_hp` → Hard
- 入口条目写 `Vitals(HP/体力/饥饿/...)` → Hard

### Soft（设计级软引用）

**判断标准**（满足任一条即为 Soft）：
1. 消费方文档中提及概念名但不引用具体字段或类型
2. 消费方文档描述了一个依赖此概念的设计行为，但不绑定具体实现
3. 概念变更可能影响消费方的设计假设，但不是直接的类型不兼容

**同步策略**: 标注影响 + 建议检查。置信度中，标注"纯技术同步"或"可能涉及设计意图"。

**示例**:
- "NPC 健康状态影响劳动生产力" → Soft（没写怎么影响）
- "战后伤势恢复期间 Vitals 修正" → Soft（没写修正公式）
- "音频模块内部直读 Vitals 做心跳触发" → Soft（触发条件没列具体字段）

### Ref（仅wikilink）

**判断标准**: 消费方文档中只通过 `[[wikilink]]` 提及，没有实质使用。

**同步策略**: 仅记录，不标注修改。用户可自行决定是否更新 wikilink 目标。

## 置信度标注

每处建议修改必须标注置信度：

| 置信度 | 标注 | 含义 |
|--------|------|------|
| 高 | 纯技术同步 | 类型签名/字段名/数值范围的机械更新，不需要设计判断 |
| 中 | 可能涉及设计意图 | 需要检查消费方的设计假设是否仍成立 |
| 低 | 推断 | 从上下文推断的关联，需要用户自行验证 |
