# audit — 旧文件审计修复

## 触发

```
/woworldidea-design audit <文件路径>
```

## 场景

你发现一个设计文档已经过时——其他模块改了、概念变了、接口更新了，但这份文件还是旧版本。你怀疑它和当前系统有很多不一致，但不想手工逐条比对。

## 执行流程

### S1: 提取引用

读取目标文件 → 提取所有外部概念引用：

1. **wikilink**: `[[01-生命/004-Vitals|Vitals]]` → 提取目标文档路径
2. **Rust 类型/字段名**: `woworld_life::Vitals`、`Vitals.hp`、`Vitals.stamina` → 正则提取
3. **trait 名**: `BuildingQuery`、`WeatherQuery` → 匹配已知 trait 列表
4. **结构体/枚举名**: `DeathCause`、`LifeEntity` → 匹配已知类型列表
5. **常数值**: 硬编码数字如"伤害公式中 spirit 消耗系数 0.5" → 如果 owner 改了值

### S2: 逐条比对

对每个引用 → 查 owner 模块的当前定义 → 比对文件中使用的版本：

| 比对维度 | 问题 |
|---------|------|
| 类型 | f32 vs u32？ |
| 字段名 | 文件用 `thirst`，owner 已改名 `thirst_level`？ |
| 存在性 | 文件引用的概念已被删除？ |
| 数值范围 | 文件说 0-100，owner 定义 0-1？ |
| 约束 | 文件说不为空，owner 允许 None？ |
| 接口签名 | trait 方法签名变了？ |

**方法**: 使用子代理，每个外部引用启动一个比对。和 sync 的 S5 共用引擎。

### S3: 分级标注

每个不一致标注 Hard/Soft/Ref + 建议修改 + 置信度：

```
🔴 Hard: 第45行 "Vitals.thirst (0-100)" → Life模块当前定义 thirst 已是 f32 0-1 归一化
   建议: 将 0-100 改为 0-1 归一化，或确认此处使用百分比模型有特殊理由
   置信度: 高 (纯技术同步)

🟡 Soft: 第78行 "NPC死亡时记录 DeathCause (25种)" → Life模块 CHG-041 已扩展为 30种6类
   建议: 更新数量描述，检查新增5种死亡原因是否影响此处逻辑
   置信度: 高 (纯技术同步)

🟢 Ref: 第12行 "详见 [[01-生命/005-旧版本文档]]" → 目标文件已不存在
   建议: 更新或删除此 wikilink
   置信度: 高 (纯技术同步)
```


## v1 限制

- 只做报告，不修改被审计的文件
- 不修改消费方文档
- 如果审计文件是某个模块的一部分，不自动更新该模块的接头总览
- v2 将加入自动修复选项
