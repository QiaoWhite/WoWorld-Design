# gate — 新模块三层闸门

## 触发

```
/woworldidea-design gate <模块名> [--level 1|2]
```

默认跑 L1。用户可选 `--level 2` 跳级。

## 执行流程

### L1: 轻触（草稿期，每次改完就跑）

**检查1: 内部文档间矛盾**

读取新模块所有文档 → 提取所有定义性陈述 → 交叉比对：
- 文档 A 说"BlockFamily 有 9 个核心族"，文档 B 说"8 个核心族" → 矛盾
- 文档 C 定义了字段 `width: u32`，文档 D 的伪代码写了 `block.width as f32` → 类型矛盾

**检查2: 断链检测**

提取所有 `[[wikilink]]` → 逐一检查目标文件是否存在 → 不存在的报告为断链。

**检查3: 外部依赖签名验证**

提取所有对外部概念的引用（类型名/trait名/字段名）→ 查 owner 模块的接头总览出口条目：
- 引用 `Vitals.hydration_level`？→ 查 Life 模块出口条目 → Vitals 有 `hydration` 无 `hydration_level` → 字段名不匹配
- 引用 `BuildingQuery::query_blueprint()`？→ 查 Building 模块出口 → BuildingQuery 不存在 → 引用了不存在的 trait

**输出（不改文件）**:
```
━━━━━━━━━━━━━━━━━━━━━━
L1 轻触报告: <模块名>
━━━━━━━━━━━━━━━━━━━━━━

🔴 内部矛盾 (2):
  • 003-建筑类型.md vs 005-WFC求解.md: "核心族数量"不一致 (9 vs 8)
  • ...

🟡 断链 (3):
  • 002-数据模型.md: [[../经济/不存在文件]] → 目标不存在
  • ...

🟠 外部签名不匹配 (1):
  • 001-总纲.md: 引用了 "Vitals.hydration_level" → Life 模块实际字段名为 "hydration"
  • ...

━━━━━━━━━━━━━━━━━━━━━━
💡 方向稳了后跑 /woworldidea-design gate <模块名> --level 2 做完整对接。
━━━━━━━━━━━━━━━━━━━━━━
```

### L2: 对接（方向稳了，看看外部世界）

在 L1 全部基础上增加：

**检查4: 完整 owner-vs-consumer 对齐**

对每个本模块消费的外部概念 → 子代理比对：
- 本模块的使用方式 ↔ owner 的当前定义
- 类型、范围、约束是否一致？
- 不一致 → 报告冲突

**检查5: 出口重叠检查**

本模块定义的每个新概念 → 在所有已有模块接头总览出口中搜索同名/相似概念：
- 发现重叠 → 报告"概念 X 与已有模块 Y 的 Z 概念重叠，是否吸收/派生/重命名？"
- 防止概念重复定义

**检查6: 冲突检测 + 影响链**

本模块的所有外部消费 + 出口 → 拼出完整影响链。

**输出（不改文件）**:
```
━━━━━━━━━━━━━━━━━━━━━━
L2 对接报告: <模块名>
━━━━━━━━━━━━━━━━━━━━━━

L1 遗留:
  (L1 报告摘要)

🟠 外部消费冲突 (4):
  • 本模块消费 Vitals.stamina 做"建造体力消耗" → 
    Life 模块定义 stamina 为瞬时值(刻级消耗) → 
    你可能需要的是"长期耐力"而非"瞬时体力" (置信度: 中)

🔵 出口重叠 (1):
  • 本模块定义 "BuildingMaterial" → 
    物品系统已定义 "MaterialResource" (09-物品/003-材料.md) → 
    建议检查是否可以复用 MaterialResource 而非新建

📊 影响链预览:
    本模块导出 5 个概念 → 影响 6 个已有模块的 12 处消费点

━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━
```
