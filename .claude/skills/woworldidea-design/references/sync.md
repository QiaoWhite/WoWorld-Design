# sync — 改后自动同步（★主力命令）

## 触发

```
/woworldidea-design sync [--dry-run]
```

## 前提

用户在本模块已完成一轮设计修改（未 commit）。CC 帮助改了多个文件，可能涉及多个概念的新增/修改/删除。

## 完整执行流程

### S0: 安全点

自动执行 `git stash -u -m "woworldidea-design sync安全点-<timestamp>"`。

如果 git stash 失败（如无改动），跳过此步继续。
如果 `--dry-run` 模式，跳过此步。

### S1: 改动识别

**输入来源**:
1. 优先读 `.claude/temp/modified_files.txt`（PostToolUse hook 积累的标记文件）
2. 备用：从对话 transcript 解析 Write/Edit 工具的 `file_path`、`old_string`、`new_string`
3. 双重过滤：只保留 (a) `开发阶段/` 路径下 (b) 最后一次 sync/impact/gate 调用之后的改动

**输出**: 改动文件清单，展示给用户确认：
```
识别到以下文件在本轮被修改：
  • 01-生命/004-Vitals与生理状态.md (新增 hydration 字段)
  • 01-生命/001-接口出口.md (同步更新)
确认这是你要同步的改动范围吗？[y] [n-缩小范围] [e-手动指定]
```

### S2: 解析改动

从每个被改文件提取改动摘要：
- 新增/修改/删除了哪些概念？
- 新增/修改/删除了哪些字段？
- 类型、范围、约束有无变化？

**方法**: 对比文件当前内容与 git working tree vs HEAD（如果有 commit），或与 transcript 中记录的 old_string 对比。

**输出**: 改动摘要表：
```
概念: Vitals
  改动: 新增字段 hydration (f32, 0-1 归一化)
  改动: 字段 thirst 描述从"口渴程度"扩展为"饮水欲望(与hydration水分含量区分)"

概念: Physiology
  改动: from_vitals() 新增 hydration 派生逻辑
```

### S3: 抽查验证

从改动概念中随机抽 1-2 个消费方：
- 读消费方文档中引用该概念的段落
- 与接头总览中记录的消费条目比对

**如果不一致** → 警告：
```
⚠️ 抽查发现: 接头总览记录"经济模块消费 Vitals"，但经济模块文档中未找到引用。
接头总览可能已腐化，本次分析仅供参考。建议先跑 audit。
```

**如果一致** → 继续。

### S4: 被动区 — 更新本模块管理文件

对每个改动的概念，自动更新（不询问）：

1. **001-接口出口.md**: 
   - 概念条目中的字段列表、类型、描述与模块文档同步
   - 新增概念 → 新增出口条目
   - 删除概念 → 标记 `~~已废弃~~`（不删除条目）

2. **002-接口入口.md**:
   - 如果本次改动涉及本模块新增/修改对外部概念的消费 → 更新入口条目

3. **003-变更影响链.md**:
   - 如果出口变了 → 更新消费方列表
   - 新增出口 → 新增影响链段落

4. **000-变更日志.md**:
   - 追加条目: `<日期> | <改动摘要> | 出口变更/入口变更/内部重构 | <关联CHG编号>`

### S5: 冲突扫描（子代理并行）

对每个被改概念，为每个消费方启动一个子代理：

**子代理输入**:
```
概念: Vitals
Owner定义: (从模块文档和接头总览出口条目提取的完整定义，含字段列表/类型/约束)
消费方: 06-战斗
消费方文档路径: 开发阶段/06-战斗/...
接头总览入口条目: (消费方记录的引用方式)

请做以下检查:
1. 消费方文档中对 Vitals 的使用，与 owner 新定义是否一致？
2. 如果有不一致，是 Hard/Soft/Ref 哪个级别？
3. 如果一致，引用原文证明。
4. 如果不一致，引用双方原文，标注差异。

输出格式:
{
  "consistent": true/false,
  "level": "Hard/Soft/Ref",
  "owner_quote": "...",
  "consumer_quote": "...",
  "diff_description": "...",
  "suggested_fix": "...",
  "confidence": "纯技术同步/可能涉及设计意图"
}
```

**并行策略**: 所有消费方子代理同时启动。并发上限由系统自动管理。

**退路机制**: 如果子代理被拒绝或失败，主代理串行比对。在报告末尾标注"退路模式：X/Y 项使用子代理，Z 项为主代理串行"。

### S6: 铁匠学徒分级排序

汇总所有子代理输出 → 按 Hard → Soft → Ref 排序。

每组内按模块编号排序。

### S7: 手动纸条（逐条确认）

对每个消费方的每处不一致，按标准纸条格式展示：

```
━━━━━━━━━━━━━━━━━━━━━━
纸条 3/12: 06-战斗/002-接口入口.md
━━━━━━━━━━━━━━━━━━━━━━
📍 文件: 开发阶段/06-战斗/002-接口入口.md
📝 当前: "Vitals/Physiology/BodyPartDamage (存储端)"
📝 当前: "伤势持久化到 LifeEntity.damage_pool. 恢复期间 Vitals 修正"
⚠️ Vitals 新增 hydration 字段，战斗模块"恢复期间 Vitals 修正"可能需纳入 hydration
🔧 建议: 入口条目字段列表补 hydration。恢复逻辑段落标注"hydration影响恢复速度"待定
📊 Hard (接口签名) + Soft (设计逻辑) | 置信度: 高/中
[y]应用 [n]跳过 [e]编辑
```

**执行逻辑**:
- `y` → 立即应用建议修改
- `n` → 跳过，在报告中记录"用户确认跳过"
- `e` → 用户提供自定义修改内容，skill 执行
- 如果用户连续 `n` 超过 3 次，不再逐条确认，批量列出剩余纸条让用户自选

### S8: 被动区续 — CLAUDE-INTERFACES 契约表

如果本次改动涉及跨模块契约变更：
- 在 CLAUDE-INTERFACES.md 中对应 CHG 段落更新契约条目
- 如果新增契约关系 → 新增条目
- 递纸条给用户确认（ClAUDE-INTERFACES 是"宪法"）

### S9: 生成 CHG 初稿

汇总本次所有改动 → 写入 `Change/CHG-XXX-简短描述-YYYYMMDD.md`。

格式遵循现有 CHG 模板（一、变更概述 / 二、新增文件 / 三、修改文件 / ... / 十一、关联 CHG）。

CHG 编号自动取当前最大值 + 1。

初稿生成后 → 展示给用户审阅 → 用户可编辑。

### S10: 更新 CLAUDE.md

在 CLAUDE.md 中：
- 追加新 CHG 条目到"后续 CHG 契约摘要"表
- 如果新模块 → 追加模块条目

### S11: 自检验证

跑三个检查：

**P1 出口保鲜**: 
- 对比本模块文档中的概念定义 ↔ 接头总览出口条目
- 字段数、类型、描述是否一致？
- 不一致 → 报告并自动修正

**P2 入口一致性**:
- 对所有 Hard 级消费方，对比其文档中的引用 ↔ owner 新定义
- 仍有不一致 → 标记为"遗留冲突"

**P3 消费方入口条目**:
- 如果概念签名变了，消费方的接头总览入口条目更新了吗？
- 未更新 → 报告

**输出**:
```
自检完成:
  ✅ P1 出口保鲜: 通过
  ⚠️ P2 入口一致性: 2个Hard级消费方仍有遗留冲突（详见报告）
  ✅ P3 入口条目: 通过
```

### S12: 清理

- 删除 `.claude/temp/modified_files.txt`
- 删除所有 session-scoped 临时状态文件
- 输出最终报告摘要

## --dry-run 模式

执行 S1-S6（到纸条生成为止），但：
- S0 不执行 git stash
- S4/S8/S9/S10 不修改任何文件
- S7 展示纸条但不执行
- S12 正常清理临时文件

## 注意事项

- sync 是**有副作用的**——它会自动更新本模块管理文件。跑之前确认你已保存其他正在编辑的文件。
- 如果你在同一轮对话中多次调用 sync（针对不同批次的改动），每次只处理**自上次 sync 以来的新改动**。
- 如果接头总览模块文件夹不存在（新模块还没建接头总览），sync 会在被动区自动创建。
