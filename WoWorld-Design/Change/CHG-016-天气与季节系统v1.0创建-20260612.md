## 关键设计决策

| # | 决策 | 结论 |
|---|------|------|
| 1 | 数据流模式 | 统一轮询(WeatherQuery trait) — 零事件总线。天气变化(分钟级)远慢于轮询频率(帧级) |
| 2 | 解耦原则 | 天气只提供客观物理事实。NPC SkyPerception(主观)/魔法元素浓度(多因子)/战斗环境判定 — 各自模块自行推导 |
| 3 | 温度模型 | 双层 — regional_temperature(大气) + ground_temperature(群系修正)。NPC根据高程上下文自选 |
| 4 | 空间粒度 | 两层 — ClimateRegion(~50km,每游戏日) + LocalWeather(~1km,每游戏分钟)。砍掉全球气压层 |
| 5 | 状态机 | 6状态Markov链(Clear→HeavyStorm) + 雾独立布尔维度。转移矩阵参数化生成 |
| 6 | 极端天气 | 参数组合判定 — 不命名枚举。三层NPC响应(L1温和→L2警告→L3灾难硬约束) |
| 7 | 季节 | 120天/年·48分钟/天·均匀四季·春分开局·虚数年号 |
| 8 | 历史气象 | 天气产出气象异常(纯数据) → 历史模块转化为事件。约束:历史必须消费异常数据,不能独立随机 |
| 9 | 视觉 | WeatherVisualPacket(~200 bytes/帧) → 调制已有渲染系统,不新增 |
| 10 | 性能 | Rust ≤0.25ms + GPU ≤0.45ms + VRAM ≤2.5MB — 全部在技术栈v3预算内 |
| 11 | 跨模块 | 13消费方完整数据合同,全部通过WeatherQuery trait轮询 |

## 模块契约确立

天气与季节系统成为以下概念的权威 Owner：

| 概念 | 权威 Owner | 关键约定 |
|------|-----------|---------|
| 天气数据 | 天气系统 001/004 | WeatherQuery::sample() 统一入口 — 零事件总线 |
| 季节 | 天气系统 003 | SeasonClock 纯时间函数 — 120天/年[TUNING] |
| 温度 | 天气系统 001 | 双层(regional/ground) — NPC自选基准 |
| 极端天气 | 天气系统 002 | 参数组合判定 — 三层NPC响应 |
| 视觉输出 | 天气系统 001/004 | WeatherVisualPacket — 调制已有shader |
| 气象异常 | 天气系统 002 | 纯数据产出 — 历史系统消费转化 |
