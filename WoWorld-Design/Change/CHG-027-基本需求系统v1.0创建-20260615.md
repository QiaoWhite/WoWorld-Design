## 背景

NPC ver2.0 中，生理需求通过三条独立路径处理：硬阈值 GOAP 安全网（hunger<0.3→吃）、模糊的 physiology_modifier（未展开细节）、以及情绪引擎 physiology_pull。四个关键缺陷：

1. 悬崖效应——hunger=0.31 vs 0.29 的差异微小但行为截然不同
2. 无个体差异——所有 NPC 在同一 hunger 值反应一致
3. 需求间无耦合——真实生理学中的 ghrelin/leptin 调节链不存在
4. 无预测性——有智慧的 NPC 应预判需求而非被阈值驱动反应

本 CHG 将 4 个扁平阈值扩展为统一的 7 维需求框架，引入 3 个新需求维度,重构决策-需求映射。

## 核心设计

### 7 维需求框架

| # | 需求 | 性质 | 决策层 | 数据源 |
|---|------|------|--------|--------|
| ① | 饥饿 | 已有,机制升级 | GOAP<0.3 + PROB | Life.Vitals.hunger |
| ② | 口渴 | 已有,机制升级 | GOAP<0.25 + PROB | Life.Vitals.thirst |
| ③ | 疲劳/睡眠 | 已有,机制升级 | GOAP>0.9 + PROB | Life.Vitals.fatigue |
| ④ | 个人安全 | 已有,不改 | GOAP only | health + combat_threat |
| ⑤ | **元素平衡** | ★ 累积驱动 | PROB only | Life.Vitals.element_surplus[8] |
| ⑥ | **性/libido** | ★ 周期驱动 | PROB only | Life.Vitals.libido |
| ⑦ | **社交/归属** | ★ 心理驱动 | PROB only | NPC.NpcData.social_deficit |

### 统一数学：urgency = deviation × sensitivity

所有需求共享同一线性公式。个体差异来自大五人格查表（NeedSensitivity），多样性涌现自一次乘法+一次查表。

### 元素平衡——血元素合成的瓶颈模型

基于魔法系统公理（血元素=八元素复合材料），进食→吸收元素→血元素合成→多余元素累积→需有意识释放。瓶颈模型：最稀缺元素决定血元素合成效率→效率<1.0→饱腹恢复下降→饥饿和元素平衡产生张力。释放方式：主动释放(全清,瞬间)/睡眠被动释放(5%/晚)。

### ConsumableEffect — 消耗品效果数据合同

Life 模块定义 ConsumableEffect，物品模块存储。对齐已有 EnchantmentSchema/AssemblySchema 的"消费模块定义 schema, 物品模块存储"模式。

### 性能

- 存储增量: +48 bytes/NPC (100K NPC ≈ 4.8 MB, +0.34%)
- 热路径增量: ~60ns/NPC (< 0.5% L1 预算)
- VRAM: 0 (全在 CPU 侧)

## 与已有模块的关系

```
Life 模块 ──── 定义 Vitals (权威数据源) ──→ NPC 模块消费
              定义 ConsumableEffect ──→ 物品模块存储
              定义 RaceTraits 新字段

NPC 模块 ──── Physiology 派生 Vitals (主观感知层)
              need_action_match 消费 7 维 urgency
              GOAP 安全网保持不变 (仅生存需求)

物品模块 ─── ItemRegistry 存储 ConsumableEffect
              查询服务——对数据不解释
```

**依赖方向**：Life → Items (存储 ConsumableEffect, 正向依赖) / NPC → Life (消费 Vitals) / NPC → Items (查询 ConsumableEffect)。无循环。
