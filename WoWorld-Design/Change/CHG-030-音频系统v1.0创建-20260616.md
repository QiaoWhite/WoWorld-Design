## 一、概述

**音频系统 v1.0 创建**——WoWorld 的声学真理引擎。定义声音如何从物理动作中涌现、如何在介质中传播衰减、如何被听者感知和掩蔽、如何被打断。

此前音频设计碎片化分布在 6+ 个模块中（语言表达 TTS/VoiceProfile、天气环境音、生命听觉/发声、战斗听觉信息、魔法雷元素、文化节日音乐），无统一权威。本次创建统一了全部音频相关设计，建立模块间接口契约。

---

## 二、核心设计决策

1. **双层模型**：Rust 模拟层（声音作为物理现象）+ Godot 渲染层（声音作为玩家体验）——对标天气系统
2. **拉模型**：各业务模块不知道音频存在——写入 SoundFootprint/CurrentSpeech/action_log，音频模块主动轮询
3. **所有共享类型放 woworld_types**：17 个模块零直接依赖音频 crate
4. **SoundFootprint 描述物理非语义**：表面接触+运动+发声——音频模块拥有映射逻辑
5. **声音五分类**：WorldSound/Voice/Music/Notification/Accessibility
6. **对话时间维度**：内容(语言表达) 与 声音(音频模块) 分离——fraction_heard + clarity
7. **VoiceProfile 所有权迁至音频模块**：声学参数(模拟层)←音频，TTS渲染(渲染层)←语言表达保留
8. **跨 L 层传播**：强度门控 + 传播延迟——L2 龙吼可传至 L1
9. **11 子插件内部架构**：每个通过 pub(crate) trait 隔离——替换组件=重新 impl
10. **完全 TOML 化**：4 个配置文件驱动所有规则——Mod 替换=换 TOML+文件

## 五、关键架构决策

### 依赖方向

```
woworld_types（所有共享类型——零模块依赖）
     ↑
     ├── 各业务模块 import（写 SoundFootprint/CurrentSpeech/action_log）
     └── 音频模块 import（读 + 处理 → AudioQuery）
              ↓
         AudioRenderPacket → GDExtension → Godot 渲染
```

**★ 17 个业务模块零直接依赖音频 crate。**

### 所有权迁移

| 概念 | 旧归属 | 新归属 |
|------|--------|--------|
| VoiceProfile（声学参数） | 语言表达 012 | **音频模块** |
| VoiceEmotionModulation | 语言表达 012 | **音频模块** |
| VoiceManager（队列+打断） | 语言表达 012 | **音频模块** |
| audible_radius 六因子 | 语言表达 011 | **音频模块** |
| effective_audible_radius() | 语言表达 011 | **音频模块** |
| TtsEngine trait | 语言表达 012 | **语言表达保留**（渲染层） |
| TtsConfig | 语言表达 012 | **语言表达保留** |

### 对话分离

| 层 | 归属 | 内容 |
|----|------|------|
| 对话内容（说什么） | 语言表达 | ExpressionRef + 文本 + resolve_partial() |
| 对话声音（怎么被听到） | **音频模块** | 传播 + 时间 + 清晰度 + fraction_heard + 打断 |

对标：书的内容归语言表达，书的物理属性归物品系统。

## 六、性能影响

| 指标 | 预估值 | 对标 |
|------|--------|------|
| Rust CPU | ~0.17ms/frame | 天气 0.25ms 预算内 |
| Rust RAM | ~10.4 MB | 经济系统 ~8MB |
| Godot 音频资产 | ~50 MB（已有预算） | — |
| AudioRenderPacket | ~3 KB/frame | WeatherVisualPacket ~200B |

新增字段对已有模块的内存影响：每 NPC +~1.2KB（ActionRingBuffer 1024B + SoundFootprint 128B + 其他 ~80B）。10,000 NPC ≈ +12MB——已在 NPC 模块内存预算基础上扩充。
