## §一、变更动机

v4.0 §二十一 LOD 统一架构在 2026-06-18 确立（KCQ-022），但停留在纸面概要阶段。经本次深度审查，发现以下缺口迫使全面深化：

| # | 缺口 | 严重度 |
|---|------|--------|
| 1 | 距离→LOD 映射表缺失——各维度距离带不一致（NPC L1=50m vs 动画 L0=10m vs 植被 L0=200m） | 🔴 |
| 2 | `LODCoordinator` 无实现算法——冲突解决顺序、降级优先级、预算分配策略全空白 | 🔴 |
| 3 | `audio_lod` 仅 4 档（0-3），与其他角色维度不对齐；无意图驱动的远距语音覆盖 | 🔴 |
| 4 | 跨维硬约束不完整——缺少 `animation_lod ≥ skeleton_lod`、VRAM 压力输入、级联交互约束 | 🔴 |
| 5 | `BuildingLOD → scene_lod` 映射标记为 TBD（CHG-043） | 🟡 |
| 6 | VRAM 4.2GB/6GB=70%——多文档警告"若 LOD 架构不完整落地，VRAM 先于 CPU/GPU 成为瓶颈" | 🔴 |
| 7 | 跨 LOD 交互（级联问题）——NPC A(LOD 0) 与 NPC B(LOD 3) 物理碰撞/对话/战斗如何处理 | 🔴 |
| 8 | 过渡时序不一致——skeleton/render/animation/ai 各维度的升级/降级延迟未定义 | 🟡 |

### 2.1 场景线 — 8 层（0-7），倍增因子 ~2.7×，12km 覆盖

|-----------|------|------|------|------|------|------|-----|

**设计依据**：体素尺寸以 2 的幂递进（Transvoxel 数学约束）。各元素使用 scene_lod 的子集（建筑 6 层有效 + 2 层融合、植被 5 层有效 + 3 层融合、海洋 5 层、云 4 层）——scene_lod 是上限，不强制所有元素填满 8 层。

### 2.2 角色线 — 5 层（0-4），倍增因子 ~4×，800m+ 覆盖

| char_lod | 距离 | 骨骼 | 动画 | 渲染 | 物理 | AI | 音频（基础） |
|----------|------|------|------|------|------|-----|------------|
| 0 | 0-15m | 35骨 | 9层全栈 | 1500面 | 全碰撞+IK | 全 GOAP | 全传播 |
| 1 | 15-60m | 33骨 | 6核心层 | 800面 | 简化碰撞(5胶囊) | 轻量决策 | 简化衰减 |
| 2 | 60-200m | 28骨 | 4行为层 | 300面 | AABB | 统计模板 | 仅事件声 |

**设计依据**：基于 Hall 人际距离学（亲密 0-0.5m / 个人 0.5-1.2m / 社会 1.2-3.6m / 公共 3.6m+）+ 视觉角分辨率。LOD0 从原设计的 10m 扩展到 15m——战斗/广场/市场的最远有意义互动距离。

### 2.3 音频基础距离带（独立于 char_lod 其他维度——声音传播比视觉远）

| 4 | 1000m+ | 静默（无 intent boost 时） |

## §三、七维 LodPrescription（修订版）

每个实体一条，~8 字节。所有字段对齐原设计的 u8 布局：

```
scene_lod: u8      (0-7,  地形/建筑/海洋/云/植被统一)
skeleton_lod: u8   (0-4,  35→33→28→15→0 骨)
animation_lod: u8  (0-4,  9层全栈→6核心→4行为→2层→无)
render_lod: u8     (0-4,  1500面→800→300→impostor→无)
physics_lod: u8    (0-4,  全碰撞+IK→简化胶囊→AABB→AABB仅→无)
audio_lod: u8      (0-4,  ★修订: 4档→5档，与其他角色维度对齐)
ai_lod: u8         (0-4,  全GOAP→轻量决策→统计模板→统计聚合→仅存在)
```

## §四、跨维硬约束

```
skeleton_lod = 4  →  animation = 4, render = 4, physics = 4    (0骨不可逆)
scene_lod = N     →  skeleton_lod ≥ max(0, N-2)                 (场景降了骨架必须跟)
ai_lod ≥ 3        →  physics ≥ 3, animation ≥ 3                 (统计NPC不需碰撞/动画)
ai_lod = 4        →  animation = 4                              (无AI不播动画)
skeleton_lod ≥ 2  →  physics_lod ≥ 2                            (骨骼粗→物理粗, CHG-042)
physics_lod = 4   →  animation_lod = 4                          (无碰撞无动画, CHG-042)
animation_lod ≥ skeleton_lod  (动画模块内部 clamp — 0骨不播动画)
```

### 5.3 冲突解决算法（8 步，每帧执行）

```
Step 1 — 基础分配:
  对每个实体:
    distance = |entity.position - camera.position|
    scene_lod = scene_distance_table(distance)   // §2.1 查表
    char_lod  = char_distance_table(distance)    // §2.2 查表
    audio_lod_base = audio_distance_table(distance) // §2.3 查表

Step 2 — 硬约束:
  玩家 → 全维度 LOD 0（永久）
  战斗中 NPC → ai=0, physics=0, anim≤1（最低战斗保真度）
  玩家当前交互目标 → 拉升到 InteractionIntent 要求的最低 LOD

Step 3 — 级联拉升（预算制，见 §六）:
  InteractionIntent 驱动目标拉升
  AudioBroadcast 驱动听众 audio_lod 提升（广播优化：一次计算 → N 个接收）

Step 4 — Attention 加成:
  玩家视线锥（30°）内 → scene_lod -1, render_lod -1, anim_lod -1（各提升 1 档）
  玩家聚焦目标 → 全维度拉满到交互所需 LOD
  focus_listening 开启 → 视线方向 ±10° 内 audio_lod -1
  玩家转头朝向声源 → 自动对该声源实体施加 render_lod -1

Step 5 — VRAM 压力降级（先降场景——视觉代价最低）:
  VRAM ≥ 70% (0.70):
    scene_lod 5-6 → +1 (提前切 Signed Heightfield)
    植被 LOD0→LOD1 距离从 200m 缩至 150m
    非地标建筑在 scene_lod 4 处 → 强制 LOD2
  VRAM ≥ 85% (0.85):
    scene_lod 3-4 → +1
    植被再缩 25% (LOD0→LOD1 距离缩至 110m)
    非地标建筑全降 1 级
    纹理 mipmap bias +1
  VRAM ≥ 95% (0.95):
    全局 scene_lod +1（上限 7）
    全 NPC render_lod +1
    纹理 mipmap bias +2

Step 6 — 帧预算降级（先降实体计算——从远到近）:
  预算 < 3.0ms: scene_lod 3+ 区域内 NPC → anim +1, skel +1
  预算 < 1.5ms: scene_lod 2+ 区域内 NPC → ai +1, anim +1, skel +1
  预算 < 0.5ms: 全非关键 NPC → 全维度 +1（全局紧急降级）

Step 7 — 跨维约束 Clamp:
  对每个实体应用 §四 全部硬约束
  修正 Step 5-6 可能产生的约束违反

Step 8 — 迟滞:
  降级: 500ms 延迟（防 LOD 边界抖动）
  升级: 无延迟（响应性优先——不能让人等着 NPC "加载"）
  与 prev_prescriptions 比较: 同一实体同一维度在 500ms 内的降级请求被忽略
```

## §六、级联交互机制

### 6.1 核心原则

**单向·被动·按需拉升**：

- **单向**：只有高 LOD 实体能拉升低 LOD 实体。LOD 2 NPC 不能主动发起交互——不产生反向级联
- **被动**：被拉者只升到交互所需的最低 LOD，不是升到 LOD 0
- **按需**：不同交互类型需要的最低 LOD 不同（见 `InteractionIntent` 各变体的 required_lod）

### 6.2 预算制

```rust
const MAX_UPGRADE_COST_MS: f32 = 0.3;  // 级联拉升总预算上限

fn process_cascade_upgrades(
    interactions: &[InteractionIntent],
    prescriptions: &mut HashMap<EntityId, LodPrescription>,
    frame_budget: &FrameBudget,
    player_relations: &RelationMap,
) {
    let mut budget_ms = (frame_budget.remaining_ms * 0.5).min(MAX_UPGRADE_COST_MS);

    // 按优先级排序: Combat > PlayerDirect > NPCConversation > Trade > Broadcast
    let sorted: Vec<_> = interactions.iter()
        .sorted_by(|a, b| cascade_priority(b, player_relations)
                           .partial_cmp(&cascade_priority(a, player_relations))
                           .unwrap())
        .collect();

    for intent in &sorted {
        let required = intent.required_lod();
        let current = &prescriptions[&intent.target];
        let cost = estimate_upgrade_cost(current, &required);

        if cost <= budget_ms {
            prescriptions.get_mut(&intent.target).unwrap().upgrade_to(&required);
            budget_ms -= cost;
        }
        // 预算耗尽 → 剩余 targets 保持低 LOD
        // → "太远了/太忙了够不着" —— 涌现，非 bug
    }
}

fn cascade_priority(intent: &InteractionIntent, relations: &RelationMap) -> f32 {
    let type_priority = match intent.kind {
        Combat => 100.0,
        PlayerDirect => 80.0,
        Conversation => 60.0,
        Trade => 50.0,
        PublicSpeech => 40.0,    // 单个听众优先级低于直接对话
        CasualAcknowledgment | Theft => 0.0,  // 不参与拉升
    };
    let importance = relations.importance(intent.target); // 已有数据
    let relevance = 0.5 + importance;                     // [0.55, 1.5]
    type_priority * relevance
}

fn estimate_upgrade_cost(from: &LodPrescription, to: &LodPrescription) -> f32 {
    let ai_gap = to.ai_lod.abs_diff(from.ai_lod) as f32;
    let anim_gap = to.animation_lod.abs_diff(from.animation_lod) as f32;
    let phys_gap = to.physics_lod.abs_diff(from.physics_lod) as f32;
    ai_gap * 0.05 + anim_gap * 0.08 + phys_gap * 0.03
}
```

**注意**：预算上限不是固定数字——来自帧预算剩余容量的 50%。正常帧可能撑 20 个轻微拉升（ai 2→1），战斗帧可能连 3 个全拉升都勉强。数字由预算决定，不写死常量。

### 6.3 升级重建

```
NPC B (ai_lod=2) 被 NPC A (ai_lod=0) 搭话:

1. A 的 GOAP 产生 InteractionIntent { target: B, kind: Conversation }
2. LODCoordinator 计算: Conversation → B 需要 ai ≤ 1, skeleton ≤ 2, anim ≤ 2
3. 预算检查通过 → B 的 LodPrescription 被拉升
4. B 的 AgentSnapshot 从以下来源确定性重建（1-3 帧, <50ms）:
   — 上次缓存的 AgentSnapshot（CHG-042 §7.1: LOD 2 每 1-5s 刷新, 缓存可用）
   — LifeEntity 状态（年龄/健康/职业——一直都在）
   — 位置: 从 ProfessionSchedule + NpcAnchor + game_time 确定性生成
5. B 的 GOAP 被临时激活（ai_lod=1），对"被搭话"做出反应
6. 互动结束 → B 的 LOD 回落 → AgentSnapshot 再次压缩缓存
```

## §七、音频 LOD 完整公式

### 7.1 有效音频 LOD

```
effective_distance = raw_distance × env_acoustic_modifier

env_acoustic_modifier:
  峡谷 0.6 / 密林 2.5 / 石殿 0.4 / 水下 3.0 / 平原 1.0 / 市集 1.8 / 山顶 0.7

base_audio_lod = audio_distance_table(effective_distance)  // §2.3

effective_audio_lod = max(0, base_audio_lod
                           - intent_boost_steps
                           - hearer_attention_bonus)

intent_boost_steps (来自 CommunicationIntent, CHG-018):
  whisper: 0, normal: 0, loud: 1, shout: 2, declamation: 3

hearer_attention_bonus:
  睡眠=0, 专注工作=0, 闲散=1, 专注聆听=1
  hearing_acuity: 年龄派生, 0.0-1.0 (Gompertz 模型已有)
  ambient_noise: 市集高衰减 / 森林低衰减 (WeatherQuery 已有)
```

### 7.2 非语言声源（钟声/打铁/瀑布/兽吼）

不通过 CommunicationIntent——通过 **`AcousticProjection`**（CHG-030 音频系统扩展）：

```rust
struct AcousticProjection {
    base_loudness: f32,           // dB
    directionality: Directionality, // Omni / Cone(angle) / Beam(angle)
    freq_profile: FreqProfile,    // LowFreqDominant(远传) / HighFreqDominant(近衰减快)
}
```

`AcousticProjection` 修正 `effective_distance`：
- base_loudness 每 +10dB → effective_distance × 0.7
- LowFreqDominant → effective_distance × 0.8

### 7.3 广播优化

```rust
struct AudioBroadcast {
    source_position: Vec3,
    source_entity: Option<EntityId>,
    acoustic: AcousticProjection,
    intent: Option<CommunicationIntent>,  // None = 非语言声源
    max_range: f32,
}

// LODCoordinator 处理:
// 对每个 broadcast:
//   for entity in entities within max_range:
//     计算 effective_distance → effective_audio_lod
//     写入 entity.audio_lod = min(entity.audio_lod, effective_audio_lod)
// 单次传播计算 → N 个接收端共享
```

### 7.4 玩家"专注聆听"

```
触发: 按住 Alt / 手柄 LB
效果:
  — 视线方向 ±10° 锥内所有声源 → audio_lod -1
  — 非视线方向 → audio_lod 不变 (注意力聚焦 = 周边听觉下降)
  — 移动速度 ×0.3
  — UI: 暗角 + 声源方向波形指示器
实现:
  PlayerInput.alt_held → PlayerAttention.focus_listening = true
  → LODCoordinator Step 4 处理
```

## §八、过渡时序

### 8.1 分维度延迟

| 维度 | 升级延迟 | 降级延迟 | 机制 |
|------|---------|---------|------|
| scene_lod | 0ms | 500ms 迟滞 | Clipmap 预计算，指针切换 |
| skeleton_lod | ≤200ms | 0ms | 骨骼矩阵瞬间切换，升级时等 render dither |
| animation_lod | 200-500ms | 0ms | 逐层 blend_weight 渐增，降级直接关层 |
| render_lod | 0ms（触发 dither） | 200ms alpha dither | dither 期间视觉过渡，升级时 render 先升 |
| physics_lod | 0ms | 0ms | 碰撞胶囊无状态，瞬间创建/销毁 |
| audio_lod | 0ms | 0ms | 传播模型无状态 |
| ai_lod | 1-3 帧 (<50ms) | 0ms | AgentSnapshot 确定性重建 |

### 8.2 时序规则

```
降级（远离）: skeleton 先降 → render 跟降
  理由: 避免高精度骨骼驱动已降级 mesh 的浪费
  代价: ≤16ms (1帧) 的不匹配，不可见

升级（靠近）: render 先升 (alpha dither 开始) → skeleton 在 dither 期间跟进
  理由: 低面数 mesh 被高精度骨骼驱动 → "低清但动作流畅" → 可接受
  最坏: 200ms (dither 期间) 低清+流畅 → 自然过渡
```

## §十一、生命周期 L1-L4 ↔ ai_lod 映射

```
生命周期 L1 (0-50m)   ↔  ai_lod 0-1     advance_age: 每帧, 逐NPC
       (50-150m)      ↔  ai_lod 1        生命周期无 L2→此带按 L1 处理(advance_age 极廉价)
生命周期 L3 (150m-5km) ↔  ai_lod 2-3     advance_age: 月批, 批处理
生命周期 L4 (5km+)     ↔  ai_lod 4       advance_age: 年批, 仅统计
```

**核心不变量**：所有层级共用完全相同的时间流速。批处理只是把累积的 delta 一次算完，不是加速时间。`advance_age` 是纯函数，对 LOD 架构零感知。

## §十二、VRAM 监控

### 12.1 VramLedger（Rust 侧记账，非 GPU 查询）

```rust
// woworld_core
struct VramLedger {
    categories: [AtomicU64; 6],  // terrain/npc_mesh/building/vegetation/textures/misc
    total_used: AtomicU64,
}

impl VramLedger {
    fn record_load(&self, category: Category, bytes: u64) { ... }
    fn record_unload(&self, category: Category, bytes: u64) { ... }
    fn pressure_ratio(&self) -> f32 { ... }
    fn predict_10fr(&self, pending: &[AssetLoadRequest]) -> f32 { ... }
}
```

埋点位置：所有资产加载/卸载路径（chunk 加载器、NPC 实例化器、建筑 WFC 求解器、植被 TC 展开器）。±15% 精度足够——VRAM 降级关心趋势和量级，不关心精确字节数。

### 12.2 降级路径

VRAM 和帧预算是两个独立压力源，走不同的降级路径：
- **VRAM 优先降场景**（场景吃掉最多 VRAM——地形 1.2GB + 建筑 0.3GB + NPC 0.3GB）
- **帧预算优先降实体计算**（NPC AI/动画吃掉最多 CPU）

两个压力源可以同时作用于同一个实体——这是正确行为。如果场景被激进降级到 scene_lod=6（32m 体素），远处 NPC 确实应该被双重惩罚——它们在低精度场景中已经不可见了，保留动画/AI 是纯浪费。跨维约束强制执行这一点。

## §十三、设计原则（不可违反）

1. **分离关注点**：LODCoordinator 管资源分配（"这个实体本帧有多少计算预算"）。各模块管实现策略（"给定预算，砍哪些细节"）。消费者管信息后果（"给定退化输入，还能读到什么"）。
2. **零领域知识**：LODCoordinator 不直接理解 RitualDef、CombatStyle、ProfessionSchedule。重要性信号通过 ai_lod、InteractionIntent、relation_importance 上传。
3. **确定性**：同一输入 → 同一输出。VRAM 和帧预算的预测值使用上一帧快照，不实时查询 GPU。
4. **不删除原有设计**：所有现有模块的 LOD 约定通过映射/派生关系收敛到统一框架，不强制合并。
5. **玩家 = NPC**：同一套 LOD 机制——玩家永远 LOD 0（全精度），但玩家的感知也受 LOD 约束（远处 NPC 看不清/听不清，除非专注聆听）。
