## 一、动机

物品系统 v1.0 (CHG-011~018) 预留了 `FurnitureItem`(0x54) 类别占位——"家具作为物品形态，放置后变成 Furniture 实体"。但：

- **Furniture 实体从未被定义**——它是什么？它的数据模型是什么？它在哪个模块？
- **建筑模块**明确说"家具不属于建筑模块"，但提供了 `Surface` trait 和 `BuildingQuery::surfaces_in_region()`——给谁用？
- **世界生成**提到"内饰渐进生成"——但怎么生成？在哪个 Pass？消费什么数据？
- **NPC** 需要知道面前的东西能不能坐——当前没有 affordance 查询
- **魔法**需要知道家具的元素构成——当前 MaterialProperties 有 `elemental_affinity` 但总纲中物品到材质的引用链断了

本次重写彻底填补这一概念缺口，将早期的 [[../WoWorld-Design/Happy Game/开发阶段/家具系统设计-旧|家具系统设计 (旧·v0.1)]]（手工枚举 50+ 种家具）替换为符合 WoWorld 设计哲学的**参数化家族 + 文化派生 + 种子确定性生成**。

## 四、核心设计决策摘要

1. **统一放置实体连续谱**——不区分"建筑组件"和"家具实体"，差异在属性梯度（Graspability: Immovable/Attached/Free）而非类型
2. **一切皆物品**——所有被摆放实体有 ItemEntId，物理原子层的 35 个原子统一操作 EntityId
3. **参数化家族**——9 家具族 × ~27 mesh 模板 × 连续参数 × 文化扭曲 = ~12,000+ 视觉变体。开发和 Mod 通过 TOML 数据驱动，非 Rust 代码
4. **三层文化派生**——CultureCoreParams → CultureFurnishProfile → FurnishEnsemble → 逐件采样
5. **双 Pass 种子确定性**——Pass A(附着家具)/Pass B(自由家具)/Pass C(野外)，全量预生成写入 LMDB
6. **工作站 u16 注册制**——物品系统只存 u16 标签，领域模块注册并解释。完全解耦
7. **雕塑例外**——底座参数化 + 主体标记化目录(~66 AI mesh)，通过 SubjectTag 选择

## 五、woworld_core 新增类型概览

| 类型 | 用途 |
|------|------|
| `EntityId` (变更) | 1-bit 物品命名空间 + 3-bit entity_kind |
| `AffordanceSet` (新增) | 64-bit 物理供给标记——<5ns 位过滤 |
| `SurfaceId` (新增) | 4 变体紧凑编码——Building/Terrain/Feature/Furniture |
| `Surface` trait (新增) | 可放置表面的统一查询接口 |
| `SurfaceProvider` trait (新增) | 表面提供方——BuildingStore 和 PlacementStore 均实现 |
| `EntityIndex` trait (扩展) | 新增 affordances_of/item_def_of/workstation_tag_of/graspability/try_acquire |
| `PlaceError` / `PickUpError` / `Graspability` (新增) | 放置/拾取/抓取相关错误与枚举 |
| `WorldEvent` 扩展 | 新增 ItemPlaced/ItemPickedUp/ItemMoved/SurfaceDestroyed/ContainerOpened/WorkstationUsed |

## 六、性能影响

|---|-----|
