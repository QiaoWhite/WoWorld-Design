## 一、背景与动机

### 1.1 存档系统长期缺失

截至 CHG-054，WoWorld 的 26 个独立模块中**没有独立的存档系统设计文档**。持久化约定散落在各模块的性能预算文档中，引用一个"似乎存在但未完整编写的规范"：

- NPC 模块定义了 `SaveSystem` 结构体（含 `lmdb::Environment`）
- 历史模块设计了独立 LMDB 文件 `world_<seed>_history.lmdb`
- 性能分析文档提到"NPC 数据：JSON Lines 格式，增量写入"
- 世界生成 P13 包含 `write_to_lmdb()` 函数
- CHG-054 原计划中包含"存档系统 v1.0（C2 项 · Tier 0 最高优先级）"但未创建

### 1.2 核心矛盾

在 `/grill-me` 访谈和 11 轮迭代审计中，识别出 7 个关键矛盾：

1. **存储预算不一致**：物品系统 1.7GB vs 世界生成 140MB vs 性能分析 <3GB——累计预算对不上
2. **序列化格式冲突**：性能分析说"JSON Lines"，所有其他文档说 bincode
3. **LMDB 架构不统一**：历史模块独立 LMDB 文件 vs NPC 模块暗示统一 LMDB 环境
4. **存储策略不一致**："种子确定性零存储"vs"全量预生成写入 LMDB"——不同模块采用不同策略
5. **Wear 状态机缺失**：技术栈声明存在但无设计文档
6. **数据老化策略碎片化**：不同数据的归档/过期策略不统一（建筑 10/50 年、死亡 30 天、认知无过期）
7. **PerceptualCache 不持久化**：读档不是真正的"暂停/继续"——确认这是设计选择而非 bug

## 二、核心设计决策

### 2.1 存储策略：全量快照 + 脏数据增量

- **否定了"种子零存储"方案**（种子重现作为主要存储策略）
- 原因：版本迁移地狱、确定性脆弱、读档时间不可控
- 采用：初始状态全量存储 + 后续只写脏数据增量
- 种子从"空间优化手段"重新定位为"容错恢复手段"

### 2.2 文件架构：单文件 LMDB + 多 named_db

- 否定了历史模块的"独立 LMDB 文件"方案
- 原因：跨模块原子写入优先于独立文件的损坏隔离
- 采用：一个 `.woworld` = 一个 LMDB 环境，内含 14+ 个 named_db

### 2.3 模块契约：SaveableModule trait

- 存档系统不知道任何模块的具体类型
- `SaveableModule` trait：8 个方法，2 个必覆（`module_name`、`current_version`），6 个有默认实现
- `snapshot_dirty(&self)` + `confirm_snapshot_written(&mut self)` 两阶段消除竞态

### 2.4 其他关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 序列化格式 | 存档系统不规定（各模块自选） | 解耦最大化 |
| 版本管理 | 全局（文件结构）+ 模块级（数据格式）两级 | 改一个模块不影响其他 |
| 存档槽 | 手动无上限 + 自动轮转(3) + 快存(1) + 退出(1) + Initial(1) | 玩家完全自由 + 崩溃恢复 |
| 玩家死亡 | PendingInheritance → 血缘继承或灵魂转世 | 世界永远可继续 |
| 世界发现 | 目录扫描（无 worlds.json） | 文件系统即真相 |
| Mod 兼容 | 删 Mod → 冻结/替换降级 | 不丢数据 |
| 崩溃恢复 | 临时文件+原子重命名 / 覆盖前备份 / session.lock | 三件套互保 |

## 五、保留矛盾（设计选择，非 bug）

| # | 矛盾 | 选择 | 理由 |
|---|------|------|------|
| 1 | 数据老化策略不统一 | 各模块自治 + 参考基准 | 不同数据自然寿命不同 |
| 2 | PerceptualCache 不持久化 | 读档重建 | 空缓存 → 新奇探索涌现 |
| 3 | Wear 状态机不存在 | 各模块自管 | 四种磨损无统一物理规律 |
| 4 | 多个 LMDB 文件 → 单文件 | named_db 统一 | 原子写入优先 |
| 5 | 种子确定性 ≠ 存储策略 | 全量快照，种子容错 | 版本迁移和读档时间优先 |
