## 一、背景

CHG-055 创建了存档系统 v1.0（6 文档，~5,500 行）。在 `/grill-me` 启动的 10 轮迭代审计中（R1→R10），对存档系统进行了全面的跨模块深度审计。审计覆盖：

- 存档系统 6 份文档的内部一致性
- SaveableModule trait 设计的完备性
- 与 14 个消费模块的接口契约
- Rust 解耦原则、线程安全、内存安全
- 玩家体验（死亡/继承/连续存档/世界管理）
- 性能（Initial 存档 OOM、惰性迁移尖刺、流式写入）

## 二、审计发现与修正（24 项）

### 2.1 Trait 新增（9 方法）

| # | 方法 | 签名 | 说明 |
|---|------|------|------|
| P1 | `estimate_dirty_bytes` | `-> Option<u64>` | 存档前空间估算，±50% 精度 |
| P2 | `named_dbs` | `-> &[(&str, &str)]` | **(必须覆盖)** 声明写入目标 (db_name, key_prefix) |
| P4 | `reset_to_default` | `&mut self` | load() 失败后清理半加载状态 |
| P5 | `validate_after_migration` | `-> Result<Vec<MigrationWarning>>` | 迁移后轻量验证，<1s |
| P7 | `write_dirty` | `&self, txn: &mut RwTransaction` | 流式增量写入，避免内存峰值 |
| P15 | (文档约束) | — | validate_after_migration <1s |
| P17 | `pending_save_count` | `-> u8` | UI 连续存档提示 |
| P19 | `load` 签名变更 | `&mut self, ctx: &LoadContext` | 渐进加载支持（LoadContext 提供 txn_factory） |
| P22 | `write_initial` | `&self, txn: &mut RwTransaction` | Initial 存档直接流式写入（不经 Phase 1 内存收集） |

**新增 `LoadContext` 类型**：
```rust
struct LoadContext<'env> {
    txn: RoTransaction<'env>,
    create_txn: Box<dyn Fn() -> Result<RoTransaction<'env>> + 'env>,
}
```
模块在 `load()` 中可调用 `ctx.create_txn()` 获取独立读事务做按需/渐进加载——不依赖 `&SaveSystem` 引用（零耦合）。

## 三、SaveableModule trait v2.0（最终形态）

```rust
/// 模块与存档系统的唯一契约。共 14 方法。
/// 必须覆盖: 4 方法 (module_name, current_version, named_dbs, key_prefixes)
/// 可选覆盖: 10 方法（均有合理默认值）
trait SaveableModule: Send + Sync {
    // ══ 必须覆盖（4） ══
    fn module_name(&self) -> &'static str;
    fn current_version(&self) -> u32;
    /// (db_name, key_prefix)。空 = 无持久化。组合必须在所有注册模块中唯一。
    fn named_dbs(&self) -> &[(&str, &str)];

    // ══ 元数据（2） ══
    fn is_critical(&self) -> bool { false }
    /// ±50% 精度。None = 不参与估算。
    fn estimate_dirty_bytes(&self) -> Option<u64> { None }

    // ══ 保存（5） ══
    fn has_dirty(&self) -> bool { false }
    fn snapshot_dirty(&self) -> Result<DirtySnapshot> { Ok(DirtySnapshot::empty()) }
    /// 流式写入脏数据。大模块覆盖以避内存峰值。
    fn write_dirty(&self, txn: &mut RwTransaction) -> Result<()> { /* 默认委托 snapshot_dirty */ }
    /// Initial 存档专用——写入全部初始数据。
    fn write_initial(&self, txn: &mut RwTransaction) -> Result<()> { /* 默认委托 snapshot_dirty */ }
    fn confirm_snapshot_written(&mut self, snapshot: &DirtySnapshot) {}

    // ══ 读档（3） ══
    /// ctx 提供读事务 + 事务工厂（渐进加载）。
    fn load(&mut self, ctx: &LoadContext) -> Result<()> { Ok(()) }
    /// load() 失败后由 SaveSystem 调用。
    fn reset_to_default(&mut self) {}

    // ══ 迁移与验证（2） ══
    fn migrate(&self, from_version: u32, txn: &RwTransaction) -> Result<()> { Ok(()) }
    /// 迁移后验证。<1s 内完成。
    fn validate_after_migration(&self, txn: &RoTransaction) -> Result<Vec<MigrationWarning>> { Ok(vec![]) }
}
```

`SaveableModule` 是 object-safe 的（所有方法返回 Sized 类型或 `()`），可继续作为 `&mut dyn SaveableModule` 使用。

## 五、性能影响

所有新增逻辑均为 O(1) 或 O(named_dbs)（≤14）。无新增热路径开销：
- `named_dbs()` 一致性验证：O(14) ≈ <1µs，Phase 1 结束时执行一次
- `estimate_dirty_bytes()`：各模块自行估算，无 SaveSystem 开销
- `validate_after_migration()`：仅版本迁移时执行（<1s 预算，极低频率）
- `key_prefix` 冲突检测：模块注册时执行一次（O(模块数²) ≤ 196 次比较 ≈ <1µs）
- `pending_save_count()`：直接返回 u8 字段，零计算

## 六、向后兼容

CHG-055 的 `SaveableModule` trait（8 方法）与本次 v2.0（14 方法）的差异：
- `module_name()` 和 `current_version()` 保持不变
- 新增的 `named_dbs()` 为必须覆盖——CHG-055 中的模块尚未实现此 trait，不构成兼容性负担
- `load()` 签名从 `(&mut self, &RoTransaction)` 变为 `(&mut self, &LoadContext)`——但没有任何模块已实现此方法（存档系统 v1.0 刚创建，无消费者）

**无向后兼容性负担。**
