## 动机

WoWorld 的核心设计是"NPC 与玩家无本质区别"——NPC 可以建造、破坏、耕耘、挖掘，正如玩家一样。这要求地形不仅是静态的噪声高度场，而是一个**可被持续修改的世界级稀疏密度场**。

在 /grill-me 访谈中，用户提出了三个关键问题：
1. 地形为什么不使用 ECS 架构？
2. 大量 NPC 在超大范围内的地形修改如何高效支持？
3. 永久性修改在超远视距下如何正确呈现？

经过业界调研（Veloren MMO 体素 RPG、bevy_voxel_world、Ethertum 等）和五轮设计批判，确立了 **"地形内核不转 ECS，编排层入 ECS"** 的架构决策。

## 架构决策

### 内核不转 ECS

**决策**：地形数据（高度场、密度场、海洋深度）保持为 ECS Resource，不进入 Archetype 存储。

**理由**：
- 业界一致：Veloren（最大规模体素 MMO）将 `TerrainGrid` 作为 ECS Resource，Chunk 数据有独立的稀疏存储结构
- 地形是"世界级单例"——不是"有生命周期和状态变化"的实体
- ECS 优化的是"很多小东西"的缓存友好批处理，不是"一个大东西"的空间查询
- Godot 团队明确解释为什么不 ECS 化地形/物理/渲染子系统
- 地形查询的主要开销是密度场计算（5-20 倍于 mesh 提取），ECS 不加速此过程
- 无状态纯函数噪声查询已经是最优的——不需要 Archetype 存储的额外间接层

### 编排层入 ECS

**决策**：地形修改的**生命周期管理**走 ECS——用 Component 追踪修改请求，用 System 编排 dirty→remesh 流水线，用 Event 通知 NPC 感知。

**核心类型**（`woworld_core::edit_terrain`）：

| 类型 | 角色 |
|------|------|
| `EditDensitySnapshot` / `EditDensityBuilder` | CoW 稀疏 3D 密度修改（`Arc<HashMap<IVec3, (f32, u8)>>`） |
| `EditHeightfieldSnapshot` / `EditHeightfieldBuilder` | CoW 2D 表面高度投影（`Arc<HashMap<(i32,i32), f32>>`） |
| `EditTerrainSnapshot` / `EditTerrainBuilder` | 组合两者 + `DirtyChunkQueue` |
| `ModificationBatch` | 帧内批量修改积累（Veloren BlockChange 模式） |
| `DirtyChunkQueue` | Chunk 脏标记队列 + 两级索引 |
| `EditDensityLayer` | `DensityProvider` 桥接——使 CoW 快照可插入 `DensityStack` |

**关键设计决策**：

| 决策 | 理由 |
|------|------|
| Copy-on-Write（`Arc` 原子交换） | 一次 Transvoxel 提取做 ~36K 密度采样——每采样获取一次 `RwLock` 读锁不可接受。CoW 提供零读锁开销 |
| 双层存储（3D 密度 + 2D 高度投影） | 3D 密度场供 Transvoxel 提取和 raycast；2D 高度投影供 `height_at()` O(1) 快速查询（NPC 每帧调用） |
| 批量化修改（`ModificationBatch`） | 爆炸一次修改数百体素——逐个体素发 Component 不可行。帧内积累，帧末一次 flush |
| `DensityStack::material_at` 材质组合 | 从高 priority 向低查找"所有权"层——EditDensity (priority 10) 覆盖 CaveDensity (4) 覆盖 TerrainBaseDensity (0) |
| 密度手动叠加（非栈内插入） | `HeightfieldTerrain::density_at` 手动叠加 EditDensity 增量——避免将 EditDensity 永久插入 `DensityStack` 后的锁问题，同时 VoxelChunk rayon job 在自己的临时栈中包含 EditDensityLayer |

### 修改流水线

```
玩家挖掘 / NPC 建造 / 爆炸
    │
    ▼
[Phase 0: 积累] ModificationBatch::push()
    │
    ▼
[Phase 1: 应用] modification_collector_system (ECS) →
    验证权限 → EditDensityBuilder → EditHeightfieldBuilder →
    freeze() → EditTerrainSnapshot → DirtyChunkQueue 标记
    │
    ▼
[Phase 2: 同步] WorldDriver →
    terrain.set_edit_terrain(snapshot) →
    ocean.set_edit_heightfield(snapshot)
    │
    ▼
[Phase 3: 重提取] drain DirtyChunkQueue →
    submit_voxel_job (rayon) →
    Transvoxel + EditDensityLayer →
    ArrayMesh 更新
    │
    ▼
[Phase 4: 通知] SpatialEventBus::push(TerrainModified) →
    NPC 感知系统 → 更新世界模型
```

### 2D/3D 分裂的修复

**问题**：`HeightfieldTerrain` 原先的 `height_at()` 走裸噪声（`terrain_height()`），`density_at()` 走 `DensityStack`。修改后两者不一致——NPC 可能站在修改过的表面上但 `height_at` 返回原始高度。

**修复**：所有 9 个 `TerrainQuery` 方法现在均通过 `actual_height_at()` 或手动密度叠加感知 EditTerrain：

| 方法 | 修改路径 |
|------|---------|
| `height_at` | `actual_height_at` → EditHeightfield 优先，回退 `terrain_height` |
| `normal_at` | `actual_height_at` ×3 → 修改感知的表面法线 |
| `density_at` | `density_stack.density_at` + EditDensity 增量 |
| `terrain_raycast` | 走 `density_at` ✅ 已修复 |
| `is_walkable` | `actual_height_at` + 修改感知法线 |
| `surface_material_at` | EditDensity 材质 → DensityStack 材质 → 噪声回退 |
| `medium_at` | `actual_height_at` 判断水/空 |
| `sample_vertex` | `actual_height_at` + `actual_material_at` |
| `sample_horizon` | 走 `density_at` ✅ 已修复 |

同样修复了 `HeightfieldOcean` 的 `terrain_height`（原先直接调用 `noise.sample_height`）和 VoxelChunk 提取（原先创建不含 EditDensity 的临时栈）。
