## 动机

Sprint-061 实现"NPC 头顶自言自语气泡"MVP 时，审计发现 **"Bark" 一词在设计文档中被两个正交概念重载**，若直接编码会造成跨模块类型冲突：

1. **UI 文字气泡**（眼睛看）——`UI与UX系统/002` §"Bark 气泡" + `005` 的 `BarkEvent { speaker_id, text, emotion, priority }`。NPC 头顶浮现的**文字**。
2. **音频语声**（耳朵听）——`音频系统/007` 的 `BarkType`（12 类语声：Pain/Exertion/Anger/Fear/Joy…），是 SoundFootprint 的一个 sound_type 变体，是**非语言声音**。

二者都叫 "Bark"，但语义正交：一个是渲染到屏幕的文字，一个是播放的音效样本。按 CLAUDE.md 铁律"所有跨模块共享类型归 `woworld_core`"，两个 `BarkType` 将来必在同一 crate 真冲突。

## 消歧决策

**不删除任何原有设计**（遵循冲突修正原则）——通过词根分离消除命名碰撞：

| 概念 | 归属模块 | 词根 | 关键类型 |
|------|---------|------|---------|
| UI 文字气泡 | UI/UX 系统 | `speech_bubble` / `Bubble` | `BubbleType`、`SpeechBubbleEvent`、`SpeechBubbleState` |
| 音频语声 | 音频系统 | `bark` / `Bark` | `BarkType`（12 类，保持不变） |

- `音频/007` 的 `BarkType` 是 v1.0 权威规格，**保持不变**，占用 `Bark` 词根。
- UI 侧 `002`/`005` 的气泡类型全部改用 `SpeechBubble` / `BubbleType` 词根。

## 代码实现（Sprint-061·MVP）

### 架构决策：pull 替代 signal push

`UI/005` 原规定用 GDExtension `signal bark_emitted` **push** 到 GDScript。**本实现改用现有 pull 管线**：

- 理由：push 会把气泡节点的创建/生命周期逻辑放进 GDScript → 违反宪法 §2（Godot 侧无游戏逻辑）。
- 方案：复用 `entity_visual_system → EntityRenderer::sync` 的每帧 pull 管线（与名字系统同款），气泡数据搭 `EntityVisual` 顺风车，逻辑全留 Rust 侧。

### 新增类型

| 类型 | 位置 | 角色 |
|------|------|------|
| `BubbleType`（Normal/Emotion/Ambient/Quest/Damage）+ `color()` | `woworld_core::speech_bubble` | 气泡内容类型 → 文字颜色（对应 002 §5 类颜色表） |
| `EntityVisual.bubble_text` / `bubble_color` | `woworld_core::entity_visual` | pull 数据合同新增字段 |
| `SpeechBubbleState` / `BubbleSlot` / `ActiveBubble` | `woworld_ecs::resources::speech_bubble_state` | 跨帧气泡状态 Resource（含 duration/cooldown） |
| `speech_bubble_system` / `pick_bubble` | `woworld_ecs::systems::npc::speech_bubble` | 从 needs/emotion/goal 生成桩化文本，prng 错峰 + retain 防泄漏 |

### 关键实现约束

- **调度顺序**：`speech_bubble_system` 在 `entity_visual_system` 之前调用（同帧写→同帧读，避免滞后一帧）。
- **被夺舍 NPC 跳过**：`speech_bubble_system` 传入 `player_entity` 并跳过，防退出夺舍时闪现残留气泡。
- **第五类型名**：用 `Damage`（对齐 002"伤害反应'啊！'红字"原文），非草稿的 `Combat`。
- **层次说明**：`BubbleType` 是**渲染颜色分类**（002 颜色表的编码），非 005 数据合同层的语义类型（那是 `emotion` 字段）。MVP 桩化简化，未来由 `emotion` 驱动取代。
