## 三、依赖方向（Rust 解耦检查）

```
依赖图 (→ = "导入/知道的模块"):

物品模块 → 不导入任何模块 (只提供 ItemRegistry + 存储 ConsumableEffect)
Life 模块 → 不导入物品/文化/信仰/NPC (只定义 ConsumableEffect/函数签名)
文化模块 → 不导入 Life/NPC
信仰模块 → 不导入 Life/NPC
魔法模块 → 不导入 Life/NPC (元素浓度场是 Magic 自己的)
NPC 模块 → 导入 Life (Physiology), 调用 CultureQuery/FaithQuery
模拟引擎 → 导入所有模块 (胶水层, 负责协调)

✓ 无循环依赖
✓ Life 不依赖 NPC
✓ Life 不依赖物品 (正向依赖: 物品模块注册 Life 定义的类型)
✓ NPC 只消费 Life 数据,不修改
✓ 所有新增接口沿着已有 trait 通道走
```

**关键原则重申**：
- 本模块（基本需求系统）不创建独立模块——它是对 Life/NPC/物品已有机制的重新组织和扩展
- 所有新增数据必须以已有模块的已有结构体为载体
- ConsumableEffect 由 Life 模块定义（权威）,物品模块存储（服务）,这和对 EnchantmentSchema 的处理相同

## 四、接口修改总结

### 需要修改的已有 trait

| Trait / 结构体 | 修改类型 | 修改内容 |
|---------------|---------|---------|
| `Life.Vitals` | 新增字段 | element_surplus: [f32; 8], libido: f32 |
| `Life.RaceTraits` | 新增字段 | blood_element_ratio: [f32; 8], libido_type: LibidoType, libido_cycle_days: u32 |
| `NPC.Physiology` | 新增字段 | element_balance_urgency: f32, libido: f32 |
| `NPC.NeedColumn` ★ v1.1 | 新增 SoA 列族 | social_deficit: f32, need_sensitivity: NeedSensitivity |
| ~~`NPC.NpcData`~~ | ~~新增字段~~ | ~~social_deficit: f32~~ ★ v1.1 已迁移至 NeedColumn |
| `NPC.ActionDef` | 新增字段 | satisfied_needs: &'static [NeedTag] |
| `Items.ItemProperties` | 新增字段 | consumable: Option<ConsumableEffect> |
| `Items.ItemRegistry` | 新增方法 | get_consumable(id) -> Option<&ConsumableEffect> |
| `NPC.Physiology::from_vitals()` | 修改 | 新增字段映射 |

### 不需要修改的内容

- `FaithQuery` — 所有需要的数据已有
- `WeatherQuery` — 不涉及
- `PowerTopologyQuery` — 不涉及
- `Skill system` — 不涉及
- `GOAP planner` — 不改,只有生存需求进安全网
