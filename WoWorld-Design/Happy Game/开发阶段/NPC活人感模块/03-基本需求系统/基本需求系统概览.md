## §1 概述

### 1.1 为什么需要此系统

NPC ver2.0 中，生理需求通过三条独立路径处理：
1. **GOAP 安全网**——硬阈值触发（`hunger<0.3→找食物`），悬崖效应
2. **概率决策器**——`physiology_modifier` 黑箱（文档未展开其内部逻辑）
3. **情绪引擎**——`apply_physiological_pull()` 三者各管一域，无统一框架

**四大缺陷迫使重构**：
- **悬崖效应**：hunger=0.31 和 hunger=0.29 差异微乎其微，但前者 NPC 正常社交，后者被电击般冲向食物——不自然
- **无个体差异**：所有 NPC 在同一 hunger 值反应完全一致——抹平了人格/文化/种族多样性
- **需求间无耦合**：真实生理中睡眠不足→饥饿素上升→更饿。当前模型各需求完全独立
- **无预测性**：有智慧的 NPC 应预判"下午会饿"，而非等到血糖骤降才反应

### 1.2 系统边界

本系统不创建独立模块。它是对 **Life/NPC/物品** 已有机制的重组和扩展：

```
┌─ Life 模块（数据权威）────────────────────────────────────┐
│  Vitals 新增: element_surplus[8], libido                  │
│  RaceTraits 新增: blood_element_ratio[8], libido_type     │
│  新增函数: ingest_food(), vent_surplus(), tick_passive()  │
│  新增公开类型: ConsumableEffect (供物品模块注册)          │
└──────────────────────────────────────────────────────────┘
                    │
                    │ Physiology::from_vitals()
                    ▼
┌─ NPC 模块（行为消费者）────────────────────────────────────┐
│  Physiology 新增: element_balance_urgency, libido         │
│  ★ v1.1: NeedColumn 新增 (SoA): social_deficit + need_sensitivity │
│  重构: physiology_modifier → need_action_match           │
│  扩展: apply_physiological_pull +3行                     │
└──────────────────────────────────────────────────────────┘
                    │
                    │ ItemRegistry::get_consumable()
                    ▼
┌─ 物品模块（数据服务）──────────────────────────────────────┐
│  ItemProperties 新增: consumable: Option<ConsumableEffect>│
│  ItemRegistry 新增: get_consumable(id) → Option<&...>    │
└──────────────────────────────────────────────────────────┘
```

**关键原则**：
- Life 是全部生物数据的**唯一权威源**——不改角色
- Physiology 是 NPC 的**主观感知层**——不改角色，只加字段
- GOAP 安全网不改——只有**生存需求**有资格触发紧急规划
- 本模块是嵌入式的，**不做独立 crate**——零新增编译单元

## §2 需求维度

### 2.1 需求分类总览

```
┌─ 匮乏驱动（不满足→死亡/崩溃）─────── GOAP 安全网 ───┐
│ ① 饥饿 (Hunger)              已有, 机制升级          │
│ ② 口渴 (Thirst)              已有, 机制升级          │
│ ③ 疲劳/睡眠 (Fatigue)        已有, 机制升级          │
│ ④ 个人安全 (Safety)          已有, 不改              │
├─────────────────────────────────────────────────────┤
├─ 累积驱动（不满足→微弱不适）────── 仅进概率 ────────┤
│ ⑤ 元素平衡 (Element Balance)  ★ 新增                │
├─────────────────────────────────────────────────────┤
├─ 周期驱动（不满足→无直接惩罚）──── 仅进概率 ────────┤
│ ⑥ 性/libido                  ★ 新增                │
├─────────────────────────────────────────────────────┤
├─ 心理驱动（不满足→情绪长期下降）── 仅进概率 ────────┤
│ ⑦ 社交/归属 (Social)          ★ 新增                │
└─────────────────────────────────────────────────────┘
```

**分类逻辑**：前四个是"不处理就出事"——GOAP 安全网必须管。后三个是"不处理也死不了"——只能在概率决策器中作为行动权重的影响因子。

### 2.2 各需求的详细定义

#### ① 饥饿 (Hunger) — 已有，机制升级

| 属性 | 值 |
|------|-----|
| **数据来源** | `Life.Vitals.hunger: f32` — 0.0=极度饥饿, 1.0=饱腹 |
| **消耗公式** | `种族代谢率 × 活动量 × 环境温度修正`（冰冷环境消耗加快）|
| **归零链** | hunger<0.1 → 体力恢复停止 → hunger=0 → 扣 HP → 死亡 |
| **GOAP 阈值** | `< 0.3` — 不改 |
| **urgency 公式** | `(1.0 - hunger) × sensitivity`（sensitivity 从大五人格查表）|
| **昼夜调制** | 正常饭点附近 urgency 有微弱的节律性上升 (≤ ±0.05) |

#### ② 口渴 (Thirst) — 已有，机制升级

| 属性 | 值 |
|------|-----|
| **数据来源** | `Life.Vitals.thirst: f32` — 0.0=脱水, 1.0=不渴 |
| **消耗公式** | 高温→加速；湿润食物→减缓 |
| **归零链** | 比饥饿更快扣 HP → 死亡 |
| **GOAP 阈值** | `< 0.25` — **注意**：比饥饿阈值更低（口渴更快致命）|
| **urgency 公式** | `(1.0 - thirst) × sensitivity` |

#### ③ 疲劳/睡眠 (Fatigue) — 已有，机制升级

| 属性 | 值 |
|------|-----|
| **数据来源** | `Life.Vitals.fatigue: f32` — 0.0=精力充沛, 1.0=极其疲劳 |
| **累积公式** | `体力消耗速度 × (1 - 耐久)` |
| **归零链** | fatigue>0.9 → 全属性 -20%；fatigue>1.0 → 强制昏睡 |
| **GOAP 阈值** | `> 0.9` — 不改 |
| **urgency 公式** | `fatigue × sensitivity` |
| **昼夜调制** | 夜间 fatigue urgency 微升（褪黑素节律的模拟，≤ 0.1 偏移）|

#### ④ 个人安全 (Safety) — 已有，不改

| 属性 | 值 |
|------|-----|
| **数据来源** | `Life.Vitals.health` (归一化) + `CurrentState.combat_threat` |
| **GOAP 阈值** | health<0.3 → SurvivalHeal; combat_threat>0.7 → SurvivalDefend |
| **改动** | **零修改**。仅为完整性列于此处 |

#### ⑤ ★ 元素平衡 (Element Balance) — 新增

**设计理念**：
- 魔法系统公理：**血元素 = 金木水火土风雷电八元素的复合材料**（灵元素不参与消化代谢）
- 进食 → 食物携带的元素谱被吸收 → 血元素合成消耗掉"需要的部分"（种族血元素配比）→ **多余元素累积**
- 累积到一定程度 → 需要有意识地释放（无声无形，瞬间完成）
- 睡眠时缓慢被动释放（5%/游戏日）

| 属性 | 值 |
|------|-----|
| **数据来源** | `Life.Vitals.element_surplus: [f32; 8]` — 八元素的过剩累积（不包含血和灵）|
| **累积来源** | 吃了元素配比 ≠ 种族需求的偏科食物 |
| **释放方式** | 主动释放（瞬间全清，几秒，有意识）/ 睡眠被动释放（5%/晚）|
| **周期** | 均衡饮食 → 几乎不累积（睡眠被动够用）；偏科饮食 → 3-5 天需主动一次 |
| **归零后果** | 无。不处理 → 微弱不适（情绪引擎 pull 系数极小）|
| **决策层** | **仅概率决策器**（不进 GOAP）|
| **urgency 公式** | `max(surplus)×0.5 + avg(surplus)×0.5`（详 §4）|
| **与魔法连接** | v1.0 不做元素过剩对法术的增益。预留了 `blood_element_ratio` 查询接口 |

**释放观感规范**：
- 无声无形——无粒子、无动画、无音效
- NPC 自身有意识，短暂停顿 1-3 秒（游戏时间）
- 旁观 NPC 无感知事件，不产生记忆
- 玩家观察：如果盯着看 → NPC 短暂停顿（和发呆难以区分）；否则完全无感
- 玩家 UI：不需要任何提示

#### ⑥ ★ 性/libido — 新增

**设计理念**：NPC 的性驱力是周期性的身体信号，不是持续的恒量。进化多样性要求不同物种有不同的发情模式。

| 属性 | 值 |
|------|-----|
| **数据来源** | `Life.Vitals.libido: f32` — bio-signal, not survival vital |
| **三种类型** | `Continuous` / `Seasonal` / `EventTriggered`（详 §5）|
| **归零后果** | 无 |
| **决策层** | **仅概率决策器**（不进 GOAP——不是紧急事件）|
| **reproductive_urge** | `libido × culture.marriage_pressure × faith.reproduction_modifier × personality.openness` |
| **文化/信仰乘数** | 来自已有 `FertilityNorms`（文化 004）+ `ReligiousReproductionNorms`（信仰 002）——零新增接口 |

**与其他系统的关系**：
- `reproductive_urge` → 概率决策器寻找伴侣行为 → `CourtshipState::Bonded` → 激活 Life 012 `FertilityDrivers`
- NPC 模块的 `AttractionTemplate` 提供行为目标筛选
- 不定义繁殖逻辑本身——那是 Life 012 的职责

#### ⑦ ★ 社交/归属 (Social) — 新增

**设计理念**：社交匮乏是心理状态，不是身体状态。一个人在荒岛独居一年——身体完全健康，心理的社交匮乏极高。另一个内向者独居一年——同样的客观独居时长，心理的社交匮乏低于外向者。

| 属性 | 值 |
|------|-----|
| **数据来源** | `NeedColumn.social_deficit: f32` (SoA) — 0.0=社交充足, 1.0=极度孤独。★ v1.1: 已从 NpcData 迁移 |
| **累积速度** | 无社交互动 +0.02/游戏日 → 50 天独居才满 |
| **恢复** | 每次有意义社交互动 -0.03 至 -0.15（取决于互动质量×亲密度×情绪愉悦度）|
| **互动定义** | 参与对话 / 共同活动 / 在公共空间旁观他人（轻度）|
| **归零后果** | 无直接。长期 → 情绪 pleasure 下降 |
| **决策层** | **仅概率决策器** |

**与 stagnation_sense 的区别**：
```
stagnation_sense（SelfNarrative） 综合性"人生意义匮乏"指标
    ↑ 输入之一
social_deficit（此处）           具体的"社交不足"指标
```
两者相关但不相等。一个社交充足的 NPC 依然可能因缺乏人生目标而 stagnation_sense 高。一个独居的学者可能 social_deficit 中等但 stagnation_sense 低（因为有创作使命）。

## §3 统一数学框架

### 3.1 核心公式：匮乏→紧迫

所有生理需求共享同一公式。没有 sigmoid，没有幂函数——线性 + 人格查表。调参者一眼可读，玩家可直觉理解。

```
urgency_i = deviation_i × sensitivity_i
```

| 符号 | 含义 | 来源 |
|------|------|------|
| `deviation_i` | 需求当前值与理想值的偏差 | Life.Vitals（客观事实）|
| `sensitivity_i` | 此 NPC 对此需求的个体敏感度 | 大五人格查表（初始 roll，BigFive 极端冲击后重算）|

```
deviation 计算：
  hunger:         (1.0 - hunger)          // 0.0=饱→0, 0.0=饿→1.0
  thirst:         (1.0 - thirst)          // 同上
  fatigue:        fatigue                 // 0.0=精力→0, 1.0=疲→1.0
  element_balance: max(surplus)×0.5 + avg(surplus)×0.5  // 混合指标
  libido:         libido                 // 0.0→0, 1.0→1.0
  social:         social_deficit         // 0.0→0, 1.0→1.0
  safety:         (1.0 - health)         // health=1.0→0, 0.0→1.0
```

### 3.2 敏感性查表（从大五人格派生）

**设计原则**：不新增 NPC 字段。现有大五人格已经足够为所有 7 个需求提供丰富的个体差异。

```rust
/// 每个 NPC 的需求敏感度——从大五人格一次性派生
/// 存储在 NPC 模块本地（或每决策周期重新计算——成本 ≈ 8 次查表 = 可忽略）
/// ★ v2.0: 扩展至 8 字段（+esteem +competence），详见 [[04-进阶需求系统]]
struct NeedSensitivity {
    hunger: f32,
    thirst: f32,
    fatigue: f32,
    element_balance: f32,
    libido: f32,
    social: f32,
    // ★ v2.0 新增
    esteem: f32,
    competence: f32,
}

impl NeedSensitivity {
    /// 从大五人格一次性派生全部敏感度
    pub fn from_big_five(p: &BigFive) -> Self {
        Self {
            // 神经质高的人饿时焦虑，更快找食物
            hunger:    0.5 + p.neuroticism * 0.5,                 // 0.5~1.0
            // 自律的人定时喝水，对脱水更敏感
            thirst:    0.5 + p.conscientiousness * 0.25 + p.neuroticism * 0.25,
            // 外向者精力旺盛，稍感疲劳不急着睡
            fatigue:   0.5 + (1.0 - p.extraversion) * 0.3,        // 内向者更易感疲劳
            // 自律者更关注身体平衡，神经质者更容易感到不适
            element_balance: 0.4 + p.conscientiousness * 0.45 + p.neuroticism * 0.15,
            // 外向开放者对社交/亲密行为更活跃
            libido:    0.3 + p.extraversion * 0.4 + p.openness * 0.3,
            // 外向者社交需求强
            social:    0.2 + p.extraversion * 0.8,                // 0.2~1.0

            // ★ v2.0 新增:
            // 尊重敏感度: 外向者在意社会评价, 竞争性强者(低宜人)在乎输赢
            esteem:    (0.2 + p.extraversion * 0.5 + (1.0 - p.agreeableness) * 0.3).clamp(0.2, 1.0),
            // 胜任敏感度: 尽责者标准高(落差感强), 神经质放大能力焦虑
            competence: (0.2 + p.conscientiousness * 0.5 + p.neuroticism * 0.3).clamp(0.2, 1.0),
        }
    }
}
```

**多样性效果**（示例）：

| NPC | neuroticism | hunger sensitivity | 相同 hunger=0.3 时的 urgency | 行为表现 |
|-----|------------|-------------------|------|---------|
| 平和矮人 | 0.2 | 0.6 | 0.42 | 不急，打完这批铁再吃 |
| 焦虑精灵 | 0.9 | 0.95 | 0.67 | 放下活马上去找食物 |
| 普通人类 | 0.5 | 0.75 | 0.53 | "差不多该吃饭了"，再过一会儿去 |

**同一个身体状态，三个不同行为——涌现自一次查表 + 一次乘法。**

### 3.3 跨需求耦合（极简——真实生理学基线）

只保留有实验证据的交叉通路。**稀疏矩阵，非 O(N²)**：

```rust
/// 跨需求调制——在 decision_engine 中每决策周期应用
fn apply_cross_need_modulation(
    urgencies: &mut [f32; 7],
    phys: &Physiology,
    social_deficit: f32,  // ★ v1.1: social_deficit 在 NeedColumn 上, 不在 Physiology 上
) {
    // 睡眠债务 → 放大饥饿压力 (ghrelin/leptin 激素调节)
    urgencies[HUNGER]  *= 1.0 + phys.fatigue * 0.25;
    // 脱水 → 放大疲劳压力
    urgencies[FATIGUE] *= 1.0 + (1.0 - phys.thirst) * 0.2;
    // 社交匮乏 → 放大各类生存需求 (孤独→暴食/失眠)
    let social_amp = 1.0 + social_deficit * 0.1;
    urgencies[HUNGER]  *= social_amp;
    urgencies[FATIGUE] *= social_amp;
}
```

三个乘法，总成本 ~2ns。v1.0 使用全局耦合系数。后续可扩展为 `RaceTraits` 每物种独立系数（如矮人睡眠-饥饿耦合更强）。

### 3.4 昼夜节律调制

```rust
/// 昼夜节律对需求 urgency 的微弱调制
/// chronotype: 从种子生成——"早鸟型" / "夜猫型" / "中间型"
fn circadian_modulation(hour: f32, chronotype_phase: f32) -> f32 {
    let phase = (hour - chronotype_phase) / 24.0 * std::f32::consts::TAU;
    1.0 + 0.08 * phase.sin()  // ±8% 最大偏移
}
// 只影响 hunger 和 fatigue 的 urgency
```

## §4 元素平衡需求——完整设计

### 4.1 血元素合成的瓶颈模型

**前提公理**（来自 [[../魔法/01-基础层/002-十元素|魔法十元素]]）：
> 血元素 = 金木水火土风雷电八元素按特定比例合成

**核心公式**：

```
给定:
  food_element_yield[i]  for i in 0..7  (食物携带的元素谱)
  race_blood_ratio[i]    for i in 0..7  (种族血元素的八元素配比, Σratio ≈ 1.0)
  portion: f32                           (进食份量)

步骤 1: 为合成血元素各元素的需求量
  needed[i] = race_blood_ratio[i] × portion

步骤 2: 瓶颈计算——最稀缺元素决定合成效率
  // 瓶颈率 ≤ 1.0。=1.0 意味着所有元素都充足，血元素合成完整
  bottleneck = min(1.0, min_over_i(absorbed[i] / needed[i]))

步骤 3: 各元素去向
  used[i]    = needed[i] × bottleneck    // 实际用于合成血元素
  surplus[i] = absorbed[i] - used[i]     // 正=过剩累积, 负=元素不足
  element_surplus[i] += max(0.0, surplus[i])

步骤 4: 合成效率反哺饥饿
  血元素合成效率 = bottleneck
  → 效率<1.0 → 同等食物量恢复的饱腹度下降
  → 效率=1.0 → 饱腹度正常恢复
```

### 4.2 瓶颈模型的涌现意义

**场景**：矮人矿工连着三天吃烤肉

| 天 | 吃了什么 | 火元素 | 木元素 | bottleneck | hunger 恢复 | surplus 变化 |
|----|---------|-------|-------|-----------|------------|-------------|
| 1 | 烤肉 (火偏重) | 充足 | 充足 | 1.0 | 正常 | 火+0.15 |
| 2 | 烤肉 | 充足 | 充足 | 1.0 | 正常 | 火+0.15 |
| 3 | 烤肉 | 充足 | 刚好 | 0.85 | **下降 15%** | 火+0.1 (速率减慢) |
| 4 | 蔬菜 (木/水偏重) | 充足 | 充足 | 1.0 | 正常 | 木/水+0.1, 火被动释放-0.05 |

**结论**：矮人矿工不需要脚本说"连着吃几天肉就腻了去吃菜"。瓶颈模型让"饮食均衡"成了生物驱动力——不吃菜血元素合成效率下降 → 需要吃更多才能同样饱 → 但火元素越堆越多 → 最终主动释放。**张力来自数学，不是 if 语句。**

### 4.3 食物元素谱的三层默认值

不是所有食物都需要手写元素谱。三层回退覆盖全部食物：

```
Level 1 — 手写值 (ItemProperties.consumable.element_profile)
  龙肉：         {火:0.7, 血:0.1, 金:0.05, ...}
  魔法蘑菇：     {灵:0.3, 木:0.2, 风:0.1, ...}
  异域香料：     {火:0.4, 风:0.2, ...}
```

### 5.2 周期计算

```rust
/// 计算当前 libido 值 (0.0-1.0)
/// game_day: 当前游戏日数
/// age_factor: 年龄修正 (0.0-1.0, 来自 LifeEntity)
/// health_factor: 健康修正 (0.0-1.0, hp / max_hp)
pub fn compute_libido(
    libido_type: &LibidoType,
    cycle_days: u32,
    game_day: f64,
    age_factor: f32,
    health_factor: f32,
) -> f32 {
    let raw = match libido_type {
        LibidoType::Continuous => {
            let phase = (game_day % cycle_days as f64) / cycle_days as f64;
            // 温和正弦: 0.3 + 0.3 × sin(2π×phase) → 范围 0.0~0.6
            0.3 + 0.3 * (std::f64::consts::TAU * phase).sin() as f32
        }
        LibidoType::Seasonal { onset_day_of_year, duration_days } => {
            let day_of_year = game_day as u32 % 365;
            if is_in_window(day_of_year, *onset_day_of_year, *duration_days) {
                // 窗口期内剧烈波动: 0.5 + 0.5 × sin(...) → 范围 0.0~1.0
                let progress = (day_of_year - *onset_day_of_year) as f32 / *duration_days as f32;
                0.5 + 0.5 * (std::f32::consts::TAU * progress * 3.0).sin()
                // sin 频率 3× 使窗口期内有多个起伏
            } else {
                0.0  // 非窗口期——归零
            }
        }
        LibidoType::EventTriggered => {
            // 默认 0.0。触发事件由外部系统注入(天象/魔法浓度/生命阶段变化)
            // ★ v1.0 日历后备: 每 N 年某固定季节自动触发（种子确定性日历），附带天象检查
            //   · 龙 (Dragon): 每 12 年夏至 → 天象检查(特定星座) → 触发=0.8 否则 0.0
            //   · 神明 (Divine): 每 100 年 → 触发=1.0（极稀有事件）
            //   · 其他 EventTriggered 生物: 每 5 年春分 → 触发=0.5
            //   · full_dynamic 延后至 v2.0，届时日历后备退化为 fallback 分支
            // v1.0: 日历后备（种子确定性日历）替代完整动态触发
            calendar_trigger(seed, species_id, current_year, current_season)
        }
    };
    (raw * age_factor * health_factor).clamp(0.0, 1.0)
}
```

### 5.3 reproductive_urge 合成

```rust
/// 每决策周期计算, 不存储

    culture: &CultureQuery,
```

### 6.1 社交互动定义

"社交互动"的精确定义——区分于"路过"。

```rust
/// 社交互动的质量由参与深度和情绪愉悦度共同决定
enum SocialInteractionQuality {
    /// 主动对话——双方参与 DialogueContext
    Conversation { depth: f32, closeness: f32, pleasure: f32 },
    /// 共同活动——一起劳作/打猎/仪式
    GroupActivity { intensity: f32, familiarity: f32 },
    /// 旁观——在酒馆/市场/广场有他人在可闻距离内
    /// 极度内向者也从中获得微弱社交满足(知道"世界还在运转")
    ProximityOnly,
}

fn interaction_recovery(quality: &SocialInteractionQuality) -> f32 {
    match quality {
        Conversation { depth, closeness, pleasure } => {
            // 深度对话 + 亲密关系 + 愉悦情绪 → 高质量社交
            (depth * 0.3 + closeness * 0.3 + pleasure * 0.2).clamp(0.0, 0.15)
        }
        GroupActivity { intensity, familiarity } => {
            (intensity * 0.5 + familiarity * 0.5).clamp(0.0, 0.12)
        }
        ProximityOnly => 0.03,  // 轻度——但累积起来也很可观
    }
}
```

### 6.2 累积与恢复

```rust
/// 每游戏日 tick (冷路径——不在决策热路径)
/// ★ v1.1: social_deficit 存储在 NeedColumn 上，参数改为 &mut NeedColumn
fn tick_social_deficit(need_col: &mut NeedColumn, personality: &BigFive, dt_days: f32) {
    // 累积: 无互动时上升
    // isolation_rate = 0.02 / day —— 50 天独居才满
    need_col.social_deficit += 0.02 * dt_days;

    // 人格调制: 内向者累积更慢(独处自在)
    let introversion = 1.0 - personality.extraversion;
    need_col.social_deficit -= introversion * 0.005 * dt_days;  // 内向者自带社交缓冲

    need_col.social_deficit = need_col.social_deficit.clamp(0.0, 1.0);
}

/// NPC 参与社交互动后——事件驱动, 非每帧轮询
/// ★ v1.1: social_deficit 存储在 NeedColumn 上
fn on_social_interaction(need_col: &mut NeedColumn, quality: &SocialInteractionQuality) {
    let recovery = interaction_recovery(quality);
    need_col.social_deficit -= recovery;
    need_col.social_deficit = need_col.social_deficit.max(0.0);
}
```

**重要**：入睡不恢复 social_deficit——睡眠恢复的是 fatigue。社交匮乏只能靠社交来缓解。这和真实心理学一致：孤独的人睡再多也不解决孤独。

#### 核心匹配算法

```rust
/// 计算行动对当前需求的匹配度
///
/// 对 action.satisfied_needs 中每个需求取 urgency → 算术平均 → 权重乘数
/// 严谨性: 平均而非最大——避免"超饿的 NPC 会去酒馆只喝酒(只有口渴需求)"
///   如果 action 声明了多个需求, 它应该同时缓解这些需求。平均确保部分匹配也有适度权重。
///
/// 返回值: 1.0 (需求无关) ~ 2.0 (需求极其紧迫)
pub fn need_action_match(
    action: &ActionDef,
    phys: &Physiology,
    social_deficit: f32,
    esteem_deficit: f32,              // ★ v2.0 新增
    competence_frustration: f32,      // ★ v2.0 新增
    sensitivity: &NeedSensitivity,
) -> f32 {
    let needs = action.satisfied_needs;
    if needs.is_empty() {
        return 1.0;  // 该行动不满足任何需求——不受需求影响
    }

    let mut total_urgency: f32 = 0.0;
    for need in needs {
        total_urgency += match need {
            NeedTag::Hunger         => (1.0 - phys.hunger)         * sensitivity.hunger,
            NeedTag::Thirst         => (1.0 - phys.thirst)         * sensitivity.thirst,
            NeedTag::Fatigue        => phys.fatigue                 * sensitivity.fatigue,
            // ★ 元素平衡: deviation (已在 Physiology 合并) × sensitivity
            NeedTag::ElementBalance => phys.element_balance_urgency * sensitivity.element_balance,
            NeedTag::Libido         => phys.libido                  * sensitivity.libido,
            NeedTag::Social         => social_deficit               * sensitivity.social,
            NeedTag::Safety         => (1.0 - phys.health),  // safety 无敏感度差异——人人怕死
            // ★ v2.0 新增 — 进阶需求
            NeedTag::Esteem        => esteem_deficit               * sensitivity.esteem,
            NeedTag::Competence    => competence_frustration        * sensitivity.competence,
        };
    }
    let avg_urgency = total_urgency / needs.len() as f32;

    // 需求越迫切, 权重越高: 1.0 + urgency = 范围 1.0~2.0
    (1.0 + avg_urgency).clamp(1.0, 2.0)
}
```

### 7.3 情绪引擎：apply_physiological_pull

在现有函数中新增 3 个条件。

```rust
pub fn apply_physiological_pull(
    emotion: &mut EmotionState,
    phys: &Physiology,
    social_deficit: f32,  // ★ 新增参数
) {
    // === 已有, 不改 ===
    if phys.hunger < 0.4 {
        emotion.pleasure -= 0.001 * (0.4 - phys.hunger);
    }
    if phys.fatigue > 0.7 {
        emotion.arousal -= 0.001 * phys.fatigue;
    }
    if phys.health < 0.5 {
        emotion.control -= 0.001 * (0.5 - phys.health);
    }

    // === ★ 新增 ===
    // 元素不平衡 → 微弱不适
    if phys.element_balance_urgency > 0.5 {
        emotion.pleasure -= 0.0005 * (phys.element_balance_urgency - 0.5);
    }
    // 高 libido → 轻度焦躁
    if phys.libido > 0.7 {
        emotion.arousal += 0.001 * (phys.libido - 0.7);
    }
    // 孤独 → 快乐下降
    if social_deficit > 0.5 {
        emotion.pleasure -= 0.001 * social_deficit;
    }
}
```

系数极小——经过数十游戏日的累积才会产生可感知的情绪变化。这是**长期背景色调**，不是"几小时不社交就抑郁"。

```rust
// NPC 模块: Physiology 主观感知层
pub struct Physiology {
    // ... 已有字段 (hunger, thirst, fatigue, health, stamina, temperature,
    //              body_part_damage) ...

    /// ★ 新增: 元素平衡紧迫度. 从 Vitals.element_surplus 派生
    pub element_balance_urgency: f32,

    /// ★ 新增: 性驱力（直通 Vitals.libido）
    pub libido: f32,
}

impl Physiology {
    pub fn from_vitals(v: &Vitals, ctx: &PerceptionContext) -> Self {
        Self {
            hunger:     v.hunger,       // 已有, 直通
            thirst:     v.thirst,       // 已有, 直通
            fatigue:    v.fatigue,      // 已有, 直通
            health:     v.hp / v.max_hp.max(1.0),  // 已有, 归一化
            stamina:    v.stamina / v.max_stamina.max(1.0), // 已有, 归一化
            temperature: Self::perceived_temperature(v.body_temperature, ctx), // 已有
            body_part_damage: v.body_part_damage.clone(), // 已有

            // ★ 新增:
            // urg = 最大过剩×0.5 + 平均过剩×0.5
            // 既捕获"某一个极高"，也捕获"多项都在上涨"
            element_balance_urgency: {
                let max_s = v.element_surplus.iter().cloned().fold(0.0f32, f32::max);
                let avg_s = v.element_surplus.iter().sum::<f32>() / 8.0;
                max_s * 0.5 + avg_s * 0.5
            },
            libido: v.libido,  // 直通
        }
    }
}
```

## §11 玩家体验

### 11.1 玩家也是同一套系统

玩家的角色在技术上拥有相同的 NpcData（共享同一套 Rust 代码路径）。但：

- **饥饿/口渴**：已有生存 UI（和现有生存游戏一致）
- **疲劳**：屏幕边缘模糊/角色语音"累了"（和现有生存游戏一致）
- **元素平衡**：无 UI 提示。玩家通常饮食多样化 → 几乎不累积。法师玩家(未来)可通过 Godot 侧的微弱的 UI 提示感知（v2.0）
- **libido**：无直接 UI。转化为"对话时有额外浪漫选项"——完全可选，不选没惩罚
- **社交**：无 UI。独行太久 → 旅伴 NPC 会说"好久没见到其他人了"，或玩家注意到自己角色的情绪标记偏灰暗

### 11.2 各需求的玩家负担

| 需求 | 玩家负担 | 说明 |
|------|:---:|------|
| 饥饿 | 🔴 高 | 必须管理。生存游戏的核心循环 |
| 口渴 | 🔴 高 | 同上 |
| 疲劳 | 🟡 中 | 已有疲劳系统 |
| 元素平衡 | 🟢 无 | 完全自动——多样化饮食自然不累积 |
| libido | 🟢 无 | 可选深度玩法 |
| 社交 | 🟢 无 | 玩家持续和 NPC 交互自然不累积；荒野独居→沉浸感(非惩罚) |

**设计原则**：新增需求不应增加玩家的操作负担。它们的价值在 NPC 侧——驱动 NPC 的日常行为涌现。

## §13 SoA 列族架构与实现约束 ★ v1.1 (2026-06-23)

> **设计裁决**：03-基本需求系统试点 方案 D — 持久化字段从 `NpcData`（AoS）迁移至独立 `NeedColumn`（SoA），推导函数保持纯函数零存储。此设计取代 §8.7 中 "NpcData 新增 social_deficit 字段" 的旧方案。

### 13.1 列族所有权

03-基本需求系统 **own 一个 SoA 列**：

```rust
/// NeedColumn — 03-基本需求系统 own 的持久化列族
/// 存储在 NpcStore 中，按 NpcId (u32 index) 索引
/// SoA 布局：所有 NPC 的 NeedColumn 连续存储在一个 Vec<NeedColumn> 中
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct NeedColumn {
    /// 社交匮乏 (0.0=社交充足, 1.0=极度孤独)
    /// 累积: 无社交 +0.02/游戏日, 50天独居才满
    /// 恢复: 有意义社交互动 -0.03~0.15
    /// 仅进概率决策器, 不进 GOAP
    pub social_deficit: f32,           // 4B

    /// 需求敏感性——从大五人格一次性派生。缓存防重复查表。
    /// BigFive 在极端冲击后漂移 → NeedSensitivity 随之重算
    /// (创伤>0.9 或 长期环境冲突>100日)
    pub need_sensitivity: NeedSensitivity,  // 32B (8×f32)
}
// 总计 36B/NPC。100K NPC = 3.6MB SoA，单次线性扫描全在 L2 缓存内。
```

**不在 NeedColumn 中的 03 数据**（纯函数导出）：

```rust
/// 03 提供的纯函数——每决策周期调用，不存储
impl NeedSystem {
    /// 元素平衡紧迫度 — 从 Vitals 派生，max*0.5 + avg*0.5
    /// ⛔ 禁止缓存此值。Vitals 是唯一权威源。
    pub fn element_balance_urgency(vitals: &Vitals) -> f32;

    /// libido 直通 Vitals — 周期计算在 Life 模块完成
    /// ⛔ 禁止在 NPC 侧缓存副本。Vitals.libido 是唯一权威源。
    pub fn libido(vitals: &Vitals) -> f32;
}
```

**与 §8.6 `Physiology::from_vitals()` 的关系**：
- `Physiology` 是每决策周期组装的**栈上临时视图**——不持久化。`from_vitals()` 调用时，`element_balance_urgency` 和 `libido` 字段通过调用 `NeedSystem::element_balance_urgency(v)` 和 `NeedSystem::libido(v)` 填充。
- 两套 API 不冲突：`NeedSystem` 是函数的**权威定义者**（03 own），`Physiology` 是函数的**消费方**（NPC 模块组装）。和 `ConsumableEffect`（Life 定义 schema，Items 存储）是同一模式。

**关键原则**：`NeedColumn` 只存**持久化字段**。每决策周期可派生的值不进列——从权威源计算，即用即弃。

### 13.2 NpcData → NeedColumn 的迁移

旧方案（v1.0, §8.7 原文）：
```rust
// ⛔ 旧: social_deficit 直接挂在 NpcData 上
pub struct NpcData {
    // ... 已有字段 ...
    pub social_deficit: f32,  // §8.7 仅定义此字段
}
// 注: §3.2 的 NeedSensitivity 在 v1.0 中未明确指定存储位置
//     ("存储在 NPC 模块本地或每决策周期重新计算")
//     v1.1 明确将其持久化至 NeedColumn，与 social_deficit 同列
```

新方案：
```rust
// ✅ 新: NpcData 不持有个体需求数据。按需从 NpcStore 取列引用。
// 决策器签名（和 04/06 列族统一注入）:
fn select_action(
    npc: &NpcData,
    need_col: &NeedColumn,              // 03 own 的持久化列
    growth_col: &GrowthColumn,          // 04 own 的持久化列 (★ v2.1 已迁移)
    cognition_col: &CognitionColumn,    // 06 own 的持久化列 (★ v1.1 已迁移)
    world: &SubjectiveWorld,
    rng: &mut impl Rng,
) -> Action;
```

**NpcData 主文档中**：`social_deficit` 和 `need_sensitivity` 字段从 `NpcData` struct 中移除。改为注释引用 `→ NeedColumn (03-基本需求系统)`。主文档 `NPC活人感开发文档ver2.0.md` §1.1 的 NpcData 定义同步更新。

### 13.3 类型级防错约束（⚠️ 实现禁忌）

> **设计意图**：以下约束不是"建议"——违反任一条会导致 §雷点1-5 的硬错误。类型系统无法捕获的逻辑错误用编译期契约文档覆盖。

#### ⛔ C01 — 禁止缓存跨模块查询结果

```
❌ 禁止: pub static CULTURE_CACHE: Lazy<Arc<Mutex<HashMap<...>>>>
❌ 禁止: pub static FAITH_REPRODUCTION_CACHE: Lazy<...>
❌ 禁止: 在 NeedColumn 上添加 culture_query_results 字段

✅ 必须: 每决策周期从 &dyn CultureQuery / &dyn FaithQuery 参数注入
✅ 理由: 文化/信仰数据可能被其他系统（CHG-024, CHG-025）修改。
        缓存 = 幽灵依赖。参数注入让 Rust borrow checker 保证数据一致性。
```

**直接影响**：`reproductive_urge()` 的 `culture`/`faith` 参数来自调用者注入的 `WorldQueries`，
不是来自全局缓存。`compute_libido()` 的 `age_factor`/`health_factor` 来自 `Life.Vitals` 当前值，不缓存。

#### ⛔ C02 — 禁止对 NeedTag 匹配使用 _ 通配分支

```
❌ 禁止: match need_tag { NeedTag::Hunger => ..., _ => 0.5 }
❌ 禁止: match need_tag { ... => /* 默认分支 */ }

✅ 必须: 穷举所有 9 个 NeedTag 变体（含 04 的 Esteem, Competence）
✅ 理由: 新增 NeedTag 变体时编译器强制检查所有 match 分支。
        _ 通配吞掉新变体 → 静默回归 → "为什么 NPC 的 Competence 需求完全被忽略了？"
✅ Rust 强制: 移除 _ 分支后编译器报 exhaustive match 错误 → 新变体不会漏
```

**具体影响**：`need_action_match()` 中的 `match need { ... }` 必须穷举 9 个变体。`urgency` 计算中的 sensitivity 查表同样穷举。

#### ⛔ C03 — 禁止 social_deficit 参与布尔逻辑

```
❌ 禁止: if npc.need_col.social_deficit > 0.7 { force_social_action(); }
❌ 禁止: if need_col.social_deficit > SOME_THRESHOLD { ... }

✅ 必须: social_deficit 仅作为 need_action_match() 中 urgency 计算的乘数因子
✅ 理由: 布尔阈值 = 悬崖效应。这正是 §1.1 指出的"hunger=0.31 vs 0.29"问题。
        社会行为的涌现来自 urgency × sensitivity 的连续乘积累——
        不是"孤独 > 0.7 → 强制去酒馆"。
```

#### ⛔ C04 — 禁止对 element_surplus 做索引越界假设

```
❌ 禁止: element_surplus[7] // 硬编码索引
❌ 禁止: for i in 0..8 { surplus[i] = ... } // 魔法数字

✅ 必须: 使用命名常量 ELEMENT_METAL..ELEMENT_LIGHTNING (附录 A 已定义)
✅ 必须: 用 const ELEMENT_COUNT: usize = 8 替代硬编码
✅ 理由: 十元素体系（魔法 002）可能在未来版本扩展。
        硬编码索引在新增元素时全部失效。
```

#### ⛔ C06 — 禁止在 impl NpcData 块中实现消费 SoA 列族的逻辑

```
❌ 禁止: impl NpcData { fn update_needs(&mut self) { self.social_deficit += ... } }
❌ 禁止: fn tick_social_deficit(npc: &mut NpcData, ...)

✅ 必须: 独立系统函数，仅借用所需列族：
         fn need_system(need_col: &mut NeedColumn, personality: &BigFive, dt: f32)
✅ 理由: impl NpcData 中的 &mut self = 锁全部 NpcData 字段 = 禁止 rayon 并行。
        设计文档中的 impl NpcData 块表达概念归属——不是 Rust 实现策略。
        Rust 代码中，每个系统函数只 borrow 它实际 consume 的列族。
        NeedColumn + GrowthColumn + CognitionColumn 可被三个 rayon task 同时借用。
```

#### ⛔ C05 — 性能预算理解正确性

```
§10.2 当前预算（per-NPC，已验证可靠）：
  from_vitals()        ~5ns
  need_action_match()  ~50ns
  apply_physiological_pull() ~6ns
  总计 per-NPC          ~60ns（@ 1000 L1 NPC × 5 决策/s = 300μs）

❌ 禁止: 为了"优化"这个预算使用 unsafe 裸指针或手动 SIMD
❌ 禁止: 对小循环（<100 迭代）使用 rayon::par_iter()
✅ 必须: 标准 Vec 迭代器。60ns/NPC ≈ 在 4GHz CPU 上约 240 条指令，编译器自动向量化足矣。
✅ 理由: per-NPC 预算是为了让开发者放松——不是紧张。
        任何试图"凑进 60ns"而牺牲可读性的优化 = 过早优化。
```

### 13.4 与主文档的接口契约

03-基本需求系统向 NpcStore 注册的列族：

```
NpcStore::register_column::<NeedColumn>("need")
    .with_initializer(|npc_id, rng| NeedColumn {
        social_deficit: 0.0,  // 新 NPC 社交充足
        need_sensitivity: NeedSensitivity::from_big_five(rng.gen_big_five()),
    })
    .with_migration("v1.0 → v1.1: social_deficit 从 NpcData 迁移至 NeedColumn")
```

**其他模块消费 NeedColumn 的方式**（读接口，不是写接口）：

| 消费模块 | 消费方式 | 消费字段 |
|---------|---------|---------|
| NPC 概率决策器 | `need_action_match(action, phys, need_col.social_deficit, ..., &need_col.need_sensitivity)` | `social_deficit`, `need_sensitivity` |
| NPC 决策器（跨需求耦合）| `apply_cross_need_modulation(urgencies, phys, need_col.social_deficit)` (§3.3) | `social_deficit` |
| NPC 情绪引擎 | `apply_physiological_pull(emotion, phys, need_col.social_deficit, ...)` | `social_deficit` |
| 04-进阶需求系统 | `frustration_regression(goals, &mut need_col.need_sensitivity)` | `need_sensitivity`（临时调制，不持久化） |
| NPC 社交系统 | `on_social_interaction(need_col, quality)` | `social_deficit`（写入——恢复） |
| NPC 每日 tick | `tick_social_deficit(need_col, personality, dt_days)` | `social_deficit`（累积） |
| 世界生成（Bootstrap）| `NeedColumn::seed(rng, big_five)` | 全部字段（初始化） |

**零新增 trait 方法**：所有消费沿已有函数签名。`NeedColumn` 是具体类型（非 trait object）——直接引用，零虚函数开销。
