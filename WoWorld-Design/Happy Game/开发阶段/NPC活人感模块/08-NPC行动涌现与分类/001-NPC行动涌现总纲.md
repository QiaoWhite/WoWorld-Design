## §一、设计哲学

### 1.1 核心命题

**WoWorld 不是故事容器，是故事生成器。** 开发者只定义世界的底层规则——人格参数、行动规律、材料属性、物理法则。具体的故事由 NPC 之间的互动以概率和统计的方式自然涌现。

本模块回答的问题是：**NPC 究竟能做什么？** 但答案不是一份清单，而是一套**通用物理语法**。

### 1.2 五项核心原则

| # | 原则 | 含义 | 反例（我们绝不做的） |
|---|------|------|-------------------|
| **1** | **零年龄门控** | 不设 `if age < X { cannot_do(Y) }`。所有行为差异从连续参数涌现 | `if npc.age < 12 { remove_from_work_pool() }` |
| **2** | **零硬编码禁止** | 不写"中毒 → 不能劳动"的规则。中毒 → health_penalty → 全原子效用下降 → GOAP 自然选择买药 > 打工 | `if npc.is_poisoned { disable_work() }` |
| **3** | **动画是物理的输出** | IK 解算从武器轨迹生成所有战斗动画。零预设。 | 为每种武器×每种招式做关键帧 |
| **4** | **材料属性是数据** | "铁受热变软"不是规则——是铁的热软化曲线自动计算。新增材料 = 新增一行 TOML | `if material == "Iron" and temp > 800 { deformable = true }` |
| **5** | **玩家 = NPC** | 同一套原子引擎、同一套 AgentSnapshot、同一套 GOAP | 玩家有独立的"特殊动作系统" |

### 1.3 涌现哲学

```
❌ 硬编码: "铁匠可以锻造"、"中毒不能劳动"、"老人不能战斗"
✅ 涌现:
  ┌─────────────────────────────────────────────────────────┐
  │ AgentSnapshot 连续参数                                   │
  │   strength=0.3 (老人) / health_penalty=0.6 (中毒)       │
  │         ↓                                               │
  │ 物理原子: GRASP(大锤) → force_check: 0.12 < 0.32(锤子阈值) → TooHeavy │
  │         → "力气不够，抓不起锤子"                           │
  │         ↓                                               │
  │ GOAP 评估: 买药效用 0.95 > 打铁效用 0.096                │
  │         → 选择去药铺                                      │
  └─────────────────────────────────────────────────────────┘
```

> **单位归一化约定**: AgentSnapshot 的所有 26 个字段均为 **0-1 归一化值**（无物理单位）。所有物理原子的 force_check/stability_check 使用归一化阈值比较，确保不同物理量在比较前经过归一化: `normalized_threshold = physical_value / reference_max`。例如，8kg大锤的抓取力阈值 = `8kg / NPC最大举重(≈25kg) = 0.32`。GRASP 的 force_check 比较 `agent.strength × (1-health_penalty)` vs `0.32`——双方都是归一化值。

**关键**: 没有一行代码说"老人不能锻造"或"中毒者应该买药"。结果从物理约束+效用计算中**自然涌现**。

## §二、三层原子架构

### 2.1 架构总图

```
┌─────────────────────────────────────────────────────────────┐
│  第三层: 社会抽象原子 (~25个)                                │
│  GOAP 规划器操作 — 领域模块注册的 ActionCandidate             │
│  NPC 决策: "今天打工"、"找人聊天"、"去神殿祷告"                │
│  由各领域模块实现 execute()，内部消费第二层原子                 │
└───────────────┬─────────────────────────────────────────────┘
                │ 每个第三层原子 = 一个领域 execute() 函数
                │ 内部实现为: 若干第二层原子的序列
┌───────────────▼─────────────────────────────────────────────┐
│  第二层: 领域复合原子 (~40个)                                │
│  有意义的动作组合 — 跨多个物理原子的复合                       │
│  例: CRAFT(Smithing) = HEAT → STRIKE×N → DIP → WIPE        │
│       HARVEST = GRASP + CUT + SCOOP + STACK                 │
│       CAST = POSTURE + CHANT + CONDUCT                      │
└───────────────┬─────────────────────────────────────────────┘
                │ 每个领域复合原子 = 一个 execute() 函数
                │ 内部实现为: 若干物理基元的序列
┌───────────────▼─────────────────────────────────────────────┐
│  第一层: 物理基元 (35个)                                     │
│  身体与世界的最小可交互单元 — 零领域知识                       │
│  定义在 woworld_core — SoA 友好批量执行                       │
│  MOVE · GRASP · STRIKE · ATTACH · IGNITE · OBSERVE ...      │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 三层之间的关键关系

**第一层 → 第二层**: 第一层原子不关心自己在"打铁"还是"打仗"。STRIKE 只做物理计算——力 × 硬度差 = 变形。第二层（CRAFT、PARRY）编排第一层原子的序列+时机，并加上领域规则（技能检定、品质计算）。

**第二层 → 第三层**: 第二层和第三层是**同一实体的两个视图**。第二层 CRAFT = 第三层 CRAFT 的 execute_fn 内执行体——它们不是两个独立实现。第三层为这个实体包裹了 GOAP 可见的 preconditions（"附近有锻炉吗？"）和 utility_fn（"打铁对满足饥饿有没有帮助？"）。当文档中 CRAFT 同时出现在 §2.4 和 §2.5，它描述的是同一样东西——前者从"怎么做"角度，后者从"什么时候选"角度。模块注册 ActionCandidate → GOAP 自动可见。

**跨层贯穿**: AgentSnapshot（连续能力参数）贯穿全部三层。不设门控——原子自身的 force_check 自然拒绝参数不足的尝试。

### 2.3 第一层完整目录: 35 个物理基元

### §二.6 两阶段延迟审议 (CHG-058) ★ v1.1

当 deliberation_depth > 0.2 时，触发两阶段审议:
- **阶段1** (廉价筛选): 检查 depth > 0.2 阈值 — O(1)，零 World 引用
- **阶段2** (predict_outcome): 为全体 ActionCandidate 调用 predict_outcome() — 每个候选 O(log n)。填充 ActionCandidate.prediction: Option<OutcomePrediction>
- **权重调制**: 每个候选的 GOAP 权重 *= 1.0 + expected_pleasure * final_depth * 1.2 (clamp [0.3, 1.7])
- **生存压制**: 当 max_urgency > 0.7 时，survival 行动不受审议调制影响

CognitiveTide 3 字段每决策周期映射至 AgentSnapshot 认知段:
- cognitive_load 上升 → 全原子 execution_noise 放大 + 反应延迟
- rumination_pressure 上升 → 分心 + 战术精度衰减
- mind_quietude 上升 → 战略层深度 + 施法稳定性 + 专注力

## §三、AgentSnapshot — 贯穿全三层的连续参数快照

### 3.1 设计动机

传统的"年龄门控"写死了 NPC 能做什么。在 WoWorld 中，年龄/中毒/技能/疲劳/情绪**不产生二进制的是/否判断**——它们连续地调制所有原子的执行参数和评估效用。

AgentSnapshot 是这一哲学的物理实现：每个 NPC 每决策周期将其状态刷新为一个紧凑快照（~108 bytes SoA），所有原子执行和 GOAP 评估都消费这个快照。

### 3.2 完整结构定义

详见 [[004-AgentSnapshot与连续参数涌现]]。核心字段：

```rust
struct AgentSnapshot {  // ~108 bytes, SoA 友好
    // === 身体 (7 fields, 从 Life + 生命周期派生) ===
    strength: f32,           // 0-1 → GRASP/LIFT/STRIKE 力度
    dexterity: f32,          // 0-1 → DODGE/THROW 精度
    stamina: f32,            // 0-1 → MOVE 持续/STRIKE 频率
    health_penalty: f32,     // 0-1 中毒/受伤/疾病累积
    mobility_factor: f32,    // 0-1 年龄+体型+负重 → MOVE 速度
    fatigue: f32,            // 0-1 → 所有原子效率衰减
    hunger: f32,             // 0-1 → Eat 目标权重
    
    // === 认知 (8 fields, 从 技能+认知系统+NPC人格 派生) ===
    skill_vector_compressed: u64,
    mental_model_count: u8,
    cognitive_damping: f32,  // 0-1 → 学习新概念速率
    planning_horizon: u8,    // GOAP 搜索深度
    literacy: f32,           // 0-1 → READ/WRITE 可行性
    cognitive_load: f32,     // ★ 0-1 → 全原子执行噪声放大·决策延迟
    rumination_pressure: f32,// ★ 0-1 → 分心·战术精度衰减
    mind_quietude: f32,      // ★ 0-1 → 战略层深度·专注力
    
    // === 社交 (4 fields, 从 权力+名声+文化 派生) ===
    legal_capacity: f32,     // 0-1 → SIGN/INVEST/PROCLAIM
    reputation_flags: u32,
    social_standing: f32,    // 0-1 → 说服力/交涉成功率
    religious_participation: f32,  // 0-1 → PRAY/SACRIFICE 效能
    
    // === 生命阶段 (4 fields, 从 生命周期 CHG-041 派生) ===
    developmental_phase: u8, // Infant/Toddler/.../Elder
    age_ratio_in_phase: f32, // 0-1
    fertility_potential: f32,
    gompertz_mortality_risk: f32,
    
    // === 环境 (3 fields, 从 天气+感官 派生) ===
    wetness: f32,            // → 摩擦力/导电性/可燃性
    temperature_exposure: f32,
    intoxication: f32,       // → 全原子精度×噪声
}
```

### 3.3 与其他模块的派生关系

```
Life.Vitals          ──→ AgentSnapshot.physical (strength, stamina, health_penalty...)
生命周期.AgeClock     ──→ AgentSnapshot.lifecycle (developmental_phase, age_ratio...)
技能.SkillEntry       ──→ AgentSnapshot.skill_vector_compressed
认知.CognitiveStyle   ──→ AgentSnapshot.cognitive_damping, planning_horizon
认知.CognitiveTide    ──→ AgentSnapshot.cognitive_load, rumination_pressure, mind_quietude ★ v1.1
认知.predict_outcome() ──→ ActionCandidate.prediction (两阶段审议·阶段2) ★ v1.1
权力.PowerTopology    ──→ AgentSnapshot.legal_capacity
天气.WeatherQuery     ──→ AgentSnapshot.wetness, temperature_exposure
```

AgentSnapshot 不存储在任何模块中——它是每次决策周期从各模块数据**派生**的临时视图。生成方式: `AgentSnapshot::from_sources(&NpcState, &LifeEntity, &SkillEntries, &WeatherSample, ...)`。

## §四、MaterialProperties — 涌现的真正引擎

### 4.1 为什么 MaterialProperties 是关键

物理原子很简单——STRIKE = 力施加。让 STRIKE 产生无限多样化行为的是**材料属性的差异**:

```
STRIKE(铁锤, 玻璃瓶, force=0.3):
  铁锤.hardness(0.9) >> 玻璃瓶.hardness(0.4) × 2 → BREAK → 碎裂
  铁锤.toughness(0.8) >> 玻璃瓶.toughness(0.1) → 铁锤无损

STRIKE(铁锤, 热铁锭, force=0.8):
  铁锤.hardness(0.9) ≈ 热铁锭.hardness_at_temp(0.35) → 塑性变形
  热铁锭.elasticity(0.3) → 只部分回弹 → 产生永久形变
  → 铁锤温度上升（动能→热能） → 声音: 响亮金属撞击声 → 火花: 少量

STRIKE(木棍, 石头, force=0.5):
  木棍.hardness(0.3) < 石头.hardness(0.7) → 木棍受损
  木棍.toughness(0.4) → 可能断裂
```

STRIKE 的代码中没有任何"玻璃会碎"、"铁会变形"、"木头会断"的知识。它只做材质属性对比和物理计算。行为从数据中涌现。

### 4.2 MaterialProperties 结构

详见 [[008-MaterialProperties与物品系统对接]]。核心结构:

```rust
struct MaterialProperties {
    // 力学: GRASP/STRIKE/BREAK/LIFT/PUSH/PULL/ATTACH/CRUSH/CUT
    density_kgm3: f32, hardness: f32, toughness: f32,
    elasticity: f32, friction: f32,
    
    // 热学: HEAT/COOL/IGNITE/EXTINGUISH
    specific_heat: f32, thermal_conductivity: f32,
    melting_point_k: f32, ignition_point_k: f32, combustibility: f32,
    
    // 电学/魔导: CONDUCT
    electrical_conductivity: f32, mana_conductivity: f32,
    
    // 化学/表面: DIP/POUR/SPREAD/WET/WIPE
    flammability: f32, solubility: f32, acidity: f32,
    toxicity: f32, porosity: f32,
    
    // 感官: OBSERVE/LISTEN
    reflectivity: f32, sound_damping: f32, scent_intensity: f32,
    
    // 魔法: CAST/INFUSE/TRANSMUTE/SENSE_AETHER
    elemental_affinity: [f32; 10],
    enchantment_capacity: f32, aether_retention: f32,
    
    // 特殊标记
    oxidizes_when_heated: bool,
}
```

### 4.3 与物品系统的集成模式

遵循物品系统现有的 `Option<ConsumableEffect>` 和 `Option<AudioMaterial>` 委托模式：

- `ItemProperties` 新增 1 个字段: `material: Option<MaterialDefId>`（u32 引用，非内嵌）
- 仅 Material-category 物品 (0x20-0x2F) 填充
- 武器/盔甲通过装配树从组件材料**继承**
- ~200 种材料 × 120B = 24KB → L2 缓存常驻 → 查询成本 <5ns
- **物品系统不解释 MaterialProperties**——只存储和提供查询

## §五、年龄与生命阶段的连续涌现

### 5.1 核心原则: 零年龄门控

AgentSnapshot 的 `developmental_phase` 是一个描述性标签——**不是行为开关**。以下是各阶段的涌现行为特征（全部从 AgentSnapshot 连续参数推导，非硬编码）:

| 生命阶段 | AgentSnapshot 特征 | 涌现的行为特征 |
|---------|-------------------|-------------|
| **Infant (婴儿)** | strength=0.05, mental_model_count=0, planning_horizon=1, legal_capacity=0, literacy=0, fertility=0 | GOAP只生成 cry/seek_mother; GRASP 任何东西都失败(太重); 完全依赖照料者 |
| **Toddler (幼儿)** | strength=0.15, mental_model_count=2-5, planning_horizon=1-2 | 开始GRASP小物体; 模仿成年人POSTURE; 玩耍涌现(低风险探索); 不能CRAFT(物理参数太低) |
| **Child (儿童)** | strength=0.3, cognitive_damping=MIN(≈0.1), planning_horizon=3, literacy=emerging | 认知阻尼最低=学习速率最高; 开始简单CRAFT(学徒); PLAY 主导行为; READ/WRITE 取决于教育; 不能INVEST/SIGN(legal_capacity=0) |
| **Adolescent (少年)** | strength=0.6, fertility=rising, aesthetic_openness=MAX | AestheticTaste 开始结晶; Courtship 出现; 社交从 Play 转向 Status |
| **YoungAdult (青年)** | strength=0.9, fertility=PEAK, planning_horizon=5-6 | 全原子能力巅峰; 核心婚育期; 职业确立期 |
| **Adult (成年)** | strength=0.95, cognitive_damping=基线 | 全原子能力维持; 权力角色承担(JUDGE/PROCLAIM); 文化传承(Teach) |
| **MiddleAge (中年)** | strength=0.85(缓降), aesthetic_complexity=PEAK | 审美复杂度达峰; 社会资本累积→更频繁使用权力原子 |
| **Elder (老年)** | strength=0.6→0.3, gompertz=指数上升, cognitive_damping=升高, fertility→0 | STRIKE 力量不足→转轻工具/策略; 从生产者→导师/裁判/祭司; 临终整理(WriteLifeTrace); 哀悼丧偶(Mourn) |

### 5.2 涌现的具体例子

**儿童不能锻造** → 不是因为 `if age < 12 { cannot_smith() }` → 是因为 `strength=0.3` 无法满足 `GRASP(8kg大锤)` 的最小力量要求 `8kg/25kg(最大举重)=0.32(归一化阈值) > 0.3(strength)` → 尝试了但拿不动。可能用小锤子打小东西——自然涌现的"学徒做小件"。

**老人不打仗** → 不是因为 `if age > 60 { cannot_fight() }` → 是因为 `strength 从 0.95 连续下降到 0.5` + `stamina 恢复率连续降低` + `gompertz_mortality 指数上升` → STRIKE 效用评估越来越低 → GOAP 自然选择其他行动。但老人可以成为**指挥官/策略师**——因为 strategic 认知不受 strength 影响。

**中毒者不劳动** → 完整走通见 §1.3。

## §六、关键用户提问的记录与回应

以下记录用户在设计讨论中提出的核心问题及设计回应。详见 [[011-关键问题回应与设计论证]]。

### 问题 1: KnowledgeSeed TOML —— 知识起源

**问**: 世界生成时 NPC 已经有了技艺。但"第一个铁匠"的手艺从哪来？在世界刚生成的瞬间，这些 NPC 没有师傅——但在世界内的视角中，他们是从传承中学到的。

**答**: 引入 KnowledgeSeed —— TOML 数据表定义每个技术时代×区域的基础已知技艺。**时代框架和技艺树是手工设计的**（类似 SkillDef），**具体传播路径是历史引擎自动生成的**。世界生成时，的历史模拟阶段（P9）产生传播事件链: "铁器冶炼于 year -847 在聚落 A 被发现" → "经贸易路线传至聚落 B" → "聚落 B 建立铁匠行会 year -650"。当代 NPC 的技能由 `generate_starting_skills()` 赋予，历史事件链提供叙事一致性。详见 [[005-知识种子与历史引擎集成]]。

### 问题 2: 技能精度 —— 高技能 vs 低技能

**问**: 把技艺拆成原子序列后，高技能 NPC 和低技能 NPC 做同一件事——差别在哪？还能用数值表示技艺高低吗？

**答**: 技能 = 执行精度，不是序列门控。Lv 10 和 Lv 90 铁匠执行相同的 HEAT→STRIKE→DIP 序列，但 Lv 10 的温度噪声 ±120°C(实际~243°C×30%)，力度变异 ±32%，OBSERVE 感知准确度 60%——导致 Rough 品质或失败。Lv 90 的温度噪声 ±3°C(实际~3°C×0.3%)，力度变异 ±5%，OBSERVE 感知准确度 99.5%——产出 Refined/Perfect 品质。技能数值仍然有意义——它映射到 `execution_noise_std(level)` 函数。详见 [[006-技能精度与原子执行噪声]]。

### 问题 3: 战斗系统的动画和碰撞

**问**: 战斗到底预设多少动画？碰撞箱怎么做？

**答**: **零预设战斗动画。** 所有攻击/格挡/闪避/受伤动画由 IK 解析解从武器轨迹自动生成。武器碰撞体（capsule/sphere，从 ItemParams 尺寸派生）× 身体碰撞体（15 capsules，从 33/35 骨骨架派生）→ 轨迹扫掠求交 → 命中判定。IK 解算比预设动画混合**更快**（~0.67μs/NPC vs ~6μs/NPC for 9-layer animation stack blend）。详见 [[007-碰撞箱战斗与IK动画管线]]。

### 问题 4: 战斗基元、物理原子和动画的关系

**问**: 同样是枪刺，可以用好几种动画吗？如果我做斧头砍的动画但挂"点"这个动作标签，系统当它是点吗？

**答**: 同一个"刺"基元（物理参数相同），不同 NPC 的 CombatStyleParams 产生完全不同的 IK 解算结果——朴素老兵 vs 花哨年轻人 vs 大师级自创风格。三种不同的视觉，但同一个物理判定。挂错标签 → 系统按错误的物理参数算伤害（斧砍挂"点"=穿刺伤害），但原子引擎不阻止——武器类型→基元的约束是设计层的事。详见 [[011-关键问题回应与设计论证]]。

### 问题 5: MaterialProperties 与涌现

**问**: 是不是意味着物品、动作原子、环境条件构成一套"合成表"？这样能涌现吗？

**答**: MaterialProperties 是**数据**（MaterialDef TOML），不是"if 铁+锤+800°C → 可锻"的规则表。物理原子只做物理公式计算——它从数据中**推导**行为，而非从规则表中**查找**行为。但 CraftingRecipe（配方）确实以 TOML 形式存在——它是**文化知识的形式化**，代表"这个文明发现了这个配方"，可以由师傅教给学徒，可以失传，可以被创新管线实验出新变体。**物理必然性为涌现提供边界，文化知识为涌现提供起点，认知创新为涌现提供增长点。** 详见 [[008-MaterialProperties与物品系统对接]]。

### 问题 6: 动作原子是否需要手动映射到行动？

**问**: 行为层的每个行动都要手动用原子层的动作映射吗？NPC 要等待灌注原子序列才能行动，还是自己领悟？

**答**: 两者都有，但在不同层。**物理必然**不需要任何人知道——冷铁+大锤=打不动，这是 MaterialProperties 自动计算的。**基础工艺**属于文化知识——NPC 通过学徒系统从师傅那里学到 CRAFT 这个 ActionCandidate。**创新工艺**通过认知系统的创新管线涌现——高开放性 NPC 尝试新材料组合，实验出新的配方变体。不是"往 NPC 脑子里灌原子序列"——是 NPC 学到领域复合原子（如 CRAFT），其 execute() 内部已经编排好了物理原子序列。详见 [[011-关键问题回应与设计论证]]。

## §七、性能预算

详见 [[009-性能预算与LOD联动]]。摘要:

| 新增系统 | 每帧成本 | 备注 |
|---------|---------|------|
| 碰撞扫掠 | ≤0.1ms | 单次 0.3μs, capsule-capsule 测试 ~5ns |
| IK 解算 | ≤0.5ms (零增量) | 解析 IK 比预设动画快 ~9 倍, 在现有骨骼预算内 |
| MaterialProperties 查询 | <0.05ms | 24KB 表 L2 缓存常驻 |
| 非 Rigid 武器 PBD | ≤0.3ms | 使用稀有, 动态降级 |
| **总 CPU 增量** | **≤0.45ms** | 6.4% Rust 预算, 被峰值互斥吸收 |
| **总内存增量** | **~10.2 MB** | 0.7% NPC 数据 |
| **VRAM 影响** | **零** | 碰撞全部 CPU 侧 |

典型和平帧 12.05ms (72%), 战斗帧 12.80ms (77%), 最坏重载帧 14.95ms (90%)。

### 10.2 玩家特殊处理

"玩家=NPC"原则下，玩家的 AgentSnapshot 从玩家角色状态中派生——与 NPC 完全相同的数据源。区别在于:
- 玩家的 `execution_noise` 恒为 0（输入精度由玩家操作决定，不由技能噪声调制）
- 玩家的 GOAP 输出被 `ControlMode` 覆盖——玩家输入直接调用 ActionCandidate.execute_fn，绕过 GOAP
- 玩家的 AgentSnapshot.skill_vector_compressed 从玩家角色数据中获取（与 NPC 一样取 5 大类的 max 等级）

### 10.3 调试与可观测性

所有物理原子的执行通过 `SpatialEventBus` 推送 `AtomEvent`:
- 每个 AtomEvent 包含: `{entity_id, atom_type, input_hash, output_hash, timestamp, frame_number}`
- 历史系统的大日志可选消费 AtomEvent 用于事后调试
- 开发工具 (`woworld_devtools`) 可订阅 AtomEvent 流进行实时可视化
- 确定性种子重放: 相同 seed + 相同初始状态 → 相同的 AtomEvent 序列（用于回归测试）

### 10.4 Rust→Godot 数据通道（简要）

物理原子执行→Entity状态变更→变更标记到 entity dirty_flags→帧末 GDExtension 将脏实体序列化为 PackedByteArray→Godot 侧更新 Transform/Mesh/Animation 状态。此通道由现有 `woworld_gdext` crate 实现，物理原子层不直接处理。详见 [[../../技术栈方案/001-WoWorld正式技术栈方案v3|技术栈 v4.0 §三]]。
