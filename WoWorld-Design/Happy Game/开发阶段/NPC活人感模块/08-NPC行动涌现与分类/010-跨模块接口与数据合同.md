# 010-跨模块接口与数据合同

> **最后更新**: 2026-06-18
> **权威声明**: 本模块（08-NPC行动涌现与分类）是物理原子层、AgentSnapshot、MaterialProperties 接口定义的**唯一权威来源**。其他模块中涉及上述概念的描述，应以本文档为准，不应让本模块迁就其他模块中片面、冲突或有遗漏的参考性描述。

---

## 一、与其他模块的双向接口汇总

### 1.1 提供侧：本模块提供给其他模块的 trait / 类型 / 数据

| 接口 | 类型 | 消费方模块 | 关键约定 |
|------|------|-----------|---------|
| **PhysicalAtom** (35个枚举) | enum + execute() 函数 | 全部模块 | 定义在 `woworld_core::atoms`。原子不包含领域知识——仅做物理计算。各领域模块的 execute() 内部调用 |
| **AgentSnapshot** | struct (~108B SoA, v1.1 26字段含CognitiveTide 3字段) | NPC活人感、战斗、经济、信仰、文化... | NPC 能力的连续参数快照。Life/生命周期/技能/天气→AgentSnapshot。各模块的 ActionCandidate evaluator 消费。单 cache line 友好 |
| **MaterialProperties** | struct (24字段,33标量值含elemental_affinity[10]) | 物品系统、战斗、魔法、经济 | 材料的物理/化学/魔法属性。定义在 `woworld_core::materials`。物品系统提供 MaterialDef 注册表 |
| **GripHandle** | struct | 战斗、经济（CRAFT） | 稳定持握句柄。GRASP 产出，供 STRIKE/THROW/RELEASE/DIP 等后续原子使用 |
| **ActionCandidate** | struct | NPC活人感（GOAP） | GOAP 可见的行为单元。各模块注册自己的候选行动，NPC 模块不枚举 |
| **CompoundEntity** | struct | 战斗（武器装配）、建造 | ATTACH 产出的复合物理体。存储连接类型和子实体引用 |
| **StrikeResult / BreakResult / CutResult / ...** | 各原子返回值 enum | 战斗、经济、感官、音频 | 每个物理原子有统一的返回类型。包含成功/失败状态+物理效果数据 |
| **AtomEvent** | struct (推送至 SpatialEventBus) | 感官、音频、历史 | 每个物理原子执行后自动产生。包含: 声音 footprint、视觉标记、AetherImprint 触发条件 |
| **execution_noise()** | 纯函数 | 全部消费物理原子的模块 | `fn execution_noise_std(skill_level: f32) → f32`。技能→原子执行噪声的连续映射 |

### 1.2 消费侧：本模块消费的其他模块 trait / 类型 / 数据

| 接口 | 提供方模块 | 本模块使用方式 |
|------|-----------|-------------|
| **SkillEntry.proficiency()** | 技能系统 | 传入 execution_noise() 计算原子执行精度 |
| **MaterialDef** (TOML 注册表) | 物品系统 | 物理原子执行时查询材料属性（硬度、密度、热导率...） |
| **ItemAssembly** (组件树) | 物品系统 | resolve_material() 遍历装配树以确定物品的整体材料属性 |
| **Vitals / Physiology** | 生命系统 | 派生 AgentSnapshot.physical 参数 |
| **AgeClock / Gompertz** | 生命周期系统 | 派生 AgentSnapshot.lifecycle 参数 |
| **CognitiveStyle / MentalModel** | 认知系统 | 派生 AgentSnapshot.cognitive 参数 |
| **EmotionEngine** | NPC活人感 | 派生情绪对原子执行的调制 |
| **WeatherQuery** | 天气系统 | 派生 AgentSnapshot.environment 参数（wetness, temperature_exposure） |
| **TerrainQuery** (空间查询) | 模型/物理系统 | MOVE/JUMP 原子的地形查询 |
| **EntityIndex** (空间查询) | 模型/物理系统 | STRIKE/THROW 原子的实体索引 |
| **SpatialEventBus** (空间查询) | 模型/物理系统 | 所有原子推送执行事件 |
| **VisibilityQuery** (空间查询) | 模型/物理系统 | OBSERVE 原子的视线查询 |
| **AudioQuery** | 音频系统 | LISTEN 原子的音频查询 |
| **AetherQuery** | 历史系统 | SENSE_AETHER 原子的历史查询 |

## 二、其他模块中涉及本模块概念的现有内容

### 2.1 音频系统 — ActionAtom（58变体）

**现状**: `开发阶段/音频系统/002-声音足迹与发射模型.md` 定义了 58 个 ActionAtom 变体作为声音产出的触发器。

**冲突/需对齐**: 音频系统的 ActionAtom 枚举是"声音分类标签"——按声音特征分类（Swing声/Impact声/Whoosh声）。本模块的 35 个物理原子是"物理交互分类"——按力学特征分类。两者的映射关系已在讨论中明确（见本文件夹 002-物理原子层定义与签名），但音频系统文档应更新以引用本模块原子层为权威声音触发源——**每个物理原子执行时自动产生声音足迹，音频系统根据 AtomEvent 的物理参数自动选择声音，无需维护独立的 ActionAtom 分类。**

**权威归属**: 物理原子定义权在本模块。音频系统的"什么动作产生什么声音"应从物理原子的 AtomEvent 中自动派生。

### 2.2 战斗系统 — 招式基元（10个）

**现状**: `开发阶段/战斗/005-招式积木与流派生成.md` 定义了 10 个招式基元（刺/斩/撩/劈/扫/点/崩/缠/引/化），并描述为"身体可以做什么的最小单位"。

**冲突/需对齐**: 战斗基元是本模块 STRIKE 原子的参数化变体——STRIKE(arc, downward) = 劈，STRIKE(linear, forward) = 刺。战斗系统的文档描述 "不由武器决定" 与本模块的 "武器类型约束可用基元" 一致。但战斗文档中的非 Rigid 武器基元（甩/缠/卷/锁/绞/抡）标记为"尚未设计"——应由本模块和战斗系统联合设计。详见本文件夹 007-碰撞箱战斗与IK动画管线。

**权威归属**: STRIKE 原子的物理定义在本模块。招式基元的参数化方案（轨迹曲线、力度、方向）在战斗模块。两者是叠加关系——战斗模块定义 STRIKE 的"怎么打"，物理原子层定义 STRIKE 的"打出去什么效果"。

### 2.3 物品系统 — MaterialProperties（不存在）

**现状**: `物品系统/` 中没有 MaterialProperties 概念。NPC 认知系统文档（006-创新管线与跨领域对接.md）前向引用了 MaterialProperties 为"已有基础"——现已由 CHG-042 正式设计（本模块 008 + 物品系统 010）。下文描述 CHG-042 之前的现状以供参考。

**需要的改变**: 物品系统需要新增 MaterialDef TOML 注册表（详见本文件夹 008-MaterialProperties与物品系统对接）。这不推翻物品系统的委托哲学——物品系统只存储和提供 MaterialDef 查询，不解释材料属性的含义。

### 2.4 技能系统 — 知识起源空白

**现状**: `技能系统/003-教学与跨模块接口.md` 的 `generate_starting_skills()` 函数接受 `profession_skills` 参数，但从未定义这些技能的初始来源。知识起源被完全留给世界生成模块。

**需要的改变**: 引入 KnowledgeSeed TOML 注册表作为技术时代×区域的知识初始条件。详见本文件夹 005-知识种子与历史引擎集成。不修改 TeachingSession、三层天赋、XP 公式。仅新增 KnowledgeSeed 作为 generate_starting_skills() 的数据源。

### 2.5 技能系统 — 技能精度缺失

**现状**: 技能系统使用 `proficiency() = level/100.0` 做线性映射，用于公式中的乘法因子（格挡成功率、品质影响权重等）。没有"执行噪声"概念。

**需要的改变**: 技能系统新增 `execution_precision()` 函数，映射到本模块的 `execution_noise_std()`。详见本文件夹 006-技能精度与原子执行噪声。

### 2.6 经济系统 — 生产劳动

**现状**: `经济系统/001` 定义了 ProfessionTag（Blacksmith/Farmer/Weaver...），但生产的物理实现被留给"物品系统/技能系统"。

**需要的改变**: 经济模块的 ProfessionTag 应映射到本模块的领域复合原子（CRAFT/HARVEST/DIG...）。ProfessionTag 变成了一个标签——它告诉 GOAP "这个 NPC 的首选 ActionCandidate 是什么"。详见本文件夹 003-领域复合原子与模块注册。

### 2.7 感官系统 — 感知与 OBSERVE/LISTEN

**现状**: 感官系统定义了 SensoryProvider::perceive() 管道，产出 PerceptBatch。感知是被动的——环境刺激→感知。

**对齐**: 本模块的 OBSERVE/LISTEN 原子是**主动的**感知行为——NPC 可以"决定去看"或"决定去听"。这与感官系统的被动管道是互补关系。详见本文件夹 002-物理原子层定义与签名。

## 四、权威性声明

1. **物理原子层的定义权**属于本模块。其他模块中关于"NPC 的身体能做什么"的描述，应以 35 个物理原子为准。其他模块可以定义"在特定情境下用什么参数调用这些原子"（如战斗模块定义招式基元的轨迹参数），但不能定义"存在一个挥砍原子"而物理原子层没有。

2. **AgentSnapshot 的结构定义权**属于本模块。各模块可以定义"AgentSnapshot 的某个字段如何从自己的数据派生"（如生命周期模块定义 age_ratio 的计算方式），但不能新增或修改 AgentSnapshot 的字段。

3. **MaterialProperties 的数据模型定义权**属于本模块。物品系统可以定义 MaterialDef TOML 的具体条目（铁=hardness=0.45），但字段的语义和物理公式由本模块权威定义。

4. **ActionCandidate 注册机制**属于本模块（定义在 woworld_core）。各领域模块可以注册自己的候选行动，但 ActionCandidate 的结构（id, domain, precondition_fn, utility_fn, execute_fn, spatial_target, duration_category）由本模块定义。
