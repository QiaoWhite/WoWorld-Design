## 一、关于行动分类与涌现架构

### Q1: "NPC在不同年龄段下能做的事情是不一样的。尽量不要硬编码，坚持涌现原则。应该怎么组织？"

**用户原始表述**: "接下来我想讨论NPC究竟可以做哪些具体的事情，我们要将他们分门别类。你必须遍历 开发阶段中几乎所有文件。NPC在不同年龄段下能做的事情是不一样的。尽量不要硬编码，坚持涌现原则。"

**设计回应**: 不做"NPC 能做什么"的清单，而是建立三层原子架构。年龄不是门控——年龄→AgentSnapshot 连续参数→所有原子执行精度连续变化→GOAP 效用评估自然分化。不写 `if age < X { cannot_do(Y) }`。详见 [[001-NPC行动涌现总纲#§五、年龄与生命阶段的连续涌现|001 §五]]。

### Q2: "如果NPC有不能做的事，不用硬编码他不能做，应该自然涌现。比如中毒了，不是从设计层面不让他劳动，应该是他产生劳动的想法，评估后放弃了/尝试去劳动，然后失败了。"

**用户原始表述**: "如果NPC有不能做的事，不用硬编码他不能做，应该自然涌现。比如一个NPC中毒了，不用从设计层面不让他劳动，应该是他产生劳动的想法，评估后放弃了/尝试去劳动，然后失败了。"

**设计回应**: 中毒 → AgentSnapshot.health_penalty = 0.6 → 所有原子效用 × 0.4。GOAP 评估: 买药效用 0.95 > 打铁效用 0.096 → 选择买药。如果买不起药 → 尝试打铁 → GRASP(大锤) 的 force_check 失败 → "锤子都拿不动"。全程零 `if poisoned`。详见 [[001-NPC行动涌现总纲#1.3 涌现哲学|001 §1.3]]。

### Q3: "我们归纳的行动，包括寻路吗？"

**设计回应**: 寻路是引擎服务，不在行动归约中。但移动（MOVE 原子）是可观察行为——走到市场、骑马到邻城、游泳过河——三种 MOVE 的 locomotion_type 变体，消耗不同体力，暴露不同风险。详见 [[002-物理原子层定义与签名|002]] 中 MOVE 原子的完整签名。

### Q4: "我们有没有必要从比较原子的、积木式的角度去归纳一些非常基础的行动？"

**设计回应**: 有必要。但边界划在"物理模拟层"而非"行为决策层"。物理原子（MOVE, GRASP, STRIKE...）是极薄极稳的引擎服务层。GOAP 规划层操作的是领域模块注册的 ActionCandidate。三层架构正是这个问题的答案。详见 [[001-NPC行动涌现总纲#§二、三层原子架构|001 §二]]。

### Q5: "塞尔达传说：旷野之息的元素反应、王国之泪的余料建造、博德之门3的抓取/蘸取/投掷——请以这些游戏为参考完善物理原子层。"

**设计回应**: 参考游戏的核心洞察是"极少数规则 × 极丰富的属性空间 = 玩家自己发现涌现行为"。物理原子层从最初 19 个扩展到 35 个，增加了 VENT, OBSERVE, LISTEN, SENSE_AETHER, CONDUCT, WET 等，以及 MaterialProperties 的 30 字段属性表。详见 [[002-物理原子层定义与签名|002]] 和 [[008-MaterialProperties与物品系统对接|008]]。

## 二、关于战斗系统

### Q6: "战斗系统中的动作和这份方案中的动作原子的关系是怎么样的？交易、锻造、种植等动作和动作原子的关系是什么？"

**设计回应**: 战斗基元 = STRIKE 原子的参数化变体。劈 = STRIKE(arc, downward, body_commitment=0.9)，刺 = STRIKE(linear, forward, body_commitment=0.2)。10个已定义基元(目标15-20) × 4 种轨迹曲线 × 8 方向 × 力度连续值 = 无限招式空间。交易/锻造/种植是领域复合原子——EXCHANGE/CRAFT/PLANT——内部编排第一层物理原子的序列。详见 [[001-NPC行动涌现总纲#§二|001 §二]] 和 [[003-领域复合原子与模块注册|003]]。

### Q7: "同样是枪刺击，可以开发用好几种动画吗？如果做斧头砍的动画但挂'点'这个动作，系统当它是点吗？战斗系统的动作、动作原子和动画到底是什么关系？"

**设计回应**: 同一个"刺"基元 × 不同 CombatStyleParams = 无限视觉变体。老兵（flourish=0.1）vs 花哨年轻人（flourish=0.8）vs 流派大师（body_commitment=0.9）——同一个物理判定，三个完全不同的 IK 解算结果。如果斧头砍的骨骼路径挂到"点"标签 → 系统按"点"的物理参数（小接触面+穿刺伤害+精准判定）计算——但伤害数字蹦出来是"穿刺伤害 15"——玩家会困惑。这不是BUG——是设计错误。武器类型→可用基元的约束是物品系统+战斗系统的设计层责任，不是原子引擎的责任。详见 [[007-碰撞箱战斗与IK动画管线|007]]。

### Q8: "战斗系统我到底应该怎么设计？我在多大的程度上要预设动画？我希望战斗能够基于碰撞箱。"

**设计回应**: **零预设战斗动画。** 武器碰撞体（capsule/sphere，从 ItemParams 尺寸派生）× 身体碰撞体（15 capsules，从 33/35 骨骨架派生）→ 轨迹扫掠求交 → 命中判定。IK 解析解从武器轨迹反推: Hand IK → Elbow → Shoulder → Spine Bridge(7 bones) → Hip → Foot IK。需要预设的是数据（碰撞体层级、IK 约束参数、CombatStyleParams 默认分布表），不是动画。详见 [[007-碰撞箱战斗与IK动画管线|007]]。

### Q9: "什么叫IK，什么叫IK解算"

**设计回应**: IK（Inverse Kinematics，逆向运动学）是已知想要的手/脚位置，反推关节角度的算法。正向运动学是"我转了肩30°+肘45°→手在哪"，IK 是"手要在那→肩和肘各转多少"。2-bone 链（手臂、腿）有解析解——余弦定理直接出结果，不需迭代。WoWorld 用 IK 替代预设动画的核心原因: 同一招式意图 × 不同武器×不同敌人×不同地形 → IK 自动生成无限视觉变体，无需制作任何额外动画。详见 [[007-碰撞箱战斗与IK动画管线#§五、IK 动画管线|007 §五]]。

## 三、关于知识起源与技能系统

### Q10: "世界生成时，整个世界在设定上已经是一个存在了很多年的世界。'第一个铁匠'的手艺从哪来？在玩家和开发者的角度，世界刚生成，这些是第一批拥有技艺的NPC。但在NPC生活的世界里，他们认为自己是从别人那里传承的技艺。"

**设计回应**: 这是两套时间尺度的张力——开发者的"世界刚生成"与NPC的"我师父教的传承五百年"。解决方式: KnowledgeSeed TOML 定义技术时间线（"铁器冶炼于 -1500至-500年间在各区被发现"），历史引擎自动生成传播事件链——"Settlement_A 于 year -847 首次炼铁" → "贸易路线传至 Settlement_B" → "Settlement_B 建立铁匠行会"。当代 NPC 的技能由 `generate_starting_skills()` 赋予，其"我师父教的"信念由历史事件链拼出。两个系统不需要数值精确匹配——只需叙事一致。详见 [[005-知识种子与历史引擎集成|005]]。

### Q11: "KnowledgeSeed TOML是什么，里面要填写什么内容，自动生成还是手动填写？"

**设计回应**: KnowledgeSeed 是技术史的时间线元数据——不是往 NPC 脑子里灌动作序列的文件。内容: 时代定义（名称/起止年）、技艺发现类型（Inevitable/Conditional/Breakthrough/Prerequisite）、触发条件（生物群系/资源/前置技艺）、传播速度、基础技能上限。**时代框架和技艺树是手工设计的**（类似 SkillDef TOML），**具体传播路径是自动生成的**（历史引擎根据贸易路线/文化接触/战争自动计算）。详见 [[005-知识种子与历史引擎集成|005]]。

### Q12: "当我们把技艺拆解成这么细致的步骤，低技能和高技能的人的差别在哪，还能用数值的高低描绘人的技艺吗？"

**设计回应**: 技能 = 执行精度，不是序列门控。Lv 10 和 Lv 90 铁匠执行相同的 HEAT→STRIKE→DIP 序列，但 `execution_noise_std(level) = BASE_NOISE × (1 - level/100)²` 决定了原子参数的稳定性。Lv 10: 温度噪声 ±100°C, 力度变异 ±30%, OBSERVE 感知准确度 60%。Lv 90: 温度噪声 ±10°C, 力度变异 ±5%, OBSERVE 感知准确度 99.5%。技能数值（level 0-100）仍然有意义——它映射到连续噪声函数。详见 [[006-技能精度与原子执行噪声|006]]。

### Q13: "什么是技能精度，详细说明。"

**设计回应**: 技能精度不是新数值——是现有 `proficiency() = level/100.0` 在原子执行层的物理体现。噪声标准差 = BASE_NOISE × (1 - proficiency)²。二次衰减: 新手噪声大（proficiency=0.1 → noise_std=0.81×BASE），大师噪声极小（proficiency=0.9 → noise_std=0.01×BASE）。不同的原子有不同的 BASE_NOISE（STRIKE力 ±40%, STRIKE位置 ±15%, HEAT温度 ±30%, OBSERVE感知 ±50%）。高技能工匠不仅手更巧——眼也更毒（OBSERVE 准确度也受同一噪声调制）。详见 [[006-技能精度与原子执行噪声|006]]。

## 四、关于 MaterialProperties 与物品系统

### Q14: "MaterialProperties相关的功能深入讲解，这对物品系统的影响非常大。"

**设计回应**: 参见 [[008-MaterialProperties与物品系统对接|008]]。

MaterialProperties 当前在物品系统中**不存在**——被 NPC 认知文档前向引用了但从未设计。当前材料属性分裂在 ItemProperties、AestheticProps、ParamSchema、EnchantRuneProps 四处分存。

设计方案遵循物品系统现有的 `Option<ConsumableEffect>` 和 `Option<AudioMaterial>` 委托模式:
- 新增 `MaterialDef` TOML 注册表（30 字段，~200 条目，24KB L2 缓存常驻）
- `ItemProperties` 新增 1 个字段: `material: Option<MaterialDefId>`（u32 引用，非内嵌）
- 仅 Material-category 物品（0x20-0x2F）填充；武器/盔甲通过装配树从组件材料**继承**
- **物品系统不解释 MaterialProperties**——只存储和提供查询。消费模块各自读取关心的字段
- 锻造配方: `RecipeInput.acceptable` 新增材料约束字段（"Blade 需要 Metal, hardness>=0.3, melting_point<2000K"——铁/钢/铜/秘银都可，金/铅不行）
- 内存: ItemProperties +8B × 5,000 DefId = 40KB。仅 Material 类 ~500 条实际填充

### Q15: "这种设计是不是意味着，物品、动作原子、环境条件构成一套套合成表？这样能涌现吗？"

**设计回应**: 有两层"表"——但性质完全不同:
1. **物理必然层**（MaterialProperties）: 不是"合成表"——是物理公式。冷铁硬、热铁软——不是"铁+800°C=可锻"的规则，是热软化曲线自动计算。STRIKE 代码中没有"铁会变形"的知识。新增材料=新增一行 TOML 数据，不写新规则。
2. **文化知识层**（CraftingRecipe）: 确实是"配方表"——但它是**文化知识的形式化**，代表"这个文明发现了这个配方"。TOML 数据，可 mod。NPC 通过学徒系统学到。可失传、可被认知创新管线实验出新变体。

**物理必然性提供涌现的边界条件。文化知识提供涌现的起点。认知创新提供涌现的增长点。** 这三者共同构成涌现的完整图景。

### Q16: "反复思考物品系统到底怎么做才能满足我们提出的涌现的要求。"

**设计回应**: 物品系统保持其"基础设施，不解释含义"的核心哲学。为支持涌现，仅需三项新增:
1. **MaterialDef 注册表**: TOML 数据，物品系统存储+查询，不解释。消费模块读取。
2. **材料引用字段**: `ItemProperties.material: Option<MaterialDefId>`。装配树材质继承工具函数。
3. **配方材料约束**: `RecipeInput.acceptable` 支持属性范围而非具体 ID。

不需要推翻现有设计。不需要在物品系统中添加"硬度影响锻造"的逻辑——那是物理原子层的事。

## 六、关于性能

### Q18: "深入评估性能压力"

**设计回应**: 详见 [[009-性能预算与LOD联动|009]]。核心结论:
- 碰撞扫掠: ≤0.1ms/帧。单次 capsule-capsule 相交 ~5ns（20 FMA 指令, SIMD 友好）
- IK 解析解: 零增量（在现有骨骼 ≤0.5ms 预算内）。2-bone analytical IK ~17.5ns/肢体 vs 预设动画混合 ~180ns/骨骼 → IK 实际上更快
- MaterialProperties: <0.05ms/帧（24KB 表 L2 缓存常驻）
- 非 Rigid 武器 PBD: ≤0.3ms/帧（使用稀有，可动态降级迭代次数）
- **总 CPU 增量 ≤0.45ms**（6.4% Rust 预算），被峰值互斥规则吸收
- **总内存增量 ~10.2MB**（0.7% NPC 数据），可忽略
- **VRAM 零影响**（碰撞全部 CPU 侧）
- 最坏重载帧: 14.95ms（90%），余量 1.75ms

### Q19: "行为层的每一个行动都要用原子层的动作来手动映射吗"

**设计回应**: 不需要手动映射每一个行动到物理原子序列。**物理必然**不需要任何人知道——MaterialProperties 自动计算。**基础工艺**以第二层复合原子（如 CRAFT）的形式存在于文化知识中——NPC 学到的是"锻造"这个复合动作，其 execute() 内部已经编排好了 HEAT→STRIKE→DIP 序列。NPC 不需要逐原子理解。**创新工艺**通过认知系统的创新管线涌现——不是预设的配方。

## 七、全局设计原则回应

### 关于 Rust 解耦

物理原子层放在 `woworld_core`——不依赖任何领域模块。各领域模块依赖 `woworld_core::atoms`，但物理原子不反向依赖它们。MaterialProperties 是数据（TOML），不是代码依赖。AgentSnapshot 是临时派生视图——各模块不持有 AgentSnapshot 的持久副本。

### 关于玩家游玩便利性

玩家=NPC=同一套原子引擎。玩家界面上的"攻击"→GOAP 选择 Fight ActionCandidate→执行 STRIKE 原子。玩家不需要手动执行物理原子——GOAP+领域复合原子封装了原子序列。

### 关于可维护性和优雅

三层架构中每一层的变化不会泄漏到其他层。新增材料=新增 TOML 数据行。新增武器类型=注册新的 slot_type 和碰撞体映射。新增文化仪式=注册新的第三层 ActionCandidate 并实现其 execute()。不需要修改原子引擎。

### 关于数学工具

- collision sweep: capsule-capsule intersection (analytical, ~20 FMA)
- IK: law of cosines (analytical, ~70 FMA for 2-bone chain)
- execution_noise: quadratic mapping (1 - proficiency)²
- PBD: Gauss-Seidel iterative constraint solving
- skill XP: exponential deceleration `base × (e^(k×L) - 1) / k`
- MaterialProperties 热响应: linear softening between ambient and melting_point
- CombatStyleParams: 6+ dimensional continuous parameter space
