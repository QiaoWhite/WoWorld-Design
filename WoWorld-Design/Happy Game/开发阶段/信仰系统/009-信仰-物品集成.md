## 一、核心原则

**信仰系统不新增 ItemCategory 枚举值。所有集成走物品系统的现有扩展机制。**

| 策略 | 说明 |
|------|------|
| 不新增 `ItemCategory::HolyItem` | 宗教物品分散在现有类别中——圣剑=武器、经书=书籍、圣遗物=材料 |
| 不定义"神圣伤害"类型 | 伤害类型以 [[../战斗/001-战斗系统总览|战斗系统]] 为权威 |
| 不修改物品核心 struct | `mod_tags` + `DirectEnchantment` + `ItemState.inscription` 已足够 |

## 二、六项集成清单

### 集成 1：神遗物生成

**使用机制**：物品系统 `MagicMat` 类别、子类 神遗物（物品 002 已提及）

```rust
/// 信仰系统提供。世界生成 P3（资源分布）消费。
pub struct DivineRelicTemplate {
    /// 关联的神或信仰
    pub associated_deity: Option<String>,
    pub associated_faith: Option<FaithId>,
    /// 生成条件
    pub spawn_condition: DivineRelicSpawn,
    /// 属性范围
    pub properties: DivineRelicProperties,
}

pub enum DivineRelicSpawn {
    GodfallSite，    // 仅神陨之地——世界生成 002 定义（极稀有 <0.1%）
    AncientTemple，  // 古老神殿遗址——世界生成 P9 历史模拟的废弃神殿
    SaintBurial，    // 圣人埋葬地——生命系统 DeathCustom 标记
    DivineEvent，    // 神迹发生地——A 模式下 GodEntity 显现过的位置
}

pub struct DivineRelicProperties {
    pub rarity_min: Rarity，     // 最低稀有度——神遗物至少 Epic
    pub material: String，       // "神血结晶"/"神骨碎片"/"神器残片"
    pub innate_effect: Option<String>, // 天生效果描述——"持有者缓慢恢复HP"
}
```

世界生成 P3 消费 `DivineRelicTemplate` → 在符合条件的坐标放置 `MagicMat` 神遗物物品实体。

### 集成 2：仪式场所制造

**使用机制**：物品系统 `LocationRequirement::RitualSite`（物品 006 已定义）

信仰系统定义哪些类型的宗教物品制造需要 RitualSite：
- 祝圣武器（圣剑/圣枪）——需要祭坛 + 祭司主持
- 圣物匣（Reliquary）——需要神殿 + 圣人遗骨
- 圣水/圣油——需要神殿 + 祝福仪式
- 诅咒移除——需要 RitualSite + 祭司/萨满

```rust
/// 信仰系统提供——TOML 配方数据消费。
fn requires_ritual_site(item_type_tag: &str, faith_id: FaithId) -> bool {
    matches!(item_type_tag,
        "consecrated_weapon" | "reliquary" | "holy_water" |
        "blessed_armor" | "curse_removal" | "sacred_icon"
    )
}
```

物品系统的制造配方 TOML 在 `LocationRequirement::RitualSite` 字段上引用此判断。

### 集成 3：神圣祝福——叠加 DirectEnchantment

**使用机制**：物品系统 `DirectEnchantment`（物品 008 已定义——永久 + 铭刻名 + 历史记录）

```rust
/// 信仰系统注册的附魔类型。不新增物品系统枚举——通过魔法系统的附魔扩展点。
/// DirectEnchantment 的 provenance 自动记录"谁在何时何地奉何神之名祝福了此物"。
pub struct DivineBlessingEnchantment {
    /// 祝福类型
    pub blessing_type: DivineBlessingType,
    /// 赐福者——神（A 模式）、高阶祭司（任何模式）
    pub blesser: Option<NpcId>,
    /// 以何神之名
    pub in_name_of_deity: Option<String>,
    /// 祝福日期
    pub blessed_at: GameTimestamp,
    /// 祝福效果
    pub effect: DivineBlessingEffect,
}

pub enum DivineBlessingType {
    /// 祝圣——此物现在具有宗教神圣性（对标 RitualDef 的祝圣仪式）
    Consecration,
    /// 祝福——轻微的持续正面效果（如 +5% 对亡灵伤害）
    Blessing,
    /// 神选——A 模式下神亲自祝福（效果强度 ×1.5-2.0）
    DivineChoice,
}

pub enum DivineBlessingEffect {
    /// 对抗特定生物类型的伤害加成
    BaneDamage { creature_type: String， bonus: f32 }，
    /// 持有者某种状态抗性
    StatusResistance { status: String, resistance: f32 }，
    /// 轻微持续恢复（"神圣光环"）
    AuraHealing { hp_per_hour: f32 }，
    /// 信仰标识——不影响属性，但被识别为该信仰的圣物
    FaithMarker,
}
```

`DivineBlessingEnchantment` 叠加在物品系统现有的 `DirectEnchantment` 机制之上——信仰系统注册 `EnchantmentType::DivineBlessing`，物品系统照常处理。区别：DivineBlessing 的 provenance 包含神学信息（以谁之名祝福）。

### 集成 4：天启配方

**使用机制**：物品系统 `RecipeSource::Revealed { event_id }`（物品 006 已定义）

```rust
/// 信仰系统发出 Revelation 事件 → 物品系统发现新配方。
/// 配方内容由物品系统的 TOML 数据驱动——信仰系统只提供发现事件。
pub struct DivineRevelationEvent {
    /// 获启示者
    pub recipient: NpcId,
    /// 启示来源（神/灵/祖先灵/觉悟体验）
    pub source: RevelationSource,
    /// 发现的配方 TOML ID
    pub revealed_recipe_id: String,
}

pub enum RevelationSource {
    DivineVision { deity: String },     // A 模式神显或 B/C 模式的神秘体验
    AncestralWisdom，                   // 祖先崇拜——"祖先在梦中告诉我"
    MeditationInsight，                 // 禅修/萨满出神中的领悟
    SacredTextDecipher，                // 从经文中解读出隐藏配方
}
```

### 集成 5：圣物铭文

**使用机制**：物品系统 `ItemState.inscription: Option<String>`（物品 001 已定义）

信仰系统提供祷文/祝福文本模板 → 语言表达系统生成具体文本 → 写入 inscription。

```rust
/// 信仰系统提供的铭文模板——语言表达系统填充内容。
struct FaithInscriptionTemplate {
    template_id: u32,
    /// 铭文类型
    inscription_type: InscriptionType,
    /// 包含的命名占位符
    placeholders: Vec<NamedPlaceholder>,
}

enum InscriptionType {
    Prayer，         // 祷文——"主啊，请守护此剑的持有者"
    BlessingFormula， // 祝福句式——"奉圣父、圣子、圣灵之名祝圣"
    ScriptureQuote，  // 经典引用——"《圣典》第X章第Y节"
    DeityName，       // 仅神名——"宙斯"（刻在武器上祈求庇护）
}
```

### 集成 6：信仰审美标签

**使用机制**：物品系统 `StyleTag::Religious`（物品 003 已定义——纯美学分类）

信仰系统通过 `FaithQuery` 输出信仰审美偏好。物品在世界生成/制造时，根据制造者的信仰决定是否添加 `StyleTag::Religious`。

## 三、不做什么（明确边界）

| 不做 | 原因 |
|------|------|
| 不新增 `ItemCategory::HolyItem` | 宗教物品分散在现有类别——武器/书籍/材料/消耗品 |
| 不定义"神圣伤害"类型 | 战斗系统为伤害类型权威——如有神圣伤害，由战斗模块定义 |
| 不修改 ItemProperties 核心 struct | `mod_tags` 已为扩展预留空间 |
| 不新增附魔 slot 类型 | `DirectEnchantment` 已支持信仰祝福（permanent + provenance） |
| 不定义"圣物"品质轴 | 现有的 Quality×Rarity 双轴已足够——圣物表现为高 Rarity（Epic/Legendary）+ 特殊 provenance |
| 不创建独立的"仪式物品"清单 | RitualSite + TOML 配方数据驱动——不硬编码 |

## 四、物品系统的修改

唯一的修改需求：物品系统 009 的跨模块接口表中**新增信仰系统为第 8 个接口伙伴**（原为 7 个模块）。新增 3 行：

| 接口 | 提供方 | 消费方 | 约定 |
|------|--------|--------|------|
| `DivineRelicTemplate` | 信仰系统 | 世界生成 P3 | 神遗物的生成条件与属性范围 |
| `DivineBlessingEnchantment` | 信仰系统 | 物品系统（DirectEnchantment 扩展） | 通过魔法系统的附魔扩展点——信仰注册 EnchantmentType::DivineBlessing |
| `FaithItemExtension` | 信仰系统 | 物品系统（ItemProperties.mod_tags） | 信仰标签（Blessed/Consecrated/Cursed/Profane/Relic）的语义定义 |
