## 二、WeatherQuery — 唯一轮询接口

```rust
/// 天气系统对外的唯一数据接口
/// 所有消费模块通过此 trait 获取天气数据
/// 零事件总线——全部同步轮询
trait WeatherQuery {
    /// ─── 核心查询 ───
    
    /// 查询任意位置、任意时间的天气快照
    /// pos: 世界坐标 (x, y, z)  — y=海拔 m
    /// time: 游戏时间（历史查询用过去时间，运行时用当前时间）
    /// 返回: 该位置该时间的完整天气样本
    fn sample(&self, pos: Vec3, time: GameTime) -> WeatherSample;
    
    /// ─── 历史气象查询（低频——历史模块月度调用） ───
    
    /// 查询某区域一段时间内的极端天气记录
    /// 用于慢轮询消费者不遗漏短期极端天气
    fn extreme_events_since(
        &self,
        region: ClimateRegionId,
        since: GameTime,
    ) -> &[ExtremeWeatherRecord];
    
    /// ─── 便捷查询（委托给 SeasonClock 的子 trait） ───
    
    /// 获取当前季节
    fn current_season(&self, time: GameTime) -> Season;
    
    /// 获取某纬度+时间的日长（小时）
    fn day_length_hours(&self, latitude: f32, time: GameTime) -> f32;
    
    /// 获取某纬度+时间的太阳高度角 (rad)
    fn solar_elevation(&self, latitude: f32, time: GameTime) -> f32;
    
    /// 获取天文学天象事件
    fn sky_events(&self, latitude: f32, time: GameTime) -> SkyEvents;
    
    /// 获取日长（含午休修正）的 NPC 有效活动时长
    fn effective_daylight_hours(&self, latitude: f32, time: GameTime) -> f32;
}
```

**为什么 season 查询也在 WeatherQuery 中？**
季节本质上是天气的驱动力，且绝大多数消费方同时需要天气和季节信息。将两者合并进一个 trait 减少依赖注入参数数量。接口隔离原则在理论上是正确的，但实践中每个消费方都同时需要两者——分开反而产生碎片化。

### 2.1 ExtremeWeatherRecord

```rust
/// 极端天气记录——供历史模块低频查询
struct ExtremeWeatherRecord {
    start_time: GameTime,
    end_time: GameTime,
    weather_type: ExtremeWeatherType,
    max_intensity: f32,
    max_wind_speed: f32,
    affected_area_km2: f32,
    climate_region: ClimateRegionId,
    /// 是否已由历史模块转化为 HistoricalEvent
    recorded_by_history: bool,
}
```

天气系统内部为每个 ClimateRegion 维护最近 3 个月的极端天气循环缓冲（~50条 × 64 bytes = 3.2KB）。历史模块每月调用 `extreme_events_since()`，天气系统遍历缓冲返回匹配记录。零分配。

## 三、依赖注入 — 天气系统从外部读取的 trait

```rust
/// ─── 世界生成 → 天气系统 ───

/// 气候参数场查询
trait ClimateParamsQuery {
    /// 查询任意位置的气候基线参数
    fn climate_params_at(&self, pos: Vec3) -> ClimateParams;
}

struct ClimateParams {
    /// 纬度
    latitude: f32,
    /// 年均温度 °C
    avg_temperature: f32,
    /// 年均湿度 0-1
    avg_humidity: f32,
    /// 季节性变化幅度 — 热带~5°C，温带~25°C
    seasonality_amplitude: f32,
    /// 年均降水量 mm（归一化）
    annual_precipitation_normalized: f32,
}

/// 高程查询
trait ElevationQuery {
    fn elevation_at(&self, pos: Vec3) -> f32;        // m
    fn slope_aspect_at(&self, pos: Vec3) -> Vec2;    // 坡向法线 (水平分量)
    fn terrain_type_at(&self, pos: Vec3) -> TerrainType;
    fn is_valley(&self, pos: Vec3) -> Option<Vec2>;  // 山谷轴线方向
}

/// 群系微气候查询
trait BiomeMicroclimateQuery {
    fn biome_microclimate_at(&self, pos: Vec3) -> BiomeMicroclimate;
}

struct BiomeMicroclimate {
    canopy_closure: f32,        // 冠层郁闭度 0-1
    albedo: f32,                // 地表反照率 0-1
    ground_type: GroundType,     // BareSoil/Sand/Rock/Urban/Water/Ice
    wetland_fraction: f32,      // 湿地比例 0-1
    urban_fraction: f32,        // 城市建筑覆盖率 0-1
    distance_to_water_km: f32,  // 最近开放水体距离 km
}

/// 洋流查询
trait OceanCurrentQuery {
    fn current_at(&self, pos: Vec3) -> Option<OceanCurrent>;
}

struct OceanCurrent {
    temperature_deviation: f32,  // 洋流温度偏差 °C
    direction: Vec2,             // 洋流方向
    speed: f32,                  // m/s
}

/// 世界边界查询
trait WorldBoundaryQuery {
    /// 是否在世界边界风暴带内
    fn is_in_storm_zone(&self, pos: Vec3) -> bool;
    /// 风暴带强度 0-1（进入越深越强）
    fn storm_zone_intensity(&self, pos: Vec3) -> f32;
}

/// ─── 技术栈 → 天气系统 ───

trait PerformanceBudget {
    fn weather_cpu_budget_ms(&self) -> f32;    // ≤0.2ms
    fn weather_particle_budget_ms(&self) -> f32; // ≤0.5ms
    fn weather_vram_budget_mb(&self) -> f32;     // 增量≤5MB
}
```

### 4.1 NPC 活人感模块

```
NPC 轮询频率: 每决策周期 (0.5-5s)

├─ cloud_cover → SkyPerception.cloud_cover (人格过滤)
├─ cloud_type → SkyPerception.cloud_type

├─ ground_wind_speed → 户外活动意愿调节
├─ surface_state → 移动速度修正
├─ is_extreme_weather + extreme_weather_type → 行为权重强制偏移
├─ precipitation_trend + weather_stability → "快要下雨了"社交对话触发
└─ lightning_last_strike → 惊吓情绪触发

NPC 自己维护:
├─ SkyPerception (主观天象感知——人格+情绪过滤)
├─ perceived_temperature (体感温度——衣着/代谢修正)
├─ 连续低温/高温天数计数器
└─ 极端天气暴露计数器

三层极端天气响应:
├─ L1 (普通天气): 温和偏移——"多云→户外工作意愿-5%"
├─ L2 (极端天气警告): 强制偏移——"无防护户外活动权重×0.1"
│   └─ 但 NPC 仍可出门——如果觉得值得 (救命/埋伏)
└─ L3 (灾难级): 硬约束——"所有非紧急户外活动取消"
    └─ NPC 人格不可覆盖 (台风天躺在外面晒太阳不合理)
```

### 4.2 植物系统 (Life 009)

```
植物轮询频率: 每游戏时 (~48s 实时)

消费字段:
├─ ground_temperature → 生长度日累积 / 冻害判定 / 冬季休眠触发
│   ├─ GDD (Growing Degree Days) = Σ max(0, ground_temp - species.base_temp)
│   ├─ 超过GDD阈值 → 开花/结果/落叶
│   └─ ground_temp < frost_tolerance → 冻害→健康度下降
├─ ground_humidity + precipitation_intensity → 水分胁迫
│   ├─ 有效水分 = f(降水强度, soil_moisture, canopy_interception)
│   └─ 水分胁迫 = max(0, species.water_min - effective_water)
├─ surface_state (Snow) → 冠层积雪截留 / 雪下存活
├─ wind_direction + ground_wind_speed + gust_factor → 风致破坏
│   ├─ gust_factor > 阈值 → 树枝断裂概率
│   └─ 持续风向 → 树冠偏斜生长
├─ cloud_cover → 光合有效辐射修正
├─ solar_elevation → 光周期感知 (短日→落叶触发)
├─ is_extreme_weather → 冰雹击落果实 / 风暴倒树
└─ current_season() → 开花/结果/落叶的季节窗口

季节/天气交互的七个通道 (已在植物 009 §二十二定义):
├─ 风 (vertex shader sway + 破坏概率)
├─ 雨 (土壤水分 → 生长速率)
├─ 冠层截留 (降水到达林下的比例)
├─ 雪 (冠层载雪 → 滑落/枝断)
├─ 极端温度 (冻害/热枯)
├─ 雾 (叶面湿润 → 真菌/附生植物)
└─ 组合效应 (暴风雪/冰雹/雨后暴晒)

植物只消费 ground_* 字段——植物不关心大气层发生了什么。
```

```

└─ is_extreme_weather → 暴风雨→鸟群迫降 / 寒流→集体南迁

动物自己维护:
├─ 连续低温/高温天数计数器
├─ 迁徙能量累积
└─ 冬眠准备状态
```

```

消费字段:
├─ visibility_m → 视觉侦查距离 / 远程武器有效射程

├─ is_extreme_weather → 环境剧变推送给战斗感知
│   ├─ 初始快照: 战斗开始时的环境印象
│   ├─ 事件驱动推送: 环境剧变时世界推给NPC
│   └─ 战略层重新评估: "暴风雨来临→撤退/趁乱突袭"
└─ precipitation_type (Snow) → 地面足迹可见 (追踪优势)

不消费的字段:
└─ lightning_risk — 概率太低，不是战术决策的输入。
   雷电在战斗中的唯一意义是感官干扰 (闪光/雷声→瞬间注意力偏移)
   → 由 NPC 感知系统处理，非战斗系统

环境元素联动 (不在 WeatherSample 中——由魔法模块自行推导):
├─ 雨天→电系优势 (水导电)
├─ 干燥→火势蔓延
├─ 水域→电导电范围扩大
├─ 暴风雪→水(冰)系增强
└─ 大风→火势扩散

### 4.5 魔法系统

```
魔法轮询频率: 施法时 / 环境元素浓度每游戏10分钟

天气系统只提供基础物理参数 → 魔法模块自行聚合多因子推导元素浓度:

水元素浓度 at (pos, time):
├─ 天气因子: precipitation_type + intensity (雨→水↑)
├─ 地理因子: distance_to_water (世界生成)
├─ 魔法因子: 附近水元素法阵 (魔法模块)
├─ 事件因子: 最近水元素法术 AetherImprint (历史模块)
└─ 地脉因子: 水元素地脉节点 (世界生成/魔法模块)

天气系统提供的参数:
├─ precipitation_type + intensity → 天气对元素浓度的贡献
├─ ground_temperature → 火元素浓度基线 (高温↑)
├─ wind_speed → 风元素浓度基线
├─ cloud_cover + cloud_type (积雨云) → 雷元素浓度基线
├─ surface_state (Snow/Ice) → 冰元素可用性
└─ ground_humidity → 水元素浓度补充

仪式魔法:
└─ 天气操控仪式: 查询目标区域天气→计算操控难度
    难度 = f(目标天气与当前天气的"距离", 区域面积, 施法者能力)
    不属于天气系统的职责——魔法模块自行定义

魔力恢复:
└─ 环境元素浓度影响恢复速率——魔法模块的魔力体系 (005) 决定
    天气只提供基础参数，不做任何魔力计算
```

```

├─ ground_humidity → 食物腐败加速
│   └─ 湿度×温度→腐败速率 (物品系统内部公式)
├─ lightning_risk → 金属装备引雷 (极低概率——万分之一/暴风雨帧)
│   └─ 概率足够低，不是玩家可预期的策略——纯意外事件

所有修正公式在物品系统内部——天气只提供物理事实。
物品系统也不通过天气系统做锈蚀/腐败——自己维护计数器。
```

├─ surface_state → 攀爬/跳跃/骑乘检定修正
│   ├─ Ice → 攀爬 DC +10
│   ├─ Mud → 骑乘 DC +5
│   └─ Snow → 追踪检定 DC -5 (足迹清晰)
├─ ground_wind_speed + gust_factor → 飞行/射箭检定修正
│   ├─ 阵风 > 阈值 → 飞行 DC +8
│   └─ 侧风 → 射箭偏转 (叠加战斗系统的弹道)
├─ precipitation_intensity → 潜行听觉掩护
│   ├─ 中雨: 听觉感知距离×0.7
│   └─ 大雨: 听觉感知距离×0.4
├─ solar_elevation → 夜间技能检定 (黑暗惩罚独立于天气)
└─ cloud_cover → 观星/导航检定修正 (阴天无法靠星导航)

技能的检定修正公式在技能系统内部——天气系统不定义 DC。
```

### 4.8 海洋系统 (技术栈方案)

```
海洋轮询频率: 每游戏10分钟

消费字段:
├─ regional_wind_speed → Gerstner 波高
│   └─ 浪高 = f(风速, 水深, 离岸距离) — 海洋模块内部计算
├─ wind_direction → 浪方向
├─ is_extreme_weather → 暴风雨→浪高×3-5
├─ storm_front_distance → 航海危险预警
├─ surface_state (Ice) → 极地海域冰封判定
└─ gust_factor → 异常大浪 (Rogue Wave) 概率

天气系统不控制海洋——海洋模块通过 OceanProvider trait 消费天气数据，
自行推导波高/浪向/洋流。天气只提供风速和风向。
```

### 4.9 载具系统

> ★ 经 [[../../../参考文档/023-载具系统设计探讨-20260615/设计草稿/001-总纲与核心概念|载具系统设计]] 确认——
> 日常航行不受自然力影响（NPC想走什么路线就走什么路线），仅灾害级天气施加硬约束。
> 动力来源多样性（魔力/人力/畜力/风力/混合）不影响此规则。

```
载具轮询频率: 每决策周期 (每载具每1-10s)

消费字段（仅保留灾害级——日常航行不受影响）:
├─ is_extreme_weather + storm_front_distance → 航行危险评估
│   ├─ 暴风雨→浪高×3-5→载具可能受损→需要靠岸维修
│   ├─ 台风→载具摧毁风险→所有海上载具强制返回港口
│   └─ 对所有动力类型统一作用——差异仅在于损伤通道
│       帆船: propulsion(桅杆折断), 魔力船: propulsion(引擎过载)
├─ visibility_m → 航行安全距离 / 触礁风险
├─ surface_state (Ice) → 内河/近海冰封→航线中断
├─ precipitation_type → 甲板湿滑→NPC移动速度修正 (已在NPC文档定义)
│   └─ 暴风雨: 乘客移动速度×0.6 + 随机方向扰动×0.2
├─ wave_height (海洋模块从风速推导) → 晕船/货物受损/NPC安抚任务
└─ gust_factor → 突发横风→小型船倾覆风险

  ~~regional_wind_speed + wind_direction → 帆船航速~~     (已删除——日常不受风力影响)
  ~~魔力驱动船/帆船区分~~                                 (已删除——灾害对所有动力统一作用)

载具 AI (NPC在载具上):
├─ MaintainVehicle (暴风雨后维修)
├─ AssistPassengers (安抚/引导乘客)
└─ 船长决策: 绕行/靠岸/继续航行——基于天气+船况评估（L3战略层，NPC人格驱动）
```

### 4.10 历史系统

```
历史轮询频率: 每游戏月 (运行时tick) / 一次性 (世界生成P9)

世界生成 P9 (历史模拟):
├─ HistoricalWeatherQuery::all_anomalies()
│   → 全部 HistoricalWeatherAnomaly 序列 (5K-20K条)
├─ 历史模块转化为:
│   ├─ 根事件: 气象异常→调整 Drought/Flood/ClimateShift 根事件概率
│   ├─ 因果传播: 异常作为 world_context_modifier 输入
│   │   (连续3年干旱→Famine概率×3.0)
│   ├─ ClimateCycle 趋势参数化: amplitude+period+phase 来自天气系统
│   └─ 重大异常→HistoricalEvent (EventType::Drought/Flood/etc.)
│
└─ 约束: 历史模块的自然事件生成必须消费气象异常数据，
         不能独立随机。避免"历史说大旱但气象记录显示正常"

运行时 (月度tick):
├─ WeatherQuery::extreme_events_since(region, last_month)
│   → 过去一个月的极端天气记录
├─ 评估 impact_score:
│   ├─ 受影响人口 > 500? → 写入 event_log
│   ├─ 建筑损毁 > 阈值? → 生成 BuildingDamage WorldStateDelta
│   └─ 否则 → 不记录为独立事件 (走NPC个体记忆)
└─ 记录格式: GenerationSource::ImpactTrigger

ClimateCycle 接口:
├─ 天气系统提供: ClimateCycleParams { amplitude, period_years, phase }
├─ 历史模块消费: Trend::ClimateCycle 用此参数初始化 natural_rate
└─ 趋势层在 200-500 年周期上波动——影响全部事件概率
```

### 4.11 世界生成

```
世界生成消费天气数据:

体素密度场 L10 (天气临时层):
├─ snow_accumulation_rate (来自 WeatherVisualPacket)
│   → 写入密度场材质——雪盖
├─ surface_state → 判断积水/积雪/冰
└─ 更新频率: 每 Chunk 每 30s (技术栈 v3 §4.4)

游牧迁徙走廊:
├─ 消费 SeasonState → 确定季节性草场位置
└─ 迁徙路线 = f(季节, 草场生产力)

海上交通网络:
├─ 季风方向 → 航线权重
├─ 季节冰封 → 航线季节限制
└─ 风暴频率 → 航线风险等级

注意: 世界边界的风暴带不属于天气系统——是世界生成的软边界逻辑。
     天气系统通过 WorldBoundaryQuery 查询风暴带，返回区域内覆盖天气。
```

### 4.12 Godot 渲染层

```
渲染轮询频率: 每帧 (16.7ms)

天气系统生产 WeatherVisualPacket → Godot 渲染层消费:

struct WeatherVisualPacket {
    // ── 天空色调 ── 8 floats = 32 bytes
    sky_color_grade: SkyColorGrade {
        rayleigh_color_mult: Vec3,   // 晴天=深蓝，沙尘暴=棕黄
        mie_color_mult: Vec3,        // 地平线色调
        exposure_mult: f32,          // 阴天降低
        saturation_mult: f32,        // 雨天降低
    },
    
    // ── 云配置 ── 推给混合体积云 compute shader
    cloud_config: CloudConfig {
        coverage: f32,              // 云覆盖率→密度场阈值
        base_altitude: f32,         // 云底高度
        thickness: f32,             // 云层厚度
        density_scale: f32,         // 密度乘数
        horizon_coverage: f32,      // 2D impostor
        cirrus_intensity: f32,      // 卷云强度
        cloud_type: CloudType,      // 积云/层云/积雨云/卷云
        update_interval: u8,        // 1=每帧, 2=每2帧
    },
    
    // ── 降水粒子 ──
    precipitation_config: PrecipitationConfig {
        particle_type: PrecipitationParticle,
        intensity: f32,             // 粒子密度
        drop_size: f32,             // 粒子大小
        fall_speed: f32,            // 下落速度
        wind_drift: Vec2,           // 风致偏移
        camera_space_only: bool,    // 仅相机周围（优化）
    },
    
    // ── 雾 ──
    fog_params: FogParams {
        mode: FogMode,              // Linear/Exponential
        density: f32,
        start_distance: f32,
        color: Vec3,                // 辐射雾=白, 沙尘=棕, 海雾=灰蓝
        height_fog: bool,           // 高度雾（山谷雾）
        height_fog_base: f32,
    },
    
    // ── 风视觉 ──
    wind_visual: WindVisualParams {
        direction: Vec2,
        base_speed: f32,
        gust_amplitude: f32,
        gust_frequency: f32,
    },
    
    // ── 闪电 ──
    lightning_flash: Option<LightningFlash> {
        intensity: f32,
        position: Option<Vec3>,     // 云地闪位置
        delay_seconds: f32,         // 距上次闪电
        branch_count: u8,
    },
    
    // ── 地表 ──
    surface_wetness: f32,           // 0-1 控制 specular
    snow_accumulation_rate: f32,     // mm/游戏时 (→世界生成 L10)
    
    // ── 过渡 ──
    transition_blend: f32,          // 0-1 天气变化混合系数
}

总大小: ~200 bytes/帧
通过 GDExtension PackedByteArray 传递
```

**云更新策略** (对齐技术栈 v3 §5):
- 每 2 帧更新一次体积云（分摊 ≤0.75ms/帧）
- `update_interval` 字段让 Godot 决定跳过帧

**降水粒子性能**:
| 类型 | 粒子数 | GPU预算 | 备注 |
|------|--------|---------|------|
| 小雨 | ~200 | <0.1ms | GPU particles shader |
| 中雨 | ~500 | <0.2ms | +简单溅射贴片 |
| 大雨 | ~800 | <0.3ms | +地面涟漪贴片+屋顶滴水 |
| 暴风雨 | ~1200 | <0.4ms | 全效果 |
| 雪 | ~300-600 | <0.2ms | 更慢下落，更大尺寸 |
| 冰雹 | ~200 | <0.1ms | 快速下落+弹跳音效 |
| 无降水 | 0 | 0 | 跳过 |

### 4.13 Godot 音频层

> **★ v1.0 权威归属**: 环境音频的渲染规则以 [[../音频系统/001-音频系统总纲|音频系统]] 为权威。
> 天气系统只提供 `WeatherVisualPacket` 数据（对标——数据触发信号），音频模块决定如何映射为具体的声音资产、混合规则和优先级。
> 风矢量同时作为传播引擎的调制输入（音频模块直接消费 `WeatherQuery::sample()` 获取 `wind_speed`/`wind_direction`）。

```
音频轮询频率: 每帧 (16.7ms)

消费 WeatherVisualPacket（数据触发——具体播放规则以音频模块为权威）:
├─ precipitation_config.particle_type 变化 → 触发雨声/雪声切换
├─ precipitation_config.intensity → 降水音量
├─ wind_visual.base_speed → 风声强度
├─ wind_visual.gust_amplitude → 阵风呼啸声
├─ lightning_flash 出现 → 安排雷声延迟播放
│   └─ 延迟 = distance(position, listener) / 340.0 (m/s)
└─ transition_blend → 天气音效平滑过渡

★ 以下管理职责已迁移至音频模块:
音频系统自行管理（见 [[../音频系统/004-混合与掩蔽引擎|音频 004]]）:
├─ 环境音效混合 (雨声+风声+雷声+树叶沙沙)
├─ 室内/室外衰减
├─ 音效优先级和最多同时播放数
└─ 音频资源加载/卸载
```
