## 一、设计原则

### 1.1 热冷分离

数据的「使用频率」决定存储方式：

```rust
/// 热数据——每帧的空间查询、渲染
struct ComponentHot {
    spatial: Transform,       // 世界空间变换
    bounds: Aabb,             // 包围盒
    mesh_key: MeshKey,        // 渲染资源标识
    health: f32,              // 结构健康度 0-1
}
// 连续内存排列，利于 CPU 缓存

/// 冷数据——偶尔查询的历史、风格
struct ComponentCold {
    family: ComponentFamilyId,
    placed_at: GameTime,
    placed_by: Option<NpcId>,
}
// 与热数据分离存储——历史查询不污染渲染缓存线

/// 参数——求解时和放置时才需要
struct ComponentParams { /* 见 001 */ }
```

### 1.2 主存储 = 组件列表 + 空间索引

大多数查询（坐标→建筑，建筑→组件列表）不需要图遍历。连接图按需构建——只在结构完整性求解和翻修分析时需要。

## 二、ComponentInstance

```rust
/// 组件实例——ComponentFamily 的运行时具体化身
pub struct ComponentInstance {
    pub id: ComponentInstanceId,     // 全局唯一 u64
    pub hot: ComponentHot,
    pub cold: ComponentCold,
    pub params: Box<dyn ComponentParams>,
    pub floor: i8,                   // 所在楼层 (0=地面, -1=地下, 1=二楼)
    pub vertical_span: u8,           // 跨多少层（高墙可能从层0跨到层1）
}
```

**为什么组件而非房间是基本物理单元**：房间是设计的语义单位（BSP 分区），但没有物理本质——没有质量、没有连接面、没有载荷。组件是「物理上真实的东西」——能承重、能连接、能损坏。房间由组件围合而成。

## 三、Building

```rust
/// 建筑：组件图 + 空间索引 + 所有权
pub struct Building {
    pub id: BuildingId,                   // 全局唯一 u64
    pub components: Vec<ComponentInstance>,  // 主存储——热数据连续
    pub spatial_index: BuildingSpatialIndex, // 快速空间查询
    pub foundation_type: FoundationType,
    pub structural_score: f32,            // 结构完整性评分 0-1
    pub build_at: GameTime,               // 建造时间
    pub owner: OwnershipState,
    pub source: BuildingSource,
}

/// 建筑来源
pub enum BuildingSource {
    WorldGen {
        seed: u64,
        template_id: Option<TemplateId>,
    },
    PlayerBlueprint {
        hash: [u8; 32],         // Blueprint TOML 的 SHA-256
        placed_by: PlayerId,
    },
    NpcConstruction {
        initiated_by: NpcId,
    },
    /// 从简易庇护所升级
    UpgradedFromShelter {
        original: ShelterId,
    },
}

/// 所有权状态
pub enum OwnershipState {
    OwnedBy(EntityId),           // NPC / 公会 / 玩家
    SharedBy(Vec<EntityId>),     // 夫妻、合伙人
    Abandoned { since: GameTime },
    Public,                      // 广场、市政厅
    Contested {                  // 所有权争议中
        claimants: Vec<EntityId>,
        dispute_since: GameTime,
    },
}
```

**重要设计决策**：Building 没有 `building_type` 字段。世界生成时使用的 `BuildingTypeTag` 只在生成过程中使用，建成后消失——建筑就是建筑，标签不是建筑的属性。

## 四、BuildingSpatialIndex

建筑内部空间索引——不是为了整个世界的 octree（那是 World Gen/Terrain 的职责），而是为了**建筑内部**的快速查询。

```rust
/// 建筑内部空间索引——轻量分块网格哈希
pub struct BuildingSpatialIndex {
    /// 建筑在世界空间中的 AABB
    outer_bounds: Aabb,
    /// 子网格尺寸 (m)——通常 = 主网格 1.0m
    cell_size: f32,
    /// 稀疏网格：cell_key → 该 cell 内的组件索引列表
    cells: HashMap<(i32, i32, i32), Vec<usize>>,
}

impl BuildingSpatialIndex {
    /// 给定世界坐标，返回此建筑内包含该坐标的所有组件索引
    pub fn components_at(&self, pos: WorldCoord) -> Vec<usize> { ... }

    /// 给定区域，返回相交的组件索引
    pub fn components_in_aabb(&self, min: WorldCoord, max: WorldCoord) -> Vec<usize> { ... }

    /// 重建索引（翻修后）
    pub fn rebuild(&mut self, components: &[ComponentInstance]) { ... }
}
```

建筑组件通常 <500，简单的分块网格哈希即可。不需要通用 octree。

## 五、Region（空间分区——BSP 的产物）

BSP 房间分区产生 Region。Region 不存储功能标签——它只存储空间边界。

```rust
/// 建筑内的空间分区
pub struct Region {
    pub id: RegionId,
    pub building_id: BuildingId,
    pub floor: i8,
    pub boundary: Aabb2D,           // 此层内的 2D 边界
    pub ceiling_height: f32,
    pub adjacent_regions: Vec<(RegionId, Portal)>,
}

/// 区域之间的通道
pub struct Portal {
    pub position: Vec3,
    pub portal_type: PortalType,
    pub connects_to: PortalDestination,
    pub is_locked: bool,
    pub dimensions: (f32, f32),     // 宽×高——用于通过性检查
}

pub enum PortalType {
    Door, Gate, Hatch, Window, OpenArchway, Stairway,
}

pub enum PortalDestination {
    Outside,
    Region(RegionId),
    Building(BuildingId),
}
```

## 六、BuildingHistory（建筑历史）

双层时间窗口存储。建筑随时间「讲故事」——不同时代的修改留下可见痕迹。

```rust
/// 建筑修改历史——双层存储
pub struct BuildingHistory {
    pub building_id: BuildingId,
    /// 活跃层：过去 10 游戏年的所有修改，完整保留
    pub active: VecDeque<HistoryEntry>,
    /// 归档层：每 50 年一个完整快照 + 聚合摘要
    pub archived: ArchiveSummary,
}

pub struct HistoryEntry {
    pub time: GameTime,
    pub entry_type: HistoryEntryType,
    pub components_changed: Vec<ComponentInstanceId>,
    pub triggered_by: Actor,
    pub style_snapshot: Arc<StyleSnapshot>,
    pub reason: Option<String>,  // NPC 为什么做此修改（从认知系统捕获）
}

pub enum HistoryEntryType {
    Built,          // 初始建造
    Renovated,      // 翻修
    Expanded,       // 扩建（增加组件）
    Damaged,        // 破坏（战斗/灾害）
    Repaired,       // 修复
    Demolished,     // 拆除部分组件
    OwnershipChanged,
    ShelterUpgraded, // 从庇护所升级
}

pub enum Actor {
    Npc(NpcId),
    Player(PlayerId),
    Weather,
    Combat,
    Magic,
    Decay,         // 自然风化
}

/// 风格快照——建造/修改时的文化参数冻结
pub struct StyleSnapshot {
    pub culture_fingerprint: u64,
    pub faith_fingerprint: u64,
    pub climate_fingerprint: u64,
    pub race_fingerprint: u64,
    pub timestamp: GameTime,
}
```

### 6.1 归档层

```rust
pub struct ArchiveSummary {
    /// 每 50 年一个完整快照（建筑状态概要）
    pub snapshots: Vec<BuildingSnapshot>,
    /// 快照之间的聚合事件
    pub events_between: Vec<AggregatedEvent>,
}

pub struct BuildingSnapshot {
    pub time: GameTime,
    pub component_count: u32,
    pub structural_score: f32,
    pub ownership: OwnershipState,
    pub floor_count: u8,
    pub total_area: f32,
}

pub struct AggregatedEvent {
    pub time_range: (GameTime, GameTime),
    pub event_type: HistoryEntryType,
    pub count: u32,
    pub description: String,
}
```

### 6.2 历史查询

```rust
impl BuildingHistory {
    /// 查询某时间点的组件风格（用于考古学查询）
    pub fn style_at(&self, component_id: ComponentInstanceId, time: GameTime)
        -> Option<&StyleSnapshot> { ... }

    /// 重建某时间点的建筑状态概要
    pub fn state_at(&self, time: GameTime) -> BuildingSnapshot { ... }

    /// 此建筑是否有著名历史事件
    pub fn has_notable_events(&self) -> bool { ... }
}
```

### 6.3 与 AetherImprint 的关系

- `BuildingHistory` = 建筑模块内部存储（自己的事）
- `AetherImprint`（[[历史系统]]）= 全局历史总线
- 重要事件（建成、毁灭、名人居住过）→ 由 [[历史系统]] 的定期 crawler 从 BuildingHistory 扫描提升为 AetherImprint
- 日常维修、小翻修 → 只在 BuildingHistory 内

## 七、Shelter（简易庇护所）

不是所有人为结构都走完整的 Building 生成管线。野外猎人临时棚子、难民营帐篷、市场摊——这些是 `Shelter`。

```rust
/// 简易庇护所——无组件图、无 WFC、无施工调度
pub struct Shelter {
    pub id: ShelterId,
    pub position: WorldCoord,
    pub shelter_type: ShelterType,
    pub owner: Option<EntityId>,
    pub built_at: GameTime,
    pub durability: f32,          // 0-1，快速衰减
    pub area: f32,                // m²
}

pub enum ShelterType {
    LeanTo,         // 斜棚
    Tent,           // 帐篷
    Stall,          // 市场摊位
    Campfire,       // 篝火+简陋围栏
    Burrow,         // 地穴
}
```

当 `Shelter` 长期存在且被扩建时 → NPC 行为触发 `ShelterUpgraded` → 走完整的 `BuildingGenerator` 生成正式 Building。`BuildingHistory` 记录 `ShelterUpgraded` 事件。

## 八、持久化格式

- Building 序列化为二进制（`rkyv` 用于热加载——零拷贝反序列化）
- BuildingHistory 活跃层完整条目，归档层聚合 struct
- ComponentRegistry 不持久化——编译期常数
- Blueprint TOML 是文本格式——不受 LMDB 影响
- 存档格式带 `version: u16`——加载时版本检查 + 需要时迁移
