## 一、体系概览

四种求解策略，共享同一个 `ConstraintChecker`：

| 求解器 | 输入 | 输出 | 使用场景 |
|--------|------|------|---------|
| `WfcBuildingSolver` | `BuildingSite` + `BuildContext` | `Building`（组件图） | 世界生成、玩家编辑器预览 |
| `BlueprintValidator` | `Blueprint`（TOML）+ `BuildContext` | `ValidatedBlueprint` 或 `Vec<ValidationError>` | 玩家 Blueprint 验证 |
| `StructuralSolver` | `Building`（组件图） | `f32`（完整性）+ `Vec<StructuralIssue>` | WFC 后验证、翻修检查、破坏后重算 |
| `LocalIncrementalSolver` | 修改目标 + 现有 `Building` + `BuildContext` | `Vec<AddRemoveOp>` | NPC 翻修/扩建 |

## 二、WfcBuildingSolver — 建筑级 WFC（2.5D 三阶段）

### 2.1 总览

```
BuildingSite { boundary, orientation, building_type, BuildContext }
        │
        ▼
[阶段1] BSP 房间分区
        ≡ 二叉空间分割 → 房间边界 + 类型提示
        │
        ▼
[阶段2] 逐层 2D WFC（0.25m grid, ~100 tile, ~2400 cell）
        ≡ 权重驱动的约束满足，逐层约束传播
        │
        ▼
[阶段3] 3D 组装
        ≡ cell 合并 → ComponentInstance + ConnectionFace 边图 + 屋顶 + 结构验证
        │
        ▼
Building { components, structural_score }
```

### 2.2 阶段1：BSP 房间分区

不是 WFC——是带约束的二叉空间分割：

```rust
fn partition_rooms(
    boundary: Aabb2D,
    building_type: BuildingTypeTag,
    ctx: &BuildContext,
    seed: u64,
) -> Vec<RoomHint> {
    let target_rooms = building_type.typical_room_count(ctx);
    // BSP: while 叶节点数 < target_rooms
    //   - 选面积最大的叶
    //   - aspect_ratio > 1.8 → 沿短轴分（避免瘦长房间）
    //   - 否则 → 随机 H/V（种子确定性）
    //   - 分割面受 min_room_width 约束（来自 ctx.culture.privacy）
    //   - 分割面 = 未来的内墙位置
    // 输出: 叶节点 = 房间边界

    // 房间类型提示分配
    for room in &mut rooms {
        room.type_hints = assign_type_hints(room, building_type, &boundary);
        // 临街面 → 公共空间权重高
        // 最大房间 → 主厅/锻造间权重高（取决于建筑类型）
        // 最小房间 → 储藏/私室权重高
    }

    rooms
}

struct RoomHint {
    boundary: Aabb2D,
    type_hints: Vec<(FunctionTag, f32)>,  // 软提示——WFC 可以偏离
    neighbor_doors: Vec<DoorHint>,
    is_atrium: bool,     // 天井/中庭——开放空间
    is_stairwell: bool,  // 楼梯井——所有层相同位置
}
```

**多层建筑关键修正**：BSP 只在地面层运行一次。上层继承地面层的房间分区 + 楼梯井位置，允许有限修改（合并相邻房间、加隔墙），不能移动承重墙。

### 2.3 阶段2：2D WFC

#### Tile 定义

WFC tile 是简约的——不包含具体尺寸，只包含族标签和材质。

#### 基础类型

```rust
/// WFC 邻接方向
#[derive(Clone, Copy, PartialEq, Eq, Hash)]
pub enum Direction {
    North, East, South, West,
    Up, Down,
}
impl Direction {
    pub const ALL: [Direction; 6] = [North, East, South, West, Up, Down];
    pub fn opposite(self) -> Direction { ... }
    pub fn to_face_dir(self) -> FaceDir { ... }
}

/// WFC cell 中的族标识——9 变体，用于 ComponentRegistry 查找
#[derive(Clone, Copy, PartialEq, Eq, Hash)]
pub enum FamilyTag {
    ExteriorOpen,
    InteriorOpen,
    ExteriorWall,
    InteriorWall,
    Floor,
    Door,
    Window,
    Stairs,
    Column,
}

/// WFC tile 变体——ComponentFamily 注册时声明的完整 tile 表示
pub struct WfcTileVariant {
    pub kind: WfcTileKind,      // 完整 tile 标识（含子类型参数）
    pub material: MaterialClass,
    pub weight: f32,             // 基础权重——会被 BuildContext 调整
}

/// WFC 的完整 tile 标识——含子类型
/// 10 变体，其中 Door/Window/Stairs 带参数
#[derive(Clone, Copy, PartialEq, Eq, Hash)]
pub enum WfcTileKind {
    ExteriorOpen,
    InteriorOpen,
    ExteriorWall,
    InteriorWall,
    Floor,
    Door(DoorType),          // Single / Double / Arch / Portcullis / Trapdoor
    Window(WindowType),      // NarrowSlit / Standard / Wide / Bay / Round / ArrowSlit
    Stairs(StairDir),        // Straight / LShaped / Spiral / Ladder
    Column,
}
impl WfcTileKind {
    /// 剥离子类型参数 → FamilyTag
    pub fn family_tag(&self) -> FamilyTag {
        match self {
            Self::ExteriorOpen => FamilyTag::ExteriorOpen,
            Self::InteriorOpen => FamilyTag::InteriorOpen,
            Self::ExteriorWall => FamilyTag::ExteriorWall,
            Self::InteriorWall => FamilyTag::InteriorWall,
            Self::Floor => FamilyTag::Floor,
            Self::Door(_) => FamilyTag::Door,
            Self::Window(_) => FamilyTag::Window,
            Self::Stairs(_) => FamilyTag::Stairs,
            Self::Column => FamilyTag::Column,
        }
    }
}

/// WFC cell 内存储的 tile 实例
struct WfcTile {
    family_tag: FamilyTag,
    material: MaterialClass,
    face: Option<FaceDir>,    // 仅 Wall/Window/Door —— 外立面的朝向
    subtype: u8,              // 族内子类型（编码 DoorType/WindowType/StairDir 等）
}
// 实际 tile 数 ≈ 146，BitSet<256> = 32 字节——留足余量
// 10m×15m @ 0.25m = 40×60 = 2400 cell = 76.8KB 状态
```

#### 邻接约束

从 `ComponentRegistry` 的 `ComponentFamilyBase::face_on_side()` 自动生成。不是手写。`Registry` 通过依赖注入传入——不是全局变量。

```rust
fn compatible(
    tile_a: &WfcTile, side_a: Direction,
    tile_b: &WfcTile,
    registry: &ComponentRegistry,  // ← 依赖注入，非全局
) -> bool {
    let family_a = registry.resolve_tag(tile_a.family_tag);
    let family_b = registry.resolve_tag(tile_b.family_tag);

    let face_a = family_a.face_on_side(tile_a, side_a);
    let face_b = family_b.face_on_side(tile_b, opposite(side_a));

    match (face_a, face_b) {
        (Some(fa), Some(fb)) => registry.compatible(&fa, &fb),
        (None, None) => true,  // 双方都无连接面 → 不连接但可以邻接
        _ => false,
    }
}
```

#### WFC 主循环

```rust
struct WfcGrid<'a> {
    cells: Vec<WfcCell>,
    width: u32, height: u32,
    perimeter: Perimeter,
    room_hints: Vec<RoomHint>,
    registry: &'a ComponentRegistry,  // ← 依赖注入
    rng: Pcg64,
}

struct WfcCell {
    possible: BitSet<256>,    // 146 tile → 256 位 = 32 字节，留足余量
    collapsed: Option<WfcTile>,
    entropy: f32,
}

impl WfcGrid {
    fn step(&mut self) -> WfcResult {
        let idx = self.lowest_entropy_cell()?;
        let weights = self.tile_weights(idx);
        let chosen = sample_weighted(&weights, &mut self.rng);
        self.cells[idx].collapse(chosen);
        self.propagate(idx)
    }

    fn tile_weights(&self, cell_idx: usize) -> Vec<(WfcTile, f32)> {
        // ★ 不包含材料可用性——后处理检查
        // 权重来源：
        //   1. 基础频率（门比墙少——自然的建筑统计比例）
        //   2. 文化偏好（石墙 ×1.3 vs 木墙 ×0.7）
        //   3. 气候约束（沙漠→木材耐久低但不是不能用，权重降低）
        //   4. 房间提示（在厨房提示区→Floor和灶台相关 tile 权重高）
        //   5. 边界约束（在建筑边界→Empty 权重为 0）
        //   6. 朝向偏好（南面→窗户权重高）
        //   × 7. 不包含材料可用性——后处理再累计材料需求检查
    }

    fn propagate(&mut self, from_idx: usize) -> WfcResult {
        let mut dirty = VecDeque::new();
        dirty.push_back(from_idx);

        while let Some(current) = dirty.pop_front() {
            for dir in Direction::ALL {
                let neighbor = self.neighbor_index(current, dir);
                if self.cells[neighbor].collapsed.is_some() { continue; }

                let before = self.cells[neighbor].possible.count();
                self.cells[neighbor].possible.retain(|tile| {
                    compatible(&self.cells[current].collapsed.unwrap(), dir, tile)
                });
                let after = self.cells[neighbor].possible.count();

                if after == 0 { return WfcResult::Contradiction(current, neighbor); }
                if after < before {
                    self.cells[neighbor].update_entropy();
                    dirty.push_back(neighbor);
                }
            }
        }
        WfcResult::Progress
    }
}
```

#### 矛盾处理

```
1. 回溯 1 步（最多 5 次）：恢复上次坍缩前的状态，从候选集中移除上次的选择
2. 重启（最多 3 次）：用不同种子重新初始化
3. 局部松弛：接受当前权重最高的非法 tile，标记为后处理修复
```

99% 的合理输入不会出现矛盾。WFC 是成熟算法，在建筑约束下高度稳定。

#### 种子派生

`BuildingGenerator::generate()` 接受 `seed: u64`。WFC 内部 3 个子种子派生：

```
seed → BSP partition seed       = seed
seed → WFC tile collapse seed   = seed ^ 0xA1B2_C3D4_E5F6_0718
seed → post-process jitter seed = seed ^ 0xF1E2_D3C4_B5A6_9780
```

确定性：同一 `(BuildingSite, BuildContext, seed)` → 必定产生相同 `Building`。

#### 屋顶约束前置

**屋顶类型在 WFC 求解前已知**（从 `BuildingTypeTag` + `BuildContext.culture.roof_style` 派生）。但屋顶具体尺寸（梁跨、椽间距）后处理时才确定。WFC 求解时：

- 最顶层 `Wall` cell 的 `PosY` face → 必须兼容 `RoofBeam` 的 `NegY` face（roof_type 已知 → RoofBeamFamily 的 ConnectionFace 可预查）
- 屋顶坡度影响高层墙的承重需求（陡坡 = 更大侧推力）
- 这些约束在 WFC 权重和邻接表中已生效——屋顶**放置**在后处理，但屋顶**约束**在前
- 屋顶后处理在 `WfcBuildingSolver::generate()` 内部执行——此时 Building 尚未存入全局 Store。可直接遍历 `Building.components` 过滤 top-floor Wall，无需通过 `BuildingQuery`

```rust
// WFC 求解前
let roof_type = ctx.culture.roof_style.unwrap_or(RoofStyle::Gable);
let roof_pitch = ctx.climate.roof_pitch();
// 预查 RoofBeamFamily 的 ConnectionFace
let roof_beam_face = registry.resolve_tag(FamilyTag::Beam)
    .face_on_side(&RoofBeam::wfc_tile(roof_type), Direction::Down, ctx);
// → WFC 顶层 wall cell 的 PosY face 必须 compatible with roof_beam_face
```

#### 多层约束传播

每层独立 WFC，但上层受下层 `VerticalConstraint` 约束：

```rust
struct VerticalConstraint {
    required_supports: HashSet<(u32, u32)>,     // 必须放承重组件的 cell
    no_heavy_wall: HashSet<(u32, u32)>,         // 下方是窗/门，不能放承重墙
    required_openings: HashSet<(u32, u32)>,     // 挑空/中庭贯通
    stair_shaft: HashSet<(u32, u32)>,           // 楼梯位置——所有层相同
}

// 规则:
// F0 的 ExteriorWall 的 cell → F1 可以是 ExtWall/IntWall/Column/Window（退台时是 Empty）
// F0 的 Window/Door 的 cell → F1 不能是承重组件
// F0 的 Column 的 cell → F1 必须是 Column（柱贯穿全高）
// F0 的 Floor 的 cell → F1 可以是任意类型（但放墙需要额外梁或加厚地板）
```

### 2.4 阶段3：3D 组装

#### cell 合并（泛洪填充 + Union-Find）

4 个相邻的 `ExteriorWall + Stone + NorthFace` → 一面 1m×2.5m×0.5m 的石墙组件。这是标准的光栅化反向操作。

#### 尺寸派生

WFC 输出材质+类型标签，不是具体尺寸。尺寸在后处理中从 `BuildContext` 计算：

```rust
let thickness = ctx.climate.wall_thickness(floor, total_floors);
// 底层承担全重 → 最厚, 顶层只承担自己 → 最薄

```

#### 结构完整性（见 §三）

**调用链**：WFC 生成 Building 后 → 立即调用 `StructuralSolver::compute()` → 将 `structural_score` 写入 `Building` → 若 `score < 0.3`（结构危险），此解被拒绝并触发 WFC 回溯（最多 5 次）或重启（最多 3 次）。

## 三、StructuralSolver — 结构完整性验证

从组件图计算载荷传播。**不是有限元分析——是简单的迭代求解。**

```rust
fn structural_integrity(graph: &StableGraph<ComponentInstance, ConnectionEdge>) -> f32 {
    // 1. 从屋顶向下传播重力载荷
    //    顶层组件: load = self_weight
    //    下层: load = self_weight + Σ(上方组件下传的重量 / num_supports)
    // 2. 每个 ConnectionFace 检查: load_transfer ≤ face.load_bearing
    // 3. 返回值 = min(所有面的 capacity / actual_load)
    //    1.0 = 完美, 0.0 = 塌了

    // 迭代求解（一般 3-5 次收敛）
    let mut loads = HashMap::new();
    for _ in 0..MAX_ITER {
        let mut changed = false;
        for edge in graph.edges() {
            let load_from_above = loads.get(&edge.source).unwrap_or(0.0);
            let shared = load_from_above / edge.num_supporting_children();
            if update_load(&mut loads, edge.target, shared) { changed = true; }
        }
        if !changed { break; }
    }

    // 检查超载
    let mut min_ratio = 1.0;
    for edge in graph.edges() {
        let actual = loads.get(&edge.target).unwrap_or(0.0);
        let capacity = edge.connection.load_bearing();
        if capacity > 0.0 {
            min_ratio = min_ratio.min(capacity / actual.max(1.0));
        }
    }
    min_ratio.clamp(0.0, 1.0)
}
```

## 四、BlueprintValidator — Blueprint 验证器

玩家 Blueprint（TOML）在施工前必须通过验证：

impl<'a> BlueprintValidator<'a> {
    fn validate(&self, bp: &Blueprint, ctx: &BuildContext) -> Result<ValidatedBlueprint, Vec<ValidationError>> {
        let mut errors = Vec::new();

        // 1. 组件族名有效性
        // 2. 参数 schema 验证
        // 3. 尺寸合理性（门不能比墙大，……）
        // 4. 连接面兼容性（从几何自动推导——不检查手写连接）
        // 5. 结构完整性检查（调用 StructuralSolver）
        // 6. 网格对齐检查

## 五、LocalIncrementalSolver — 局部增量求解

NPC 翻修/扩建时使用——不跑完整 WFC，只做局部修改：

```rust
struct LocalIncrementalSolver;

impl LocalIncrementalSolver {
    fn solve(
        goal: &ModificationGoal,
        existing: &Building,
        ctx: &BuildContext,
    ) -> Result<Vec<AddRemoveOp>, SolveError> {
        match goal {
            ModificationGoal::AddRoom { adjacent_to, size_hint, function } => {
                // 1. 在现有建筑边界找可扩建的方向
                // 2. 新房间的 BSP 融合进现有 Region 图
                // 3. 生成新组件 + 修改受影响的现有组件（打通门洞）
                // 4. 验证不会导致结构完整性下降到危险水平
            }
            ModificationGoal::RemoveWall { component_id } => {
                // 1. 检查此墙是否承重——承重墙禁止移除除非加柱
                // 2. 更新邻接的 ConnectionFace
                // 3. 合并新连通的 Region
                // 4. 重算结构完整性
            }
            ModificationGoal::ReplaceDoor { component_id, new_params } => { ... }
            ModificationGoal::AddWindow { on_wall_id, position, window_type } => { ... }
            ModificationGoal::AddFloor { ... } => { ... }
        }
    }
}
```
