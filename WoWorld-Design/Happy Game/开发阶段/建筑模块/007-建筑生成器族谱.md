## 一、BuildingGenerator trait

多种生成策略的抽象——不是「一个 WFC 生成所有建筑」。

```rust
/// 建筑生成器——多种策略，同一个 trait
pub trait BuildingGenerator: Send + Sync {
    /// 此生成器能处理的建筑类型
    fn accepts(&self, building_type: BuildingTypeTag) -> bool;

    /// 核心：从场地 + 上下文 → 完整的建筑组件图
    fn generate(
        &self,
        site: &BuildingSite,
        ctx: &BuildContext,
        seed: u64,
    ) -> Result<Building, GenerationError>;

    /// 能力声明
    fn supports_radial(&self) -> bool { false }
    fn supports_multi_building(&self) -> bool { false }
    fn max_floors(&self, ctx: &BuildContext) -> u8 { 2 }
}

/// 生成器注册表
pub struct GeneratorRegistry {
    generators: Vec<Box<dyn BuildingGenerator>>,
    default_generator: Box<dyn BuildingGenerator>,
}

impl GeneratorRegistry {
    pub fn select(&self, building_type: BuildingTypeTag) -> &dyn BuildingGenerator {
        self.generators.iter()
            .find(|g| g.accepts(building_type))
            .unwrap_or(&*self.default_generator)
    }
}
```

**v1.0 包含**：WfcRectangular、WfcRadial、Cathedral、Shelter、Complex（接口+简单实现）

**v1.1+ 包含**：WallGenerator、BridgeGenerator、UndergroundGenerator、ComplexGenerator 完整体

## 二、WfcRectangularGenerator——80% 建筑

**适用**：住宅、商店、工坊、酒馆、仓库、市政厅、兵营、磨坊、马厩……

**覆盖**：[[建筑模块/004-约束求解系统|004 约束求解系统]] 完整描述。2.5D 三阶段（BSP → 2D WFC → 3D 组装），约束传播多层支持。

**能力**：`supports_radial=false`, `max_floors=8`（石结构）或 4（木结构）

## 三、WfcRadialGenerator——圆形/塔楼/竞技场

**适用**：塔楼、灯塔、土楼、圆形神殿、竞技场/角斗场、螺旋塔

### 3.1 RadialWfcGrid

```rust
struct RadialWfcGrid {
    center: Vec2,
    inner_radius: f32,        // 最内环半径（0 = 中心单 cell）
    outer_radius: f32,
    rings: Vec<RadialRing>,
    height: u8,               // 层数（逐层分别跑）
}

struct RadialRing {
    radius: f32,
    cell_count: u32,
    cell_arc: f32,            // 每个 cell 的弧度
    cells: Vec<WfcCell>,
}

// 邻接方向:
//   Inward:  ring[r][θ] → ring[r-1][θ']  (内环 cell 少, 1对多)
//   Outward: ring[r][θ] → ring[r+1][θ']
//   CW:      ring[r][θ] → ring[r][θ+1]     (顺时针邻)
//   CCW:     ring[r][θ] → ring[r][θ-1]     (逆时针邻)
```

**径向 tile 族标签**：仍使用相同的 `FamilyTag`（Wall、Floor、Door 等），但 `FaceDir` 变为 `Inward / Outward / CW / CCW`。

生成后同一 `WallFamily` 生成弧墙——`dimensions` 变为 `{ inner_radius, outer_radius, arc_angle, height, thickness }` 而非 `{ width, height, thickness }`。

### 3.2 径向+矩形混合

万神殿模式——圆形主体 + 矩形门廊：

```
1. 圆形主体 → 径向 WFC
2. 标记圆形主体朝南的弧面为「连接面」
3. 矩形门廊 → 矩形 WFC，边界一端附着在连接面上
4. 交接处 → 过渡柱（PortalColumn）自动插入
```

### 3.3 螺旋塔

每层比下层旋转 N°（N 从 BuildContext 的 Style 偏好派生）。实现：径向 WFC 逐层运行，每层的 `rotation_offset` = `floor × 5°`。

## 四、CathedralGenerator——十字平面宗教建筑

**适用**：主教座堂、大教堂、修道院教堂、朝圣教堂

**特殊之处**：十字平面是固定程序（SacredArchitectureParams，[[信仰系统]] CHG-025），不是 BSP 随机分区。但十字平面内的具体实现（拱高、柱粗、窗大小）仍是涌现的——由 BuildContext 文化+气候参数派生。

```rust
struct SacredLayout {
    nave_width: f32,
    nave_length: f32,
    aisle_width: f32,
    transept_width: f32,
    apse_radius: f32,
    orientation: f32,          // 朝向圣地
    crossing_type: CrossingType,  // 塔楼/穹顶/尖塔
    tower_count: u8,           // 0/1/2/4
}

enum CrossingType { Tower, Dome, Spire, Open }
```

**生成流程**：

```
1. 从 FaithBuildProfile 派生 SacredLayout
2. 十字 = 5 个矩形区域（中殿+双侧廊+翼廊+后殿）
   每个区域 → WfcRectangularGenerator 子调用
3. 交叉处 → 特殊处理（塔楼/穹顶）
4. 钟楼 → 如果 tower_count > 0，作为附加的 WfcRectangular（高塔）
5. 朝向全部对齐 orientation
```

## 五、ComplexGenerator——建筑群

**适用**：皇宫、城堡、修道院、大学、神庙群

```rust
struct BuildingComplex {
    id: ComplexId,
    layout: ComplexLayout,
    buildings: Vec<ComplexBuilding>,
    connections: Vec<ComplexConnection>,
    shared_spaces: Vec<SharedSpace>,
}

struct ComplexBuilding {
    site: BuildingSite,
    role: BuildingRole,  // MainHall / SideWing / GateHouse / Tower / ...
}

struct ComplexConnection {
    from: (usize, PortalSpec),  // (building_index, portal)
    to: (usize, PortalSpec),
    connection_type: ConnectionType,
    elevation: f32,
}

enum ConnectionType {
    CoveredCorridor,  // 封闭连廊
    SkyBridge,        // 空中廊桥
    SharedWall,       // 共享墙
    Tunnel,           // 地下通道
}

enum ComplexLayout {
    Axial { axis: Vec2, symmetry: SymmetryType },  // 中轴对称（皇宫、神庙）
    Radial { center: Vec2 },                        // 放射状（城堡）
    Organic { seed_points: Vec<Vec2> },             // 有机生长
    Grid { spacing: f32 },                          // 网格（军营）
}
```

### 5.1 皇宫生成流程

```
1. ComplexLayout::Axial { axis: NorthSouth, symmetry: Bilateral }
2. 沿轴线分配建筑角色（太和门→太和殿→中和殿→保和殿→乾清门→乾清宫→坤宁宫）
3. 左右对称镜像 BuildingSite
4. 共享空间：殿前广场 = SharedSpace { type: CeremonialPlaza }
5. 逐个建筑调 WfcRectangularGenerator
6. 生成连接结构（宫墙、连廊）
7. 对每栋建筑——传连接点硬约束 → WFC 内为连接点预留 Door/Portal cell
```

### 5.2 双子楼生成

```
1. ComplexLayout: 塔A 在 (-15, 0), 塔B 在 (+15, 0)（镜像）
2. 连接: 塔A 第5层东面 → 塔B 第5层西面 → SkyBridge
3. WFC 在该 cell 输出 Portal
4. SkyBridge 自身：30m×3m 窄格 WFC
```

### 5.3 建筑群所有权——生成后解散

`BuildingComplex` 只是生成约束——不是运行时实体。生成完成后，每栋 `Building` 独立所有权：

| 建筑 | 所有权 |
|------|--------|
| 太和殿 | `OwnedBy(皇帝)` |
| 内阁房 | `OwnedBy(内阁大臣)` |
| 宫墙 | `Public` |
| 连廊/空中廊桥 | `Public`（或归属相邻建筑中较小的一方） |

`BuildingComplex` 生成后即销毁——它不存储在存档中。连廊和宫墙是独立的 `Building`（`building_type: CoveredCorridor` / `WallSegment`），走独立的所有权。

## 八、多样性保证——同参数也能产出不同建筑

三重多样性来源：
1. **BSP 随机**（种子驱动）→ 房间布局不同
2. **WFC 权重噪声** × (1 + noise × 0.3) → 微调但不破坏约束
3. **后处理抖动**——门窗位置 ±0.25m，屋顶坡度 ±5°，材质颜色微调

一个城镇的 30 栋同类住宅各自不同。不是 30 次完全随机的结果——是同一个约束空间下的 30 个有效解。
