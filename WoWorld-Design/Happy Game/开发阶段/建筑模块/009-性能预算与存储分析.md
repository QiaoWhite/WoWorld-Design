| **合计** | | ~25,000 聚落 | | **~1,780,000 建筑** |

### 2.2 三层 LOD 生成策略

| LOD | 覆盖范围 | 建筑数量 | 生成方式 | 单栋耗时 | 总耗时 |
|-----|---------|---------|---------|---------|--------|
| **LOD2**（统计） | 全图 | ~178 万等效 | 聚落级聚合 | ~0.01ms | ~17s |
| **LOD1**（简化） | 全图 | ~50,000 | BSP 分区 + 外壳 | ~0.2ms | ~10s |
| **LOD0**（完整） | 出生区域 3km | ~500 | 完整 WFC | ~4ms | ~2s |
| **全图总计** | | | | | **~30s** |

30 秒世界生成 + 进度条——可接受。

### 2.3 LOD 升级

```
玩家移动 → 新区域进入 LOD1 范围
  → LOD2→LOD1（聚落级生成外壳）
  → LOD1→LOD0（玩家靠近的建筑，分帧升级）
  → 每帧最多升级 2 栋 → 20 帧完成一个村庄 (~30 栋)

并发模型：
  后台线程：读取 LOD1 SimplifiedBuilding → 跑 WFC → 生成完整 Building
  主线程：  帧末通过 channel 接收 Arc<Building> → 安全写入 BuildingStore
  后台线程不直接持有 RwLock write——避免死锁和争用。
```

## 三、内存预算

### 3.1 LOD0 区域（玩家 3km，500 栋）

| 数据 | 单栋 | 合计 |
|------|------|------|
| Building 本体 | ~128B | 64KB |
| ComponentInstance × 200 | ~200 × 96B | 9.6MB |
| BuildingSpatialIndex | ~2KB | 1MB |
| BuildingHistory 活跃层 | ~5KB | 2.5MB |
| BuildingHistory 归档层 | ~10KB | 5MB |
| **LOD0 合计** | **~36KB** | **~18MB** ✅ |

### 3.2 LOD1 区域（3-30km，50,000 栋）

| 数据 | 单栋 | 合计 |
|------|------|------|
| SimplifiedBuilding | ~48B | 2.4MB |

`SimplifiedBuilding` = `BuildingId`(8B) + `Aabb`(24B) + `BuildingTypeTag`(2B) + `room_count`(1B) + `floor_count`(1B) + `structural_score`(4B) + `orientation`(4B) + `style_hash`(4B) = 48B ✅

### 3.3 全图 LOD2 存储

25,000 聚落 × 200B = **5MB** ✅

### 3.4 建筑总内存

LOD0 (18MB) + LOD1 (2.4MB) + LOD2 (5MB) = **~25MB** ✅

技术栈总内存预算 32GB——占比 < 0.1%。

## 六、WFC 求解器内部性能

| 操作 | 数据量 | 计算量 | 耗时 |
|------|--------|--------|------|
| BSP 分区 | n=50 叶 | O(n log n) | < 0.1ms |
| WFC 2D（2400 cell）坍缩 | 2400 × 100 tile | ~240,000 比较 | ~0.2ms |
| WFC 传播（~500 步） | 500 × 4 方向 | ~2000 次 BitSet AND | < 0.1ms |
| cell 合并 | ~2400 cell | 单次扫描 + Union-Find | ~0.3ms |
| 连接边构建 | ~200 边 | 空间哈希 + 面匹配 | ~0.3ms |
| 结构迭代 | 200 节, 3-5 迭代 | O(E) = ~200 | < 0.1ms |
| 屋顶生成 | ~10 组件 | 简单几何 | ~0.1ms |
| **单栋 WFC 总计** | | | **< 2.5ms** |

实际耗时在 2-5ms 范围——取决于建筑复杂度和矛盾处理次数。技术的 5ms 预算是充裕的上限。

### 6.1 高层建筑额外成本

| 楼层数 | WFC cell | 单层耗时 | 约束检查开销 | 总耗时 |
|--------|----------|---------|------------|--------|
| 1 层 | 2,400 | 2.5ms | 0 | ~2.5ms |
| 3 层 | 7,200 | 2.5ms×3 | +15% | ~5ms |
| 6 层 | 14,400 | 2.5ms×6 | +25% | ~8ms |
| 10 层 | 24,000 | 2.5ms×10 | +30% | ~13ms |

六层在预算边界。十层略超——但十层建筑极其罕见（只有大型城市核心区）。可接受。

## 七、并发安全

- **读路径**：`BuildingQuery` 所有方法 → `RwLock::read()`——多线程并发查询安全
- **写路径**：`ConstructionScheduler::tick()`、翻修、破坏 → `RwLock::write()`——写事件稀疏（<10/s）
- **LOD 升级**：后台线程仅跑 WFC（纯 CPU）→ `Arc<Building>` 通过 channel 回主线程 → 主线程帧末写入 `BuildingStore`。**后台线程不直接持有写锁**——避免死锁。
