## 一、严重缺失系统 🔴

以下系统在 `07-设计整合分析.md` 已标记为缺失，但仍未得到补充。**在进入对应开发阶段前必须完成设计。**

### 1.1 物品数据库

| 维度 | 缺失程度 |
|------|---------|
| 物品ID体系 | 🔴 完全缺失 |
| 物品属性定义（重量/耐久/堆叠/稀有度） | 🔴 完全缺失 |
| 物品分类体系（武器/防具/消耗品/材料/杂物/任务品） | 🔴 完全缺失 |
| 制造/配方表 | 🔴 完全缺失 |
| 物品获取途径（掉落/采集/制造/购买/盗窃） | 🔴 完全缺失 |

**建议**：在 `开发阶段/` 创建 `物品系统设计.md`。优先定义 50-100 种核心物品（覆盖铁器时代到魔法时代的跨度），建立 ID 编码规范。

### 1.2 天气与季节系统

`总设计草稿` 和 `01-世界生成总流程` 中有"气候带"的概念，但完全是静态的——没有动态天气、没有季节变化。

| 缺失内容 |
|---------|
| 天气类型定义（晴/阴/雨/雪/暴风/沙尘暴/魔法风暴……） |
| 天气对 NPC 行为的影响（雨天回家、风暴躲屋内、雪天穿厚衣） |
| 天气对战斗的影响（雨天火魔法减弱、雾天远程命中率下降） |
| 季节循环（春耕秋收、冬季河流结冰、夏季干旱风险） |
| 季节对农业/采集/狩猎的影响 |
| 极端天气作为"微型事件"（洪水、旱灾、暴雪封路） |

### 1.3 对话系统

完全没有设计。当前唯一提到对话的地方是 `冒险小队.md`（LLM 增强对话），以及总设计草稿中提到的"预制选项+关键词匹配"。

| 需要设计的内容 |
|---------------|
| 对话 UI 布局（对话框位置/样式/选项排列） |
| 对话树数据结构（节点→选项→分支） |
| 关键词/主题匹配系统（NPC 能回应哪些话题） |
| 情绪对对话选项的影响（愤怒时没有礼貌选项） |
| 关系对对话的影响（陌生人→朋友→恋人的对话差异） |
| 非 LLM 对话的内容量（至少 200+ 预制对话模板） |
| 语言/方言系统接口（为文化系统的语言差异预留） |

### 1.4 技能系统细节

`总设计草稿` §5.2 提出了"用进废退"的设计原则和技能分类，但缺乏：

| 缺失内容 |

| 技能之间的协同/冲突关系 |

| 技能上限（一个人最多精通几个技能） |
| 教学/传授机制（NPC 能否教玩家技能） |
| 技能与 NPC 职业/社会身份的关系 |

### 1.5 动物与怪物设计

`动物.md` 仅占位。没有怪物设计。

| 缺失内容 |
|---------|

## 二、需要深化的系统 🟡

以下系统已有初步设计，但深度不足。

### 2.1 疾病与医疗

当前仅有 `health` 一个浮点值。缺少：

- 疾病类型（感冒/感染/瘟疫/魔法诅咒/慢性病）
- 受伤类型（骨折/出血/烧伤/冻伤/中毒）——不同治疗方式
- 医疗技能深度（草药→放血→外科→魔法治疗）
- 医生 NPC 的社会角色
- 瘟疫作为大规模事件的传播模型
- 卫生条件对健康的影响（污水、尸体处理）

### 2.2 犯罪与执法

`道德.md` 和 `总设计草稿` §7.4 有概念性描述，但缺乏：

- 犯罪类型定义（偷窃/抢劫/谋杀/欺诈/纵火/叛国……）
- 犯罪侦查机制（目击者→举报→调查→追捕）
- 证据系统（凶器、目击证词、不在场证明）
- 惩罚体系（罚款/劳役/囚禁/流放/处决——具体数值）
- 越狱/劫狱的可能
- 玩家犯罪后的追责机制（不同于 NPC 犯罪？）
- 跨国犯罪与引渡

### 2.3 儿童、家庭与世代

NPC 系统没有设计：
- 儿童 NPC（不同的属性/行为/保护机制）
- 怀孕与生育
- 成长过程（儿童→青少年→成人→中年→老年）
- 衰老对属性的影响
- 继承（父母死亡后财产/头衔归谁）
- 家族谱系追踪

这直接影响"活的世界"的长期可信度——没有世代更替，100,000 个 NPC 最终会全部老死而世界没有新人。

### 2.4 交通与载具

`冒险小队` 提到了飞行坐骑（用于空战），但没有：
- 陆地坐骑（马/驴/骆驼/奇幻坐骑）
- 马车/牛车（货运）
- 船舶（河船/海船/渔船/商船）
- 传送魔法（成本/限制/传送网络）
- 旅行速度与疲劳
- 道路质量对旅行速度的影响

### 2.5 通信与信息传播

当前记忆传播仅考虑面对面闲聊（`B5b 社交传播`）。但缺少：
- 书信/信使系统（跨远距离信息传播）
- 公告/公示（官方信息发布方式）
- 魔法通信（成本高昂的远距离即时通信）
- 信息传播速度的世界地图
- 谣言与失真（信息每传播一次有概率被扭曲）

### 2.6 植物与农业系统

植物在 `06-体素设计决策` §2.1 中仅作渲染对象，缺乏：
- 农作物生长周期（播种→生长→成熟→收获→休耕）
- 作物种类（主食/蔬菜/水果/纤维作物/药草/染料植物）
- 产量影响因素（土壤肥力/水分/温度/病虫害）
- 轮作/休耕制度
- 畜牧（放牧/圈养/饲料/繁殖）
- 采集（野生植物的可再生机制）
- 饥荒逻辑（粮食歉收→NPC 挨饿→社会动荡）

## 三、文档间的一致性问题 ⚠️

在审查过程中发现了以下跨文档的矛盾或模糊之处：

### 3.1 世界规模的双重描述

- `总设计草稿.md` §3.2 有两个重复的"3.2"章节——一个说体素世界，一个说世界规模。内容有重叠但不完全一致。
- `01-世界生成总流程.md` 说"围绕玩家加载范围进行跨区块自然景观分布规划"（ver0.5 设计），但又说 ver0.6 移除了此逻辑。建议明确当前采用哪个版本。

**建议**：清理 `总设计草稿.md` 的重复 §3.1 / §3.2，统一为一个"世界系统"章节。

### 3.2 NPC 三层 LOD 的触发距离

- `总设计草稿.md` §4.3：L1 ≤50m, L2 50-150m, L3 >150m
- `技术架构.md`：L1_DISTANCE=50m, L2_DISTANCE=150m

数值一致，但 **L1 上限当前50人、远期目标~1,000人**——一个城市可能有 2000+ NPC，如果玩家站在城市中心广场，50m 内可能远超当前上限。**L1 溢出策略**当前未定义，且随L1目标扩展至1,000人变得更加紧迫。

### 3.3 资源类型编号跳跃

`资源类型.md` 章节编号从"第一类"跳到"第三类"再跳到"第五类"。缺少第二类和第四类（或编号错误）。建议修正。

### 3.4 版本差异未清理

`01-世界生成总流程.md` §五专门讨论了 ver0.5 vs ver0.6 的差异并"建议以 ver0.5 为蓝本"。这个决策应该明确写在 `总设计草稿.md` 中，而非仅在一份子文档中提及。

## 四、奇思妙想 💡

以下创意值得在设计文档中预留接口或在未来阶段纳入考虑。

### 4.1 世界机人格化

`世界机.md` 的概念非常迷人——不同的"绝对意志"影响世界走向。建议将其发展为类似 RimWorld 的 **Storyteller 系统**，在创建存档时可选：

| 世界机类型 | 风格 | 游戏体验 |
|-----------|------|---------|
| **老好神兰迪** | 经典随机，难度递增 | 公平但有挑战 |
| **战争狂人** | 冲突概率大幅提高 | 战斗密集、高风险 |
| **丰收女神** | 灾害减少、资源丰富 | 建设种田向 |
| **诡诈之神** | 随机事件极不稳定 | 混乱、不可预测 |
| **叙述者** | 事件按叙事逻辑排列 | 故事感强 |

### 4.2 考古与遗迹

世界生成阶段九（历史模拟）快进了 N 年，产生了聚落兴衰。**衰亡的聚落不应消失——它们应变成遗迹。**

- 玩家可以在世界各地发现不同年代的文明遗迹
- 遗迹中有上个时代 NPC 的遗物（可阅读的日记、遗留的物品）
- 考古作为探索玩法：发掘→鉴定→了解世界历史→发现宝藏
- 上一个存档的世界可以作为"先史文明"出现在新存档中

### 4.3 NPC 梦境

利用 NPC 已有的记忆系统和情绪引擎：

- NPC 在睡眠时从记忆中抽取元素生成"梦境"
- 梦境内容反映 NPC 的潜意识（隐藏的恐惧/愿望/内疚）
- 玩家通过魔法手段可能"进入"或"窥视"NPC 的梦境
- 这为任务线索、NPC 深层性格揭示提供了独特的叙事入口

### 4.4 味觉/美食系统

- 不同文化有不同的食材组合和烹饪方式
- NPC 有口味偏好（甜的/咸的/辣的/酸的）
- 高品质的美食可以提升情绪、促进社交
- 传说中的食材（龙肉、凤凰蛋、世界树的果实）——极稀有，极强大
- 美食家/厨师作为 NPC 职业

### 4.5 语言与文字

- 不同文化区域可能使用不同的语言/文字系统
- 语言不通时对话选项受限（需要翻译或手势）
- 语言技能——学习外语需要时间
- 书籍包含实际可阅读的文字（非纯装饰）
- 古代文献需要"古语学"技能才能解读
- 诗人/作家 NPC 创作新的文学作品

### 4.6 地图绘制与知识迷雾

- 玩家的地图初始是空白的
- 走过的地方才被记录在地图上
- NPC 可以出售他们绘制的地图（有误差、有过时信息）
- 不同文化的地图风格不同（实用 vs 装饰性 vs 宗教宇宙观）
- 错误的地图可能引领玩家走向意想不到的地方

### 4.7 火与光的物理

- 火会蔓延——木屋能被点燃、森林可能发生火灾
- 光驱散黑暗——洞穴中没有光=完全漆黑（不仅是视觉效果，实际影响战斗和探索）
- 不同类型的照明工具（火把/油灯/魔法灯/夜视药水）有不同的光照范围
- NPC 对黑暗的反应——恐惧、谨慎、回避
- 夜行性生物在夜间更活跃

### 4.8 书信与文字遗产

- NPC 之间互相写信（重要的社交行为）
- 信的内容反映 NPC 的关系和当前状态
- 已故 NPC 的书信成为"文献遗产"
- 玩家可以截获/偷读他人的信件（获得情报，但有道德代价）
- 图书馆——聚落的知识存储，包含该地区的历史和知识

### 4.9 收藏与博物馆

- NPC 有个人收藏爱好（稀有矿石/蝴蝶标本/古代钱币/名画）
- 玩家也可以发展收藏
- 大型聚落可能有博物馆展示该地区的珍贵藏品
- 完整的收藏 = 社会地位的象征
- 盗贼可以偷窃私人收藏品（高风险高回报）

### 4.10 竞技与赛事

- 定期举办的各种比赛：
  - 武术大会（战斗竞技）
  - 烹饪大赛
  - 选美比赛（受审美系统影响）
  - 赛马/坐骑比赛
- NPC 和玩家都可以参赛
- 赛事期间大量人流涌入举办城市（经济繁荣+社交密集）
- 赌博系统

### 4.11 时间感知与日历

- 不同文化可能有不同的历法（太阳历/太阴历/魔法纪元）
- 同一个日期在不同文化中有不同的含义
- NPC 根据日历规划活动（春耕、秋收、冬藏、节日）
- 占星术——天体运行影响某些魔法效果
- 玩家的"生日"在游戏世界中可以被 NPC 记住和庆祝

### 4.12 水体互动

- 游泳（体力消耗、溺水风险、装备重量影响）
- 潜水（水下探索、水下洞穴、沉船）
- 船舶（建造→下水→航行→贸易/战斗/探索）
- 水下呼吸魔法/药水
- 水生生物（鱼、水生怪物、水下文明）
- 海啸/洪水作为自然灾害

### 4.13 名声的多维度

当前设计了"著名度"系统，但可以更丰富：
- 玩家在不同区域有不同的名声（在 A 城是英雄，在 B 城是通缉犯）
- 名声的维度：武勇/智慧/慷慨/残忍/可靠/欺诈
- NPC 对玩家的第一印象受名声影响
- 改变装束/使用化名可以暂时隐藏身份
- 吟游诗人传唱玩家的故事（进一步传播名声）

### 4.14 地下的世界

- 除了地表，地下也有一套完整的生态系统和文明
- 地下城市（矮人/暗精灵/地底种族）
- 独特的生物发光生态系统
- 地下与地表的贸易/冲突/外交
- 深渊——愈深愈危险的梯度设计

### 4.15 死亡的深度

当前死亡 = NPC 标记 `is_dead=true` 并归档。但可以更丰富：
- 葬礼——不同文化/信仰的葬礼形式完全不同
- 遗嘱——NPC 的财产分配
- 哀悼——亲友 NPC 的情绪和关系受影响
- 墓地/陵园——可探访的地点
- 亡灵魔法——死者可能以某种形式"回来"
- 幽灵——未完成心愿的 NPC 可能以幽灵形式存在

### 4.16 宠物与动物伙伴

- NPC 饲养宠物（狗/猫/鸟/奇幻宠物）
- 宠物影响 NPC 的情绪和社交
- 玩家可以收养/购买/驯服动物伙伴
- 战斗宠物（猎犬、战马、幼龙）
- 宠物与主人的关系随互动增长

### 4.17 无聊与创造

`技术架构.md` 提到了 `boredom_innovation.gd`（无聊与创新）模块，但没有任何设计文档：
- NPC 无聊时会做什么？（闲逛/找乐子/搞破坏/创造新事物）
- 无聊是否触发创新？（NPC 发明新的制作方法、新的娱乐形式）
- 无聊与生产力的关系（无聊→生产力下降 or 无聊→创造性突破？）
- 无聊是文化变革的驱动力之一？

### 4.18 "灵"元素的社会应用

`元素.md` 中对"灵"元素的定义极富想象力——它是生物连接其他元素的桥梁。这可以扩展到：
- 灵媒——能与死者之灵沟通的 NPC
- 心灵感应——高灵元素天赋者之间的非语言交流
- 灵元素共鸣——两个关系深厚的 NPC 可以感知对方的情绪/危险
- 灵元素附魔——将一段记忆或情感注入物品中

### 4.19 血元素的深度

`元素.md` 将血元素定义为"金木水火土风雷电的组合体"——这非常独特：
- 血魔法——用自己的血强化魔法（消耗生命换威力）
- 血脉——某些家族的血元素构成特殊（龙血/神血/精灵血）
- 输血/献血——血元素的转移
- 血液疾病——血元素失衡导致的特殊疾病

### 4.20 冒险小队的多人体验

`冒险小队.md` 的设计可以扩展：
- 不同人格组合的队伍有不同的化学反应（两个高神经质 NPC 在一起容易吵架）
- 队伍士气系统
- 队伍内部的角色分工（队长/厨师/侦察兵/战斗员）
- 队伍记忆——队伍一起经历的事成为集体记忆
- 队伍解散后的影响（前队员的关系、对队长的评价）

### 5.1 清理重复内容

`总设计草稿.md` 中有两个 §3.1 和两个 §3.2，应将体素世界和世界规模合并为一个清晰的章节。

建议为各子系统建立"依赖设计文档"清单，确保在开始实现前所需的设计文档已经就绪。

  - [ ] YY系统数据结构已定义
  - [ ] ZZ系统接口已预留
```
