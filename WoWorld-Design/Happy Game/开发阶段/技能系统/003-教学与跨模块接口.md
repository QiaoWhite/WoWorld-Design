## 一、教学系统

### 1.1 TeachingSession —— 通用数据模型

```rust
/// 一次教学会话 —— 覆盖所有技能类型的通用机制
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TeachingSession {
    pub session_id: SessionId,
    pub teacher: NpcId,
    pub student: NpcId,
    pub skill: SkillId,
    pub path: LearningPath,
    pub intensity: f32,           // 0.0–1.0（学院=0.5 标准化, 师承=0.8-1.0 个性化）
    pub started_at: GameTimestamp,
    pub accumulated_hours: f32,   // 累计教学时长
    pub terminated: bool,
}

/// 四种学习路径
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum LearningPath {
    /// 自学 —— 实践 + 阅读魔导书
    SelfStudy,
    /// 师承 —— 一对一私人教学
    Apprenticeship,
    /// 学院 —— 系统化标准化教育
    Academy,
    /// 秘传 —— 仪式/契约注入（由 Magic 模块管辖）
    /// 技能系统不执行秘传逻辑——仅提供 add_xp_direct() 入口
    ArcaneTransmission,
}
```

### 1.2 四种路径对比

| | 自学 | 师承 | 学院 | 秘传 |
|------|------|------|------|------|
| **XP倍率**（相对自学） | 1.0×（基准） | 1.2–3.0× | 1.3× | 瞬时大量XP（数量由Magic仪式决定） |
| **上限** | 无硬上限（指数减速自然形成） | 不超过导师 level × 0.95 | 无硬上限 | 极高 |
| **导师需求** | 无 | 1 teacher（≥ student+5 levels） | 1+ teachers + 设施 | 仪式主持人 |
| **进度灵活性** | 最高——完全自主 | 高——师承个性化 | 低——标准化进度 | 一次性——不可控 |
| **风险** | 无 | 低-中（取决于导师水平） | 低 | **由 Magic 模块定义**（契约条款） |
| **社交收益** | 无 | 与导师关系加深 | 同学网络（未来商业/政治资本） | 仅与契约方 |
| **适用技能类别** | 全部 | 全部（但需找到导师） | Combat/Magic/Academic/Artisan | **仅 Magic** |

### 1.3 师承教学公式

```rust
/// 师承路径 —— 每小时获得的教学 XP
fn teaching_xp_per_hour(
    teacher_skills: &BTreeMap<SkillId, SkillEntry>,
    student_skills: &BTreeMap<SkillId, SkillEntry>,
    skill_id: SkillId,
    teacher_mental: &dyn MentalAccess,
    student_mental: &dyn MentalAccess,
) -> f32 {
    let teacher_entry = match teacher_skills.get(&skill_id) {
        Some(e) => e,
        None => return 0.0,  // 导师不会这个技能→无法教学
    };
    let student_entry = student_skills.get(&skill_id);

    let teacher_level = teacher_entry.level;
    let student_level = student_entry.map(|e| e.level).unwrap_or(0.0);

    // 导师必须比学生显著高才能有效教学
    let knowledge_gap = teacher_level - student_level;
    if knowledge_gap <= 5.0 {
        return 0.0;  // 差距太小——无教学价值
    }

    // 教学能力 —— 由导师自身水平和表达能力决定
    let teaching_ability = (teacher_level / 100.0)
        * teacher_mental.intelligence()
        * (0.5 + teacher_mental.charisma() * 0.5);

    // 学习能力 —— 由学生智力和天生天赋决定
    let student_aptitude = student_entry
        .map(|e| e.innate_aptitude)
        .unwrap_or(1.0);  // 从未接触→默认1.0
    let learning_ability = student_mental.intelligence() * student_aptitude;

    // 基础教学速度
    let base_xp_per_hour = knowledge_gap * 0.5;

    base_xp_per_hour * teaching_ability * learning_ability
}

/// 教学上限 —— 学生不能无限接近导师
fn teaching_cap(teacher_level: f32) -> f32 {
    teacher_level * 0.95
}

/// 示例计算:
/// 导师锻造 Lv85, 学生锻造 Lv20, 导师智力0.7, 学生智力0.6, 学生天赋1.15
///   teaching_ability = 0.85 × 0.7 × 1.0 = 0.595
///   learning_ability = 0.6 × 1.15 = 0.69
///   base_xp = (85 - 20) × 0.5 = 32.5 XP/h
///   实际 = 32.5 × 0.595 × 0.69 = 13.3 XP/h
///   → 对比自学锻造:~7 XP/h（取决于材料/任务频率）
///   → 师承效率 = 13.3/7 = 1.9×
```

### 1.4 学院路径

学院是师承的多人标准化变体：

```rust
fn academy_xp_per_hour(
    course_level: f32,           // 课程目标等级
    student_level: f32,
    student_mental: &dyn MentalAccess,
    student_aptitude: f32,
) -> f32 {
    if student_level >= course_level + 10.0 {
        return 0.0;  // 学生已远超课程水平
    }

    let knowledge_gap = (course_level - student_level).max(1.0);
    let learning_ability = student_mental.intelligence() * student_aptitude;

    // 学院效率低于顶级师承（标准化vs个性化）但高于劣质师承
    let base_xp_per_hour = knowledge_gap * 0.35;

    base_xp_per_hour * learning_ability
}

/// 学院一班最多30人，一师对多生
/// 学生之间的技能互助未建模（留给未来社交/经济系统）
```

### 1.6 TeachingRisk trait —— 默认无风险

```rust
/// 教学风险 —— 默认空实现
///
/// 大多数技能教学无特殊风险:
///   - 锻造教学: 可能浪费材料（经济问题，非技能系统管辖）
///   - 历史教学: 无风险
///   - 剑术教学: 对练可能有轻伤（Combat 可选实现）
///
/// 实现方（可选）:
///   - Combat → 实现 on_session_tick: 轻伤概率(1-3%/h)
///   - Magic → 实现 on_session_end (秘传): 契约条款触发
pub trait TeachingRisk {
    fn on_session_start(&self, _session: &TeachingSession) -> Vec<SideEffect> {
        vec![]
    }
    fn on_session_tick(&self, _session: &TeachingSession) -> Vec<SideEffect> {
        vec![]
    }
    fn on_session_end(&self, _session: &TeachingSession) -> Vec<SideEffect> {
        vec![]
    }
}

pub enum SideEffect {
    MinorInjury { probability: f32 },       // 轻伤
    WeaponDamage { probability: f32, amount: f32 },  // 武器磨损
    ContractClause { clause_id: u32 },       // Magic: 契约条款触发
    StatBoost { stat: String, amount: f32 }, // 正面效果
}
```

## 四、魔法模块接口

### 4.1 秘传路径 —— add_xp_direct

Magic 016 的秘传路径通过以下流程与技能系统交互：

```
1. 启动仪式 → Magic 模块计算: ritual_xp_amount + contract_clauses
2. 调用技能系统 add_xp_direct(skill, xp_amount, student.aptitude, ...)
3. 技能系统写入 XP + 触发交叉训练
4. Magic 模块记录: 契约条款 → NPC的 obligations/debts
5. 返回 CrossTrainingEvent 列表 → Magic 可选记录
```

**技能系统不参与**：仪式条件（月相/季节/地点）、契约内容（隐藏条款/代价）、注入知识的类型限制。

### 4.2 本能化法术阈值

Magic 016 的六阶段成长模型:

```
阶段一: 依赖魔导书 → times_used < 100
阶段二: 缩短吟唱   → times_used ≥ 100
阶段三: 完整记忆   → times_used ≥ 250
阶段四: 精简吟唱   → times_used ≥ 500
阶段五: 默发       → times_used ≥ 1000 + level ≥ 60
阶段六: 瞬发       → times_used ≥ 2000 + level ≥ 85
```

技能系统提供 `times_used` 字段 → Magic 模块读取并判断阶段。

### 4.3 元素组边界

Magic 002 定义的元素兼容/对立关系 → 技能系统在 TOML 中配置 cross_training_ratio。修改不需要重新编译。

## 七、世界生成模块接口

### 7.1 generate_starting_skills 完整算法

World Gen P8（NPC 初始化）调用 `generate_starting_skills()`：

```rust
fn generate_starting_skills(
    age_years: u32,
    profession_skills: &[(SkillId, f32, f32)],  // (skill, min_level, max_level)
    mental: &dyn MentalAccess,
    physical: &dyn PhysicalAccess,
    skill_defs: &HashMap<SkillId, SkillDef>,
    settlement_tech: &dyn SettlementTechQuery,  // ★ v1.0 TechEra 幽灵类型替换——见 文化系统 005
    starting_biome: BiomeParams,
    rng: &mut impl Rng,
) -> BTreeMap<SkillId, SkillEntry> {
    let mut skills = BTreeMap::new();

    // Step 1: 为每个职业主技能分配等级
    for &(skill_id, min_lvl, max_lvl) in profession_skills {
        // 年龄修正: 18岁→偏min, 40岁→偏max
        let age_factor = ((age_years as f32 - 12.0) / 50.0).clamp(0.0, 1.0);
        let target = min_lvl + (max_lvl - min_lvl) * age_factor;

        // 天赋修正: 高天赋→倾向于更高等级
        let innate = generate_innate_aptitude(rng);  // N(1.0, 0.15)
        let adjusted = (target * innate).clamp(min_lvl, max_lvl);

        // 随机噪声 ±5
        let noise = rng.gen_range(-5.0, 5.0);
        let final_level = (adjusted + noise).clamp(0.0, 100.0);

        let xp = total_xp_for_level(final_level);
        let def = &skill_defs[&skill_id];
        let mental_factor = compute_mental_factor(mental, physical, def.category);

        skills.insert(skill_id, SkillEntry {
            level: final_level,
            xp,
            innate_aptitude: innate,
            total_xp_earned: xp,  // 初始技能的XP都算"已获得"
            times_used: estimate_times_used(final_level, age_years, mental_factor, innate, rng),
        });
    }

    // Step 2: 模拟交叉训练效果（职业主技能→副技能）
    for (&primary_skill, &primary_entry) in skills.clone().iter() {
        let def = &skill_defs[&primary_skill];
        let cross_events = simulate_cross_training_passive(
            primary_skill, primary_entry.xp, def, &skill_defs, &skills, rng
        );
        for event in cross_events {
            skills.entry(event.skill_id)
                .and_modify(|e| {
                    e.xp += event.cross_xp;
                    e.level = level_from_xp(e.xp).min(cross_training_cap(primary_entry.level));
                })
                .or_insert_with(|| SkillEntry {
                    level: level_from_xp(event.cross_xp).min(cross_training_cap(primary_entry.level)),
                    xp: event.cross_xp,
                    innate_aptitude: generate_innate_aptitude(rng),
                    total_xp_earned: 0.0,  // 交叉训练的XP不算 total_xp_earned
                    times_used: 0,
                });
        }
    }

    // Step 3: 添加出生环境自然获得的生存技能
    add_environmental_skills(&mut skills, starting_biome, age_years, rng);

    skills
}
```

#### 7.1.1 `simulate_cross_training_passive` 关键不变式

此函数从主技能的累积 XP 反推同组/跨子组技能的交叉训练 XP。必须遵守的不变式：

```
不变式 1（非递归）：
  交叉训练产生的 XP 不再触发新的交叉训练。
  实现：迭代主技能列表（profession_skills），对每个主技能计算交叉，
        但交叉结果写入时不加入"主技能列表"——不重新进入迭代。

不变式 2（天花板）：
  交叉训练不能将目标技能推到超过 min(40, source_level × 0.5)。
  如果 target 已有更高等级（来自另一个主技能的交叉或自身分配），不覆盖。

不变式 3（处理顺序）：
  1. 收集所有主技能 → 计算对每个目标技能的裸交叉 XP
  2. 对每个目标技能，取所有来源贡献的 XP 之和（cap 到天花板）
  3. 加入随机噪声（±交叉XP的20%）
  顺序保证：多个主技能对同一目标的贡献不会因处理顺序不同而产生不同结果。

不变式 4（不乘 mental_factor）：
  交叉训练的 XP 计算使用源主技能的 base_xp（未修正的），
  因为目标技能有自己的 mental_factor 在 level_from_xp 中已体现——
  此处只传输"XP量"不二次应用属性修正。
```

#### 7.1.2 `add_environmental_skills` 环境→技能映射

出生环境决定初始生存技能的有无和基础等级——纯数据驱动：


### 7.2 SkillProfile —— 职业→技能配置（暂存）

```rust
/// SkillProfile —— 描述一个职业"应该有什么技能"
///
/// ⚠️ 此类型暂存于技能系统。
/// 未来 Profession 系统成型后迁出。
/// TOML 数据驱动——不是代码。
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SkillProfile {
    pub name: String,                              // "铁匠"
    pub primary: Vec<(SkillId, f32, f32)>,         // (skill, min_level, max_level)
    pub secondary: Vec<(SkillId, f32, f32)>,       // 副技能
```

## 九、生命模块接口

### 9.1 PhysicalAccess 实现

> **PhysicalAccess 有意仅暴露 4/6 身体属性**（strength/dexterity/agility/endurance）。Life 004 的 `constitution`（体质）和 `perception`（感知）不由技能系统消费——constitution 影响 HP/疾病抵抗（Life 管辖），perception 由 survival·感知组的 wisdom(mental) 覆盖。只有被 `mental_factor()` 使用的属性才加入此 trait。

Life 模块实现 PhysicalAccess trait：

```rust
// Life 模块中
impl PhysicalAccess for LifeEntity {
    fn strength(&self) -> f32 {
        self.physical_attributes.strength  // Life 004 §二
    }
    fn dexterity(&self) -> f32 {
        self.physical_attributes.dexterity
    }
    fn agility(&self) -> f32 {
        self.physical_attributes.agility
    }
    fn endurance(&self) -> f32 {
        self.physical_attributes.endurance
    }
}
```

### 9.2 灵兽技能

灵兽使用与 NPC 相同的 `SkillEntry` 数据模型——同一套 XP 管线，但调用方不同（驯养者训练而非自主行动）：

```rust
/// 灵兽可用技能白名单——由物种定义约束
struct CreatureSkillConstraints {
    allowed_skills: Vec<SkillId>,     // 白名单（如狗: 追踪/侦察/游泳/攀爬）
    mental_override: Option<CreatureMentalProfile>,
}

/// 灵兽的心智属性——从物种本能+驯养纽带派生
impl MentalAccess for WolfCompanion {
    fn intelligence(&self) -> f32 { 0.55 }    // 狼的物种智力
    fn wisdom(&self) -> f32 { 0.70 }          // 本能驱动强
    fn willpower(&self) -> f32 { self.loyalty } // 忠诚=服从训练
    fn charisma(&self) -> f32 { 1.0 }         // 对灵兽不相关
}
```

**与 NPC 的区别**：

| | NPC | 灵兽 |
|------|-----|------|
| XP 来源 | add_xp（自主行动） | add_xp_direct（被训练/被奖励） |
| MentalAccess | 完整 MentalAttributes | 物种本能+驯养纽带映射 |
| 技能白名单 | 无（全技能可用） | 物种约束（狗不能学锻造） |
| 教学 | 可作为 teacher | 不可教学——仅被训练 |
| 交叉训练 | 完全支持 | 同组交叉训练（追踪→侦察）仍生效 |

**老猎犬追踪 Lv70 vs 新猎犬追踪 Lv5 —— 同一套 SkillEntry，涌现出差异。**

灵兽原有的 `TrainedSkill` 枚举（Life 006）作为驯服事件的**触发入口**保留——"此灵兽学会了追踪"通过灵兽模块的事件系统触发。但追踪的熟练度用技能系统的 `SkillEntry` 追踪。驯养者奖励灵兽→调用 `add_xp_direct(追踪, xp_amount)`。

> ⚠️ **与 Life 006 的协调**：灵兽的 `trained_skills: Vec<TrainedSkill>` 字段未来可迁移为 `skills: BTreeMap<SkillId, SkillEntry>`——由技能系统管理。迁移时机由 Life 模块决定，不在此 v1.0 范围。

## 十、性能预算

| 项目 | 数值 | 备注 |
|------|------|------|
| SkillDef 加载 | ~5KB (~100条定义) | TOML → 启动时一次性解析 |
| SkillEntry per NPC | ~10-20 entries × ~40 bytes = 400-800 bytes | 稀疏存储 |
| 100K NPCs 技能总存储 | ~60MB | 15技能/NPC × 40B × 100K |
| add_xp 单次调用 | <10μs | XP公式纯数学 + 交叉训练查表(每组≤6技能) |
| 交叉训练查询 | O(cross_group_size) | 每组≤7技能, 哈希表O(1)定位 |
| teaching_xp_per_hour | <1μs | 纯数学公式 |
| generate_starting_skills | <100μs/NPC | World Gen离线计算 |
| MentalAccess 调用 | <0.1μs | 直接字段访问 |
| level_from_xp / total_xp_for_level | <0.5μs | ln/exp 硬件指令 |
| skills_snapshot (全量) | <10μs/NPC | 筛选≥50的条目 |

### 存储（LMDB）

技能数据作为 NPC 存档的一部分存储——不独立存储：

```
NPC存档 (LMDB key = NpcId):
  ├─ npc_identity      (~200B)
  ├─ physiology        (~128B)
  ├─ emotion           (~64B)
  ├─ skills             (~800B)  ← BTreeMap<SkillId, SkillEntry> bincode序列化
  ├─ relationships     (~2KB)
  └─ ...                (~4KB)
```

## 十一、[TUNING] 可调参数

所有参数在 `config/skills.toml` 中可调——不需要修改代码：

```toml
[tuning]
xp_base = 100.0          # 累积XP公式 base
xp_k = 0.04              # 累积XP公式 k（减速系数）
hard_cap = 100.0         # 硬上限

[mental_factor]
range_min = 0.7          # 最小心智因子（最差心智→减速30%）
range_max = 1.0          # 最大心智因子（最好心智→加速30%）

[innate_aptitude]
mean = 1.0               # 正态分布均值
std_dev = 0.15           # 标准差
clamp_min = 0.7          # 下限
clamp_max = 1.3          # 上限

[cross_training]
same_subgroup_ratio = 0.20
same_category_ratio = 0.10
cross_category_ratio = 0.05
opposing_elements_ratio = 0.04
cap_absolute = 40.0      # 交叉训练绝对天花板
cap_ratio = 0.5          # 交叉训练相对天花板（source_level × 0.5）
max_cross_boost = 0.5    # 同组高技能对主技能学习的最大加速

[teaching]
knowledge_gap_min = 5.0         # 有效教学的最小等级差距
knowledge_gap_xp_mult = 0.5     # 差距→XP转换系数
teaching_cap_ratio = 0.95       # 教学上限（导师 level × 0.95）
academy_xp_mult = 0.35          # 学院效率系数（vs师承的0.5）

[starting_skills]
noise_range = 5.0               # 初始技能随机噪声 ±5级
age_curve_steepness = 50.0      # 年龄曲线陡度
```

## 十二、与 CHG-013 契约兼容

本模块不修改任何已有接口契约：

- 不修改 Physiology/Vitals 派生关系
- 不修改魔力单位（u32 刻）
- 不修改十元素顺序（金木水火土风雷电血灵）
- 不修改部位伤害渐进模型
- `SkillId` / `SkillEntry` 是**新增**命名空间——不作为任何已有 ID/属性的替代
- NPC 的 `SkillCategory` 从 7 类缩减为 5 类（删除 Social/Economic）——**以技能系统为权威**。这是对 NPC 文档的更新，不破环已有数据流
