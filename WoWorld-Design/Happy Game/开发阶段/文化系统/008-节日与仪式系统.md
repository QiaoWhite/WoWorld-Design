## 一、设计哲学

### 1.1 定义

- **仪式（Ritual）**：结构化的集体行为——有主祭、参与者、地点、言语和身体动作。仪式是原子——不可再分的仪式行为单元。
- **节日（Festival）**：一个或多个仪式按日历循环重复的大规模文化活动。节日是仪式在时间轴上的组织框架。
- **个人仪式（Personal Ritual）**：由生命事件触发的单次仪式（婚礼、葬礼、成年礼）——使用同一套 `RitualDef` 结构，但不由日历调度。

### 1.2 核心主张

**仪式是统一的。节日是仪式在日历上的循环。权力的强制是权力系统的事——不耦合到节日系统。**

```
RitualDef —— 所有仪式的通用原子（文化系统 008）
    │
    ├─── + 日历调度 → FestivalFramework（节日）
    │
    ├─── + 生命事件触发 → PersonalRitual（个人仪式——由 NPC 生命模块调度）
    │
    └─── + 魔法成分 → 魔法仪式（MagicRitualRef 引用魔法系统 010）
```

### 1.3 模块边界

| 文化系统 008 管 | 其他模块管 |
|----------------|----------|
| RitualDef 的通用结构定义 | 仪式魔法的魔力执行逻辑 → [[../魔法/06-人文层/010-仪式魔法|魔法 010]] |
| FestivalFramework 的日历调度 | 个人仪式的 GOAP 触发 → [[../NPC活人感模块/NPC活人感开发文档ver2.0|NPC 活人感]] |

## 二、RitualDef —— 仪式原子

### 2.1 数据结构

```rust
/// 仪式定义——所有仪式的通用原子结构
/// 
/// 无论是一个国王的加冕、铁匠的婚礼、还是收获节的开场祭祀——
/// 都使用同一套 RitualDef。区别仅在于参与者的规模和仪式的调度方式。
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RitualDef {
    /// 仪式在文化中的名称（如"加冕礼"、"婚誓"、"丰收祭"）
    pub name: GeographicName,
    
    /// 主祭——谁主持仪式
    pub celebrant: CelebrantRole,
    
    /// 参与者角色及数量要求
    pub participants: Vec<ParticipantRequirement>,
    
    /// 仪式地点类型
    pub location: RitualLocation,
    
    /// 预计持续时长（实际分钟）
    pub duration_minutes: f32,
    
    /// 言语部分——与 TurnMode::Ritual 对接
    pub speech: Vec<RitualSpeechStep>,
    
    /// 身体动作部分
    pub actions: Vec<PhysicalAction>,
    
    /// 可选的物品要求
    pub required_items: Vec<ItemRequirement>,
    
    /// 可选的超自然成分——引用魔法系统
    pub magic: Option<MagicRitualRef>,
}

/// 主祭角色
pub enum CelebrantRole {
    /// 统治者主持——权力合法性的展示
    Ruler { power_edge_id: Option<PowerEdgeId> },
    /// 宗教人员——祭司/萨满/僧侣
    Clergy { npc_id: Option<NpcId>, faith_id: FaithId },
    /// 长者——宗族/氏族传统
    Elder { npc_id: Option<NpcId> },
    /// 轮流——每年由不同人担任
    Rotating { selection: SelectionRule },
    /// 无主祭——全体参与者共同完成
    Collective,
    /// 仪式主体自身——婚礼中新郎/新娘自己宣誓
    Subject,
}

/// 参与者要求
pub struct ParticipantRequirement {
    pub role: ParticipantRole,
    pub count: Range<u32>,
    pub eligibility: Vec<EligibilityCriterion>,
}

pub enum ParticipantRole {
    Assistant,      // 助祭
    OfferingBearer,  // 祭品携带
    Choir,           // 诵经/唱诗
    Witness,         // 见证人
    Audience,        // 观礼群众
}

/// 仪式地点
pub enum RitualLocation {
    /// 建筑物内——神殿/王宫大厅
    Building { building_id: BuildingId },
    /// 露天——广场/圣泉/古树下
    OpenAir { geographic_entity_id: GeographicEntityId },
    /// 游行——沿特定路线移动
    Procession { waypoints: Vec<GeographicEntityId> },
    /// 由当时上下文决定——在节日举办地举行
    FestivalDefault,
}

/// 言语步骤——与 语言表达 005 TurnMode::Ritual 的 RitualStep 对齐
pub struct RitualSpeechStep {
    pub speaker_role: SpeakerRole,
    pub text_template_id: u32,
    pub params: FxHashMap<String, String>,
    pub duration_seconds: f32,
}

pub enum SpeakerRole {
    Celebrant,
    SpecificParticipant { role: ParticipantRole, index: u32 },
    AllParticipants,    // 群众齐声
    CallAndResponse,    // 主祭↔参与者 应答
}

/// 身体动作
pub enum PhysicalAction {
    Kneel,
    Prostrate,
    RaiseHands,
    OfferItem { item_slot: u8 },
    Pour { substance: String },
    Light { target: String },  // 点燃火炬/蜡烛
    ProcessionStep { from: u8, to: u8 },
    Dance { dance_type: String },
    Silence { duration_seconds: f32 },
}

/// 魔法成分——引用魔法系统，不定义魔法逻辑
pub struct MagicRitualRef {
    /// 魔法系统 010 中定义的仪式规格 ID
    pub magic_spec_id: u32,
    /// 预期魔力消耗
    pub expected_mana_cost: u32,
    /// 风险等级
    pub risk_level: MagicRiskLevel,
}

pub enum MagicRiskLevel { Safe, LowRisk, ModerateRisk, HighRisk, Catastrophic }
```

### 2.2 RitualDef 的三种使用语境

| 语境 | 调度方式 | 规模决定 | 示例 |
|------|---------|---------|------|
| **节日中的仪式** | FestivalFramework.schedule 绑定到 day_of_year | 节日规模 × 文化参数 | 收获节开场祭祀、新年祈福 |

## 四、节日生命周期

### 4.1 阶段流转

```
Dormant → Announcement → Preparation → Celebration → Aftermath → Concluded
  ↑                                                                    │
  └──────────────────── 下一年重新开始 ───────────────────────────────┘
```

| 阶段 | 触发 | 持续 | 关键行为 |
|------|------|------|---------|
| **Announcement** | `day_of_year - announcement_lead_days` | 3-7天 | 信息传播系统发出节日宣告；NPC 获知 |
| **Preparation** | `day_of_year - preparation_lead_days` | 1-3天 | 购物激增、服饰准备、装饰出现、外地 NPC 旅行 |
| **Celebration** | `day_of_year` | 1-7天 | 仪式执行、竞技、宴饮、舞蹈——日常 Routine 暂停 |
| **Aftermath** | 庆祝期结束后 | 1-2天 | 装饰拆除、经济结算、情绪回落、历史记录 |

### 4.2 跨年边界处理

```rust
/// 判断给定 day_of_year 是否在节日的指定阶段内
fn is_day_in_phase(framework: &FestivalFramework, phase: FestivalPhase, day: u16) -> bool {
    let (start_offset, end_offset) = match phase {
        FestivalPhase::Announcement => (-(framework.announcement_lead_days as i32), 0),
        FestivalPhase::Preparation => (-(framework.preparation_lead_days as i32), 0),
        FestivalPhase::Celebration => (0, framework.duration_days as i32),
        FestivalPhase::Aftermath => (framework.duration_days as i32, framework.duration_days as i32 + 2),
        _ => return false,
    };
    
    let start = modulo_120(framework.day_of_year as i32 + start_offset);
    let end = modulo_120(framework.day_of_year as i32 + end_offset);
    
    if end >= start { day >= start && day < end }
    else { day >= start || day < end } // 跨越年边界
}

fn modulo_120(n: i32) -> u16 {
    ((n % 120 + 120) % 120) as u16
}
```

### 4.3 天气对节日规模的影响

```rust
/// 每类活动的天气敏感性——热带/户外/室内不同
fn weather_sensitivity(activity: &FestivalActivityType) -> WeatherSensitivity {
    match activity {
        Procession | Tournament | Race | Fireworks | Dance | Market
            => WeatherSensitivity { heavy_rain_cancels: true, wind_sensitive: true, ..indoor(false) },
        Feast | Music | Theatre | Contest | GiftGiving
            => WeatherSensitivity { heavy_rain_cancels: false, wind_sensitive: true, ..indoor(false) },
        Prayer | Sacrifice | RitualCleansing | Matchmaking | Auction
            => WeatherSensitivity::indoor(),
        Fireworks | Decoration
            => WeatherSensitivity { heavy_rain_cancels: true, wind_sensitive: true, ..indoor(false) },
        _ => WeatherSensitivity::default(),
    }
}

/// 将天气敏感活动在恶天气下标记为取消——降低节日 scale
fn apply_weather_to_festival(instance: &mut FestivalInstance, weather: &WeatherSample, framework: &FestivalFramework) {
    let mut cancelled_count = 0u32;
    let total = framework.activities.len() as u32;
    
    for activity in &framework.activities {
        let sens = weather_sensitivity(activity);
        if sens.is_cancelled_by(weather) { cancelled_count += 1; }
    }
    
    instance.weather_impact = 1.0 - (cancelled_count as f32 / total.max(1) as f32) * 0.7;
    instance.scale *= instance.weather_impact;
}
```

```rust
/// 每文化节日数量——归一化密度指数
fn festival_count_for_culture(core: &CultureCoreParams, rng: &mut impl Rng) -> u32 {
    let density = (
        core.religiosity * 0.30
        + core.artistry * 0.25
        + core.indulgence * 0.20
        + core.competition_orientation * 0.15
        + (1.0 - core.uncertainty_avoidance) * 0.10
    ).clamp(0.0, 1.0);
    
    let count = 2.0 + density * 5.0;
    let jitter = rng.gen_range(-0.5..0.5);
    (count + jitter).round().clamp(2.0, 7.0) as u32
}

/// 节日类型的文化驱动权重
fn festival_type_weights(core: &CultureCoreParams) -> Vec<(FestivalTypeTag, f32)> {
    let mut w = vec![(CalendarTag, 1.0)]; // 所有文化都有日历节日
    
    if core.religiosity > 0.3 {
        w.push((ReligiousTag, core.religiosity * 1.5));
    }
    if core.militarism > 0.4 || core.competition_orientation > 0.5 {
        w.push((CompetitiveTag, (core.militarism + core.competition_orientation) / 2.0));
    }
    if core.artistry > 0.5 {
        w.push((ArtisticTag, core.artistry));
    }
    w
}
```

## 六、NPC 参加决策

### 6.1 多因子公式

```rust
fn will_attend_festival(
    npc: &NpcData, festival: &FestivalInstance, framework: &FestivalFramework,
    culture: &dyn CultureQuery, power: &dyn PowerTopologyQuery,
    season: &SeasonState, piety: f32,
) -> f32 {
    // 1. 权力义务——PrescribeRule 创建的强制参加 Duty
    let obligation = if power.has_duty_for(npc.id, DutyType::Obligation, 
        &DutyContent::MandatoryObservance { festival_id: festival.festival_id }) 
        { 0.95 } else { 0.0 };
    
    // 2. 虔诚度——宗教节日的核心因子
    let piety_factor = if framework.bound_faith.is_some() { piety * 0.4 } else { 0.0 };
    
    // 3. 人格因子
    let personality = npc.big_five.extraversion * 0.20
        + npc.big_five.openness * 0.15
        - npc.big_five.neuroticism * 0.10;
    
    // 4. 社交因子——有亲密关系的人决定去？
    let social = if npc.has_close_relations_attending(festival.festival_id) { 0.25 } else { 0.0 };
    
    // 5. 距离衰减——指数
    let dist_km = npc.position.distance_km(festival_location(festival).center());
    let distance = (-dist_km / 10.0).exp().clamp(0.05, 1.0);
    
    // 6. 季节气候期望（非当天实际天气——NPC 无法预知）
    let weather_expect = match season.season {
        Season::Summer => 0.95, Season::Spring => 0.85,
        Season::Autumn => 0.80, Season::Winter => 0.60,
    };
    
    let raw = obligation.max(piety_factor + personality * 0.25 + social 
        + distance * 0.20 + weather_expect * 0.10 + 0.1);
    raw.clamp(0.0, 1.0)
}
```

### 6.2 NPC 行为覆盖

庆祝期到场 NPC 的日常 Routine 被覆盖——非必要行为暂停，节日活动权重 ×3.0。NPC 服装自动切换：NPC 模块自身在决策循环中检测"正在庆祝+我在参加"→ `OutfitMode::Ceremonial`（由 NPC 模块拉取，不由节日系统推送）。

### 6.3 人群聚集优化

- **分帧错开**：宣告期+预备期 3-7 天内 NPC 行动计划自然分散——不是所有人在 Day 1 早晨同时冲向广场
- **L1 (<50m)**：个体 A* 寻路
- **L2 (50-200m)**：预计算流场（居住区→广场的方向场），NPC 沿流场移动
- **L3+L4**：统计模型——密度贴图，不模拟个体

## 九、与权力系统的桥接

### 9.1 合法性 ritual 因子

权力系统在每日合法性重算时主动拉取：

```rust
// 权力系统 004::recalc_legitimacy() 中
let ritual_factor = if let Some(record) = ritual_query.recent_ritual_involving(edge.holder, 3650) {
    let days_since = (current_time - record.performed_at).as_days();
    if days_since < 30 { 0.15 }
    else if days_since < 365 { 0.08 }
    else if days_since < 3650 { 0.03 }
    else { 0.0 }
} else { 0.0 };
```

权力系统使用 `RitualQuery` trait 拉取数据——不反向依赖节日系统。`edge.last_ritual_at` 是权力系统内部缓存——由权力系统根据 RitualQuery 的结果自行更新。

### 9.2 强制庆祝——PrescribeRule

统治者创建强制庆祝义务时，权力系统不需要知道节日：

```rust
// 统治者行使 PrescribeRule 原子
PrescribeRuleParams {
    scope: RuleScope::Everyone,
    content: RuleContent::MandatoryObservance {
        occasion: "国王生日",
        expected_behavior: "参加庆祝活动",
        annual_day: 90,  // day_of_year
    },
    sanction_on_violation: SanctionSpec { ... },
}
// → 权力系统为所有 subject 创建 Duty { trigger: Periodic { interval: 365_days } }
// → NPC 在每日 Duty 检查中知道自己有义务
// → NPC 的节日参加决策中 obligation_factor = 0.95
```

### 9.3 PowerSource::Divine 仪式参照

`PowerSource::Divine { ritual: String }` —— 此处的 `ritual` 字符串是 `RitualRegistry` 中已注册仪式的名称。权力系统通过 `RitualQuery` 验证该仪式确实被执行过。仪式定义的权威在文化系统——权力系统不定义仪式。

## 十、节日命名

复用 [[007-地名系统|地名 007]] 的 `GeographicName` 基础设施。节日名称从文化的音系规则和描述词库生成——与山河命名使用同一套底层。

```rust
fn generate_festival_name(
    festival_type: &FestivalType,
    emotional_tone: &EmotionalTone,
    lexicon: &CultureToponymyLexicon,
    rng: &mut impl Rng,
) -> GeographicName {
    match festival_type {
        FestivalType::Calendar { anchor } => {
            GeographicName::from_descriptive(
                lexicon.season_word(anchor.season()),
                lexicon.festival_suffix(emotional_tone),
                lexicon.phonology(),
            )
        }
        FestivalType::Religious { .. } => {
            GeographicName::from_associative(
                lexicon.deity_name(deity_id),
                lexicon.festival_suffix(emotional_tone),
            )
        }
        // ...
    }
}
```

不同文化对同一节日的他称通过 exonym 机制处理（与地名 exonym 相同的五源模型）。

## 十一、节日演变

### 11.1 文化反馈

节日每年完成后对文化参数的温和推力：

```rust
fn festival_cultural_feedback(festival: &FestivalFramework, instance: &FestivalInstance, 
                                culture: &mut CultureCoreParams) {
    let novelty = |param: f32| 1.0 - param; // 已很高的参数→再推更难
    let strength = instance.scale * (instance.actual_participants as f32 
        / instance.planned_participants.max(1) as f32) * 0.003;
    
    match festival.emotional_tone {
        EmotionalTone::Ecstatic => culture.indulgence += strength * 0.5 * novelty(culture.indulgence),
        EmotionalTone::Solemn => culture.religiosity += strength * 0.3 * novelty(culture.religiosity),
        EmotionalTone::Competitive => culture.competition_orientation += strength * 0.4 * novelty(culture.competition_orientation),
        _ => {}
    }
    
    culture.clamp_all();
}
```

### 11.2 节日诞生/衰落/融合/漂移

| 路径 | 机制 | 详见参考 |
|------|------|---------|
| 诞生 | 种子生成 / 历史事件结晶 / 涌现检测 / 文化接触引入 | 022-设计探讨 007 |
| 衰落 | 连续 5 年参与率 <20% → Declining → 5 年观察 → Defunct | 优化审查 002 |
| 融合 | 两个文化日期接近+类型相似+长期接触 → FestivalModification | 优化审查 001 |
| 漂移 | 外围元素(食物/舞蹈/装饰)逐渐变化——核心仪式保持稳定 | 022-设计探讨 007 |

## 十三、玩家体验

| 层次 | 行为 | 后果 |
|------|------|------|
| **旁观** | 观看仪式/竞技/表演 | 无直接后果 |
| **参与** | 宴饮、舞蹈、赠礼、逛集市 | NPC 关系提升、经济消费 |
| **竞技** | 报名比赛——与 NPC 同规则判定 | 获胜 → 名声事件(勇武+荣誉) |
| **干预** | 做演讲、救场、捣乱 | 名声显著改变 / 法律后果 |
| **创立** | 史诗任务 → 当地自发纪念 3 年 → 系统检测并正式化 | 间接——玩家不会直接点击"创立节日" |

玩家通过五层线索发现节日：旅行者对话 → 远距离灯火 → 街道装饰 → NPC 服装 → 直接与 NPC 对话询问。跨文化节日体验受文化距离(similarity)调制——文化距离大 → 新奇或困惑。
