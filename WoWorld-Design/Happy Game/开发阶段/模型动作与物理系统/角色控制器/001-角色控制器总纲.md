## 〇、核心哲学

**1. 控制器统一，意图分叉。**
NPC 和玩家共享同一套 `MovementSystem` + `ActionController`。差异只在意图来源——玩家推摇杆，NPC 的 GOAP 规划路径——身体怎么移动是同一个函数。

**2. 涌现式物理——定义规则，让结果涌现。**
开发者定义质量、冲量、加速度曲线、摩擦力。击退距离从 `impulse / mass` 涌现。体型压制从同一条公式涌现。攀爬握力从 `surface_material × weather × fatigue` 涌现。不逐招硬编"此招击退=3m"。

**3. 玩家 = NPC + I/O 适配层。**
玩家不是独立的 Entity 类型。一个"被玩家控制的铁匠 NPC"的 Component 集合 = `[NpcCore, Vitals, SkillHandle, PlayerTag, ControlMode::Manual]`。释放控制 → 拆 `PlayerTag` → GOAP 在下一帧自然接管。夺舍是 Component 的装拆，不是实体复制。

**4. 承诺中断系统——非状态图。**
WoWorld 已有 ~40 个玩家动作 × 10+ 个游戏系统。传统 FSM 的显式状态转换是 O(n²) 的维护灾难。承诺中断系统将所有复杂度移到 TOML 数据中——每个动作声明自己的承诺级别、取消窗口、可被谁取消。核心仲裁逻辑 ~40 行 Rust。

## 一、为什么不使用状态图

### 对比

| | 传统 FSM（Veloren 等） | 承诺中断系统（本模块） |
|---|---|---|
| 状态数 | ~40+ 枚举变体 | 仅 3 个 ActionPhase |
| 转换定义 | 显式 pairs——O(n²) | TOML `cancel_set`——每个动作列出可取消方 |
| 新动作加入 | 修改状态机代码 | 加一段 TOML |
| 仲裁逻辑 | 分散在各状态的 Enter/Exit | 集中在 `ActionController::tick`（~40 行） |
| NPC 与玩家 | 通常需不同状态机 | 同一套——差异在 `ActionRequest.source` |
| 复杂度增长 | 随系统数指数增长 | 随系统数线性增长（TOML 行数） |

### 决策记录

选择承诺中断系统而非 FSM 的原因：
- WoWorld 有 10+ 个独立游戏系统（战斗/魔法/制作/建造/攀爬/骑乘/对话/…），每个系统都有自己的动作。
- FSM 需要为"攀爬时被击中→硬直→滑脱→坠落→着地→站起来"这条链定义显式转换——每加入一个新动作就要更新状态转移矩阵。
- 承诺中断系统只需定义每个动作的 `commitment` level 和 `cancel_set`。被打断的控制流是涌现的，不需要显式编码。

**Veloren 的 FSM 方案**（对比参考）：Veloren 使用 `CharacterState` 枚举（~40+ 变体），每个变体实现自己的 `update()` 逻辑。这在自己的边界内工作良好——但 Veloren 的动作空间远小于 WoWorld（无复杂制作/建造/攀爬/社交/魔法交互）。当动作数膨胀到 WoWorld 级别，FSM 的转换矩阵会超出可维护阈值。

> ⚠️ **这不是贬低 FSM**——在动作空间较小的游戏中，FSM 比承诺系统更直观、更易调试。WoWorld 选择承诺系统是因为其动作多样性不可削减。

## 三、与 CHG-067 LocomotionMode 三态机的关系

CHG-067 定义了三态物理模式。本模块的 ActionController 和 MovementSystem 是其在玩法层的消费者。

### 正交性

| | LocomotionMode（物理层） | MovementState（玩法层） |
|---|---|---|
| 定义方 | CHG-067 运动学地基 | 本模块 002 |
| 语义 | 身体在什么物理状态 | 身体在做什么活动 |
| 变体 | Grounded / PhysicsBody / Attached | Stance × Pace × SpecialMode |
| 变更触发 | 物理事件（着地/击飞/挂载） | 玩家按键 / GOAP 意图 / 介质检测 |

两层通过三个标志交互：
- **`MovementLock`**：ActionController 写入，MovementSystem 读取——决定移动速度和方向是否被动作覆盖
- **`RotationLock`**：ActionController 写入，MovementSystem 读取——决定身体朝向的约束
- **物理中断**：`ImpulseQueue` 命中 → `LocomotionMode` 切 `PhysicsBody` → MovementSystem 自动停止读 MovementState，转 COM 积分

### 控制权交接

```
常态（Grounded + Free）： MovementState.pace 决定速度，MoveIntent.direction 决定方向
动作中（Grounded + Full）：MovementState 被忽略，摩擦力减速，动作根运动接管
被击飞（PhysicsBody）：    MovementState 被忽略，COM 物理接管
落地（PhysicsBody → Grounded）：CMovementRecovery 栈弹出→恢复 MovementState
挂载（Attached + Mounted）：local→world 每帧计算，MovementState 被忽略
```

## 六、整合验证

以下四个场景证明所有子系统可协同工作。每个场景触碰了 3+ 个文档的机制。

### 场景 A: 玩家冲刺中跳跃 → 空中拉弓 → 射箭 → 着地

```
帧 0:  MovementState { Sprinting, Standing }; loco=Grounded
帧 1:  按空格 → ActionRequest { Jump, priority=20 }
帧 2:  ActionController: accept → Windup(50ms) → Active
       MovementModeSystem: loco→PhysicsBody → Airborne(Jumping)
       MovementRecovery: push { Sprinting, Standing }
帧 5:  空中推摇杆——AirState::Jumping { control_ratio: 0.7 }
帧 8:  按左键(弓) → ActionResolver: NotInWater → accept AimBow
       ActionController: Windup(200ms) → Active(∞) → 持续消耗 Stamina
帧 15: 释放左键 → Charged(900ms → full_draw) → pending_follow_up=ShootArrow
帧 16: ShootArrow 执行 → 箭矢发射
帧 18: 着地 → MovementSystem: 三级着地→Grounded
       MovementRecovery: pop → { Sprinting, Standing }
       MovementModeSystem: 恢复 Sprinting
✅ 全程无缝
```

### 场景 B: NPC 被攻击 → 防御 → 被击飞 → 着地 → 逃跑

```
帧 0:  MovementState { Walking, Standing }; ActionController: idle
帧 1:  被攻击命中 → CombatHitEvent(Hit)
帧 2:  GOAP: 战斗模式 → ActionRequest { Block }
帧 3:  ActionController: accept Block → Windup(50ms) → Active(∞)
       MovementLock: Partial(1.0m/s); 体力消耗 3.0/s
帧 8:  被重击 → caused_knockback=true; ImpulseQueue.push
帧 9:  ImpulseSystem drain → loco=PhysicsBody → Airborne(KnockedBack)
       ActionController: Interrupted(Block, PhysicsKnockback)
       MovementRecovery: push { Standing, Walking }
帧 15: GOAP: 逃跑 → CGoapIntent { desired_movement: Sprinting }
       但 MovementModeSystem: PhysicsBody → 不应用
帧 25: 着地 → loco=Grounded
       MovementRecovery: pop { Standing, Walking }
       MovementModeSystem: 现在可以应用 desired_movement → Sprinting
帧 26: NPC 以最高速度逃离
✅ 全程无缝
```

### 场景 C: DomainDelegated { Combat: Manual, Movement: Auto }

```
帧 0:  玩家在战斗中，GOAP 写 MoveIntent (向敌人奔跑)
帧 1:  MovementModeSystem: GOAP → MovementState { Running }
       玩家按左键 → ActionResolver: Combat域=Manual → ActionRequest { LightAttack }
       ActionController: accept LightAttack → MovementLock: Full
       MovementSystem: 忽略 Running → 减速到零 → 攻击根运动生效
帧 5:  Recovery 完成 → MovementLock: Free → 恢复 Reading MovementState { Running }
帧 6:  GOAP 想 Dodge → GoapIntentDispatchSystem: Combat∈manual_domains → 不发
       敌人来袭 → InstinctLayerSystem: 不受 ControlMode 影响 → ActionRequest { Dodge }
       ActionController: source=Instinct > source=Player → accept Dodge
✅ 本能层始终自动，不被 DomainDelegated 抑制
```

### 场景 D: 玩家 AimBow 充能被敌人箭矢打断

```
帧 0:  AimBow Active(∞), elapsed=600ms, SustainPhase: Normal
帧 1:  被箭矢命中 → CombatHitEvent(Hit, caused_knockback: false, damage: 15)
       StaggerSystem: damage > stagger_threshold → ActionRequest { Stagger, priority=35, source=System }
帧 2:  ActionController: Stagger(35) > AimBow(12) + source=System → accept
       → Interrupted(AimBow, Staggered, progress=0.45)
       → Stagger 开始 → MovementLock: Full
帧 3:  玩家仍在按左键 → ActionResolver 继续发 AimBow
       ActionController: 在 Stagger 中(commitment=Hard) → 拒绝 → ActionRejectedEvent
帧 5:  Stagger 结束 → ActionController 空闲
帧 6:  新 AimBow → accept
✅ 系统中断优先级 > 玩家意图
```

## 八、明确排除

| 排除项 | 理由 |
|--------|------|
| **通用刚体引擎** | WoWorld 只需要空间查询 + COM 积分——不需要刚体求解器 |
| **N 体动量守恒求解** | 飞行中离散接触为非递归（深度上限 1）——详见 CHG-067 §五 |
| **真实流体/摩擦μ滑行/热胀冷缩** | 超出游戏物理所需的保真度 |
