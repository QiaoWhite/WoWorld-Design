## 一、热/冷分离——与 ComponentInstance 一致的架构

```rust
/// 热数据 — 每帧空间查询走这里
/// 32 bytes, Copy type — 一个 L1 cache line 装两个
#[derive(Copy, Clone)]
pub struct PlacedEntry {
    pub item_id: u64,              // 8 bytes — ItemEntId 的 raw u64
    pub pos_x: f32,                // 12 bytes — 世界空间中心 (x,y,z)
    pub pos_y: f32,
    pub pos_z: f32,
    pub yaw: u8,                   // 1 byte — 绕表面法线的旋转 (0-255 → 0-2π)
    pub normal_oct: u8,            // 1 byte — 表面法线 octahedral 编码
    pub surface_ref: u64,          // 8 bytes — SurfaceId compressed (含 FaceDir)
    pub flags: u8,                 // 1 byte — 位域（见下）
    pub bounding_radius_cm: u8,    // 1 byte — 包围半径 0-2.55m, 精度 1cm
}
// 总计: 8 + 12 + 1 + 1 + 8 + 1 + 1 = 32 bytes ✓

// flags layout:
//   bit 0: has_container     — 此物品放置后创建了 Container
//   bit 1: provides_surface  — 此物品提供至少一个可放置面(FurnitureSurface)
//   bit 2: is_attached       — 挂墙/挂天花板 → GRASP 需要先 DETACH
//   bit 3: is_broken         — 已被 BREAK 破坏，仅保留碎片视觉
//   bit 4: is_stackable      — 松散堆叠的材料（黏土堆/矿堆）
//   bits 5-7: reserved
```

```rust
/// 冷数据 — 低频查询（历史、所有权），不在渲染路径上
/// 32 bytes
#[derive(Clone)]
pub struct PlacedCold {
    pub placed_at: GameTime,                // 8 bytes
    pub placed_by: Option<EntityId>,        // 12 bytes (u64 + Option tag + padding)
    pub original_owner: Option<EntityId>,   // 12 bytes — "最初为谁制造"
    pub conjured_ttl: Option<GameDuration>, // 4 bytes — 召唤系限时家具
}
// 总计: 8 + 12 + 12 + 4 = 36 → 但对齐到 8 → 40 bytes
// 优化: 如果 conjured_ttl 和 original_owner 合并到 packed Option → 24 bytes
// 但冷数据不是性能瓶颈——保持清晰 > 节省 8 bytes
```

**与 ComponentInstance 的热冷分离模式完全一致**（见 [[../../建筑模块/002-建筑数据模型|建筑 002 §一-B]]）。

## 二、PlacementChunk — 分块存储

```rust
pub type ChunkKey = (i32, i32);  // chunk_x, chunk_z — 与体素 32m chunk 对齐

/// 只有包含放置物品的 chunk 才分配此结构
pub struct PlacementChunk {
    /// 热数据 — 按 item_id 排序（binary_search）
    hot: RwLock<Vec<PlacedEntry>>,
    
    /// 冷数据 — 同索引对应 hot
    cold: RwLock<Vec<PlacedCold>>,
    
    /// 派生表面数据 — 从 hot 中 provides_surface=true 的条目 + ItemPlacementProps 派生
    surfaces: RwLock<Vec<FurnitureSurfaceEntry>>,
    
    /// 脏标记 — 序列化时检查
    dirty: AtomicBool,
}

/// 家具提供的表面 — 纯派生（非持久化）
struct FurnitureSurfaceEntry {
    item_id: u64,                  // ItemEntId
    face: FaceDir,
    area: (f32, f32),
    load_bearing_kpa: f32,
}
```

**为什么不是 octree**：放置物品的 chunk 稀疏度 ~12.5%（3万/24万 chunk）。`HashMap<ChunkKey, Arc<PlacementChunk>>` 比 octree 简单，查询速度相同。chunk 内只有 ~65 件物品——线性扫描 > 树遍历。

**hot 按 item_id 排序**：插入 O(log n) 查找 + O(n) 平移。n=65 → O(n) shift < 200ns。不值得维护更复杂的数据结构。

**surfaces 重建**：chunk 加载时扫描 hot 中 provides_surface=true 的条目，查 `ItemRegistry` 获取 `FurnitureProvidedSurface`。~15 次 HashMap 查找，< 5μs。

## 三、PlacementStore — 主存储

```rust
pub struct PlacementStore {
    /// 分块表 — Arc 允许读路径共享，写路径 clone + replace
    chunk_table: RwLock<HashMap<ChunkKey, Arc<PlacementChunk>>>,
    
    /// ItemEntId → ContainerId (仅 has_container=true 的物品，~5%)
    /// 独立存储——节省 8 bytes × 95% 非容器物品的 hot 空间
    container_map: RwLock<HashMap<u64, ContainerId>>,
    
    /// ItemEntId → ChunkKey 辅助索引 — 启动时重建，非持久化
    item_chunk_index: RwLock<HashMap<u64, ChunkKey>>,
}
```

### 3.1 EntityIndex trait 实现

```rust
impl EntityIndex for PlacementStore {
    fn entities_in_radius(&self, center: WorldCoord, radius: f32) -> Vec<EntityId> {
        let chunks = self.covered_chunks(center, radius);
        let table = self.chunk_table.read();
        let mut result = Vec::with_capacity(64);
        
        // 每个目标 chunk
        for ck in chunks {
            if let Some(chunk) = table.get(&ck) {
                let hot = chunk.hot.read();
                // 级联 chunk 级剔除
                if !chunk_intersects_sphere(&ck, center, radius) { continue; }
                
                for entry in hot.iter() {
                    // 球-AABB 相交测试
                    let dist_sq = center.distance_squared_to(entry.pos());
                    let r_total = radius + entry.bounding_radius_cm as f32 * 0.01;
                    if dist_sq <= r_total * r_total {
                        result.push(EntityId::from_item(ItemEntId(entry.item_id)));
                    }
                }
            }
        }
        result
    }
    
    fn affordances_of(&self, id: EntityId) -> AffordanceSet {
        // 从 PlacedEntry → ItemDefId → ItemPlacementProps.affordances
        // 热路径——可缓存到 per-chunk 的简易 HashMap<u64, AffordanceSet>
        // 但 < 5ns 不需要——直接查 ItemRegistry
        let item_id = id.as_item_id()?;
        ITEM_REGISTRY.get_placement_props(item_id)
            .map(|p| p.affordances)
            .unwrap_or(AffordanceSet::empty())
    }
    
    fn material_of(&self, id: EntityId) -> Option<MaterialProperties> {
        let item_id = id.as_item_id()?;
        let item_def = ITEM_REGISTRY.get_def(item_id)?;
        let mat_def_id = item_def.material_instance?;
        MATERIAL_DEF_REGISTRY.get(mat_def_id)
    }
    
    fn graspability(&self, id: EntityId) -> Graspability {
        let props = ITEM_REGISTRY.get_placement_props(id.as_item_id()?)?;
        if props.is_fixed {
            Graspability::Immovable
        } else if props.is_attached {
            Graspability::Attached { 
                mass_kg: props.mass_kg, 
                detachment_force_n: props.attachment_force_required() 
            }
        } else {
            Graspability::Free { mass_kg: props.mass_kg }
        }
    }
    
    fn try_acquire(&self, id: EntityId, acquirer: EntityId) -> Result<AcquireGuard, AcquireError> {
        match self.graspability(id) {
            Graspability::Immovable => Err(AcquireError::Immovable),
            Graspability::Attached { detachment_force_n, .. } => {
                Err(AcquireError::NeedsDetach { force_required_n: detachment_force_n })
            }
            Graspability::Free { .. } => {
                // 获取成功 → 暂时从 EntityIndex 的可见集合隐藏
                Ok(AcquireGuard::new(id, acquirer))
            }
        }
    }
    
    // ... 其他方法 ...
}
```

## 四、WorldEntityIndex — 启动时胶水代码组装

```rust
pub struct WorldEntityIndex {
    placement: Arc<PlacementStore>,
    building: Arc<BuildingStore>,
    creatures: Arc<CreatureStore>,
    terrain: Arc<TerrainStore>,
}

impl EntityIndex for WorldEntityIndex {
    fn entities_in_radius(&self, center: WorldCoord, radius: f32) -> Vec<EntityId> {
        let mut result = Vec::new();
        result.extend(self.placement.entities_in_radius(center, radius));
        result.extend(self.building.entities_in_radius(center, radius));
        result.extend(self.creatures.entities_in_radius(center, radius));
        result.extend(self.terrain.entities_in_radius(center, radius));
        result
    }
    
    fn transform_of(&self, id: EntityId) -> Option<Transform> {
        if id.is_item() {
            self.placement.transform_of(id)
        } else {
            match id.entity_kind()? {
                EntityKind::Creature => self.creatures.transform_of(id),
                EntityKind::BuildingComponent => self.building.transform_of(id),
                EntityKind::TerrainFeature => self.terrain.transform_of(id),
                EntityKind::DroppedItem => self.placement.dropped_item_transform_of(id),
                _ => None,
            }
        }
    }
    
    fn affordances_of(&self, id: EntityId) -> AffordanceSet {
        if id.is_item() {
            self.placement.affordances_of(id)
        } else if id.entity_kind() == Some(EntityKind::BuildingComponent) {
            self.building.affordances_of(id)
        } else {
            AffordanceSet::empty()
        }
    }
    
    // ... 其他方法 ...
}
```

**这是编译期静态分发**——每个方法只做 2-4 次 `Arc` 解引用 + 内联调用。开销 ≤ 一次 L3 cache miss。物理原子管道看到的是 `&dyn EntityIndex`。

## 五、序列化 — LMDB

### 5.1 键值格式

```
LMDB key: [b'P'][b'L'][chunk_x: i32 BE][chunk_z: i32 BE]  — 10 bytes
LMDB value: bincode(ChunkPlacementBlob)
```

```rust
#[derive(Serialize, Deserialize)]
struct ChunkPlacementBlob {
    version: u16,                     // 格式版本
    items: Vec<PackedPlacedEntry>,    // 只存 hot
    cold: Vec<PackedPlacedCold>,      // 只存 cold
}

#[derive(Serialize, Deserialize)]
struct PackedPlacedEntry {
    item_id: u64,
    pos_x: f32, pos_y: f32, pos_z: f32,
    yaw: u8, normal_oct: u8,
    surface_ref: u64,
    flags: u8,
    bounding_radius_cm: u8,
}
// 32 bytes packed — 与内存布局一致，bincode 零拷贝

#[derive(Serialize, Deserialize)]
struct PackedPlacedCold {
    placed_at: GameTime,              // 8 bytes
    placed_by_entity: u64,            // 0 = None
    original_owner_entity: u64,       // 0 = None
    conjured_ttl_seconds: u32,        // 0 = None
}
// 32 bytes packed

// per-item serialized: 32 + 32 = 64 bytes
// 2M items × 64 = 128 MB LMDB — 在 140MB 存档预算内
```

### 5.2 container_map 单独序列化

```
LMDB key: [b'P'][b'L'][b'C'][b'M']  — 4 bytes (constant)
LMDB value: bincode(HashMap<u64, ContainerId>)
```

## 六、启动与流式加载

```
启动:
1. LMDB 扫描 [b'P'][b'L']* key → 构建 chunk_inventory:
   HashMap<ChunkKey, (item_count, lmdb_offset)>
2. 确定玩家出生点
3. 加载 birth_point ± 3×3 chunks → 反序列化 → 注入 PlacementStore
4. 重建 item_chunk_index（扫描已加载 chunk 的 hot）
5. 重建所有已加载 chunk 的 surfaces

运行时:
- ChunkStreamer 检测新 chunk 进入加载半径
- 后台线程: LMDB read → bincode deserialize → Arc::new(PlacementChunk)
  → chunk_table.write().insert(chunk_key, arc)
- 离开半径的 chunk: 
  if dirty → bincode serialize → LMDB write → chunk_table.write().remove(chunk_key)
- 每 30s 批量 flush dirty chunks
```

## 七、掉落物 (DroppedItem) — 轻量空间记录

与 PlacedItem 不同——掉落物无摆放语义，只有位置+速度+存在时间：

```rust
/// 掉落物 — 物理刚体，不在 PlacementStore 中
/// 但注册在 EntityIndex (namespace=DroppedItem)
pub struct DroppedItemRecord {
    pub item_id: u64,
    pub pos: [f32; 3],
    pub velocity: [f32; 3],
    pub dropped_at: GameTime,
    pub lifetime_remaining: Option<GameDuration>,  // 自然消失时间
}
// 32 bytes — 存入独立的 DroppedItemStore
// 掉落物 < 1000 件（被拾取/自然消失后清理）→ < 32KB 内存
```

**PlacedItem vs DroppedItem 的判定**: `ItemRegistry::get_placement_props(def_id).is_some()` → 可摆放，走 PlacementStore。`.is_none()` → 只能作为掉落物存在。

## 八、性能核算终表

| 项 | 值 | 备注 |
|---|-----|------|
| PlacedEntry | 32 bytes | Copy type |
