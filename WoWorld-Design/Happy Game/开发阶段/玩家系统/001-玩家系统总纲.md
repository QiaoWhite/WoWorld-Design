## 〇、模块定位

玩家系统是**独立一级模块**，但它不定义新实体类型。WoWorld 的核心哲学是**玩家 = NPC + I/O 适配层**——玩家操控的角色和 NPC 共享同一套 BodyState、RaceTraits、部位伤害、死亡流程、GOAP 心智引擎。玩家系统的职责是定义**玩家如何与这些共享系统交互**——输入映射、信息展示边界、角色创建流程、双角色托管、成长引导、死亡继承。

```
共享层（生命/NPC/技能/魔法/经济/战斗/…）—— 玩家 & NPC 共用
        ↑                ↑
    NPC 自主          玩家 I/O 适配层 ← 本模块定义
    (GOAP+概率)      (输入/渲染/UI/信息边界)
```

**本模块不定义的内容**（属于其他模块）：
- BodyState/Vitals/RaceTraits → [[../生命/001-生命总纲|生命模块]]
- GOAP/记忆/情绪/关系/BigFive → [[../NPC活人感模块/NPC活人感开发文档ver2.0|NPC活人感]]
- 技能成长（用进废退）→ [[../技能系统/README|技能系统]]
- 魔法/战斗/交易——全部走 NPC 同一代码路径
- UI 具体布局/控件 → [[../UI与UX系统/README|UI/UX系统]]（轨B.1）

## 一、核心哲学：玩家 = NPC + 控制层

### 1.1 本体论

```rust
// Player ≠ 独立实体类型。Player = SapientMind + ControlMode。
// 玩家的 LifeEntity 和其他 NPC 完全相同。

struct LifeEntity {
    // ... 所有 NPC 共享的字段（生命/001）
    mind: MindLayer,  // SapientMind —— 与 NPC 同一结构
}

// 差异仅在此字段：
struct SapientMind {
    // ... 所有 Sapient NPC 已有的字段
    control_mode: Option<ControlMode>,  // Some = 可被玩家接管
}

enum ControlMode {
    Auto,                                 // 完全托管——GOAP 正常运转
    Manual { action_override: Action },   // 完全手操——GOAP 后台运行但不被执行
    DomainDelegated {                     // 域选择托管
        manual_domains: HashSet<ActionDomain>,
    },
}

enum ActionDomain {
    Movement,
    Combat,
    Speech,
    ItemUse,
    Interaction,
    MagicUse,
}
```

**关键性质**：
- **被控无感知**：被操控的角色不知道"被操控"。GOAP 在后台持续运转。实际动作与控制输入有差异时，角色的自我叙事倾向于**合理化**而非痛苦——"我今天去了那个地方……仔细想想，我确实想去看看。"玩家操控 = 人格塑造，不是附身对抗。
- **所有系统正常运行**：审美/认知/情感/关系不受 ControlMode 影响。玩家操控角色的移动，但角色的情感仍然来自他自己的情感引擎。
- **同 PC 同死**：玩家操控的角色和任何其他 NPC 一样——同样的 DeathCause、同样的 Gompertz 死亡率、同样的死亡后处理。

### 1.2 玩家 ≠ NPC 的唯二差异

| 维度 | NPC | 玩家 |
|------|-----|------|
| **心智驱动** | GOAP + 概率决策引擎 | 人类玩家输入（ControlMode 覆盖） |
| **物理** | Rust 空间查询四 trait | Godot PhysicsServer3D (CharacterBody3D) — 唯一保留 PhysicsServer3D 的实体 |

其余全部——BodyState、Vitals、伤害、技能、魔法、战斗 AI、交易、对话管道——走同一代码路径。详见 [[006-玩家专属I-O适配层]] 逐系统差异清单。

## 二、两种进入模式

> 来源：[[../../Change/hand/011 玩家进入游戏|hand/011 玩家进入游戏]]

### 2.1 原住民模式

玩家接管既有一个 NPC 的人生——继承其身份、家庭、工作、人际关系、财产、名声。

- 世界生成完成 → 玩家从大日志中选择一个 NPC → 接管
- 角色已经在做某事（铁匠在工作、农夫在田间、士兵在巡逻）——玩家切入的第一刻听到 NPC 的日常对话
- 角色的家庭关系、社会义务、财产、名声——**全部继承**。是否甩掉——玩家自己决定，后果由系统涌现
- 角色原本的技能等级、已知法术、语言 proficiency——全部继承

### 2.2 穿越者模式

玩家自定义全部参数——角色从前与这个世界毫不相干。

- 玩家在进入游戏前完成 7 步角色创建向导（详见 [[002-角色创建与外观]]）
- 参考日式异世界奇幻冒险的进入感——从零开始（或含金汤匙）
- 随机出生点在野外——第一个交互可能是树、兔子、石头。最近聚落可能要走两天
- 穿越者的"前世"在大日志中由玩家自己书写/编造/留白——用同一套大日志数据结构

### 2.3 两种模式的哲学统一

两种模式都兼容"玩家 = NPC"：
- **原住民**：你接管的那个 NPC 本来就活在这个世界——你只是"醒了"
- **穿越者**：世界生成时预留了"外来者"的叙事空间（漂泊者/远方来客），角色创建参数直接写入 LifeEntity + SapientMind 字段。穿越者进入后，NPC 对他的反应由系统涌现——没有特殊对待，没有"勇者降临"光环

## 四、架构速览

```
玩家系统（本模块）
│
├── 角色创建 ── 7步向导 → LifeEntity + SapientMind 字段直写
│   └── 外观动态 ── 着装可见性 / 伪装 / 身份隐藏（无头衔标签）
│
├── 双角色系统 ── 1-2 个可控角色，随时切换
│   ├── ControlMode: Auto / Manual / DomainDelegated
│   ├── 非操控角色 = Auto 模式（完全自主，和其他 NPC 一样）
│   └── 两个角色交互走 NPC-NPC 同一代码路径
│
├── 成长与目标 ── PlayerGoal → GOAP 自动展开 sub_goals
│   ├── 引导: 小精灵 + 大日志 + 菜单百科（Minecraft 式）
│   └── 技能: 用进废退，透明度由小精灵 + 大日志提供
│
├── 死亡与继承 ── 血亲继承 → 大日志选择 → 全新角色
│   └── 技能/关系不可继承（新角色是新的人）
│
└── I/O 适配层 ── 输入映射 + 信息展示边界 + 逐系统差异清单
```

## 七、设计原则

1. **玩家 = NPC**：不新建玩家专属实体类型。所有数据模型归属原有模块。
2. **涌现优先**：认知（名声/好感/信任）仅从 NPC 言行涌现，系统不向玩家暴露数值。
3. **人格塑造方向**：玩家操控 → 角色倾向合理化 → 角色逐渐与玩家意图塑造的形象趋近。不产生"对抗性认知张力"。
4. **信息诚实边界**：自身 Vitals/技能/大日志可见——他人内心数据不可见。小精灵可读大日志，但不知道 NPC 在想什么。
5. **等权选择**：单角色 = 双角色 = 完整游戏。不因少一个角色而损失内容。
6. **无主线**：无强制任务、无通关条件。目标由玩家自定，引导极简（Minecraft 式）。
