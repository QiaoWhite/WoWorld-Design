## 〇、定位

本文档定义双角色系统的**调度机制、ControlMode 模型、切换规则**。核心哲学来自 [[../../Change/hand/009 玩家角色本质|hand/009]]：玩家可以有两个可操控角色，随时切换，确保玩家总是有事做。两个角色都和 NPC 平等。

---

## 一、单角色 vs 双角色

### 1.1 等权选择

**单角色和双角色是等权选择**——不是"单角色=不完整游戏"。

- 角色创建时（或原住民模式选择时），玩家选择：单角色开局 或 双角色开局
- 游戏中途可以随时**增加第二个角色**（从大日志中选择一个 NPC）或**放弃一个角色**（该角色回到 Auto 模式，继续他的人生）
- 放弃 ≠ 杀死——他只是不再接收玩家输入，和其他 NPC 一样活着

### 1.2 双角色调度结构

```rust
struct PlayerAgent {
    active_1: EntityId,
    active_2: Option<EntityId>,       // None = 单角色
    current_control: EntityId,        // 当前活跃屏幕
    other_last_seen: Option<GameTime>, // 另一个角色上次被关注的时间
}
```

## 二、ControlMode 模型

### 2.1 三种模式

```rust
enum ControlMode {
    Auto,                                 // 完全托管——GOAP 正常运转
    Manual { action_override: Action },   // 完全手操——GOAP 后台运行但不被执行
    DomainDelegated {                     // 域选择托管
        manual_domains: HashSet<ActionDomain>,
    },
}

enum ActionDomain {
    Movement,     // 移动方向/速度、跳跃、攀爬、游泳
    Combat,       // 目标选择、攻击时机、风格切换、技能释放
    Speech,       // 对话选项选择、声量、语气
    ItemUse,      // 物品使用/装备切换/丢弃
    Interaction,  // 门/箱子/机关/采集/制作
    MagicUse,     // 法术选择与释放
}
```

- **Auto**：角色按自己的 GOAP 决策引擎行事——和任何其他 NPC 一样。玩家旁观。
- **Manual**：玩家完全接管。GOAP 在后台持续运转但不产出动作。后台 GOAP 仍产生"假设动作流"——供自我叙事使用。
- **DomainDelegated**：混合模式。勾选的域由玩家接管，其余域由角色的 GOAP 自主。例如：玩家接管 Movement 和 Combat，角色自己决定说什么话、用什么物品。

### 2.2 托管冲突不解决——冲突是涌现

玩家和角色 GOAP 之间的域冲突**不被系统解决**：

- 角色 GOAP 想逃跑，玩家手动控制往前冲 → 角色产生"我冲向了危险"的记忆
- 角色的自我叙事**倾向于合理化**，而非产生痛苦——"我今天冲向了危险……仔细想想，那是正确的选择。"

**玩家操控 = 人格塑造**。玩家持续操控角色往某个方向走，角色的 GOAP 预期和自我叙事会逐渐与玩家的意图趋近。不是"附身对抗"，而是"耳濡目染"。

### 2.3 非操控角色的行为

当玩家操控角色 A 时，角色 B 在 Auto 模式下自主行事——和任何其他 NPC 一样：
- 继续他的日常生活（工作/社交/出行）
- 遇到值得注意的事件 → 大日志自动记录（轻量感官持续运转）
- **不会主动"通知"玩家**——玩家想了解 B 的近况，需要切换过去或查阅大日志

**切换瞬间**，系统显示角色 B 的**状态摘要**：
- 当前位置（聚落名/地形描述）
- 当前活动（"正在酒馆喝酒"）
- 最近一条关键事件（"3 小时前与老战友重逢"）

这不是推送通知——是切换时的上下文帮助。系统不主动打断玩家。

## 三、切换机制

### 3.1 基本规则

| 项目 | 规格 |
|------|------|
| 切换触发 | 玩家按下 CharacterSwitch 键 → 选择目标角色 |
| 冷却 | **无**——随时切换 |
| 距离上限 | **无**——两个角色可分布于世界任意位置 |
| 同时操控 | **否**——同一时刻只有一个角色接收玩家输入 |
| 切换后 | 旧角色回到之前的 ControlMode（Auto 或 保留 DomainDelegated），新角色切换到 Manual |
| 远处加载 | 目标角色在远处 → 加载目标区域 → **游戏暂停** → 加载完摄像机切过去 |

### 3.2 任何状态下均可切换

切换**不受目标角色所处状态限制**：

- 战斗中 → **更要切换**——正是生死攸关的关头，玩家应该手操化解危机（或至少看着角色死去）
- 溺水/坠落 → 同上——玩家可以尝试救人
- 睡觉/昏迷 → 切换后看到黑屏或模糊意识流，等待醒来或选择切回去

### 3.3 时间跳过

当**两个角色都处于跳时间活动**（睡觉/冥想/昏迷/长途旅行）时：

```rust
fn next_player_event(active_1: EntityId, active_2: Option<EntityId>, world: &World) -> GameTime {
    let now = world.game_time();

    let wake_1 = expected_wake_time(active_1, world);
    let wake_2 = active_2.map(|id| expected_wake_time(id, world)).unwrap_or(wake_1);
    let nearest_wake = wake_1.min(wake_2);

    let entities = if let Some(id) = active_2 { vec![active_1, id] } else { vec![active_1] };
    let external = world.next_significant_event_for(&entities);
    // significant_event: 战斗开始/火灾/有人来敲门/婴儿哭了/...

    nearest_wake.min(external)
}
// 时间跳到 nearest_wake_or_event
// 跳过后，如果有外部事件 → 通知玩家
// 否则，切换到第一个醒的角色
```

单角色模式下同理——只是没有第二个角色的 wake time。

## 四、两个角色之间的交互

### 4.1 走 NPC-NPC 同一代码路径

两个可操控角色之间的对话/交易/协作/冲突——走**完全相同的 NPC-NPC 交互代码路径**：

| 交互类型 | 机制 |
|----------|------|
| **对话** | A 发起对话（玩家输入）→ B 的 SapientMind（Auto 模式）自主生成回应 → A 看到 B 的回应 → 玩家选择下一句。**不会自言自语**——B 是独立人格 |
| **交易** | A 发起交易 → B 的 GOAP 自主决定接不接受、讨价还价 |
| **合作战斗** | A 和 B 在同一场战斗中 → B 的 AI 自主战斗（Auto）——玩家只能直接控制 A |
| **冲突/攻击** | A 攻击 B → B 的反应由 B 的大五人格 + 情绪 + 关系涌现——可能逃跑/反击/质问 |

### 4.2 关系涌现

两个角色之间有一条 `Relationship`（和所有 NPC 间关系同一数据结构）：
- affection / trust / familiarity / attraction —— 全是这两个角色互动中涌现出来的
- 角色 A 对角色 B 的感情**不是玩家手动设定的**——是相处时间、对话质量、共同经历的结果
- 玩家可以试图让 A 喜欢 B——但 B 喜不喜欢 A 不是玩家决定的，是 B 的大五人格 + 情绪 + 关系涌现的

### 4.3 结伴与分开

- **一起活动**：两个角色在物理空间中结伴。未被操控的那个跟随 + 自主行动（Auto 或 DomainDelegated）
- **分开活动**：一个在北山屠龙、一个在苍山打铁。切换——你看到的已经是另一个人的生活流

## 五、大日志共享

两个角色共享同一本大日志——像共享的外置大脑：

- 角色 A 在北地发现霜铃兰 → 大日志自动记录 → 角色 B 在南海立刻可见此条目
- 时间线混排：角色 A 的条目和角色 B 的条目用颜色区分来源
- 大日志不随个体死亡消亡——旧条目永久保留（除非玩家手动清除）
