## 〇、定位

本文档定义**玩家视角相机**——玩家看世界的那只"眼睛"，也是驱动 LOD/剔除/音频衰减的**视点权威**。它是**玩家专属**表现层：NPC 使用同一套角色控制器（[[../模型动作与物理系统/角色控制器/001-角色控制器总纲|角色控制器]]）但**不拥有相机**（[[006-玩家专属I-O适配层]] §1.4）。

相机是**三向枢纽**：

- **消费**：跟随被控实体 ECS `Position`；把鼠标/右摇杆转 yaw/pitch；缩放、FP/TP 切换、地形遮挡碰撞。
- **反馈（移动/朝向）**：相机 yaw 作为 `camera_transform` 回喂 [[006-玩家专属I-O适配层]] §2.2 `MoveDirection` 与 [[../模型动作与物理系统/角色控制器/002-MovementState与连续移动|角色控制器 002]] §十一 `RotationLock::CameraForward`。
- **反馈（视点）**：产出 `CameraState`（位置/朝向/FOV）供 `LODCoordinator`、视锥剔除、音频 LOD 消费（[[../技术栈方案/001-WoWorld正式技术栈方案v3|技术栈方案]] §21.3）。

**一句话原则**：相机跟随的是**被控实体的 ECS Position（Rust 权威值）**。走动、飞行、夺舍、换角色——只要"谁被控制"改变，相机换目标即可，无特判。这是"玩家=NPC"在视角层的直接体现。

## 一、设计范式与决策

### 1.1 核心范式：自由环绕 + 角色朝移动方向

现代开放世界标准（WoW / 原神 / 塞尔达）：鼠标独立驱动相机 pivot yaw/pitch 环绕角色；角色朝向由移动方向决定（`RotationLock::InputDirection`，ECS 写 `Rotation`）；WASD 相对相机 yaw；战斗/防御/瞄准切 `RotationLock::CameraForward`（角色背对相机、可 strafe），对齐 [[../模型动作与物理系统/角色控制器/002-MovementState与连续移动|角色控制器 002]] §十一。

> **UX 取舍（无倒退）**：自由环绕下 `InputDirection` 意味着按 S 角色**转身走开**（无 backpedal），符合 WoW/原神惯例。精确后退/横移由 `CameraForward`（战斗/瞄准态）提供。此为刻意设计，见 §六。

### 1.2 决策速查表

| 决策点 | 结论 | 主导考量 |
|--------|------|----------|
| 相机范式 | 自由环绕 + 角色朝移动方向 | 手感/涌现/可达性 |
| 化身渲染 | 走 `entity_renderer` 统一路径（玩家=NPC 单路径） | 解耦/涌现/可维护 |
| 相机 rig 归属 | 独立顶层节点，读被控实体 ECS Position | 解耦/Godot 惯例 |
| 角色转向 | 落地 `RotationLock`，复用 008 `turn_smoothing` | 模块协作/一致性 |
| **相机碰撞** | Rust `TerrainQuery::terrain_raycast` 夹紧臂长 | 引擎无关/性能/CHG-033 |
| **视点契约** | 产出 `CameraState` 供 LODCoordinator | 模块协作（关键贯通） |
| 缩放/FP 切换 | 臂长连续缩放；隐藏阈值防相机入体 | 可达性 |
| `camera_transform` 来源 | CameraRig yaw basis（**不再是 body**） | 模块协作（载重修复） |
| 相机手感 | MVP：落地下沉 + 疾跑 FOV；007 自持 | 模块特色/记忆点 |
| 相机输入 | 暂用原始键；`InputAction` 登记变体、接线延后 InputMap 冲刺 | 与既有迁移债一致 |

## 二、架构总览（架构 P）

```
Main (Node3D)
 ├─ WorldDriver ............... Rust ECS 权威；Phase 2 驱动相机跟随+碰撞+发布 CameraState
 ├─ InputBridge .............. 输入采集转发（priority -100）
 ├─ Entities/ ................ entity_renderer 容器
 │    ├─ 玩家化身胶囊 ......... controlled=true → 无头顶名字/气泡；近距/FP 隐藏
 │    └─ NPC 胶囊 × N ......... 同一渲染路径
 ├─ Player (CharacterBody3D) . 保留：G 飞行 + 碰撞盒 + 夺舍瞬移锚点 + legacy 物理；**无 mesh、无相机子节点**
 └─ CameraRig (Node3D) ....... ★ 独立相机 rig（priority -200，先于 InputBridge 更新 yaw）
      └─ PitchArm (Node3D)
           └─ Camera3D
```

### 2.1 为什么相机 rig 独立、且读 ECS Position

1. **一条规则覆盖所有模式**：`CameraRig.position = 被控实体 ECS Position + 眼高`。走动（Block A0）、飞行（节点→ECS 回灌）、夺舍（`player_ecs_entity`=NPC，`sync_possessed_position` 节点→ECS）——ECS Position 始终最新。
2. **对未决物理问题免疫**："玩家最终用 Godot 物理还是 Rust 驱动"（[[006-玩家专属I-O适配层]] §1.1）是独立张力；相机只依赖 ECS Position，不碰 `CharacterBody3D` 物理。
3. **社区惯例**：相机跟随"目标 transform"，独立于会做动画/root-motion/LOD 换模的 mesh（Cinemachine/GTA/Genshin 皆然）。
4. **天然支持后续**：死亡留机（[[005-死亡继承与衰老]]）、夺舍换目标过场（[[003-双角色与托管模式]] / [[../模型动作与物理系统/角色控制器/012-夺舍过渡|CC 012]]）= 相机目标解绑/切换。

## 四、相机跟随、目标与视点契约

### 4.1 跟随目标

```
target_pos = 被控实体 ECS Position + Vec3(0, PIVOT_HEIGHT, 0)
```

- 被控实体 = `WorldDriver.player_ecs_entity`（夺舍时自动指向 NPC）。
- **不要用 `get_player_position()` 作跟随源**：它读 `Player` 节点（多一跳同步、存在相位陈旧风险）。新增 `WorldDriver.get_camera_target() -> Vector3` 直接读 `player_ecs_entity` 的 ECS `Position`（夺舍/裸玩家两模式均正确）。
- `PIVOT_HEIGHT` 默认 **1.5 m**；未来按物种/体型从化身参数派生（[[002-角色创建与外观]]）。

### 4.2 平滑

- 位置跟随用 **SmoothDamp**（§十四 smooth_damp 助手），`follow_smooth_time ≈ 0.08 s`。
- ⚠️ 抖动约束：ECS 在 Godot 帧循环内步进；相机跟随在 ECS 步进**之后**读取 Position（§五.3），同源同帧，无 physics/render 插值错位。若后续玩家改 `CharacterBody3D` 物理（`_physics_process` 频率≠帧率），需启用 Godot 4 physics interpolation 或移入 `_physics_process`。
- **GD-002 契约**：`camera_pos` uniform 取 `get_viewport().get_camera_3d().get_global_position()`（真实相机世界位）。缩放/碰撞拉近时自动跟随。见 bugs GD-002。
- **大世界精度**：LOD 视点用 f64（既有 `CameraState.position: DVec3`，§四.3）。相机跟随与偏移计算以 target 为原点做**局部位移**（非两大坐标相减），发布 CameraState 时转 f64 世界坐标。渲染侧浮点原点由 shader（camera-relative）单独处理，与本层解耦。

### 4.4 ★ 目标不连续 → SNAP（不 SmoothDamp）

夺舍 / 换角色 / 瞬移 / 重生时 `get_camera_target()` 会跨千米跳变，SmoothDamp 会把相机拖出长 smear。规则：

- WorldDriver 检测**被控实体变更**（`player_ecs_entity` 变化）或**单帧目标位移 > SNAP_THRESHOLD**（默认 5 m）→ 当帧**吸附**（跳过位置平滑，重置 SmoothDamp 速度）。
- 缩放/yaw/pitch 平滑不受影响（它们不跨越世界）。
- 死亡留机（§十三）是反例——那里刻意**解绑**目标，相机保持原位而非 snap，属独立分支。

### 4.3 ★ CameraState 视点契约（关键贯通）

`LodCoordinator::prescribe`（[[../技术栈方案/001-WoWorld正式技术栈方案v3|技术栈方案]] §21.3）**已实现且已按帧消费** `CameraState`（`lod.rs` 第 404 行 `entity.position - input.camera.position`、第 460 行 `camera.forward.dot(...)` 视线锥）。相机是 **CameraState 生产者**。

**复用既有类型，不新造**（`woworld_core::lod`）：

```rust
pub struct CameraState { pub position: DVec3, pub forward: DVec3, pub fov_radians: f32 }  // f64 世界坐标 + 弧度
```

- **LOD/视锥剔除/音频衰减的视点 = 相机位置**（第三人称后移最多 8m），**非角色位置**。碰撞夹紧后的真实相机位与朝向填入 `CameraState`，经 `WorldDriver.publish_camera_state(...)` 供 `prescribe` 消费。
- **`PlayerAttention` 是独立结构，相机不拥有它**（`lod.rs:156`，`LodInput` 里 `camera` 与 `attention` 分列）。相机可为 `PlayerAttention.聚焦方向` **提供数据源**（相机 forward），但注意力语义（注视目标/聚焦听觉）归感官/注意力模块。
- ⚠️ **实现期核验（不臆测现状）**：`prescribe` 已存在、其调用是否已接入 Godot 每帧循环、当前喂入的 `CameraState` 是 stub 还是真实值——须在编码时确认，并接真实相机。若尚未按帧调用，则本模块产出即为其首个真实输入源。
- 当前实时消费方 `entity_renderer.set_player_pos`（标签距离裁剪）可暂保持角色位（"离我近"语义），LodCoordinator 接入后按 CameraState 统一。

## 五、旋转模型、`camera_transform` 来源与更新定序

### 5.1 yaw / pitch

| 量 | 载体 | 输入 | clamp |
|----|------|------|-------|
| yaw | `CameraRig.rotation.y` | 鼠标 X/右摇杆 X × 灵敏度 | 归一化 [-π, π] |
| pitch | `PitchArm.rotation.x` | 鼠标 Y/右摇杆 Y ×（可反转） | TP ±80°；FP ±89° |

- 灵敏度默认 `0.003 rad/px`；X/Y 独立；`invert_y`（默认关）。归设置菜单（[[../UI与UX系统/004-面板与管理系统|UI/UX 004]]）。

### 5.2 ★ 载重修复：`camera_transform` 改传 CameraRig yaw

```gdscript
# input_bridge.gd（改动）
var rig = get_node_or_null("../CameraRig")
if rig != null:
    driver.input_set_camera_transform(rig.global_transform)   # 原为 player.global_transform
```

`player_input_system`（[[../模型动作与物理系统/角色控制器/004-ActionResolver与输入解析|CC 004]]）用 `camera_transform.x_axis`(右)/`-z_axis`(前) 转 WASD——**逻辑零改动，只换来源**。`RotationLock::CameraForward` 消费同一 `camera_transform`，单一来源无二义。

### 5.3 ★ 两相更新定序（解决帧内一致性）

yaw 是**输入派生**（须早，供移动方向同帧使用）；跟随位置是**模拟派生**（须晚，ECS 步进后）。二者分两相：

```
① CameraRig._process(priority = -200)  ── 早于 InputBridge(-100)
     累积鼠标 delta → 设 yaw(rig) + pitch(PitchArm)
     ⇒ camera_transform 就绪，供 InputBridge 同帧转发（WASD 方向不差帧）

② InputBridge._process(priority = -100)
     读 CameraRig.global_transform → input_set_camera_transform（新鲜）

③ WorldDriver._process(priority = 0，ECS 步进) → Phase 2（步进后）
     CameraRig.position = SmoothDamp(→ get_camera_target()+眼高)
     arm = camera_resolve_arm(...)（碰撞）
     应用缩放/FP/手感偏移 → 设 Camera3D 局部 -Z
     publish_camera_state(pos, forward, fov)
```

跟随位置差一帧无感（平滑本引入滞后）；移动方向差帧被避免。Godot 定序规则："process_priority 越小越先执行"。

## 六、角色转向（落地 RotationLock）

### 6.1 现状缺口

`RotationLock` 五态已定义（[[../模型动作与物理系统/角色控制器/002-MovementState与连续移动|CC 002]] §十一），`action_system` 已从 action 映射写入 `CMovementControl.rotation_lock`，但 `movement_system.rs` 的 `_apply_rotation_lock` **是空 stub**。本模块首次执行它。

### 6.2 `character_facing_system`（Rust/ECS，Phase 1，位于 `player_input_system` 之后、`entity_visual` 之前）

```
求目标朝向 target_dir(XZ)：
  按 CMovementControl.rotation_lock：
    InputDirection  → CMoveIntent.direction
    CameraForward   → camera_transform 的 -Z 投影 XZ
    TargetDirection → (locked_target.pos - self.pos) XZ   （预留，§十三）
    Locked          → 保持
    Free            → ★ 若为玩家实体 + Manual 移动域 + 有移动输入 → 按 InputDirection；
                       否则保持当前朝向
若 target_dir 非零：
    target_quat = Quat::from_rotation_arc(Vec3::Z, target_dir.normalize())
    Rotation.0  = smooth_damp_quat(Rotation.0, target_quat, feel.turn_smoothing.smooth_time)
```

- **复用既有参数**：转向平滑用 `input_feel.toml` 的 `[feel.turn_smoothing] smooth_time = 0.1`（[[../模型动作与物理系统/角色控制器/008-手感系统|CC 008]] §一），**不新增常数**。用 `glam::Quat::slerp` 实现 SmoothDamp 语义（§十四 smooth_damp_quat）。
- **`Free` 当 `InputDirection` 的规则**：因 `action_system` 在动作结束会把 `rotation_lock` 复位为 `Free`（`action_system.rs:146`），休息态恒为 Free。故 facing_system 对玩家休息态按 InputDirection 处理，实现自由环绕转向。
  > ⚠️ 实现期须核验 `action_system` 的 rotation_lock 复位时机，确认不与 facing_system 规则冲突。
- 结果写 `Rotation` → `entity_visual_system` 采集 → `entity_renderer` 应用化身朝向。NPC 未来复用同一系统（无相机时 CameraForward 退化）。

## 七、缩放与第一/第三人称切换

### 7.1 缩放（CameraZoom）

- `arm_length` 连续缩放，滚轮 ±；范围 **[0, 8] m**，默认 **4 m**，步进 ≈ 0.5 m/格。
- 缩放平滑：SmoothDamp，`zoom_smooth_time ≈ 0.12 s`。

### 7.2 FP/TP 切换与 ★ 隐藏阈值几何

胶囊半径 0.4 m、相机近裁剪 0.2 m。若相机-角色中心距 < 约 0.9 m，相机进入胶囊内部。故**化身隐藏阈值须 > 碰撞近界**：

| 状态 | arm_length | pitch clamp | 化身可见 | 触发 |
|------|-----------|-------------|----------|------|
| 第三人称 | 1.0–8 m | ±80° | 显 | 默认 |
| **近距过渡** | 0.4–1.0 m | ±80° | **隐**（防相机入体） | 缩放进入 |
| 第一人称 | 0（相机置眼高） | ±89° | **隐** | `FirstPersonToggle` 或 arm<0.4 自动吸附 |

- 化身隐藏阈值 = `capsule_radius + near + margin ≈ 0.9 m`，取 **1.0 m**。MVP 硬隐；抖动淡出（dither 淡入 [1.2→0.6]m）作 §十三 预留。
- 显隐经 `EntityVisual.controlled` + FP/近距标记，由 `entity_renderer` 执行，**不在 GDScript hack mesh**。
- 退出 FP 恢复上次 TP 臂长（缓存）。

## 八、相机碰撞（Rust 地形射线）

### 8.1 决定性 Godot 真相

**地形没有 Godot 碰撞体**（玩家贴地走 `WorldDriver.query_height()` 后手动钳 y，CHG-033"仅玩家保留 PhysicsServer3D"）。因此 `SpringArm3D` 对地形**无物可撞**——用它会穿地。

### 8.2 方案：`TerrainQuery::terrain_raycast` 夹紧臂长

```rust
// woworld_core（纯函数，可单测；TerrainHit.distance 已存在于 types.rs）
pub fn resolve_camera_arm(
    pivot: WorldPos, dir: Vec3, desired: f32,
    terrain: &dyn TerrainQuery, margin: f32,
) -> f32 {
    match terrain.terrain_raycast(pivot, dir, desired) {
        Some(hit) => (hit.distance - margin).clamp(MIN_ARM, desired),
        None      => desired,
    }
}
```

- `dir` = 从 pivot 指向"无碰撞时相机应在位置"。`margin` 默认 **0.3 m**，`MIN_ARM ≈ 0.3 m`。
- 调用：CameraRig 算 `dir`+`desired` → `WorldDriver.camera_resolve_arm(...)`（新 `#[func]`）→ 设 Camera3D 局部 -Z。
- **优点**：引擎无关、复用现有 trait、无需地形碰撞体（守 CHG-033）、纯函数可测、每帧一次射线（DDA ~16 步）成本可忽略。
- **3D 洞穴/悬崖**：`terrain_raycast` 走密度场（3D），自然处理洞穴顶/悬崖遮挡，非纯高度场。

### 8.3 后续增强（预留）

建筑/大型放置物若带 Godot 碰撞体，叠加 `SpringArm3D` 或 Godot raycast 对**这些几何**求距，与地形射线取 `min`。

### 8.4 实体/植被遮挡（预留，§十三）

角色被 NPC/树/建筑挡住时的处理（遮挡物 dither 淡出，Cinemachine 风格）不在 MVP。MVP 仅地形射线夹紧。

## 九、可见化身（走 entity_renderer 统一路径）

### 9.1 改动

1. `entity_visual_system` **语义从"排除玩家"改为"标记玩家"**：删除 `player_entity == Some(entity)` 的 `continue`，改为对该实体置 `controlled = true`。玩家 CC 实体已含 `Position`+`EntityKind::Creature`+`LodLevel`，去排除后即被采集。
   > ⚠️ **测试波及（O4）**：既有 `test_visual_system_excludes_player`（断言传入 player 时 `len==1`）会失效——须改为断言 `len==2` 且该实体 `controlled==true`。函数签名不变（`player_entity` 参数保留，仅语义从过滤转为打标）。
2. `EntityVisual` 增字段 `controlled: bool`（= 是否当前被控角色，`player_ecs_entity == entity` 判定）。
3. `entity_renderer`：`controlled==true` → 不显示头顶名字/气泡；FP 或近距（arm<1.0m）+ `controlled` → 隐藏胶囊 mesh（近距标记由 WorldDriver 传入）。
4. **不退休 `sync_bare_player_render`**：其 ECS→节点推送维持 `Player` 节点与 ECS 一致，`get_player_position()` 的消费方（标签距离裁剪、G 飞行起点、夺舍瞬移锚点）依赖它。相机**另行**经 `get_camera_target()` 读 ECS，二者并存。

### 9.2 涌现收益（维度 6/7）

夺舍：`player_ecs_entity` 换 NPC → 该 NPC 变 `controlled`（隐头顶名字、近距/FP 隐胶囊），相机自动跟它；原裸玩家（驻停）恢复为普通具名 NPC 渲染。**零特判**。"控制权=渲染差异"的自洽联动是本模块记忆点。

## 十一、相机手感（MVP：落地下沉 + 疾跑 FOV）

相机是"juice"最大杠杆，也是第三人称最鲜明的体感记忆点。008 手感系统管**角色输入-输出延迟**（不同关注点），相机手感由本文档自持。均用 SmoothDamp 语义。

### 11.1 落地下沉（Landing Dip）

- 触发：`movement_mode_system` 的着地分支（`was_airborne && landing_ok`，即 `vel.y<=0 && pos.y<=terrain+0.15`）**已存在但为内部信号**（仅 pop 恢复栈）。
  > **须显式暴露（O2）**：新增一次性落地事件（组件 `CJustLanded { impact_speed: f32 }` 或事件），在着地分支写入 `impact_speed = |vel.0.y|`。相机据此触发；下一系统帧清除。
- 效果：相机 pivot 短暂下沉 `dip = k × min(impact_speed, CAP)`，SmoothDamp 弹回。

### 11.2 疾跑 FOV 微推

- 触发：ECS `pace == Sprinting`。FOV 从基准 `lerp` +`SPRINT_FOV_KICK`（默认 +7°），SmoothDamp 过渡；停疾跑回落。
- **最终 FOV 钳制到 [60, 120]（O3）**：基准 FOV 为玩家设置值（[[../UI与UX系统/004-面板与管理系统|UI/UX 004]]），kick 叠加后 `clamp(base + kick, 60, 120)`，避免玩家设到上限时越界。

### 11.3 预留（§十三）

受击震屏、命中顿帧（hitstop）、相机后坐——归战斗冲刺（脉冲式，Cinemachine Impulse 风格）。

## 十二、输入门控与 player.gd 迁移

### 12.1 门控

CameraRig 的环顾输入在以下情形冻结、释放鼠标：
- **控制台开启**（`WorldDriver.is_console_open()`）——沿用现制。
- **模态面板开启**（物品栏/地图/大日志/设置）——需 `WorldDriver.is_ui_capturing()`（或等价）。
- **对话中不冻结**：玩家可边对话边行动（[[006-玩家专属I-O适配层]] §1.5），相机保持活动。

### 12.2 player.gd 收缩

| 职责 | 迁移 |
|------|------|
| 鼠标环顾（yaw→body / pitch→cam） | **移除** → CameraRig 接管（yaw→rig，body 不再被鼠标转） |
| 鼠标捕获 + ESC 切换 | **移出** → CameraRig（owns 环顾） |
| 控制台/UI 门控 | 由 CameraRig 自查 |
| Camera3D 子节点 | **移除**（相机迁至独立 CameraRig） |
| G 飞行 / legacy 走 / 夺舍物理 | **保留**（Player 节点仍 owns 物理/飞行/夺舍锚点） |

结果：player.gd 退化为纯物理脚本（飞行/legacy/夺舍），不再触碰相机。
