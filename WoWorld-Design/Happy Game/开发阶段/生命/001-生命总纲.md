## 一、核心架构：Life 与 NPC 心智的深层融合

### 1.1 为什么融合

当前 NPC 活人感模块 ver2.0 是独立模块——它对 NPC 的全貌做了完整定义（记忆/情绪/GOAP/关系）。但它隐含的假设是"NPC = 人类"。当世界中有精灵、矮人、程序化种族、灵兽、怪物、亡灵——它们各有不同的灵元素含量、魔力天赋、饥饿速度、体温耐受——NPC 模块缺少表达这些的基础设施。

**方案 B（深层融合）**：Life 定义所有生物共享的生物学基础，NPC 心智（SapientMind）在此之上叠加。NPC 活人感模块 ver2.0 的心智逻辑（GOAP/记忆/情绪/关系）完整保留——但它不再"拥有"身体和种族，而是从 Life 基类继承。

```rust
/// Life 是所有生物的最小公共结构
/// SapientMind(完全智能) / InstinctiveMind(动物) / 无(植物) 分层叠加
struct LifeEntity {
    // === 核心标识 ===
    id: EntityId,
    species_id: SpeciesId,
    display_name: String,            // 个体名称（"老瘸腿"、"白鬃"、"艾尔达"）

    // === 生命阶段 ===
    life_stage: LifeStage,
    age_years: f32,
    max_lifespan: f32,
    growth_progress: f32,            // 当前阶段进度 0-1

    // === 核心数值（所有生命共享）===
    vitals: Vitals,                  // HP/体力/饥饿/口渴/疲劳/体温/灵元素
    physical: PhysicalAttributes,    // 力量/敏捷/体质/感知/灵巧/耐久
    damage_pool: DamagePool,         // 各部位积累伤害
    spirit: SpiritState,             // 灵元素容量/当前灵素/十元素亲和
    magic: MagicAttributes,          // 魔力强度/控制/抗性/恢复速度
    mental: MentalAttributes,        // 智力/智慧/意志力/魅力/记忆容量 ← 原 NPC 模块 §1.1.2 过渡字段，现已融合
    appearance: PhysicalAppearance,  // 身高/体型/肤色/发色/面部特征等 ← 原 NPC 模块性别/吸引力文档过渡字段，现已融合

    // === 生态属性 ===
    ecology: EcologicalAttributes,   // 体型/食性/节律/社会性/领地性……

    // === 心智——按生命类型分层选择 ===
    mind: MindLayer,

    // === 繁衍 ===
    reproduction: Reproduction,

    // === 产物 ===
    produce: Vec<CreatureProduce>,
    inherent_slots: Vec<CreatureSlot>,

    // === 标签（可叠加、不互斥）===
    tags: HashSet<Tag>,
    status_effects: Vec<StatusEffectInstance>,

    // === 种族特质（查种族表——不存个体上）===
    race_traits: RaceTraitsId,
}
```

### 1.2 心智分层

这是关键架构决策——不同生命类型搭载不同深度的 AI：

| 心智层 | 拥有者 | 内容 | 计算成本 |
|--------|--------|------|---------|
| **无** | 植物 | 纯被动——无任何决策 | 0 |
| **MinimalMind** | 亡灵 | 仅残留的目标本能（攻击/守护/漫游） | ~0.01ms |
| **InstinctiveMind** | 动物/怪物 | 本能行为树 + 简单条件反射学习 | ~0.05ms |
| **InstinctiveMind+** | 灵兽 | InstinctiveMind + 指令理解 + 协作任务执行 | ~0.1ms |
| **SapientMind** | 智能种族 | InstinctiveMind + GOAP + 2000条记忆 + 情绪引擎 + 关系网络 | ~0.5ms |
| **DivineMind** | 神明(A模式)/龙 | 全知或近似全知——超越常规决策树 | 极高(少实例) |
| **PlayerMind** | 玩家 | 共享 BodyState/RaceTraits——心智由人类玩家输入驱动 | 0 |

```rust
enum MindLayer {
    None,
    Minimal(MinimalMind),
    Instinctive(InstinctiveMind),
    Sapient(SapientMind),    // ← 这就是当前 NPC 活人感模块 ver2.0 的内容
    Divine(DivineMind),
    Player(PlayerMind),
}
```

### 1.3 与 NPC 活人感模块的融合关系

NPC 活人感模块 ver2.0 在融合后：

| 原 NPC 模块内容 | 融合后 |
|---|---|
| NPC 大五人格 | 保留——在 SapientMind 中 |
| 情绪引擎（30种复合情绪） | 保留——新增输入：读取 BodyState 的 hunger/fatigue/spirit 等 |
| 2000条事件记忆 | 保留 |
| GOAP 规划器 | 保留——新增目标：食物/水源/睡眠/取暖（由 BodyState 驱动） |
| 社会关系网 | 保留 |
| NPC 身体参数（硬编码"人类"） | **废弃**——改为读取 Life.RaceTraits |
| 无灵元素/魔力 | **新增**——读取 Life.SpiritState + MagicAttributes |
| 无饥饿/疲劳/体温 | **新增**——读取 Life.Vitals，身体状态修改情绪和行为权重 |

```
NPC 的 GOAP 新增目标来源（从 Life.Vitals 读取）:
  hunger < 0.3  → 新增 GOAP 目标 "找到食物"
  thirst < 0.3  → 新增 GOAP 目标 "找到水源"
  fatigue > 0.8 → 新增 GOAP 目标 "找到地方睡觉"
  body_temperature 偏离 > 物种范围的 60% → 新增 "避寒/避暑"(GOAP缓冲——提前预警，危险阈值在80%)
  spirit < 0.2   → 新增 "去魔法浓度高的地方恢复"（仅高灵元素种族）
```

## 二、生命类型完整分类

| 类型 | 心智层 | 灵元素含量 | 可驯服 | 程序化生成 | 实例数(25万km²) | 说明 |
|------|--------|-----------|--------|-----------|----------------|------|
| **植物** | 无 | 低(0.05-0.15) | — | ✅ | ~数百万 | 架构分离——无 BodyState 完整继承 |
| **动物** | InstinctiveMind | 低(0.05-0.3) | 部分(灵兽) | ✅ | ~50-100万 | 含巨兽 |
| **灵兽** | InstinctiveMind+ | 中-高(0.4-0.8) | ✅ 5种驯服路径 | ✅ | ~1-5万 | 数据量小——每只可独特 |
| **怪物** | InstinctiveMind | 异常(0.2-0.9) | ❌ | ✅ | ~5-20万 | 7种来源+自然动物变体 |
| **亡灵** | MinimalMind | 异常残留(衰减中) | ❌ | 部分 | ~动态 | 由世界状态(战争/魔力/死灵法师)驱动 |
| **智能种族** | SapientMind | 中-高(0.3-0.7) | — | ✅ | ~100K+(L1-L3), 数百万(L4统计) | 人类/精灵/矮人/程序化种族 |
| **玩家** | PlayerMind | 种族决定 | — | — | 1 | 智能种族特例 |
| **神明(A模式)** | DivineMind | 极高(>0.9) | — | — | 0-数十 | 仅A模式存在实体 |
| **特殊造物** | 不定 | 不定 | 不定 | — | 极少 | 开发者手搓（龙/小精灵） |

## 三、生成顺序

```
世界生成 P2: 生物群系完成
  │
  ├─ 每个群系参数筛选合法积木候选集
  ├─ 12维积木采样（环境约束 × 历史塑形 × 种子随机）
  ├─ 第一层：硬约束检查（类型/力学/功能矛盾 → 拒绝重采）
  ├─ 第二层：生态学过滤（食物/防御/繁殖/气候/生态位 → 拒绝重采）
  ├─ 第三层：模式模板匹配（~15-20模板 → 比例范围约束）
  ├─ 第四层：美学评分（比例和谐+色彩+功能一致性+轮廓+生态位）
  │        0.8-1.0→常见  0.5-0.8→不常见  0.3-0.5→稀有  <0.3→极稀有
  ├─ 物种池完成
  │
  └─ 按稀有度×密度投放个体 → 世界中的实际生物分布
```

## 四、核心设计原则

| 原则 | 说明 |
|------|------|
| **所有伤害削减数值，所有数值可恢复** | 无永久性伤害。治愈=时间+休息+药草+魔法 |
| **特质从历史涌现，不预置** | "精灵敏捷+2"这种手写表不存在——种族参数 = f(环境, 历史, 种子) |
| **灵元素普遍存在** | 除亡灵外每个生命都有灵元素。含量决定魔力天赋。含量低≠不能学魔法——只是起点低 |
| **标签可叠加、不互斥** | 一个生命可同时被标记为 `Natural | Venomous | Nocturnal | Skittish` |
| **群系驱动生物** | 群系参数筛选积木合法空间——不是"先生成所有生物再往群系塞" |
| **动物种群用动态模型** | 物种层面驱动——L1 才模拟个体行为（幼崽/巢穴/捕猎） |
| **智能种族繁衍用社会关系** | 不是本能——是经济+伴侣+文化+个人意愿的综合决策 |

## 五、关键 trait 接口

```rust
/// 生命系统对外暴露的统一接口
trait LifeSystem {
    /// 生成世界的全部物种（世界生成阶段调用）
    fn generate_all_species(seed: u64, biomes: &BiomeMap, history: &WorldHistory) -> Vec<Species>;

    /// （运行时）实体更新——由 Rust 模拟核心每帧/每批调用
    fn tick_entity(&mut self, entity: &mut LifeEntity, delta: f32, world: &WorldState);

    /// 按群系+生态位查询可能出现的物种
    fn query_species_for_biome(biome: BiomeType, niche: EcologicalNiche) -> Vec<SpeciesId>;

    /// 验证：给定一组积木+群系，此生物能否存活
    fn validate_viability(candidate: &CreatureSpec, biome: &BiomeType) -> ValidationResult;

    /// 美学评分
    fn aesthetic_score(candidate: &CreatureSpec) -> f32;

    /// 繁衍——智能种族
    fn evaluate_fertility_desire(npc: &LifeEntity, society: &SocietyState) -> f32;
}
```
