# 008 — 货币、通胀与经济稳定器

## 一、货币三管道

### 1.0 设计原则

货币不是"玩家背包里的数字"——它是一个在开放世界中持续流动的动态系统。货币总量随挖矿、交易、消费、税收、跨区流动而实时变化。三个管道共同决定每个Market的货币供给量，进而通过订单簿影响价格。

**核心约束**：货币系统不创造"凭空的钱"——铸币管道的唯一来源是贵金属矿床的物理开采。所有货币流动都有物理原因和可追踪路径。货币永远守恒——从一个口袋流向另一个口袋，或在消费回收中被系统销毁。

```
世界贵金属矿床（世界生成决定规模）
  → 矿场开采（NPC + 玩家）
    → 铸币厂铸造（需 IssueCurrency PowerAtom）
      → 铜币进入当地 Market → 管道A完成

铜币在市场间流动
  → 商队套利（管道C）→ 铜币从高价区流向低价区
  → NPC旅行/搬迁（管道C）→ 铜币随人物移动
  → 玩家跨区交易（管道C）

铜币退出流通
  → 食物被吃掉（管道B）→ 铜币回收到"世界池"
  → 仓储租金 / 驮畜饲料 / 房屋维护（管道B）
  → 税收回收（管道B）→ 铜币回收到政府金库
```

### 1.1 管道A：铸币（货币进入流通）

货币的唯一来源是贵金属矿床的开采与铸造。不存在"系统凭空印钱"。

**铸币流程**：

```
1. 矿场产出金/银/铜矿石
   - 矿石产量 = f(矿床规模, 矿工技能, 矿场设施等级)
   - 矿床规模由世界生成决定——每个地区的地质禀赋不同
   - 矿场本身是NPC经营的经济实体（非"系统刷怪点"）

2. 矿石 → 铸币
   - 需要拥有 IssueCurrency PowerAtom 的实体操作铸币厂
   - 通常由领主/国王/行会控制——但也可以是私铸（地下铸币）
   - 铸造比例：金:银:铜 = 1:20:400（参考中世纪欧洲汇率）
   - 1铜币 = 1单位交易基准（最小不可分割单位）

3. 新铸铜币进入当地 Market
   - 铸币厂所在聚落的 Market 最先获得新货币
   - 通过管道C逐步扩散到其他 Market
   - 货币注入速度受 base_money_creation_rate 限制
```

**铸币速度限制**：

```rust
/// 货币创造速度配置
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MoneyCreationConfig {
    /// 基础日铸币率——每日新增货币占当前M2总量的比例
    /// TOML可配：0.0001-0.001（即日0.01%-0.1%）
    pub base_money_creation_rate: f64,

    /// 矿床规模系数——矿床越大，可支撑的铸币率越高
    /// 矿床规模由世界生成提供（ResourceMap）
    pub deposit_scale_multiplier: f64,

    /// 铸币厂效率——受技术水平影响
    /// 初期=1.0，随技术进步可提升至1.5-2.0
    pub mint_efficiency: f64,

    /// 铸币权分配——哪些实体拥有铸币权
    /// 运行时从 MarketAuthority 查询 IssueCurrency PowerAtom
    pub minting_authorities: Vec<EntityId>,
}
```

**全局铸币上限**：世界贵金属矿床总储量 = 货币总量的物理上限。当所有已探明矿床被充分开采后，铸币速度自然趋近于零——除非发现新矿床。这是经济系统的"金本位"约束——不是设计偏好，而是世界生成数据的内在结果。

```rust
/// 货币供给——全局状态
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MoneySupply {
    /// 当前全球铜币总量（流通中 + 金库存储）
    pub total_copper_in_circulation: u64,

    /// 当前全球铜币流通量（在NPC/玩家钱包中，不含金库）
    pub copper_in_wallets: u64,

    /// 政府金库存储（税收回收后暂存，待再分配）
    pub copper_in_treasuries: BTreeMap<EntityId, u64>,

    /// M2 = 流通量 + 金库存储
    /// （WoWorld没有银行派生货币，M2 = M0）
    pub m2: u64,

    /// 各Market的本地货币供给估计
    pub local_m2: BTreeMap<MarketId, u64>,

    /// 已开采矿床累计产出（用于追踪资源消耗）
    pub cumulative_mined_copper: u64,

    /// 全球贵金属矿床总估计储量（世界生成提供，随探矿更新）
    pub estimated_total_deposit_copper: u64,

    /// 铸币统计——按日滚动
    pub daily_mint_stats: VecDeque<DailyMintRecord>,

    /// 上月快照——用于计算月度增长率
    pub last_month_m2: u64,
    pub last_month_goods_value: u64,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DailyMintRecord {
    pub date: GameDate,
    pub minted_copper: u64,
    pub source_market: MarketId,
    pub authority: EntityId,
}

/// 铸币——管道A入口
///
/// 由拥有 IssueCurrency PowerAtom 的实体调用。
/// 每日tick时，系统遍历所有铸币权持有者，按其设施产能自动铸币。
fn mint_currency(
    supply: &mut MoneySupply,
    config: &MoneyCreationConfig,
    deposit_data: &ResourceMap,       // 世界生成提供的矿床数据
    authority: EntityId,
    target_market: MarketId,
    game_time: GameTimestamp,
) -> Result<u64, MintError> {
    // 1. 检查铸币权
    if !has_power_atom(authority, PowerAtom::IssueCurrency { amount_copper: 0 }) {
        return Err(MintError::NoMintingAuthority);
    }

    // 2. 计算本日最大可铸量
    let max_daily = (supply.m2 as f64 * config.base_money_creation_rate) as u64;

    // 3. 矿床约束——不能铸造超过剩余可采储量
    let remaining_deposit = deposit_data.remaining_copper_in_region(target_market.region());
    let deposit_constrained = remaining_deposit.min(max_daily);

    // 4. 铸币厂产能约束
    let mint_capacity = calc_mint_capacity(authority, config.mint_efficiency);
    let actual_mint = deposit_constrained.min(mint_capacity);

    // 5. 执行：新增铜币进入目标Market
    supply.total_copper_in_circulation += actual_mint;
    supply.m2 += actual_mint;
    supply.cumulative_mined_copper += actual_mint;
    *supply.local_m2.entry(target_market).or_insert(0) += actual_mint;

    // 6. 记录
    supply.daily_mint_stats.push_back(DailyMintRecord {
        date: game_time.date(),
        minted_copper: actual_mint,
        source_market: target_market,
        authority,
    });

    // 7. 新铸铜币进入铸币厂钱包（随后通过薪资/采购进入流通）
    deposit_to_treasury(authority, actual_mint);

    Ok(actual_mint)
}
```

### 1.2 管道B：消费回收（货币退出流通）

货币不是永远存在的——消费行为会让货币被系统"回收"（销毁），退出流通。管道B模拟以下情形：

- **必需品消费后无再交易价值**：食物被吃掉、药水被喝掉、蜡烛燃尽——对应的铜币没有"下一手"可流通的物品，货币在此终结
- **服务消费**：住店、理发、看病——服务被即时消费，货币从流通中移除
- **维护折旧**：房屋维修、驮畜饲料、工具磨损更换——维护成本是持续的货币漏出
- **税收回收**：政府征税后，税收收入一部分用于支付公职薪资（回到流通），一部分用于公共工程（材料采购→回到流通），净回收部分为税收中未被再支出的余额

**货币回收率公式**：

```
回收率 = f(经济活跃度)

经济活跃度 = 近30日交易笔数 / 市场历史峰值交易笔数

当经济活跃度↑ → 消费量↑ → 回收率↑
当经济活跃度↓ → 消费量↓ → 回收率↓（经济萧条时回收自然减少）
```

```rust
/// 消费回收配置
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ConsumptionReclaimConfig {
    /// 基础回收率——占每日消费总额的比例
    /// 食物/药水等消耗品 ≈ 0.95（几乎全部回收）
    /// 耐用品 ≈ 0.05（工具折旧部分回收）
    pub base_reclaim_rate: f64,

    /// 经济活跃度→回收率的弹性
    /// 0.5 = 活跃度每增加1%，回收率增加0.5%
    pub activity_elasticity: f64,

    /// 税收净回收率——税收中不被再支出的比例
    /// 高效政府≈0.1（90%再支出），腐败政府≈0.5（50%被贪墨/囤积）
    pub tax_net_reclaim_rate: f64,
}

/// 消费回收——管道B出口
///
/// 在每笔消费交易完成后调用。
/// 仅对"终端消费"执行回收——中间商采购不回收。
fn reclaim_currency(
    supply: &mut MoneySupply,
    config: &ConsumptionReclaimConfig,
    transaction: &Transaction,
    buyer: &NpcEconomicState,
) -> u64 {
    // 1. 判断是否为终端消费（非中间商转售）
    if !is_final_consumption(transaction, buyer) {
        return 0; // 中间商采购——货币仍留在流通中
    }

    // 2. 根据物品类别确定回收率
    let base_rate = match transaction.item_category {
        ItemCategory::Food | ItemCategory::Potion | ItemCategory::Consumable => {
            config.base_reclaim_rate * 0.95 // 消耗品：几乎全回收
        }
        ItemCategory::Tool | ItemCategory::Equipment => {
            config.base_reclaim_rate * 0.05 // 耐用品：仅折旧部分
        }
        ItemCategory::Service => {
            config.base_reclaim_rate * 1.0  // 服务：全部回收
        }
        _ => 0.0, // 原材料/半成品——不回收（会再加工）
    };

    // 3. 经济活跃度修正
    let activity = calc_economic_activity(transaction.market_id);
    let activity_modifier = 1.0 + (activity - 0.5) * config.activity_elasticity;

    // 4. 计算回收量
    let reclaim_amount = (transaction.total_copper as f64 * base_rate * activity_modifier) as u64;

    // 5. 执行回收
    supply.total_copper_in_circulation -= reclaim_amount;
    supply.m2 -= reclaim_amount;
    *supply.local_m2.entry(transaction.market_id).or_insert(0) -= reclaim_amount;

    reclaim_amount
}

/// 税收回收——管道B的政府子管道
///
/// 在税收征收时调用。净回收 = 税收总额 × net_reclaim_rate。
/// 剩余部分通过政府支出回到流通。
fn reclaim_tax(
    supply: &mut MoneySupply,
    config: &ConsumptionReclaimConfig,
    tax_collected: u64,
    government_entity: EntityId,
) -> (u64, u64) {
    let net_reclaim = (tax_collected as f64 * config.tax_net_reclaim_rate) as u64;
    let returned_to_circulation = tax_collected - net_reclaim;

    // 净回收部分——退出流通
    supply.total_copper_in_circulation -= net_reclaim;
    supply.m2 -= net_reclaim;

    // 可再支出部分——进入政府金库（待后续支出回到流通）
    *supply.copper_in_treasuries.entry(government_entity).or_insert(0) += returned_to_circulation;

    (net_reclaim, returned_to_circulation)
}
```

### 1.3 管道C：跨区流动（货币在不同Market间移动）

货币在不同Market间的流动由三种驱动力推动：

**驱动力一：商队套利**

中间商（满足004四条件的NPC）扫描价差 → 在低价Market买入 → 运到高价Market卖出 → 货币从高价区流向低价区。这是最强劲的跨区流动驱动力。

**驱动力二：NPC旅行/搬迁**

NPC携带钱包旅行或搬迁 → 货币跟随人物物理移动。NPC迁入城市 → 其钱包余额加入当地M2。NPC死亡 → 遗产由继承者获得（可能在不同Market）。

**驱动力三：玩家跨区交易**

玩家在Market A用铜币买物品 → 带到Market B卖出 → 货币从B流向A（与商队套利同向——总是从买方市场流向卖方市场）。

```rust
/// 跨区货币转移——管道C
///
/// 由以下事件触发：
/// - 商队完成跨Market交易
/// - NPC搬迁
/// - 玩家跨区买卖
fn transfer_between_markets(
    supply: &mut MoneySupply,
    amount: u64,
    from_market: MarketId,
    to_market: MarketId,
    reason: TransferReason,
) {
    // 从来源Market扣除
    *supply.local_m2.entry(from_market).or_insert(0) =
        supply.local_m2[&from_market].saturating_sub(amount);

    // 加入目标Market
    *supply.local_m2.entry(to_market).or_insert(0) += amount;

    // 全局M2不变——货币只是换了位置
    // 注意：如果跨区运输途中货币随商队遭遇损失（劫掠/事故），
    // 损失部分由物品系统的物品丢失事件处理——货币并不"凭空消失"，
    // 而是被劫掠者获得或随残骸留在野外（等待被发现）
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum TransferReason {
    CaravanTrade { caravan_id: CaravanId, profit_copper: u64 },
    NpcRelocation { npc_id: NpcId, new_settlement: SettlementId },
    PlayerTrade { from_settlement: SettlementId, to_settlement: SettlementId },
    Inheritance { heir_npc_id: NpcId, deceased_npc_id: NpcId },
}
```

## 二、五大自动稳定器

所有稳定器在运行时**不可见**——玩家察觉不到数学公式在运行，但可感知经济效果（价格不会无限制飙升、不会出现恶性通缩、一个玩家无法垄断全服经济）。

### 2.1 稳定器A：供给弹性稳压

当价格偏离基准时，供给侧自动调整产能以拉回价格。这是最基础的稳定器——模拟真实经济中"价高→更多生产者进入→供给增加→价格回落"的市场规律。

**触发条件**：
- 当前价格 > 基准价格 × 1.5 → 该物品的所有生产者效率 +20%（加速生产）→ 供给增加 → 价格回落
- 当前价格 < 基准价格 × 0.6 → 生产者减产（效率 -20%）→ 供给减少 → 价格回升

**机制细节**：
- 增产不是瞬间的——有产能爬坡延迟（2-5游戏日）
- NPC生产者根据自己的 EconomicCognition.market_understanding 决定是否响应价格信号——不是所有生产者都理性增产
- 减产是即时的（NPC倾向于快速减少亏损业务）

```rust
/// 供给弹性稳压器
fn apply_supply_elasticity(
    market: &mut Market,
    producers: &mut [&mut NpcEconomicState],
    config: &StabilizerConfig,
) {
    for (item_def, &last_price) in &market.last_price {
        let base_value = get_base_value_copper(*item_def); // 来自物品系统

        if last_price as f64 > base_value as f64 * config.price_ceiling_trigger {
            // 价格过高 → 激励增产
            let efficiency_boost = config.production_elasticity; // 默认0.2 = +20%
            for producer in producers.iter_mut() {
                if producer.produces_item(*item_def) {
                    // NPC自主决定是否响应（受market_understanding影响）
                    if producer.cognition.market_understanding > 0.3 {
                        producer.apply_production_boost(*item_def, efficiency_boost);
                    }
                }
            }
        } else if (last_price as f64) < base_value as f64 * config.price_floor_trigger {
            // 价格过低 → 减产
            let efficiency_cut = config.production_elasticity;
            for producer in producers.iter_mut() {
                if producer.produces_item(*item_def) {
                    producer.apply_production_cut(*item_def, efficiency_cut);
                }
            }
        }
    }
}
```

### 2.2 稳定器B：累进税自动回收

税收不只是"系统收钱"——它是货币再分配的内置机制。累进税率让高财富者贡献更多，税收收入通过政府支出（公职薪资、公共工程）回到流通——货币从富人流向政府，再从政府流向全社会。

**税率表**（TOML可配）：

| 财富等级 | 判定标准（相对当地中位数） | 税率 | 说明 |
|---------|------------------------|------|------|
| 赤贫 | < 中位数 × 0.25 | 0% | 免税 |
| 贫困 | 中位数 × 0.25-0.75 | 3% | 象征性 |
| 小康 | 中位数 × 0.75-2.0 | 5% | 基准 |
| 富足 | 中位数 × 2.0-5.0 | 15% | 累进加重 |
| 富裕 | > 中位数 × 5.0 | 30% | 最高档 |

**再分配机制**：
- 税收进入政府金库（`copper_in_treasuries`）
- 政府金库支出优先级：公职薪资 > 公共工程材料采购 > 军费 > 宫廷消费
- 公职薪资是最大的再分配渠道——让铜币从金库回到普通NPC钱包
- 净回收率（`tax_net_reclaim_rate`）决定有多少税收被真正"回收"而不再支出

```rust
/// 累进税自动回收
fn apply_progressive_tax(
    npc: &NpcEconomicState,
    local_median_wealth: u64,
    config: &StabilizerConfig,
) -> u64 {
    let wealth = npc.wallet.balance_copper();
    let ratio = wealth as f64 / local_median_wealth.max(1) as f64;

    let tax_rate = if ratio < 0.25 {
        0.0                                    // 赤贫 — 免税
    } else if ratio < 0.75 {
        config.tax_bracket_low                  // 3%
    } else if ratio < 2.0 {
        config.tax_bracket_middle               // 5%
    } else if ratio < 5.0 {
        config.tax_bracket_high                 // 15%
    } else {
        config.tax_bracket_top                  // 30%
    };

    (wealth as f64 * tax_rate * config.tax_progressive_factor) as u64
}
```

### 2.3 稳定器C：必需品价格刚性

已在 [[002-市场与订单簿]] 的分层定价中体现。此处仅做概要说明。

必需品（食物、基本工具、基础衣物）的价格粘性高（stickiness = 0.6-0.8），意味着价格不会因短期供需波动大幅跳跃。奢侈品（珠宝、艺术品、稀有材料）的粘性低（stickiness = 0.1-0.2），价格可剧烈波动。

**效果**：即使发生粮食短缺，食物价格也不会瞬间翻10倍——NPC会先减少购买量、搜寻替代品、动用储备，然后价格才逐步上升。这为管道A的增产响应提供了缓冲时间。

### 2.4 稳定器D：运输成本壁垒

已在 [[002-市场与订单簿]] 和 [[003-价格形成与信息不对称]]（待完成）中体现。此处做经济稳定视角的补充。

商队套利需要价差 > 运费 + 关税 + 风险溢酬，才有利可图。这意味着：
- 两个Market间的价格差天然存在一个"地板"——低于运费+关税的价差不会被套利抹平
- 偏远地区的价格天然高于中心地区（运费叠加）
- 战争/天气恶劣 → 风险溢酬上升 → 有效价差阈值提高 → 套利减少 → 区域价格分化加剧

这是"物理事实充当经济稳定器"——不需要额外的数学公式，运费本身就是价差的天然制动器。

```rust
/// 计算跨Market套利的最低可行价差
fn minimum_arbitrage_spread(
    from_market: &Market,
    to_market: &Market,
    item_def: ItemDefId,
    quantity: u32,
    weather: &WeatherSample,    // 来自天气系统 WeatherQuery
    security_status: f32,       // 0-1，商路安全度（来自政治/军事系统）
) -> u64 {
    let base_price = get_base_value_copper(item_def);

    // 运费 = 距离 × 单位运费率 × 天气修正
    let distance = calc_trade_route_distance(from_market, to_market);
    let weather_modifier = 1.0 + weather.precipitation_intensity * 0.5; // 雨雪增加运费
    let transport_cost = (distance * TRANSPORT_COST_PER_KM * weather_modifier) as u64;

    // 关税（来自MarketRegulations）
    let tariff = to_market.regulations.import_tariff_rate * base_price;

    // 风险溢酬 = f(商路安全度, 物品价值)
    let risk_premium = (base_price as f64 * (1.0 - security_status) * RISK_PREMIUM_FACTOR) as u64;

    // 最低可行价差
    transport_cost + tariff as u64 + risk_premium
}
```

### 2.5 稳定器E：货币-商品总量对齐

防止"钱多了但东西没多"的通胀根源。当商品总量（所有在售物品的总基准价值）增长跟不上M2增长时，自动减缓铸币速度。

```
商品总量增长率 = (本月总商品基准价值 - 上月) / 上月
M2增长率 = (本月M2 - 上月) / 上月

if M2增长率 > 商品总量增长率 × 1.5:
    铸币速度自动减半（base_money_creation_rate *= 0.5）
    持续至 M2增长率 ≤ 商品总量增长率
```

这确保货币供给不会脱离实体经济无限制膨胀。

```rust
/// 货币-商品总量对齐稳定器
fn align_money_to_goods(
    supply: &mut MoneySupply,
    mint_config: &mut MoneyCreationConfig,
    all_markets: &[&Market],
) {
    // 1. 计算本月总商品基准价值
    let current_goods_value: u64 = all_markets.iter()
        .flat_map(|m| m.asks.values().flatten())
        .filter(|listing| listing.side == OrderSide::Ask && listing.status == ListingStatus::Active)
        .map(|listing| {
            let base = get_base_value_copper(listing.item_def);
            base as u64 * listing.quantity as u64
        })
        .sum();

    // 2. 计算增长率
    let goods_growth = if supply.last_month_goods_value > 0 {
        (current_goods_value as f64 - supply.last_month_goods_value as f64)
            / supply.last_month_goods_value as f64
    } else {
        0.0
    };

    let m2_growth = if supply.last_month_m2 > 0 {
        (supply.m2 as f64 - supply.last_month_m2 as f64) / supply.last_month_m2 as f64
    } else {
        0.0
    };

    // 3. M2增速超过商品增速1.5倍 → 自动减速铸币
    if m2_growth > goods_growth * 1.5 && goods_growth >= 0.0 {
        mint_config.base_money_creation_rate *= 0.5;
        log_economic_event(
            EconomicEvent::MoneySupplyDecelerated {
                reason: "M2 growth exceeds goods growth",
                previous_rate: mint_config.base_money_creation_rate * 2.0,
                new_rate: mint_config.base_money_creation_rate,
            }
        );
    }

    // 4. 恢复正常速度的条件：M2增速回落至商品增速以下
    if m2_growth <= goods_growth && mint_config.base_money_creation_rate < MINT_RATE_NORMAL {
        mint_config.base_money_creation_rate =
            (mint_config.base_money_creation_rate * 1.5).min(MINT_RATE_NORMAL);
    }

    // 5. 更新上月快照
    supply.last_month_goods_value = current_goods_value;
    supply.last_month_m2 = supply.m2;
}
```

## 三、通胀检测

### 3.1 通胀定义

WoWorld 经济系统中，通胀 = M2增长显著快于商品总量增长，导致价格普遍持续上涨。

**检测指标**（四级）：

| 级别 | 条件 | 含义 |
|------|------|------|
| 正常 | M2月增 < 3% | 健康增长 |
| 关注 | M2月增 3%-5% | 密切观察，暂不干预 |
| **警报** | **M2月增 > 5%** | **触发自动响应** |
| 恶性 | M2月增 > 15% | 紧急状态——触发所有可用反制措施 |

### 3.2 自动响应机制

当触发通胀警报（M2月增 > 5%）时，系统自动执行以下措施（按优先级）：

```
1. 增加消费回收率
   consumption_reclaim_rate *= 1.5
   效果：更多货币从流通中移除

2. 提高生产弹性
   production_elasticity *= 1.5
   效果：高价格更快刺激增产 → 供给更快增加 → 价格更快回落

3. 缩减铸币速度
   base_money_creation_rate *= 0.25
   效果：新货币注入速度降至原来的1/4

4. 记录经济事件 → 写入历史系统
   - 事件类型：EconomicEvent::InflationAlert
   - 包含：触发时M2、月增长率、受影响Market列表、自动措施清单
   - 后续可被历史系统消费 → 生成"大通胀时代"叙事
```

**响应解除条件**：连续2个月M2月增 < 3% → 逐步恢复正常参数（每次恢复25%，避免过度震荡）。

```rust
/// 通胀检测
fn detect_inflation(
    supply: &MoneySupply,
    mint_config: &mut MoneyCreationConfig,
    config: &EconomicStabilizerConfig,
) -> Option<EconomicEvent> {
    let m2_monthly_growth = calc_m2_monthly_growth(supply);

    if m2_monthly_growth > config.inflation_threshold_monthly {
        // 触发通胀警报
        let severity = if m2_monthly_growth > 0.15 {
            InflationSeverity::Hyperinflation
        } else if m2_monthly_growth > 0.10 {
            InflationSeverity::Severe
        } else {
            InflationSeverity::Moderate
        };

        // 自动响应
        mint_config.base_money_creation_rate *= 0.25;

        let event = EconomicEvent::InflationAlert {
            timestamp: current_game_time(),
            m2_monthly_growth,
            current_m2: supply.m2,
            severity,
            auto_measures: vec![
                AutoMeasure::ReclaimRateIncreased { multiplier: 1.5 },
                AutoMeasure::ElasticityIncreased { multiplier: 1.5 },
                AutoMeasure::MintSpeedReduced { multiplier: 0.25 },
            ],
            affected_markets: supply.local_m2.keys().cloned().collect(),
        };

        // 写入历史系统
        push_to_history(EconomicEventSink::Inflation(event.clone()));

        Some(event)
    } else {
        // 检查是否可以恢复正常参数
        if m2_monthly_growth < 0.03
            && mint_config.base_money_creation_rate < config.mint_rate_normal
        {
            mint_config.base_money_creation_rate =
                (mint_config.base_money_creation_rate * 1.25).min(config.mint_rate_normal);
        }
        None
    }
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
enum InflationSeverity {
    Moderate,        // 5%-10%
    Severe,          // 10%-15%
    Hyperinflation,  // >15%
}

/// 计算月度M2增长率
fn calc_m2_monthly_growth(supply: &MoneySupply) -> f64 {
    if supply.last_month_m2 == 0 {
        return 0.0;
    }
    (supply.m2 as f64 - supply.last_month_m2 as f64) / supply.last_month_m2 as f64
}
```

## 四、玩家垄断反制

### 4.1 设计原则

玩家和NPC使用完全相同的经济规则——玩家没有"系统保护"也没有"系统打压"。但玩家拥有NPC没有的优势（跨区信息整合、目标导向的长期规划、不受GOAP日常循环约束），可能形成事实垄断。

**三层防御的目标不是"禁止垄断"**——垄断本身是合法的涌现结果。防御目标是让垄断变得"困难但可能"——成为有意义的经济成就，而非系统漏洞。

### 4.2 第一层：市场容量限制

每个Market对每个交易实体的挂单数量有上限——防止单一实体用数千条挂单淹没订单簿。

### 4.3 第二层：垄断检测 + 反制税

当单一实体控制某物品在某个Market超过40%的供给时：

1. **垄断税**：该实体在该物品上的利润加征30%垄断税
2. **广播信息**：通过 [[../语言表达系统/009-信息传播与谣言|语言表达 009]] 的五通道向外界广播"某Market某商品供不应求"——吸引外地商队前来套利。外地商队携带该物品涌入 → 供给增加 → 该实体份额下降 → 垄断自然消解
3. **广播延迟**：信息传播需要时间（商队物理旅行），外地供给不会瞬间到达——垄断者有一个窗口期获利

```rust
/// 垄断检测
fn detect_monopoly(
    market: &Market,
    item_def: ItemDefId,
) -> Option<MonopolyReport> {
    // 1. 统计该物品的总供给量（所有活跃Ask挂单）
    let total_supply: u32 = market.asks.get(&item_def)
        .map(|listings| listings.iter()
            .filter(|l| l.status == ListingStatus::Active)
            .map(|l| l.quantity)
            .sum())
        .unwrap_or(0);

    if total_supply == 0 {
        return None;
    }

    // 2. 按实体聚合供给
    let mut entity_supply: BTreeMap<EntityId, u32> = BTreeMap::new();
    for listing in market.asks.get(&item_def).iter().flat_map(|v| v.iter()) {
        if listing.status == ListingStatus::Active {
            *entity_supply.entry(EntityId::Npc(listing.owner)).or_insert(0) += listing.quantity;
        }
    }

    // 3. 检测 > 40% 的单一控制者
    for (entity, supply) in &entity_supply {
        let share = *supply as f64 / total_supply as f64;
        if share > MONOPOLY_THRESHOLD {
            return Some(MonopolyReport {
                market_id: market.market_id,
                item_def,
                monopolist: *entity,
                supply_share: share,
                total_supply,
                detection_time: current_game_time(),
            });
        }
    }

    None
}

const MONOPOLY_THRESHOLD: f64 = 0.40;

#[derive(Debug, Clone, Serialize, Deserialize)]
struct MonopolyReport {
    market_id: MarketId,
    item_def: ItemDefId,
    monopolist: EntityId,
    supply_share: f64,
    total_supply: u32,
    detection_time: GameTimestamp,
}

/// 垄断反制措施
fn apply_monopoly_countermeasure(
    report: &MonopolyReport,
    market: &mut Market,
    language_system: &mut dyn InformationPropagation, // 语言表达009的信息传播接口
) {
    // 1. 垄断税——后续交易自动扣除
    market.regulations.special_tax_rates.insert(
        (report.monopolist, report.item_def),
        MONOPOLY_TAX_RATE, // 30%
    );

    // 2. 广播信息——吸引外地商队
    let propagation_payload = PropagationPayload {
        content: format!(
            "{}的{}供不应求，价格持续走高",
            market.name, get_item_display_name(report.item_def)
        ),
        source_market: report.market_id,
        item_in_shortage: Some(report.item_def),
        price_signal: Some(market.last_price[&report.item_def]),
        timestamp: report.detection_time,
    };

    // 通过语言表达系统的信息传播通道广播
    language_system.propagate_information(
        propagation_payload,
        PropagationChannel::Official,  // 公告/商队消息
        report.market_id,
    );

    // 3. 记录经济事件
    push_to_history(EconomicEventSink::MonopolyDetected {
        market: report.market_id,
        item: report.item_def,
        monopolist: report.monopolist,
        share: report.supply_share,
    });
}

const MONOPOLY_TAX_RATE: f64 = 0.30;
```

### 4.4 第三层：NPC记忆抵抗

NPC商人不是价格接受机器——他们用 [[007-NPC经济心智|NPC经济心智]] 的全部能力判断交易合理性。当某个实体持续高价收购某物品时：

1. **锚定效应**：NPC记住"这个东西最近一直有人高价收"→ 自动涨价15-25%
2. **有限理性**：NPC不一定理解这是在垄断——但他们的"满意阈值"会自动提高（"既然有人愿意出高价，我为什么要低价卖？"）
3. **记忆传播**：通过 [[../语言表达系统/009-信息传播与谣言|语言表达 009]] 的五通道，关于"某人大量收购XX"的信息在NPC间传播（失真算子加成）

```rust
/// NPC对异常购买行为的记忆和价格抵抗
fn npc_price_resistance_to_monopoly(
    npc: &mut NpcEconomicState,
    buyer: EntityId,
    item_def: ItemDefId,
    recent_purchase_volume: u32,  // 该买家近7日对该物品的总收购量
    normal_volume: u32,           // 该物品在该市场的日均交易量
) -> f32 {
    // 收购量超过正常量3倍 → NPC警觉
    if recent_purchase_volume > normal_volume * 3 {
        // NPC人格驱动价格上调
        let base_uplift = 0.15;  // 15%基础涨价
        let neuroticism_bonus = npc.cognition.financial_literacy * 0.10; // 金融素养越高，涨得越多
        let extraversion_penalty = (1.0 - npc.cognition.market_understanding) * 0.05;

        let price_uplift = base_uplift + neuroticism_bonus - extraversion_penalty;
        // 范围：10%-25%

        // 更新NPC对该物品对该买家的心理价位
        npc.known_prices
            .entry((buyer, item_def))
            .and_modify(|pk| {
                pk.price = (pk.price as f64 * (1.0 + price_uplift)) as u32;
                pk.confidence *= 0.8; // 信心下降——"不太确定这个价格合不合理"
            });

        price_uplift
    } else {
        0.0
    }
}
```

### 4.5 三层防御的协同效果

```
玩家试图垄断铁矿市场
  → 第一层：挂单数上限 → 不能一次挂1000单
  → 玩家分批收购 + 提价 → 逐渐控制35%供给（仍低于阈值）

玩家继续收购 → 超过40%
  → 第二层触发：垄断税30% → 利润大幅缩水
  → 广播信息 → 外地商队带铁矿涌入（约3-7日后到达）
  → 同时：第三层触发 → 本地NPC矿主开始涨价15-25%

结果：
  - 3-7日内：玩家在垄断窗口期可获利（垄断税后利润）
  - 7日后：外地供给涌入 + NPC涨价 → 玩家份额降到30%以下 → 垄断自动瓦解
  - 玩家在此期间的利润 = 垄断成就的"奖励"——不是系统漏洞
```

## 六、性能预算

| 操作 | 频率 | 预算 | 说明 |
|------|------|------|------|
| 铸币tick | 每日 | <0.02ms | 仅遍历铸币权持有者（通常<10个） |
| 消费回收 | 每笔终端消费 | <0.005ms | 简单算术，交易完成后同步执行 |
| 跨区转移 | 事件驱动 | <0.01ms | 商队完成/搬迁时触发，低频 |
| 供给弹性tick | 每3日 | <0.5ms | 需遍历所有Market的last_price |
| 累进税计算 | 征税日 | <0.1ms | 已在NPC经济状态tick中摊销 |
| 货币商品对齐 | 每月 | <1ms | 需聚合所有Market挂单统计 |
| 通胀检测 | 每月 | <0.05ms | 简单月增率计算 |
| 垄断检测 | 每日 | <0.5ms | 每Market每物品遍历挂单聚合 |
| **总计** | **摊销** | **<0.2ms/帧** | vs 技术栈方案 Rust CPU预算 ~7ms/帧 |

| 资源 | 预算 |
|------|------|
| MoneySupply 内存 | ~200 bytes + per-Market条目(~100 bytes × 3000 = ~300KB) |
| EconomicStabilizerConfig | ~150 bytes（纯配置） |
| DailyMintRecord 滑动窗口（365天） | ~40 bytes × 365 = ~14.6KB |
| MonopolyReport（临时） | 栈上分配，不持久 |

## 七、跨模块接口契约

| 概念 | 权威 Owner | 消费方 | 关键约定 |
|------|-----------|--------|---------|
| MoneySupply | **经济系统 008** | 001总纲/历史系统/006权力 | 全局单例——通过 `MoneySupplyQuery` trait 只读查询。修改仅通过三管道 |
| base_money_creation_rate | **经济系统 008** | 稳定器E/通胀检测 | [TUNING] TOML可配。运行时仅稳定器E和通胀响应可修改 |
| 铸币权 (IssueCurrency) | **经济系统 006** (PowerAtom) | 经济系统 008 | 008消费006的铸币权查询——不自行定义谁能铸币 |
| 垄断检测 | **经济系统 008** | 001总纲/006权力/历史系统 | 检测到垄断→写入历史系统+触发信息传播（语言表达009）——不自行决定政治后果 |
| 税收回收 | **经济系统 008** | 001总纲 | 净回收率由008定义——各税收征收点（006权力、005体制）消费此参数 |
| 价格基准值 (base_value_copper) | **物品系统 001** | 经济系统 008 | 供给弹性/垄断检测/必需品刚性中的"基准价格"——全部以物品系统为权威 |
| 商路距离/运费 | **经济系统 003**（价格形成·待完成） | 经济系统 008 | 008消费运费计算结果——不自行计算 |
| 信息传播（垄断广播） | **语言表达 009** | 经济系统 008 | 008调用 `InformationPropagation::propagate()`——不自行实现传播机制 |
| NPC价格抵抗（记忆） | **经济系统 007** (NPC经济心智) × **NPC活人感** (MemoryStore) | 经济系统 008 | 008提供"异常收购检测"信号→007的锚定效应+记忆机制自动响应 |
| 经济事件写入 | **经济系统 008** | **历史系统** | `EconomicEventSink` trait ——历史系统实现。通胀警报/垄断检测→历史系统记录 |
