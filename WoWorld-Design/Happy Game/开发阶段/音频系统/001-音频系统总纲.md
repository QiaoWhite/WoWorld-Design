# 001 — 音频系统总纲 v1.0

> **核心设计哲学**：声音首先是物理现象，其次才是玩家体验。WoWorld 中任何有听觉的实体都独立感知世界中的声音，并据此做出行为反应——这发生在模拟层，不论玩家是否在场。音频模块是 WoWorld 的**声学真理引擎**——它定义声音如何从物理动作中涌现、如何在介质中传播衰减、如何被听者感知和掩蔽、如何被打断。游戏中的一切声音现象（脚步声、对话、音乐、提示音、环境声）均从这个引擎中涌现，零硬编码。

> **与渲染层的本质区别**：音频模块的 Rust 模拟层负责声音的**物理真实**（传播/衰减/掩蔽/打断/NPC 感知）——对标天气系统。Godot 渲染层负责声音的**播放真实**（AudioServer 空间化/混响/HRTF/播放调度）——对标天气的云渲染。两层通过 `AudioRenderPacket`（对标 `WeatherVisualPacket`）穿越 GDExtension 边界。

> **设计原则**：
> 1. **模拟优先**：声音首先存在于模拟层——玩家只是另一个听者。对标天气系统。
> 2. **拉模型**：各业务模块不知道音频存在。它们维护自己的物理数据（SoundFootprint/CurrentSpeech/action_log），音频模块主动轮询读取。零推送。
> 3. **零反向依赖**：音频模块不 import 任何业务模块——只通过 `woworld_types` 共享类型 + 已有 trait 查询。
> 4. **物理而非语义**：`SoundFootprint` 描述表面接触+运动+发声——不描述"挥剑"或"锻造"。映射规则完全在音频模块内部。
> 5. **涌现优先**：信息不对称（距离→听不清→误解）、中断（话说一半→只有前半句的记忆）、跨 L 层感知（L2 龙吼传至 L1）——全部从物理规则中自然涌现。
> 6. **完全 TOML 化**：材质→声音映射、提示音规则、音乐情境映射、传播算法选择——全部 TOML 可配置。Mod 替换 = 换 TOML + 换文件。
> 7. **对标已有模式**：WeatherQuery 轮询模式、ConsumableEffect schema 模式、HasAestheticSignal trait 模式、大日志全量记录模式、SettlementFaithSnapshot 聚合模式、WeatherVisualPacket 跨边界模式。

## §一 核心架构：双层模型 + 拉模型

### 1.1 双层模型

```
Rust 模拟层（声音作为物理现象）          Godot 渲染层（声音作为玩家体验）
─────────────────────────────────────  ─────────────────────────────────
SoundFootprint 写入（各业务模块）       AudioRenderPacket 消费（对标 WeatherVisualPacket）
  ↓                                       ↓
传播引擎 + 掩蔽引擎 + 打断仲裁            AudioStreamPlayer3D 空间化播放 + AudioServer 混音
  ↓                                       ↓
AudioQuery trait（30 个只读方法）         混响/HRTF/多普勒（Godot 原生）
  ↓
NPC/生物感知并反应（不论玩家是否在场）
```

**对标**：天气系统的 `WeatherSample`（Rust 模拟） → `WeatherVisualPacket`（Godot 渲染）。NPC 自主查天气→避雨；同理，NPC 自主查声音→警觉。

### 1.2 拉模型核心

各业务模块**零 import 音频**。它们维护自己的物理数据：

| 数据 | 谁写入 | 音频怎么拿到 |
|------|--------|-------------|
| `SoundFootprint`（实体物理声学快照） | 各实体模块（NPC/Life/Combat/Magic/...） | 音频模块每帧轮询读取 |
| `ActionRingBuffer`（瞬时动作记录） | 各实体模块（本来就有——给决策器/记忆用） | 音频模块多读一个消费者 |
| `CurrentSpeech`（正在说的话） | 语言表达模块 → `NpcData` | 音频模块每帧轮询检测 |
| `HasSoundEmitter::active_sounds()` | 各模块 impl trait | 音频模块每帧轮询 |
| `PlayerSoundFootprint` | Godot 填充 → GDExtension → Rust | 音频模块每帧接收 |
| `EnvironmentSoundSource` | 世界生成（Chunk 静态数据） | 音频模块按空间查询 |

## §三 声音溯源：从物理到声音

### 3.1 核心原则

**一个声音不是"脚步"——它是"表面接触×运动×环境"的物理组合。** 业务模块描述物理，音频模块映射声音。

### 3.2 SoundFootprint（定义在 woworld_types）

```rust
/// 实体的当前物理声学状态。
/// 各业务模块维护并更新——不知道音频存在。
/// 音频模块每帧轮询读取。
#[derive(Clone)]
pub struct SoundFootprint {
    /// 当前表面接触（双脚/多足踩在不同材质上）
    pub surface_contacts: SmallVec<[SurfaceContact; 4]>,
    /// 当前运动状态
    pub motion: MotionSnapshot,
    /// 当前主动发声
    pub vocalization: Option<VocalizationState>,
    /// 当前物品交互
    pub item_interaction: Option<ItemInteraction>,
    /// 是否在主动抑制声音
    pub silence: Option<SilenceIntent>,
}
```

### 3.3 映射流程示例：盔甲哐当

```
NPC 穿板甲行走在湿石板路上：
  ┌─ 物品系统: ItemProperties.audio_material = PlateArmor
  ┌─ 世界生成: SurfaceMaterial = Stone
  ┌─ NPC 模块: SoundFootprint { motion: { gait: Walk, velocity: 1.2 }, ... }
  ┌─ 生命系统: mass_kg = 95
  └─ 天气系统: precipitation = Rain(0.3)

        ↓ 音频模块组合（全在 propagation + assets 内部）

  SoundDef {
    base: sfx/materials/footstep/stone_walk,     // 石板路脚步
    intensity_mult: 95 × 1.2 = 1.3,
    overlays: [(sfx/materials/armor/plate_clank, 0.7)],  // 板甲哐当
    wet_modifier: { vol: 0.85, low_punch: 1.2, high_damp: 0.7 },
  }
```

**关键**：物品系统只存 `audio_material: PlateArmor` 这一个字段——对标 `ConsumableEffect` schema 模式（Life定义schema，物品存储不解）。音频模块完全拥有映射逻辑。

## §四 全局环形缓冲区与跨 L 层传播

### 4.1 环形缓冲区（对标大日志）

```rust
/// 全局环形缓冲区——记录最近 ~30s 的所有 SoundEvent。
/// 空间索引（16m 网格哈希）支持快速空间查询。
/// 单写者（音频模块主循环），零锁。
struct SoundEventBuffer {
    events: Vec<SoundEvent>,        // 预分配环形
    spatial_index: GridHashIndex,   // 16m cell → event_ids
    head: u32,
    capacity: u32,                  // ~15,000（30s 窗口峰值）
}

/// 世界中的声音事件——音频模块的唯一权威定义。
/// 所有声音（瞬时/持续/对话/环境）在进入环形缓冲区时归一化为 SoundEvent。
/// Copy 类型——~64B，栈上传递。
#[derive(Clone, Copy)]
pub struct SoundEvent {
    pub event_id: SoundEventId,
    pub source_entity: Option<EntityId>,   // None = 环境声/系统声
    pub source_pos: Vec3,
    pub source_velocity: Vec3,             // 用于多普勒计算（透传 Godot）
    pub sound_type: SoundType,             // Instant/Continuous/Voice/Ambient
    pub intensity_db: f32,                 // 基准声压级 @ 1m
    pub frequency: FrequencyProfile,       // 频率特征
    pub medium: Medium,                    // 声源所在介质
    pub emitted_at: GameInstant,           // 发射时间戳
    pub duration: Option<Duration>,        // 持续型：None = 瞬时
    pub cancel_reason: Option<CancelReason>,  // 被打断后写入
    pub metadata: u64,                     // 模块专用（压缩存储）
}

#[derive(Clone, Copy, PartialEq, Eq)]
pub enum SoundType {
    Instant,     // 瞬时——脚步/碰撞/机关（~10ms-1s）
    Continuous,  // 持续——引擎/结界/瀑布（数秒-永久）
    Voice,       // 话语——NPC说话/喊叫/唱歌
    Ambient,     // 环境——河流/风声（静态·永久）
}
```

### 4.2 跨 L 层传播

声音的 L 层独立于实体 L 层：

| 声源 L 层 | 入全局缓冲区条件 |
|-----------|----------------|
| L1 | 全部（无门槛） |
| L2 | `can_reach_l1` 判定：intensity - attenuation > MIN_AUDIBLE_SPL(10dB) |
| L3 | intensity ≥ 120dB（龙吼/爆炸/火山级） + 标传播延迟 |
| L4 | 不逐实体——EnvironmentSoundSource 覆盖 |

**L2 听者用聚落聚合**：SettlementAudioAggregate（每游戏日更新一次，惰性），对标 SettlementFaithSnapshot 模式。

### 4.3 强度门控公式

```rust
fn can_reach_l1(event: &SoundEvent, nearest_l1_listener: Vec3) -> bool {
    let distance = (event.source_pos - nearest_l1_listener).length();
    let attenuation = 20.0 * distance.log10().max(0.1)
        + atmospheric_absorption(event.frequency, distance, event.medium);
    event.intensity_db - attenuation > MIN_AUDIBLE_SPL  // 默认 10 dB
}
```

## §五 AudioQuery trait

### 5.1 设计

音频模块定义 `AudioQuery` trait（在 `woworld_types` 中声明签名，在音频 crate 中实现）。30 个只读方法，零分配，对标 `WeatherQuery` / `CultureQuery`。

### 5.2 核心方法分组

**感知查询（per-listener）**：
```rust
pub trait AudioQuery: Send + Sync {
    /// 听者感知到的所有声音（瞬时+持续+环境，统一处理）
    fn perceived_sounds(&self, listener_pos: Vec3, hearing: &HearingModel,
                        time_window: Duration, attention: &AttentionFilter,
                        head_medium: Medium) -> Vec<PerceivedSound>;

    /// 对特定说话者的话语感知状态（含时间维度）
    fn perceived_speech(&self, listener_pos: Vec3, speaker: EntityId, now: GameInstant,
                        hearing: &HearingModel) -> Option<SpeechPerception>;

    /// 当前位置的环境噪声水平（per-band）
    fn ambient_noise_level(&self, pos: Vec3, time: GameInstant) -> AmbientNoiseLevel;

    /// 当前位置的声学空间（混响参数）
    fn acoustic_space(&self, pos: Vec3) -> AcousticSpace;
}
```

**传播查询**：

## §六 对话的时间维度

### 6.1 内容与声音分离

| 层 | 归属模块 | 内容 |
|----|---------|------|
| 对话内容（说什么） | 语言表达 | ExpressionRef + 文本 + resolve_partial() |
| 对话声音（怎么被听到） | **音频模块** | 传播 + 时间 + 清晰度 + fraction_heard |

**对标**：书的内容归语言表达，书的物理属性（木板/羊皮纸）归物品系统。

### 6.2 时间差

```
NPC A 开始说："我昨天在森林里发现了一座废弃的神殿"（12词，3词/秒 = 4秒）
                                                      
t=0.0s  A 开始说话
t=0.3s  3m外的B — 声音传到（延迟 0.009s ≈ 0）— 听到"A我昨"（1词传到）
t=1.0s  B — 声音传到 — 听到"我昨天在森林"（3词传到）— fraction_heard = 0.25
t=2.0s  B — 听到"我昨天在森林里发现了"（6词）— fraction_heard = 0.5
t=2.5s  A 被桌子砸中昏迷 — cancel_reason = CombatImpact
        语言表达设 current_speech = None
t=2.5s  B 查询 perceived_speech → fraction_heard = 0.5, clarity = 0.85, is_ongoing = false, cancel_reason = CombatImpact
        B 调 lang.resolve_partial(0.5, 0.85) → "我昨天在森林里发现了"
        B 写入记忆：PerceivedUtterance { text: "我昨天在森林里发现了", was_interrupted: true }
```

### 6.3 话语清晰度五档

| 档位 | clarity | resolve_partial 返回 | 例子 |
|------|---------|---------------------|------|
| Perfect | ≥0.9 | 完整文本截取到 fraction | "我昨天在森林里发现了一..." |
| Clear | 0.7-0.9 | 完整文本，边缘模糊 | "我昨-天在森林里发现了..." |
| Garbled | 0.4-0.7 | 关键词保留，连接词丢失 | "我...森林...发现...废弃...神殿" |
| Fragment | 0.1-0.4 | 仅主题方向 | "好像说了关于森林的什么" |
| Inaudible | <0.1 | 无内容——仅方位感知 | "隐约听到说话声（来自东边）" |

## §七 打断机制

### 7.1 打断原因

```rust
pub enum CancelReason {
    SourceDied,              // 源死亡
    SourceUnconscious,       // 源昏迷
    SourceTeleported,        // 源传送
    SourceSubmerged,         // 源入水
    CombatImpact,            // 外部物理打断
    Preempted(SoundEventId), // 更高优先级抢占
    MagicallySilenced,       // 沉默法术
    SourceVoluntaryStop,     // 正常说完（非打断，是完成）
}
```

### 7.2 五种打断来源

| 来源 | 触发 | 仲裁方 |
|------|------|--------|
| 源实体状态变化 | 死亡/昏迷/传送/入水 | 音频模块每帧检查 |
| 更高优先级事件 | 战斗警报/Critical | VoicePriority 五级队列 |
| 距离变化 | 源超出听觉范围 | 传播模型动态计算 |
| 环境覆盖 | 雷声/爆炸（环境噪声覆盖） | 掩蔽引擎 |
| 听者状态变化 | 听者昏迷/传送 | 听者侧过滤 |

| ④ | perception | 感知查询 | PerceptionEngine |

## §十三 audio_lod 五档退化契约 ★CHG-049

> **关联**: [[../../../Change/CHG-049-LOD架构全面深化-20260620|CHG-049 §七]] · [[../模块接头总览/00-全局基础设施/001-核心类型注册表|LodPrescription]]

audio_lod 是 `LodPrescription` 的 7 个维度之一。CHG-049 将其从原 4 档(0-3)扩展为 5 档(0-4)，与其他角色维度对齐。

### 13.1 基础距离带（被动衰减——纯距离驱动）

| audio_lod | 距离 | 传播精度 | 失真类型 | NPC 记忆后果 |
|-----------|------|---------|---------|-------------|
| 0 | 0-30m | 全传播——完整 Utterance + 语气 + 音色 + 方向 | 无失真 | 精确记忆 |
| 1 | 30-100m | 简化衰减——内容清晰可懂，丢失精细音色/语气副词 | 轻微降采样 | 高保真记忆 |
| 2 | 100-300m | 仅事件声——可辨声源类型和内容方向，不可逐字理解 | 语义压缩 | 可能错记修饰语 |
| 3 | 300-1000m | 可检测无内容——仅方向+响度级别 | 主题抽取 | 谣言级记忆（"好像有人在说X"） |
| 4 | 1000m+ | 静默——不可检测（无意图提升时） | — | 无记忆 |

**设计依据**：声音传播天然比视觉远——听觉在动物演化中是远程预警系统。因此 audio_lod 的基础距离带比 char_lod 其他维度宽：

| 参照 | 0-15m | 15-60m | 60-200m | 200-800m | 800m+ |
|------|-------|--------|---------|----------|-------|
| char_lod 其他维度 | LOD 0 | LOD 1 | LOD 2 | LOD 3 | LOD 4 |
| **audio_lod** | **LOD 0** | **LOD 1 (30-100m)** | **LOD 2 (100-300m)** | **LOD 3 (300-1000m)** | **LOD 4 (1000m+)** |

### 13.2 意图驱动覆盖（主动投射——声源提升有效 LOD）

audio_lod 可以被声源的**交际意图**（CHG-018 `CommunicationIntent`）主动拉升：

```
effective_audio_lod = max(0, base_audio_lod
                           - intent_boost_steps
                           - hearer_attention_bonus)

intent_boost_steps:
  whisper:      0   (比正常衰减更快——见 §13.3)
  normal:       0
  loud:         1   (穿透 1 个 LOD 带)
  shout:        2   (穿透 2 个 LOD 带)
  declamation:  3   (广场公讲/战场指挥/钟声警报——穿透 3 个 LOD 带)

hearer_attention_bonus:
  睡眠=0, 专注工作=0, 闲散=1, 专注聆听=1
  hearing_acuity: 年龄派生 0.0-1.0 (Gompertz 衰老模型已有)
  ambient_noise: 市集高衰减/森林低衰减 (WeatherQuery 已有)
```

**实例**：
- NPC 广场公讲（declamation），距离 150m → base=2, -3 boost → effective=0 (完整传播)
- 战场喊"冲锋"（shout），距离 300m → base=3, -2 boost → effective=1 (简化可懂)
- 酒馆正常聊天（normal），距离 8m → base=0, -0 boost → effective=0

### 13.3 耳语的反向修正

耳语是**衰减更快**而非传播更远。通过 `env_acoustic_modifier` 实现：

```
whisper → env_acoustic_modifier × 3.0  (有效距离缩至 1/3)
  → 正常在 5m 外可听到的声音，耳语时 1.7m 外就衰减到不可闻
```

### 13.4 非语言声源 — AcousticProjection

非语言响亮声源（铁锤锻打、钟楼钟声、瀑布轰鸣、巨兽吼叫）不通过 CommunicationIntent——通过 **`AcousticProjection`** 结构（见 [[003-传播引擎#§X AcousticProjection|003 §X]]）——基于声学物理而非语言意图：

- `base_loudness: f32` (dB) — 每 +10dB → 有效距离 ×0.7
- `directionality: Directionality` — 全向/扇形/束状
- `freq_profile: FreqProfile` — 低频远传(×0.8) / 高频快衰减(×1.5)

### 13.5 失真链 → 谣言系统（故事涌现驱动力）

audio_lod 的失真不是 bug——是 WoWorld 故事生成的**核心驱动力**：

```
NPC A 在 250m 外听到 NPC B 说"国王...受伤...北方..."
  → audio_lod=2 → 语义压缩 → 关键名词保留，修饰语丢失
  → NPC A 的认知系统补全： "国王在北方受伤了"
  → NPC A 告诉 NPC C："国王在北方战死了" (脑补升级)
  → 谣言扩散 → 城市恐慌 → 经济波动 → 故事涌现
```

这不是需要修复的"信息丢失"——这是需要保留和模型化的**涌现机制**。CHG-018 的"五传播通道+失真算子"已经定义了失真语法——audio_lod 决定了失真算子的强度。

### 13.6 AudioBroadcast 广播优化

广场公讲/钟声/警报——单一声源 → 大量听众。不需要逐对计算：

```rust
struct AudioBroadcast {
    source_position: Vec3,
    source_entity: Option<EntityId>,
    acoustic: AcousticProjection,
    intent: Option<CommunicationIntent>,  // None = 非语言声源
    max_range: f32,
}

// LODCoordinator 处理（每帧）：
// 对每个 AudioBroadcast：
//   for entity in entities within max_range：
//     计算 effective_audio_lod → 写入 LodPrescription
// 单次传播计算 → N 个接收端共享
```
