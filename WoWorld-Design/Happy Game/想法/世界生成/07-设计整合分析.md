## 一、设计冲突与解决方案

### 🔴 冲突1：Marching Cubes 连续地形 vs. "放置/移除方块"

**来源**：`总设计草稿.md` §3.1 vs. NPC交互  

**问题**：设计文档多处说：
- "连续体素地形（类似Valheim/雾锁王国），非方块状" 
- "使用Marching Cubes进行等值面提取"
- 但同时阶段〇说 "玩家能放置/移除体素方块"
- 同时NPC有"挖矿""建造"行为

Marching Cubes 地形**不是方块网格**——你无法像 Minecraft 那样"挖掉一个方块"。MC等值面提取的是连续表面，修改它需要 SDF 操作（加/减密度函数），不是"删除方块"。

**严重程度**：🔴 高——这是整个世界的物理基础。

**解决方案**：

```
区分两种编辑模式：

模式A：地形雕刻（SDF加减）
  → 玩家用镐/铲"挖掘"地形
  → 实现：对体素密度场做 SDF 减法
  → 结果：自然凹陷/洞穴/斜坡
  → 对应：挖矿、挖地基、挖战壕

模式B：建筑放置（网格叠加）
  → 玩家"放置"建筑模块
  → 实现：在地形上实例化网格节点 + 碰撞体
  → 结果：房子、墙壁、地板
  → 对应：建造系统
  → 建筑模块不是体素，是网格（见 体素设计决策.md）

废弃概念："体素方块"——这个词从设计文档中删除。替换为"地形雕刻"。
```

**文档修改**：
- `总设计草稿.md` §3.1：删除"体素方块"提法，统一为"连续体素地形+地形雕刻"
- "玩家能放置/移除体素方块" → "玩家能用工具雕刻地形"

### 🔴 冲突2：世界规模 vs. 体素精度 vs. 存储

**来源**：`总设计草稿.md` §3.2 + `技术架构.md` §2

**问题**：
- 世界：16km × 16km
- 体素精度：0.25-0.5m³
- 以 0.5m 精度计算：32000 × 32000 = 1,024,000,000 个水平格
- 假设平均高度128格 → 约 1300亿个体素
- 每体素4字节 → 约 500GB 未压缩
- 即使 SVO 压缩 20:1 也是 25GB——**对于独立游戏太大了**

**严重程度**：🔴 高——不解决则无法实现。

**解决方案**：

```
改世界规模为渐进式：

初始可玩区域：4km × 4km
  → 约 64,000,000 个地表体素格
  → SVO压缩后约 500MB-1GB

可扩展区域：最多 16km × 16km（后期）
  → 但不是一次性生成
  → 使用"世界边界扩展"机制
  → 玩家到达边界时，从种子生成新区块
  → 类似 Minecraft 的动态加载但带边界

实际游玩中，玩家在单个存档内活跃范围很少超过 4km²
（Kenshi 地图约 870km² 但极少有玩家走遍全图）

存储预算：
  区块文件（活跃区域）：~500MB
  区块文件（非活跃）：压缩存档，~200MB
  NPC 数据：~200MB
  网格资产：~1-2GB
  总存档大小：< 3GB
```

**文档修改**：`总设计草稿.md` §3.2 世界规模改为渐进式。`技术架构.md` 更新区块常量。

### 🟡 冲突3：NPC 复杂度 vs. 帧预算

**来源**：`总设计草稿.md` §4.3 + `技术架构.md` §7

**问题**：L1 NPC（当前阶段≤50个，远期目标~1,000个，距玩家50m内）每个都跑完整的：
- 感官外壳（视觉射线检测）
- 情绪引擎（8+30种情绪更新）
- 记忆检索（200条记忆搜索）
- GOAP规划器（A*搜索行动链）
- 概率决策（加权采样）

声称"每0.2s更新一批"，但 50 个 NPC ÷ 10个/帧 ÷ 60fps = 每0.3秒才能更新完一轮。而 GOAP 规划在最坏情况下（搜索深度8，每步6个可选动作）是 6⁸ ≈ 1.6M 种可能——即使在 C++ 中也不能每帧跑。

**严重程度**：🟡 中——当前参数不现实但方向正确。

**解决方案**：

```
重新预算 NPC 更新频率：

决策更新（真正费CPU的部分）：
  GOAP 规划：仅在目标变化时运行 —— 平均每 30-120 秒/NPC一次
  概率决策：每次决策 < 0.05ms —— 每 10-30 秒/NPC一次
  → 日常行为一旦选定就持续执行数分钟

情绪更新（纯数学，便宜）：
  每 1-2 秒/NPC → 50 NPC 分摊到 60 帧 ≈ 1个/帧

记忆写入（偶发）：
  仅在事件发生时触发 —— 平均每 NPC 每日 3-10 条

感官更新（射线检测，较贵）：
  视觉射线：每 2-3 秒/NPC，每帧 ≤ 2 条射线
  听觉：事件驱动，非轮询

修正后的每帧CPU预算：
  感官射线：2条 × 2ms = 4ms
  情绪更新：1个 × 0.1ms ≈ 不计
  决策：平均每帧 < 1ms（分摊后）
  记忆：平均每帧 < 1ms
  → 总计：< 8ms/帧 — 可行
```

**文档修改**：`技术架构.md` §7 更新NPC更新预算。

### 🟡 冲突4：模板建筑 vs. "一切从种子生成"

**来源**：`建筑生成规则0.5` Canvas + `03-建筑生成规则.md` vs. `总设计草稿.md` "种子决定一切"

**问题**：建筑生成规则说90%民居用模板——但模板从哪儿来？模板是手建的还是程序化的？如果手建模板，那就不是"种子生成一切"。

**解决方案**：

```
模板 = 参数化生成规则，不是预制3D模型。

"模板"的含义：
  → 一个"铁匠铺模板" = 规则集：
    { 必须有: 锻炉(核心), 铁砧, 淬火槽
      通常有: 工具架, 煤炭堆
      面积: 30-60m²
      布局规则: 锻炉居中, 铁砧紧邻, 淬火槽靠墙
      文化覆盖: 屋顶→文化屋顶样式, 墙→文化材料 }
  → 3D模型由这些规则实时生成，不是加载.glb文件

手建内容仅限：
  - 关键文化资产（如"哥特式教堂"的屋顶模块）
  - 特殊物品模型（如龙的模型）
  - UI元素
```

**文档修改**：`03-建筑生成规则.md` 加入模板定义澄清。`总设计草稿.md` §3.2 明确"模板=生成规则"。

### 🟡 冲突5：战斗即时制 + 信息战心理模块 = 巨大实现复杂度

**来源**：`战斗.md` + `总设计草稿.md` §6.5

**问题**：战斗设计要求：
- 物理碰撞判定（不是简单的hitbox检测）
- NPC心理模块持续运作
- 信息战（实力推测/虚张声势/战斗意志）
- 空战
- 环境交互
- 10元素魔法

这是 AAA 级别的战斗系统设计。独立开发者做这么多系统 = 每个都做不深。

**解决方案**：

```
战斗分三个层次逐步实现（战斗系统内部的逐步完善）：

战斗 MVP：
  - 简单碰撞盒判定（非物理模拟）
  - 近战：轻攻击/重攻击/格挡
  - 远程：射线检测
  - 血条 + 基础伤害公式
  - 心理模块：仅"战斗意志"一个值（士气高/低）
  → 目标：打架能玩

战斗进阶：
  - 物理碰撞→hitbox + 惯性考虑
  - 元素魔法：火/冰/雷三种先行
  - 战斗风格：从人格推导（简单if-else）
  - 信息战基础：敌方HP不可见，通过动画判断
  → 目标：打架有深度

战斗完整：
  - 完整10元素魔法
  - 完整信息战
  - 空战
  → 目标：打架有史诗感
```

**文档修改**：`总设计草稿.md` §6 加入战斗渐进实现说明。

### 🟢 冲突6：LLM 增强 vs. 离线完整可玩

**问题较小**：设计已经清楚说明LLM是可选的。但具体交互设计（对话界面、NPC决策）中，需要始终提供非LLM的fallback路径。

**解决方案**：每个支持LLM的交互都有对应的非LLM版本。对话选项=预制选项（基础）+LLM生成（可选增强，显示为"更多对话..."按钮）。

## 四、核心障碍与对策

### 🔴 障碍1：Solo 知识广度

**问题**：需要掌握的领域：
- 体素算法（Marching Cubes/Transvoxel/SDF操作）
- AI规划（GOAP/效用AI/概率采样）
- 程序化生成（噪声函数/水文模拟/L-system）
- 3D渲染管线
- 游戏设计
- AI美术管线
- LLM集成
- 数据库/存档
- UI/UX

每项都是独立学科。

**对策**：
- 按模块逐个攻克。不要一次做太多
- 体素部分：Zylann Voxel Tools 已经解决了最难的部分。在它基础上做，不是自己写体素引擎
- GOAP：经典GOAP算法有大量开源实现参考
- 程序化生成：已有成熟噪声库（FastNoiseLite），不需要自己推导数学
- **关键策略：站在巨人肩膀上。用插件、用开源、用教程。**

### 🔴 障碍2：资产生产瓶颈

**问题**：一个人需要制作：
- 角色模型（多种族×多体型）
- 服装（多文化×多阶层）
- 建筑模块（多文化×多类型）
- 家具/道具（数百种）
- 植被（多生物群系）
- 武器/工具/物品
- UI图标
- 特效

即使 AI 辅助，这也是巨大的工作量。

**对策**：
- 早期开发：**纯色块 + 简单几何体**（胶囊体NPC、立方体建筑、球体树）
- 逐步替换为 AI 生成模型
- 建筑模块：先只做 1 种文化风格，验证通过后扩展到 3-4 种
- 使用 Geometry Nodes (Blender) 批量生成变体
- **"先丑后美"策略——让游戏先能玩，好看是后来的事**

### 🟡 障碍3：涌现测试不可自动化

**问题**：涌现行为是否正确、有趣、不崩溃——这些只能通过长时间观察发现。

**对策**：
- 内置"模拟快进"模式——加速游戏时间观察长期行为
- "涌现行为观察日志"（已在开发流程中提及）——记录所有惊喜/奇怪的行为
- 每次 playtest 后记录：NPC A 为什么做了 X？这合理吗？
- 设置自动化边界检测：NPC不可能的行为（同时死亡+存活、负数量物品等）

### 🟡 障碍4：Godot 体素插件长期的未知

**问题**：Zylann Voxel Tools 是社区维护的。如果某天停止更新……

**对策**：
- 从第一天起就封装体素 API（`terrain_interface.gd`）
- 所有体素操作通过接口调用，不直接依赖插件 API
- 如果换体素方案，只改封装层

### 🟢 障碍5：资金与时间

**问题**：3-6年的开发周期，独立开发者需要维持生活。

**对策**：
- 首个可玩版完成后开始发布 devlog
- 达到足够完成度后开启 Early Access
- 阶段性成果可以产生社区关注
- 考虑 Patreon / 爱发电 等支持渠道
