# 005-GOAP与分层AI决策系统参考

> 对应设计：WoWorld NPC 决策架构——90%概率行为树（日常）+ 9% GOAP规划（安全网）+ 1% LLM增强（可选）

---

## 一、决策架构总览

WoWorld 的 NPC 决策分为三个层次：

```
┌──────────────────────────────────────────┐
│ L3: LLM增强 (~1%)                        │
│ 复杂社交对话、道德困境、创意问题解决       │
│ 异步调用、结果验证、离线兜底               │
├──────────────────────────────────────────┤
│ L2: GOAP安全网 (~9%)                      │
│ 生存危机、目标受阻、环境剧变               │
│ A*搜索行动链、效用评估、分帧异步           │
├──────────────────────────────────────────┤
│ L1: 概率行为树 (~90%)                     │
│ 日常行为、习惯、社交常规                   │
│ 加权采样、低延迟、无需规划                 │
└──────────────────────────────────────────┘
```

**为什么不是 100% GOAP？**
- GOAP 的每次规划都是 A* 搜索——即使优化后，10万NPC同时规划也不可行
- 大多数日常行为不需要规划——NPC去常去的酒馆、走常走的路、做常做的事
- 概率决策给人的感觉并不"笨"——人类的大量行为本身就是习惯性的

## 二、GOAP（Goal-Oriented Action Planning）参考

### 2.1 GOAP 的基本原理

GOAP 的本质是逆向规划：
1. 有一个**目标**（如：饥饿值降到安全线以下 → 目标：吃饱）
2. 有一个**当前世界状态**（我在铁匠铺，我认识面包师，我有25枚铜币）
3. 规划器从目标倒推：要吃饱→需要食物→获得食物的方式有几种（买、讨、偷、自己做）→每种方式的前置条件是什么→选择最优路径

与行为树的关键区别：
- 行为树：设计师预定义了"NPC饿了的应对方式"
- GOAP：NPC根据当前世界状态，自己找到一条从现状到达成目标的路径

### 2.2 GOAP 的性能挑战与对策

| 挑战 | 描述 | WoWorld 的应对 |
|------|------|-------------|
| **搜索空间爆炸** | 深度8 × 每步6选项 = 1.6M种可能 | 硬超时2ms；裁剪明显不可行的动作；缓存常用规划 |
| **同时规划数** | 1000 L1 NPC 同时在规划 | 仅目标变化时触发（平均30-120秒/NPC）；同时异步规划上限8个 |
| **GDScript 性能** | 脚本语言的CPU限制 | 先GDScript实现，性能瓶颈确认后考虑C++/C#迁移 |
| **状态快照一致性** | 规划期间世界状态可能已变化 | 规划开始时对世界状态做快照；规划完成后验证快照是否仍然有效 |

### 2.3 GOAP 的动作设计原则

每个 GOAP 动作必须定义：
- **前置条件**（Preconditions）：什么世界状态必须为真才能执行
- **效果**（Effects）：执行后世界状态如何改变
- **成本**（Cost）：执行这个动作的代价（时间、资源、风险）

```
示例动作：去面包店买面包
  前置条件：
    - 面包店正在营业
    - NPC 有 ≥ 5 铜币
    - 面包店在可到达范围内
  效果：
    - 获得1个面包
    - 铜币 -5
    - 饥饿值 -40
  成本：
    - 时间：10分钟
    - 金钱：5铜币
    - 社交：与面包师短暂互动
```

**设计注记**：
- 动作的成本函数应该考虑人格参数（高宜人性的NPC给乞丐食物成本更低；高神经质的NPC认为"偷"的成本更高因为被抓的恐惧）
- 同一个动作对不同NPC有不同的"成本感知"——这让人格影响实际决策

### 2.4 GOAP 参考资源

- Jeff Orkin "Applying Goal-Oriented Action Planning to Games"（2003）— 原始GOAP论文
- F.E.A.R. (Monolith, 2005) — 首个商业GOAP实现
- Killzone 系列 — 商业游戏中的GOAP应用
- 开源GOAP实现：ReGoap (Unity)、GDGOAP (Godot)

## 三、概率行为树参考

### 3.1 为什么是"概率"行为树

传统行为树是确定性的——条件→动作。WoWorld 的日常行为是概率性的——条件+概率→动作。

```
传统行为树：
  IF 饥饿 AND 有食物 → 吃
  （在相同条件下总是做相同的事）

概率行为树：
  IF 饥饿 AND 有食物：
    吃 (60%)
    先做完手头的事再吃 (25%)
    忍着，等更好的食物 (10%)
    去找朋友一起吃 (5%)
  （在相同条件下可能做不同的事——不可预测=看起来聪明）
```

### 3.2 概率权重的影响因素

每次行为选择的概率权重受以下因素动态调整：

| 因素 | 示例影响 |
|------|---------|
| 人格 | 高尽责性→"先做完手头的事再吃"权重↑ |
| 当前情绪 | 心情好→"去找朋友一起吃"权重↑ |
| 最近行为历史 | 刚刚吃过→"忍着"权重↑；很久没社交→社交类行为权重↑ |
| 时间/地点 | 夜间→"去酒馆"权重↑；集市日→"逛街"权重↑ |
| 社会关系 | 附近有朋友→社交行为权重↑；附近有仇人→回避行为权重↑ |

### 3.3 行为持续时间

行为一旦选定，NPC 进入"执行模式"——持续数分钟到数十分钟不等。这避免了"每帧都在犹豫该做什么"的感知。

- 微行为（喝水、看窗外）：1-3分钟
- 中等行为（吃一顿饭、锻造一件小物件）：10-30分钟
- 长行为（种一季田、建造房屋）：数小时到数天

在行为执行期间，只有高优先级事件（生存危机、紧急社交）才能中断。

## 四、分层AI架构的参考

### 4.1 LOD AI 的核心思想

不同距离的NPC需要不同深度的AI——不是性能妥协，而是"注意力经济学"：

- 玩家真正在看的NPC：需要完整的可信度
- 玩家余光中的NPC：只需要看起来在做合理的事
- 玩家视野之外的NPC：只需要统计数据保持一致性

### 4.2 参考案例：GTA V 的 NPC LOD

GTA V 的 NPC 在玩家视野之外时会大幅简化甚至消失——但它们的位置和状态被记录，当玩家转回时"恢复"。

WoWorld 的不同在于：我们的NPC即使在L3状态下也需要保持数据的连续性——因为离线模拟和命运推演在玩家不在场时仍然在运行。

### 4.3 参考案例：The Sims 的"故事推进"

The Sims 3 的"故事推进"系统在后台处理不在当前家庭的 Sims 的生活：
- 他们可能换工作、结婚、生子、搬家
- 当玩家切换到这个家庭时，发现世界已经变了

这是 WoWorld L3模拟的核心参考。

## 五、LLM 增强的边界设计

LLM（大语言模型）是可选增强，仅对 L1 NPC 开放且使用受限：

**适合 LLM 的场景**：
- 玩家主动与 NPC 深度对话（非战斗）
- NPC 面临道德困境需要"思考"
- 生成个人日志/日记（异步、不阻塞）

**不适合 LLM 的场景**：
- 战斗决策（延迟不可接受）
- 日常行为选择（概率驱动就够了）
- 任何实时性要求高的场景

**LLM 输出的安全边界**：
- LLM 只能输出对话文本或行为倾向（如"NPC倾向于接受这笔交易"）
- 不能直接操作游戏状态（NPC不能因为LLM说"我杀了他"就真的杀人——那个行为必须通过正常的GOAP/战斗系统）
- 所有LLM输出被记录日志，方便 debug

> **对 WoWorld 而言**：AI 系统的目标不是"让NPC看起来最聪明"——而是"让NPC看起来最像人"。人类的大部分行为是习惯性的、概率性的、有时候自己都说不清为什么。完美的理性规划反而让人觉得不像人。90%的概率驱动不是性能妥协——它是人性模拟的正确方式。
