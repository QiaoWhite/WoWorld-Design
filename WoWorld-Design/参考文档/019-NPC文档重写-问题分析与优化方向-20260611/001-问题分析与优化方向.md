## 一、重写中发现的潜在问题

### 1.1 trait 边界的模糊性

**问题**：`DecisionEngine` trait 的 `select_action` 同时需要 `NpcData`、`SubjectiveWorld` 和 `Rng`。在 rayon 并行上下文中——`NpcData` 是互斥借用（每个 NPC 独立），但 `SubjectiveWorld` 是所有 NPC 共享的只读数据。Rust 的生命周期标注在 trait 层面可能产生复杂的泛型约束。

**影响**：如果 `SubjectiveWorld` 需要在 `select_action` 中传递 `&self` 引用，每个并行线程都需要访问同一个世界状态快照——这在 rayon 中可行（只读共享），但需要确保世界状态在整帧内冻结。

**建议**：每个帧开始时生成一次 `WorldSnapshot`（只读 Arc），在帧内所有 NPC 更新中共享。字段命名从 `SubjectiveWorld` 改为 `WorldSnapshot` 以明确其"帧级只读"的语义。

### 1.2 情绪感染的竞争条件

**问题**：§2.1.5 的情绪感染在 NPC 并行更新后串行执行。但如果某个 NPC 在并行更新中被标记为 L1→L2 降级（距离变化），而情绪感染仍在操作该 NPC——可能访问到已经不正确的模拟等级。

**影响**：情绪感染需要明确的操作对象范围——"仅 L1 NPC"——且升降级调度必须在情绪感染之前完成。

**建议**：调整帧内执行顺序：
```
1. 升降级调度 (确定本帧 L1/L2/L3/L4 集合)
2. NPC 心智并行更新 (L1 + L2)
3. 情绪感染串行执行 (仅 L1, <15m 距离)
4. 战斗 AI 并行更新
5. 渲染数据打包 → Godot
```

### 1.3 L4 NPC 的"存在感"表达

**问题**：L4 NPC（超远距数百万）没有个体实例，仅有区域宏观统计。但玩家需要感知到"远方有文明"——当前通过城市灯光和炊烟。然而炊烟/灯光的密度和位置如何从宏观统计（区域人口/繁荣度）反向生成视觉锚点，尚未明确。

**影响**：如果 L4 → 视觉锚点的映射不自然——玩家可能觉得"远方的城市看起来都一样"——失去"远方的文明有差异"的感觉。

**建议**：定义 `L4VisualAnchor` 生成规则：
```
城市灯光密度 = f(区域人口密度, 繁荣度, 文化技术等级)
炊烟密度 = f(区域人口密度, 季节, 是否用餐时间)
天际线轮廓 = f(城市规划类型, 文化种子)
```
这些锚点不需要精确对应个体 NPC——只需要在统计上是正确的。

### 1.4 载具上 NPC 的 NavMesh 更新频率

**问题**：载具在移动时——载具上的局部 NavMesh 需要随载具位置更新。如果船在海上以 15 m/s 移动，载具上的乘客在甲板上走动——他们的寻路网格需要反映船体的摇摆和倾斜。

**影响**：如果 NavMesh 更新太慢——乘客可能在甲板上"穿透"船舷或在暴风雨中正常行走（看起来不自然）。

**建议**：
- 载具局部 NavMesh: 仅在舱体结构变化时完全重建（罕见）
- 船体摇摆: 在 NPC 的骨骼矩阵中直接施加 `PoseOffset`（倾斜补偿），不重建 NavMesh
- 暴风雨: 乘客移动速度 ×0.6，随机方向扰动 ×0.2，模拟"甲板摇晃行走"

### 1.5 天象感知的"可观测性"差距

**问题**：§5.3 定义了天象感知的完整数据流（天气→SkyPerception→NPC 决策器权重偏移）。但如 017 测试 Phase 3 的"人格可观测性"问题所示——数据上的差异必须在 A 面可观测才有意义。NPC 因为天象而调整了行为——但玩家能否感知到"这个 NPC 因为暴风雨前兆而加速了收割"？

**影响**：如果天象感知的 A 面可观测性不足——这个系统就只是一个"后台数值游戏"。

**建议**：
- 在 NPC 对话模板中加入天象感知触发词（"暴风雨要来了，我得赶紧收庄稼"）
- 在天象剧烈变化时（暴风雨前兆 / 彩虹出现）——触发 NPC 的"观天"微姿态（抬头、停顿），让玩家看到"他们在注意天气"
- 多人同时做出天象反应 → 集体行为的视觉信号（"整个市场的人都在抬头看天"）

### 1.6 蓝图施工的"等待"体验

**问题**：玩家委托 NPC 建一栋房子——需要 5-10 个游戏日。在这期间——玩家"无事可做"地看着施工进度条慢慢涨？如果施工期间没有任何可观察的变化——体验会很差。

**影响**：施工是"生活的一部分"（铁匠档的核心内容之一）——如果施工过程不可见，就失去了"我在建立自己的生活"的感觉。

**建议**：
- 施工分阶段可视化：
  - Day 1-2: 地基开挖（地面变化 + NPC 工人在场）
  - Day 3-5: 框架搭建（建筑骨架可见）
  - Day 6-8: 封顶+外墙（建筑轮廓完整）
  - Day 9-10: 内饰+细节（窗户、门、屋顶瓦片）
- 玩家可以在任何阶段走到施工现场——看到 NPC 在工作、听到施工音效
- 施工期间 NPC 工人会在工地附近活动（临时增加该区域的 NPC 密度）

### 1.7 战斗 AI 的"参与感"落差

**问题**：半自动战斗的理念是"玩家做高层决策，AI 执行战术"。但如果一场 45 秒的战斗中——玩家只做了 2 次决策（切换风格+释放技能）——其余 40 秒是看着 AI 在打——"参与感"是否足够？

**影响**：如果决策密度太低 → 玩家觉得自己是"观众"而非"参与者"。

**建议**：
- 增加"微决策"频率——在战斗的关键时刻暂停或减速（类似子弹时间），让玩家做决策：
  - 敌方释放大招前 0.5s → 弹出抉择："格挡/躲避/硬接"
  - 队友 HP<20% → 弹出抉择："掩护队友/继续进攻/撤退"
  - 敌方士气崩溃 → 弹出抉择："追击/放走/招降"
- 这些微决策不打断战斗节奏——但保证了玩家在战斗中的"参与密度"。

## 二、架构层面的优化方向

### 2.1 WorldSnapshot 模式

将当前设计中分散的"世界状态查询"统一为 `WorldSnapshot`：

```rust
/// 每帧开始时从 WorldState 冻结的只读快照
/// 所有 NPC 决策器共享同一个 Arc<WorldSnapshot>
pub struct WorldSnapshot {
    pub time: GameTime,
    pub weather: WeatherState,
    pub positions: BTreeMap<NpcId, Vec3>,     // 所有 NPC 位置 (L1+L2)
    pub public_events: Vec<PublicEvent>,       // 本帧的公开事件
    pub spatial_index: SpatialIndex,           // 快速空间查询
    pub road_network: Arc<RoadNetwork>,
    pub ocean_state: OceanSnapshot,            // v3
    pub sky_state: SkySnapshot,                // v3
}
```

**好处**：消除决策器中的"世界状态可变借用"问题，所有 NPC 并行读取同一份不可变数据。

### 2.2 事件类型系统统一

当前 v2.0 中事件类型分散在多处：记忆的 `EventType`、GOAP 的 `Goal`、决策器的 `Action`。三者之间有隐含的映射关系但未显式定义。

**建议**：定义统一的 `ActionCategory` 枚举（已存在于决策器中），并为每种 `ActionCategory` 显式定义：
- 对应的 `EventType`（执行后产生什么记忆事件）
- 对应的 GOAP `Goal`（该行动类别能满足哪些目标）
- 对应的决策权重来源（文化/习惯/人格/情绪/天象/载具 各自贡献多少）

### 2.3 性能的进一步优化路径

| 优化项 | 当前状态 | 目标 | 优先级 |
|--------|---------|------|--------|
| L4 批量统计的分帧 | 每周/月一次 | 分摊到每日微批量——避免月度尖峰 | 🟡中 |
| SoA 布局的 SIMD | glam 自动 SIMD | 情绪引擎中 emotion.update_basic_emotions() 可以显式 SIMD | 🟢低 |
| 决策器查表缓存 | 每次重新计算权重 | 同一 NPC + 相同情境 → 缓存 1-2 帧的权重结果 | 🟡中 |
| LMDB 热记忆分区 | 全量在热缓存 | 按"最近 7 天" + "重要标记"分区——减少冷数据搜索 | 🟡中 |

### 2.4 文档的可持续维护

**问题**：ver2.0 是单一文件（~8000+ 行预期）。随着系统增加，单文件维护将变得困难。

**建议**：当文档超过 10000 行时，拆分为：
```
NPC活人感模块/
├── NPC活人感开发文档ver2.0.md     ← 总纲（数据合同 + trait 索引）
├── 01-情绪引擎.md                  ← Part 2 展开
├── 02-记忆系统.md
├── 03-行为决策.md
├── 04-分层模拟与调度.md
├── 05-战斗AI.md                    ← Part 5 展开
├── 06-载具NPC行为.md
├── 07-天象感知与NPC.md
└── 08-工程实现.md                  ← Part 6 展开
```
每个子文档独立维护，总纲通过 wikilink 引用。拆分的触发条件是"单文件超过 LLM 上下文窗口的 30%"——确保 LLM 辅助开发时能一次性加载相关章节。

## 三、对 017 测试框架的追加建议

### 3.1 NPC 专项测试维度

017 v2.0 已包含远景/聚落/战斗/海洋/体积云/载具/蓝图专项。建议为 NPC 系统本身增加一个"NPC 活人感"综合维度：

| 层级 | 要求 | 首次测试 |
|------|------|---------|
| N1 | NPC 需求驱动可观测 | Phase 2 |
| N2 | 人格差异可观测 | Phase 3 |
| N3 | 关系+记忆可观测 | Phase 4 |
| N4 | NPC-天象联动可观测 | Phase 6 |
| N5 | NPC-载具行为可观测 | Phase 6 |
| N6 | NPC-施工行为可观测 | Phase 7 |

### 3.2 半自动战斗参与感测试

新增"战斗参与密度"指标：
```
参与密度 = 玩家做决策的次数 / 战斗总时长 (秒)
目标: ≥0.08 次/秒 (约每 12 秒一次决策)
低于此阈值 → 玩家感觉自己是"观众"
```
