## 错误 1：FestivalId 位布局逻辑漏洞

**原文**（001 §四）：
```rust
pub struct FestivalId(pub u32);
// 种子生成的节日：u32 高 16bit = CultureId, 低 16bit = 节日序号
// 运行时创建的节日（纪念/涌现）：最高位置 1 标记
```

**错误**：运行时创建的节日（涌现型、纪念性）可能不属于单一文化——鲑鱼节可能涉及两个文化区域的 NPC。且 `source_culture()` 对运行时节日返回 `None`——但下游代码大量假设每个节日都有 source_culture。16bit 序号空间（65,535）也远超过实际需求——浪费了位空间。

**修正**：
```rust
/// FestivalId —— 全局唯一，32位
/// 
/// 布局（与 SkillId 对标）：
///   [31]     = source: 0=种子生成, 1=运行时创建
///   [30:24]  = reserved (7 bits)
///   [23:0]   = festival_index (24 bits, 16M 容量——远超实际需求)
/// 
/// CultureId 不编码在 FestivalId 中——通过 FestivalFramework.primary_culture 查询。
/// 这样涌现型节日（可能跨文化）也不受限于单个 CultureId。
#[derive(Copy, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct FestivalId(pub u32);

impl FestivalId {
    pub fn is_seed_generated(&self) -> bool { self.0 >> 31 == 0 }
    pub fn is_runtime_created(&self) -> bool { self.0 >> 31 == 1 }
    // 删除 source_culture() —— CultureId 从 framework 中获取，不编码在 ID 中
}
```

## 错误 2：`festival_count_for_culture` 公式缺乏归一化

**原文**（001 §3.1）：
```rust
let modifiers = 
    core.religiosity * 1.5       // max +1.5
    + core.artistry * 1.0        // max +1.0
    + core.indulgence * 0.8      // max +0.8
    + core.competition_orientation * 0.5  // max +0.5
    + (1.0 - core.uncertainty_avoidance) * 0.3; // max +0.3
(base + modifiers).round().clamp(2, 7) as u32
```

**错误**：modifiers 在极端情况下可达 1.5+1.0+0.8+0.5+0.3 = 4.1，加上 base=3.0 = 7.1。恰好接近 clamp 上限 7——但这并非有意设计，只是巧合。如果后续调整权重，可能突破或远低于上限。且 clamp 为整数意味着对基础参数不敏感。

**修正**：
```rust
/// 每文化节日数量——归一化到 [2, 7]
fn festival_count_for_culture(core: &CultureCoreParams, rng: &mut impl Rng) -> u32 {
    // 归一化到 0-1 的"节日密度指数"
    let density = (
        core.religiosity * 0.30
        + core.artistry * 0.25
        + core.indulgence * 0.20
        + core.competition_orientation * 0.15
        + (1.0 - core.uncertainty_avoidance) * 0.10
    ).clamp(0.0, 1.0);
    
    // 映射到 [2, 7]——使用乘法而非加法，保证边界可控
    let count = 2.0 + density * 5.0; // 2 + [0,5] = [2, 7]
    let jitter = rng.gen_range(-0.5..0.5); // 温和随机波动
    
    (count + jitter).round().clamp(2.0, 7.0) as u32
}
```

权重总和恒为 1.0——任何权重的调整都不会突破 [2, 7] 边界。

## 错误 3：FestivalInstance 未处理跨年边界

**原文**（002 §3.3）：节日庆祝期从 `day_of_year` 开始，持续 `duration_days` 天。

**错误**：如果 `day_of_year = 118` 且 `duration_days = 5`，节日的第 3-5 天落在 day 120→1→2。如果 `day_of_year = 1` 且宣告期 7 天前开始——宣告期从上一年的 day 114 开始。当前 `day_of_year: u16` 无 modulo 120 语义。

**修正**：
```rust
/// 判断给定 day_of_year 是否落在节日的某个阶段内
fn is_festival_active_on_day(festival: &FestivalFramework, day: u16) -> bool {
    let start = festival.day_of_year as i32;
    let end = (start + festival.duration_days as i32) % 120;
    
    if end > start {
        // 正常情况——节日不跨越年边界
        day as i32 >= start && day as i32 < end
    } else {
        // 跨越年边界——节日从年末延续到新年
        day as i32 >= start || day as i32 < end
    }
}

/// 同样处理宣告期（announcement_lead_days 天前开始）
fn announcement_start_day(festival: &FestivalFramework) -> u16 {
    let raw = festival.day_of_year as i32 - festival.announcement_lead_days as i32;
    ((raw % 120 + 120) % 120) as u16 // 正确处理负值取模
}
```

## 错误 4：NPC 参加决策使用当天实际天气

**原文**（003 §2.1）：
```rust
let weather = match current_weather.condition { ... }
```

传入的参数是 `current_weather: &WeatherSample`——意味着 NPC 在决策时查询**当前**天气。

**错误**：NPC 在宣告期（节前 7 天）就做出参加决策——此时无法预测节日当天的天气。应该使用气候期望（该季节的历史平均天气）而非当天天气快照。

**修正**：
```rust
fn will_attend_festival(
    npc: &NpcData,
    festival: &FestivalInstance,
    culture: &dyn CultureQuery,
    faith: &dyn FaithQuery,
    season: &SeasonState,        // ← 替换 current_weather
    weather_climate: &ClimateNormals, // ← 该季节的预期天气
) -> f32 {
    // ...
    // 6. 天气期望因子——NPC 基于季节预期做决策，而非实际天气
    let weather_expectation = match season.season {
        Season::Spring => 0.85,  // 春季——可能有雨但总体温和
        Season::Summer => 0.95,  // 夏季——最可靠
        Season::Autumn => 0.80,  // 秋季——收获季，可能有风暴
        Season::Winter => {
            if festival.location.is_indoor() { 0.90 } // 室内节日不受冬寒影响
            else { 0.50 } // 户外冬季节日——预期寒冷
        }
    };
    
    // 节日当天实际天气在庆祝期 phase 切换时修正参加率——
    // NPC 可以在最后一天因暴雨决定"算了不去了"
}
```

## 错误 5：文化反馈 0.002/年无效

**原文**（007 §五）：
```rust
let feedback_strength = instance.scale * instance.participant_ratio() * 0.002;
// 最大每年 0.002 —— 比代际漂移（σ=0.003）稍弱但持续累积
```

**错误**：`scale=1.0 × ratio=1.0 × 0.002 = 0.002` 每年。需要 500 年才能累积 1.0。但大多数节日不会持续 500 年。且公式中说 σ=0.003/年——这不是正确的比较对象。σ=0.003 是标准差，不是每年的期望漂移量。实际代际漂移每年为 0（均值为零的随机游走），累积 100 年约为 `0.003×√100 = 0.03`。与之相比，节日反馈 100 年累积为 `0.002×100 = 0.2`——这实际上比代际漂移**大得多**。

但问题不在这里——问题是 0.002/年/节日 × 2-7 个节日 × 不同方向（有的节日 +indulgence，有的 +religiosity）→ 方向互相抵消 → 净效应可能接近零。**公式本身无误，但使用场景描述有误导**。

**修正**：不做数值改变——但重新校准解读：
```
代际漂移：每年无方向的随机游走。100 年累积 ~0.03（各方向均可能）
节日反馈：每年有方向的温和推力。100 年单节日累积 ~0.20，多节日互相制衡

净效应：节日的反馈是文化稳定性的维护机制——阻止代际漂移的随机性
        把文化参数推向"适应节日的方向"，而不是剧烈的文化改变。
        
这正符合"节日是文化的维护机制"的设计意图——不是改变文化，而是加固它。
```

**同时修正公式**——节日反馈应有衰减（边际效益递减）：
```rust
fn festival_cultural_feedback(
    festival: &FestivalFramework,
    instance: &FestivalInstance,
    culture: &mut CultureCoreParams,
) {
    let novelty = 1.0 - culture.get_param(param).abs(); // 已经很高的参数→再推更难
    let feedback = instance.scale * instance.participant_ratio() * 0.003 * novelty;
    culture.shift_param(param, feedback);
}
```

## 错误 6：`mandatory_attendance` 冗余存储

**原文**（001 §三 + 003 §2.1）：`FestivalFramework.mandatory_attendance: bool` 和 `will_attend_festival` 中的 obligation 检查。

**错误**：`mandatory_attendance` 是一个**派生属性**——它应该从"是否有统治者对该节日创建了 Duty"中派生，而不是存储在 FestivalFramework 上。否则可能出现不一致——节日标记为强制性但没有 Duty 存在，或者 Duty 存在但节日标记为非强制。

**修正**：
- 从 `FestivalFramework` 中删除 `mandatory_attendance` 字段
- `will_attend_festival` 中查询权力系统："对于给定的节日+位置，是否存在义务要求参加？"
- 权力系统通过 `PowerTopologyQuery::has_duty_for(entity, DutyType::FestivalAttendance, festival_id)` 回答

```rust
// 删除此字段：
// pub mandatory_attendance: bool,  ← 移除

// 改为消费方查询：
fn is_festival_mandatory_for(
    npc_id: NpcId, 
    festival_id: FestivalId,
    power: &dyn PowerTopologyQuery,
) -> bool {
    power.has_duty_for(npc_id, DutyType::FestivalAttendance, &festival_id)
}
```

## 错误 7：`ConquerorFestivalPolicy::Syncretize` 语义不明

**原文**（005 §4.1）：
```rust
Syncretize { mapping: FxHashMap<FestivalId, u16> },
```
注释说"强制融合——被征服文化的节日在征服者的日期举行"。

**错误**：`mapping` 的 value 类型是 `u16`——但这个 `u16` 是 day_of_year？还是 duration？还是什么？类型和语义不匹配。

**修正**：
```rust
enum ConquerorFestivalPolicy {
    Preserve,     // 保留被征服文化的节日
    Suppress {    // 压制特定节日（→地下节日）
        suppressed_ids: Vec<FestivalId>,
    },
    Replace {     // 用征服者的节日替换
        removed: Vec<FestivalId>,
        introduced: Vec<FestivalId>, // 来自征服者文化
    },
    Syncretize {  // 融合——日期、内容或名称的混合
        modifications: Vec<FestivalModification>,
    },
}

struct FestivalModification {
    target_festival: FestivalId,
    new_date: Option<u16>,              // 日期变更
    name_override: Option<String>,      // 征服者语言的名称覆盖
    activity_additions: Vec<FestivalActivityType>, // 征服者文化加入的活动
    activity_removals: Vec<FestivalActivityType>,  // 被征服文化的活动被禁止
}
```

## 错误 8：`price_multipliers` 硬编码——违反经济系统"价格从交易涌现"契约

**原文**（004 §1.2）：
```rust
price_multipliers: FxHashMap<ItemCategory, f32>,
// 食物 ×1.3, 酒 ×1.5, 礼物 ×1.2, 装饰品 ×1.8
```

**错误**：经济系统 CHG-022 契约明确规定："价格从订单簿撮合交易中涌现——不存在中央定价公式。**任何模块不得直接写入价格**。"节日系统预定义的价格乘数是直接写入价格——违反了这条契约。

**修正**：节日系统不设定价格——它只影响**需求端**。价格的上升是经济系统订单簿的自然结果：

```rust
/// 节日对经济的影响——只设定需求修正，不设定价格
struct FestivalEconomicImpact {
    /// NPC 的消费倾向修正——消费倾向（consumption_propensity）的临时加成
    consumption_propensity_boost: f32,  // +0.15 常规，+0.30 大型节日
    /// 特定物品类别的额外需求指令（NPC 经济心智生成额外 bid 订单）
    additional_demand: Vec<(ItemCategory, f32)>, // (类别, 额外需求量)
    /// 外地商人供给期望——增加的市场参与者（临时的 Storefront 数量）
    external_supplier_count: u32,
}
```

消费激增 → 需求曲线右移 → 订单簿中 bid 增多 → 价格自然上升（通过经济系统自身的撮合引擎）。价格升多少取决于供给弹性——不由节日系统定义。

## 错误 9：仪式组件与魔法仪式冲突

**原文**（002 §4.1 + 002 §4.2）：节日系统中定义了 `RitualComponents`——包含 `SupernaturalComponent`——而魔法系统 010 中也定义了仪式组件模型。两个模型有重叠但无协调机制。

**错误**：同一种仪式（如"祈求丰收的魔法阵"）既是节日中的社会仪式（需要人群聚集、祭司主持），又是魔法仪式（需要法力注入、法阵激活）。两个系统各自建模了部分——但没有定义谁调谁、谁拥有什么。

**修正**：
- 社会仪式的**通用组件**（主祭/参与者/地点/时长/言语/动作）由**节日系统**定义
- 魔法效果层由**魔法系统**定义
- 节日系统中的 `SupernaturalComponent` 不定义魔法的实现——只定义"此仪式有魔法成分"和一个 `MagicRitualHandle` 引用到魔法系统

```rust
/// 节日系统定义的仪式——纯社会层面
pub struct RitualDef {
    pub celebrant: CelebrantRole,
    pub participants: Vec<ParticipantRequirement>,
    pub location: RitualLocation,
    pub duration_minutes: f32,
    pub speech: Vec<RitualSpeechStep>,    // 与 TurnMode::Ritual 对齐
    pub actions: Vec<PhysicalAction>,
    /// 可选的魔法成分——如果存在，交由魔法系统执行魔力逻辑
    pub magic_component: Option<MagicRitualRef>,
}

/// 魔法成分——引用魔法系统 010 的仪式规格
struct MagicRitualRef {
    ritual_spec_id: MagicRitualSpecId,  // ← 由魔法系统定义的仪式规格
    expected_mana_cost: u32,            // 预期魔力消耗
    risk_level: MagicRiskLevel,
}
```

**责任划分**：
- 节日系统：管理人群、调度时间、引导参与者行为、触发对话
- 魔法系统：在 `magic_component` 存在时，执行魔力流、判定成功/失败、施加魔法效果或反噬
- 两个系统通过 `MagicRitualRef` 单向关联——节日系统不导入魔法系统的任何内部类型
