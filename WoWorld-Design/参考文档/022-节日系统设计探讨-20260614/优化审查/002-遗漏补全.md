## 遗漏 1：节日取消与中止机制

**原始方案缺失**：002 的运行时因素表中提到"瘟疫流行→大型聚集节日被取消"，但没有被取消节日的状态模型，也没有中止进行中节日（"战争在第 2 天爆发"）的逻辑。

**补全设计**：

```rust
/// 节日取消原因
enum FestivalCancellation {
    /// 提前取消——在宣告期或预备期被取消
    Cancelled {
        reason: CancellationReason,
        cancelled_at: GameTime,
        /// 是否可能明年恢复
        will_resume_next_year: bool,
    },
    /// 进行中被中止——庆祝期中被中断
    Aborted {
        reason: AbortReason,
        aborted_at: GameTime,
        /// 中止时已完成的天数 / 总天数
        completion_ratio: f32,
    },
}

enum CancellationReason {
    War,                    // 战争爆发或逼近
    Plague,                 // 瘟疫流行
    Famine,                 // 饥荒——"没心情"
    RulerDeath,             // 统治者突然死亡——国丧
    NaturalDisaster,        // 地震/洪水/火山
    ConquerorSuppression,   // 征服者禁止
    LackOfResources,        // 实在没有物资举办
}

enum AbortReason {
    EnemyAtGates,           // 敌军兵临城下
    FireOrDisaster,         // 火灾/建筑坍塌
    RitualFailure,          // 核心仪式失败（魔法反噬/神祇震怒的迹象）
    ViolenceOutbreak,       // 群集暴力——节日斗殴升级
}

/// 取消后的 NPC 反应
struct CancellationAftermath {
    /// 情绪影响——取消的节日导致 collective mood 下降
    mood_penalty: f32,           // -0.05（天灾，可理解）到 -0.20（统治者无故取消）
    /// 如果原因是 RitualFailure → 宗教虔诚度可能被影响
    faith_crisis_probability: f32, // 0-1
    /// 如果原因是 ConquerorSuppression → 增加反抗情绪
    resistance_boost: f32,
    /// 历史记录——取消可能比正常举办的节日更值得记录
    historical_significance: f32,
}
```

**中止后的行为**：
- 所有 `attending_npcs` 的状态从 `Attending` 切换为 `Done`（或被中断的特殊状态）
- 节日装饰的清除加速（立即拆除而非正常的 2-3 天渐变）
- NPC 生成 `EventMemory`："收获节被战争打断的那年"
- 如果中止原因是 RitualFailure → 魔法系统记录魔法反噬事件

## 遗漏 2：室内/户外活动的天气敏感性

**原始方案缺失**：002 §3.5 的活动枚举中所有活动被同等处理，但 `Procession`（游行）和 `Tournament`（户外竞技）对暴雨的敏感度远高于 `Feast`（宴饮——通常在室内）和 `Prayer`（祈祷——神殿内）。

**补全设计**：

```rust
/// 每类节日活动对天气的敏感度
fn weather_sensitivity(activity: FestivalActivityType) -> WeatherSensitivity {
    match activity {
        // 户外活动——天气影响大
        Procession | Tournament | Race | Fireworks | Dance | Market => {
            WeatherSensitivity {
                heavy_rain_cancels: true,
                snow_cancels: false,       // 雪中游行/竞技反而有独特美感
                wind_sensitive: true,       // 大风→烟火取消、帐篷吹倒
                heat_sensitive: false,
                cold_sensitive: false,
            }
        }
        // 半户外活动——有遮盖但非完全室内
        Feast | Music | Theatre | Contest | GiftGiving => {
            WeatherSensitivity {
                heavy_rain_cancels: false,  // 可在帐篷/棚架下进行
                snow_cancels: false,
                wind_sensitive: true,        // 大风→帐篷危险
                heat_sensitive: false,
                cold_sensitive: true,        // 严寒→宴饮移入室内
            }
        }
        // 室内活动——基本上不受天气影响
        Prayer | Sacrifice | RitualCleansing | Matchmaking | Auction => {
            WeatherSensitivity::indoor() // 全部 false
        }
        // 灯火/烟火——特殊：需要黑暗和微风
        Fireworks | Decoration => {
            WeatherSensitivity {
                heavy_rain_cancels: true,
                snow_cancels: false,
                wind_sensitive: true,        // 大风→灯火熄灭/危险
                heat_sensitive: false,
                cold_sensitive: false,
            }
        }
    }
}
```

**节日规模修正**：天气敏感的活动在恶天气下取消 → 节日 scale 降低（而非整个节日取消）。一个 7 天的节日可以在第 2 天暴雨中取消户外活动但保留室内宴饮——节日的总体情绪降低但未中断。

## 遗漏 3：节日音频设计

**原始方案缺失**：006-语言表达与节日只覆盖了文本/对话，但完全没有提及音频。节日是音频体验最丰富的场景之一。

**补全设计**：

```rust
/// 节日的音频层——叠加在正常环境音频之上
struct FestivalAudioProfile {
    /// 节日专属音乐（该文化的音乐风格 + 节日情绪调式）
    music_preset: MusicPreset,
    
    /// 环境音频层
    ambient_layers: Vec<FestivalAmbientLayer>,
    
    /// 时间触发音频事件
    timed_audio_events: Vec<TimedAudioEvent>,
    
    /// 情绪驱动的音频——响应 SocialField 的情绪强度
    emotion_driven_audio: EmotionDrivenAudio,
}

struct FestivalAmbientLayer {
    sound_type: AmbientSoundType,
    /// 该音频层的可闻半径（m）
    audible_radius: f32,
    /// 随节日阶段变化
    phase_intensity: [f32; 4], // [Announcement, Preparation, Celebration, Aftermath]
}

enum AmbientSoundType {
    Bells,           // 钟声——宣告节日开始/结束
    CrowdMurmur,     // 人群低语——庆祝期间
    MusicStreet,     // 街头音乐——街道艺人
    RitualChant,     // 仪式诵唱——神殿区
    FireworksCrackle, // 烟火噼啪
    FeastSounds,     // 宴饮声——碰杯、欢笑
    TournamentClash, // 竞技碰撞声
    ChildrenPlaying, // 儿童玩耍——节日中儿童更活跃
}

/// 时间触发事件——"正午钟声响起"
struct TimedAudioEvent {
    game_hour: f32,          // 游戏时间（小时）
    sound_event: SoundEvent,
    repeats_daily: bool,     // 节日期间每日重复？
}

/// 情绪驱动音频——SocialField 情绪强度 → 音频强度映射
struct EmotionDrivenAudio {
    /// 欢呼声——随 SocialField.approval 强度变化
    cheering: VolumeCurve,      // approval 0→silence, 0.5→murmur, 1.0→roar
    /// 音乐强度——随群体情绪变化
    music_intensity: VolumeCurve,
}
```

**性能**：节日音频 = 已有环境音频系统的**额外音频层**——不需要新的音频引擎。VRAM 增量（节日音乐样本）≤ 5MB。

## 遗漏 4：节日竞技具体规则

**原始方案缺失**：003-004 多次提及"竞技"（Tournament/Race/Contest）但从未定义 NPC 如何比赛、如何判定胜者。

**补全设计**：

```rust
/// 节日竞技——一个节日中的竞技活动实例
struct FestivalCompetition {
    competition_type: CompetitionType,
    /// 参赛者列表
    contestants: Vec<CompetitionContestant>,
    /// 赛制
    format: CompetitionFormat,
    /// 当前轮次
    current_round: u8,
    /// 裁判（用于判定，特别是主观评分类竞技）
    judges: Vec<Judge>,
    /// 投注池（如果文化允许博彩）
    betting_pool: Option<BettingPool>,
}

enum CompetitionType {
    /// 武艺竞技——使用战斗系统的 proficiency 判定
    Martial {
        skill: SkillId,          // 剑术/斧术/弓术……
        rules: MartialRuleSet,   // 点到为止/直到一方倒下/禁用特定招式
    },
    /// 速度竞技——基于物理属性和技能
    Race {
        race_type: RaceType,     // 徒步/骑马/赛船
        course: Vec<WorldPos>,   // 赛道途径点
    },
    /// 评分类竞技——由裁判评分
    Judged {
        skill: SkillId,          // 歌唱/烹饪/诗歌
        criteria: Vec<JudgingCriterion>,
    },
    /// 力量竞技——基于 PhysicalAccess trait 的力量/耐久
    Strength {
        event_type: StrengthEvent, // 举重/投掷/摔跤
    },
}

/// 判定胜者的公式
fn determine_winner(competition: &FestivalCompetition) -> Vec<CompetitionContestant> {
    match competition.competition_type {
        Martial { skill, rules } => {
            // 使用战斗系统的 proficiency 判定
            // 但竞技不是真打——使用"模拟战斗"而非实际 Combat 系统
            // 每个参赛者的"预期得分" = proficiency(skill) × (1 + innate_aptitude/10) + random(±0.15)
            for c in &competition.contestants {
                let proficiency = c.npc_skills.get(&skill).map(|s| s.proficiency()).unwrap_or(0.1);
                let aptitude = c.npc_skills.get(&skill).map(|s| s.innate_aptitude).unwrap_or(1.0);
                c.score = proficiency * (1.0 + (aptitude - 1.0) * 0.5) + rng.gen_range(-0.15..0.15);
            }
        }
        Judged { criteria, .. } => {
            // 每位裁判独立打分 → 去掉最高最低 → 取平均
        }
        Race { .. } => {
            // 竞速：NPC 的 agility × endurance → 基础速度 → 叠加赛道难度 → 模拟时间
        }
        Strength { .. } => {
            // 力量竞技：strength × (1 - fatigue × 0.5)
        }
    }
    
    // 按得分/完成时间降序排列
    competition.contestants.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap());
    competition.contestants.clone()
}

/// 竞技奖励
struct CompetitionReward {
    /// 第一名获得的物品或货币
    first_prize: Reward,
    /// 名声增益
    reputation_gain: ReputationEffect,
    /// 是否获得头衔
    title_award: Option<TitleId>,
}
```

**性能**：竞技判定是**离线计算**——`determine_winner` 不逐帧执行。只在比赛开始时一次性运行，winner 产生后存储结果。

## 遗漏 5：聚落间节日协调

**原始方案缺失**：如果相邻两个聚落的节日在同一天——NPC 只能选一个参加。这会产生"节日竞争"。原始方案中每个聚落独立生成节日日期，没有跨聚落协调。

**补全设计**：

```rust
/// 节日日期分配——考虑周边聚落的已有节日日期
fn assign_festival_date_with_coordination(
    target_date_range: Range<u16>,
    culture_id: CultureId,
    settlement_id: SettlementId,
    existing_festivals: &[(FestivalId, u16)],  // 周边聚落已有的节日日期
    rng: &mut impl Rng,
) -> u16 {
    let mut candidates = Vec::new();
    
    for day in target_date_range {
        // 与周边聚落节日的距离（天数）
        let min_distance = existing_festivals.iter()
            .map(|&(_, d)| circular_distance_120(day, d))
            .min()
            .unwrap_or(120);
        
        // 距离 ≥ 3 天 → 候选日期。距离 < 3 天 → NPC 被迫选择 → 降低双方的参与率
        if min_distance >= 3 {
            candidates.push((day, 1.0));
        } else {
            candidates.push((day, 0.3)); // 低权重——尽量避免但非不可能
        }
    }
    
    weighted_pick(&candidates, rng)
}

/// 120 天循环日历的距离
fn circular_distance_120(a: u16, b: u16) -> u16 {
    let diff = (a as i32 - b as i32).abs() as u16;
    diff.min(120 - diff)
}
```

**NPC 面对同日两个节日时的选择**：
- 优先选择本聚落的节日（`distance=0` 的）
- 如果两个都不是本聚落的 → 选择有亲密关系的人去的那个
- 选择距离较近的
- 如果都不去 → 两个节日的参与率都下降

## 遗漏 7：玩家日历/日志集成

**原始方案缺失**：009 提到"日志自动标记"但未定义数据在哪、如何持久化。

**补全设计**：

```rust
/// 玩家的节日知识——存储在玩家存档中
struct PlayerFestivalKnowledge {
    /// 已发现的节日及其日期
    known_festivals: FxHashMap<FestivalId, KnownFestivalInfo>,
    /// 已参加过的节日（含年份和记忆）
    attended_festivals: Vec<FestivalAttendanceRecord>,
}

struct KnownFestivalInfo {
    festival_id: FestivalId,
    /// 玩家学习到此节日的途径
    learned_from: KnowledgeSource,
    /// 学到的日期
    learned_at: GameTime,
}

enum KnowledgeSource {
    /// NPC 对话中告知
    NpcTold { npc_id: NpcId },
    /// 布告板阅读
    NoticeBoard { settlement_id: SettlementId },
    /// 亲自参加
    FirsthandExperience { year: u64 },
    /// 书籍/游记
    Book { expression_ref: ExpressionRef },
}

/// 节日参加记录——玩家的节日记忆
struct FestivalAttendanceRecord {
    festival_id: FestivalId,
    year: u64,
    /// 玩家做了什么角色
    role: PlayerFestivalRole,
    /// 印象最深刻的事件

enum PlayerFestivalRole {

    Participant,
    Competitor { result: CompetitionResult },

    Troublemaker,
}
```

**日志 UI 集成**：玩家的日志面板中新增"节日历"分页——展示已发现文化的已知节日日期。如果玩家购买了某文化的节日历法知识（作为书籍或 NPC 传授），则显示该文化的完整年历。
