## 一、当前技术栈概览

WoWorld v4.0 技术栈的架构可以概括为"**Rust 大脑 + Godot 身体**"：

```
+----------------------------------------------------------------------+
|                        Rust Simulation Core                           |
|  NPC AI · 世界生成 · 战斗 · 经济 · 文化 · 信仰 · 权力 · 魔法       |
|  生命 · 历史 · 载具 · 建造 · 感官 · 认知 · 音频模拟 · 动画计算     |
|  22 个模块 · SoA 存储 · rayon 并行 · LMDB 持久化                    |
+----------------------------------+-----------------------------------+
                                   | GDExtension FFI (PackedByteArray)
+----------------------------------+-----------------------------------+
|                        Godot 4.6 LTS Client                          |
|  渲染 · UI · 音频输出 · 输入 · 玩家物理(仅CharacterBody3D)          |
|  资产管线 · glTF导入 · 多平台导出                                   |
+----------------------------------------------------------------------+
```

该方案经过了多轮严格的技术栈比较才最终确立：

| 备选方案 | 综合评分 | 被否决的关键原因 |
|----------|---------|-----------------|
| **Godot + Rust GDExtension** | **40 分** | ✅ 选中 |
| Unity 6 + C# DOTS | 35 分 | 体素支持需自建、运行时费用争议损害信任 |
| Unreal Engine 5.6 + C++ | 28 分 | 为 100 人 AAA 团队设计，Solo 不可承受的编译时间和复杂度 |
| Bevy 纯 Rust | 26 分 | 无编辑器、API 不稳定、生态 immature（详见下文） |
| 纯 Rust + wgpu（自建引擎） | 22 分 | 预计 17-36 人月工作量，Solo 不可行 |

## 二、Godot 4.6 LTS 提供了什么

### 2.1 渲染管线（~8ms 帧预算）

| 渲染任务 | 预算 | 实现方式 |
|---------|------|---------|
| 体素地形 (~200 Mesh + Clipmap LOD) | ~2.5ms | Rust Transvoxel → Godot ArrayMesh |
| NPC 渲染 (MultiMeshInstance3D, L1-L3) | ~1.0ms | GPU skinning 双骨蒙皮 |
| GPU 蒙皮 (≤1000 L1 NPC) | ~0.5ms | Rust CPU 批量骨骼矩阵→Godot shader |
| 海洋 (Gerstner 波) | ~1.0ms | 8-12 波叠加 shader |
| 体积云 (混合方案) | ~1.5ms | 2D impostor + 3D 体积密度场 ray march |
| 城市建筑 | ~1.5ms | MultiMesh + LOD |
| 光照/阴影 (CSM ×3 级联) | ~1.5ms | Godot 内置 |
| 后处理/UI/粒子/玩家/载具 | ~1.0ms | Godot 内置 |

**画面风格**：3D 低多边形 + flat/cel 渲染（TABS 风格——单 diffuse pass，无 PBR）。

### 2.2 UI 系统

Godot **Control 节点系统**——容器、按钮、文本、滚动、主题、StyleBox、BBCode 富文本——为 WoWorld 的复杂 UI 需求提供了开箱即用的支撑：
- 背包/物品面板
- 对话系统界面
- 技能树/天赋面板
- 蓝图编辑器（三路径建造系统）
- HUD（战斗/探索/建造模式切换）

### 2.3 音频输出

Godot **AudioServer** 处理最终音频输出：3D 空间音频、混音、效果总线。注意：音频**模拟**（声音传播、衰减、遮挡建模）在 Rust 侧完成，Godot 仅负责渲染最终结果。

### 2.4 输入

键盘、鼠标、手柄的完整输入处理管线。

### 2.5 物理（严重精简后）

⚠️ CHG-033 重大变更后：**仅玩家 CharacterBody3D 保留 PhysicsServer3D (Jolt Physics)**。所有 NPC、动物、物品、弹射物的物理交互使用 Rust 侧空间查询（`TerrainQuery` / `EntityIndex` / `SpatialEventBus` / `VisibilityQuery` 四 trait）。

### 2.6 资产管线

- glTF 2.0 导入（模型/材质/纹理/动画/blendshape）
- 拖放导入 + 可配置导入预设
- 纹理压缩/mipmap 自动生成
- 音频流式加载/预加载切换
- `.tres`/`.res` 资源系统（引用计数、场景间共享、继承）
- 一键导出 Windows/Linux/macOS/Android/iOS/Web

## 三、Rust 模拟核心提供了什么

Rust (stable 1.80+) 是 WoWorld 的"大脑"——所有非玩家直接感知的系统在此运行。**该核心被刻意设计为引擎无关**，不导入 Godot API。

### 3.1 22 个模拟模块

| 模块 | 核心职责 | 性能预算 |
|------|---------|---------|
| NPC AI | 记忆(≤2000条)/情感/GOAP(9%)/行为树(90%)/社交关系 | ≤2.5ms |
| 战斗 | 三层模型(本能→节奏→战略)/半自动/招式积木 | ≤1.5ms |
| 骨骼矩阵 | 批量 CPU 计算(rayon 6线程)→GPU skinning | ≤0.5ms |
| 空间查询 | TerrainQuery/EntityIndex/SpatialEventBus/VisibilityQuery | ≤1.0ms |
| 感官 | VisionQuery/ScentQuery/PerceptBatch/DarkAdaptation | ≤1.0ms |
| 世界生成 | 程序化地形/Transvoxel/Clipmap LOD/城市布局 | 后台线程 |
| 经济 | 订单簿价格涌现/两阶段提交/中间商涌现 | <0.1ms/帧 |
| 权力 | 17 PowerAtom/PowerTopology/合法性/Duty制裁 | <0.1ms/帧 |
| 文化 | CultureCoreParams/Voronoi障碍空间/四路径演变 | <0.05ms |
| 信仰 | FaithTheology/实践优先/NPC→NPC接触传染 | <0.05ms |
| 认知 | CognitiveStyle/ MentalModel/睡眠正则化/创新管线 | <0.15ms |
| 音频模拟 | SoundFootprint/传播引擎/话语清晰度 | ≤0.17ms |
| 动画 | 9层动画栈/38姿态/15轨迹/步态涌现/面部图集 | (含骨骼矩阵) |
| 生命 | Vitals/Physiology/Spirit/Mana/死亡原因(25种) | 集成在各模块 |
| 历史 | AetherImprint/AetherQuery/编年史生成 | <0.1ms/帧 |
| 载具 | 移动参考系/Crew GOAP/5动力类型 | ≤0.3ms |
| 建造 | 蓝图解析/三路径/集体施工调度 | ≤0.2ms |
| 天气 | Markov 6状态/双层温度/120d年·48min日 | ≤0.2ms |
| 物品 | ItemDefId/Assembly组件树/Enchantment | 集成 |
| 技能 | SkillId/三层天赋/TeachingRisk/CrossTraining | 集成 |
| 语言 | ExpressionRef/ContentResolver/片段组合/应酬层 | 集成 |
| LLM | LlmSceneConfig/本地5+云端6后端/VoiceProfile | 可选 |

### 3.2 存储架构

- **SoA (Struct of Arrays)**：`NpcRegistry` 按列存储，对齐 NPC 文档架构。非外部 ECS 框架
- **LMDB**：流式 Chunk 存储 + 种子确定性（支撑无限世界）
- **事件总线**：同步(帧内) + 异步(跨帧)，FIFO + 优先级队列

### 3.3 并行策略

rayon 1.x 数据并行——批量骨骼矩阵、感官查询、寻路请求。

### 3.4 分层模拟 L1-L4

| 层级 | 数量 | 更新频率 | 模拟深度 |
|------|------|---------|---------|
| L1 | ≤1,000 | 每 0.2s | 完整心智（记忆/情感/GOAP/关系/审美） |
| L2 | ≤10,000 | 每 1-2s | 轻量（行为树/基础需求/社交更新） |
| L3 | ~89,000 | 每日/每周 | 统计级（经济参与/文化漂移/人口统计） |
| L4 | 百万级 | 每周/每月 | 宏观统计（文明演化/历史事件） |

## 四、GDExtension 桥接的代价

GDExtension 通过 `godot-rust` crate (v0.2+) 连接两个世界。数据通道为 `PackedByteArray`。

### 4.1 已知成本

| 代价 | 量级 | 详情 |
|------|------|------|
| FFI 序列化开销 | ~0.5ms/1000 NPC | bincode 编解码，Rust → Godot 变换/动画数据 |
| 跨 FFI 调试困难 | 定性 | 两套独立调试工具（rust-gdb + Godot debugger） |
| 三组件独立演进 | 定性 | Rust 结构体 → godot-rust crate → GDExtension C API → Godot 引擎内部 |
| API 调用开销 | 定性 | Rust GDExtension API 调用约为 C++ GDExtension 的 2.8x（社区基准测试） |
| 混合语言认知负担 | 定性 | GDScript（胶水/场景）+ Rust（性能关键模块）双语言上下文切换 |

### 4.2 v4.0 缓解措施

- 锁定 Godot 4.6 LTS（减少 Godot 侧变动）
- 桥接层保持极薄（仅数据传递，无业务逻辑）
- Rust 核心零 Godot API 依赖（引擎无关设计）
- 热路径数据预打包为 `PackedByteArray`（减少 FFI 调用频率）

## 五、既有 Bevy 评估结论回顾

WoWorld 在技术栈选型阶段已经对 Bevy 进行了深入评估。详见 `参考文档/第一部分-设计演进存档-20260610/005-技术栈深入讨论-20260610/003-Bevy全栈方案与Godot方案的深度对比.md`。

### 5.1 当时认可的优势（至今有效）

1. **统一类型系统**——无 FFI，零序列化开销。模拟写入组件，渲染同一帧直接读取
2. **统一调试**——`bevy-inspector` 可检查任何 Entity 的 Components
3. **ECS 原生**——模拟和渲染共存于同一 ECS 世界
4. **全 Rust 类型安全**——编译期消除空指针/数据竞争
5. **单体二进制**——无外部引擎依赖
6. **WGSL 自定义渲染**——完全控制 GPU 管线

### 5.2 当时认为不成熟的方向（部分可能已有改善）

| 当时判断 | 2026.06 现状 | 变化？ |
|---------|-------------|--------|
| 无 GUI 编辑器 | 仍无官方编辑器。Jackdaw v0.4.1 (2026.05) 最接近 | 改善中，但远未成熟 |
| bevy_ui 基础薄弱 | 仍为基础 Flexbox 布局。无 Theme/StyleBox/BBCode | 无实质变化 |
| 动画系统 primitive | `AnimationGraph` 早期阶段，无 `AnimationTree`/`BlendSpace` 等价物 | 无实质变化 |
| 音频管线基础 | 仍仅有基础空间播放。kibaudio (2025) 填补部分空白 | 社区改善 |
| 资产导入需手动配置 | `AssetProcessor` 有改善，但仍远不如 Godot 拖放式 | 小幅改善 |
| API 不稳定（每 3-4 月 breaking） | 仍是 pre-1.0。0.18.1→0.19 每 3-4 月发布 | 无变化 |
| 社区小 | GitHub ~46.5K stars, ~1,967 依赖 crate, ~12 商业游戏 | 增长中，但仍远小于 Godot |

### 5.3 v4.0 最终结论（当时）

> "如果这个项目在 2028 年启动，我会推荐 Bevy 全栈。但 2026 年，Bevy 的 UI、动画、音频等方面仍需大量自建。"

**同时刻意留下了迁移通道**：Rust 模拟核心被设计为引擎无关。从架构上预设了未来可能切换渲染前端的路径：

```
现在:     Rust Core -- GDExtension -- Godot
未来:     Rust Core -- bevy_ecs -- Bevy (当 Bevy 足够成熟时)
```

## 六、当前方案的综合评估

### 优势

1. **最低完成风险**——Godot 数千款商业游戏的 track record 是 Solo 开发者最可靠的安全网
2. **成熟编辑器**——拖放式场景编辑、可视化 UI 设计、动画混合树、资产管线
3. **Rust 性能兜底**——性能关键模块（体素、AI、物理查询）在 Rust 侧，不受 GDScript 性能限制
4. **LTS 稳定性**——Godot 4.6 LTS 无需追逐 API 变动
5. **社区生态**——十年积累的教程/论坛/插件，任何 RPG 子系统都有现成参考
6. **迁移保险已买**——Rust 核心引擎无关，降低未来切换成本

### 已知风险

| 风险 | 等级 | 详情 |
|------|------|------|
| **VRAM 压力** | 🔴 #1 | 4.2GB/6GB = 70% 利用率。若 LOD 架构未完全到位，VRAM 先于 CPU 成为瓶颈 |
| **GDExtension 桥接** | 🟡 技术债务 | 三组件独立演进，跨 FFI 调试困难。已通过锁定 LTS + 薄桥接层缓解 |
| **godot-rust 社区维护** | 🟡 依赖风险 | 社区 crate，非 Godot 官方。v0.2+ 可用但偶尔 breaking |
| **Godot 5.x 兼容性** | 🟢 远期风险 | GDExtension API 可能在 5.x 中变动。薄桥接层策略降低修复成本 |
| **混合语言认知负担** | 🟡 效率成本 | GDScript ↔ Rust 双语言上下文切换。对 Solo 开发有实际影响 |

## 七、为什么现在重新讨论 Bevy

尽管 v4.0 已得出"Godot 最优"的结论，重新审视 Bevy 有几个合理原因：

1. **Bevy 在快速演进**——0.18 (2026.01) 带来 Atmospheric Scattering + PBR 改进 + Fullscreen Materials；0.19 即将发布（Solari 路径追踪升级）
2. **商业化验证增强**——Tiny Glade (100 万+ Steam 愿望单) 证明了 Bevy ECS 可以支撑商业成功
3. **Voxelis SVO-DAG** (2025.04) 展示了 Bevy 体素方案的架构先进性
4. **godot-bevy 混合方案**日趋成熟 (v0.8.x)，提供了第三条路
5. **Solo 开发的时间窗口**——如果项目还需要多年才能进入生产，现在重新评估技术栈比中途切换成本低
