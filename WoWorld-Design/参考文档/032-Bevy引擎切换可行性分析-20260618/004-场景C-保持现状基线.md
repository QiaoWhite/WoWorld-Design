## 场景定义

**不做任何引擎层面的改变。** 继续以 Godot 4.6 LTS + Rust stable 1.80+ (GDExtension) 作为 v4.0 技术栈推进 WoWorld 的开发。

这个场景是本分析的**对照基线**——其他两个场景的价值需要和它比较才能判断。

---

## 一、当前方案的核心优势（为什么它之前胜出）

### 1.1 最低完成风险

这是 Solo 开发中最被低估的因素。在 Reddit/HN/Lemmy 上的数十个 Godot vs Bevy 讨论中，反复出现的共识是：

> "Both engines are perfectly capable. Shipping is more about sticking with something than finding the perfect tool."

Godot 之所以在 v4.0 综合评分中拿到 40 分（远超 Bevy 的 26 分），不是因为它在某个维度上出类拔萃，而是因为**它在所有维度上都"够用且稳定"**：

| 维度 | Godot | Bevy | 差距 |
|------|-------|------|------|
| 渲染 | 成熟 PBR + 自定义 shader | WGSL 灵活但需自建 | Godot 胜 |
| 编辑器 | 生产级 | 社区原型 | Godot 大胜 |
| 资产管线 | 拖放 + 预设 | 代码驱动 | Godot 胜 |
| UI | Control 节点系统 | Flexbox 基础 | Godot 大胜 |
| 动画 | AnimationTree + BlendSpace | 基础关键帧 | Godot 大胜 |
| 音频 | AudioServer + AudioBus | 基础播放 | Godot 胜 |
| 物理 | Jolt (仅玩家) | Rapier/Avian | 持平(玩家物理) |
| 寻路 | NavMesh 内置 | 需自建 | Godot 胜 |
| 平台导出 | 一键多平台 | cargo build --target | Godot 胜(便捷性) |
| ECS 性能 | SoA + rayon 手动 | ECS 自动并行 | Bevy 胜 |
| 语言纯粹性 | GDScript + Rust | 100% Rust | Bevy 胜 |
| 类型安全 | 跨边界丢失 | 全栈 | Bevy 胜 |

**Godot 在 10/12 维度中持平或领先。** Bevy 的胜利维度（ECS 性能、语言纯粹性、类型安全）是架构优雅性方面的——重要，但不是决定 Solo 项目能否完成的关键。

### 1.2 社区和生态的规模效应

- Godot 数千款 Steam 游戏 → 几乎所有游戏类型都有参考项目
- 十年积累的教程、论坛帖、Discord 问答 → 遇到问题时大概率有人遇到过
- 丰富的插件生态（godot_voxel, TerrainCrafter, Chunx 等）→ 核心功能不必从零开始
- W4 Games 提供的企业级支持和主机移植 → 商业化路径明确

这不是"理论上更好"的问题，是**Solo 开发的保险单**——当你卡在某处时，能找到答案的概率更高。

### 1.3 迭代速度

GDScript 热重载（保存即生效）是 Solo 开发的加速器：
- UI 布局调整：拖放 + 实时预览，秒级
- 动画混合参数：AnimationTree 滑块实时调节
- 游戏手感微调：修改参数 → 保存 → 立即在运行中的游戏看到效果
- 不需要编译、不需要重启、不需要等待

对于需要大量调优和试错的 RPG 系统（战斗手感、UI 交互、动画过渡、NPC 行为节奏），这个迭代速度优势是实质性的。

### 1.4 迁移保险已就位

Rust 模拟核心的引擎无关设计意味着：
- 今天用 Godot 渲染，不需要改一行 Rust 逻辑代码
- 未来如果 Bevy 成熟到可以替换 Godot，迁移成本主要集中在渲染桥接层
- 不需要在今天"赌"一个引擎

这个设计决策本身就是对"Godot vs Bevy"不确定性的最佳对冲。

## 二、当前方案的已知风险与成本

### 2.1 VRAM 压力 (#1 风险)

```
VRAM 预算: 4.2GB / 6GB = 70%
- 体素网格 (视锥内):       ~1.2 GB
- 城市建筑网格+纹理:       ~0.8 GB
- NPC MultiMesh 实例数据:  ~0.3 GB
- 环境纹理 (地面/植被/海洋): ~0.5 GB
- 体积云密度纹理:          ~0.2 GB
- 阴影贴图:               ~0.2 GB
- 后处理/UI/其他:          ~1.0 GB
```

70% 是**静态预算**——未计及 LOD 降级、不可见区域剔除、纹理压缩优化后的实际值。但余量不大（1.8 GB 头寸），如果 LOD 架构未按设计完全实现，VRAM 先于 CPU/GPU 成为帧率瓶颈。

这个风险与引擎选择**无关**——无论是 Godot 还是 Bevy，都需要管理体素 + 城市 + NPC + 环境纹理的总 VRAM 占用。但它确实是任何引擎方案都必须面对的硬约束。

### 2.2 GDExtension 桥接层（已知技术债务）

详见 [[001-现状回顾与既有评估#四、GDExtension 桥接的代价|001 §四]]。核心风险：
- **三组件独立演进**：Rust struct → godot-rust → GDExtension C API → Godot 引擎。任何一环断链都需修复
- **跨 FFI 调试困难**：两套独立调试工具
- **godot-rust 社区维护**：非 Godot 官方，依赖个别维护者的持续投入
- **API 调用开销**：Rust GDExtension API ~2.8x 慢于 C++ GDExtension

**v4.0 缓解措施**（锁定 LTS + 薄桥接层 + 批量数据传递）将风险控制在可接受范围内，但未消除。

### 2.3 混合语言认知负担

GDScript（胶水代码/场景/UI 逻辑）+ Rust（性能关键模块）的双语言栈意味着：
- 上下文切换：调试 NPC 行为可能需要在 GDScript UI 层和 Rust 模拟层之间来回
- 两种类型系统：GDScript 的动态类型 + Rust 的静态类型的心理模型切换
- 两种构建流程：`cargo build` + Godot 编辑器运行

这是 GDExtension 方案的固有成本，但相比混合方案的三层架构，它仍是更简单的两层。

### 2.4 未来不确定性

- **Godot 5.x** 的 GDExtension API 可能变动。薄桥接层策略降低修复成本，但不能完全免疫
- **godot-rust** 的社区维护能否持续到项目完成
- **Godot 路线图**是否持续支持 Rust 作为一等 GDExtension 语言（目前 C++ 优先级更高）

## 三、当前方案的性能是否"够用"

这是评估是否需要切换引擎的核心问题。

### 3.1 帧预算分析

```
60fps 帧预算: 16.7ms

典型和平帧:
  Rust 模拟: ~4.0ms
  Godot 渲染: ~5.5ms
  玩家物理: ~0.3ms
  TOTAL: ~9.8ms ✅ 充足余量 (~6.9ms)

典型战斗帧:
  Rust 模拟: ~4.5ms (含战斗 AI)
  Godot 渲染: ~5.5ms
  玩家物理: ~0.5ms
  TOTAL: ~10.5ms ✅ 充足余量 (~6.2ms)

最坏重帧 (峰值叠加):
  Rust 模拟: ~6.5ms
  Godot 渲染: ~8.0ms (非峰值独立)
  玩家物理: ~0.5ms
  TOTAL: ~15.0ms ✅ 仍在预算内 (~1.7ms)

帧率瓶颈排名:
  #1 VRAM → 影响 Godot 渲染 (任何引擎都有)
  #2 Godot 渲染 (8ms 预算) → Bevy 可降低，但幅度有限
  #3 Rust 模拟 (6.5ms 最坏) → Bevy ECS 可改善，但当前方案已优化到位
```

### 3.2 关键结论

**当前方案的性能对于 WoWorld 的设计目标来说是最充足的。** 6.9ms（和平帧）和 6.2ms（战斗帧）的帧预算余量意味着：
- 有空间容纳未来新增的模拟/渲染特性
- 不会在日常开发中触及性能瓶颈
- 性能优化不是当前的紧迫任务

**切换到 Bevy 带来的"性能余量"目前是过剩的。** 10.4x boids 基准优势听起来很大，但当 L1 NPC 上限是 1000 时，Godot SoA+rayon 的 2.5ms 已经够用了。边际性能收益递减。

## 四、时间成本评估

### 4.1 三种方案的时间投入预估（5 年项目周期）

| 时间类别 | Godot+GDExtension (现状) | 纯 Bevy | Godot+Bevy 混合 |
|---------|--------------------------|---------|-----------------|
| **初始工具链搭建** | ✅ 已完成 (v4.0 审计通过) | 10-18 月 | 3-6 月 |
| **API 维护 (每 3-4 月)** | 极少 (LTS) | 1-3 天/次 | 1-3 天/次 (godot-bevy 适配) |
| **5 年累计 API 维护** | ~0 | ~40-120 天 | ~40-120 天 |
| **桥接层维护** | 已有 (薄) | 无 (全 Rust) | godot-bevy 维护跟踪 |
| **Rust 编译等待 (日常)** | 少 (仅模拟模块) | 多 (含 bevy 依赖) | 多 (含 bevy + godot-bevy) |
| **可投入游戏内容开发的时间** | ✅ 从 Day 1 | ⚠️ 1-1.5 年后 | ⚠️ 3-6 月后 |

### 4.2 机会成本

对于 Solo 开发者，花在引擎建设上的每一年都是**不花在游戏设计、内容创作、趣味性打磨上的一年**。WoWorld 的核心挑战——让 NPC 真的像"人"、让世界真的涌现出有趣的故事——与引擎选择关系不大。这些挑战需要的是大量的迭代、测试、和打磨。

## 五、小结

| 维度 | 评级 | 说明 |
|------|------|------|
| 渲染能力 | ⭐⭐⭐⭐⭐ | Godot 成熟渲染 |
| 编辑器/工具 | ⭐⭐⭐⭐⭐ | 生产级工具链 |
| 模拟性能 | ⭐⭐⭐⭐☆ | SoA+rayon 够用有余 |
| 迭代速度 | ⭐⭐⭐⭐☆ | GDScript 热重载 + Rust 编译 |
| 生态成熟度 | ⭐⭐⭐⭐⭐ | 最低风险 |
| API 稳定性 | ⭐⭐⭐⭐⭐ | LTS 锁定 |
| 架构纯粹性 | ⭐⭐⭐☆☆ | FFI 边界是技术债务 |
| 长期可维护性 | ⭐⭐⭐⭐☆ | 有两层，但薄桥接降低风险 |
| **完成项目的可能性** | ⭐⭐⭐⭐⭐ | Solo 开发的最优风险收益比 |

**保持现状不是"不作为"——它是经过评估后的主动选择。** v4.0 技术栈经过了与 Bevy、Unity、Unreal、wgpu 的多轮比较后胜出。除非出现了实质性的新信息改变了评分依据，否则不更换引擎是合理的默认策略。
