## 一、跨维度评分总表

| 维度 | Godot+GDExtension (现状) | 纯 Bevy | Godot+Bevy 混合 |
|------|--------------------------|---------|-----------------|
| **渲染能力** | ★★★★★ 开箱即用 | ★★★☆☆ 需自建/社区 crate | ★★★★★ 保留 Godot |
| **编辑器/工具** | ★★★★★ 生产级 | ★★☆☆☆ 社区原型 | ★★★★★ 保留 Godot |
| **模拟性能** | ★★★★☆ SoA+rayon 够用 | ★★★★★ ECS 原生 | ★★★★★ Bevy ECS |
| **迭代速度** | ★★★★☆ GDScript 热重载 | ★★★☆☆ Rust 编译 | ★★★☆☆ 双构建 |
| **生态成熟度** | ★★★★★ 数千款游戏 | ★★★☆☆ ~12款商业游戏 | ★★☆☆☆ 小众 |
| **API 稳定性** | ★★★★★ LTS | ★★☆☆☆ 每 3 月 breaking | ★★☆☆☆ 双重风险 |
| **架构纯粹性** | ★★★☆☆ FFI 边界 | ★★★★★ 全 Rust | ★★☆☆☆ 三层架构 |
| **长期可维护性** | ★★★★☆ 薄桥接降低风险 | ★★★★☆ Rust 类型安全 | ★★★☆☆ 三重耦合 |
| **完成项目的可能性** | ★★★★★ **最低风险** | ★★★☆☆ 高不确定性 | ★★★☆☆ 中等 |

### 评分说明

- **纯 Bevy** 在架构纯粹性和模拟性能上领先，但在对 Solo 开发最关键的维度（编辑器、迭代速度、生态成熟度、API 稳定性）上全面落后
- **混合方案** 试图取两家之长，但在最关键的两个维度（API 稳定性、长期可维护性）上创造了自己的弱点
- **现状方案** 没有最高分项，但也没有最低分项——**它是最均衡的选择**

## 二、针对 WoWorld 特定需求的逐项评估

### 2.1 体素世界生成 (Transvoxel)

| 方案 | 可用方案 | 成熟度 | 自建工作量 |
|------|---------|--------|-----------|
| Godot | godot_voxel (Zylann) — 完整 Transvoxel + LOD + 流式加载 + 碰撞生成 | ⭐⭐⭐⭐⭐ 年量级 | 低 |
| Bevy | Voxelis SVO-DAG (2025.04) — 更先进的压缩架构，99.999% 压缩率 | ⭐⭐⭐☆☆ 月量级 | 中 (集成+适配) |
| Bevy | bevy_cube_marcher / bevy_meshem — 体素网格生成基础 | ⭐⭐⭐☆☆ | 高 (需自建 Transvoxel LOD 过渡) |

**结论**：Godot 的 godot_voxel 是**唯一开箱即用的完整 Transvoxel 方案**。Bevy 的 Voxelis 架构更先进（SVO DAG 压缩率惊人），但需要更多集成工作。如果体素是世界生成的核心，Godot 的现有方案降低了巨大的初始风险。

### 2.2 NPC 大规模模拟 (L1-L4 分层)

| 方案 | L1 (≤1,000) | L2 (≤10,000) | L3 (~89,000) | L4 (百万) |
|------|------------|-------------|-------------|----------|
| Godot SoA+rayon | ≤2.5ms ✅ | 轻量 ✅ | 统计级 ✅ | 宏观 ✅ |
| Bevy ECS | 更优 ✅ | 更优 ✅ | 可提升活跃度 | 可提升活跃度 |

**结论**：当前 L1-L4 分层设计已经是对性能约束的合理响应——在资源有限的情况下，不是所有 NPC 都需要完整心智。Bevy ECS 提供了"让更多 NPC 更活跃"的**未来可能性**，但这在当前设计中不是急需的。L1 上限 1,000 + L2 上限 10,000 的设定在 Godot SoA+rayon 下完全可行。

### 2.3 9层动画栈 + 双骨蒙皮

| 方案 | 动画混合 | 面部图集 | GPU 蒙皮 |
|------|---------|---------|---------|
| Godot | AnimationTree + BlendSpace2D + StateMachine ✅ | 自定义 shader + 512² 图集 ✅ | GPU skinning shader + Rust CPU 骨骼矩阵 ✅ |
| Bevy | bevy_animation 基础关键帧 | 需自建面部图集系统 | 需自建双骨蒙皮 shader |

**结论**：这是 Bevy 方案中最显著的缺口之一。WoWorld 的动画系统设计的复杂度（9 层栈/38 姿态/15 轨迹/涌现步态/512² 面部图集）直接对标了 Godot AnimationTree 的能力。在 Bevy 中复制这套系统需要**2-4 个月**的全职开发，且需要深厚的图形学和骨骼动画专业知识。

### 2.4 UI 系统

WoWorld 需要的 UI 复杂度和 Bevy 当前能力之间的差距：

| UI 需求 | Godot Control 节点 | Bevy (bevy_ui + egui) |
|---------|-------------------|----------------------|
| 背包 (网格/拖放/提示) | ✅ GridContainer + Drag&Drop | ⚠️ 需大量自建 |
| 对话 (富文本/分支/头像) | ✅ RichTextLabel + Theme | ⚠️ 无内置富文本 |
| 技能面板 (树状/进度) | ✅ GraphNode + ProgressBar | ⚠️ 无内置树状图 |
| 蓝图编辑器 (网格/旋转/验证) | ✅ 自定义 Control 节点 | ⚠️ 需全部自建 |
| HUD (动态切换/小地图) | ✅ Control 层级系统 | ⚠️ 需自建 |
| 主题系统 | ✅ Theme + StyleBox | ⚠️ 代码驱动 |
| 国际化字体 | ✅ 内置支持 | ⚠️ 需额外配置 |

**结论**：这是 Bevy 方案中**最大的实际障碍**。WoWorld 的 RPG 特性决定了复杂 UI 是核心体验的一部分。Bevy 的 UI 系统在为编辑器工具类的界面设计，而非沉浸式游戏 UI。在 Bevy 中达到 Godot Control 节点的便利性，需要非常大量的投入。

### 2.5 音频系统

WoWorld 的音频需求（CHG-030）包括 AudioQuery 30 方法、传播引擎（衰减/吸收/风/温度/遮挡/多普勒）、话语清晰度五档、音乐三层模型。其中：
- **模拟端**（传播建模、AudioQuery trait）已在 Rust 侧，引擎无关 ✅
- **渲染端**（3D 空间音频、混音、效果总线、音乐播放）需要引擎支持

| 方案 | 音频渲染 |
|------|---------|
| Godot | AudioServer + AudioBus 成熟 ✅ |
| Bevy | bevy_audio (基础) + kibaudio (社区填补) ⚠️ |

### 2.6 物理

CHG-033 后，仅玩家 CharacterBody3D 使用 Godot Jolt Physics。Bevy 有 Rapier/Avian 作为替代。这部分的迁移成本很低（1-2 周），因为有成熟的社区 crate。

### 2.7 资产管线与 AI 生成美术

WoWorld 的美术管线是 **AI 生成 (Stable Diffusion) + 手动调整 (Blender)**。这意味着：
- 资产是代码化/批量生成的，而非手动逐个创作
- glTF 2.0 格式对两个引擎都是标准格式
- Godot 的拖放导入在这个管线中优势有限（因为资产是批量生成的）
- Bevy 的代码驱动资产加载可能更适合 AI 生成的批量资产

**但**：Godot 的资产预览/浏览器在**管理**大量 AI 生成的资产时仍有价值——可视化的组织、筛选、预览比纯文件系统管理更高效。

## 三、时间窗口分析

这是从"今天做决定"到"未来做决定"的成本变化：

```
2026 ─── Godot 压倒性优势（编辑器/社区/LTS）
  │        Bevy pre-1.0，每 3-4 月 breaking
  │        切换成本：10-18 月纯迁移
  │
2027 ─── Bevy 可能达到 1.0？编辑器 Phase 1-2？
  │        切换成本降低，但仍需 6-12 月
  │        混合方案的 godot-bevy 集成可能更成熟
  │
2028 ─── Bevy 生态显著成熟
  │        切换成本进一步降低（3-6 月？）
  │        纯 Bevy 方案开始有吸引力
  │
2030 ─── Bevy 可能是成熟的、有完整编辑器的引擎
          纯 Bevy 方案达到与今天 Godot 相当的生产力
```

### 关键洞察

**今天切换 Bevy 的代价最高，收益最小。** Bevy 的每一次发布都在降低未来的迁移成本。推迟决定不是犹豫——是战略性的时机选择：

- **现在迁移**：付出 10-18 月的"引擎税"，在 Bevy 最不成熟的版本上开发
- **3 年后迁移**：Bevy 更成熟 → 迁移成本更低 → Rust 核心保持引擎无关 → 迁移不影响模拟代码
- **不迁移**：Godot 一直在变得更好。Godot 4.6 LTS 不会消失

## 四、结论与建议路径

### 4.1 推荐路径：保持现状，持续观察

```
                     Godot 4.6 LTS + Rust GDExtension
                     (v4.0 技术栈，持续开发)
                              │
                              │  保持 Rust 核心引擎无关
                              │  建立 Bevy 观察清单
                              │
                     ┌────────┴────────┐
                     │                 │
              每 6 个月评估         遇到实质性能瓶颈时
              Bevy 生态进展         评估 ECS 迁移收益
                     │                 │
                     ▼                 ▼
              阅读 Bevy 发布说明    如果瓶颈确认 →
              检查关键 crate 成熟度  评估混合方案渐进迁移
              更新本文档             或等待 Bevy 成熟度达标
```

### 4.2 Bevy 观察清单（触发重新评估的条件）

当以下条件**同时**满足时，重新打开引擎切换讨论：

| 条件 | 当前状态 (2026.06) |
|------|-------------------|
| ✅ Bevy 达到 1.0+ 或 API 稳定承诺 | ❌ pre-1.0 (0.18.1) |
| ✅ 官方编辑器进入可用阶段 (≥Phase 2) | ❌ Phase 1 原型 |
| ✅ UI 系统有成熟 widget 库 (树/表/标签/滚动/富文本) | ❌ 基础 Flexbox |
| ✅ 动画系统支持 BlendSpace/多层混合 | ❌ 基础关键帧 |
| ✅ 音频系统有完整混音/DSP 管线 | ⚠️ kibaudio 填补部分 |
| ✅ 至少 1 个商业成功的开放世界 RPG 或体素游戏 | ❌ 0 个 |
| ✅ Voxelis 或其他体素方案成熟到开箱可用 | ⚠️ v0.x (2025.04) |

### 4.3 如果最终还是决定切换

**选择纯 Bevy（场景 A）而非混合方案（场景 B）。**

理由：混合方案的"两全其美"承诺在实践中变成了"双重维护负担"。对于 Solo 开发者：
- 纯 Bevy 的架构纯粹性在长期会降低维护成本（一套类型系统、一套调试工具、一套构建流程）
- 混合方案的三层架构在持续演进中累积的维护成本，可能超过一次性迁移到纯 Bevy 的成本
- 混合方案适合**渐进迁移**的过渡期（3-12 月），不适合作为**永久架构**

具体路径：
1. 等待 Bevy 满足"观察清单"条件（预计 2027-2028）
2. 在 Rust 模拟核心中预留 Bevy ECS 适配层（`#[cfg(feature = "bevy")]`），保持编译可选
3. 先在**副本分支**上执行概念验证迁移（Transvoxel + 100 NPC + 基础 UI）
4. 确认迁移可行性和实际工时后，再决定正式切换

### 4.4 底线

> **WoWorld 最大的风险不是技术栈不够优雅，而是项目无法完成。**
>
> Godot + Rust GDExtension 是当前时间点（2026.06）让 Solo 开发者**最有可能完成项目**的技术方案。它可能不是架构上最漂亮的，但它是工程上最务实的。
>
> Bevy 是未来——但不是今天。保持 Rust 核心的引擎无关性，你已经在为那个未来做好了准备。当 Bevy 成熟的那一天到来时，迁移不会是从零开始。
>
> 在那一天到来之前，把精力花在让 WoWorld 的 NPC 真的像"人"、让世界真的涌现出故事上——这些挑战与引擎选择无关，它们才是 WoWorld 真正的核心。
