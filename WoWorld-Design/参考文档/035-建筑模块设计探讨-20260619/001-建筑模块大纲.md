## 一、模块定位

建筑模块（`woworld_building`）是 WoWorld 中**建筑物理与建造规则**的权威模块。它是 Rust 侧的薄层 crate，定义建筑的组件化数据模型、约束求解、施工调度，通过 trait 与其他模块解耦。

### 核心职责（本模块做什么）

1. **组件族定义**：参数化的结构组件族（墙、地板、门、窗、柱、梁、楼梯、屋顶等），支持 Mod 扩展
2. **建筑数据模型**：`Building`（组件图）、`ComponentInstance`、`BuildingHistory`（增量修改链）
3. **约束求解**：WFC 建筑级求解（世界生成）+ Blueprint 验证 + 结构完整性验证 + 局部增量求解（NPC 翻修）
4. **施工调度**：`ConstructionScheduler`——声明式任务 + 阶段级渲染，不碰材料获取
5. **Blueprint 格式**：玩家 DIY 建筑的 TOML 文件格式，跨存档可分享

### 核心里程碑（本模块不做什么）

- ❌ 家具管理 → [[物品系统]] 模块（`PlacedItem`）
- ❌ 房间功能标签 → NPC [[认知与智慧系统]] 的 `MentalModel`（涌现）
- ❌ 建筑风格决策 → [[文化系统]] / [[信仰系统]] / [[审美系统]]
- ❌ 材料获取规划 → NPC 自身的 `EconomicBehavior`
- ❌ 建筑好坏评判 → NPC 社交 + [[审美系统]]

## 二、架构总览

三层 + 一个注册中心 + 一个生成器族：

```
woworld_building crate:
  ComponentRegistry（组件族注册 + 连接兼容性矩阵）
  ├─ L1 数据层: Building, ComponentInstance, BuildingHistory, BuildingQuery trait
  ├─ L2 求解层: WfcBuildingSolver, BlueprintValidator, StructuralSolver, LocalIncrementalSolver
  └─ L3 施工层: ConstructionScheduler, ConstructionTask, ConstructionModifier trait

  BuildingGenerator trait（多种生成策略）:
    ├─ WfcRectangularGenerator（80% 建筑）
    ├─ WfcRadialGenerator（圆形/塔楼）
    ├─ CathedralGenerator（十字平面宗教建筑）
    ├─ ComplexGenerator（建筑群——皇宫、城堡、修道院）
    ├─ WallGenerator（线性城墙）
    ├─ BridgeGenerator（桥梁）
    ├─ UndergroundGenerator（地下建筑）
    └─ ShelterGenerator（简易庇护所）
```

## 三、对外三扇门

| 门 | 类型 | 方向 | 说明 |
|----|------|------|------|
| **BuildContext** | struct（建筑模块定义） | 入 | 六个外部参数（种族体型/气候/资源/文化/信仰/业主偏好）聚合为一个 struct，调用方填充后传入 |
| **BuildingQuery** | trait | 出 | 建筑世界数据的唯一只读接口——空间查询、结构查询、表面查询、历史查询 |
| **ConstructionTask + MaterialRequirementList** | struct | 出 | 施工需求的声明式产出——NPC/玩家自行规划材料获取方式 |

## 四、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 建筑基本单元 | **参数化组件族**（非房间、非体素） | 房间是 BSP 分区的设计单位；组件是物理单位；体素是地形的 |
| 网格系统 | **1.0m 主网格 + 0.25m 子网格** | 对齐精度，不限制组件尺寸（组件尺寸由种族/文化参数驱动） |
| WFC 层归属 | 城市级 WFC 留在 [[世界生成]]，建筑级 WFC 在建筑模块 | 城市级 WFC 关心功能分区→建筑类型（世界生成的职责）；建筑级 WFC 关心组件布局（建筑模块的职责） |
| 风格驱动 | **BuildContext 参数化**，不硬编码 | 文化/气候/种族/材料可用性四个维度驱动组件参数，任何风格都是参数组合的涌现 |
| 蓝图 vs WFC | Blueprint = 玩家 DIY 设计文件；WFC = 世界生成时的自动生成器 | 共享同一套 ComponentRegistry 和 Building 数据模型 |
| 家具 | 归 [[物品系统]] 模块 | 数量级不同（22 万件 vs 200 组件）、物理不同（无结构约束）、生命周期不同 |
| 房间功能 | **涌现标签**，不存储 | 从家具分布 + NPC 使用行为实时派生，每个 NPC 的 MentalModel 各自判断 |
| 表面放置 | `Surface` trait（建筑模块定义）→ [[物品系统]] 消费 | 建筑模块提供可放置表面信息，不关心放了什么 |
| 解耦策略 | **BuildContext 依赖反转** | 建筑模块不 import Culture/Faith/Weather/Life——只定义自己需要的参数 struct |

## 五、WFC 生成方法（建筑级）

### 5.1 矩形建筑（2.5D 三阶段）

1. **BSP 房间分区**：二叉空间分割 → 每个房间的边界 + 类型提示（来自 `BuildingTypeTag`）
2. **逐层 2D WFC**（0.25m grid）：~100 tile × 2400 cell，权重来自 BuildContext 文化/气候参数。约束传播逐层求解（多层时上下层对齐）
3. **3D 组装**：cell 合并 → ComponentInstance + ConnectionFace 边图 + 屋顶添加 + 结构完整性验证

单栋耗时 < 5ms。

### 5.2 径向建筑

`RadialWfcGrid`：环+扇形 cell。解锁圆形塔楼、土楼、竞技场。支持径向+矩形混合（如万神殿：圆形主体+矩形门廊）。

### 5.3 建筑群

`ComplexGenerator`：多栋建筑 + 轴线/对称约束 + 共享空间（庭院/广场）+ 跨建筑连接（连廊/空中廊桥）。

| 操作 | 预算 | 频率 |
|------|------|------|
