## 〇、问题

是否需要一个独立的"法律/犯罪/惩罚系统"模块？

**结论：不需要。** 法律从现有四个系统的交互中涌现。但现有模块间存在接口缺口和连接标注遗漏需要补齐。

---

## 一、法律涌现的完整链路

法律不是预设的规则集合——它是 PrescribeRule + Sanction + Adjudicate + NPC 自主决策 + 信息传播的涌现结果。每一环都已经由现有模块覆盖：

### 1.1 链路全景

```
① 规则存在
   权力/002 PrescribeRule 原子 → 权力/005 Duty 自动生成
   例: "苍山领主 PrescribeRule(禁止盗窃, scope=苍山城辖区)"

② 违规发生
   玩家或NPC做出被 Prohibition 型 Duty 禁止的行为

③ 目击
   NPC感官系统（NPC活人感/08 · SensoryProvider trait）
   视觉/听觉范围内的事件 → PerceptBatch

④ 记忆编码
   NPC活人感 §2.2 · MemoryEncoder
   事件类型 Combat/Ceremony/Revelation → 必然编码
   其他事件 impact_score > 0.1 → 阈值编码

⑤ 推理归因
   NPC活人感 §1.1 · MentalModel（世界运作信念，≤20条）
   NPC活人感 §1.1 · MindAttribution（Theory of Mind，≤16条）
   概念与语言地基/011 · deductive_chain() 演绎推理
   "谁在附近？谁的 MentalModel 模式匹配这个行为？"

⑥ 决策：告发还是沉默？
   NPC活人感 §2.3 · 概率决策器
   权重 = 文化习俗 × 个人习惯(HabitSystem) × 人格偏移 × 情绪修正
        × 心境修正 × need_action_match × 与受害者/加害者的关系
   文化/003 · CommunicationNorms.directness —— 委婉文化更倾向间接处理
   语言表达/005 · DialogueIntent::GoalDriven { action: "report_crime" }

⑦ 权威受理
   权力/002 · PrescribeRule 的 holder 收到报告
   holder 自己的 GOAP/概率决策器决定：
   - 调查（高尽责性 + 高关系亲密度 → 优先）
   - 无视（低尽责性 或 与加害者关系更好）
   - 立即制裁（高冲动性 + 高愤怒 + 证据确凿）
   - 政治化利用（高宜人性低 + 高战略思维 → 借机打压对手）

⑧ 调查
   = NPC与证人对话（语言表达/005）+ MentalModel推理
   权威可以 Compel(witness_testimony) —— 强制证人作证
   权威可以 Access(search)——搜查嫌疑人的住所
   每个证人的记忆有独立的 confidence 和 decay_factor

⑨ 裁决
   权力/002 §4.5.1 · Adjudicate 原子
   holder 对 subject 作出有约束力的裁定
   裁决依据 = 证人证词 + 权威的 MentalModel + 文化规范内化 + 人格

⑩ 制裁执行
   权力/005 §2.3 · 制裁塌缩链
   阶段1: 找到所有对此 obligor 持有 Sanction 权力的 holder
   阶段2: 过滤有意愿者（调用 NPC 决策引擎 → 见下文 §3.2 缺口）
   阶段3: 规范层级排序
   阶段4: 执行或 legitimacy 下降

⑪ 惩罚形式
   Extract(罚金/没收财产)
   Constrain(监禁 = behavior=leave_prison · 放逐 = behavior=enter_territory)
   Compel(赔偿劳动/社区服务)
   Remit(赦免 —— 权力/002 §4.5.1)

⑫ 记录与传播
   所有参与者编码 EventMemory
   语言表达/009 · 信息传播系统 —— 逐跳衰减（每跳 ×0.7，信任修正）
   旁听者形成二阶记忆
   名声涌现（分散式 NPC 记忆 × 传播跳数 → 共识）
```

### 1.2 为什么"法律"不是独立模块——17个原子已覆盖全部尺度

| 法律概念 | 权力原子 | 说明 |
|---------|---------|------|
| 法律条文 | `PrescribeRule` + `Duty` 自动生成 | scope=subject群体, 规则内容=Prohibition/Obligation |
| 犯罪 | `DutyBreach` | 对 Prohibition 型 Duty 的违反 |
| 执法权 | `Sanction` + `Constrain` + `Compel` | holder 可对违规者施加后果 |
| 管辖权 | `PowerDomain::Territory` + `DefineMembership` | 空间范围 + 人员范围 |
| 审判 | `Adjudicate` | 有约束力的争议裁定 |
| 赦免 | `Remit` + `Derogate` | Remit=对已判惩罚的豁免；Derogate=对特定个体的规则豁免 |
| 豁免权 | `ImmunitySet` | 外交豁免/议会免责/贵族特权 |
| 执法链 | `Delegate` | 国王→法官→治安官→守卫 的权力委托链 |
| 引渡 | `Represent` | 两政权代表交涉——把逃犯作为交涉标的 |
| 程序正义 | 元规则（约束 holder 的 PrescribeRule） | "领主须经审判方可制裁" |
| 法律冲突 | 规范层级（权力/005 §三） | 委托链深度 > 领域专属 > legitimacy |
| 监狱 | `Constrain(behavior=leave_building)` | 物理建筑 + 守卫 Compel(guard_duty) |

**尺度的统一**：亲子宵禁和帝国刑法用的是同一套原子——`Constrain(behavior=night_outside)` vs `Constrain(behavior=murder)`。差异仅在 scope（一个孩子 vs 全体国民）和 sanction（训斥 vs 死刑）。这是 17 个原子优雅性的核心证明——不需要为每种社会尺度写不同代码。

## 二、为什么独立模块反而是错误的——七维分析

### 2.1 Rust 解耦

独立法律模块需要依赖四个模块：

```
法律模块
  ├── 权力系统（PrescribeRule · Adjudicate · Sanction · Delegate · Territory）
  ├── NPC 系统（记忆 · 推理 · 决策 · 情绪 · 关系）
  ├── 对话系统（信息传播 · 告发行为）
  └── 文化系统（CommunicationNorms · NormInternalization）
```

四个依赖的模块 = 高耦合节点。任何依赖方的接口变更都可能传播到法律模块。而现有方案：**零新依赖**——每个模块只消费已有 trait，NPC 决策时组合调用。

### 2.2 玩家体验

玩家不应该感受到"我触发了犯罪系统"——他应该感受到：

> "老王的剑丢了。他很生气。他在到处问人。他看我的眼神不一样了。"

这就是活人感。独立模块会引入"通缉等级""犯罪记录""悬赏金额"之类 UI 概念——恰好是 WoWorld 刻意不要的层。法律从 NPC 的脸上和话里涌现，不从 HUD 上涌现。

### 2.3 性能

零新系统 = 零新增计算。

- Duty 违约检查：已在权力系统的 Duty 触发机制中（Continuous/Periodic/Conditional）
- NPC 决策（是否告发）：已在每决策周期的概率决策器 + GOAP 中
- 信息传播：已在语言表达/009 的传播系统中

独立模块如果做，要么冗余（重复已有计算），要么替代 NPC 自主决策（破坏涌现）。NPC 模块现有性能预算 < 0.5ms/决策周期，不应加新层。

### 2.4 可维护性

17 个普适权力原子已经覆盖从亲子到帝国的全部权力关系。权力/002 已经写了核心理念：

> "法律涌现：PrescribeRule + Sanction + Adjudicate + Delegate + legitimacy → 法律体系。法律不是预设的——它从规则的制定、执行、被认可中涌现。"

再加"法律专用"数据结构 = 重复表达同一事物 = 两份代码维护同一个概念 = 分歧风险。法律模块的每个"新"数据结构都会是已有原子的**重命名包装**。

### 2.5 数学优雅

制裁塌缩链（权力/005 §2.3）已经是一个干净的数学模型：

```
resolve_breach(duty, breach, topology):
  阶段1: 有权者集合 S1 = {h | h 对 obligor 持有 Sanction 且 domain 覆盖此违规类型}
  阶段2: 有意愿者集合 S2 = {h ∈ S1 | will(h, breach) > θ}
         其中 will() = f(人格, 情绪, 关系, 文化规范, legitimacy变化预期)
  阶段3: 排序 S2 按 规范层级优先级
  阶段4: 对 S2[0] 执行 Sanction → 成功
          若失败 → S2[1] → ... → 全部失败 → legitimacy 批量下降
```

这是一个数学上自洽的流水线。独立法律模块会在上面再套一层状态机——破坏了这个干净的单向数据流。

### 2.6 模块特征

各模块的正交关注点：

| 模块 | 关注点 | 是否定义法律相关新基元 |
|------|--------|----------------------|
| 权力系统 | 谁对谁能做什么 | ✅ PrescribeRule/Sanction/Adjudicate——通用基元 |
| NPC 系统 | 个体如何感知/记忆/决策/行动 | ✅ 记忆/推理/决策——通用基元 |
| 对话系统 | 人之间如何交换信息 | ✅ 对话/传播——通用基元 |
| 文化系统 | 群体的行为规范和沟通准则 | ✅ 规范内化/沟通规范——通用基元 |
| **法律模块（假设）** | — | ❌ 只有复合——"调查""告发""判决"是已有基元的拼装 |

"调查犯罪"是 NPC 行动涌现模块（08/003-领域复合原子）的职责——它是已有物理原子 + 对话 + GOAP 的组装，不属于独立模块的新基元。

### 2.7 涌现式交互

这是最强的论点。法律从个体行为中涌现——这个设计理念已经在权力系统中写死：

> "Polity 从权力边模式的连通分量中涌现，不是预设的政治单元。"

同理，法律体系从 PrescribeRule + Adjudicate + Sanction + 合法性中涌现——不是预设的法规列表。每座城市的"法律"不同，因为 PrescribeRule 的集合不同、holder 的人格不同、文化规范不同、执法意愿不同。

## 三、现有模块间的接口缺口与连接遗漏

虽然不需要新模块，但以下接口缺口和连接标注遗漏需要在现有模块中补齐。

### 3.1 缺口一：PowerTopology 缺少管辖权查询接口

**问题**：当 NPC 决定"告发"时，需要查询"谁对此违规有管辖权"。当前 PowerTopology 的四维索引（出边/入边/空间/原子类型）没有暴露这个复合查询。

**需要的接口**（在 PowerTopology 或 PowerQuery trait 中定义）：

```rust
/// 权力拓扑查询 trait —— 供 NPC 决策和对话系统消费
pub trait PowerQuery: Send + Sync {
    /// 查询对特定位置+违规类型拥有管辖权的 holder 列表
    /// 
    /// 查询逻辑:
    ///   1. 通过空间索引找到覆盖此位置的 PrescribeRule 边
    ///   2. 筛选 PrescribeRule 内容匹配此违规类型的边
    ///   3. 对每条边，找到其 holder
    ///   4. 按 legitimacy × 规范层级优先级 排序
    /// 
    /// 返回: Vec<(holder, edge_id, legitimacy, atom_types)> 
    ///   —— 排好序的管辖权持有者列表
    fn query_jurisdiction(
        &self,
        position: WorldPos,
        violation_type: ViolationType,
    ) -> Vec<JurisdictionHolder>;

    /// 批量查询 —— 同一位置多种违规类型
    fn query_jurisdictions(
        &self,
        position: WorldPos,
        violation_types: &[ViolationType],
    ) -> HashMap<ViolationType, Vec<JurisdictionHolder>>;
}

pub struct JurisdictionHolder {
    pub holder_id: EntityId,
    pub edge_id: EdgeId,
    pub legitimacy: f32,
    pub atom_types: Vec<AtomTypeTag>,  // 该 holder 对此违规持有的权力原子类型
    pub enforcement: f32,              // 执行能力 0-1
}
```

**影响文件**：`权力系统/008-跨模块接口与性能预算.md` —— 在导出 trait 列表中补充 `PowerQuery`。

### 3.2 缺口二：制裁塌缩链"有意愿者"未引用 NPC 决策接口

**问题**：权力/005 §2.3 的阶段 2 "过滤有意愿者"需要调用 NPC 决策引擎，但两边都没有定义这个跨模块调用的 trait 方法签名。

**当前状态**：
- 权力/005 写"阶段 2: 过滤有意愿者"——没有引用任何 NPC 模块接口
- NPC 模块没有"评估制裁意愿"的方法

**需要补充的接口**：

```rust
// 在 NPC 模块的 DecisionEngine trait 或新增的 SanctionWillingness trait 中
pub trait SanctionWillingness: Send + Sync {
    /// 评估此 NPC 对特定违规执行制裁的意愿
    /// 
    /// 意愿 = f(
    ///   人格（高尽责→+意愿, 高宜人→-惩罚性制裁/+修复性制裁）,
    ///   情绪（愤怒→+惩罚意愿, 恐惧→-行动意愿）,
    ///   与受害者的关系（affection + trust）,
    ///   与加害者的关系（affection + trust → -意愿）,
    ///   文化规范内化（NormInternalization.behavioral_compliance）,
    ///   对 legitimacy 变化的预期（制裁执行→legitimacy↑, 制裁忽视→legitimacy↓）,
    ///   个人风险（加害者的权力/报复能力 → -意愿）
    /// )
    fn sanction_willingness(
        &self,
        npc: &NpcData,
        breach: &DutyBreach,
        victim: Option<EntityId>,
        perpetrator: EntityId,
        topology: &dyn PowerQuery,
    ) -> f32;

    /// 评估此 NPC 对特定违规的告发意愿
    /// （用于目击者决策——是告发还是沉默）
    fn report_willingness(
        &self,
        npc: &NpcData,
        witnessed_event: &EventMemory,
        perpetrator: EntityId,
        topology: &dyn PowerQuery,
    ) -> f32;
}
```

**影响文件**：
- `权力系统/005-义务、免疫与规范层级.md` —— §2.3 制裁塌缩链明确引用 `SanctionWillingness` trait
- `NPC活人感模块/NPC活人感开发文档ver2.0.md` —— §1.2 补充 `SanctionWillingness` trait 或标注 DecisionEngine 覆盖此职责

### 3.3 缺口三：NPC 行动涌现模块未注册法律相关复合原子

**问题**：`DialogueIntent::GoalDriven { action: "report_crime" }` 在对话系统文档中作为示例出现，但未在 NPC 行动涌现模块（08/003-领域复合原子）中正式注册。

**需要注册的复合原子**：

| 复合原子 | 组装来源 | 注册位置 |
|---------|---------|---------|
| `ReportToAuthority` | 移动到 holder 位置 + 发起 DialogueIntent(GoalDriven, "report_crime") + 提供目击记忆作为话题 | 08/003 |
| `Investigate` | 对话(NPC←→证人) + MentalModel 推理 + 物理检查现场（如果有物质痕迹）+ 整合证据更新 MindAttribution | 08/003 |
| `Apprehend` | Combat 半自动（目标=制服非致命）+ Compel(surrender) 或 物理拘束 | 08/003 |
| `ExecuteSanction` | 依据 Sanction 参数 → Extract(罚金) / Constrain(监禁) / Compel(赔偿) / Remit(赦免) | 08/003 |
| `Patrol` | 沿预定义路径移动 + 感官警觉度提升 + 定期检查 Duty 违约 | 08/003 |
| `GuardPrisoner` | 维持与囚犯的空间邻近 + 感官警觉度提升 + 对囚犯的 Constrain 检查 | 08/003 |

每个复合原子的组装说明应包括：
- 前提条件（什么触发此原子）
- 所需物理原子列表
- 所需领域复合原子列表
- 参数化（从 NPC 数据 + 世界状态中提取的参数）
- 结果和副作用（Duty 状态变化、记忆编码、信息传播）

**影响文件**：`NPC活人感模块/08-NPC行动涌现与分类/003-领域复合原子与模块注册.md` —— 在法律/执法分类下注册以上复合原子。

### 3.4 缺口四：玩家不对称性

**问题**：NPC 受 GOAP 约束（家庭/工作/社交目标限制其行动范围），玩家不受约束。这导致：

1. **完美犯罪**：玩家杀死唯一目击者 → 零目击记忆 → 不可破案。系统性可重复。
2. **跨辖区逃逸**：玩家跑出领主辖区 → 领主无法追捕（除非跨辖区合作——见 §3.8）。
3. **嫁祸**：玩家把赃物放在无辜 NPC 家 → 权威的 MentalModel 推理可能错误归因。

**判断**：这些不是 bug——是设计理念的自然结果。真实世界中确实存在完美犯罪和跨辖区逃逸。WoWorld 不提供"游戏平衡"来阻止聪明/残忍的玩家——如果玩家想当连环杀手，世界会如实反映他的行为后果（但仅限被目击/推理出的那些）。

**需要确认的对齐点**：将来设计名声系统时，需要确定是否需要一个"不可抹除的痕迹层"——比如 AetherImprint（历史/004）记录重大事件，即使所有目击者死亡。这不需要现在决定，但值得标注为名声系统与法律涌现的交叉关注点。

### 3.5 缺口五：程序正义 = 元规则 —— 链路已通但未文档化

**问题**：程序正义（"定罪前必须听取两方证词"、"死刑须经国王批准"）是约束权力行使者（holder）的 PrescribeRule。这不是新原子——是已有原子的递归应用。

```
元规则示例:
  领主制定: PrescribeRule("禁止盗窃", scope=苍山城民)
  国王制定: PrescribeRule("领主须经审判方可制裁任何人", scope=全国领主)
             ↑ 这是一条 scope 是"领主"的规则——元规则
             
  当领主想制裁盗窃者:
    → 国王的元规则约束领主的 Adjudicate/Sanction 行使方式
    → 规范层级（权力/005 §三）: 国王 > 领主（委托链深度 + 领域专属）
    → 领主必须遵循元规则——否则领主自己也产生 DutyBreach（对国王）
```

法律体系从规则 + 元规则 + 元元规则的递归中涌现——这正是权力系统自指的优雅性。但当前文档没有显式写出这条递归链路。

**影响文件**：`权力系统/005-义务、免疫与规范层级.md` —— §三 规范层级补充"元规则递归"说明。

### 3.6 缺口六：物理连接标注缺失 —— 监狱、守卫、囚犯

**问题**：监狱是 Constrain(behavior=leave_building) 的权力边 + 一个物理建筑 + 守卫 NPC。这是三个已有系统的交汇，但交汇点没有在任何地方标注：

| 关联 | 模块A | 模块B | 当前标注状态 |
|------|-------|-------|------------|

### 3.7 缺口七：记忆衰减 = 自然追诉时效 —— 机制已存在但未声明

**问题**：旧的犯罪因为目击者记忆衰减（decay_factor 每天 λ=0.05，高冲击力慢衰减）、证人死亡、证据消失，自然变得不可追诉。这是涌现式"追诉时效"——不需要规则定义。

但 NPC 模块 §2.2.4 有 `important_event_ids` —— 情绪峰值事件（Δ>0.3）自动标为重要，不受冷记忆压缩影响。这意味着：

- 普通盗窃（低情绪冲击）→ 衰减快 → 几个月后证人记忆模糊 → 自然不可追诉 ✓
- 谋杀目击（极高情绪冲击）→ `important_event_ids` → 几乎永久保留 → 永远可追诉 ✓

这两个机制之间隐含的政策含义：**严重犯罪实际上没有追诉时效**。值得在相关文档中声明这一涌现性质。

**影响文件**：`NPC活人感模块/NPC活人感开发文档ver2.0.md` —— §2.2.4 补充说明：重要记忆的永久保留 → 严重犯罪的自然永久追诉。

### 3.8 缺口八：跨辖区引渡 —— 外交行为表未列出

**问题**：权力/007 定义了 Polity 和 DiplomaticRelation，但没有列出"引渡"作为一种外交操作。跨辖区的罪犯追捕场景：

- **引渡请求**：Polity A → Polity B："交出逃犯 X"（Represent 原子——A 的代表向 B 提出交涉）
- **外交庇护**：Polity B 拒绝引渡："X 在我们这里受保护"（B 行使 Derogate 或 ImmuniteSet）
- **跨境追捕**：Polity A 的执法者进入 B 领土——在权力/006 §2.2 的 `on_entity_enter_territory` 中触发什么？

引渡作为一种外交行为类型，应该和宣战/结盟/通商一样在 权力/007 中列出：

| 外交操作 | 权力原子 | 说明 |
|---------|---------|------|
| 引渡请求 | Represent | A 代表 B："请交出 X" |
| 引渡同意 | Compel(return_to_A) | B 对 X 行使强制返回 |
| 引渡拒绝 | — | B 不对 X 采取行动，A 与 B 关系可能恶化 |
| 跨境追捕 | Constrain 冲突 | A 执法者进入 B 领土 → B 的 auto_apply Constrain 可能触发 DutyBreach |

**影响文件**：`权力系统/007-政治实体与外交涌现.md` —— 补充引渡作为外交操作类型。

### 3.9 缺口九：玩家作为执法者的操作映射

**问题**：玩家可以加入权力结构，成为持有 Sanction 权力的执法者。玩家如何执行逮捕/裁决/制裁？

NPC 走 Combat 半自动（目标=制服而非致命）→ Compel(surrender)。但玩家侧的物理输入（按键映射）需要区分"攻击"和"逮捕"——这是战斗系统的玩家输入映射，不属于新模块：

- **战斗系统**：玩家的 CombatCommand 枚举需要包含 `Apprehend` 选项（制服目标，非致命）
- **玩家**：玩家选择 `Apprehend` 战斗风格 → 战斗 AI 自动做非致命攻击
- **对话系统**：逮捕后玩家可以对被制服者发起对话（"你被捕了"——`DialogueIntent::GoalDriven { action: "announce_arrest" }`）

这属于战斗系统和玩家模块的现有职责，不需要新模块。但需要在战斗/玩家文档中标注。

**影响文件**：`战斗/001-战斗系统总览.md` 或 `生命/011-玩家.md` —— 补充玩家执法操作模式说明。

### 3.10 缺口十：财产犯罪的物品标识

**问题**：NPC 看到玩家拿着他三天前丢失的剑——和看到玩家拿着任何一把剑——反应是否不同？

当前设计中的相关机制：
- NPC 的 EventMemory 记录了"我的剑丢失了"
- NPC 的感官系统看到玩家物品栏中的剑
- MentalModel 可以推理："这把剑看起来像我的 → 可能就是他偷的"

但如果要和普通剑区分——需要物品系统支持：
1. **纯记忆驱动**（当前方案）：NPC 通过 EventMemory + 视觉匹配来识别。不修改物品系统。
2. **物品标签**（可选增强）：`ItemEnt` 增加 `stolen_from: Option<Vec<(EntityId, GameTimestamp)>>` 追踪盗窃来源链——但这引入了物品系统和 NPC 系统的耦合。

**推荐**：保持方案 1（纯记忆驱动）。NPC 通过视觉识别 + 记忆推理来认出失物——这是"活人感"的来源（"这把剑...我怎么看着眼熟？"）。方案 2 的物品标签会引入跨模块耦合，违背解耦原则。

**影响文件**：无需修改。但在物品系统的外部接口段落标注：物品系统不追踪"偷来的"标签——失物识别由 NPC 记忆 + 感官系统负责。
