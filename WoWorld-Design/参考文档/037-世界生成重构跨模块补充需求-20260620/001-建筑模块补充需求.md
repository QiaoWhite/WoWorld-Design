# 001 — 建筑模块补充需求

> **目标模块**: [[../../Happy Game/开发阶段/建筑模块/README|建筑模块]]（CHG-043）
> **紧迫度**: 🔴 高 — P6 建筑生成阶段必须消费
> **日期**: 2026-06-20

---

## 〇、总因：为什么建筑模块需要补充

> **核心矛盾**: 世界生成重构后，P5 产出社会结构骨架——它知道"这个矿业镇需要 3 个铁匠铺、每个 80-120m²、文化风格为北方石砌"。但当前建筑模块的 `BuildingGenerator` trait 只接受单个建筑的功能类型参数——没有标准化的"生成合同"类型来传达这些跨域约束。世界生成需要告诉建筑模块"建什么、建多少、建成什么样"，但不能侵入建筑模块的 WFC 求解逻辑。
>
> **解耦要求**: 建筑模块定义 `BuildingSpec`（"我需要什么参数"），世界生成构造其实例（"我给你这些参数"）。建筑模块内部如何求解 WFC → 世界生成不关心。这就是"合同模式"。

---

## 一、新增类型：BuildingSpec（生成合同）

> **★ 为什么需要**: 当前建筑模块没有标准化的批量生成入口。世界生成 P5 产出的建筑规格包含数量、面积范围、文化风格、功能区偏好等跨域信息，这些信息不属于建筑模块的 WFC 内部知识。`BuildingSpec` 是世界生成→建筑模块的数据合同——它让世界生成能说"我需要这些建筑"，而不需要知道建筑模块怎么求解。

```rust
/// 世界生成→建筑模块的生成合同
/// 定义在 woworld_building crate 中
#[derive(Debug, Clone)]
pub struct BuildingSpec {
    pub function: BuildingFunction,
    pub area_range: (f32, f32),
    pub floor_range: (u32, u32),
    pub culture_style: CultureStyleId,
    pub plot_size: (f32, f32),
    pub count: u32,
    pub preferred_zone: Option<FunctionalZoneType>,
    pub require_street_access: bool,
    pub expected_quality: u8,
    pub household_size_hint: Option<HouseholdSize>,
    pub constraints: Vec<BuildingConstraint>,

    /// ★ AcousticTag — 建筑声学标签。CHG-030 音频系统要求世界生成为每栋建筑预设。
    /// P5 从 建筑功能+文化风格+墙体材质 推导: 石教堂→StoneCathedral, 木屋→WoodenRoom
    pub acoustic_tag: AcousticTag,
}

pub enum HouseholdSize { Small, Nuclear, Extended, Compound }

pub enum BuildingConstraint {
    RequireRoom(RoomFunction, u32),
    FaceDirection(f32),
    AdjacentTo(BuildingFunction),
    RequireBasement,
    TallestInZone,
}
```

## 二、新增接口：generate_buildings()

> **★ 为什么需要**: 建筑模块当前没有暴露一个可供世界生成编排调用的批量入口。世界生成不运行 WFC——它只传入参数，获取结果。`generate_buildings()` 是这个编排边界的体现。

```rust
/// 建筑模块暴露的批量生成接口
/// 纯确定性函数——相同参数 → 相同结果
pub fn generate_buildings(
    specs: &[BuildingSpec],
    terrain: &dyn TerrainQuery,
    settlement_layout: &SettlementLayout,
    rng: &mut DeterministicRng,
) -> Vec<BuildingData>;
```

**设计约束**：
- 此函数是纯确定性函数——相同参数 → 相同结果
- 不依赖运行时状态（无天气查询、无 NPC 查询）
- building_id 从 `hash(spec_index, building_index)` 确定性派生

## 三、BuildingData 新增字段

> **★ 为什么需要**:
> - `built_year`: 世界生成需要为每栋建筑赋予"建造年代"，这是 P9 历史事件推导和 P8 NPC "爷爷建了铺子"推导的基础事实。当前 BuildingData 无此字段——建筑模块不应计算年代（它不知道聚落的历史深度），但数据结构需要承载这个值。
> - `owner_npc_id`: P8 将 NPC 绑定到房间——需要知道"谁拥有/居住在这栋建筑里"。初始为 None，P8 填充。
> - `condition`: 当前仅有 `quality_level` (0-4)，不支持"废墟"状态。P9 历史事件可能烧毁/废弃建筑。
> - `culture_style_id`: 建筑风格同时受文化参数和本地材料影响——世界生成选择，建筑模块执行。

```rust
// BuildingData 新增字段:
pub built_year: i32,                              // ★ P8 记忆推导 + P9 历史一致性需要
pub owner_npc_id: Option<NpcId>,                   // ★ P8 NPC-房间绑定需要
pub condition: BuildingCondition,                  // ★ P9 历史损毁事件需要扩展废墟状态
pub functional_zone: FunctionalZoneType,            // ★ P5 城市级布局需要
pub culture_style_id: CultureStyleId,               // ★ 文化→建筑风格映射链需要
```

## 四、丧葬区：墓碑/坟墓生成

> **★ 为什么需要**: P8 生成已死祖先——这些死者需要物理痕迹（墓碑）存在于世界中。004-聚落选址 §5.2 定义了"丧葬区"功能区，但没有个体墓碑生成逻辑。没有这个接口，100K NPC 中的已死祖先（估计 30-50K）将没有任何物理存在——玩家走在聚落中看不到任何死亡的文化证据。墓碑的排列（老墓在中心、新墓在外围）也是一种可感知的历史深度。

```rust
pub struct GraveSpec {
    pub deceased_npc_id: NpcId,
    pub death_year: i32,
    pub social_standing: f32,
    pub is_notable: bool,
}

pub fn generate_graves(
    specs: &[GraveSpec],
    cemetery_zone: &CemeteryZone,
    rng: &mut DeterministicRng,
) -> Vec<BuildingData>;
```
