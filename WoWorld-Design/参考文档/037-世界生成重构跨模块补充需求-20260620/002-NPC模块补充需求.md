> **紧迫度**: 🔴 高 — P8 人口生成必须消费
> **日期**: 2026-06-20

---

## 〇、总因：为什么 NPC 模块需要补充

> **核心矛盾**: 当前 NPC 模块将 NPC 视为"已存在的个体"——它有 BigFive、EmotionState、MemoryStore、GOAP 规划器，但**没有"这个 NPC 是怎么被创建出来的"机制**。世界生成需要同时产出一个聚落的所有 NPC（100K+）和他们的家族关系——不是每个 NPC 独立随机生成，而是作为家族和社会网络中的节点。
>
> **缺失的核心能力**:
> 1. **家族结构** — NPC 模块没有 `FamilyTree` 类型。NPC 之间的关系（父子/配偶/兄弟）在内存中仅作为 `RelationEdge` 存在——但这不足以支撑逆向人口投影和"爷爷建了铺子"推导。
> 2. **批量构造路径** — `NpcData` 的 4.3KB 字段（CHG-044）没有一个让世界生成可以高效批量填充的构造签名。
> 3. **预存记忆的冗余** — 原设计倾向在世界生成时预填充记忆，但这和信息推导哲学矛盾。
>
> **解耦要求**: NPC crate 拥有 `FamilyTree` 类型和运行时方法。世界生成批量构造实例并填充边——但不拥有类型定义。

## 一、新增类型：FamilyTree

> **★ 为什么需要 FamilyTree（而不仅仅是 Vec<RelationEdge>）**
>
> 1. **逆向人口投影需要图结构**: P8 的逆向投影算法从活着的 NPC 反推祖先——父亲、祖父、曾祖父。这个过程产出的祖先节点需要和有向边（father_of, mother_of）建立 DAG。`Vec<RelationEdge>` 可以存储这些边，但没有 DAG 的图查询能力（parents_of / children_of / blood_distance / can_marry / household_members）。
> 2. **"爷爷建了铺子"需要代际查询**: NPC 被问及过去时，调用 `family_tree.parents_of(npc)` → 找到父亲 → 递归找到祖父 → 查询祖父的职业（从 FamilyEdge 关联的 NpcId 查找祖父的 ProfessionTags）→ 推导"你爷爷是铁匠"。这个查询链不存在于现有的扁平 RelationEdge 中。
> 3. **婚姻配对需要血缘距离检查**: 跨聚落婚姻配对需要检查 `blood_distance(a, b) >= min_distance`——防止近亲结婚。这个查询需要 DAG 上的最短路径。
> 4. **家庭居住聚合需要 household_members()**: P5 的 `HouseholdSize`（核心家庭 vs 扩展家族）影响 BuildingSpec 的面积——需要知道"谁和谁住在同一个屋檐下"。这个信息仅从 FamilyTree 的 DAG 结构 + 文化 individualism 参数推导。

```rust
pub struct FamilyTree {
    pub edges: Vec<FamilyEdge>,
    outgoing: HashMap<NpcId, Vec<usize>>,
    incoming: HashMap<NpcId, Vec<usize>>,
}

pub struct FamilyEdge {
    pub from: NpcId,
    pub to: NpcId,
    pub relation: FamilyRelation,
    pub established_at: i32,
    pub dissolved_at: Option<i32>,
}

pub enum FamilyRelation {
    FatherOf, MotherOf,
    MarriedTo { start_year: i32, end_year: Option<i32> },
    SiblingOf,
}

impl FamilyTree {
    pub fn parents_of(&self, npc: NpcId) -> Vec<(NpcId, FamilyRelation)>;
    pub fn children_of(&self, npc: NpcId) -> Vec<NpcId>;
    pub fn blood_distance(&self, a: NpcId, b: NpcId) -> Option<u32>;
    pub fn can_marry(&self, a: NpcId, b: NpcId, min_blood_distance: u32) -> bool;
    pub fn household_members(&self, npc: NpcId, culture_individualism: f32) -> Vec<NpcId>;
}
```

## 二、NpcData 构造友好性

> **★ 为什么需要批量构造函数**
>
> 世界生成需要在 P8 一次性创建 100K NpcData 实例。每个实例需填充 ~4.3KB 字段（CHG-044 + 认知系统 + 生命周期）。如果使用逐个 setter 模式，100K × ~50 个字段 = 500 万次函数调用——在加载画面中不可接受。
>
> 需要暴露一个接受紧凑参数表的批量构造函数。同时明确区分三类字段：
> - **生成时填充**: 年龄、性别、BigFive、技能、职业、家族位置、建筑绑定、语言proficiency
> - **运行时派生**: 情绪状态（EmotionEngine 首帧计算）
> - **运行时增量**: 记忆存储（MemoryStore 从空开始）

```rust
impl NpcData {
    pub fn new_generated(/* 所有生成时需要的字段 */) -> Self;
}

// 明确的三类字段标记:
// === 世界生成填充 ===
pub family_position: FamilyPosition,
pub death_year: Option<i32>,
pub death_cause: Option<DeathCause>,

// === 运行时填充 ===
pub memory_store: MemoryStore,     // ★ 生成时为空——运行时增量追加
pub emotion_state: EmotionState,   // ★ 运行时首帧计算
```

## 三、RelationEdge 的初始社会关系

> **★ 为什么需要初始 RelationEdge**
>
> 玩家进入的世界不是"所有人刚见面"——NPC 之间已经存在雇主-雇员、师徒、邻里、熟人关系。这些关系影响：
> - 权力边的选择：统治者更可能从已有 social_standing 的候选人中选择
> - 经济交互：NPC 倾向于和已知的 TradePartner 交易（降低信息不对称）
> - GOAP 社交目标：NPC 不会去和"完全陌生"的人社交——他们已有 Acquaintance 网络
>
> 这些初始关系从 P5 的 RoleTemplate + P8 的 FamilyTree 自动推导——不需要手写。

```rust
pub enum SocialRelationType {
    Employer, Employee, Colleague, Apprentice,
    Neighbor,
    Acquaintance, Friend, Rival,
    TradePartner, Creditor, Debtor,
}
```

## 四、废弃：SelfNarrative 预存方案

> **★ 为什么废弃 SelfNarrative**
>
> **原设计**: 世界生成时预计算每个 NPC 的 `SelfNarrative`（~200B 紧凑摘要），包含 life_events[5]、social_trajectory、identity_anchors u64 bitmask。
>
> **废弃原因（三重）**:
> 1. **冗余**: SelfNarrative 的所有信息都可以从 FamilyTree + ProfessionTags + BuildingData + EventLog 中实时推导。预存储意味着同样的信息存了两份。
> 2. **上限矛盾**: 5 个 life_events 槽位不足以覆盖一个 47 岁 NPC 的人生。选择"哪 5 个事件最重要"本身就是文化偏见——不同文化对"重要事件"的定义不同。
> 3. **硬编码风险**: identity_anchors 的 64 位 bitmask 需要定义"铁匠""父亲""幸存者"等位——谁决定哪些身份有专属位？这本质上就是硬编码身份类别。

**替代**: 无状态查询函数 `query_personal_history()`，从结构化事实实时推导，不缓存。

```rust
pub fn query_personal_history(
    npc: &NpcData, family_tree: &FamilyTree, event_log: &EventLog,
    building_data: &BuildingData, settlement_tech: &SettlementTechState,
    query_context: QueryContext, rng: &mut DeterministicRng,
) -> PersonalHistoryFragment;
```

## 五、NpcAnchors — 家·工作·产权（★ 新增 — 解决缺口 11/12/14）

> **★ 为什么需要**
>
> NPC 与物理空间的关系是多层的：住哪、在哪工作、拥有哪些房产。当前 `BuildingData.owner_npc_id` 只能表达"谁拥有这栋建筑"——无法表达"谁租住在里面"、"谁在那打工但不是主人"。反过来，NPC 端也不知道自己的家和工作场所在哪。
>
> 解法: 不让建筑声明"谁在里面"——让 NPC 声明"我在哪里"。

```rust
/// 定义在 woworld_npc crate 中
pub struct NpcAnchors {
    pub home: Option<HomeAnchor>,
    pub workplace: Option<WorkplaceAnchor>,
    pub owned_buildings: Vec<BuildingId>,     // 产权
}

pub enum HomeAnchor {
    Building { building_id: BuildingId, room_id: RoomId },
    NomadicCamp { current_camp: Vec2 },
    Homeless { usual_spot: Vec2 },
    Transient { inn_building: BuildingId },
}

pub enum WorkplaceAnchor {
    Building { building_id: BuildingId, room_id: Option<RoomId> },
    Farmland { plot_index: usize },           // → P5 FarmlandPlot
    Pasture { zone_index: usize },            // → P5 PastureZone
    Mine { mine_id: MineId },
    Waterfront { position: Vec2 },
    Roaming { waypoints: Vec<Vec2> },
    MarketStall { market_id: MarketId },
}
```

**建筑→NPC 匹配算法**（P8 Phase D 执行，非预存）：
```rust
fn assign_workplaces(npcs: &mut [NpcData], buildings: &[BuildingData], outdoor: &[OutdoorWorkplace]) {
    for occupation_group in group_by_occupation(npcs) {
        let candidates = matching_workplaces(occupation_group, buildings, outdoor);
        for npc in occupation_group {
            let best = candidates.iter()
                .filter(|w| !w.is_fully_occupied())
                .map(|w| (w, score(npc, w)))           // skill×distance×family_history
                .max_by_key(|(_, s)| *s);
            npc.anchors.workplace = best.map(|(w, _)| w.into());
        }
    }
}
```

## 六、P8 分四阶段生成（★ 新增 — 解决缺口 15 鸡生蛋）

> **★ 为什么需要分阶段**
>
> 单次遍历同时决定职业、家族、技能会产生循环依赖。四阶段打破循环。

```
Phase A — 骨架: 按年龄/性别生成占位 NPC（职业=未分配）。输出 NpcSkeleton × N
Phase B — 家族: 逆向人口投影。婚姻配对→子女→反推祖先。输出 FamilyTree
Phase C — 职业世系: 整数规划全局分配。满足 RoleTemplate 约束。
            从 家族继承倾向 + settlement_tech + BigFive适配度 → 职业标签
Phase D — 实例化: 技能等级+BigFive+NpcAnchors+初始动作状态。输出完整 NpcData
```

**Phase C 的整数规划**：匈牙利算法或最小费用流。约束: Σ(职业=铁匠) ≥ P5 RoleTemplate 铁匠需求数。目标函数: 最大化 Σ(家族继承 bonus + BigFive适配度)。

## 七、NPC 初始动作与位置（★ 解决冷启动）

> **★ 为什么需要**
>
> 世界生成完成时，所有 NPC 的 `current_action` 和 `current_position` 未被设置。玩家进入首帧，GOAP 第一规划周期才会产生动作——玩家看到 NPC 从"静止在家"过渡到"开始工作"。这破坏"世界已运转千年"的沉浸感。
>
> **方案**: 不做预模拟。从 ProfessionTag 的典型日程模板 + 当前时间 + 建筑绑定 → 直接推导首帧初始状态。

```rust
pub fn initial_action_state(
    npc: &NpcData,
    profession_tags: &[ProfessionTagId],
    time_of_day: TimeOfDay,          // SeasonClock(world_time=0)
    home: Option<BuildingId>,
    workplace: Option<BuildingId>,
    rng: &mut DeterministicRng,
) -> InitialNpcState {
    // 从 ProfessionTag TOML 的 daily_schedule 字段推导，非硬编码:
    // 铁匠 daily_schedule = [{06-18: workplace, WORK}, {18-06: home, SLEEP}]
    // 无日程→从同 category 职业的默认模板继承
}

pub struct InitialNpcState {
    pub current_position: Vec3,
    pub current_action: ActionId,
    pub current_location_type: LocationType,
}
```

**数据驱动**: ProfessionTag TOML 新增可选 `daily_schedule`。不定义 = 从同 category 默认模板继承。所有职业可通过 TOML 自定义——零硬编码。
