## 〇、总因：为什么经济系统需要补充

> **核心矛盾**: 经济系统声明"价格从交易中涌现，不由函数计算"——这是正确的运行时哲学。但世界生成需要**初始经济状态**。没有初始订单簿 → 没有价格信号 → NPC 无法做消费决策 → 经济无法启动。经济涌现需要初始条件，就像物理模拟需要初始位置和速度。
>
> 目前经济系统有运行时的 `Market` 撮合引擎，但没有**从资源禀赋+NPC职业分布推导初始供需**的 bootstrap 机制。同时，ProfessionTag TOML 定义了职业名称和关联技能，但**没有定义每个职业消费什么原材料、产出什么产品**——这正是"一个聚落有多少铁盈余"的数据驱动来源。

---

## 一、ProfessionTag TOML 新增字段：consumes / produces

> **★ 为什么需要 consumes/produces 字段**
>
> 当前 ProfessionTag 只定义了职业名称、分类、关联技能。但"铁匠"这个标签不携带"铁匠每年消费 800kg 铁锭 + 400kg 煤，产出 500kg 铁工具"的信息。而这个信息是：
>
> 1. **聚落供需推导的唯一数据驱动源**: 不需要手写"聚落 A 产铁 1000 吨/年"——只需知道聚落 A 有 3 个铁匠（proficiency=0.7, 0.5, 0.3），然后 `800×(0.7+0.5+0.3) = 1200kg` 自动推导。
> 2. **新增职业 = 新增 TOML 条目，代码零修改**: 如果 Mod 加一个"玻璃匠"职业，只需在 TOML 中定义它消费沙子/产出玻璃——供需聚合自动工作。
> 3. **essential 标记驱动涌现式危机**: 铁匠标记 coal 为 essential=true → 如果缺煤，铁匠无法工作 → GOAP 自动选择替代行动（买煤/转行/迁徙）。不是代码中的 `if no_coal { disable_blacksmith() }`。
> 4. **避免硬编码聚落经济特征**: 如果不走数据驱动，就必须手写"矿业镇=产铁，农业村=产粮"——这是硬编码。

```toml
# ★ 如果不在 TOML 中定义，就必须在代码中硬编码每个职业的投入产出
[[profession_tag.consumes]]
item = "iron_ingot"
annual_kg_per_proficiency = 800
essential = true

[[profession_tag.produces]]
item = "iron_tools"
annual_kg_per_proficiency = 500
quality_base = 0.6
```

## 三、新增接口：bootstrap_economy()

> **★ 为什么需要 bootstrap**
>
> 经济涌现需要初始条件。如果没有 bootstrap：
> - 所有 Market 的订单簿为空 → 没有价格信号
> - NPC 的 known_prices 为空 → 无法做消费决策
> - NPC 钱包为 0 → 无法交易
> - 经济陷入"冷启动死锁"
>
> `bootstrap_economy()` 的作用是从供需盈余 + 贸易网络 → 构造"已交易千百年的世界中的当前市场快照"。这不是破坏涌现——是设置涌现的初始条件。运行时第一次交易即可更新价格。
>
> **哲学澄清**: 初始价格 = `base_value_copper × abundance_ratio` 不是"定价公式"。它是对"该商品在已运行千年的市场中的统计概括"的估算。它承认自己是一个近似——运行时订单簿撮合立即取代它。

```rust
pub fn bootstrap_economy(
    settlements: &[SettlementData],
    npcs: &[NpcData],
    supply_demand: &HashMap<SettlementId, SupplyDemandSummary>,
    transport: &TransportNetwork,
    item_defs: &ItemRegistry,
    rng: &mut DeterministicRng,
) -> EconomyState;
```

## 四、NPC 初始钱包分配

> **★ 为什么用 Pareto 分布**
>
> 中世纪社会的财富分配不是均匀的——它是高度集中的（前 20% 拥有 80% 财富）。Pareto 分布（α=1.8）是对这种"自然不平等"的数学近似。这不是"预设谁有钱"——它是从统计规律中抽样。具体的财富值由抽样 + 阶层修正 + 年龄修正 + 继承修正决定——而不是"贵族=10000 铜币, 农民=100 铜币"的硬编码表。

## 附注：CarryingCapacity 归属（★ — 解决缺口 6）

> **★ 为什么需要明确归属**
>
> `CarryingCapacity` 被 P4 选址、P8 人口分配、经济系统供需推导共同消费，依赖地形/水文/群系/资源（四个模块的数据）。但没有任何 CHG 契约指定其 Owner。
>
> **方案**: `CarryingCapacity` 定义为 `woworld_core` 中的纯函数——所有输入在世界生成 P4 时已可用（terrain, resources, culture）。经济系统和 NPC 系统通过 `woworld_core::CarryingCapacity` 消费结果。世界生成 P4 计算并存入 `SettlementData.carrying_capacity`——其他模块不重算，只读取。

## 五、MoneySupply 总量推导（★ 新增 — 解决缺口 4）

> **★ 为什么需要**
>
> 经济 bootstrap 分配了 NPC 钱包和各 Market 的初始订单簿，但没有确保货币总量的一致性。如果货币总量远小于所有商品的总价值，经济会因流动性不足而死锁。

```rust
/// 世界生成 P11 调用——推导初始货币供应量
pub fn initial_money_supply(
    total_npc_count: u64,
    total_settlement_gdp_estimate: u64,  // Σ(每聚落年经济产出 × 流通速度倒数)
    item_defs: &ItemRegistry,
    rng: &mut DeterministicRng,
) -> MoneySupply {
    // GDP 估算: 从所有 ProfessionTag.produces 的年产出 × base_value_copper 聚合
    // 流通速度: 中世纪经济 ~3-5 次/年
    // M0 = GDP / velocity
    // M2 = M0 × (1 + credit_multiplier)  // 借贷创造的广义货币
    //
    // 铸币管道: M0 × minting_ratio → 初始铸币量 (铜币/银币/金币 分布)
    // 消费回收管道: 初始为 0 (运行时从税收/费用中累积)
    // 跨区流动: 各 Market 间货币分布从贸易量推导
}

pub struct MoneySupply {
    pub total_m0: u64,             // 基础货币总量（铜币当量）
    pub total_m2: u64,             // 广义货币（含信贷）
    pub regional_distribution: HashMap<SettlementId, u64>,  // 各聚落货币量
    pub coin_composition: CoinComposition,  // 铜/银/金币分布
}
```

## 六、EconomicRelation 生成（★ 新增 — 解决缺口 7）

> **★ 为什么需要**
>
> 经济系统 README 列出世界生成为 `EconomicRelation` 的提供者（初始商路种子）。P7 生成物理道路，P11 创建 Market——但二者之间的经济语义层（谁和谁贸易什么）未被显式创建。
>
> `EconomicRelation` 承载 Trade/Competition/Dependency 关系——不只是"有路连接"，而是"因为 A 产铁过剩而 B 缺铁，所以 A→B 铁贸易"。这是经济系统理解聚落间关系的核心数据结构。

```rust
/// 世界生成 P11 调用——从供需盈余+交通连接推导聚落间经济关系
pub fn derive_initial_economic_relations(
    settlements: &[SettlementData],
    supply_demand: &HashMap<SettlementId, SupplyDemandSummary>,
    transport: &TransportNetwork,
    rng: &mut DeterministicRng,
) -> Vec<EconomicRelation> {
    for (a, b) in settlement_pairs_within_range(settlements, transport, MAX_ECONOMIC_RANGE) {
        let a_sd = &supply_demand[&a.id];
        let b_sd = &supply_demand[&b.id];

        // 检查互补: B 缺的是不是 A 盈余的？
        for item in a_sd.surplus_items() {
            if b_sd.is_deficit(item) {
                relations.push(EconomicRelation {
                    from: a.id, to: b.id,
                    relation_type: EconomicRelationType::Trade,
                    goods: vec![item],
                    volume: estimate_trade_volume(a_sd.surplus(item), b_sd.deficit(item), distance),
                });
            }
        }

        // 检查竞争: A 和 B 是否在同一市场上竞争同一稀缺资源？
        for item in a_sd.deficit_items() {
            if b_sd.is_deficit(item) && item_is_scarce_in_region(item, a, b) {
                relations.push(EconomicRelation {
                    from: a.id, to: b.id,
                    relation_type: EconomicRelationType::Competition,
                    contested_resource: Some(item),
                    ..
                });
            }
        }
    }
}
```

**关键**: 这不是"指定贸易路线"——是从供需盈余的数学关系中**自动推导**。新增资源 → 新增 ProfessionTag → 经济关系自动更新。

## 七、第一笔交易验证（★ 新增 — 解决缺口 16）

> **★ 为什么需要**
>
> 经济 bootstrap 填充了初始订单簿。但如果所有供需恰好平衡 → 没有交易发生 → 价格永远停在初始估算 → 经济静止。
>
> 验证三个自然不平衡源是否足够驱动首帧交易:

**来源 1 — 空间不平衡**: 矿业镇（铁过剩 Ask + 粮食赤字 Bid）+ 农业村（粮食过剩 Ask + 铁赤字 Bid）→ P11 创建的 TradeNetwork Market 中 Bid ≥ Ask → 自动撮合。

**来源 2 — 财富-消费不平衡**: Pareto 钱包分布 → 底层 30% NPC 钱包不足以覆盖`self_need` → 必须出售劳力/物品换取粮食 → Bid。顶层 10% 资金过剩 → 购买奢侈品/囤积资源 → Ask。

**来源 3 — 职业代谢不平衡**: ProfessionTag consumes/produces 自动产生交叉需求。铁匠不产铁（必须从矿工买），矿工不产食物（必须从农民买）。只要有 ≥ 2 个不同职业的 NPC，消费-产出交叉必然产生交易需求。

**Bootstrap 后自检**:
```rust
fn verify_bootstrap_liquidity(state: &EconomyState) -> Result<(), BootstrapError> {
    for market in &state.markets {
        // 每个 LocalBazaar 至少有一对可撮合的 Bid-Ask
        let executable = market.bids.iter().any(|(item, bids)| {
            market.asks.get(item).map_or(false, |asks| {
                bids.first().zip(asks.first())
                    .map_or(false, |(b, a)| b.limit_price >= a.limit_price)
            })
        });
        if !executable {
            return Err(BootstrapError::DeadMarket(market.market_id));
        }
    }
    Ok(())
}
```

如果检验失败 → 日志警告 + 自动注入最低流动性（为每个空 Market 创建基础的 food bid 和 food ask，价格=base_value_copper × noise）。此降级措施仅在 bootstrap 时可用——运行时永不注入。
