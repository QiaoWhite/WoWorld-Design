> **目标模块**: [[../../Happy Game/开发阶段/历史/001-历史系统总纲|历史系统]]
> **紧迫度**: 🔴 高 — P9 历史事件生成 + P4.5 KnowledgeSeed 传播必须消费
> **日期**: 2026-06-20

## 〇、总因：为什么历史系统需要补充

> **核心矛盾**: 当前历史系统有 `EventLog` 数据结构和事件因果链的理论框架，但它的"预生成历史"逻辑仍在世界生成 008 的 §二（P9 旧版）中——那个版本只有 6 类事件的简单表，没有因果图，没有 KnowledgeSeed 传播，没有事件参与人分配。世界生成 008 将被完全重写——历史事件的生成逻辑需要迁移到历史系统自身。
>
> **缺失的能力**:
> 1. **KnowledgeSeed 正式格式** — 005-知识种子 中有草图，但不是可解析的 TOML 规范
> 2. **技术传播算法** — 传播概率公式在 NPC 行动模块 005 中有描述，但需要历史系统拥有并执行
> 3. **因果事件模板** — 战争/灾害/繁荣/文化/政治/探索 6 类事件如何从种子+地理+文化中自动采样
> 4. **SettlementTechState** — 传播的最终产物（每聚落每技艺的水平）没有类型定义

---

## 一、KnowledgeSeed TOML 正式格式

> **★ 为什么需要正式化 KnowledgeSeed**
>
> 当前草图（NPC 行动模块 005）有时代/技艺的 TOML 片段，但不是完整的可解析规范。需要正式化的原因：
> 1. **TOML 是数据驱动的唯一途径**: 新增技术时代 = 新增 TOML 条目，不需要改 Rust 代码。
> 2. **Mod 支持**: Mod 可以添加新的技术时代（如"发条时代 ClockworkAge"），只需提供 TOML 文件。
> 3. **发现类型（Inevitable/Conditional/Breakthrough）决定传播行为**: 三类发现的语义必须由历史引擎解释——不是 TOML 中的字符串标签。
> 4. **Condition 语法需要标准化**: `{ resource = "CopperOre", biome = ["Mountain"] }` ——条件系统需要解析这种格式并匹配到世界生成的资源/群系数据。

```toml
[[era]] name = "IronAge", start_year = -500, end_year = 0
[[era.tech]]
domain = "IronSmelting"
discovery = "Breakthrough"
condition = { prerequisite = ["CopperSmelting"], tech_level = 0.6 }
base_skill_ceiling = 40
propagation = "VerySlow"
```

## 二、新增接口：generate_pre_history()

> **★ 为什么需要这个接口**
>
> 当前历史事件生成逻辑（世界生成 008 §二 P9）是嵌入在世界生成文档中的伪代码。世界生成重构后，这个逻辑属于历史系统——世界生成只传入参数，获取 `EventLog` + `SettlementTechState`。
>
> 接口需要消费 NPC 和 PowerTopology——这意味着 P9 在 P8+P10 之后。事件的参与者引用具体的 NpcId。

```rust
pub fn generate_pre_history(
    knowledge_seeds: &KnowledgeSeedRegistry,
    settlements: &[SettlementData],
    npcs: &[NpcData],
    families: &[FamilyTree],
    power_topology: &PowerTopology,
    culture_params: &HashMap<SettlementId, CultureCoreParams>,
    years_to_simulate: u32,
    rng: &mut DeterministicRng,
) -> (EventLog, SettlementTechState, WorldStateDelta);
```

## 三、新增类型：SettlementTechState

> **★ 为什么需要**
>
> KnowledgeSeed 传播算法的最终产物——每个聚落每项技艺在 year 0 时的最高水平——必须有一个类型承载。P8 人口生成需要消费 `SettlementTechState` 作为该聚落 NPC 的技能天花板。如果这个类型不存在，P8 不知道"这个聚落的铁匠技能最高能到多少"——只能硬编码一个全局天花板。
>
> 没有 SettlementTechState → P8 无法从数据推导技能水平 → 必须回退到硬编码。

```rust
pub struct SettlementTechState {
    pub tech_levels: HashMap<String, u8>,          // domain → 0-100
    pub tech_arrival_years: HashMap<String, i32>,  // domain → 传入年份
    pub tech_origins: HashMap<String, SettlementId>,
    pub lineage_chains: Vec<PropagationChain>,
}
```

## 四、世界状态 Delta（WorldStateDelta）

> **★ 为什么需要**
>
> P9 生成的历史事件可能标记建筑为废墟、聚落为废弃、地点为大规模死亡地。但这些修改影响的是世界生成产出的数据（BuildingData、SettlementData）——历史系统不能直接修改它们（违反解耦）。
>
> `WorldStateDelta` 是历史系统→世界生成的"副作用报告"。世界生成 P13 读取 Delta 并应用到相应数据——解耦且可追踪。

```rust
pub struct WorldStateDelta {
    pub buildings_damaged: Vec<(BuildingId, BuildingCondition)>,
    pub settlements_abandoned: Vec<SettlementId>,
    pub mass_death_sites: Vec<MassDeathSite>,
    pub ruin_sites: Vec<RuinSite>,
}
```
