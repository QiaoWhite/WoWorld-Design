## 〇、总因：为什么魔法系统需要补充

> **核心矛盾**: 魔法系统定义了 10 元素体系、魔力池、施法机制——但 NPC 的"魔法天赋"从何而来？如果世界生成不分配初始魔法亲和力 → 所有 NPC 的 magic_affinity=0 → 没有人能施法 → 魔法系统在运行时永远无法启动。魔法天赋需要初始分布，如同铁匠技能需要初始分布。
>
> 魔法天赋与普通技能不同——它不是通过学徒制"学习"的（你必须有先天亲和力才能感知魔力），也不是通过 KnowledgeSeed 传播的（魔法是被"发现"的，不是被"发明"的）。魔法亲和力从以下因素推导：
> 1. **地理位置** — 靠近元素裂隙/魔法森林 → 高亲和力
> 2. **遗传** — 父母的魔法天赋传给孩子（部分遗传 + 噪声）
> 3. **个体随机** — 对数正态分布（大多数为 0，极少数极高——天才涌现）
>
> **解耦要求**: 魔法系统定义 `MagicAffinity` 类型和 `select_primary_element()` 函数。世界生成 P2 产出 `MagicParamField`，P8 调用魔法系统函数分配每个 NPC 的初始魔法天赋。

---

## 一、MagicAffinity 类型定义

> **★ 为什么需要**
>
> 当前魔法系统文档没有定义"一个 NPC 的魔法天赋状态"的类型。没有此类型 → 世界生成无法为 NPC 赋值魔法亲和力 → NPC 行动系统的 CAST/CHANT 原子无法被任何人使用 → 魔法永远不发生。

```rust
pub struct MagicAffinity {
    pub strength: f32,                  // 0-1 天赋强度
    pub primary_element: Element,       // 主元素
    pub secondary_element: Option<Element>,
    pub elemental_profile: [f32; 10],   // 10元素归一化向量
    pub mana_pool: u32,                // 当前魔力
    pub mana_max: u32,                 // 最大魔力
    pub known_spells: Vec<SpellId>,     // 运行时填充
    pub spell_proficiencies: HashMap<SpellId, f32>, // 运行时填充
}
```

## 二、select_primary_element()

> **★ 为什么需要**
>
> 10 元素体系的元素倾向不是均匀随机——它从地理位置（火山=火，深海=水，高山=风）+ 父母遗传 + 个体随机中加权选择。此选择逻辑属于魔法系统——它定义了元素亲和力的语义。如果世界生成硬编码选择逻辑，10 元素的交互规则变化时世界生成会断裂。

```rust
pub fn select_primary_element(
    location_affinity: &ElementAffinityMap,
    parents_elements: Option<Element>,
    rng: &mut DeterministicRng,
) -> Element;
```

## 三、initial_mana_pool()

> **★ 为什么需要**
>
> 魔力池不是常数——它是亲和力强度 × 年龄增长 × 技术修正的函数。年轻 NPC 的魔力池小（未完全开发），成年 NPC 满，老年 NPC 可能略有衰减。技术修正来自 KnowledgeSeed——如果该聚落尚未发现魔法（BasicElementalMagic tech_level=0），即使 NPC 有天赋，魔力池也为 0。

```rust
pub fn initial_mana_pool(
    affinity_strength: f32,
    age: u8,
    settlement_tech: f32,  // 0-1 聚落魔法技术水平
    rng: &mut DeterministicRng,
) -> u32;
```

## 四、MagicParamField — 世界生成 P2 新增产出

> **★ 为什么需要**
>
> `MagicParamField` 是群系参数场的补充——定义世界每个位置的元素亲和力和魔法浓度。它属于世界生成 P2 的产出，但它定义了魔法系统消费的数据合同。需要魔法系统声明：`MagicParamField` 应包含什么字段、每个字段的语义和取值范围。
>
> 当前世界生成的自然景观文档定义了幻想地形（8.1-8.7 类），其中"能量浸染型"（永燃之地、永冻区、元素倾泻区）直接影响魔法参数场。但两者之间没有正式的数据映射。

```rust
pub struct MagicParamField {
    pub elemental_affinity: [f32; 10],   // 每位置的 10 元素亲和
    pub magic_concentration: f32,        // 0-1 魔法浓度
    pub mana_regen_modifier: f32,        // 魔力恢复速率修正
}
```

## 五、Life 模块魔法属性初始化（★ 新增 — 解决缺口 5）

> **★ 为什么需要**
>
> 魔法系统（001-魔法总纲）定义每个生命实体有 5 个魔法相关属性存储在 Life 模块的 `Vitals` 中。我们的 `MagicAffinity`（在 NPC crate）是高层概念——这 5 个 Life 属性是其物理底层。如果 Life 属性不初始化，即使 NPC 有 `MagicAffinity`，运行时 CAST/CHANT 原子会因 `spirit_capacity=0` 而无法工作。

**5 个 Life 魔法属性 → MagicAffinity 映射**:

| Life 属性 | 从 MagicAffinity 推导 | 公式 |
|-----------|---------------------|------|
| `spirit_capacity` | strength + 年龄 | `strength² × age_modifier × 1000` |
| `magic_power` | strength + primary_element | `strength × element_power_multiplier[element]` |
| `magic_control` | strength × dexterity | `strength × (1 - execution_noise)` |
| `magic_resistance` | elemental_profile | `Σ(elemental_profile[i] × ELEMENT_RESIST[i])` |
| `mana_regen_rate` | strength × settlement_tech | `strength × magic_concentration × base_regen` |

```rust
/// 定义在 woworld_life crate 中（Life 模块拥有这些属性）
impl Vitals {
    pub fn init_magic_attributes(
        affinity: &MagicAffinity,
        age: u8,
        dexterity: f32,
        settlement_magic_concentration: f32,   // 来自 MagicParamField
        settlement_tech: f32,                  // 来自 SettlementTechState
    ) {
        self.spirit_capacity = (affinity.strength.powi(2) * age_modifier(age) * 1000.0) as u32;
        self.magic_power = affinity.strength * element_power_multiplier(affinity.primary_element);
        self.magic_control = affinity.strength * dexterity;
        self.magic_resistance = dot_product(&affinity.elemental_profile, &ELEMENT_RESIST);
        self.mana_regen_rate = affinity.strength * settlement_magic_concentration * BASE_MANA_REGEN;
    }
}
```

**所有权**: `spirit_capacity`/`magic_power`/`magic_control`/`magic_resistance`/`mana_regen_rate` 属于 Life 模块的 `Vitals`。`MagicAffinity` 属于 NPC 模块。世界生成 P8 创建 NPC 时先构造 `MagicAffinity`，再调用 `Vitals::init_magic_attributes()` 初始化 Life 侧属性。
