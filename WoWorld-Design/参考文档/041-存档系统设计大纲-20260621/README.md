## 一、模块定位

存档系统是 WoWorld **世界状态的持久化层**。它不参与游戏模拟，不定义任何业务逻辑——只负责将内存中的世界状态保存到磁盘，并在需要时恢复。

**一句话**：存档系统是"世界的快照相机"——按下快门存世界，翻开相册读世界。

---

## 二、核心职责

| 职责 | 说明 |
|------|------|
| **保存** | 增量/全量将模块数据序列化并写入 LMDB 文件 |
| **加载** | 从 LMDB 文件反序列化世界状态到内存 |
| **版本迁移** | 旧格式存档自动迁移到当前格式 |
| **崩溃恢复** | 临时文件 + 原子重命名保证写入中途崩溃不损坏旧存档 |
| **会话检测** | 检测上次游戏是否正常退出，提示玩家 |
| **Mod 兼容** | 检测 Mod 增删，提供降级/冻结选项 |
| **存档管理** | 多槽位（手动无上限 + 自动轮转 + 快速 + 退出）的枚举、删除 |

---

## 三、模块边界

### 存档系统拥有

```
├── SaveableModule trait（模块与存档系统的唯一契约）
├── SaveSystem API（LMDB 环境管理 + 保存/加载流程编排）
├── 存档文件格式（Header + named_db 键空间约定）
├── 世界注册与发现（目录扫描 + world_meta.json）
├── 崩溃恢复协议（临时文件 + session.lock）
└── 版本迁移调度（调用模块的 migrate() 方法）
```

### 存档系统不拥有

```
├── ❌ 任何模拟数据——模块数据归各自模块所有
├── ❌ 序列化格式选择——各模块自选（bincode/rkyv/...）
├── ❌ 数据老化策略——各模块自治
├── ❌ 跨模块引用解析——运行时懒解析（ID 引用）
├── ❌ 世界生成——P0-P13 输出内存 WorldState，存档系统负责首次持久化
├── ❌ 玩家设置——settings.json 归游戏主循环管理
└── ❌ UI——存档系统提供 API，Godot 层构建 UI
```

## 四、核心设计决策速览

| 决策 | 选择 | 理由 |
|------|------|------|
| 存储策略 | **全量快照 + 脏数据增量** | 读档时间可控，版本迁移可行，不依赖确定性重放 |
| 文件架构 | **单文件 LMDB + 多 named_db** | 跨模块原子写入，文件可复制分享 |
| 种子定位 | **容错恢复手段**（非空间优化） | 初次生成存全量，种子在损坏时可重建种子可重现层 |
| 存档文件 | **一个存档 = 一个 .woworld 文件** | 自包含，可复制传输 |
| 序列化契约 | **存档系统不规定格式** | 模块自选序列化库，存档系统只传递 bytes |
| 写入安全 | **临时文件 + 原子重命名** | 操作系统保证原子性 |
| 版本管理 | **模块级版本号** | 改一个模块不影响其他模块的迁移 |
| 加载方式 | **渐进式 + 按需加载** | 大模块（物品 1.7GB）不全量反序列化 |
| 存档触发 | **tick 边界** | 避免半截状态快照 |
| 并发保护 | **MVCC 读句柄常驻 + 写事务短命** | 无切换开销 |

## 五、与其他模块的关系

```
存档系统 ←→ 14 个模块（通过 SaveableModule trait）
   ↑
   └── 每个模块实现 trait，自管序列化/反序列化/迁移
   
存档系统 ←→ 世界生成
   └── 世界生成产出内存 WorldState → 存档系统首次持久化
   
存档系统 ←→ 文件系统
   └── 目录扫描发现世界、管理 .woworld 文件、session.lock
```
