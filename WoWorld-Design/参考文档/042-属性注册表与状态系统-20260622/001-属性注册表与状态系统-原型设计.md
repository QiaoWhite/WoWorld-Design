# 001 — 属性注册表与状态系统 — 原型设计

> **开发代号**: WoWorld (Wonder World)
> **引擎**: Godot 4.7
> **开发者**: 独立游戏开发者（Solo）
> **原型状态**: 开发侧草稿 — 允许频繁变动，未来可能大幅重构

---

## §0 执行摘要

### 问题

WoWorld 当前约 **200+ 实体属性** 分散在 ~25 个模块中——NPC 的生命值在 [[../Happy Game/开发阶段/生命/README|Life]]、人格特质在 [[../Happy Game/开发阶段/NPC活人感模块/README|NPC 活人感]]、物品属性在 [[../Happy Game/开发阶段/物品系统/README|物品系统]]、技能熟练度在 [[../Happy Game/开发阶段/技能系统/README|技能系统]]。不存在：

- **统一的属性目录** — 开发者不知道"一个实体总共有哪些属性可以查"
- **统一的检查接口** — 开发者没有控制台来 `inspect` 一个实体
- **玩家角色面板** — 玩家看不到自己角色的完整状态
- **统一的状态效果框架** — buff/debuff/标签/部位伤害/元素效果散落在 Life::Tag（200+ HashSet）、Combat 部位伤害、Magic 元素效果三处

### 方案

本原型设计 **属性注册表与状态系统** — 一个全局基础层模块，提供两样东西：

1. **属性注册表**（Property Registry） — 所有实体属性的 TOML 目录 + 统一查询接口。属性存储保持分散（各模块自主），但目录和查询通道统一。
2. **状态系统**（State System） — StatusQuery trait 统一跨模块状态查询，四种状态类别（Tag / Condition / Effect / Modifier）各模块独立实现，StatusAggregator 合并输出。

### 设计约束

- **开发侧优先** — 当前功能为开发者调试服务，玩家面板是后续迭代
- **容忍变动** — TOML 目录可热重载，属性增删不强制重新编译
- **预留重构** — 核心接口（PropertyProvider / StatusQuery）刻意保持最小表面积，方便未来大改
- **不删已有设计** — 所有现有模块的属性存储保持不变，仅新增统一的"查询视角"

## §1 场景走查：开发者控制台检查 NPC

> **这是驱动整个设计的核心场景。** 本节走查开发者从打开控制台到完成一次完整实体检查的全部步骤。

### 1.1 打开控制台

开发者按 `~` 键。控制台覆盖层从底部滑入——半透明深色背景，白色等宽字体，下方式命令行输入行，上方是滚动输出区。游戏默认暂停（可在设置中改为不暂停）。

光标自动定位到输入行。提示符为 `>`。

### 1.2 选择目标

开发者有多种方式指定目标实体：

| 方式 | 语法 | 示例 | 说明 |
|------|------|------|------|
| **鼠标点击** | 点击画面中实体 | — | 最常用。控制台回显 `[已选中] NPC_001 (铁匠 老张)` |
| **ID 直指** | `target <EntityId>` | `target NPC_001` | 脚本/自动化场景 |
| **名称搜索** | `target name <关键词>` | `target name 老张` | 模糊匹配，多个结果时列出候选项 |
| **范围选择** | `target @circle <半径>` | `target @circle 50m` | 选中范围内的所有实体 |
| **类型过滤** | `target type=<EntityKind>` | `target type=NPC` | 选中当前视野内所有 NPC |

本次开发者用鼠标点击了一个 NPC 铁匠。

### 1.3 检查实体（默认深度）

```

```

`inspect` 不带参数，默认作用于当前选中目标。默认输出深度为 `summary`：

## §2 属性分类树

属性按 **13 个分类** 组织，作为查询语言点号路径的第一级。这些分类是 **查询视图的标签，不是 Rust 模块**——一个模块可以为多个分类贡献属性。

| 分类键 | 显示名称 | 属性数（估） | 主要内容 | 属主模块 |
|--------|----------|-------------|----------|----------|
| `identity` | 身份标识 | ~6 | EntityId, 名称, 物种, EntityKind, 年龄, 性别 | Life |
| `vitals` | 生命体征 | ~18 | HP, 魔力, 体力, 饥饿, 口渴, 疲劳, 体温, 精神, libido, element_surplus[8] | Life |
| `physical` | 身体与外观 | ~20 | BodyPlan, 外观参数, RaceTraits, DamagePool(按部位), 身高体重 | Life |
| `mental` | 心智与人格 | ~25 | BigFive(5维), EmotionalState, 当前心情, 压力, MentalAccess, 认知风格, 认知潮汐 | NPC 活人感 |
| `social` | 社会关系 | ~15 | 人际关系列表, 声望聚合值, 派系归属, 社会地位, 荣誉权重 | NPC 活人感 / 权力系统 |
| `skills` | 技能 | 稀疏(每技能~10字段) | SkillEntry(xp/level/aptitude/total_xp/times_used), 交叉训练加成 | 技能系统 |
| `equipment` | 装备与物品 | ~12 | 装备槽位映射, Outfit(双套切换), 背包摘要, 容器树 | 物品系统 |
| `status` | 状态效果 | 动态, 无上限 | Tags(二进制/可叠加), Conditions(阶段递进), Effects(定时持续), Modifiers(数值乘数) | Life / Combat / Magic / 天气 |
| `knowledge` | 知识与记忆 | ~6(聚合) | 记忆总数, known_recipes, known_locations, known_entities, known_spells | NPC 活人感 / 魔法 |
| `combat` | 战斗 | ~10 | 战斗状态, 当前目标, 姿态, 部位伤害, 连击计数 | 战斗 |
| `magic` | 魔法 | ~15 | 法力池, 已知法术, 元素亲和度[10], SpiritState, 印记 | 魔法 / Life |
| `aesthetic` | 审美 | ~8 | AestheticTaste(32B), 近期审美判断, 外观评价 | NPC 活人感 / 文化系统 |
| `economy` | 经济 | ~8 | 钱包, 拥有财产, 交易历史计数, 信用评级 | 经济系统 |

### §2.1 分类的扩展性

分类列表通过 TOML 目录定义，**增删分类不需要改 Rust 代码**。新增一个分类只需在 TOML 目录中添加 `[category.<新键>]` 条目。查询语言自动适配。

## §3 查询语言参考

### 3.1 设计原则

- **点号路径** — 直觉性强，像文件路径。`NPC_001.vitals.hp`
- **无 JOIN，无子查询，无聚合**（v1） — 保持查询语言最小化
- **默认只读安全** — `set` 受 safety_level 门控
- **中文友好** — 命令名英文（简洁），属性名中文（可读），但 `get`/`set` 都接受中英文路径

### 3.2 命令参考

| 命令 | 语法 | 示例 | 安全约束 |
|------|------|------|----------|
| `inspect` | `inspect [目标] [--depth=summary\|detailed\|full]` | `inspect NPC_001 --depth=full` | 只读，受 visibility_level 过滤 |
| `get` | `get <目标>.<路径>` | `get NPC_001.vitals.hp` | 只读，受 visibility_level 过滤 |
| `set` | `set <目标>.<路径> <值>` | `set NPC_001.status.tags add "blessed"` | 受 safety_level 门控 |
| `watch` | `watch <目标>.<路径>` | `watch NPC_001.vitals.hp` | 只读，持续输出 |
| `list` | `list <目标>.<分类>` | `list NPC_001.status.effects` | 只读 |
| `filter` | `filter [type=<EntityKind>] where <条件>` | `filter type=NPC where vitals.hp < 50` | 只读，最多 100 条结果 |
| `target` | `target <方式> <参数>` | `target name 老张` | 无 |
| `help` | `help [命令]` | `help set` | 无 |

### 3.3 路径语法

```
路径         ::= 根 ("." 段)*
段           ::= 标识符 ("[" 索引 "]")?
索引         ::= 整数 | "*" | 十六进制 | 范围
范围         ::= 整数 ".." 整数

示例:
  NPC_001.vitals.hp                          — 整数字段
  NPC_001.vitals.element_surplus[3]           — 数组索引
  NPC_001.skills[0x03_01_0001].xp            — 十六进制键
  NPC_001.skills[*].level                     — 通配：返回所有技能的等级
  NPC_001.equipment.slots[0..5]               — 范围：前5个槽位
  NPC_001.physical.damage_pool["left_arm"]    — 字符串键
```

### 3.4 类型显示格式

| Rust 类型 | 控制台输出格式 | 示例 |
|-----------|---------------|------|
| `u32` / `i32` | 整数 | `87` |
| `f32` | 2 位小数 | `36.70` |
| `String` | 双引号 | `"老张"` |
| `bool` | `是` / `否` | `是` |
| `Enum` | 变体名 | `Human` |
| `Vec<T>` | `[item1, item2, ...]` | `[火, 土, 金]` |
| `HashMap<K,V>` | `{key: val, ...}` | `{锻造: 34, 武器锻造: 28}` |
| `Option<T>` | 值 或 `无` | `无` |

### 3.5 filter 条件表达式

```
条件   ::= 属性路径 操作符 值
操作符 ::= == | != | < | > | <= | >= | contains | has_tag | has_effect
组合   ::= 条件 ("and" | "or" 条件)* | "not" 条件 | "(" 组合 ")"

示例:
  vitals.hp < 50
  status.tags has_tag "bleeding"
  status.tags has_tag "bleeding" and vitals.hp < 30
  type=NPC and (vitals.hp < 30 or combat.stance == "unconscious")
```

### 3.6 安全门控

| safety_level | 标签含义 | 允许操作 |
|--------------|----------|----------|
| 0 | 核心生存属性 | 只读，`set` 总是拒绝 |
| 1 | 重要但不致命 | 只读，`set` 需要 `--force` 标志确认 |
| 2 | 状态/标签/外观 | 可读写，`set` 正常执行 |
| 3 | 调试/隐藏 | 完全可读写，仅开发者构建可见 |

属性在 TOML 目录中声明自己的 safety_level。控制台在执行 `set` 前检查目标属性的 safety_level。

## §4 TOML 属性目录 Schema

### 4.1 设计理由

- **TOML** 已是项目的标准数据格式（制造配方、技能定义、Mod 覆盖都用 TOML）
- **人可读** — 开发者可直接编辑
- **热重载** — 增删属性不需要重新编译 Rust
- **Mod 友好** — Mod 可叠加自己的属性目录文件

### 4.2 属性条目定义

```toml
# 示例：来自 Life 模块的属性
[property.vitals.hp]
type = "u32"
category = "vitals"
display_name = "生命值"
display_name_en = "HP"
visibility_level = 0       # 总是可见
safety_level = 0           # 核心生存属性 → 控制台禁止修改
source = "struct"          # 存储在强类型 struct 中
path = "vitals.hp"
description = "当前生命值。降至 0 触发死亡流程。"
min = 0
max = "derived_from: physical.max_hp"
default = 100
unit = ""
tags = ["core", "save", "combat_relevant"]

# 示例：动态属性（来自 PropertyBag）
[property.status.tags]
type = "HashSet<Tag>"
category = "status"
display_name = "标签"
visibility_level = 1       # 条件可见（需要感知技能/近距离/长时间观察）
safety_level = 2           # 控制台可修改
source = "bag"             # 存储在 PropertyBag 中
path = "status.tags"
description = "实体当前持有的所有标签。可叠加，不互斥。"

# 示例：来自 NPC 模块的心理属性
[property.mental.big_five.o]
type = "f32"
category = "mental"
display_name = "开放性 (Openness)"
visibility_level = 1       # 需要深入社交互动才能感知
safety_level = 1           # --force 才能修改
source = "struct"
path = "mental.big_five.o"
description = "BigFive 人格模型——开放性维度。影响 NPC 对新事物的接受度、好奇心和想象力。"
min = 0.0
max = 1.0
default = "random"         # 世界生成时随机
```

### 4.3 derived_from 引用

`min`/`max` 字段可引用其他属性路径：

```toml
max = "derived_from: physical.max_hp"     # 引用同实体属性
max = "derived_from: race_traits.base_hp" # 引用种族基础值
```

控制台在展示时解析这些引用。属性的 `description` 也会标注"上限取决于 <路径>"。

### 4.4 扩展性

- 各模块在 `property_catalog/<模块名>.toml` 中定义自己的属性
- Mod 在 `mods/<mod_id>/property_catalog/*.toml` 中添加新属性
- 新属性默认 `source = "bag"`，自动进入 PropertyBag
- 重复路径 → 加载时报错

## §5 StatusQuery Trait 设计

### 5.1 问题与方案

状态效果是跨模块聚合的最高难度问题——一个 NPC 可能同时拥有来自 Life 的"流血"标签、来自 Combat 的"左臂重伤"阶段状态、来自 Magic 的"火焰附魔"定时效果、来自 Weather 的"失温"数值修正。

**方案**：不统一存储，统一查询接口。每个模块保留自己的状态存储，但实现同一个 `StatusQuery` trait。`StatusAggregator` 遍历所有实现者，合并输出。

### 5.2 Trait 签名

```rust
/// 定义在 woworld_core。由任何产生状态数据的模块实现。
pub trait StatusQuery: Send + Sync {
    /// 二进制或可叠加标签（如 "on_fire", "blessed", "poisoned"）
    fn query_tags(&self, entity: EntityId) -> Vec<TagInfo>;

    /// 有阶段的递进状态（如 "出血: 2/5 阶段"）
    fn query_conditions(&self, entity: EntityId) -> Vec<ConditionInfo>;

    /// 有持续时长和强度曲线的定时效果（如 "加速: 30s, +15%"）
    fn query_effects(&self, entity: EntityId) -> Vec<EffectInstance>;

    /// 作用于派生属性的数值修正（如 "伤害加成: 1.15x，来源: 狂暴"）
    fn query_modifiers(&self, entity: EntityId) -> Vec<ModifierInfo>;
}
```

### 5.3 返回结构

```rust
pub struct TagInfo {
    pub tag_id: u32,
    pub display_name: &'static str,     // "流血"
    pub source_module: &'static str,    // "Life"
    pub source_id: Option<u64>,         // 施加来源的 SkillId / ItemId / SpellId
    pub stack_count: u32,               // 可叠加标签的层数
    pub applied_at: u64,                // 施加时间戳
}

pub struct ConditionInfo {
    pub condition_id: u32,
    pub display_name: &'static str,     // "左臂重伤"
    pub body_part: Option<&'static str>, // "left_arm"
    pub current_stage: u8,              // 2
    pub max_stage: u8,                  // 5
    pub recovery_progress: f32,         // 0.0-1.0 恢复进度
    pub source_module: &'static str,
}

pub struct EffectInstance {
    pub effect_id: u32,
    pub display_name: &'static str,     // "火焰附魔"
    pub intensity: f32,                 // 当前强度
    pub intensity_curve: EffectCurve,   // 强度随时间变化的曲线类型
    pub remaining_ticks: u32,           // 剩余 tick 数
    pub total_ticks: u32,               // 总 tick 数
    pub tick_interval_ms: u32,          // 每 tick 间隔
    pub source_module: &'static str,
    pub source_id: Option<u64>,
}

pub struct ModifierInfo {
    pub modifier_id: u32,
    pub display_name: &'static str,     // "狂暴伤害加成"
    pub target_property: &'static str,  // "combat.damage_mult"
    pub operation: ModifierOp,          // Add / Mul / Override
    pub value: f32,                     // 1.15
    pub source_module: &'static str,
    pub source_id: Option<u64>,
    pub priority: u8,                   // 冲突时高优先级胜出
}
```

### 5.4 实现者矩阵

| 模块 | query_tags | query_conditions | query_effects | query_modifiers |
|------|-----------|-----------------|---------------|-----------------|
| **Life** | ✅ 生理标签(流血/饥饿/中毒等) | ✅ 部位伤害阶段 | — | — |
| **Combat** | ✅ 战斗姿态标签 | ✅ 伤势阶段 | — | ✅ 战斗加成 |
| **Magic** | ✅ 元素亲和标签 | — | ✅ 法术 buff/debuff | ✅ 魔法倍率 |
| **NPC 活人感** | ✅ 社交标签(已婚/就业等) | ✅ 情绪状态 | — | — |
| **物品系统** | ✅ 装备加成标签 | — | — | ✅ 物品属性修正 |
| **技能系统** | — | — | — | ✅ 技能加成 |
| **天气系统** | ✅ 环境暴露标签 | — | ✅ 天气持续效果 | ✅ 温度修正 |

### 5.5 StatusAggregator

`PropertyRegistry` 内置一个 `StatusAggregator`：

```rust
pub struct StatusAggregator {
    providers: Vec<Box<dyn StatusQuery>>,
}

impl StatusAggregator {
    pub fn collect(&self, entity: EntityId) -> UnifiedStatusView {
        UnifiedStatusView {
            tags: self.providers.iter().flat_map(|p| p.query_tags(entity)).collect(),
            conditions: self.providers.iter().flat_map(|p| p.query_conditions(entity)).collect(),
            effects: self.providers.iter().flat_map(|p| p.query_effects(entity)).collect(),
            modifiers: self.providers.iter().flat_map(|p| p.query_modifiers(entity)).collect(),
        }
    }
}
```

控制台调用 `StatusAggregator::collect()` → 一次遍历→得到统一的状态视图。每个 provider 的实现是轻量的——只返回对已有数据的引用或拷贝，不做计算。

## §6 集成地图

### 6.1 受影响模块

| 模块 | 变更 | 优先级 | 风险 |
|------|------|--------|------|
| **Life** | 实现 StatusQuery (tags + conditions) | 🔴 关键 | 低 |
| **Combat** | 实现 StatusQuery (tags + conditions + modifiers) | 🔴 关键 | 低 |
| **Magic** | 实现 StatusQuery (tags + effects + modifiers) | 🟡 高 | 低 |
| **NPC 活人感** | 实现 StatusQuery (social tags + mood conditions)；GOAP 可从 PropertyRegistry 查询世界状态 | 🔴 关键 | 中 |
| **物品系统** | 实现 StatusQuery (equipment modifiers) | 🟡 高 | 低 |
| **技能系统** | 实现 StatusQuery (skill modifiers) | 🟡 高 | 低 |
| **天气系统** | 实现 StatusQuery (exposure tags + effects + modifiers) | 🟢 中 | 低 |
| **存档系统** | 属性目录的 `tags` 字段定义 saveable_fields | 🟡 高 | 中 |
| **模块接头总览** | 新接头条目（27 号槽位） | 🟡 高 | 低 |
| **CLAUDE-INTERFACES.md** | 新 CHG 条目：StatusQuery trait + PropertyRegistry 所有权 | 🟡 高 | 低 |

### 6.2 模块边界

**属性注册表与状态系统 拥有：**
- PropertyProvider trait 定义（在 woworld_core 中）
- StatusQuery trait 定义（在 woworld_core 中）
- 属性目录 TOML 的 schema 规范
- StatusAggregator 合并逻辑
- 控制台查询语言解释器
- 可见性规则引擎（VisibilityResolver trait + 条件评估）

**属性注册表与状态系统 不拥有：**
- 任何实体数据 — 属性值始终由属主模块存储
- 任何状态效果的具体逻辑 — Life/Combat/Magic 各自管理自己的状态
- 任何序列化格式 — 存档系统不规定格式的约定不变
- 玩家角色面板 UI — 这是表现层消费者，不在本模块范围内

## §7 原型到正式模块的毕业路径

### 7.1 本原型做了什么

- 定义了具体的使用场景（控制台检查 NPC）
- 建立了 13 类属性分类树
- 设计了查询语言语法和命令参考
- 定义了 TOML 属性目录 schema
- 设计了 StatusQuery trait 签名和返回结构
- 绘制了跨模块集成地图

### 7.2 本原型没做什么

- 没有为所有 13 个分类定义完整的 PropertyProvider trait（只给了模式）
- 没有分配全部 ~200+ 属性到具体模块（只给了分类框架和示例）
- 没有写正式模块的完整规格文档（那是步骤 3 的内容）
- 没有定义性能预算（开发侧早期，数据量不确定）
- 没有设计玩家角色面板 UI（后续迭代）

### 7.3 毕业条件与 8 步映射

| 步骤 | 内容 | 状态 |
|------|------|------|
| 1. 参考文档草稿 | 本原型 | ✅ 当前 |
| 2. 跨模块审计 | 逐模块验证无冲突 | ⬜ 待进行 |
| 3. 正式开发规格 | 创建 `开发阶段/属性注册表与状态系统/` — 含总纲/TOML规范/查询语言参考/StatusQuery接口/集成指南 | ⬜ |
| 4. 修改关联文档 | Life/Combat/Magic/NPC 等模块的 README 补充 StatusQuery 实现说明 | ⬜ |
| 5. 审查 | 内部一致性 + 跨文档对齐 + 与 CLAUDE-INTERFACES.md 无冲突 | ⬜ |
| 6. CHG 文档 | `CHG-060-属性注册表与状态系统-v1.0-YYYYMMDD.md` | ⬜ |
