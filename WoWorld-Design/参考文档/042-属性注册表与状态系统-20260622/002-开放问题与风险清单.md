## §1 需要用户裁决的问题

### Q1: 控制台 UI 层 — Rust 还是 Godot？

控制台覆盖层有两个技术路径：
- **Rust 侧**（egui / custom）：直接访问 Rust 数据，无需跨 FFI 序列化。开发成本高（窗口管理/输入处理/文本渲染需独立实现）。
- **Godot 侧**（Control 节点 + GDExtension 桥接）：开发快，可利用 Godot 内置 UI 系统。但每次查询需跨 FFI 传数据。

**建议**：先 Godot 快速原型（验证交互流程），性能瓶颈出现后迁移到 Rust。控制台的查询频率极低（手动输入），FFI 开销在初期不是问题。

### Q2: PropertyProvider 粒度 — 批量还是按属性？

- **批量**：`snapshot(entity) -> PropertySnapshot`，一次返回所有属性。优点：控制台 `inspect` 一次调用。缺点：`get single_property` 也需构建完整快照→浪费。
- **按属性**：`get_property(entity, path) -> PropertyValue`，单次调用。优点：精准、省内存。缺点：`inspect --depth=full` 需 ~200 次调用→可能慢。

**建议**：双模式。`PropertyProvider` trait 同时提供 `get_one()` 和 `snapshot()` 两个方法。`get_one()` 默认实现是调用 `snapshot()` 再提取——各模块可按需针对性优化热路径。

### Q3: filter 是否支持空间查询？

`filter type=NPC where distance < 500m` 的价值很高（区域调试），但添加 `distance` 操作符意味着 PropertyRegistry 需要依赖 SpatialIndex。这建立了"全局基础层 → 世界生成"的反向依赖。

**建议**：v1 不支持空间查询。`filter` 只在已加载的实体范围内查询。空间查询由 `target @circle 50m`（依赖 SpatialIndex，但它是一个独立的定位命令，不耦合到查询语言的条件表达式中）满足。

### Q4: 玩家角色面板 — 共用 PropertyRegistry 还是独立视图？

- **共用**：玩家面板数据从 PropertyRegistry 读取。一致性好，一处改动全局生效。
- **独立**：玩家面板有独立的数据源和格式化逻辑。UX 更优（专门为玩家策划的信息层级），但两套系统需要维护同步。

**建议**：共用底层 PropertyRegistry，但玩家面板有自己的"策划视图层"——从同一 Registry 拉数据，但只展示策划指定的属性子集 + 自定义排版 + 教学提示。开发者控制台走"原生视图"。

### Q5: 控制台语言 — 中文/英文/双语？

命令名（`inspect`/`get`/`set`/…）用英文（编程风格，简洁）。属性路径和输出标签呢？

**建议**：路径接受中英文（`vitals.hp` 和 `生命值` 都解析到同一属性）。输出默认中文（`display_name`），`--lang=en` 切换英文。属性目录 TOML 中 `display_name` 和 `display_name_en` 同时维护。

## §2 识别的风险

### R1: TOML 目录与 Rust 代码漂移

当属性在 TOML 目录中声明了，但 Rust struct 中不存在（或类型不一致）时，运行时才会发现。**缓解**：未来有代码时，build.rs 或 proc macro 在编译时检查 TOML 与 struct 定义的一致性。当前设计阶段，这是文档维护纪律问题。

### R2: 跨模块状态合并冲突

当两个 StatusQuery 实现者返回冲突的 modifier（比如 Combat 说 +10% 伤害，Magic 说 伤害 = 固定 50）时，StatusAggregator 需要冲突解决策略。**当前设计**：按 `priority` 排序，高优先级胜出。但这可能不符合游戏设计意图（应该是叠加而非替代）。**需要明确叠加规则后再定案**。

### R3: PropertyBag vs Struct 边界模糊

哪些属性进强类型 struct (快速+安全)，哪些进 PropertyBag (灵活+慢)？当前边界是"核心生存/战斗依赖的→struct，扩展/Mod/展示的→bag"。但在开发迭代中属性可能跨边界迁移。**缓解**：迁移路径明确——TOML 中改 `source = "bag"` → `source = "struct"`，同时模块内部新增 struct 字段。迁移期间旧路径保留为兼容别名。

### R4: 控制台范围蔓延

控制台最初只有 `inspect`/`get`/`set`，但交互设计（查询语言、`filter`、`watch`、空间查询…）很容易扩展成完整的世界编辑器。**缓解**：v1 明确范围——仅实体属性检查和基本修改。世界编辑（地形/建筑/NPC生成）不属于本模块。

## §3 原型范围外（明确不代表遗漏）

| 项目 | 理由 |
|------|------|
| 性能预算 | 无代码，无法 benchmark。正式模块时补充 |
| 控制台自动补全 | 查询语言稳定后再做 |
| `watch` 命令的实时更新率 | 取决于 tick 架构，正式模块时定义 |
| JSON/SNBT 导出 | 后续迭代 |
| 远程控制台（网络） | WoWorld 当前是单机游戏 |
| 属性变更历史 / 时间线图表 | 超出 v1 范围——但这是很棒的调试功能 |
| 多人游戏权限系统 | 单机游戏，无此需求 |
| 玩家角色面板 UI 具体设计 | 本原型聚焦开发侧。玩家面板后续迭代 |
| 物品属性的完整 TOML 目录 | 本原型给了 schema 和示例。全量目录在正式模块步骤 3 时完成 |
| PropertyProvider trait 的 13 分类完整签名 | 本原型给了 StatusQuery (状态) 的完整签名。PropertyProvider (全部属性) 的全量签名在正式模块时完成 |
