## 二、为什么现在开始编码

1. **地基已稳**：`woworld_core` 的类型注册表、空间查询四 trait（TerrainQuery/EntityIndex/SpatialEventBus/VisibilityQuery）、SaveableModule trait（14方法）已完整设计，有具体 Rust 伪代码签名
2. **世界生成管线已定**：P0-P13 共 15 阶段，每阶段输入/输出类型明确，可直接翻译为 Rust 代码
3. **技术栈已锁定**：Godot 4.7 + Rust stable 1.80+ GDExtension，经 Bevy 切换可行性分析（三种路径评估）确认
4. **跨模块契约已建立**：CLAUDE-INTERFACES.md 935 行 + 模块接头总览 102 文件覆盖全部接口
5. **再拖下去也不会更"完美"**：设计永远在演化（CHG-057/058/059 今日刚落地），起步编码本身会反向暴露设计缺陷，比继续纸上谈兵更高效

## 三、核心设计原则（编码时严格遵守）

1. **玩家 = NPC**：同一套 Rust 代码驱动。玩家只是"夺舍"了一个 NPC，多了 I/O 层（输入/渲染/UI）
2. **涌现优先**：不轻易建立独立模块。法律从权力原子+制裁链涌现，名声从分散式记忆×信息传播涌现
3. **NPC 不能"查询数据库"**：只能通过模拟感官（视听嗅触+内感觉）获取不完整的、可能有误差的信息
4. **零年龄门控·零硬编码禁止**：NPC 行为能力从 AgentSnapshot 连续参数涌现，不设 if age >= 18 之类硬门槛
5. **Rust 模拟核心保持引擎无关**：所有游戏逻辑在 Rust 侧，Godot 只负责渲染/UI/音频/输入/玩家物理
6. **仅玩家保留 PhysicsServer3D**：NPC/物体全部 Rust 侧空间查询四 trait

## 四、关键决策汇总

| # | 决策 | 来源 | 影响 |
|---|------|------|------|
| 1 | Rust workspace monorepo 结构 | 本规划 | 所有 crate 在 `woworld/crates/` 下，共享 Cargo.lock |
| 2 | 先做 `woworld_core` → `woworld_spatial` → `world_gen` | 依赖链 | 一切从地形开始 |
| 3 | 最低可行原型：地形+玩家+昼夜 | 技术栈方案 Phase 1 | 4-6周，不包含 NPC/物品/战斗 |
| 4 | Phase 1 不做海洋物理 | 复杂度控制 | 海面用平面+shader，OceanProvider trait 预留 |
| 5 | 暂不接入 LMDB | 复杂度控制 | 第一个原型不需要存档 |
| 6 | UI/UX **文档**优先于 UI **代码** | 风险控制 | 326 处交叉引用假设了不存在的 UI 层 |
| 7 | 不追求设计"冻结" | 哲学 | 设计→编码→反馈→修正，循环推进 |
| 8 | `woworld_core` 中所有类型与模块接头总览保持一致 | 契约 | 核心类型注册表为权威蓝图 |

## 五、风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 编码发现设计重大缺陷 | 高 | 大 | 设计文档同步更新，小修即时，大修走 CHG |
| Rust 学习曲线 | 中 | 中 | 从简单类型开始，逐步引入 trait/泛型/并行 |
| Godot GDExtension 集成坑 | 高 | 中 | 先用纯 Rust 单元测试验证逻辑，再桥接 Godot |
| 设计文档与代码脱节 | 中 | 大 | 每完成一个开发 Phase 同步更新接头总览 |
| 四轨并行注意力分散 | 中 | 中 | 每轮对话聚焦 1-2 轨，按优先级交替 |
| 性能目标不达标（VRAM 6GB） | 中 | 大 | 原型阶段即建立 RenderDoc 验证流程 |
