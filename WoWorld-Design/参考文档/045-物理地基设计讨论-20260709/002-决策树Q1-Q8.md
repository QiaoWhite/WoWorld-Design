## Q1 — 扩展自建 vs 引入通用物理引擎

**选择**: 不引入通用刚体物理引擎（Rapier/Jolt/Avian），扩展现有 Rust 运动学 + 空间查询。

**否决**: 引入引擎作为新支柱。

**理由**:
- 用户列的 5 场景，4 个引擎反而做得更差（角色控制器、移动平台行走、爬墙重力重定向、坐骑跟随——刚体求解器最易抖/打架/穿模处）；业界（塞尔达/旷野之息/汪达与巨像）这些都是自建运动学。
- "飞多远""武器加速度加成"本质是读速度做积分，不需求解器。
- 引擎 = 一整根新支柱，与确定性存档、SoA/rayon、GDExtension 跨边界、LMDB 快照全面冲突。
- **全项目取向已经是"无求解器"**（建筑倒塌状态切换、战争方阵统计、动物飞行砍空气动力学、死亡骨架松弛、帆船风驱动删除）——引擎逆势。

**翻盘条件（未触发）**: 若愿景是 Teardown/Besiege 式"物理破坏作为核心玩法动词"。用户确认建造=放置式、地形=密度场编辑，不是。

## Q1.5 — 是否需要"大规模涌现式刚体动力学"

**选择**: 不需要。

**理由（三个硬冲突）**:
1. **持久化**——涌现在 WoWorld = 能沉淀进 LMDB 跨百万年存活的东西（记忆/名声/社会结构）。刚体碎块落点无法/不应持久化，读档即蒸发 = 装饰非涌现。
2. **确定性**——同 seed 可复现 + 跨 Win/Linux/macOS；刚体求解器跨帧率/平台不确定。
3. **LOD 预算**——25 万 km²、L1-L4、GTX 1660，无余量；且 LOD 意味着没人看的堆叠根本不模拟，谈不上涌现。
- **不教条的边界**: 滑坡→CHG-065 密度场编辑；建筑倒塌→Godot 纯视觉碎块 VFX（不模拟/不持久化/秒消）；能推的巨石→至多几个特例化运动学 hero 物体。

## Q2 — 是否要统一的「质量→冲量→单体 COM」力学地基

**选择**: 要，但只做"单体"版本（2 组件 `Mass`+复活 `Velocity` + 1 冲量事件 + 2 系统 + 快/慢路分离）。多体连锁明确排除。

**否决**: 每招手写"飞多远"常量。

**证据（不统一已在伤害）**: 战斗 009 写"落差 >10m 重伤/>30m 致命"，物理模块写">8m/s 重伤/>20m/s 致命"。Python 实算：落 10m≈14.0 m/s、落 30m≈24.3 m/s；反解 8m/s 只等于 3.26m 落差。**两套阈值差约 3 倍**——两人背对背编的物理常量永远对不齐。统一地基第一价值 = 消灭这类无声漂移。

**理由**:
- **涌现/记忆点**: `impulse/mass → Δv → COM 积分器`，击退距离/推搡胜负/小妖撞踉跄大怪/坠落/落水全涌现自 2 个数。逐招硬编是"故事容器"，反哲学。玩家感知到的记忆点 = 一致性（把你甩下悬崖摔死的数学 = 你自己失足跌落的数学）。
- **ECS 几乎免费**: `Velocity` 已存在（死代码）复活；定向冲量走专用一次性冲量队列 `ImpulseQueue`（drain 一次即弃，非组件；感知通知另走 `SpatialEventBus`，因现有 `WeaponImpact` 只带标量力+位置、不带目标与冲量向量）；纯数学放 core，glam 即可，脱离 ECS/Godot 可单测。
- **性能**: 常态零成本——99% NPC 留贴地快路，只有正在飞的几十个进 COM 慢路。

## Q3 — 玩家被击退是否统一走 COM

**选择**: 统一走 COM，落地即还控。

**否决**: 玩家保留 Godot 侧专属击退。

**理由**: 守住"玩家与 NPC 无本质区别"立项哲学；响应性用"落地即还控 + 动画预告"补。日常移动仍走纯 Godot（玩家专属），只有被击飞这一下交给物理体模式。

## Q4 — 击退冲量从哪来

**选择**: 真实动量涌现 + 冲量分流。`冲量 = 武器质量 × 挥击末端速度 × 传递系数`，`Δv = 冲量 / 目标质量`；命中瞬间动量按武器分流 `penetration = 动量×穿透系数(锋利)` / `knockback = 动量×击退系数(钝重)`，两系数 TOML。

**否决**: 招式手写 `launch_power` 常量。

**理由**:
- **一次读数喂两消费者**: capsule 扫掠本就带出末端速度 → 一路给伤害（场景 2"加速度加成"= 伤害∝动量）、一路给击退冲量（场景 4"飞多远"）。同一物理量解决两场景。
- **手感涌现**: 钝锤→动量灌击退（人飞）、锐剑→动量灌穿透（扎入不飞），玩家能感觉区别却零硬编。
- **体型差自动成立**: 目标质量作分母，同锤打小妖飞老远/打巨魔几乎不动——场景 2"体型质量差异影响碰撞"免费涌现（前提补盲区 #1 mass）。
- **最小反噬**: `AttackTrajectory::UpwardStrike{launch_power}` 已存在字段重新解释为"上挑传递系数"，不删旧设计。

## Q5 — 飞行中撞到其他实体怎么办

**选择**: 检测 + 离散非递归反应（B），LOD 门控，递归深度上限 1。飞行体撞上 → 停/翻滚 + 给旁观者离散踉跄事件（可选封顶二次小冲量，旁观者撞第三者只触发踉跄不再传冲量）。

**否决**: A 完全无视（鬼穿人群，观感假）；C 动量守恒连续求解（求解器兔子洞，不确定/不可复玩/吃满 CPU）。

**理由**:
- 近场/LOD 0-1 开 B（"踹进人群人群炸开"记忆点）；远场退化成 A。成本：飞行体本就每步 raycast，顺手一次 `entities_in_aabb`。
- 非递归 = 拿到"多米诺一下"观感却永不掉进 N 体守恒求解。一条 `if depth==0` 护栏。
- 一套"检测+离散"盖三场景：生物正常对撞走避让快路、生物高速撞走 B、载具相撞走 B（+ 接建筑 `component_health` 结构伤害）。

## Q6 — 会动表面上的移动（甲板/坐骑/爬巨兽）

**选择**: 父级挂载（A）+ 薄契约 `SurfaceAnchor` trait + `AttachedTo`/`WalkableAnchor` 组件，统一甲板/坐骑/巨兽，不做超类型，不触"载具≠坐骑"意志分界。甲板位姿用双层（粗导航 1-10s + 载人时每帧合成 Gerstner 俯仰横摇），**Rust 单一权威源**同喂渲染与挂载。

**否决**: B 速度继承（旋转漂移、需持续修正）；重量级 DynamicSurface 超类型（把故意解耦的载具/坐骑焊死）。

**文档依据（既有设计已选 A）**: `VehicleState.world_transform: Mat4`（载具 001 L273，注释"用于移动参考系"）；`collision_bodies/cover_surfaces` 明确"载具本地空间"（载具 010）；`walkable_area`（载具 001 L136）；`MotionVector{velocity,angular_velocity,sway}`（载具 001 L478）；骑乘"骨盆=马鞍 transform"（空间查询 §4.2）。

**Python 预算**: 50 可见船 × 4 船体波采样 + 30 挂载实体 local→world ≈ **1.98 µs/帧**（16.6ms 的 0.012%）。

**玩家路径（需原型验证，见 [[003-待验证与开放问题]]）**: 玩家保留 move_and_slide + 平台速度继承 + 偏航修正；"角色站旋转平台"是 Godot 已知难题。

## Q7 — 投射物（火球/箭/投掷物）

**选择**: 复用 core 纯运动学内核（integrate/segment-vs-capsule/terrain_raycast），**不复用 PhysicsBody 重表示**；投射物 = 轻量 SoA 组件 `Projectile{vel,gravity_scale,drag,kind,shooter,payload}` + 逐步 CCD 扫掠（segment-vs-terrain-DDA + segment-vs-capsule，glam 手写，不引 parry）。命中喂回 Q4 伤害结算 + `ImpulseQueue` 冲量（Q5）。

**否决**: "投射物 = PhysicsBody 实体"（数量/结构/碰撞/生命周期全不同）；hitscan 瞬时射线（给不了弧线/风偏/可闪避/可拾回/AOE 击退）；预建对象池（先 ECS 实体，profiling 显示抖动再迁，YAGNI）。

**三空白一并填**: 投射物(#3) + 火球 AOE 击退（火球命中发径向冲量事件→炸飞周围，魔法空白）+ 掉落物积分(#4，DroppedItem = `gravity_scale=1,hit=Rest` 的投射物)。

**Python 预算**: 300 支箭同飞 ≈ **4.66 µs/帧**（0.028%）。

## Q8 — 爬墙/攀爬

**选择**: 攀爬 = 把 `movement_system` 从"假设法线朝上"泛化到"任意表面法线"——贴面查询 `terrain_raycast(向墙)`+`normal_at`、切平面移动 `move - n*(move·n)`、每实体 `Climbing.max_grip_angle` 替代全局坡度常量。握力从 材质×湿度×体力 涌现；失手 detach → 接回 COM 坠落链。壁面重力重定向（蜘蛛）= 同系统参数极值，延后。魔法/战斗仅通过可改写的 `Climbing` 能力字段插入，现在不耦合。

**否决**: AC 式 authored parkour 系统（反涌现哲学）；仅靠 Godot 物理（NPC 是 Rust 侧，必须 Rust 实现）。

**体素 SDF 的天生优势（记忆点）**: 高度场做不了悬垂/洞顶；体素密度场任意面有真实法线，"攀任意体素面含悬垂下方/洞顶"是地形表示的自然能力。

**Python 预算**: 20 近场逐帧攀爬者 ≈ 0.53 µs/帧。

### Q8 修正 — "大量 NPC 都具备攀爬"的规模化
用户追问后修正：Q8 运动学内核只覆盖"看得见的几十个"；"大量普遍具备"把问题从**物理**推到**导航 + LOD**（算力非瓶颈，Python 实算：150 近场逐帧=2.2µs、3000 远场粗粒度=1.8µs 且非每帧）。规模化真正承重墙：
1. **导航图必须编码"可攀爬边" + 按实体握力过滤**——A* 加能力掩码 `entity.max_grip_angle ≥ edge.required_grip`；按需/局部/随 chunk 流式生成，非全局预计算。← 真正的工作量。
2. **LOD 攀爬遍历状态**——远场粗粒度边遍历（无 raycast + 种子失手概率，统计涌现）；近场提升为逐帧运动学；粗↔细提升锚同一攀爬边保连续。
3. **ECS 能力常驻组件 + 廉价状态位**——`Climbing` 常驻不增删（防 archetype 抖动），"当前是否在爬"用状态位。
