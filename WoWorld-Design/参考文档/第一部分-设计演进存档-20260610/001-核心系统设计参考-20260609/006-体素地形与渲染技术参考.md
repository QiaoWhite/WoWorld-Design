# 006-体素地形与渲染技术参考

> 对应设计：WoWorld 体素世界——Voxel Tools (Zylann) + Transvoxel (Marching Cubes LOD变体) + SDF地形雕刻 + Godot 4.6.2

---

## 一、为什么是体素？

WoWorld 选择连续体素（Marching Cubes/Transvoxel）而非方块体素（Minecraft 风格）：

| | 方块体素 (Minecraft) | 连续体素 (WoWorld) |
|---|---|---|
| 地块单元 | 1m³ 立方体 | 0.25-0.5m³ 密度场 |
| 地形外观 | 台阶状 | 平滑曲面 |
| 雕刻方式 | 添加/移除方块 | SDF加减（布尔操作） |
| 洞穴/悬崖 | 难以实现自然感 | 天然支持 |
| 美术方向 | 像素/复古 | 低多边形但平滑 |

**我们的选择**：连续体素允许自然地形的平滑坡面、悬垂、洞穴——这些是方块体素难以优雅实现的地形特征。

## 二、Marching Cubes 与 Transvoxel

### 2.1 Marching Cubes 原理

Marching Cubes 将体素密度场转化为三角网格：

1. 每个体素格有8个角，每个角保存一个密度值（正=固体内部，负=空气）
2. 根据8个角的密度符号，共有 2⁸ = 256 种可能的等值面形状
3. 每种形状对应一个预定义的三角形布局
4. 遍历所有体素格，生成三角网格

**优点**：平滑曲面、C¹连续（如果密度场连续）  
**缺点**：每个体素格都是独立处理的 → 远处体素仍然在生成三角面 → 浪费

### 2.2 Transvoxel — Marching Cubes 的 LOD 变体

Transvoxel 解决的核心问题：**远处的体素不需要那么高的精度。**

```
LOD 0 (近处)：0.5m/体素 → 高精度三角面
LOD 1 (中距)：1m/体素   → 中等精度三角面
LOD 2 (远处)：2m/体素   → 低精度三角面
LOD 3 (极远)：4m/体素   → 极低精度三角面 + 雾/远裁
```

**Transvoxel 的关键创新**：解决了不同 LOD 级别之间的"接缝"问题——较高 LOD 的边界三角形需要特殊处理以无缝连接到较低 LOD。

### 2.3 性能考量

| LOD 级别 | 体素精度 | 覆盖范围 | 三角面数 (估算) |
|---------|---------|---------|---------------|
| LOD 0 | 0.5m | 玩家 50m 内 | ~10K-50K |
| LOD 1 | 1m | 50-150m | ~5K-20K |
| LOD 2 | 2m | 150-500m | ~2K-10K |
| LOD 3 | 4m | 500m+ | ~1K-5K |

## 三、SDF 地形雕刻

### 3.1 SDF（Signed Distance Field）操作

WoWorld 使用 SDF 操作实现地形修改（挖矿、挖地基、填坑）：

| SDF 操作 | 效果 | 用途 |
|---------|------|------|
| **SDF 减法** | 从密度场中减去一个形状 | 挖矿、挖洞、挖战壕 |
| **SDF 加法** | 向密度场中添加一个形状 | 填坑、堆土、筑堤 |
| **SDF 平滑混合** | 两个密度场之间平滑过渡 | 自然的地形与建筑融合 |

### 3.2 SDF 操作的挑战

- **性能**：每次 SDF 操作需要更新影响范围内的所有体素 → 大型操作可能导致卡顿
- **存储**：被修改的区块需要单独保存（不能仅依靠种子重新生成）→ 脏区块追踪
- **Marching Cubes 重生成**：被修改的区块需要重新运行 Marching Cubes → 异步后台重生成

**建议**：
- 限制单次 SDF 操作的体积（如：单次最多修改 3×3×3 个体素格）
- 大型地形改造（如整座山被挖平）使用**批处理 + 分帧执行**
- 被修改的区块标记为"脏"，存档时单独存储

## 四、Voxel Tools (Zylann) 插件参考

### 4.1 为什么选择 Voxel Tools

Zylann 的 Voxel Tools 是 Godot 生态中最成熟的体素解决方案：
- 原生支持 Transvoxel（Marching Cubes + LOD）
- 提供 SDF 地形雕刻 API
- 集成 Godot 的碰撞、导航、渲染系统
- 活跃的社区维护

### 4.2 Voxel Tools 的关键模块

| 模块 | 功能 | WoWorld 的使用场景 |
|------|------|------------------|
| VoxelTerrain | 体素地形节点 | 主地形渲染 |
| VoxelGenerator | 噪声驱动的密度场生成 | 世界生成管线 |
| VoxelTool | SDF 地形修改 | 玩家挖掘、建造 |
| VoxelStream | 区块流式加载 | 区块管理 |
| VoxelMesh | 等值面提取 | Marching Cubes |

### 4.3 潜在的坑与对策

| 潜在问题 | 对策 |
|---------|------|
| 插件停止维护 | 从第一天起封装体素API（terrain_interface.gd），所有体素操作通过接口调用 |
| 性能不满意 | Transvoxel LOD 是成熟的算法——如果真的不够用，可以从 Voxel Tools 迁移到自己实现的 Transvoxel |
| Godot 版本兼容 | 优先使用与 Godot 版本匹配的插件版本；避免过新或过旧的版本 |

## 五、混合渲染策略

WoWorld 不是纯体素世界——网格模型（建筑、NPC、道具）与体素地形共存。

### 5.1 体素 + 网格的共存

| 对象类型 | 渲染方式 | 原因 |
|---------|---------|------|
| 地形（山、谷、洞穴） | 体素 → Marching Cubes | 需要可雕刻、平滑曲面 |
| 建筑 | 网格模型 | 需要细节、纹理、可更换样式 |
| NPC | 骨骼动画 + 网格模型 | 需要动画系统 |
| 植被（树、草） | GPU Instancing + 网格 | 数量大、需要高效渲染 |
| 道具（武器、家具） | 网格模型 | 可替换、细节丰富 |

### 5.2 建筑与地形的融合

建筑放在体素地形上时，需要处理基础（建筑底部嵌入地形）：
- 简单方案：建筑底部有一个"地基"网格，略深入地形
- 进阶方案：建筑放置时对体素地形做局部 SDF 减法（挖一个浅坑）

## 六、性能策略概要

| 策略 | 说明 |
|------|------|
| 分帧加载 | 区块加载分摊到多帧，每帧不超过2-3个新区块 |
| 预测性预加载 | 根据玩家移动方向，提前加载前方区块 |
| 后台异步生成 | 区块生成（噪声+MC+碰撞）全部在后台线程 |
| 脏区块追踪 | 仅保存被修改的区块，未修改区块从种子重生 |
| LOD 平滑过渡 | Transvoxel 自带 LOD 过渡，无需额外处理 |
