# 技术栈重新设计

> **日期**: 2026-06-09  
> **审查基础**: `技术架构.md`、`性能优化分析 20260603.md`、`NPC活人感开发文档ver1.01short.md`、`01-世界生成总流程.md`、`06-体素设计决策.md`  
> **前提**: 独立开发者（Solo），开发周期预计 3-6 年

---

## 〇、核心结论

**原技术栈（Godot 4.6.2 + GDScript 全栈）无法支撑 WoWorld 的设计目标。**

推荐架构：**Rust 模拟核心 + Godot 4.x 游戏客户端**，通过 GDExtension 集成。

这不是"用 Rust 替代 Godot"，而是将**计算密集型模拟**从 Godot 中剥离到 Rust，Godot 专注于它擅长的事——渲染、UI、音频、输入。

## 一、原技术栈为何不可行

### 1.1 致命缺陷清单

`性能优化分析 20260603.md` 由开发者自己在审查全部设计文档后撰写，它诚实地暴露了以下问题：

| #   | 问题                | 原文摘要                                                             | 严重度   |
| --- | ----------------- | ---------------------------------------------------------------- | ----- |
| 1   | **GOAP 规划器**      | "最危险的单点。A* 搜索在最坏情况下可在单次规划中消耗数百ms。50个NPC同时触发GOAP→5000ms→帧完全卡死"    | 🔴 致命 |
| 2   | **GDScript 性能边界** | "GOAP 规划器如果用纯 GDScript 实现，每次规划可能耗时 5-20ms"                       | 🔴 致命 |
| 3   | **NPC 记忆存储**      | "200条×10,000 NPC=2,000,000条记忆→~400MB。需要压缩"（远期 2000条×100,000=2亿条） | 🔴 致命 |
| 4   | **社交传播**          | "30个NPC在城市广场→435个潜在传播对"                                          | 🔴 致命 |
| 5   | **动画 CPU**        | "50个L1 NPC的骨骼动画→2.5-5ms。留给寻路和碰撞的预算只剩3ms"                         | 🟡 高危 |
| 6   | **城市 Chunk 加载**   | "骑马奔向城市→每秒0.5 chunk，城市chunk需1-3秒生成→加载赶不上移动"                      | 🟡 高危 |
| 7   | **关系矩阵**          | "10,000 NPC→约5,000万条有向关系边（完全图）"（已通过惰性创建缓解但仍是大问题）                 | 🟡 高危 |
| 8   | **NPC 内存**        | "Dictionary 格式 10,000 NPC→~400MB（含记忆），超出200MB预算"                 | 🟡 高危 |

### 1.2 根本原因

以上问题不是"优化不够好"——它们反映了一个根本性的错配：

> **Godot 是为"几百个实体、几十个AI、手工场景"设计的通用游戏引擎。WoWorld 需要的是"十万个实体、数千个AI、全程序化世界"的数据密集型模拟引擎。**

具体来说：

1. **GDScript 是脚本语言**，不适合 CPU 密集型计算。GOAP、情绪引擎、记忆检索、世界生成——这些全部是热路径上的密集计算。原文承认"GDScript 适合游戏逻辑，不适合热路径上的密集计算"，但 WoWorld 的设计恰恰将几乎所有系统都放在了热路径上。

2. **Dictionary 不是数据结构**。NPCData 当前设计为 Godot Dictionary（~12KB/NPC）。100K NPC → 1.2GB 仅 NPC 数据本体，不含记忆和关系。紧凑二进制格式可以将这个数字压缩到 ~5KB/NPC（500MB/100K）。

3. **Godot 没有数据库**。2 亿条记忆需要一个真正的嵌入式数据库来存储和检索。Godot 的 Array/Dictionary 在百万级条目上的查找是 O(n)，不可接受。

4. **Godot 的线程模型受限**。世界生成的 10 个阶段、NPC 的分帧更新、记忆的批量压缩——这些需要细粒度的并行计算。Godot 的 WorkerThreadPool 可以处理简单任务，但无法支撑复杂的数据流水线。

5. **Node 架构不适合海量实体**。Godot 的 Node 树模型在数千个节点时就会显著降速。100K NPC 即使只有 1%（L1）实例化为 3D 节点，也有 1000 个 CharacterBody3D——这是 Node 架构的极限边缘。

### 1.3 "先 GDScript 实现再迁移"的策略风险

`技术架构.md` 和 `性能优化分析` 中多次提到"先 GDScript 实现，实测后决定是否迁移到 C++/C#"的策略。这个策略在正常情况下是合理的——但对于 WoWorld，它在架构层面存在严重隐患：

- **数据结构选择是不可逆的**。如果你从第一天就围绕 Godot Dictionary 设计 NPCData，后续迁移到紧凑二进制格式意味着重写所有读写 NPC 数据的代码——而 NPC 数据是几乎所有系统的核心依赖。
- **API 设计受限于语言**。GDScript 的 API 设计天然倾向于 Godot 原生类型（Array、Dictionary、Resource）。当这些 API 被 50+ 个 `.gd` 文件调用后，迁移成本呈指数增长。
- **性能模型完全不同**。在 GDScript 中可以"正常工作"的设计（如每帧遍历所有 L1 NPC 检查条件）在编译语言中可能需要完全不同的架构。

### 2.2 职责划分

| 层 | 技术 | 负责什么 | 不负责什么 |
|----|------|----------|-----------|
| **Rust 模拟核心** | Rust + ECS + LMDB | NPC 心智模拟、GOAP 规划、记忆存储/检索、世界生成管线、经济系统、存档序列化 | 渲染、UI、音频、输入、物理碰撞 |
| **Godot 客户端** | Godot 4.x + GDScript/C# | 3D 渲染、体素 Mesh 生成、动画播放、UI、音频、输入处理、物理（Jolt） | NPC 决策逻辑、记忆数据库、世界生成 |
| **GDExtension 桥** | godot-rust crate | Rust ↔ Godot 数据传递、NPC 状态的增量同步、将模拟结果"翻译"为场景节点状态 | — |

### 2.3 核心原则

1. **模拟与表现分离**：Rust 模拟核心可以在**无头模式**（无 Godot）下运行——这对测试、调试、世界预生成至关重要。
2. **Godot 不拥有 NPC 数据**：NPCData 的唯一真相来源在 Rust 侧。Godot 只持有渲染所需的"表现层快照"（位置、动画状态、当前表情/情绪标签）。
3. **增量同步，非全量复制**：每帧只从 Rust 向 Godot 传输**发生变化**的 NPC 状态，而非全部 100K NPC。
4. **二进制优先**：NPC 数据、记忆、存档全部使用紧凑二进制格式。JSON 仅用于调试输出和 LLM 交互。

## 三、Rust 模拟核心设计

### 3.1 为什么是 Rust

| 对比维度     | GDScript         | C++ (GDExtension) | Rust                            |
| -------- | ---------------- | ----------------- | ------------------------------- |
| 性能       | 慢（脚本语言）          | 极快                | 极快（与 C++ 同级）                    |
| 内存安全     | GC（引用计数）         | 手动管理（易出错）         | 编译期保证（无 GC）                     |
| 并发       | 受限（GDScript 单线程） | 完整但易出错            | fearless concurrency            |
| 生态系统     | Godot 内置         | 成熟                | 快速增长（游戏和数据库 crate 丰富）           |
| 学习曲线     | 低                | 高                 | 高（但长期收益巨大）                      |
| 适合热路径    | ❌                | ✅                 | ✅                               |
| 适合数据结构密集 | ❌                | ✅                 | ✅（enum/pattern matching 极适合状态机） |

对 Solo 开发者而言，Rust 的编译期安全检查是一把双刃剑——初期开发速度较慢（与编译器"搏斗"），但长期维护成本极低（编译器帮你找 bug）。对于 WoWorld 这种复杂度的项目，这个 tradeoff 是值得的。

### 3.2 ECS 架构选型

NPC 模拟采用 ECS（Entity Component System）模式：

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Bevy ECS**（仅使用 ECS 部分） | 成熟、文档丰富、Rust 原生 | 与 Bevy 引擎绑定较深（但 ECS 可独立使用） | ⭐⭐⭐⭐ |
| **Flecs + flecs-rs** | 工业级、C 库性能极致、语言无关 | Rust 绑定不如原生 Rust 方案舒适 | ⭐⭐⭐⭐⭐ |
| **自定义 ECS** | 完全控制 | 造轮子——Solo 开发者不应该浪费生命在这个上 | ⭐ |

**推荐：Flecs**。理由是：
- Flecs 是用 C 写的，性能极致，可以编译为 Rust 可调用的原生库
- 支持 `flecs::sync` 等高级特性（自动将 ECS 变更同步到外部）
- 已有成熟的 Rust 绑定（`flecs-rs`）
- 不绑定任何特定引擎——未来即使放弃 Godot 换其他渲染前端，模拟核心不受影响

### 3.3 关键 Crate 选择

| 用途 | Crate | 理由 |
|------|-------|------|
| ECS | `flecs-rs` | 高性能 C ECS 的 Rust 绑定 |
| 嵌入式数据库 | `lmdb-rkv` | LMDB 的 Rust 绑定。内存映射、读极快、ACID 事务 |
| 噪声生成 | `noise` | Rust 噪声库，支持 Perlin/Simplex/Worley |
| 序列化 | `serde` + `bincode` | 二进制序列化，比 JSON 快 10-100x |
| 数学 | `glam` | SIMD 加速的向量/矩阵库 |
| 随机数 | `rand` + `rand_pcg` | 种子可控的随机数生成 |
| 寻路 | 自定义或 `pathfinding` crate | A* on navmesh |
| 并发 | `rayon` | 数据并行迭代器 |
| Godot 集成 | `godot-rust` | GDExtension 的 Rust 绑定 |
| 日志 | `tracing` | 结构化日志 |
| WFC（建筑生成）| 自定义 | WFC 实现简单，不需要外部依赖 |

### 3.4 NPC 模拟管线（Rust 侧）

```
每游戏帧（~16.7ms @ 60fps）:

L1 NPC（约 1000 个，分 10 批处理）:
  Batch N (100 NPCs):
    ├── 感官更新（视觉射线 → GPU compute shader 批量）
    ├── 情绪引擎（纯数学，极快）
    ├── 决策：
    │   ├── 90% 概率行为树（加权采样，O(n)）
    │   └── 10% GOAP（带 2ms 硬超时 + 缓存）
    ├── 行动推进（状态机 tick）
    └── 记忆检查（事件驱动，非轮询）

L2 NPC（约 10000 个，分 100 批处理）:
  Batch N (100 NPCs):
    ├── 情绪衰减（简化）
    └── 概率状态转换（极简）

L3 NPC（约 100000 个，每日/周批量推演）:
  每日凌晨（游戏时间）:
    └── 统计推演（人口/经济/势力变化的聚合计算）
```

### 3.5 记忆数据库设计

```
LMDB 数据库结构:

数据库 1: event_facts（客观事实）
  Key: event_id (u64)
  Value: { timestamp, location, event_type, participants[], objective_summary }

数据库 2: npc_memories（NPC 主观记忆）
  Key: (npc_id: u64, event_id: u64)  ← 复合键
  Value: { emotional_encoding, personal_attribution, perspective_bias, compression_level, last_access_day }

数据库 3: npc_relationships
  Key: (npc_a: u64, npc_b: u64)
  Value: { affection, trust, familiarity, status, last_interaction_day }
```

**规模估算**：
- 2 亿条事件 × 每个事件被平均 5 个 NPC 记住 = 10 亿条主观记忆条目
- LMDB 单条记录 ~64 字节（压缩后）
- 10 亿 × 64 字节 ≈ 64GB（磁盘）——**太大**
- **必须实施分层压缩**：
  - 热记忆（< 7 天）：完整存储，约 2000 条/NPC × 1000 L1 = 200 万条 → ~128MB
  - 温记忆（7-30 天）：精简存储，约 5000 条/NPC × 10000 L1+L2 = 5000 万条 → ~2GB
  - 冷记忆（30-365 天）：摘要存储，约 50000 条/NPC × 100000 = 50 亿条 → 每条约 20 字节 → ~100GB ——**仍然太大**

**重新思考**：2 亿条"事实"不是每个都会被所有 NPC 记住。实际记忆分布：
- L1 NPC（1000）：每人 ~2000 条热/温记忆 = 200 万条
- L2 NPC（10000）：每人 ~200 条温记忆 = 200 万条  
- L3 NPC（100000）：每人 ~20 条摘要 = 200 万条
- **总计约 600 万条有效记忆**，远小于 10 亿的理论上限

600万条 × 平均 80 字节（压缩后）= **~480MB**。完全可行。

### 3.6 存档设计

```
save_XXX/
├── header.bin          # 元数据（种子、天数、版本）
├── npc_data.bin        # 所有 NPC 的紧凑二进制数据
├── memories/           # LMDB 数据库文件（直接复制）
├── relationships/      # LMDB 数据库文件
├── world_diff.bin      # 被修改过的体素 chunk（增量）
├── factions.bin        # 势力数据
├── economy.bin         # 经济状态
├── events.bin          # 大事件编年史
└── player.bin          # 玩家数据
```

存档/读档策略：
- **增量存档**：LMDB 的 copy-on-write 特性使增量存档天然高效——只需复制修改过的页面
- **异步存档**：Rust 侧后台线程执行，完全不阻塞主循环
- **目标**：存档时间 < 2 秒（100K NPC 规模），存档大小 ~2-3GB

## 四、Godot 游戏客户端设计

### 4.1 Godot 保留的职责

Godot 仍然是最适合做以下事情的引擎：

1. **3D 渲染**：Forward+ 渲染器，低多边形 + 像素纹理的美术风格对 Godot 绰绰有余
2. **体素 Mesh 生成**：从 Rust 传来的体素密度数据 → Godot 侧用 `SurfaceTool` 或 `ArrayMesh` 构建渲染 Mesh。Transvoxel 的等值面提取可以在 Rust 侧完成（速度更快），Mesh 构建在 Godot 侧完成（需要 GPU 上传）
3. **动画系统**：`AnimationTree` + `Skeleton3D` 处理玩家和 L1 NPC 的骨骼动画
4. **UI 系统**：Godot 的 Control 节点体系成熟
5. **音频**：`AudioServer` 管理 3D 音频
6. **输入**：`InputMap` + 输入事件系统
7. **物理**：Jolt Physics 集成（碰撞检测、角色移动）
8. **LLM 网关**：HTTP 请求管理（Godot 的 `HTTPRequest` 节点足够）

### 4.2 Godot 侧不再拥有的数据

| 原方案（Godot 全栈）                       | 新方案（Rust + Godot）                                            |
| ----------------------------------- | ------------------------------------------------------------ |
| NPCData 是 Godot Resource/Dictionary | NPCData 是 Rust struct，Godot 侧只有 `NpcVisualState`（位置/动画/情绪标签） |
| 记忆存储在 NPC Dictionary 的 Array 中      | 记忆存储在 Rust 侧的 LMDB 中，Godot 不可直接访问                            |
| GOAP 在 GDScript 中运行                 | GOAP 在 Rust 中运行，Godot 只接收"当前行动"结果                            |
| 世界生成在 GDScript 中运行                  | 世界生成在 Rust 中运行，Godot 接收"这里应该放什么"的指令                          |
| 经济模拟在 Godot 中                       | 经济模拟在 Rust 中，Godot 只显示 UI 价格                                 |

### 4.3 GDExtension 接口设计

```rust
// godot-rust 暴露给 Godot 的 API 示例

#[gdextension]
unsafe impl ExtensionLibrary for WoWorldCore {}

#[derive(GodotClass)]
#[class(base=Node)]
struct SimulationBridge {
    // 内部持有 Rust 模拟核心的句柄
    core: SimulationCore,
}

#[godot_api]
impl SimulationBridge {
    // === 世界生成 ===
    #[func]
    fn generate_world(&mut self, seed: u64, initial_radius_km: f32);
    
    #[func]
    fn get_chunk_data(&self, chunk_x: i32, chunk_y: i32) -> ChunkData;
    
    // === NPC ===
    #[func]
    fn tick_simulation(&mut self, delta_game_seconds: f64);
    
    #[func]
    fn get_npc_visual_states(&self, player_pos: Vector3, radius: f32) -> Array<Dictionary>;
    // 返回玩家周围 NPC 的渲染所需信息：
    // [{ id, position, rotation, animation_state, emotion_label, current_action_label }]
    
    #[func]
    fn get_npc_detail(&self, npc_id: u64) -> Dictionary;
    // 当玩家与 NPC 深度交互时（对话、查看详情），获取完整心智状态
    
    // === 记忆（仅供 UI 展示，不能直接修改） ===
    #[func]
    fn get_npc_recent_memories(&self, npc_id: u64, count: u32) -> Array<Dictionary>;
    
    // === 存档 ===
    #[func]
    fn save_game(&self, save_path: String) -> bool;
    #[func]
    fn load_game(&self, save_path: String) -> bool;
    
    // === 调试 ===
    #[func]
    fn debug_get_simulation_stats(&self) -> Dictionary;
}
```

## 五、技术栈总结

### 5.1 最终选型

| 层          | 技术                             | 版本/方案          |
| ---------- | ------------------------------ | -------------- |
| **模拟核心语言** | Rust                           | stable (1.80+) |
| **ECS 框架** | Flecs (via `flecs-rs`)         | 最新稳定版          |
| **嵌入式数据库** | LMDB (via `lmdb-rkv`)          | 0.8+           |
| **游戏引擎**   | Godot                          | 4.6+           |
| **引擎集成**   | GDExtension (via `godot-rust`) | 0.2+           |

| **目标平台**   | Windows / Linux / macOS        | —              |

### 5.2 与原方案对比

| 维度 | 原方案（Godot 全栈） | 新方案（Rust + Godot） |
|------|---------------------|----------------------|
| NPC 模拟规模上限 | ~500 L1（实用上限） | ~5000 L1（仅受 CPU 限制） |
| GOAP 规划时间 | 5-20ms/次（GDScript） | <0.5ms/次（Rust 编译优化） |
| 记忆存储上限 | ~10 万条（Dictionary 内存） | 千万级（LMDB 磁盘+缓存） |
| 世界生成速度 | 慢（GDScript 单线程） | 快（Rust + rayon 并行） |
| 存档速度 | 数秒-数十秒（JSON） | <2 秒（二进制增量） |
| 内存占用（100K NPC） | ~1.8GB（Dictionary） | ~500MB（紧凑二进制） |
| 学习曲线 | 低 | 高 |
| 初期开发速度 | 快 | 慢 |
| 长期维护成本 | 高（性能问题不断涌现） | 低（编译器保证正确性） |

## 六、开发路线建议

### 6.1 阶段划分

| 阶段 | 内容 | 预估周期 |
|------|------|----------|
| **阶段 0：技术验证** | Rust↔Godot GDExtension 原型：1 个 NPC 在 Godot 场景中移动，其 AI 决策由 Rust 侧驱动。LMDB 读写 100 万条假记忆。 | 2-4 周 |
| **阶段 1：最小模拟核心** | Rust 侧：NPCData 结构体、情绪引擎、概率行为树、基础记忆系统。Godot 侧：1 个 NPC 的可视化 + 调试面板。 | 2-3 月 |
| **阶段 2：世界生成管线** | Rust 侧：噪声→生物群系→聚落→建筑→NPC 初始化的完整管线。Godot 侧：体素渲染 + 流式加载。 | 3-6 月 |
| **阶段 3：NPC 规模扩展** | 100 NPC → 1000 NPC → 10000 NPC。每轮扩展后性能测试。三层 LOD 切换。 | 3-6 月 |
| **阶段 4：完整系统集成** | GOAP、关系系统、经济、记忆集中数据库、存档/读档。 | 6-12 月 |
| **阶段 5+** | 后续系统（战斗、魔法、LLM 网关等）。规模扩展至 100K。 | 长期 |

### 6.2 关键决策点

1. **不要用 GDScript 写任何 NPC 核心逻辑。** 从第一天起，所有 NPC 决策、记忆操作、情绪计算都在 Rust 侧。
2. **Godot 侧用 GDScript 写胶水代码。** UI、场景管理、输入处理可以用 GDScript——它们的性能不重要。
3. **LMDB 从阶段 0 就引入。** 不要先用 Dictionary 再迁移——数据结构迁移的代价远超一开始就用正确方案。
4. **GOAP 先做 Rust 原型，设定严格的性能预算。** 2ms 硬超时、动作空间预裁剪、规划缓存——在原型阶段就实现这些约束。

## 七、风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| **Rust 学习曲线阻挡开发进度** | 🟡 中 | Solo 开发者已有编程基础；Rust 的学习投资在 3-6 年开发周期中会被摊销；从不涉及 unsafe 的模块开始（情绪引擎、概率 AI） |
| **godot-rust 成熟度不足** | 🟡 中 | GDExtension 是 Godot 4.2+ 官方支持的接口；godot-rust 已有活跃社区；架构设计使模拟核心**不依赖** godot-rust（可脱离 Godot 独立测试） |
| **LMDB 在游戏场景下的适用性** | 🟡 中 | LMDB 适合读多写少的场景（记忆检索正是如此）；需在原型阶段验证写入性能（记忆编码频率） |
| **Rust↔Godot 数据同步开销** | 🟢 低 | 增量同步（只传变化）、批量传输（每帧一批而非单个 NPC）、紧凑二进制格式 |
| **编译时间过长** | 🟢 低 | Rust 编译慢但在现代硬件上可控；使用 sccache + 增量编译；将模拟核心拆分为多个 crate 以并行编译 |
| **Solo 维护两套代码库** | 🟡 中 | Rust 代码和 Godot 场景/资源分离清晰；Rust 侧有完善的单元测试覆盖（独立于 Godot 运行） |
