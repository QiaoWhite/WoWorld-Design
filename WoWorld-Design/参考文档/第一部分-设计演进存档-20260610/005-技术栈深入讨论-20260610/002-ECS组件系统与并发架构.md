# 002-ECS 组件系统与并发架构

> 对应：`004-技术栈重新设计` §3.2·3.4  
> 重点：Flecs ECS 的组件定义、System 管线、分帧调度、rayon 并发

---

## 一、为什么 ECS 适合 WoWorld

NPC 模拟天然适合 ECS（Entity Component System）：

| ECS 特性 | 如何对应 WoWorld 的需求 |
|----------|----------------------|
| **Entity 仅是 ID** | 每个 NPC 是一个 Entity（u64），没有附加任何 Game Object 开销 |
| **Component 是纯数据** | NpcEmotion、NpcPhysiology、NpcAction……各自是独立 Component |
| **System 遍历匹配的 Entity** | 情绪引擎 System 遍历所有有 NpcEmotion 组件的 Entity |
| **查询天然并行化** | 同一个 System 对不同 Entity 的操作无依赖 → rayon 并行 |

**ECS 解决了 WoWorld 的核心性能矛盾**：如何让 100K 个实体各自运行独立的心智模块，而不会因为实体间的交叉依赖导致无法并行？答案是：**能并行的 System（情绪更新）并行执行，有依赖的 System（GOAP 规划→行动执行）串行管道化**。

## 二、组件定义

### 2.1 核心组件

```rust
// ===== 身份与存在 =====

/// 标记组件：此 Entity 是一个 NPC（区别于物品、建筑等）
#[derive(Component)]
struct Npc;

/// 不可变的身份数据
#[derive(Component)]
struct Identity {
    id: u64,
    name: String,          // 生成后的名字（缓存，避免每次显示时重新生成）
    birth_day: i32,
    species: Species,
    sex: Sex,
}

// ===== 心智组件（仅 L1/L2 NPC 拥有） =====

#[derive(Component)]
struct Personality(pub NpcPersonality);  // 上篇定义的结构

#[derive(Component)]
struct Physiology(pub NpcPhysiology);    // 上篇定义的结构

#[derive(Component)]
struct EmotionalState(pub NpcEmotion);   // 上篇定义的结构

// ===== 行为组件 =====

/// 当前正在执行的动作
#[derive(Component)]
struct CurrentAction {
    action_id: u16,
    remaining_ticks: u16,
    target: Option<Entity>,
    interruption_allowed: bool,
}

/// 动作完成/中断时触发的事件（一次性组件，下一帧被 System 消费）
#[derive(Component)]
struct ActionCompleted {
    action_id: u16,
    result: ActionResult,
}

/// 概率行为树的决策缓存（避免每帧重新采样）
#[derive(Component)]
struct BehaviorCache {
    last_decision_tick: u64,
    cooldown_remaining: u16,
}

// ===== 空间组件 =====

/// 世界坐标（L1 和部分 L2 NPC 拥有）
#[derive(Component)]
struct WorldPosition {
    x: f32, y: f32, z: f32,
    chunk_x: i32, chunk_y: i32,
}

/// 移动目标（导航系统写入，移动系统读取）
#[derive(Component)]
struct MoveTarget {
    x: f32, y: f32, z: f32,
    arrival_tolerance: f32,
    path: Vec<(f32, f32, f32)>,  // 预计算的导航路径
}

// ===== LOD 相关 =====

#[derive(Component)]
struct SimulationLevel {
    level: SimLod,
    level_change_cooldown: u16,  // 防止抖动
}

/// 标记：此 NPC 是"关键角色"——即使距离远也锁定 L1
#[derive(Component)]
struct ImportantNpc;

// ===== 社会关系（运行时缓存） =====

/// L1 NPC 的热关系缓存（从 LMDB 加载到内存中）
#[derive(Component)]
struct SocialCache {
    known_npcs: HashMap<u64, CachedRelation>,
    // 最近互动过的 NPC 的轻量缓存
}

struct CachedRelation {
    affection: f32,
    trust: f32,
    familiarity: f32,
    tag: u8,
}
```

### 2.2 标记组件（用于 System 查询过滤）

```rust
/// 此 NPC 当前在玩家视野中（需要渲染）
#[derive(Component)] struct Visible;

/// 此 NPC 的 GOAP 规划器当前活跃
#[derive(Component)] struct Planning;

/// 此 NPC 正在进行社交互动
#[derive(Component)] struct Socializing { partner: Entity }

/// 此 NPC 正在战斗中
#[derive(Component)] struct InCombat { target: Entity }
```

## 三、System 管线

### 3.1 每帧的 System 执行顺序

```
FRAME START
│
├─ Phase 1: 空间更新（需要玩家位置，先执行）
│   ├── sys_update_lod           # 根据玩家位置重新分配 L1/L2/L3
│   ├── sys_spawn_new_l1         # L3→L2→L1 晋升的 NPC 实例化组件
│   └── sys_despawn_l3           # L1→L2→L3 降级的 NPC 移除组件
│
├─ Phase 2: 感知（L1 only, 可并行）
│   ├── sys_vision               # 视觉射线检测（射线结果从 Godot 侧回传）
│   ├── sys_hearing              # 听觉事件处理（事件驱动，非轮询）
│   └── sys_update_subjective_world  # 构建每个 L1 NPC 的"主观世界状态"
│
├─ Phase 3: 心智更新（L1+L2, 可并行）
│   ├── sys_emotion_decay        # L2 NPC 的情绪衰减
│   ├── sys_emotion_update       # L1 NPC 的完整情绪引擎
│   ├── sys_emotion_contagion    # L1 NPC 间的情绪感染（需要 spatial query）
│   └── sys_physiology_tick      # 饥饿/口渴/疲劳的增量更新
│
├─ Phase 4: 决策（L1 only, 串行——GOAP 有全局副作用）
│   ├── sys_probabilistic_behavior  # 90% 的日常决策（加权采样，极快）
│   ├── sys_goap_check           # 检查是否需要 GOAP 介入（目标受阻/生存危机）
│   ├── sys_goap_plan            # GOAP A* 搜索（带 2ms 硬超时 + 缓存）
│   └── sys_action_selection     # 将决策结果写入 CurrentAction 组件
│
├─ Phase 5: 行动执行（L1+L2）
│   ├── sys_action_tick          # 推进当前动作的时间线
│   ├── sys_movement             # 向 MoveTarget 移动
│   └── sys_action_completion    # 处理 ActionCompleted 事件
│
├─ Phase 6: 记忆（事件驱动，非轮询）
│   ├── sys_memory_encode        # 将重要事件编码为记忆（仅当有值得记录的事）
│   ├── sys_memory_compress      # 记忆压缩（每游戏日运行一次，分批）
│   └── sys_memory_social_spread # 社交记忆传播（限速：每NPC每游戏小时 1-2 条）
│
├─ Phase 7: 关系维护
│   ├── sys_relationship_decay   # 关系随时间衰减（每游戏日）
│   └── sys_relationship_update  # 因互动而更新关系值（事件驱动）
│
└─ Phase 8: 同步到 Godot
    └── sys_sync_visual_state    # 将变化打包成 NpcVisualState 传给 Godot
```

### 3.2 分帧策略

不试图在一帧内更新所有 NPC。使用**时间分片**：

```rust
// 分帧常量
const L1_BATCH_SIZE: usize = 200;      // 每帧处理 200 个 L1 NPC
const L2_BATCH_SIZE: usize = 500;      // 每帧处理 500 个 L2 NPC
const EMOTION_UPDATE_INTERVAL: u16 = 3; // 每个 NPC 每 3 帧更新一次情绪

// 情绪引擎的 System 实现伪代码
fn sys_emotion_update(world: &World, frame: u64) {
    let mut query = world.query::<&mut EmotionalState>().with::<SimulationLevel>().build();
    
    // 仅处理本帧的批次
    let batch_start = (frame as usize * L1_BATCH_SIZE) % query.count();
    let batch: Vec<_> = query.iter_mut().skip(batch_start).take(L1_BATCH_SIZE).collect();
    
    // rayon 并行处理本批次的 NPC（它们之间无依赖）
    batch.par_iter_mut().for_each(|emotion| {
        emotion.tick();  // 纯数学计算，无副作用
    });
}
```

> **设计注记**：分帧策略意味着每个 L1 NPC 的情绪每 `(1000/200) × 3 = 15 帧 = 0.25s` 更新一次。这在感知上完全足够——人类情绪的明显变化也需要数秒到数分钟。

## 四、并发架构

### 4.1 并发模型

WoWorld 的模拟核心使用**分阶段数据并行**模型：

```
┌─────────────────────────────────────────────────────┐
│                    主线程（协调）                       │
│                                                     │
│  Phase 1: 空间更新 (sequential)                      │
│  Phase 2: sys_vision (sequential, 等待 Godot 射线结果) │
│          sys_hearing (sequential)                    │
│          sys_subjective_world (parallel via rayon)    │
│  Phase 3: sys_emotion_* (parallel via rayon)          │
│  Phase 4: 决策 (sequential, GOAP 有全局副作用)         │
│  Phase 5: 行动 (parallel via rayon)                   │
│  Phase 6: 记忆 (sequential, LMDB 写入需事务)           │
│  Phase 7: 关系 (parallel via rayon)                   │
│  Phase 8: 同步 (sequential)                          │
└─────────────────────────────────────────────────────┘
```

**为什么有些 Phase 是 sequential**：
- **决策**：GOAP 规划器修改世界状态（行动的效果）。如果两个 NPC 同时规划并同时抢同一个资源（如最后一个面包），必须串行化以避免不一致。
- **记忆写入**：LMDB 的写事务是串行的（但读可以并发）。
- **视觉**：射线检测的结果从 Godot 侧回传，这是外部依赖。

**为什么有些 Phase 可以 parallel**：
- **情绪更新**：每个 NPC 的情绪仅依赖自己的生理状态和当前情绪——与其他 NPC 无数据依赖。
- **行动推进**：每个 NPC 的行动独立推进时间线。
- **关系衰减**：每条关系独立计算衰减。

### 4.2 rayon 的使用原则

```rust
// ✅ 正确：每个 NPC 独立计算，无共享可变状态
query.par_iter_mut().for_each(|(emotion, phys, pers)| {
    emotion.pleasure -= phys.hunger * 0.01;
    emotion.pleasure += (pers.extraversion - 0.5) * 0.005;
});

// ❌ 错误：NPC 间有交互（情绪感染）时直接用 par_iter
// 需要先在 sequential 阶段建立"谁感染谁"的传播图，
// 然后在 parallel 阶段应用传播结果
```

**情绪感染的正确并发方式**：

```rust
// Step 1 (sequential): 建立感染图
let mut contagion_map: HashMap<Entity, Vec<(Entity, f32)>> = HashMap::new();
for (entity, emotion, pos) in query.iter() {
    let neighbors = spatial_index.query_radius(pos, EMOTION_CONTAGION_RADIUS);
    for neighbor in neighbors {
        let intensity = calc_contagion(emotion, &world.get::<EmotionalState>(neighbor));
        contagion_map.entry(neighbor).or_default().push((entity, intensity));
    }
}

// Step 2 (parallel): 应用感染结果
contagion_map.par_iter().for_each(|(target, sources)| {
    let mut emotion = world.get_mut::<EmotionalState>(*target).unwrap();
    for (source, intensity) in sources {
        emotion.apply_contagion(*source, *intensity);
    }
});
```

## 五、ECS 与 LMDB 的交互

Flecs ECS 负责**运行时热数据**（情绪、动作、位置）。LMDB 负责**持久化冷数据**（记忆、关系）。

```
ECS (内存)                          LMDB (磁盘+文件缓存)
─────────                          ───────────
NpcEmotion      ← 实时更新          事件事实（不变）
CurrentAction   ← 实时更新          主观记忆（追加+压缩）
WorldPosition   ← 实时更新          关系数据（更新少，读多）
Physiology      ← 实时更新
Personality     ← 启动时加载（不变运行时）
Identity        ← 启动时加载（不变运行时）

ECS → LMDB:
  - 记忆编码：当 NPC 经历值得记录的事件 → 写入 LMDB
  - 关系更新：互动完成后 → 异步写入 LMDB（不阻塞 ECS tick）
  
LMDB → ECS:
  - NPC 晋升 L1/L2：从 LMDB 加载该 NPC 的记忆和关系到 ECS 缓存
  - 记忆检索：NPC 进行社交闲聊时 → 从 LMDB 读取相关记忆
```

**关键原则**：ECS 是模拟引擎的"工作内存"，LMDB 是"长期记忆"。ECS 中的数据有寿命（NPC 降级后释放），LMDB 中的数据是永久的。

## 六、性能测算

以帧预算 16.7ms (60fps) 为基准，Rust 侧的模拟核心目标使用 ≤7ms/帧（留给 Godot 渲染+物理 ~9.7ms）：

| Phase | 并行? | 处理量/帧 | 预计耗时 |
|-------|-------|----------|---------|
| Phase 1: 空间更新 | 串行 | ~2000 次距离计算 | <0.1ms |
| Phase 2: 感知 | 并行 | 200 L1 的视觉+听觉 | <1ms |
| Phase 3: 心智更新 | 并行 | 200 L1 + 500 L2 的情绪 | <0.5ms |
| Phase 4: 决策 | 串行 | ~10 个 GOAP 规划 + 200 概率决策 | <3ms |
| Phase 5: 行动执行 | 并行 | 200 L1 + 500 L2 | <0.5ms |
| Phase 6: 记忆 | 串行 | ~20 条记忆编码 + 检索 | <1ms |
| Phase 7: 关系 | 并行 | 1000 条关系衰减 | <0.1ms |
| Phase 8: 同步 | 串行 | 打包数据 | <0.3ms |
| **总计** | | | **<6.5ms** |

> 在 Rust 的编译优化（相当于 C++ 的 -O3 级别）下，纯数学计算（情绪引擎、概率采样）的执行速度是 GDScript 的 50-100 倍。GOAP 的 A* 搜索在紧凑数据结构和编译优化下有 10-50 倍的性能提升。
