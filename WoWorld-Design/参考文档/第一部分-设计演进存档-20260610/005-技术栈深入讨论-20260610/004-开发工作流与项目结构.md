## 二、Solo 开发者的日常开发流程

### 2.1 开发纯模拟逻辑（最常见的工作）

```
1. 在 rust_core/src/npc/ 中修改/新增代码
2. cargo test -p rust_core          # 跑单元测试（< 5s）
3. cargo bench -p rust_core         # 跑基准测试（验证性能无退化）
4. cargo build -p rust_core --release
5. 将编译产物复制到 godot_client/addons/woworld_core/
6. 打开 Godot，运行场景验证
```

**优化**：步骤 4-6 应该用一个 `Makefile` 命令替代：

```makefile
.PHONY: build-and-run
build-and-run:
    cargo build -p rust_core --release
    cp target/release/woworld_core.dll godot_client/addons/woworld_core/
    godot --path godot_client --scene scenes/test_world.tscn
```

### 2.2 开发 UI/场景/动画（Godot 侧工作）

```
1. 打开 Godot Editor
2. 编辑 .tscn 场景、调整动画、修改 UI 布局
3. F5 运行 → 在 Godot 内调试
```

### 2.3 调试跨边界问题

```
情景：NPC 在 Godot 中行为异常，但不知道是 Rust 决策问题还是 Godot 渲染问题。

步骤：
1. 在 rust_core 中启用详细日志
   export RUST_LOG=woworld_core::npc::decision=trace
2. 运行 cargo test -p rust_core -- --nocapture  # 先纯 Rust 侧验证
3. 如果 Rust 侧测试通过 → 问题是 Godot 渲染/同步层
4. 如果 Rust 侧测试失败 → 加更多日志，在 Rust 侧调试
```

## 三、测试策略

### 3.1 Rust 侧（主要测试战场）

```rust
// == 单元测试：每个模块独立 ==

#[test]
fn test_emotion_hunger_drags_pleasure_down() {
    let mut emotion = NpcEmotion::default();
    let phys = NpcPhysiology { hunger: 0.8, ..default() };
    emotion.tick(&phys, &NpcPersonality::default(), 1.0);
    assert!(emotion.pleasure < 0.0, "饥饿应降低愉悦度");
}

#[test]
fn test_goap_finds_path_to_food() {
    let mut world = WorldState::new();
    world.set("has_money", true);
    world.set("bakery_open", true);
    let plan = goap_planner::plan(&world, &Goal::Eat);
    assert!(plan.is_some());
    assert_eq!(plan.unwrap().actions[0].action_id, ACTION_BUY_BREAD);
}

// == 集成测试：多系统交互 ==

#[test]
fn test_npc_full_day_cycle() {
    let mut sim = SimulationCore::new(seed: 42);
    let npc = sim.create_npc(NpcTemplate::villager());
    
    // 快进 24 游戏小时
    sim.advance_game_time(24 * 60);
    
    let npc = sim.get_npc(npc);
    // NPC 应该吃了至少一顿饭
    assert!(npc.physiology.hunger < 0.5);
    // NPC 应该有新的记忆
    assert!(!sim.get_recent_memories(npc, 10).is_empty());
    // NPC 的情绪应该不是完全中性的
    assert!(npc.emotion.pleasure.abs() > 0.01);
}

// == 性能基准 ==

#[bench]
fn bench_emotion_update_1000_npcs(b: &mut Bencher) {
    let mut sim = SimulationCore::new(seed: 42);
    for _ in 0..1000 {
        sim.create_npc(NpcTemplate::random());
    }
    b.iter(|| sim.tick_emotions());
    // 目标：< 0.5ms
}

// == 涌现行为测试（长时间运行） ==

#[test]
fn test_world_stability_100_years() {
    // 这不是每次提交都跑的测试——只是在重大变更时手动运行
    let mut sim = SimulationCore::new(seed: 42);
    sim.generate_world(4.0); // 4km²
    sim.populate_npcs(1000);
    
    // 模拟 100 年
    for _ in 0..(365 * 100) {
        sim.advance_game_time(24 * 60);
    }
    
    // 世界不应该崩溃
    assert!(sim.npc_count() > 0, "世界在100年后不应空无一人");
    assert!(sim.economy().is_stable(), "经济不应崩溃");
    assert!(sim.social().has_formed_factions(), "应该形成了势力");
}
```

  - 引入 LMDB

  - 性能测试：100 个 NPC 同时运行简单行为

## 七、最危险的陷阱

### 陷阱 1：过度设计 Rust 侧

"我要用 Rust 的类型系统完美建模整个 NPC 系统"——然后花 3 个月在设计 trait hierarchy 和泛型约束上，一行实际模拟代码都没写。

**对策**：前 3 个月用最简单的 struct + 普通函数。不做 trait abstraction，不做泛型。`npc.emotion.pleasure -= npc.physiology.hunger * 0.01;` 比 `EmotionSystem::<Plutchik<PAD<f32>>>::tick()` 更好。

### 陷阱 2：在 Godot 侧重复模拟逻辑

"Rust 返回了 NPC 状态，但我需要在 Godot 侧也做一些判断来决定显示什么。"——渐渐地在 Godot 侧积累了模拟逻辑的重复实现。

**对策**：Godot 侧只允许三种操作：渲染、UI、输入。如果发现自己想在 Godot 侧写 `if npc.emotion.pleasure < 0`，停下来——这个判断应该在 Rust 侧完成，然后通过 `NpcVisualState` 传给 Godot。

### 陷阱 3：忘记做无头测试

"我没法测试世界生成，因为它需要渲染。"——错了。Rust 模拟核心可以在**完全没有 Godot** 的情况下运行。世界生成、NPC 模拟、经济系统——全部可以在命令行中测试和验证。

**对策**：从第一天起，每个 Rust 模块都必须有独立于 Godot 的单元测试。如果某个 System "必须要开着 Godot 才能测"——那是设计有问题。
