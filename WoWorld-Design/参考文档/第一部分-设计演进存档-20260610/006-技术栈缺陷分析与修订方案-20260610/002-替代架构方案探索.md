# 002-替代架构方案探索

> 基于 001 识别的十大缺陷，探索三种替代架构

---

## 方案 A：纯 Rust + 自定义轻量渲染前端（激进方案）

### 架构

```
┌─────────────────────────────────────────────┐
│              Rust (唯一代码库)                │
│                                             │
│  ┌─────────┐ ┌─────────┐ ┌───────────────┐  │
│  │ 模拟核心  │ │ wgpu 渲染│ │ egui UI       │  │
│  │ · NPC   │ │ · voxel │ │ · 主菜单       │  │
│  │ · 世界  │ │ · mesh  │ │ · 背包面板     │  │
│  │ · 经济  │ │ · anim  │ │ · 对话界面     │  │
│  │ · 记忆  │ │ · LOD   │ │ · 调试面板     │  │
│  └─────────┘ └─────────┘ └───────────────┘  │
│                                             │
│  共享 ECS 世界（所有系统在同一进程内）         │
│  零拷贝、零 FFI、统一调试                     │
└─────────────────────────────────────────────┘
```

### 为什么考虑这个方案

001 识别的十个缺陷中，有六个直接源于"两个分离的运行时"：数据拷贝、类型系统分裂、调试无统一视图、构建复杂度、物理分裂、GDExtension 脆弱性。这个方案从根本上消除了 FFI 边界。

### 组件

| 层 | 技术 | 备注 |
|----|------|------|
| 渲染 | `wgpu` | Rust 原生的跨平台图形 API。比 Godot 的渲染器更底层，给体素渲染完全的控制权 |
| 体素 | 自建 Transvoxel + wgpu compute shader | 等值面提取可以在 GPU 上做（compute shader） |
| UI | `egui` | Immediate mode GUI。不适合做华丽的 RPG 界面，但作为基础是够用的 |
| 动画 | 自建 GPU 骨骼动画 | 使用 compute shader 批量处理骨骼矩阵计算 + vertex skinning |
| 物理 | `rapier` | 纯 Rust 物理引擎。001 缺陷九直接消失——AI 和物理在同一 Runtime |
| 音频 | `kira` 或 `rodio` | Rust 音频库。功能不如 Godot AudioServer，但对于独立游戏足够 |
| 资产 | `gltf` crate + 自定义导入器 | GLTF 加载 + 转换为自定义运行时格式 |
| ECS | Flecs (via `flecs-rs`) | 与之前方案相同 |

### 优势

1. **零 FFI**：模拟、渲染、UI、音频在同一进程，同一类型系统
2. **统一的调试视图**：`egui` 面板可以直接显示任何 ECS 组件的值
3. **GPU 驱动的动画**：compute shader 批量处理，1000 个 NPC 的骨骼动画可以在 1-2ms 内完成
4. **完全的渲染控制**：体素 chunk 的 LOD 切换、MultiMesh 实例化、自定义 culling——全部由开发者控制，不受引擎限制
5. **编译为单一二进制**：一个 `cargo build --release` 产出可执行文件。没有动态库，没有引擎版本兼容问题
6. **单一构建系统**：只有 Cargo。CI 配置极简

### 劣势

1. **没有 GUI 编辑器**：场景布局、UI 设计全部代码化——需要大量的手写布局代码
2. **egui 不适合精美 RPG UI**：Immediate mode GUI 做技能面板、背包界面、对话窗口——能做但痛苦。富文本、动画过渡、Theme 系统——全部需要自己造
3. **音频功能精简**：`kira` 能做基本的 3D 音频，但混音、效果器、音频总线——需要自己搭建
4. **资产管线从零开始**：Blender → GLTF → 自定义运行时格式。渲染管线需要自己处理材质、着色器、纹理格式
5. **初期开发速度极慢**：前 3-6 个月可能都在搭建基础设施，看不到任何"游戏"
6. **动画工具链缺失**：Godot 的 AnimationTree 有可视化编辑器可以预览 BlendSpace。纯代码——你只能在运行时看效果

### 适用判断

这个方案适合：已经熟练掌握 Rust + wgpu + GPU 编程 + 游戏引擎架构的开发者。对 Solo 开发者而言，**这等于同时写一个游戏和半个游戏引擎**。

## 方案 B：Rust 模拟核心 + Bevy 前端（折中方案）

### 架构

```
┌──────────────────────────────────────┐
│           Rust (唯一代码库)           │
│                                      │
│  ┌────────────────┐ ┌──────────────┐ │
│  │ 模拟核心 (同前)  │ │ Bevy 前端     │ │
│  │ · Flecs ECS    │ │ · bevy_render │ │
│  │ · GOAP         │ │ · bevy_ui     │ │
│  │ · LMDB         │ │ · bevy_audio  │ │
│  │                │ │ · bevy_anim   │ │
│  └────────────────┘ └──────────────┘ │
│                                      │
│  两个 ECS 世界（Flecs + Bevy）        │
│  但都在同一 Rust 进程中——无 FFI       │
└──────────────────────────────────────┘
```

### 为什么是"折中"

Bevy 解决了方案 A 的部分核心问题（渲染、UI、音频、资产系统已经有现成的），同时消除了 Godot 引入的 FFI 边界。但它引入了新的问题。

### 优势（vs Godot 方案）

1. **无 FFI**：模拟核心和渲染前端在同一 Rust 进程
2. **ECS 原生**：Bevy 本身就是 ECS 引擎——不需要像 Godot 那样把 ECS 数据映射到 Node 树
3. **统一调试**：`bevy-inspector` 可以直接查看/修改任何 Component
4. **Bevy 在快速成熟**：每 3-4 个月一个大版本，UI/动画/音频在持续改进

### 劣势（vs Godot 方案）

1. **UI 仍然不如 Godot**：`bevy_ui` 是 Flexbox 布局，功能不如 Godot Control 丰富
2. **动画系统仍在早期**：`bevy_animation` 的功能相当于 Godot 的 30%
3. **没有场景编辑器**：所有场景和 UI 用代码构建
4. **资产导入不如 Godot 成熟**：GLTF 导入基本可用但不是无缝
5. **编译速度慢**：Bevy 是大型依赖——首次 `cargo build` 可能 15-20 分钟
6. **Bevy 的 API 不稳定**：每个大版本都有 breaking change。这意味着每 3-4 个月需要更新适配代码

### 关键问题：Flecs ECS vs Bevy ECS

004 方案推荐 Flecs 作为模拟核心的 ECS。在 Bevy 方案中，有两个选择：

**选择 1：保留 Flecs，Bevy 仅做渲染前端**
- 与 Godot 方案类似，但换成了 Bevy 做渲染
- 优势：模拟核心仍然独立于渲染引擎
- 劣势：两个 ECS 世界之间的同步开销

**选择 2：完全用 Bevy ECS，模拟和渲染在同一个 ECS 世界**
- 最极致的整合——零拷贝，模拟和渲染共享 Component
- 劣势：模拟核心与 Bevy 版本深度绑定。Bevy API break → 模拟核心也要改

## 方案 C：C++ 模拟核心 + Godot 客户端（务实方案）

### 架构

```
C++ (模拟核心, 通过 GDExtension C API)
  └── GDExtension ── Godot 4.x (客户端)
```

### 为什么考虑 C++

- Godot 引擎本身是 C++ 写的。GDExtension 的原生接口是 C ABI，C++ 可以直接调用。
- Godot 的核心类和数据结构（`Vector3`、`Array`、`Dictionary`、`AABB`）在 C++ GDExtension 中可以直接使用——不需要像 Rust 那样通过 bincode 序列化。
- 性能与 Rust 同级。
- Godot 的 C++ GDExtension 有官方支持和示例。

### 优势（vs Rust 方案）

1. **与 Godot 深度整合**：C++ GDExtension 可以直接使用 Godot 的 `Vector3`、`Dictionary`、`Callable` 等类型——不需要序列化
2. **官方支持**：Godot 官方维护 C++ GDExtension 绑定
3. **不需要 godot-rust**：不依赖社区 crate
4. **Godot 社区中 C++ 用户更多**：遇到问题更容易找到帮助
5. **性能等同于 Rust**：两者编译到原生代码

### 劣势（vs Rust 方案）

1. **内存安全**：没有 borrow checker。NPC 系统的复杂度意味着悬垂指针、use-after-free、data race 的风险显著上升——而且这些 bug 极难排查
2. **编译速度**：C++ 的大型模板项目编译可能比 Rust 更慢
3. **包管理**：没有 Cargo。vcpkg/Conan 可用但不如 Cargo 方便
4. **并发更危险**：C++ 的线程安全完全是程序员的责任
5. **没有 `serde` 级别的零成本序列化**

### 适用判断

最务实的方案——如果开发者对 C++ 比对 Rust 更熟悉，并且愿意接受内存安全的 tradeoff。对于需要与 Godot 深度整合的场景，C++ 是天然的。

## 关键洞察

在所有方案中，有两个矛盾是无法同时最优解决的：

**矛盾一：引擎成熟度 vs FFI 复杂度**
- 成熟引擎（Godot）= 功能丰富 + FFI 复杂
- 不成熟引擎（Bevy）= 功能不足 + 无 FFI
- 自建引擎（wgpu）= 完全控制 + 天文数字的工作量

**矛盾二：初期开发速度 vs 长期扩展上限**
- C++ + Godot = 最快出原型，但内存安全风险随复杂度增长
- Rust + Godot = 平衡，但 FFI 开销和动画瓶颈限制 NPC 规模上限
- 纯 Rust = 最慢的起步，但 3-6 年后的总成本可能更低

**没有完美的方案。只有最适合 Solo 开发者当前能力和项目当前阶段的方案。**
