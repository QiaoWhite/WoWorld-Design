# 003-最终修订技术栈方案

> 综合 001（缺陷分析）和 002（替代方案探索）后的最终推荐

---

## 〇、核心结论

**修订后的技术栈：Rust 模拟核心 + Godot 4.x 客户端 + 关键子系统从 Godot 剥离到 Rust**

与 004/005 的原方案相比，核心变化：

| 模块 | 原方案 (004/005) | 修订方案 |
|------|-----------------|---------|
| NPC 模拟 | Rust | Rust（同） |
| NPC 动画 | Godot AnimationTree | **Rust 侧 GPU compute shader** |
| 体素渲染 | Godot Voxel Tools (Zylann) | **Rust 侧 Transvoxel + wgpu compute** |
| 物理 | Godot Jolt Physics | **Rust 侧 Rapier** |
| 渲染前端 | Godot 3D 场景 + Forward+ | **Rust 侧 wgpu 渲染器 + Godot 仅做 UI/后处理** |
| UI | Godot Control | Godot Control（同——这是 Godot 最有价值的部分） |
| 音频 | Godot AudioServer | Godot AudioServer（同） |
| 输入 | Godot InputMap | Godot InputMap（同） |
| LLM 网关 | Godot HTTPRequest | Godot HTTPRequest（同） |

**本质变化**：从"Rust 做模拟 + Godot 做一切渲染"变为"Rust 做模拟和核心渲染 + Godot 做 UI、音频、输入和最终合成"。

## 一、修订动因：致命缺陷的解决方案

001 识别了两个致命/高危缺陷，它们驱动了整个修订：

### 缺陷四：动画无法规模化（🔴 致命）

**根因**：Godot AnimationTree 是 CPU 驱动的、单角色状态机——无法规模化到 1000 个 L1 NPC。

**解决方案**：将骨骼动画的计算从 Godot 剥离到 Rust 侧，使用 GPU compute shader 批量处理。

```
Godot AnimationTree 模式（不可规模化）:
  每个 NPC 一个 AnimationTree 实例
    → 1000 个实例 × 0.01ms = 10ms+

GPU Compute Shader 模式（可规模化）:
  所有 NPC 的骨骼矩阵在一个 dispatch 中计算
    → 1000 个 NPC × 50 根骨骼 = 50000 个矩阵
    → GPU 上约 0.5-1ms（并行处理）
```

具体实现：
1. 动画数据（关键帧、骨骼层级）在 Rust 侧加载并上传到 GPU buffer
2. 每个 NPC 的动画状态（当前动画、时间、混合权重）存储在 GPU buffer 中
3. compute shader 批量计算所有 NPC 的骨骼矩阵
4. 结果矩阵直接用于 vertex shader 中的蒙皮——不需要回读到 CPU
5. Godot 侧只需要提供一个旋转的摄像机和一个方向光

### 缺陷三：Node 架构不适合海量实体（🔴 高）

**根因**：即使只渲染 1000 个 L1 NPC，每个都需要 5+ 个场景节点 → 5000+ 个 Node → SceneTree 回调开销。

**解决方案**：NPC 渲染不经过 Godot 的 Node 树。Rust 侧直接管理所有 NPC 的渲染数据，通过 wgpu 渲染到一个离屏纹理，Godot 将这个纹理合成到最终画面上。

```
Godot Node 模式（不可规模化）:
  SceneTree
    ├── World
    │   ├── NPC_001 (Node3D + Skeleton3D + AnimationTree + MeshInstance3D × N)
    │   ├── NPC_002 (...)
    │   ├── ... × 1000
    │   └── NPC_999
    ├── Terrain
    │   └── Chunk × N (VoxelTerrain × N)
    └── Player

修订后模式:
  SceneTree
    ├── RustRenderViewport (TextureRect + Viewport)
    │   └── 离屏纹理（来自 Rust wgpu 渲染的所有 NPC + 体素地形）
    ├── UI (Control 节点树)
    ├── Audio (AudioStreamPlayer 们)
    └── Player (玩家自己的 CharacterBody3D)
```

### 缺陷九：物理引擎分裂（🔴 高）

**根因**：NPC 移动决策在 Rust，物理移动在 Godot Jolt——1 帧的往返延迟。

**解决方案**：将物理也移到 Rust 侧（Rapier）。NPC 的移动决策和物理碰撞在同一帧内闭环。Godot 侧只处理玩家的物理（Jolt）。

## 二、修订后的架构

### 总体结构

```
┌─────────────────────────────────────────────────────────────┐
│                    Rust 核心（独立进程或同进程动态库）         │
│                                                             │
│  ┌───────────────────┐  ┌──────────────────────────────┐    │
│  │   模拟引擎 (Flecs)  │  │   渲染引擎 (wgpu)              │    │
│  │                   │  │                              │    │
│  │ · NPC 心智        │  │ · Transvoxel 体素 Mesh 生成    │    │
│  │ · GOAP 规划       │  │ · GPU 骨骼动画 (compute)       │    │
│  │ · 世界生成         │  │ · MultiDraw 实例化渲染         │    │
│  │ · 经济模拟         │  │ · LOD 切换                    │    │
│  │ · 记忆数据库(LMDB) │  │ · 光照 + 阴影 (简单方向光)     │    │
│  │ · 物理 (Rapier)    │  │                              │    │
│  │                   │  │  输出：离屏纹理 (RGBA + Depth)  │
│  └───────────────────┘  └──────────────┬───────────────┘    │
│                                        │                    │
│                             共享内存 / 纹理句柄              │
└────────────────────────────────────────┼────────────────────┘
                                         │
┌────────────────────────────────────────┼────────────────────┐
│                          Godot 4.x 客户端                   │
│                                        │                    │
│  ┌─────────────────────────────────────▼──────────────────┐ │
│  │              RustRenderViewport                         │ │
│  │  接收 Rust 侧的离屏纹理，显示在 TextureRect 上           │ │
│  │  处理摄像机旋转/缩放                                    │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────────┐  │
│  │ UI 层     │  │ 音频      │  │ 输入      │  │ 玩家物理    │  │
│  │ Control   │  │ Audio    │  │ InputMap  │  │ Jolt       │  │
│  │ 节点树    │  │ Server   │  │           │  │ Physics    │  │
│  └──────────┘  └──────────┘  └──────────┘  └────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 关键原则

1. **NPC 和体素地形完全不在 Godot 的场景树中**。它们在 Rust 侧渲染到离屏纹理。
2. **Godot 只处理玩家自身**。玩家的 CharacterBody3D、摄像机、UI、音频——这些在 Godot 侧。
3. **Rust 侧的渲染输出是一张纹理 + 深度图**。Godot 在 Viewport 中显示这张纹理。
4. **鼠标点击/射线检测**：InputMap 将屏幕坐标传给 Rust 侧，Rust 侧在自己的空间数据结构中做 hit test。
5. **UI 叠加**：所有 UI（背包、对话、HUD）是 Godot Control 节点，渲染在离屏纹理之上。

## 四、与原方案的关键差异

### 4.1 动画：从 CPU 到 GPU

```
原方案：
  Godot AnimationTree → 1000 个实例 → CPU 瓶颈

修订方案：
  Rust 侧加载动画数据 → GPU compute shader 批量计算骨骼矩阵 → GPU vertex shader 蒙皮
  → 1000 个 NPC 的动画在 ~1ms 内完成（GPU bound）
```

资源格式：GLTF 的动画数据在 Rust 侧解析 → 关键帧数据上传到 `StorageBuffer` → compute shader 访问

### 4.2 体素：从社区插件到自建

```
原方案：
  Zylann Voxel Tools → 在 Godot 场景树中管理 chunk → 依赖社区维护

修订方案：
  Rust 侧自建 Transvoxel → wgpu compute shader 提取等值面 → MultiDraw 渲染
  → 完全控制体素管线的每个阶段
```

放弃 Voxel Tools 的原因：
1. 它依赖 Godot 的场景树管理 chunk——这是缺陷三的放大器
2. WoWorld 对体素的修改需求（100K NPC 的挖掘/建造）远超 Voxel Tools 的设计假设
3. 自建 Transvoxel 在 Rust 侧——可以与世界生成、NPC 行为、物理在同一个 ECS 世界中交互

### 4.3 渲染集成：从 GDExtension 数据同步到共享纹理

```
原方案：
  Rust struct → bincode → GDExtension → Variant → Node3D
  每帧 1000 次序列化 + Godot 侧 GC 压力

修订方案：
  Rust wgpu 渲染到离屏纹理 → 纹理句柄传给 Godot → TextureRect 显示
  零数据拷贝（纹理在 GPU 内存中共享）
```

这是最大胆的改变。它意味着：
- Godot 看不到任何 NPC 的 Node3D——它只知道"有一张纹理从 Rust 侧传过来"
- 所有的 NPC 渲染、culling、LOD 切换在 Rust 侧完成
- Godot 处理摄像机的旋转/缩放 → 传给 Rust 侧更新 View 矩阵

### 4.4 物理：Jolt → Rapier (for NPC)

```
原方案：
  Rust AI → Godot Jolt → 碰撞回传 → Rust AI（1 帧延迟）

修订方案：
  Rust AI → Rust Rapier → 碰撞立即反馈 → Rust AI（同一帧）
  玩家物理仍在 Godot Jolt（因为玩家在 Godot 的 Node 树中）
```

NPC-NPC 碰撞和 NPC-地形碰撞在 Rust Rapier 中处理。Player-NPC 碰撞需要通过 Rapier-Jolt 之间的接口处理。

## 五、新增风险

这个修订方案消除了原方案的致命缺陷，但它引入了新的风险：

| 风险 | 等级 | 缓解 |
|------|------|------|
| **自建 wgpu 渲染器的工作量** | 🔴 高 | 分阶段：Phase 0 先用 Godot 渲染（快速原型）；Phase 3+ 再迁移到 wgpu |
| **GPU 驱动的动画调试困难** | 🟡 中 | 在模拟核心中保留 CPU 端动画的验证路径（慢但可调试） |
| **Rapier 的 NPC 角色控制器不够成熟** | 🟡 中 | Rapier 的 character controller 不如 Jolt 成熟；需要额外的打磨 |
| **GLTF 资产的自定义加载** | 🟡 中 | `gltf` crate 成熟，但材质/着色器绑定需要手动配置 |
| **共享纹理的平台兼容性** | 🟡 中 | Windows 上通过 DXGI/D3D 共享纹理可行；Linux 和 macOS 需要各自的实现 |
| **Godot 侧失去 NPC 的可视化编辑** | 🟢 低 | NPC 的渲染本来就不适合可视化编辑（100K 个 NPC 无法手动摆放） |

## 六、渐进式实施路径

修订方案不需要一次性实现。可以分阶段渐进迁移：

```
Phase 0 (当前): Godot 全栈原型
  - 1 个 NPC + 简单地形在 Godot 中运行
  - 验证游戏循环的基本概念

Phase 1: Rust 模拟核心 + Godot 渲染（原方案）
  - NPC 心智在 Rust 中，Godot 做渲染和动画
  - 验证 Rust↔Godot 通信
  - ⚠️ 在这个阶段会遇到动画瓶颈（~100-200 L1 NPC 时 AnimationTree 开始吃力）

Phase 2: GPU 动画
  - 在 Rust 侧实现 compute shader 骨骼动画
  - Godot 侧不再使用 AnimationTree for NPC——改为接收动画纹理
  - 验证 1000 个 NPC 的动画性能

Phase 3: 自建体素渲染
  - 在 Rust 侧实现 Transvoxel + wgpu 渲染
  - 放弃 Godot Voxel Tools
  - 验证 10000 个 chunk 的流式加载 + 渲染

Phase 4: Rust 侧物理
  - Rapier 物理集成
  - 替代 Godot Jolt for NPC
  - 验证 1000 个 NPC 同时移动的碰撞性能

Phase 5: 完整 wgpu 渲染前端
  - Godot 只保留 UI + 音频 + 输入
  - 所有 3D 渲染（NPC + 体素 + 建筑）在 Rust wgpu 侧
```

**每个 Phase 之间可以停顿、可以回溯。这不是"全有或全无"的迁移。**

## 七、最终技术栈总结

```
语言:      Rust (stable)
ECS:       Flecs (via flecs-rs)
数据库:    LMDB
渲染:      wgpu (NPC + 体素) + Godot Forward+ (玩家 + UI 合成)
动画:      GPU compute shader (骨骼蒙皮)
物理:      Rapier (NPC) + Jolt (玩家)
音频:      Godot AudioServer
UI:        Godot Control 节点
输入:      Godot InputMap
资产:      GLTF (Rust gltf crate 加载) + Blender
序列化:    serde + bincode
并发:      rayon + tokio
构建:      Cargo workspace + Godot Editor
```

**这是为 3-6 年开发周期、Solo 开发者、100K NPC 目标设计的"诚实"技术栈。**

它承认了原 Rust+Godot 方案的致命缺陷（动画规模化、Node 架构开销、体素插件依赖），并在 Rust 侧解决了它们。它也诚实地面对了修订方案的新增风险（自建渲染器的工作量），并提供了渐进式路径来管理这些风险。

**最关键的设计决定**：不是"选择 Godot 还是选择 Rust"——而是"把最需要性能和控制力的系统（NPC、动画、体素、物理）从 Godot 中剥离到 Rust，把 Godot 保留给它最擅长的事情（UI、音频、输入）"。
