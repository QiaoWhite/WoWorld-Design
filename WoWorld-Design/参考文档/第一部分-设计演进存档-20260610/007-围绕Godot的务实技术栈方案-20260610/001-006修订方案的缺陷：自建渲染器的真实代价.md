# 001-006 修订方案的缺陷：自建渲染器的真实代价

> 对 `006-最终修订技术栈方案` 提出的"Rust wgpu 渲染 + Godot 仅做 UI"方案进行诚实评估

---

## 〇、006 方案解决了什么

006 正确地识别了四个致命/高危缺陷——动画规模化、Node 架构开销、体素插件依赖、物理分裂——并提出了技术上正确的解决方案。**技术上正确，不等于工程上可行。**

对于一个 Solo 开发者而言，"自建 wgpu 渲染器 + GPU compute shader 动画管线 + 自建 Transvoxel + Rapier 物理"的工作量——在最乐观的估计下——相当于 **12-18 个月的全职工作**，才能达到 Godot 渲染器已免费提供的功能水平（光照、阴影、材质、后处理、粒子、天空、雾……）。

---

## 一、被低估的工作量

### 1.1 渲染器不只是"把三角形画出来"

006 方案中 Rust wgpu 渲染器需要替代的功能清单：

| Godot 免费提供的 | 需要在 wgpu 中自建 | 工作量（人·月） |
|-----------------|-------------------|---------------|
| Forward+ 渲染管线 | 自建渲染管线（Forward/Deferred） | 2-3 |
| 方向光/点光/聚光 + 阴影映射 | 多种光源 + shadow map 级联 | 2-3 |
| PBR 材质系统 | 自建 PBR shader + 材质参数 | 1-2 |
| 天空盒 + 大气散射 | 天空渲染 | 0.5-1 |
| 后处理（bloom、tonemap、SSAO） | 后处理管线 | 1-2 |
| 粒子系统（雨、雪、魔法特效） | 粒子系统 | 1-2 |
| 雾 + 远裁 | 简单 | 0.5 |
| 多平台支持（D3D12/Vulkan/Metal） | wgpu 已处理，但需测试 | 0.5-1 |
| 调试视图（线框、法线、光照） | 自建 | 0.5 |
| LOD 切换 (Mesh LOD) | 自建 | 1 |
| **合计** | | **10-16 人·月** |

> 一个 Solo 开发者花 10-16 个月搭一个渲染器，而这段期间游戏本身毫无进展。这不是"开发游戏"，这是"开发半个游戏引擎"。

### 1.2 GPU Compute Shader 动画管线的复杂

006 方案中 NPC 动画在 GPU 上完成。这需要：

1. **GLTF 动画数据解析**：骨骼层级、关键帧、插值类型、逆绑定矩阵——全部在 Rust 侧解析和存储
2. **动画状态机**：Godot AnimationTree 的 BlendSpace、状态转换、过渡混合——需要在 Rust 侧重新实现
3. **动画混合**：行走+跑步混合、上半身覆盖（边走路边挥手）——这些在 Godot 中是内置的
4. **Compute Shader**：骨骼矩阵计算在 GPU 上批量完成——shader 编写、调试、性能调优

**工作量估算**：3-6 人·月。这是一个专门的"技术动画程序员"的全职工作。

### 1.3 自建 Transvoxel 的隐藏复杂性

Zylann Voxel Tools 用了多年时间解决这些非 obvious 的问题：

- **不同 LOD 之间的三角形接缝**：Transvoxel 论文中的过渡单元格方案需要精确实现
- **碰撞体的增量更新**：地形被修改后，只重新烘焙被修改的 chunk 的碰撞
- **导航网格的重新烘焙**：NPC 寻路需要最新的地形碰撞信息
- **SDF 操作的数值稳定性**：大尺度 SDF 修改可能导致三角形翻转或退化
- **流式加载的优先级**：远处的 chunk 应该以更低的优先级生成

**工作量估算**：4-8 人·月。Transvoxel 论文只有 60 页——实现它的生产级代码可能需要数千行。

### 1.4 共享纹理的平台兼容噩梦

006 方案中 Rust wgpu → Godot 通过共享 GPU 纹理通信：

- **Windows**：D3D11 和 D3D12 的纹理共享机制不同。需要使用 `ID3D11Texture2D` 的共享句柄或 D3D12 的 shared heap
- **Linux**：Vulkan 的 `VK_KHR_external_memory` 扩展。Godot 的 Vulkan 后端需要与 wgpu 的 Vulkan 后端协商纹理格式
- **macOS**：Metal 的 `MTLSharedTexture`。但 Godot 在 macOS 上默认使用 Vulkan (MoltenVK)……这又多了一层转换

每个平台需要独立的代码路径。这在 CI 中测试、在用户机器上调试——是一项持续性的维护负担。

## 二、006 方案错失的 Godot 优势

### 2.1 RenderingDevice——Godot 自带的"低级渲染 API"

Godot 4.x 提供了 `RenderingDevice` 类——它给开发者直接访问 Vulkan/D3D12 的能力，同时保留在 Godot 的渲染管线内。

**这意味着你不需要 wgpu 就能做到 006 方案中 80% 的事情：**
- Compute shader → `RenderingDevice.compute_list_add_barrier()` + dispatch
- 自定义 buffer 上传 → `RenderingDevice.buffer_update()`
- GPU 驱动的间接绘制 → 可以做到

但文档很少，社区示例稀缺。这是一个"强大的但没人用的功能"。

### 2.2 MultiMesh——Godot 内置的批量渲染

`MultiMeshInstance3D` 用一个 draw call 渲染成千上万个实例。每个实例有自己的 Transform，可以通过 `MultiMesh.buffer` 设置自定义的 per-instance 数据。

结合自定义 shader——可以在 shader 中读取 per-instance 的动画状态并做 GPU skinning。**这消除了创建 1000 个 Skeleton3D 节点的需求。**

### 2.3 视觉着色器——不需要写 GLSL

Godot 的 Visual Shader 编辑器可以创建自定义的 GPU skinning shader——通过可视化节点连接。虽然不是必需的（手写 Godot shading language 也完全可以），但对于 Solo 开发者的生产力有显著帮助。

## 三、诚实的总成本估算

| 006 方案的新增模块 | 乐观估计（人·月） | 悲观估计（人·月） |
|-------------------|-----------------|-----------------|
| wgpu 渲染器基础 | 6 | 12 |
| GPU 动画管线 | 3 | 6 |
| 自建 Transvoxel | 4 | 8 |
| Rapier 物理集成 | 1 | 2 |
| 共享纹理平台适配 | 1 | 3 |
| GLTF 加载 + 材质系统 | 1 | 2 |
| 后处理/特效 | 1 | 3 |
| **总计** | **17** | **36** |

对于一个每周工作 40 小时的 Solo 开发者：
- 乐观：~8 个月的全部工作时间
- 悲观：~18 个月的全部工作时间

而这段时间内——**你一行 NPC 心智代码都不会写，一个涌现叙事场景都不会产生，一个世界都不会生成。** 你全在写基础设施。

这就是"技术上正确"和"工程上可行"之间的鸿沟。

## 四、006 方案何时是正确的

自建渲染器在以下情况下是正确的选择：

1. **你有一个 3-5 人的引擎团队**，可以将渲染器和游戏逻辑并行开发
2. **你的渲染需求极为特殊**，通用引擎无法满足（如特殊的全局光照方案、非欧几里得几何）
3. **你已经花了 5 年以上时间在自建引擎上**，它已经是你的生产力工具

对于 WoWorld——一个 Solo 项目，渲染需求是"低多边形 + 像素纹理 + 方向光"——上述条件都不成立。Godot 的 Forward+ 渲染器对此绰绰有余。

---

## 五、结论：需要一个 Godot 优先的务实方案

006 的方向是正确的——Rust 应该负责计算密集的模拟。但 **Rust 不应该负责渲染**。渲染应该留在 Godot 内，通过更聪明的方式使用 Godot 已经提供的功能。

**核心问题不是"Godot 的渲染器不够好"——而是"Godot 的动画/实体管理方式不适合海量 NPC"。**

解决这个问题的正确方式不是把渲染器搬到 Rust——而是**在 Godot 的渲染管线内，换一种方式管理海量实体**：用 MultiMesh 批量渲染、用自定义 shader 做 GPU 动画、用 RenderingDevice 做高级优化。Godot 提供了这些工具——我们只是没用好它们。
