# 001-LLM 辅助下的 006 方案可行性重评估

> 前提：开发者大量使用 LLM（Claude、GPT 等）生成 Rust/wgpu/shader 代码  
> 核心问题：当"写代码的速度"不再是瓶颈时，006 方案的自建渲染器路线是否变得可行？

---

## 〇、006 方案的真正瓶颈是什么

006 方案（Rust wgpu 渲染 + Godot 仅 UI）被否决的主要原因是**工作量**——自建渲染器需要 17~36 人·月的全职工作。但这 17~36 人·月可以分解为两类劳动：

| 劳动类型 | 占比 | 特征 | LLM 能帮多少？ |
|---------|------|------|--------------|
| **知识型** | ~40% | 需要理解 wgpu/Transvoxel/GPU skinning 的原理 | ⭐⭐⭐ 能教、能解释、能生成原型 |
| **搬砖型** | ~60% | boilerplate、管线配置、shader 变体、平台适配 | ⭐⭐⭐⭐⭐ 极其擅长 |

如果 LLM 能吃掉 60% 的搬砖型劳动 + 加速 40% 的知识型劳动，006 方案的实际人力成本可能从 **17~36 人·月 降至 5~10 人·月**。

这不是一个小数字——它意味着一个 Solo 开发者有可能在 6~12 个月的兼职时间内完成自建渲染器。

## 一、LLM 具体能做什么（按模块评估）

### 1.1 wgpu 渲染器基础

**LLM 擅长度**：⭐⭐⭐⭐⭐

wgpu 的管线配置是典型的"模板化代码"——创建 Device、设置 SwapChain、配置 RenderPipeline、编写 ShaderModule。LLM 对这些模式的掌握非常成熟。

```rust
// 这种代码 LLM 可以秒出：
let render_pipeline = device.create_render_pipeline(&wgpu::RenderPipelineDescriptor {
    label: Some("NPC Render Pipeline"),
    layout: Some(&pipeline_layout),
    vertex: wgpu::VertexState {
        module: &shader,
        entry_point: "vs_main",
        buffers: &[NpcVertex::desc(), NpcInstanceData::desc()],
    },
    fragment: Some(wgpu::FragmentState {
        module: &shader,
        entry_point: "fs_main",
        targets: &[Some(wgpu::ColorTargetState {
            format: config.format,
            blend: Some(wgpu::BlendState::ALPHA_BLENDING),
            write_mask: wgpu::ColorWrites::ALL,
        })],
    }),
    // ... LLM 可以完整生成 100+ 行的配置代码，零错误
});
```

**"但是我不知道这些参数是什么意思"**——这是关键。如果开发者对 wgpu 一无所知，LLM 生成的代码是一个黑箱。Debug 时你需要理解它。所以 LLM 不能替代**理解**——它能替代的是**打字**。

### 1.2 Transvoxel 实现

**LLM 擅长度**：⭐⭐⭐⭐

Transvoxel 算法有公开论文（Lengyel, 2010），其中的查找表、过渡单元格逻辑、LOD 接缝处理——LLM 可以生成一个可工作的实现。因为它是**算法**而非**黑魔法**。

```
开发者：实现 Transvoxel 的等值面提取。输入是 3D 密度场，输出是三角形列表。
        需要支持 LOD 0-3，过渡单元格使用 Lengyel 论文中的查找表。

LLM：生成 ~500 行的 Rust 实现，包括：
  - 256 种 Marching Cubes 配置的查找表
  - Transvoxel 过渡单元格的专用查找表
  - 密度场采样 + 线性插值顶点定位
  - LOD 级别之间的接缝处理
  - 三角形索引生成
```

**风险**：性能可能不是最优。需要开发者做 profiling 和调优。但"能工作"到"高性能"之间的差距，远比"从零写"到"能工作"之间的差距小。

### 1.3 GPU Compute Shader 动画

**LLM 擅长度**：⭐⭐⭐

WGSL（wgpu 的 shader 语言）比较新，LLM 的训练数据不如 GLSL/HLSL 丰富。但骨骼蒙皮的计算逻辑是标准化的。

```wgsl
// LLM 可以生成这种标准 shader：
@group(0) @binding(0) var<storage> bone_matrices: array<mat4x4<f32>>;
@group(0) @binding(1) var<storage> instance_data: array<NpcInstance>;

@vertex
fn vs_main(
    @location(0) position: vec3<f32>,
    @location(1) normal: vec3<f32>,
    @location(2) bone_indices: vec4<u32>,
    @location(3) bone_weights: vec4<f32>,
    @builtin(instance_index) instance_idx: u32,
) -> VertexOutput {
    let npc = &instance_data[instance_idx];
    let bone_offset = npc.bone_matrix_offset;
    
    // GPU skinning
    var skinned_pos = vec4<f32>(0.0);
    var skinned_normal = vec3<f32>(0.0);
    for (var i = 0u; i < 4u; i++) {
        let matrix = bone_matrices[bone_offset + bone_indices[i]];
        skinned_pos += matrix * vec4<f32>(position, 1.0) * bone_weights[i];
        skinned_normal += (matrix * vec4<f32>(normal, 0.0)).xyz * bone_weights[i];
    }
    // ...
}
```

**关键难点 LLM 帮不了的**：
- 动画状态机的逻辑需要开发者自己定义
- 动画混合（blending）的权重和过渡需要设计
- 性能调优（occupancy、register pressure）需要理解 GPU 架构

### 1.4 跨平台纹理共享

**LLM 擅长度**：⭐

这是最危险的部分。D3D11/D3D12/Vulkan/Metal 之间的纹理共享机制各不相同，代码路径高度平台相关，且错误表现为"黑屏"或"崩溃"——几乎没有可调试的中间状态。LLM 对此几乎没有可靠的训练数据。

**建议**：不要跨进程共享纹理。改为在**同一进程**中使用 GDExtension——Rust 编译为动态库，被 Godot 加载，Rust 侧用 wgpu 渲染到一个离屏纹理，然后通过某种方式传给 Godot。或者更简单的——Rust 侧渲染到像素 buffer，Godot 侧上传到 `ImageTexture`。这多了一次 GPU→CPU→GPU 的拷贝——但对于低多边形风格，每帧拷贝一张 1920×1080 的 RGBA 纹理也就是 ~8MB，在现代硬件上 < 0.5ms。

## 二、LLM 辅助下的工作量重估

| 006 方案模块 | 纯手工（人·月） | LLM 辅助（人·月） | LLM 帮了什么 |
|------------|---------------|-----------------|------------|
| wgpu 渲染器基础 | 6~12 | **1.5~3** | 管线配置、shader 模板、顶点格式 90% 由 LLM 生成 |
| GPU 动画管线 | 3~6 | **1~2** | 骨骼矩阵计算、WGSL shader、buffer 管理——LLM 生成 |
| Transvoxel | 4~8 | **1~2** | LLM 从论文生成完整实现，开发者调优 |
| 物理集成 (Rapier) | 1~2 | **0.5** | API 调用是标准模式 |
| 纹理共享 | 1~3 | **0.5** | 放弃共享纹理，改用 buffer 拷贝（简单但可行的替代方案） |
| GLTF 加载 | 1~2 | **0.5** | `gltf` crate 已是成熟方案，LLM 生成集成代码 |
| 后处理 | 1~3 | **0.5** | 基本 bloom + tonemap 有大量开源参考 |
| **总计** | **17~36** | **5.5~10** | |
| **Solo 兼职（20h/周）** | 20~43 个月 | **6~12 个月** | |

## 三、"AI 生成代码"的隐性成本

### 3.1 理解的债务

LLM 生成了 500 行 Transvoxel 代码，它能工作。三个月后出现了一个奇怪的三角形闪烁 bug——你需要 debug 这段代码。但你没有亲手写过它。你需要先花几小时到几天**理解** LLM 的输出，才能开始 debug。

**对策**：要求 LLM 生成代码时附带**详细的注释 + 引用来源（论文页码）**。每次接受 LLM 生成的模块后，立即手写一个最小测试案例来理解它的行为。这不是浪费时间——这是在偿还理解债务。

### 3.2 调试的盲区

GPU 代码的调试本身就困难（Compute Shader 没有 printf）。如果代码是 LLM 生成的，你甚至不知道"正确的行为应该是什么"——因为你对算法的理解停留在 LLM 的解释层面。

**对策**：
- 在 CPU 端保留所有 GPU 算法的验证路径（CPU 计算结果应与 GPU 一致）
- 用 RenderDoc 捕获 GPU 帧——即使你看不懂 shader 的中间值，至少能看到输入输出
- 极小步迭代——每次只让 LLM 改 5-10 行，立即验证

### 3.3 架构的一致性

十个 LLM 会话生成的十个模块，放在一起不一定能协作。它们的接口风格、错误处理模式、命名约定可能完全不同。

**对策**：
- 在每个 LLM 会话开始时先给出"项目约定"（命名规范、错误处理策略、接口风格）
- 所有模块通过明确定义的 trait 接口通信
- 人工 code review 的重点不是"有没有 bug"——LLM 在这方面已经够好了——而是"接口是否与项目的其他部分一致"

## 四、结论：LLM 让 006 方案从"不可行"变为"高风险可行"

| | 纯手工 006 | LLM 辅助 006 | 007 方案（Godot 优先） |
|---|---|---|---|
| 工作量 | 17~36 人·月 | **5~10 人·月** | 3~6 人·月 |
| Solo 兼职 | 20~43 个月 | **6~12 个月** | 4~7 个月 |
| 技术风险 | 极高 | 高（理解债务+调试盲区） | 中 |
| 3 个月能玩吗 | 不能 | 不能 | 能 |
| 长期控制力 | 极高（全自建） | 极高（但代码不完全是你写的） | 中（依赖 Godot） |
| 推荐度 | ❌ | 🟡 有条件的推荐 | ✅ 当前推荐 |

**LLM 辅助下的 006 方案适合以下条件**：
1. 开发者愿意**花时间理解** LLM 生成的代码，而不是盲目接受
2. 开发者有基本的 GPU 编程概念（管线、buffer、shader 类型）
3. 接受前 6 个月可能看不到"游戏内容"——全在搭基础设施
4. 愿意接受"这段代码能工作但我没有完全理解它"的技术债务

**对于 WoWorld**——如果开发者对"完全控制渲染管线"有强烈的执念，并且已经熟练使用 LLM 进行 Rust 开发，006 路线在 LLM 辅助下变得可行。但 007 路线（Godot 优先）仍然是更安全、更快速出成果的选择。
