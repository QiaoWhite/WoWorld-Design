# 002-理念与实践的平衡：在 007 方案下修正设计文档

> 核心矛盾：设计文档表达了"创建另一个人世"的极致野心——100K 完整 NPC、全球多元文化、涌现叙事。  
> 但 Solo 开发者的现实是：Phase 1 只有 1 个胶囊体在找苹果吃。  
> 如何在保持愿景的同时，让文档体系支撑务实的渐进式开发？

---

## 〇、问题的本质

当前设计文档存在一个结构性的张力：

**设计文档说**："每个 NPC 是一个完整的人——有 2000 条记忆、大五人格、30 种复合情绪、GOAP 规划、社会认知……"

**开发者明天要做的**："让一个胶囊体检测到饥饿值下降，然后向最近的苹果移动。"

这两者之间的距离是巨大的。如果文档体系不能弥合这个距离，开发者会陷入两种困境之一：
- **困境 A**：被宏伟愿景压垮，觉得"现在做的这点东西跟愿景比起来算什么"，失去动力
- **困境 B**：无视愿景，全力做眼前的小功能，做着做着发现架构支撑不了扩展，推倒重来

---

## 一、诊断：设计文档缺少"渐进层"

当前文档体系是一个**单层结构**：

```
总设计草稿 → NPC活人感开发文档（完整规格）→ 性能优化分析 → ...
```

所有文档都在描述"完整体的 WoWorld 应该是什么样的"。没有文档回答：
- "第一个可以跑的原型应该是什么样的？"
- "从 1 个 NPC 到 100 个 NPC，系统架构需要变什么？"
- "哪些设计决策可以在 Phase 1 推迟？哪些必须从第一天就正确？"

### 解决：引入"愿景层 → 规格层 → 实施层"三层结构

```
愿景层（已存在）：
  总设计草稿、创意.md、欢迎.md
  → 回答：这个世界应该是什么样的？

规格层（已存在）：
  NPC活人感开发文档、技术架构.md、性能优化分析
  → 回答：完整的系统应该如何设计？

实施层（需要新建）：
  Phase 1 最小原型规格、Phase 2 扩展规格、...
  → 回答：我现在应该做什么？
```

## 二、具体修改建议

### 2.1 新建文档：`开发阶段/当前实施目标.md`

这份文档是"现在应该做什么"的单一真相来源。它应该是活的——随着开发进展不断更新。

```markdown
# 当前实施目标

> 最后更新：2026-06-10
> 当前阶段：最小生存循环原型

## 我在做什么

创造一个体素世界，放入一个会饿的胶囊体 NPC，让它找食物。

## 当前目标（本次工作块）

- [ ] Godot 4.6.2 项目初始化
- [ ] 1 个 VoxelTerrain chunk（32×32×4）
- [ ] 1 个胶囊体 NPC
  - [ ] 有 hunger 值（每游戏分钟 +1）
  - [ ] hunger > 80 时触发"找食物"行为
  - [ ] 向场景中预设的苹果位置移动
  - [ ] 到达后 hunger 归零
- [ ] 玩家角色（可移动的第三人称胶囊体 + 摄像机）

## 这次不做什么

- ❌ 不做 Rust 模拟核心（先用 GDScript 原型）
- ❌ 不做 GOAP 规划器（简单的 if 判断足够）
- ❌ 不做情绪系统
- ❌ 不做记忆系统
- ❌ 不做 UI（用 Godot Label 显示 hunger 值即可）
- ❌ 不做多个 NPC

## 下次可能做什么（待本次完成后决定）

- 将 NPC 逻辑移到 Rust GDExtension
- 添加 10-20 个 NPC
- 实现基础的情绪引擎（PAD 三维轴）
```

### 2.2 在 `总设计草稿.md` §〇 增加"渐进承诺"

在当前 §〇 末尾增加一个小节，声明设计哲学中的渐进原则：

```markdown
### 关于野心与渐进

本文档描述的是 WoWorld 的**完整形态**——它最终应该成为的样子。
但不是"现在就要做出来"的样子。

开发遵循一个严格的原则：
- **每个工作块产出一个可运行的东西**。不是文档，不是设计图，是一个可以打开的、可以看到变化的程序。
- **每个系统从最简单的可行版本开始**。情绪引擎从 PAD 三维轴开始，不是 30 种复合情绪。记忆系统从 10 条事件记录开始，不是 2000 条。
- **复杂性是长出来的，不是设计出来的**。只有当简单版本确实不够用时，才添加下一层复杂度。
- **规格文档描述"天花板"，实施文档描述"下一步"**。两者之间的差距是正常的，不需要弥合——只需要诚实地标注。

如果你在阅读本文档时感到压迫感——那是正常的。这些系统描述的是长期目标。
现在要做的只是其中最小的一步。每一步走完再决定下一步。
```

### 2.3 在 NPC 文档中标注"复杂度层级"

`NPC活人感开发文档ver1.01short.md` 当前是一个"完整规格"——所有系统都在同一层级上描述。建议在每个章节开头增加复杂度层级标注：

```markdown
## 第5章：情绪引擎

> 🟢 MVP：PAD 三维轴 + 1 种情绪标签
> 🟡 进阶：8 种基本情绪 + 复合情绪
> 🔴 完整：30 种复合情绪 + 情绪感染 + 非语言表达

（然后是现有的完整规格描述……）
```

同样标注应用于：
- 记忆系统（🟢 10 条记录 → 🟡 200 条 + 遗忘 → 🔴 2000 条 + 重构 + 集中数据库）
- GOAP 规划（🟢 if-else → 🟡 简单 GOAP → 🔴 完整 GOAP + 缓存 + 长线目标）
- 社会认知（🟢 关系值衰减 → 🟡 意图识别 → 🔴 完整 B4 + 偏见 + 心智理论）

**好处**：开发者在任何时间点都知道"当前阶段我应该做到哪个复杂度"。完整规格仍然是神圣的——它描述了天花板——但标注使开发者不会在地板上被天花板吓到。

### 2.4 新建文档：`想法/开发想法草稿/开发节奏与优先级.md`

这份文档是对已删除的 `开发流程.md` 的精神替代——但它**不预设时间表和阶段顺序**。它只描述：

- 每个系统的三个复杂度层级（MVP → 进阶 → 完整）
- 系统之间的"硬依赖"（如：社会认知依赖记忆系统，记忆系统依赖 NPCData）
- **当前优先级**（由开发者根据当下判断动态调整）
- **最近完成**和**下一步考虑**的列表

关键设计：这份文档的"优先级"部分是**活的**——每次工作块结束后更新。它不是计划，是状态快照。

## 三、设计哲学中需要澄清的张力

### 3.1 "不写故事" vs "NPC 需要足够复杂才能涌现故事"

设计中存在一个隐含的循环依赖：
- 涌现有趣的故事 ← 需要 NPC 足够复杂 ← 需要实现完整的心智系统 ← 需要数年开发
- 但"涌现叙事"作为核心卖点，需要尽早验证——如果开发了 2 年发现 NPC 参数设置无法产生有趣的故事，整个项目的基础就动摇了

**建议**：在 Phase 2（20 个 NPC 运行基本情绪+决策）就开始做**微型涌现验证**。20 个 NPC 在有限规则下——他们之间会发生意外互动吗？会形成小型社交动态吗？如果 20 个 NPC 都无法产生任何有趣的行为，那 100K 个也不会。

### 3.2 "100K NPC 全部模拟" vs 实际是分层模拟

设计文档说"全部尽数模拟"，但实际上 90% 的 NPC 在 L3 层——他们每游戏日只接受一次批量统计推演。一个 L3 NPC 的"模拟"和玩家直观理解的"模拟"（每个 NPC 有自己的想法和行动）相去甚远。

**建议**：在文档中诚实地定义"模拟"的含义：
- L1（~1000 个）：全模拟——实时心智、可互动、可见
- L2（~10000 个）：轻量模拟——实时但简化、可见但不可深度互动
- L3（~89000 个）：统计存在——确保数据的连续性，但不运行实时心智

"全部模拟"= "所有 NPC 的数据在存档中存在、被维护、随世界变化而变化"。不是"所有 NPC 都在实时思考"。这种诚实对管理玩家期望至关重要。

### 3.3 "全球多元文化" vs 文化系统的实现成本

"全球多元文化自由融入"是极有吸引力的愿景。但文化系统的完整实现（文化种子参数、漂移/分化/融合、对建筑/服饰/饮食/宗教/语言的全链路影响）可能需要 12~24 个月的开发时间。

**建议**：
- Phase 1-2：单一文化（如"通用中世纪奇幻"）。先把 NPC 做活，文化可以等。
- Phase 3-4：引入文化种子参数 + 2-3 种明显不同的文化变体。验证文化系统的技术可行性。
- Phase 5+：扩展至全球多元文化 + 文化演化。

在愿景文档中保持"全球多元文化"的描述——但在实施文档中诚实标注它的优先级。

## 四、平衡的核心原则

| 原则 | 说明 |
|------|------|
| **愿景不妥协，实施不跳跃** | 设计文档保持完整的雄心。实施按复杂度层级逐步推进。两者不需要一致——需要的是标注清楚"当前在哪一层"。 |
| **每个工作块验证一个假设** | 不是"实现情绪系统"——而是"验证 PAD 三维轴能否让 NPC 的行为看起来受情绪影响" |
| **先做能用眼睛验证的东西** | 文字输出 < 数值输出 < 图表输出 < 3D 场景中的可见行为。能"看到"的东西给开发者的正反馈远超数据。 |
| **删减优先级：先砍文化，再砍社交，不砍AI** | NPC 的基本自主性（饥饿→找食物→吃）是地基。情绪、文化、记忆——它们是楼层。地基不稳之前不要盖楼。 |
| **设计文档是地图，不是合同** | 地图告诉你地形，不影响你选择今天走哪条路。规格文档告诉你可以到达哪里，不规定你必须到达哪里。 |

| 🟢 | 澄清"全部模拟"的定义 | `总设计草稿.md` §4.3 + `NPC` 相关文档 |

这些修改的核心意图不是**降低愿景**——而是**在愿景和当下之间架一座桥**。
