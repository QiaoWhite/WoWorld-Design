> 将 001（LLM 辅助可行性）和 002（理念实践平衡）的结论综合为可执行的行动建议

## 〇、一个 Solo 开发者在 2026 年的独特位置

你处于一个历史上从未有过的位置：

1. **你有 LLM 作为编程伙伴**——它可以让你的代码产出速度达到传统 Solo 开发者的 3-5 倍
2. **你有 Godot 4.6**——一个成熟的、功能丰富的开源游戏引擎
3. **你有 Rust**——给你 C++ 级的性能 + 编译期的内存安全保证
4. **你有一个极其清晰的愿景**——你知道你要创建什么，不需要在方向上犹豫

同时：
1. 你只有**一个人**
2. 你的愿景的规模通常需要一个**中型团队**
3. 你的开发周期是 **3-6 年**——技术会变化，你会疲劳，动力会波动

这个位置是挑战，也是特权。以下是在这个位置上的务实行动指南。

## 一、技术路线：坚定走 007 方案，保留 006 作为远期选项

### 1.1 立即执行：Phase 1（Godot 全栈原型）

```
目标：1 个胶囊体 NPC 在体素世界中找苹果吃
技术：纯 Godot 4.6 + GDScript
时间：2-4 周
LLM 角色：辅助——生成模板代码、解释 API、debug
```

**这不是"推翻技术决策"——这是"在做出任何重大架构决策之前，先让一个东西跑起来"。**

Phase 1 的目的不是产出可扩展的代码——而是：
- 验证"胶囊体找苹果"的游戏循环是否有趣
- 在 Godot 编辑器中快速迭代
- 积累对 Godot API 的直觉
- 发现设计文档中没有覆盖的实际问题

### 1.2 近期执行：Phase 2（Rust 模拟核心集成）

```
目标：20 个 NPC 在 Rust 驱动下自主行动
技术：Rust GDExtension + Godot Node 渲染 + Godot AnimationTree
时间：2-3 个月
LLM 角色：核心——生成 Rust NPC 数据结构、情绪引擎、概率决策器的初版代码
```

这是"LLM 的最大价值区间"——数据结构定义、纯数学计算、状态机逻辑。LLM 可以生成 80% 的 Rust NPC 代码，开发者做 20% 的架构审查和集成调试。

### 1.3 中期执行：Phase 3（MultiMesh + GPU Skinning）

```
目标：200+ NPC 同时可见，稳定 60fps
技术：MultiMeshInstance3D + 自定义 shader + PhysicsServer3D API
时间：2-3 个月
LLM 角色：关键——生成 Godot shader 代码 + MultiMesh 管理逻辑
```

这是 Godot 优先方案的决定性验证——MultiMesh + GPU skinning 能否在不搬出渲染器的情况下支撑海量 NPC。如果这个验证失败（200 NPC 时 Godot 侧仍出现不可接受的性能问题），006 方案重新进入讨论。

### 1.4 远期保留：006 路线（Rust wgpu 渲染）

**不关闭这个选项。** 当且仅当以下条件同时满足时，考虑向 006 迁移：
1. Phase 3 验证失败（Godot 渲染无法支撑 200+ NPC）
2. 开发者已积累足够的 Rust + GPU 编程经验
3. LLM 辅助可以显著降低工作量（见 001 的分析）

## 二、文档体系：建立"三层结构"

### 2.1 修改清单（优先级排序）

### 2.2 文档维护铁律

1. **`当前实施目标.md` 是活的**。每次完成一个工作块后立即更新。
2. **规格文档不因实施进展而降低**。NPC 文档仍然描述完整形态——但加上层级标注。
3. **愿景文档保持不变**。`创意.md` 和 `总设计草稿.md` 的哲学表述不需要因为"现在只有 1 个胶囊体"而打折。

## 三、LLM 使用策略

### 3.1 什么交给 LLM

| 模块类型 | 交给 LLM 的比例 | 示例 |
|---------|---------------|------|
| 数据结构定义 | 90% | Rust struct、enum、序列化 derive |
| 纯算法实现 | 80% | Transvoxel、GOAP A* 搜索、噪声生成 |
| Boilerplate 代码 | 95% | wgpu 管线配置、GDExtension 注册、build.rs |
| Shader 代码 | 70% | GPU skinning、PBR 光照、后处理 |
| 测试代码 | 85% | 单元测试、集成测试的框架和边界 case |
| 性能调优 | 20% | 需要实际 profiling 数据指导 |

### 3.2 什么必须开发者亲手做

| 活动 | 原因 |
|------|------|
| 架构决策 | LLM 不理解 WoWorld 的全局约束和长期愿景 |
| 接口设计 | 模块边界的形状决定了整个项目的耦合度 |
| 性能 profiling | LLM 只能猜测瓶颈，不能测量 |
| 涌现行为观察 | 这是游戏设计的核心——没有 AI 能替代你"看 NPC 是不是够有趣" |
| 最终代码审查 | LLM 生成的代码必须经过你的理解和批准 |

### 3.3 LLM 会话管理

为 WoWorld 项目维护一个 `.claude/prompts/` 目录，存储常用的 LLM 提示词模板：

```
.claude/prompts/
├── rust-npc-struct.md       # "生成一个新的 NPC 组件"的提示词
├── godot-shader.md           # "生成 Godot shader"的提示词
├── gdextension-bridge.md     # "生成 GDExtension 桥接代码"的提示词
├── code-review.md            # "审查这段代码"的提示词
└── design-feedback.md        # "评估这个设计与 WoWorld 愿景的契合度"的提示词
```

每次 LLM 会话结束后，把**有效的提示词**保存下来。你是在训练你的 AI 搭档。

## 四、动力管理：Solo 开发 3-6 年的心理策略

### 4.1 最重要的指标不是代码行数

而是**"可展示的进展"**。每个月结束时问自己：有什么东西是我可以打开、看到变化、甚至发给朋友看的？

如果答案是没有——这个月的工作可能过于深陷于基础设施。调整下个月的优先级，确保产出可见的结果。

### 4.2 愿景是燃料，不是负担

`总设计草稿.md` 不是一本"你现在必须做到的事情清单"。它是一本**旅行指南**——告诉你前方有什么风景。当你疲惫时，读一读它，想想 100K 个活着的 NPC 在世界中生活的画面——那是你在建造的东西。

但当你开始工作时，关掉它。打开 `当前实施目标.md`。那里只有 3-5 个勾选框。

### 4.3 允许自己"做错"

Bevy 全栈也许是对的。Godot 优先也许是对的。Rust wgpu 也许最终会成为正确答案。**现在无法知道。** 

重要的是：选一个方向，走 2-3 个月，诚实地评估结果，然后调整。这不是"推翻自己"——这是"获得新信息后做出更好的决策"。参考文档 004→005→006→007→008 的演变正是这个过程。
