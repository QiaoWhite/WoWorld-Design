# 003-最终技术栈：LLM 时代边学边做的务实路径

> 综合 001（条件重设）和 002（UAF 动画架构）——在"完全知道要什么 + LLM 生成代码 + 边学边做 + UAF 参考"的新条件下给出完整技术方案

---

## 〇、核心结论

**最终推荐：C# + Godot 4.6，分三阶段渐进演进。**

```
现在 → Phase 1: GDScript 原型 (2-4 周)
       ↓
     Phase 2: C# 迁移核心逻辑 (2-3 月)
       ↓
     Phase 3: C# 优化 + GPU 动画 (3-6 月)
       ↓
     远期: 保留 C++ GDExtension 作为性能兜底选项
```

**这一路线的核心逻辑**：
- GDScript 让开发者在最短时间内看到东西动起来（即时反馈 → 学习的正循环）
- C# 在 LLM 训练数据、类型安全、性能、学习曲线上综合最优
- UAF 启发 GPU 动画在 Godot RenderingDevice 上实现——不依赖任何外部渲染器
- C++ GDExtension 保留为最终的性能安全网——当且仅当 C# + GPU 仍然不够时启用

## 二、为什么是 C#

### 2.1 之前为什么忽略了 C#

004~007 的技术栈讨论默认在 Rust/C++/GDScript 三者之间选择，C# 被排除在外。原因是之前的假设是"开发者已经有成熟的编程能力"——在这种情况下 Rust 的内存安全和 C++ 的生态优势更突出。

但条件变了。对于"在 LLM 辅助下边学边做"的开发者，C# 的综合评分最高：

### 2.2 C# 在新条件下的优势

| 优势 | 为什么对 WoWorld 重要 |
|------|---------------------|
| **LLM 训练数据极丰富** | Unity 生态数十年的 C# 游戏代码——AI 见过的 C# 远超 GDScript |
| **类型安全但温和** | 有 GC（不需要手动管理内存），有泛型、LINQ、async/await——写 NPC 逻辑比 C++ 安全，比 Rust 容易 |
| **性能足够** | NativeAOT 编译 + `Span<T>` + `stackalloc`——在热路径上接近 C++/Rust |
| **Godot 官方支持** | `Godot.NET.Sdk` 是官方维护的——不是社区 crate |
| **学习资源充足** | C# 的中文教程、书籍、视频远远多于 Rust 和 GDScript |
| **迁移路径平滑** | GDScript（原型）→ C#（主力）→ C++ GDExtension（兜底）——三步渐进，每步都是增量改进 |

### 2.3 C# 的性能边界与应对

C# 在以下场景的性能不及 Rust/C++：
- 极高频的 CPU 计算（如每帧 1000 次 GOAP A* 搜索）
- 需要精细控制内存布局的场景

**应对**：
1. **热路径优化**：使用 `Span<T>`、`stackalloc`、`ArrayPool<T>` 减少 GC 分配
2. **NativeAOT 编译**：消除 JIT 开销
3. **分帧批处理**：GOAP 规划分摊到多帧——本来就必须这样做
4. **GPU 卸载**：动画、等值面提取移到 GPU
5. **C++ GDExtension 兜底**：如果某个模块确实达到了 C# 的性能极限，单独迁移到 C++

## 四、渐进实施路径

### Phase 1：GDScript 极简原型（2-4 周）

**技术**：纯 Godot 4.6 + GDScript

**目标**：
- 1 个 VoxelTerrain chunk
- 1 个胶囊体 NPC（hunger 值 + 简单的找食物逻辑）
- 玩家角色可移动
- 基础光照 + 天空

**LLM 角色**：生成 GDScript 模板代码，解释 Godot API，debug 协助

**学习重点**：Godot 编辑器、场景系统、信号机制、GDScript 基础语法

**不做什么**：
- 不做 C#（原型阶段 GDScript 更快）
- 不做 GOAP（简单的 if 判断足够）
- 不做 GPU 动画（1 个胶囊体不需要）

### Phase 2：C# 迁移 + NPC 系统（2-3 月）

**技术**：C# (.NET 8) + Godot 4.6

**迁移策略**：
- 不是"重写"——而是"新功能用 C#，旧 GDScript 逐步替换"
- 先迁移数据结构（NPCData → C# class/struct）
- 再迁移逻辑（情绪引擎、概率决策器、GOAP 规划器）
- GDScript 保留用于场景脚本和 UI

**目标**：
- NPCData 完整结构（C# 实现）
- 情绪引擎（PAD 三维轴 + 8 基本情绪）
- 概率行为树（90% 的日常决策）
- 简单 GOAP（生存目标：进食、饮水、睡眠）
- 20 个 NPC 在场景中自主行动
- SQLite 集成——记忆存储

**LLM 角色**：生成 70-80% 的 C# NPC 代码；解释 .NET 概念；协助配置 NativeAOT

**学习重点**：C# 类型系统、LINQ、async/await、Godot C# API、SQLite

### Phase 3：GPU 动画 + 规模化（3-6 月）

**技术**：C# + Godot RenderingDevice Compute Shader + MultiMesh

**目标**：
- 实现 UAF 启发的 Pose Database（200 个姿态）
- GPU Compute Shader 姿态混合 + 骨骼矩阵计算
- MultiMeshInstance3D 批量渲染（200+ NPC）
- PhysicsServer3D API（无 Node 碰撞）
- LOD 三级动画降级（全模块 → 简化 → 无动画）

**LLM 角色**：生成 WGSL/GLSL compute shader 初版；生成 Buffer 管理代码；协助 RenderDoc 调试

**学习重点**：GPU 管线基础、Compute Shader 编写、RenderDoc 使用

### Phase 4+：扩展与优化（长期）

- Transvoxel 等值面提取 → C# 实现（从 Voxel Tools 迁移）
- NPC 数量从 200 → 1000 → 5000 → 渐近 100K
- C++ GDExtension：仅当 C# 确认达到性能边界时启用

## 五、LLM 使用策略

### 5.1 技术栈特定的 LLM 提示词

为这个技术栈准备专用的 LLM 会话启动模板：

```
# 启动新的 C# NPC 模块开发

你是一个资深 C# 游戏开发者，熟悉 Godot 4.x C# API。项目背景：
- 游戏引擎：Godot 4.6.2，使用 .NET 8 (NativeAOT)
- 项目目标：100K 活着的 NPC 的体素开放世界 RPG
- 架构原则：数据本体与表现分离、ECS 思想、GPU 计算卸载
- 代码风格：使用 struct 作为值类型、Span<T> 减少分配、async/await 异步规划
- 序列化：MessagePack
- 数据库：SQLite (Microsoft.Data.Sqlite)

请帮我实现 [具体的 NPC 子系统]：
- 输入数据格式：[描述]
- 输出数据格式：[描述]
- 与其他系统的接口：[描述]
- 性能约束：[描述]
- 复杂度层级：当前实现 MVP 级别 [具体描述 MVP 应包含什么]

先给出架构概览，再生成完整代码。
```

### 5.2 按模块独立 LLM 会话

- **每个 NPC 子系统一个独立 LLM 会话**：情绪引擎、GOAP 规划、记忆系统、关系系统
- 好处：LLM 的上下文不会被前一个模块的细节污染
- 接口文档在每个会话开始时提供

## 六、风险矩阵

| 风险 | 等级 | 缓解 |
|------|------|------|
| C# 性能不达标（1000 NPC 时） | 🟡 中 | NativeAOT + 分帧批处理 + 最终 C++ GDExtension 兜底 |
| Godot C# 社区资源少于 GDScript | 🟡 中 | LLM 可以弥补——AI 对 C# 的掌握远超 GDScript |
| RenderingDevice API 文档不足 | 🟡 中 | 通过 LLM 理解 + Godot 源码参考 |
| SQLite 写入性能不如 LMDB | 🟢 低 | WAL 模式 + 批量事务；100K NPC 规模下测试 |
| GC 导致微停顿 | 🟢 低 | NativeAOT + `Span<T>` + 对象池——游戏领域 C# 的 GC 优化已成熟 |
| 开发者 C# 学习曲线 | 🟢 低 | AI 全程辅助；C# 本身是"对新手友好的编译语言" |

## 七、一句话总结

> **GDScript 入门。C# 扎营。GPU 动画攻坚。C++ 作为最终安全网。**
>
> 这不是"最好的技术栈"——这是**"对一个在 LLM 辅助下边学边做的 Solo 开发者最可能成功的技术栈"**。
>
> 不要一开始就 Rust。不要把自己的学习曲线变成项目的瓶颈。从最简单的开始，在需要时再变复杂。LLM 是你的加速器——用得好，C# 能做的事远比想象的多。
