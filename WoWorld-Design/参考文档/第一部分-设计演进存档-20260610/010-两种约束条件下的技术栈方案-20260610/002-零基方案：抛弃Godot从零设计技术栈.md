# 002-零基方案：抛弃 Godot 从零设计技术栈

> 约束条件：无。Godot 可以抛弃。所有假设归零。  
> 核心问题：如果今天（2026 年 6 月）重新开始，一个 Solo 开发者 + LLM 辅助 + 完全知道要什么——会选择什么技术栈？

---

## 〇、从第一性原理出发

忘掉 Godot。忘掉 Rust。忘掉所有之前的分析。只问一个问题：

**WoWorld 需要一个什么样的软件？**

### 0.1 它的核心需求

1. **渲染一个体素世界**（低多边形风格，地形可被雕刻）
2. **模拟 100K 个有独立心智的 NPC**（记忆、情绪、决策、社交）
3. **生成整个世界**（地形、聚落、建筑、文化、历史）
4. **玩家可以在这个世界中自由行动和交互**
5. **存档和读档整个世界的状态**

### 0.2 它的非需求（不需要的东西）

- ❌ 不需要照片级渲染（低多边形 + 像素纹理）
- ❌ 不需要物理级光照（方向光 + 基本阴影足够）
- ❌ 不需要网络同步（单机游戏）
- ❌ 不需要多人协作场景编辑（一个人开发）
- ❌ 不需要复杂的粒子系统（非核心）
- ❌ 不需要大世界的无缝流式加载优化（初始 4km² 足够，后期渐进扩展）

### 0.3 关键约束

- **一个开发者**。任何需要团队协作的工作流不可行
- **3-6 年开发周期**。技术栈必须在这个时间框架内保持可用
- **LLM 是主要代码产出手段**。技术栈需要与 LLM 高度兼容
- **边学边做**。不能预设开发者已掌握任何特定技术

---

## 一、候选技术栈：不预设任何选项

### 方案 A：Unity + C# + DOTS

**Unity 6 (2026) + C# + Data-Oriented Technology Stack (ECS)**

| 维度 | 评估 |
|------|------|
| LLM 训练数据 | ⭐⭐⭐⭐⭐ 最多——Unity 生态数十年 |
| NPC 模拟 | C# ECS (DOTS) 可以处理 100K 实体 |
| 体素渲染 | 需要自建或社区插件——不如 Godot Voxel Tools 成熟 |
| 动画 | Unity Animation Rigging + 可自建 GPU compute |
| UI | Unity UI (uGUI/UI Toolkit) 成熟 |
| 学习曲线 | 中——编辑器比 Godot 复杂，但教程最多 |
| 费用 | 个人免费（年收入 < $200K） |
| 风险 | Unity 的" runtime fee "争议虽已撤回，但信任已受损 |

**结论**：如果 LLM 训练数据量是唯一标准，Unity 是第一选择。AI 见过的 Unity C# 代码比任何其他游戏引擎都多。但 Unity 的体素支持不如 Godot，DOTS 的学习曲线不低。

---

### 方案 B：Unreal Engine 5.6 + C++ / Blueprints

**UE5.6 + C++（模拟）+ Blueprints（胶水）**

| 维度 | 评估 |
|------|------|
| LLM 训练数据 | ⭐⭐⭐⭐ C++ 极多，但 UE C++ 特殊（反射宏、GC） |
| NPC 模拟 | Mass Entity (UE5 的 ECS) 可处理海量实体 |
| 体素渲染 | Voxel Plugin (社区) 或自建——不如 Godot 方便 |
| 动画 | **UAF——行业最先进的动画架构** |
| UI | UMG (Unreal Motion Graphics) 成熟 |
| 学习曲线 | **极高**——UE 是工业级引擎，概念密度是所有引擎中最高的 |
| 费用 | 收入 < $1M 免版税 |
| 风险 | UE 对 Solo 开发者极其沉重——编译时间、编辑器性能、概念复杂度 |

**结论**：UAF 动画管线是世界上最先进的——它是 009 方案的动画灵感来源。但 UE5 本身对 Solo 开发者来说可能是一个"杀鸡用牛刀"级别的负担。它的概念密度（Actor/Component/Pawn/Controller/GameMode/GameState/PlayerState……）意味着学习曲线的前 6 个月可能在引擎概念中挣扎，而不是在写 WoWorld 代码。

---

### 方案 C：Bevy (Rust 全栈游戏引擎)

**Bevy 0.15+ (2026) + Rust**

| 维度 | 评估 |
|------|------|
| LLM 训练数据 | ⭐⭐⭐ Rust 通用代码很多，但 Bevy 特定代码偏少 |
| NPC 模拟 | Bevy ECS 原生——模拟核心和渲染在同一 ECS 世界 |
| 体素渲染 | 需自建——但 Bevy 的 rendering pipeline 给予完全控制 |
| 动画 | 需自建——但可以参考 UAF 设计 GPU compute 动画 |
| UI | `bevy_ui` 可用但不及 Godot/Unity/UE 成熟 |
| 音频 | `bevy_audio` 基本可用 |
| 学习曲线 | 中高——Rust 本身 + Bevy 的 ECS 概念 |
| 费用 | 免费开源 |
| 风险 | Bevy 的 API 不稳定——每个大版本有 breaking change |

**结论**：如果开发周期中有足够的耐心应对 API 变化，Bevy 是"零基方案"中最令人兴奋的选择。它给了 Solo 开发者完全的控制权——不用在引擎的"正确用法"中妥协。但代价是：UI、动画、音频、资产管线——一切自建。

---

### 方案 D：纯 Rust + wgpu（自建引擎）

**Rust + wgpu + egui + rapier + rodio + gltf crate**

这是 006 方案的完整版——不借用任何游戏引擎。

| 维度 | 评估 |
|------|------|
| LLM 训练数据 | ⭐⭐ Rust + wgpu 的组合较少见 |
| 完全控制 | ⭐⭐⭐⭐⭐ 不依赖任何外部引擎 |
| 工作量 | 🔴 17-36 人·月（但 LLM 可降至 5-10） |
| 风险 | 最高——一切自建 |

**结论**：这是技术上的巅峰——完全控制。但 006/007/008 的讨论已经充分说明了代价。

---

### 方案 E：Web 技术栈 (WASM + WebGPU)

**Rust → WASM + WebGPU + 浏览器渲染**

| 维度 | 评估 |
|------|------|
| LLM 训练数据 | ⭐⭐ 较新 |
| 跨平台 | 天然——浏览器 |
| 性能 | 受限于 WASM（接近原生但不是原生） |
| 体素渲染 | WebGPU 可用——但 compute shader 支持不如原生 |
| 存档 | 浏览器本地存储——不适合数 GB 存档 |

**结论**：不适合 WoWorld。浏览器不适合需要大量本地存储和极致性能的游戏。

## 三、零基条件下的最终推荐

### 🥇 第一推荐：仍然 C# + Godot 4.6（009 方案）

即使抛弃所有预设，009 方案仍然是最优解。原因不是"已经分析过了就懒得改"——而是**当把所有选项摊开逐项对比后，Godot + C# 的综合得分仍然最高**。

核心优势：
- 对 Solo 开发者最友好（即时反馈、编辑器辅助）
- C# 的 LLM 训练数据仅次于 C++，但学习曲线远远更温和
- 体素支持是现成的（Voxel Tools）
- 可以被抛弃——但没必要

### 🥈 第二推荐：Unity 6 + C# DOTS

如果有一天 Godot 不再适合（比如 Voxel Tools 停止维护、GDExtension C# 方向改变），Unity 是最自然的替代。C# 技能可以平移过去。DOTS 的 ECS 架构与 WoWorld 的模拟理念天然契合。

### 🥉 第三推荐：Bevy (Rust)

如果开发者在 1-2 年后已经精通 Rust，并且 Bevy 的 API 已经趋于稳定——这是一个值得认真考虑的迁移方向。Bevy 的纯 ECS 架构与 WoWorld 的海量实体需求是天作之合。

## 四、为什么 UE5 不推荐给 Solo 开发者

UE5 的 UAF 是世界上最先进的动画系统——**但它不是为一个人设计的**。

- **编译时间**：C++ 项目首次编译 30 分钟 +。每次改头文件 → 增量编译 2-5 分钟。对于 LLM 辅助开发（频繁生成→编译→验证）的工作流，这是毁灭性的效率杀手。
- **概念密度**：Actor、Pawn、Character、Controller、PlayerController、AIController、GameMode、GameState、PlayerState、GameInstance、World、Level、Subsystem……这是工业级的抽象层次，为一个 100 人团队的分工合作而设计。一个 Solo 开发者需要全部理解才能有效工作。
- **编辑器性能**：UE5 编辑器本身是一个重量级应用。32GB 内存是基本要求。对于一台普通开发 PC，同时运行 UE5 编辑器 + 浏览器（查文档 + LLM 对话）+ IDE——可能内存就会紧张。
- **LLM 对 UE C++ 的理解**：UE 的 C++ 有大量的宏（`UCLASS()`、`UPROPERTY()`、`UFUNCTION()`）、自定义的 GC 系统、反射系统——这层抽象让 LLM 生成的 UE C++ 代码比标准 C++ 更容易出错。

> UE5 是一个伟大的引擎——为 100 人团队做 AAA 游戏而设计。WoWorld 是一个人 + LLM。用 UE5 就像用航空母舰送外卖——虽然它确实能把外卖送到，但靠岸的时间和油费远超外卖本身的价值。

---

## 五、为什么 Unity 是第二但不是第一

Unity + C# DOTS 是技术上的强匹配。它的 ECS (Entities) + Jobs System + Burst Compiler 可以处理海量实体。而且 LLM 对 Unity C# 的训练数据是宇宙第一。

但它不是第一的原因：

1. **体素支持**：没有像 Godot Voxel Tools 那样成熟的开箱即用方案。需要自己写或依赖社区插件——而社区体素插件都不如 Zylann 成熟。
2. **编辑器复杂度**：Unity 编辑器比 Godot 复杂。对于"边学边做"的开发者，Godot 的编辑器更轻量、更直观。
3. **信任问题**：2023 年的 runtime fee 事件虽然已经平息，但它暴露了 Unity 作为商业公司的不可预测性。对于 3-6 年的项目，这种不确定性是真实的成本。

---

## 六、零基方案的最终结论

```
如果今天是 2026 年 6 月 10 日，从零开始：

第一选择：C# + Godot 4.6（009 方案）       — 综合最优
第二选择：Unity 6 + C# DOTS                 — 体素需要自建
第三选择：Bevy (Rust)                       — 等 Bevy 再成熟一年
不推荐：  UE5 / wgpu 自建 / Web 技术栈
```

**Godot 没有被抛弃——因为经过与所有替代方案的严格对比，它仍然是最适合这个项目的引擎。**

这不是"懒得换"——这是"比了一圈发现原来的选择确实是最优的"。当一个结论经过否定之否定后仍然成立，它就是可靠的。
