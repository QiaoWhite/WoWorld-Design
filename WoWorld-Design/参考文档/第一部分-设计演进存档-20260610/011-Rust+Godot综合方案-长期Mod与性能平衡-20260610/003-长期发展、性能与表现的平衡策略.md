# 003-长期发展、性能与表现的平衡策略

> 核心矛盾：如何在 3-6 年的开发周期中保持稳定，同时不牺牲性能优化和视觉表现力？

---

## 〇、三个时间尺度的策略

```
短期（0-12 月）   →  先跑起来，建立正反馈
中期（1-3 年）    →  规模化验证，社区萌芽
长期（3-6 年）    →  性能与表现的极致平衡，Mod 生态成熟
```

---

## 一、性能预算：从第一天就分配

### 1.1 60fps 的 16.7ms 预算分配表

| 类别 | 预算 | 负责方 | 当前目标 | 远期目标 |
|------|------|--------|---------|---------|
| **Rust 模拟核心** | ≤7ms | Rust | 20 NPC | 100K NPC |
| └─ NPC 心智（情绪/决策/社交） | ≤3ms | Rust | 20 个 | 1000 L1 + 10K L2 |
| └─ GOAP 规划（分帧） | ≤2ms | Rust | 5 个/帧 | 50 个/帧 |
| └─ 世界生成（后台） | ≤1ms | Rust | 1 chunk/帧 | 3 chunk/帧 |
| └─ 骨骼矩阵计算 | ≤0.5ms | Rust | 20 个 | 1000 个 |
| └─ 存档/读档 I/O（后台线程） | 不占帧预算 | Rust | — | — |
| **Godot 渲染** | ≤8ms | Godot | ~50 Node | < 100 Node |
| └─ 体素 Mesh 渲染 | ≤2ms | Godot | 50 chunk | 50 chunk |
| └─ NPC MultiMesh 渲染 | ≤1ms | Godot | 20 个 | 1000 个 |
| └─ GPU skinning shader | ≤0.5ms | Godot | 20 骨骼 | 最大骨骼数 |
| └─ 光照 + 阴影 | ≤2ms | Godot | 方向光 | 方向光 + 少量点光 |
| └─ 后处理 | ≤1ms | Godot | Bloom | Bloom + SSAO 可选 |
| └─ UI 渲染 | ≤1ms | Godot | 基础 HUD | 完整 UI |
| └─ 玩家 + 粒子 | ≤0.5ms | Godot | 玩家 | 玩家 + 天气粒子 |
| **Godot 物理** | ≤1.7ms | Godot | 20 碰撞体 | 1000 RID 碰撞体 |
| **总计** | **≤16.7ms** | | | |

### 1.2 性能基准的持续追踪

```rust
// Rust 侧的每帧性能统计
struct FrameStats {
    frame_id: u64,
    npc_count: NpcCounts { l1: u32, l2: u32, l3: u32 },
    timings: FrameTimings {
        emotion_update_us: u64,
        decision_update_us: u64,
        goap_planning_us: u64,
        bone_matrix_us: u64,
        sync_to_godot_us: u64,
    },
    memory: MemoryStats {
        npc_data_mb: f32,
        memory_db_cache_mb: f32,
        total_mb: f32,
    },
}
```

每次重大变更后运行性能基准测试，与上次对比。如果某项指标退化 > 20%，触发性能审查。

### 1.3 性能优化的分层策略

```
Layer 1: 算法优化（ROI 最高）
  - GOAP 动作空间裁剪（减少搜索分支）
  - 记忆检索索引优化（避免全表扫描）
  - 空间查询优化（八叉树替代暴力遍历）

Layer 2: 数据结构优化
  - SoA 布局替代 AoS（缓存友好）
  - 紧凑二进制替代 JSON（存档/网络）
  - 对象池替代频繁分配

Layer 3: 并行化
  - rayon 数据并行（情绪/关系更新）
  - 后台线程（世界生成、存档）
  - GPU compute（动画骨骼矩阵、Transvoxel）

Layer 4: 硬件特性（远期）
  - SIMD 加速（glam crate 已默认启用）
  - GPU instancing（MultiMesh）
  - DirectStorage（存档 I/O 加速）
```

## 二、视觉表现：低多边形 ≠ 不好看

### 2.1 低多边形的真正美学

低多边形风格不是在"妥协"——它是一种有意为之的美学选择。关键不是多边形数量，而是：

| 维度 | 对表现力的影响 | 策略 |
|------|-------------|------|
| **光照** | ⭐⭐⭐⭐⭐ 低多边形 + 好光照 = 惊艳 | 方向光 + 柔和阴影 + 环境光遮蔽。Godot 的 SDFGI 在低多边形场景下效果极好 |
| **色彩** | ⭐⭐⭐⭐⭐ 统一的调色板比多边形数重要得多 | 定义 16-32 色的主调色板，所有资产严格遵守 |
| **材质** | ⭐⭐⭐⭐ 像素纹理 + 低多边形 = 天然的"手工感" | 256px 以下的纹理，避免平滑过滤 |
| **动画** | ⭐⭐⭐⭐⭐ 流畅的动画比多边形的数量更能传达"生命力" | GPU 驱动的模块化动画——NPC 的姿态和微动作直接传达情绪 |
| **粒子/大气** | ⭐⭐⭐⭐ 雾、飘浮粒子、光柱、风尘 | Godot 粒子系统 + 简单的 Volume Fog |
| **后处理** | ⭐⭐⭐ Bloom + Color Grading 瞬间提升质感 | Godot Environment 节点配置 |

### 2.2 "震撼力"的来源分析

什么让玩家第一眼看到 WoWorld 时说"哇"？

不是 NPC 的面部细节——因为你不会看到每个 NPC 的脸。而是：

1. **规模感**：站在山丘上，看到下方一座完整的城镇，几十个屋顶，烟囱冒着烟，NPC 在街道上走动。这个场景的多边形数量可能不到 50K——但规模感让它震撼。

2. **生机感**：不是"NPC 的皮肤材质多逼真"——而是"那个 NPC 为什么走向那个摊位？他们在交易吗？那个 NPC 为什么在跑？发生了什么？"——行为驱动的视觉叙事。

3. **光影氛围**：夕阳下金色的光洒在低多边形的屋顶上，远处的山脉在雾中若隐若现。这不需要光线追踪——Godot 的 Forward+ 可以做到。

4. **天气与时间**：雨滴打在石板上、雪花在灯笼周围飘舞、清晨的薄雾从河面升起——粒子和大气效果的成本很低，但对氛围的贡献极大。

### 2.3 性能与表现的交换空间

| 设置 | 低配置 | 标准配置 | 高配置 |
|------|--------|---------|--------|
| L1 NPC 数 | ≤50 | ≤200 | ≤500 |
| L1 骨骼数 | 20 | 40 | 50 |
| 阴影 | 仅方向光 CSM | 方向光 + 少量点光 | 全光源阴影 |
| 后处理 | 仅 Bloom | Bloom + Color Grading | + SSAO + Motion Blur |
| 粒子 | 低预算 | 中预算 | 高预算 |
| 绘制距离 | 500m | 1000m | 2000m |
| LOD 切换距离 | 缩短 | 标准 | 延长 |
| 目标帧率 | 30fps | 60fps | 120fps (VRR) |

## 三、长期稳定的技术保障

### 3.1 依赖冻结与定期评估

```
年度常规审查（每年 1 月和 7 月）:
  - 评估所有 Rust crate 的更新
  - 评估 Godot 版本的更新
  - 评估是否需要升级

紧急审查（随时）:
  - 关键安全漏洞 → 立即评估并升级
  - 依赖被废弃 → 寻找替代方案
```

### 3.2 自愈存档设计

存档格式包含版本号和迁移脚本：

```rust
struct SaveHeader {
    magic: [u8; 4],       // b"WOWO"
    version: SaveVersion,
    creation_timestamp: i64,
    mod_list: Vec<ModInfo>,
}

struct SaveVersion {
    major: u16,
    minor: u16,
    patch: u16,
}

// 存档加载时的自动迁移
fn load_save(path: &Path) -> Result<World> {
    let header = read_header(path)?;
    let mut data = read_data(path, &header)?;
    
    // 按顺序应用迁移脚本
    if header.version < SaveVersion::new(0, 2, 0) {
        data = migrate_0_1_to_0_2(data)?;
    }
    if header.version < SaveVersion::new(0, 3, 0) {
        data = migrate_0_2_to_0_3(data)?;
    }
    // ... 未来的迁移脚本追加在这里
    
    World::from_data(data)
}
```

### 3.3 API 稳定性承诺

对 Modder 的承诺：
- Rhai API 函数一旦标记为 `#[stable]`——不会在 minor 版本中移除或改变签名
- 标记为 `#[experimental]` 的 API 可能在 minor 版本中改变
- 废弃的 API 会在至少 2 个 minor 版本中保持可用（标记 `#[deprecated]`）

## 四、Godot 编辑器作为"单人团队"的杠杆

### 4.1 Godot 编辑器不是"方便"——它是"另一个人"

对 Solo 开发者来说，Godot Editor 相当于：
- 一个不用付工资的 UI 设计师（Control 节点可视化编辑）
- 一个不用付工资的动画师（AnimationTree——仅用于玩家）
- 一个不用付工资的关卡设计师（场景编辑器）
- 一个不用付工资的材质艺术家（Shader 编辑器 + Visual Shader）
- 一个不用付工资的音频工程师（AudioServer 可视化配置）

放弃 Godot = 放弃这五个人。自建 wgpu 渲染器 = 自己当这五个人。

### 4.2 哪些用编辑器，哪些用代码

| 用 Godot 编辑器 | 用 Rust 代码 |
|---------------|------------|
| UI 布局（拖拽 Control 节点） | NPC 数据定义（TOML + Rust struct） |
| 场景布景（摆放建筑位置模板） | 世界生成（Rust 噪声 + 规则） |
| 材质/Shader（可视化调试） | GPU 计算（Rust → WGSL） |
| 动画（仅玩家——AnimationTree 可视化） | NPC 动画（Pose Database + Compute） |
| 音频混合（Audio Bus 布局） | 存档/读档（Rust bincode + LMDB） |

## 五、3-6 年开发周期的心理策略

### 5.1 "完成"的定义不是"所有系统 100% 实现"

而是——

**"在每个阶段，都有一个可以运行、可以看到变化、可以分享给他人的东西。"**

- Phase 1 完成 = 1 个胶囊体在体素世界中找苹果吃。可以拍视频发 devlog。
- Phase 2 完成 = 20 个 NPC 在城镇中各自生活。可以让朋友试玩。
- Phase 3 完成 = 200 个 NPC 的 GPU 动画 + MultiMesh 渲染。可以参加 indie game showcase。
- ...
- Phase N 完成 = 100K NPC 的活的世界。Early Access 发布。

每次完成——不是"系统完成了"——是"可以展示的东西完成了"。

### 5.2 接受"有些系统可能永远不会达到完整形态"

文化系统可能永远只有 2-3 种文化变体。全球多元文化可能永远停留在"东方元素 + 西方基底"而非真正的全球覆盖。完整的社会认知（B4）可能被简化为关系值 + 意图推测。

**这不是失败。这是 Solo 开发的本质。** 愿景是北极星——不要求你真的走到它那里。它只需要挂在天上，让你的每一步都朝向正确的方向。

---

> **最终极的平衡不是"在性能和画面之间妥协"——而是让每一个性能优化都为视觉表现创造空间，让每一个视觉决策都尊重性能的边界。这是低多边形美学的天然优势：你不需要 100 万个多边形来创造震撼——你需要正确的光影、流畅的动画、和 100 个正在"生活"的 NPC。**
