# 001-011 方案与游戏理念的冲突与自身缺陷分析

> 审查基础：011 三份文档 vs. `总设计草稿.md`、`NPC活人感开发文档`、`创意.md`、`CHG-003` 设计哲学  
> 审查目的：找出冲突、缺陷、纰漏，为修正提供依据

---

## 一、与核心设计哲学的冲突

### 冲突一：Rhai 行为脚本 ≠ 涌现

**011 的设计**（002 §2.2）：
```rust
fn on_npc_idle(npc) {
    if time > 12.0 && time < 14.0 && hunger < 0.3 {
        npc.set_goal("socialize", #{ location: tavern, priority: 3 });
    }
}
```

**游戏哲学**（总设计草稿 §〇）：
> "我们定义人格参数、行动规则、情绪规律、文化传播率；具体的故事由NPC之间的局部互动以概率和统计的方式自然涌现。"

Rhai 脚本的 `on_npc_idle` 钩子本质上是**脚本化的行为**——"如果条件A则执行B"。这与"只创造规则，不创造故事"的涌现哲学直接冲突。如果 Modder 可以通过 Rhai 写 `if time == noon { go_to_tavern() }`，那 NPC 就不是"活的人"——他是"被 Modder 远程操控的木偶"。

**严重程度**：🔴 理念冲突。这动摇了 WoWorld 的核心差异化竞争力。

### 冲突二：TOML 模板的静态人格 vs 每个 NPC 的独特性

**011 的设计**（002 §1.2）：NPC 模板在 TOML 中定义了固定的人格参数（`openness = 0.35`）。

**游戏哲学**（总设计草稿 §四）：
> "每个NPC有独立的大五人格……他们的行为由概率驱动而非脚本——不可预测所以看起来聪明。"

TOML 模板是一个**平均值**——所有"铁匠"都有 openness = 0.35。但哲学要求每个 NPC 是**独特的个体**。011 没有解释 TOML 模板的值是"基准值 + 每个 NPC 随机偏移"，还是"所有铁匠完全相同"。如果是后者——10 万个铁匠 NPC 拥有完全相同的人格——这直接违背了"每个 NPC 都是独特的"的核心承诺。

**严重程度**：🟡 表述缺失。技术上可以通过"模板基准值 + 每 NPC 随机偏移"解决——但 011 没有明确说明这一机制。

### 冲突三："从第一天为 Modder 设计" vs 项目当前阶段

**011 的主张**（002 标题）："从第一天为 Modder 设计"——Rhai 引擎、TOML 管线、事件钩子系统、Mod 分发平台、VSCode 扩展、WoWorld Mod Kit CLI。

**项目现实**（CLAUDE.md）："尚无游戏代码……项目当前处于理念设计阶段。"

在连一个胶囊体都还没有在场景中移动的时候，设计 Rhai API 的 `npc.set_goal()` 函数签名——这是把马车放在马前面。Mod 支持是长期目标，但它依赖于核心系统先被实现和稳定。你无法为还不存在的系统设计稳定的 Mod API。

**严重程度**：🟡 时序问题。Mod 架构的方向正确，但实施的优先级错位。不是"从第一天写 Mod 代码"——而是"从第一天在架构中为 Mod 预留接口"。

## 二、与现有 NPC 设计文档的技术冲突

### 冲突四：Flecs ECS vs 现有 NPC 架构

**011 的设计**（001 §1.1）：模拟核心使用 Flecs ECS——NPC 的心智组件（Emotion、Physiology、Action）作为 ECS Component。

**NPC 文档的架构**（NPC活人感开发文档 §1-2）：
- NPCData 是单一 Resource/Dictionary，不是 ECS Component
- NPCManager 单例管理所有 NPC，不是 ECS World
- 分层模拟是 L1/L2/L3 基于距离触发，不是 ECS System 的查询过滤
- 记忆系统在每个 NPC 的 Dictionary 中作为嵌套数据，不是独立的数据库

将现有 NPC 架构迁移到 ECS 是**重写整个 NPC 系统**。这不仅是技术迁移——它改变了数据流、模块边界、调试方式、序列化格式。011 将 Flecs ECS 列为"基础设施" crate，但没有讨论这个迁移的成本和与现有设计的兼容性。

**严重程度**：🔴 架构冲突。要么修改 011 以适配现有 NPC 架构，要么承认 011 隐含了一个"完整重写 NPC 系统"的前提。

### 冲突五：LMDB vs NPC 文档的记忆架构

**011 的设计**（001 §1.1）：LMDB 作为记忆数据库——key-value 存储。

**NPC 文档的架构**（NPC活人感开发文档 §3 + §16）：
- 每个 NPC 的记忆是嵌在 NPCData 中的有序列表（`Vec<EventMemory>`）
- 记忆压缩（摘要化）是 per-NPC 的独立操作
- 记忆检索考虑情绪一致性、时间、冲击力——不是简单的 key 查询
- 集中记忆数据库 + per-NPC 主观视角的双层设计在 CHG-003 中提出

LMDB 可以支撑这个双层架构（事件事实库 + 主观记忆库）——但 011 没有详细说明如何从 NPC 文档中复杂的记忆查询需求（"找一条冲击力高、情绪一致、适合分享给面前这个 NPC 的记忆"）映射到 LMDB 的 key-value 操作。

**严重程度**：🟡 设计不足。方向正确，但缺少从 NPC 文档到 LMDB 实现的映射细节。

## 三、011 方案的内部缺陷

### 缺陷一：Rust CPU 骨骼矩阵 + Godot GPU skinning 的职责分裂

**011 的设计**（001 §2.1）：骨骼矩阵在 Rust CPU 侧批量计算，然后传给 Godot shader 做蒙皮。

**问题**：这在 1000 个 NPC 时会产生一次 **3.2MB 的 CPU→GPU 数据传输**（每帧）。虽然 PCIe 带宽理论上足够，但在实际应用中：
- Godot 的 `PackedByteArray` → `TextureBuffer` 的上传路径不是为"每帧更新 3.2MB"优化的
- GDExtension 的 packed array 传递有一次额外的数据拷贝
- 如果未来骨骼数从 50 增加到 70，数据量增长到 4.5MB/帧

**严重程度**：🟡 需要实测验证。如果实测表明传输耗时 < 0.5ms，此缺陷不成立。但在 Phase 2 必须做这个验证。

### 缺陷二：TOML 只适合简单数据，不适合复杂嵌套

**011 的设计**（002 §1.2）：使用 TOML 定义 NPC 模板、物品、建筑、配方。

**问题**：TOML 对简单的键值对很友好——但对于 WoWorld 的一些数据结构，它可能不够：
- WFC 建筑约束规则——需要复杂的嵌套数组和条件
- 配方表——输入输出物品的数量和条件
- 文化参数——嵌套的浮点范围和关联规则

TOML 在嵌套深度超过 3 层时可读性急剧下降。对于这些复杂数据，可能需要自定义 DSL 或使用 Rhai 表达式嵌入 TOML。

**严重程度**：🟢 低。大部分 Mod 数据是简单的键值对。对于少数复杂情况，可以混合使用 TOML + 嵌入 Rhai 表达式。

### 缺陷三：16.7ms 帧预算缺少 Rhai 脚本执行预算

**011 的设计**（003 §1.1）：详细分配了 Rust 模拟、Godot 渲染、物理的帧预算。但没有为 Rhai 脚本执行分配任何预算。

如果 50 个 Mod 每个注册了 `on_npc_idle` 钩子，而场景中有 200 个 L1 NPC——每个 NPC 的 idle 事件触发 50 个 Rhai 脚本调用。即使每个调用 0.01ms，总计 = 200 × 50 × 0.01 = 100ms——帧预算已经爆炸。

**严重程度**：🔴 如果 Mod 系统上线后才发现这个问题，性能将是灾难性的。

### 缺陷四：Pose Database 的资产来源悬空

**011 的设计**（001 §2.2）：Pose Database ~200 个姿态，从 GLTF 动画中提取。

**问题**：谁来制作这 200 个动画？Solo 开发者没有动画预算。AI 动作捕捉工具（Plask、DeepMotion）可以生成，但质量参差不齐。011 提到"LLM 生成启发式规则"——但 LLM 不能生成 3D 动画数据。

Pose Database 的"200 个姿态"如果只包含 5-10 个手工动画 + 简单的参数混合，可能看起来非常机械。如果包含 200 个高质量 AI 生成 + 手工修整的动画——制作周期可能需要 3-6 个月（仅动画资产）。

**严重程度**：🔴 资产瓶颈。技术方案可行，但制作 Pose Database 的工作量被严重低估。

### 缺陷五：Modular Rig 要求所有 NPC 共享骨骼拓扑

**011 的设计**（001 §2.1 + 009 相关描述）：模块化骨骼——下半身/上半身/头部/面部独立计算。

**问题**：这个架构假设所有 NPC 物种有相同的骨骼模块划分。但——
- 人类：双腿行走 → 下半身模块有意义
- 蛇人（远期可能）：无腿 → 下半身模块不适用
- 龙（远期可能）：四足 + 翅膀 → 完全不同的模块划分
- 飞行生物：翅膀模块——人类没有这个模块

如果要支持多个物种，要么所有物种共享同一套骨骼（通过对不使用的骨骼做恒等变换——浪费 GPU 计算），要么每个物种有独立的模块定义和 Pose Database（资产制作量 × N）。

**严重程度**：🟡 远期问题。现在的重点是"人类 NPC"——多物种是远期目标。但架构应该从第一天就预留"每个物种的 SkeletonTemplate 定义了它的模块划分"的接口。

### 缺陷六：Mod 存档兼容性过于乐观

**011 的设计**（002 §6.3）：Mod 删除 → 物品标记为"残留物品"；Mod 更新 → 提示确认兼容性。

**问题**：这低估了 Mod 对游戏状态的影响深度：
- Mod 添加的 NPC 模板在存档中产生了 500 个 NPC 实例。如果 Mod 被删除——这 500 个 NPC 怎么办？他们的心智状态、记忆、关系、财产全部在存档中。删除他们会让世界出现 500 个"人口空缺"。
- Mod 修改了世界生成参数。存档中的世界是用旧参数生成的。新加载的区块应该用新参数还是旧参数？
- Mod 添加了文化种子。这个文化在存档中已经演化了两百年，产生了大量的建筑、习俗、NPC。删除 Mod → 这个文化"消失"——它的所有痕迹怎么办？

"标记为残留"的策略只适用于**物品**——不适用于**NPC 模板、文化种子、世界生成规则**。

**严重程度**：🔴 如果 Mod 系统要真正实现 Level 4（世界规则修改），存档兼容性需要根本性的重新设计。

## 四、遗漏的重要议题

### 遗漏一：Godot 5.0 的迁移路径

011 §4.1 说"Godot 5.0 发布后观察 6-12 个月再评估迁移"——但没有讨论这对 GDExtension 意味着什么。Godot 4→5 的 GDExtension API 极可能有 breaking change。在 3-6 年的开发周期中，几乎必然经历至少一次 Godot 大版本升级。011 需要一个具体的"GDExtension 版本迁移预案"。

### 遗漏二：godot-rust 社区的 bus factor

`godot-rust` 的核心维护者数量极少（实际上主要依赖个位数的贡献者）。如果这个 crate 停止维护，011 方案中 Rust↔Godot 的整个通信层需要替代方案。一个可能的预案是：直接从 Rust 调用 GDExtension C API（绕过 `godot-rust`），或者切换到 C++ GDExtension。

### 遗漏三：Rust 编译时间对 LLM 辅助迭代的影响

011 大量依赖 LLM 生成 Rust 代码。但 Rust 的编译时间较长——首次 `cargo build` 可能需要 5-10 分钟（依赖下载 + 编译）。对于"LLM 生成代码 → 编译验证 → 修复 → 再编译"的迭代循环，编译时间是一个重要的摩擦点。使用 `sccache` + 增量编译可以缓解——但 011 没有提到这个配置。

### 遗漏四：测试策略

011 详细讨论了性能基准、帧预算、数据格式——但完全没有提到**测试**。100K NPC 的涌现行为如何验证？Mod 脚本如何在 CI 中自动测试？存档迁移脚本如何确保不会损坏数据？这在任何技术方案中都是一个重大空白。

### 遗漏五：本地化框架

Mod 系统天然是多语言的——Modder 用英文写 TOML，但玩家可能是中文/日文/韩文。011 没有讨论 Mod 数据的本地化机制。TOML 中的 NPC 名字、物品描述、对话文本如何被多语言 Mod 覆盖？
