# 002-冲突解决方案与预期结果

> 基于 001 识别的 5 个理念冲突、2 个技术冲突、6 个内部缺陷、5 个遗漏  
> 逐项给出解决方案和修正后的预期效果

---

## 一、理念冲突的解决

### 冲突一：Rhai 行为脚本 ≠ 涌现 → 改为"参数调节 API"

**解决方案**：不暴露过程式行为 API（`npc.set_goal()`、`npc.say()`）。暴露**参数调节 API**。

```
修正前（脚本化的行为——违背涌现哲学）:
  fn on_npc_idle(npc) {
      if time > 12.0 { npc.set_goal("socialize", ...); }
  }

修正后（Modder 调节涌现参数——符合涌现哲学）:
  [mod.toml 参数覆盖]
  [npc_behavior_modifiers]
  socialize_likelihood_at_noon = 1.5     # 默认 1.0，铁匠 Mod 设为 1.5
  tavern_affinity_for_craftsmen = 1.3    # 默认 1.0
  idle_curiosity_during_lunch = 1.2     # 默认 1.0
  
  # 这些浮点乘数被 Rust 侧的概率决策器读取——不改变行为逻辑，
  # 只调节倾向性。铁匠仍然由 GOAP+概率驱动，只是午饭后去酒馆的
  # 概率变成了原来的 1.5 倍。
```

**Rhai 保留但重新定位**：Rhai 仅用于 Level 4-5 的硬核 Modder——且API 被限制为参数查询和事件响应注册，而不是行为编写。游戏的核心涌现引擎（GOAP + 概率决策器）不被 Mod 绕过。

**预期结果**：Mod 社区创造的是"不同的世界风味"（这个世界的铁匠更喜欢社交），而不是"被写死的 NPC 行为"。涌现哲学得到保护。Rhai 作为 escape hatch 仍然可用——标注为"高级/可能破坏涌现"。

### 冲突二：静态 TOML 模板 → 明确"基准值 + 偏移"机制

**解决方案**：TOML 模板不是 NPC 的最终人格——是**分布的参数**。

```toml
[personality]
openness = { base = 0.35, variance = 0.15 }     # N(0.35, 0.15)
conscientiousness = { base = 0.75, variance = 0.10 }
```

每个 NPC 实例化时从分布中采样。这样同一 TOML 模板可以产出一千个不同但相似的人格。在文档中明确标注 TOML 值是"分布均值"而非"固定值"。

**预期结果**：10 万个铁匠 NPC 拥有各不相同但统计上相似的铁匠人格。符合"每个 NPC 都是独特个体"的哲学。

### 冲突三：Mod 优先级的时序错位 → "架构预留，实现延后"

**解决方案**：区分"在架构中为 Mod 预留接口"和"实现完整的 Mod 系统"。

- Phase 1（原型）：TOML 数据文件存在但没有热加载——数据在编译时嵌入
- Phase 2（模拟核心）：Rust struct 的字段标记 `#[serde]` 而非硬编码——为数据驱动留接口
- Phase 3（规模化）：实现 TOML 热加载 + Mod 文件覆盖
- Phase 4+（社区）：Rhai 引擎集成 + Mod 发布平台

**预期结果**：架构从第一天就支持数据驱动——但完整的 Mod 工具链在核心系统稳定后才上线。避免"为不存在的系统设计 API"。

## 二、技术冲突的解决

### 冲突四：Flecs ECS vs 现有 NPC 架构 → 放弃 ECS，保持现有架构

**解决方案**：不引入 Flecs ECS。使用 Rust struct + Vec + HashMap 的简单组合。

- Flecs 的优势（查询速度、System 调度）在 WoWorld 的规模下可以通过 SoA 数据布局 + rayon 并行来达到
- 现有 NPC 文档的架构（NPCManager 单例、三层 LOD、距离触发）不需要 ECS 来支撑
- 放弃 ECS 消除了与 NPC 文档最大的架构冲突
- Rust crates 选型中去掉 `flecs-rs` 和 `hecs`

**替代实现**：用 Rust 的模块系统 + 显式的 System 函数替代 ECS。编译器仍然提供隔离——每个模块是独立的 `.rs` 文件。

```
src/npc/
├── emotion.rs      // pub fn update_all(world: &mut World)
├── decision.rs     // pub fn process_batch(world: &mut World, batch: &[NpcId])
├── memory.rs       // pub fn encode(&mut World, event: Event)
└── social.rs       // pub fn spread_memories(world: &mut World)
```

**预期结果**：与 NPC 文档描述的系统架构一致。数据流清晰可追溯。LLM 不需要理解 ECS 概念就能生成代码。性能通过 SoA 布局 + rayon 达到同等水平。

### 冲突五：LMDB 映射细节缺失 → 补充查询接口设计

**解决方案**：在 Rust 侧封装一个 `MemoryDb` trait——对上层暴露 NPC 文档中定义的查询（"找一条冲击力高、情绪一致、适合分享的记忆"），内部使用 LMDB 的索引查询实现。

```rust
trait MemoryDb {
    fn encode_event(&mut self, event: EventFact) -> Result<EventId>;
    fn encode_subjective(&mut self, npc: u64, event: EventId, encoding: EmotionalEncoding) -> Result<()>;
    fn query_for_sharing(&self, npc: u64, listener: u64, current_emotion: &Emotion, limit: usize) -> Vec<MemoryRef>;
    fn compress_cold_memories(&mut self, npc: u64, older_than_days: i32) -> Result<u32>;
}
```

这个 trait 是 LMDB 无关的——未来可以切换数据库实现。在 Phase 2 实现 LMDB 版本，Phase 3+ 可以基于实际性能数据优化。

**预期结果**：NPC 文档中的记忆系统规格与 LMDB 实现之间的鸿沟被 trait 抽象桥接。上层代码不变，下层可以切换。

## 三、内部缺陷的解决

### 缺陷一：骨骼矩阵传输量 → 加入实测验证为 Phase 2 门禁

**修正**：在 Phase 2 的验收标准中加入"传输时耗 < 0.5ms"的硬性要求。如果未达标——考虑将骨骼矩阵计算也移到 GPU（Rust → WGSL compute shader → Godot 直接读取 GPU buffer，跳过 CPU 回传）。

### 缺陷二：TOML 嵌套限制 → 混合 TOML + 嵌入 Rhai 表达式

**修正**：对于简单数据用 TOML。对于复杂数据（WFC 约束、配方条件），在 TOML 中嵌入 Rhai 表达式字符串：

```toml
[recipe.legendary_sword]
inputs = [
    { item = "iron_ingot", count = 5 },
    { item = "dragon_scale", count = 2 },
]
condition = "npc.skill('blacksmithing') > 80 && world.is_full_moon()"
```

### 缺陷三：Rhai 帧预算缺失 → 硬预算 + 钩子频率限制

**修正**：

```rust
struct ModExecutionBudget {
    max_total_us: u64,        // 所有 Mod 脚本的总执行时间上限：每帧 500μs
    max_per_mod_us: u64,      // 每个 Mod 的上限：100μs
    max_hook_calls_per_frame: u32,  // 每帧最多触发 100 次钩子调用
}

// 如果超出预算：
//   - 剩余的钩子调用推迟到下一帧
//   - 连续 3 帧超预算的 Mod → 自动禁用，通知玩家
```

在 003 的帧预算表中增加一行：`Rhai Mod 脚本执行 ≤ 0.5ms`。

### 缺陷四：Pose Database 资产来源 → 分层构建策略

**修正**：
- Phase 1-2：仅 20-30 个手工基础姿态（站立/行走/跑步/坐/吃/睡/工作/社交）——足够支撑 20 个 NPC 的基本日常
- Phase 3：使用 AI 视频→3D 姿态工具批量生成至 ~100 个姿态
- Phase 4+：Mod 社区贡献动画 + 游戏内数据积累

Pose Database 的 200 个姿态是**远期目标**，不是 Phase 1 的需求。在 011 文档中明确标注每个 Phase 需要的姿态数量。

### 缺陷五：多物种骨骼拓扑 → 抽象 SkeletonTemplate 接口

**修正**：

```rust
struct SkeletonTemplate {
    species_id: u32,
    modules: Vec<ModuleDef>,
    bones: Vec<BoneDef>,
    rest_poses: Vec<Mat4>,
}

struct ModuleDef {
    name: String,               // "lower_body", "upper_body", "head", "face", "wings"
    bone_indices: Vec<u32>,     // 该模块包含的骨骼索引
    driver: ModuleDriver,       // 由什么驱动（移动/行动/注意力/情绪/自定义）
}
```

每个物种有独立的 `SkeletonTemplate`——人类、矮人、精灵可能共享同一套模板（结构相同），但龙和蛇人有自己的模板。GPU compute shader 接受 `SkeletonTemplate` 作为参数。

### 缺陷六：Mod 存档兼容性 → 分层兼容策略

**修正**：根据 Mod 的影响层级采用不同的存档兼容策略。

| Mod 类型 | 删除后 | 更新后 |
|---------|--------|--------|
| Level 1（纹理/模型替换） | 自动回退到 vanilla 资源 | 自动应用新资源 |
| Level 2（物品/模板新增） | 已有实例标记为"来源未知"——保留但不再生成新实例 | 已有实例保留旧属性的快照，新生成的用新模板 |
| Level 3（Rhai 脚本） | 脚本引用的 NPC 回退到默认行为 | 新脚本替换旧脚本 |
| Level 4（世界规则修改） | **需要新建存档**。明确提示玩家 | **不兼容**。需要新建存档 |

Level 4（修改世界生成参数、文化演化规则）的存档兼容性不可行——这是诚实。与其装作可以兼容，不如告知玩家"这个 Mod 修改了世界规则，需要新存档"。

## 四、遗漏的补充

### 遗漏一：GDExtension 迁移预案

在 011 §4.1 增加：维护一个 `gdext-abi` crate，封装所有 Godot API 调用。当 Godot 5.0 改变 GDExtension API 时，只需修改这个 crate——Rust 模拟核心代码不受影响。如果 `godot-rust` 停止维护，后备方案是直接从 Rust 调用 GDExtension 的 C API。

### 遗漏二：godot-rust bus factor

在 011 风险矩阵中增加："`godot-rust` 核心维护者 < 5 人——如果 crate 停止维护，使用 C API 直接调用的后备方案已在架构中预留。"

### 遗漏三：编译时间摩擦

在 011 开发工作流中增加：首次 `cargo build` 前配置 `sccache`。增量编译时间从 5-10 分钟降至 10-30 秒（改动单个文件）。这是 LLM 辅助迭代循环的关键优化。

### 遗漏四：测试策略

新增测试分层：
- **单元测试**：每个 NPC 模块独立（`cargo test`）——LLM 生成
- **集成测试**：10 个 NPC 在模拟核心中运行 7 个游戏日——验证没有 panic、记忆不泄漏、关系不爆炸
- **涌现回归测试**：给定种子 42 + 100 NPC 运行 100 天 → 存档 hash 应与基线一致（检测非预期的行为变化）
- **Mod 沙箱测试**：Mod 在隔离环境中加载，自动检测崩溃

### 遗漏五：Mod 本地化

在 TOML 中支持多语言字段：

```toml
[identity]
name = { en = "Blacksmith", zh = "铁匠", ja = "鍛冶屋" }
description = { en = "A skilled metalworker.", zh = "熟练的金属工匠。" }
```

## 五、修正后的总预期结果

| 维度 | 修正前（011） | 修正后 |
|------|------------|--------|
| Mod 行为方式 | Rhai 脚本写行为（破坏涌现） | **参数调节 + 概率倾向性**（保护涌现） |
| NPC 独特性 | TOML 固定值 | **TOML 分布（base + variance）** |
| ECS 引入 | Flecs ECS 与现有架构冲突 | **放弃 ECS，用模块化 Rust** |
| 帧预算 | 缺少 Rhai 预算 | **硬预算 0.5ms + 自动禁用超预算 Mod** |
| 姿态来源 | 200 姿态（来源悬空） | **分 Phase 构建：30→100→200** |
| 多物种骨骼 | 未讨论 | **SkeletonTemplate 接口** |
| Mod 存档 | 过于乐观 | **Level 4 明确需要新存档** |
| 测试 | 完全缺失 | **四层测试策略** |
| 本地化 | 完全缺失 | **TOML 多语言字段** |

**修正后的 011 方案保留了所有核心优势（Rust 性能、UAF 动画、Mod 生态、长期稳定）——同时消除了与涌现哲学的根本矛盾，补全了遗漏的关键设计。**
