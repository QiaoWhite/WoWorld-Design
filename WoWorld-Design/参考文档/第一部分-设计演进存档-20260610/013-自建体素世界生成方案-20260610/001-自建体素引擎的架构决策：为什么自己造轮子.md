# 001-自建体素引擎的架构决策：为什么自己造轮子

> 决策：不依赖 Zylann Voxel Tools 或任何第三方体素引擎。  
> 从 Transvoxel 算法出发，构建完全适配 WoWorld 独特需求的体素世界生成管线。

---

## 〇、为什么不用现成的

### 0.1 Voxel Tools (Zylann) 不匹配的地方

| Voxel Tools 的设计假设 | WoWorld 的实际需求 | 冲突 |
|----------------------|-------------------|------|
| 玩家是唯一的地形修改者 | 100K NPC 各自挖矿、建造、踩出道路 | NPC 修改的频率和分布远超插件设计 |
| Chunk 以 `VoxelTerrain` 节点存在于场景树 | 10K+ 活跃 chunk——Godot Node 架构无法承受 | 场景树爆炸 |
| 修改后在 Godot 主线程重新生成 mesh | 高频修改需要异步后台重生成 | 主线程卡顿 |
| 单个密度场（噪声） | 基础地形 + NPC 修改 + 建筑覆盖——需要多层密度场叠加 | 单一密度场不够 |
| 基于视距的 LOD | 需要基于"NPC 活动热度"的额外 LOD 策略 | LOD 逻辑不匹配 |

### 0.2 自建的优势

| 自建可以做到 | 对 WoWorld 的价值 |
|------------|------------------|
| 密度场分层（基础地形层 + 修改层 + 文化层）独立管理 | NPC 挖矿只影响修改层——基础地形不变，存档只存修改层 |
| 异步分帧 mesh 生成完全由 Rust 控制 | 主线程零阻塞 |
| 输出为简单三角形数组 | 不受 Godot 体素插件版本影响——兼容任何 Godot 版本 |
| 为 NPC-terrain 交互优化的数据结构 | 100K NPC 的寻路和碰撞查询在 Rust 侧完成——不经过 Godot |
| 种子确定性 + 增量修改存储 | 存档只保存种子 + 被修改的 chunk——极小的存档体积 |

### 0.3 诚实的代价

- 自建 Transvoxel 实现（非算法研究——有论文 + `transvoxel-rs` reference）
- 自建 chunk 管理系统
- 自建 SDF 修改管线
- 预估额外工作量：2-4 人·月（有 `transvoxel-rs` 作为参考实现时）

**这个代价是值得的**——因为体素世界是 WoWorld 的物理基础，每个系统都依赖它。在这个层面妥协会在未来产生连锁的技术债务。

## 一、核心算法选型：Transvoxel

### 1.1 为什么不是其他选项

| 算法 | 优点 | 为什么不选 |
|------|------|----------|
| **Marching Cubes** | 简单、被充分理解 | 无内置 LOD——远处和近处同样的三角形密度 |
| **Surface Nets** | 比 MC 更平滑 | LOD 过渡不如 Transvoxel 优雅 |
| **Dual Contouring** | 保留尖锐边缘（建筑） | 实现复杂（QEF 求解器）；WoWorld 的低多边形风格不需要尖锐边缘保留 |
| **Cubical Marching Squares** | 比 MC 更简单 | 不如 Transvoxel 成熟 |
| **Transvoxel** | MC 的平滑 LOD 变体，过渡单元格处理 | ✅ 选择此方案 |

**Transvoxel = Marching Cubes + LOD + 无缝过渡。**

### 1.2 Transvoxel 的基本工作原理

```
输入：3D 密度场（每个体素格的 8 个角都有密度值）
参数：等值面阈值（密度 > 阈值 = 固体内部）

LOD 0 (近处, 0.5m/体素):
  → 完整的 Marching Cubes——256 种配置的查找表
  → 最高三角形密度

LOD 1 (中距, 1m/体素):
  → 分辨率减半——2×2×2 的 LOD 0 体素合并为 1 个 LOD 1 体素
  → 密度值通过采样 LOD 0 的对应角来计算

LOD 2 (远处, 2m/体素):
  → 再次减半

过渡单元格（LOD N 和 LOD N+1 的边界）:
  → 使用 Transvoxel 的专用查找表
  → 确保两个 LOD 级别的三角形在边界上无缝连接
```

### 1.3 参考实现

- **`transvoxel-rs` crate**（`github.com/platonvin/Transvoxel-rs`）：可直接 fork 的基础实现
- **Lengyel 2010 论文**："Transition Cells for Dynamic Multiresolution Marching Cubes"
- **策略**：不重新发明算法——在 `transvoxel-rs` 的基础上针对 WoWorld 的需求进行适配和优化

## 二、数据架构：分层密度场

### 2.1 为什么分层

WoWorld 的密度场不是单一来源——它来自多个系统的叠加：

```
最终密度 = 基础地形层（种子→噪声→生物群系）
         + 地质特征层（悬崖、洞穴、矿脉）
         + 文化覆盖层（建筑地基、道路、农田平整）
         + NPC 修改层（挖掘、填土、踩出的路径）
         + 玩家修改层
```

分层的好处：
- **存档效率**：只保存被修改过的层的 chunk。未修改的 chunk 从种子重新生成
- **修改撤销**：每一层的修改可以独立回滚
- **NPC 权限**：NPC 只能修改特定的层（如：NPC 可以在"NPC 修改层"挖掘，但不能删除文化层的建筑地基）

### 2.2 各层的详细设计

```rust
/// 密度场的层级
enum DensityLayer {
    /// 基础地形——由种子+噪声决定。不可修改。
    Base,
    /// 地质特征——悬崖、洞穴、矿脉。世界生成时确定，生成后不可修改。
    Geological,
    /// 文化覆盖——建筑地基、道路、农田平整。由文化系统管理。
    Cultural,
    /// NPC 修改——挖矿产生的大坑、踩出的土径。大量小规模修改。
    NpcModification,
    /// 玩家修改——SDF 雕刻。频率较低但影响范围可能大。
    PlayerModification,
}

/// 分层密度场
struct LayeredDensityField {
    /// 基础+地质——从种子生成的只读层（合并存储以节约内存）
    base_density: BaseDensityProvider,
    /// 文化覆盖——稀疏存储（大部分 chunk 没有文化覆盖）
    cultural_overlay: HashMap<ChunkCoord, SparseModificationBuffer>,
    /// NPC 修改——最频繁、最分散、影响范围最小的修改
    npc_modifications: LruCache<ChunkCoord, ModificationBuffer>,
    /// 玩家修改——频率低、影响范围可能较大
    player_modifications: HashMap<ChunkCoord, ModificationBuffer>,
}

impl LayeredDensityField {
    /// 采样某个世界坐标的最终密度值
    fn sample(&self, pos: Vec3) -> f32 {
        let mut density = self.base_density.sample(pos);
        
        // 按层级叠加——上层覆盖下层
        if let Some(mods) = self.cultural_overlay.get(&pos.to_chunk()) {
            density += mods.sample(pos);
        }
        if let Some(mods) = self.npc_modifications.get(&pos.to_chunk()) {
            density += mods.sample(pos);
        }
        if let Some(mods) = self.player_modifications.get(&pos.to_chunk()) {
            density += mods.sample(pos);
        }
        
        density
    }
}
```

### 2.3 修改缓冲区的内部结构

```rust
/// 单个 chunk 的修改——使用稀疏体素八叉树存储
struct ModificationBuffer {
    /// 修改操作的列表——不是密度值本身，而是 SDF 操作
    operations: Vec<SdfOperation>,
    /// 缓存的密度值——在操作列表变化后重新计算
    cached_density: Option<Box<[f32; CHUNK_VOXELS]>>,
    /// 脏标记——是否有未重新计算的操作
    dirty: bool,
}

enum SdfOperation {
    /// 添加一个球体（填土、堆石头）
    Sphere { center: Vec3, radius: f32, smooth: f32 },
    /// 减去一个球体（挖坑）
    SubtractSphere { center: Vec3, radius: f32, smooth: f32 },
    /// 添加一个长方体（建筑地基、平台）
    Box { center: Vec3, half_extents: Vec3 },
    /// 减去一个长方体（挖矿井、挖地下室）
    SubtractBox { center: Vec3, half_extents: Vec3 },
    /// 沿路径的轨迹——用于道路踩出
    Trail { points: Vec<Vec3>, width: f32, depth: f32 },
}
```

## 三、Chunk 管理：需求驱动的异步加载

### 3.1 加载优先级系统

不是"离玩家越近优先级越高"——而是"对游戏体验影响最大越优先"。

```rust
/// Chunk 加载优先级
struct ChunkPriority {
    /// 优先级分数（越高越先加载）
    score: f32,
}

impl ChunkPriority {
    fn compute(
        distance_to_player: f32,
        has_l1_npcs: bool,        // 这个 chunk 中是否有 L1 NPC
        is_in_player_view: bool,   // 是否在玩家视野中
        npc_modification_count: u32, // 有多少 NPC 正在修改这个 chunk
    ) -> Self {
        let mut score = 0.0;
        
        // 玩家视野中的 chunk → 最高优先级
        if is_in_player_view {
            score += 100.0 - distance_to_player.clamp(0.0, 50.0);
        }
        
        // 有 L1 NPC 在其中的 chunk → 高优先级
        if has_l1_npcs {
            score += 80.0;
        }
        
        // 正在被 NPC 大量修改的 chunk → 高优先级
        score += (npc_modification_count as f32).min(10.0) * 2.0;
        
        // 基础：距离衰减
        score += 50.0 / (1.0 + distance_to_player / 50.0);
        
        Self { score }
    }
}
```

### 3.2 异步生成管线

```
Chunk 生命周期：

EMPTY (不存在)
  ↓ 优先级队列触发加载
GENERATING (后台线程: 噪声生成 + 各层叠加 → 密度场)
  ↓ 密度场就绪
MESHING (后台线程: Transvoxel 等值面提取 → 三角形数组)
  ↓ 三角形数组就绪
LOADED (活跃: 三角形发送给 Godot 构建 ArrayMesh)
  ↓ 一定时间未被访问
CACHED (内存中: 保留密度场 + 修改 buffer，释放三角形数组)
  ↓ 长时间未被访问或内存压力
EVICTED (淘汰: 仅保留修改 buffer 到磁盘，其余从种子可复现)
```

### 3.3 活跃 Chunk 数量控制

```
内存中最多保持:
  - 256 个 LOADED chunk (玩家周围 50m → ~16×16 chunks)
  - 1024 个 CACHED chunk (玩家周围 150m)
  - 总内存 < 200MB (chunk 数据)

超过限制时:
  - 最久未访问的 LOADED → CACHED
  - 最久未访问的 CACHED → EVICTED (保存修改到磁盘)
```

## 四、与 Godot 的集成：最小接触面

### 4.1 数据流（Rust → Godot）

```
Rust 侧（每帧，后台线程）:
  1. 检查优先级队列——哪些 chunk 需要新生成/重新 mesh
  2. 最多处理 3 个 chunk/帧的"生成"阶段
  3. 最多处理 5 个 chunk/帧的"meshing"阶段
  4. 完成 meshing 的 chunk → 将三角形数据打包为字节数组

Rust → Godot 传递（GDExtension，主线程）:
  - PackedVector3Array vertices
  - PackedVector3Array normals  
  - PackedInt32Array indices
  - 每个 chunk 约 50-500KB（取决于三角形密度）
  - 每帧最多 3-5 个新 chunk 的 mesh 数据 = 0.25-2.5MB

Godot 侧（主线程）:
  - ArrayMesh::add_surface_from_arrays()
  - 创建 MeshInstance3D（或更新已有）
  - 创建 PhysicsServer3D 碰撞体（heightfield 或 trimesh）
```

### 4.2 最小化 Godot Node 数

```
Godot SceneTree 中的体素相关 Node:
  └─ TerrainManager (Node)                           ← 1 个
      ├─ MeshInstance3D × ~100 (活跃 LOADED chunk)   ← ~100 个
      └─ 碰撞体通过 PhysicsServer3D RID 管理          ← 0 个 Node
```

不使用 `VoxelTerrain` 节点。不使用体素插件的场景树。所有 chunk 管理在 Rust 侧。

## 五、特殊需求：NPC-Terrain 交互

### 5.1 高频小修改（NPC 挖矿）

100 个 NPC 同时在各自的矿点挖掘——每分钟产生 ~10 次 SDF SubtractSphere 操作，分布在不同的 chunk。

**策略**：
- SDF 操作先缓冲在 `ModificationBuffer.operations` 列表中
- Chunk 标记为 `dirty`
- 不是在每次修改后立即重新 mesh——而是使用**合并重生成**：
  - 如果 chunk 在过去 0.5 秒内已经被标记为 dirty → 合并新的修改
  - 0.5 秒后 → 触发一次重新 mesh（应用这段时间内的所有修改）
- 结果：挖矿操作看起来是即时的（因为操作被立即记录），但 mesh 更新是节流的

### 5.2 道路踩出（Desire Path）

这是 WoWorld 的五级道路系统中 2 级道路的形成机制。

```
NPC 在草地/泥土上频繁行走 → Trail 操作被写入 ModificationBuffer
→ 经过 100 次 Trail 操作的累积 → 地面上出现可见的凹陷
→ 经过 1000 次 → 清晰的土径
→ 如果该路径继续被高频使用 → Rust 侧标记此路径为"2 级道路"
→ 2 级道路触发 Godot 侧的视觉变化（贴图替换 + 微地形调整）
```

实现：
- `Trail` 操作的 `depth` 参数累积——不是一次性挖深，而是每次走过增加 0.001 的深度
- `ModificationBuffer` 的 `Trail` 操作按路径点自动合并（相同路径的多次行走 → 深度叠加）

### 5.3 NPC 寻路与地形

NPC 需要知道"前方是否可以通行"——这需要查询地形。

```
Rust 侧维护一个"可通行性网格"（每 chunk 一个 2D 网格）:
  - 每个 1m² 的格子标记为：可通行 / 陡坡 / 障碍物 / 水
  - NPC 寻路在可通行性网格上进行 A*
  - 当 chunk 的 mesh 更新后 → 异步重新计算可通行性网格
  - 碰撞检测：NPC vs 地形的碰撞在 Godot PhysicsServer3D 中（使用 chunk 的 trimesh 碰撞体）
```

## 六、存档设计

```
save_XXX/
├── world_header.bin       # 种子、世界大小、生成参数版本
├── chunks/
│   ├── 000_000.bin        # 被修改过的 chunk 的 ModificationBuffer
│   ├── 000_001.bin        # 未修改过的 chunk 不存在——从种子可复现
│   └── ...
└── roads.bin              # 2 级+ 道路的路径数据（用于快速查询）
```

存档体积估算（4km² 初始世界，0.5m 体素）：
- 修改过的 chunk：假设 10% 的 chunk 被修改 → ~1600 个 chunk × 平均 50KB = 80MB
- 未修改的 chunk：0 字节（从种子重新生成）
- 道路数据：~1MB
- **总计：~81MB**——远优于 Voxel Tools 的全量存档（~500MB+）
