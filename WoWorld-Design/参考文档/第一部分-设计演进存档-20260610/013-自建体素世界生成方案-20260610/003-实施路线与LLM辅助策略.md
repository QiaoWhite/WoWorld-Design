# 003-实施路线与 LLM 辅助策略

> 将 001（架构）和 002（噪声与生物群系）的技术方案转化为可执行的开发计划  
> 关键约束：Solo 开发者 + LLM 大量生成代码 + Rust + Godot

---

## 〇、总体分阶段策略

```
Phase 1 (2-4周): 最简单的密度场 → 三角形 → Godot 渲染
  → 验证 Rust ↔ Godot 的体素管线可行

Phase 2 (4-8周): 噪声叠加 → 可识别的地形特征
  → 第一个"看起来像自然地形"的世界

Phase 3 (4-8周): 分层密度场 + 修改系统
  → NPC 可以挖坑、玩家可以雕刻

Phase 4 (长期): LOD、洞穴、生物群系、性能优化
  → 完整的体素世界生成管线
```

---

## 一、Phase 1：最小可行管线（2-4 周）

### 目标

在 Godot 中看到一个由 Rust 生成的**单个 chunk** 的体素 mesh——哪怕它只是一个球体密度场。

### 具体步骤

```
Week 1: Rust 侧
  □ 创建 `voxel_core` crate
  □ 实现最简单的密度场：球体 (center=原点, radius=10m)
     fn sphere_density(pos: Vec3) -> f32 { 10.0 - pos.length() }
  □ 移植 transvoxel-rs 的 Marching Cubes 查找表（或直接引入 crate）
  □ 对单个 16×16×16 体素 chunk 运行 Marching Cubes
  □ 输出：Vec<Vec3> vertices + Vec<u32> indices
  □ cargo test: 验证三角形数量 > 0

Week 2: Godot 侧 + 集成
  □ GDExtension 桥：Rust → PackedVector3Array + PackedInt32Array
  □ Godot 侧：ArrayMesh::add_surface_from_arrays()
  □ MeshInstance3D 显示
  □ 加一个旋转的摄像机
  □ 加上方向光 → 看到一个灰色的球体
  ✅ 里程碑 1：一个由 Rust 生成的体素 mesh 在 Godot 中渲染
```

### LLM 角色

- 生成 256 种 Marching Cubes 配置的查找表代码（这是 LLM 极擅长的工作——从论文附录翻译为 Rust 数组）
- 生成 `ArrayMesh` 构建的 GDScript 代码
- 生成 GDExtension 的 `PackedVector3Array` 传递代码
- 调试：如果 mesh 不渲染——LLM 帮助检查法线方向、索引顺序、材质分配

## 二、Phase 2：噪声地形（4-8 周）

### 目标

生成一个"看起来像自然地形"的世界——有山、有谷、有海。

### 具体步骤

```
Week 3-4: 噪声层实现
  □ 引入 noise crate (Perlin + Worley + fBm)
  □ 实现分层噪声叠加（003 文档 §1.2 的公式）
  □ 多个 chunk (32×32×4 chunks = 128m × 128m 区域)
  □ 每个 chunk 32×32×32 体素 → 运行 Marching Cubes
  □ 平坦的 chunk 做 greedy meshing（合并同质的巨大三角形）

Week 5-6: Chunk 管理
  □ ChunkManager: 优先级队列 + 异步生成
  □ 玩家周围加载 ~50 个 chunk
  □ 简单的 LOD: 远处的 chunk 用更少的体素采样
  □ 流式加载: 玩家移动时加载前方 chunk，卸载后方 chunk

Week 7-8: 视觉完善
  □ 基于高度的简单材质映射（低处=绿草、中处=灰岩、高处=白雪）
  □ 基础的 Godot Environment (方向光 + 天空 + 雾)
  □ 性能测试: 50 chunk 渲染 + 玩家移动 → 稳定 60fps
  ✅ 里程碑 2：一个可以探索的自然地形世界
```

### LLM 角色

- 生成完整的分层噪声叠加代码——这是纯数学、LLM 擅长
- 生成 ChunkManager 的异步调度逻辑（`rayon` 线程池 + `crossbeam` channel）
- 生成性能测试用例（帧率监控 + 内存跟踪）
- 注意：LLM 生成的噪声参数（频率、振幅）可能不合理——需要开发者在 Godot 中"看"地形并手动调整

## 三、Phase 3：分层密度场 + 修改系统（4-8 周）

### 目标

NPC 可以挖坑，玩家可以雕刻地形，修改被持久化到存档。

### 具体步骤

```
Week 9-10: 分层密度场
  □ 实现 LayeredDensityField（基础层 + 修改层）
  □ BaseDensityProvider: 从种子+噪声生成（Phase 2 的成果）
  □ ModificationBuffer: 稀疏 SDF 操作列表
  □ density.sample(pos) → 透明地组合所有层

Week 11-12: SDF 修改操作
  □ SdfOperation: Sphere/Box/Trail 的实现
  □ ModificationBuffer::apply() → 重新计算密度值
  □ 脏 chunk 检测 + 节流重新 mesh（0.5s 合并窗口）
  □ 测试: 100 个"假 NPC"随机挖坑 → 性能不下降

Week 13-14: 存档
  □ 修改过的 chunk → bincode 序列化到磁盘
  □ 未修改的 chunk → 从种子重新生成
  □ 存档加载: 种子 → 基础地形 + 重放修改操作
  □ 测试: 挖 100 个坑 → 存档 → 读档 → 坑还在

Week 15-16: Godot 侧工具
  □ 玩家 SDF 雕刻工具（半径可调的挖/填刷子）
  □ 简单的调试 UI: 显示当前看向的 chunk 的 ID + 修改次数
  ✅ 里程碑 3：一个可以被玩家修改、持久化修改的体素世界
```

### LLM 角色

- SdfOperation 枚举 + apply 逻辑——标准几何计算，LLM 零错误
- 存档的 serde 代码——`#[derive(Serialize, Deserialize)]` + bincode 配置
- Godot 雕刻工具的输入处理代码
- 注意：脏 chunk 的节流逻辑需要开发者手动调参——LLM 只能生成初始值

## 四、Phase 4+：完整管线（长期）

### 4.1 洞穴生成

- 3D Perlin 噪声 → 地下挖空区域
- 与地表修改层的交互：如果玩家从地表挖到洞穴——两层合并
- LLM 生成 3D 噪声洞穴的完整实现

### 4.2 Transvoxel LOD

- LOD 0 (0.5m) → LOD 1 (1m) → LOD 2 (2m) → LOD 3 (4m)
- 过渡单元格：引入 `transvoxel-rs` 的 transition lookup tables
- 远 chunk 的三角形密度显著降低
- LLM 生成 LOD 切换逻辑和过渡单元格处理

### 4.3 生物群系集成

- 温度/降水 map 从噪声派生
- 群系分类 → 材质映射 + 植被密度 + 地表特征
- 群系边界平滑过渡
- LLM 生成群系分类的决策树代码

### 4.4 GPU 加速

- 噪声生成迁移到 compute shader (WGSL)
- 一个 chunk (32³ = 32768 个采样点) 在 1 次 GPU dispatch 中完成
- Marching Cubes 的 GPU 实现（每个体素格一个 GPU 线程）
- 参考：`voxel-game-rs` + `Kosmos` 的 compute shader 实现

### 4.5 NPC-Terrain 交互

- NPC 挖掘：`SdfOperation::SubtractSphere` 写入目标 chunk 的 ModificationBuffer
- 道路形成：NPC 每次走过一个位置 → `Trail` 操作的深度累积
- 寻路：可通行性网格在 chunk mesh 更新时异步重新计算
- NPC 建筑地基：文化层的 `SdfOperation::Box`

## 五、关键技术风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| **Transvoxel LOD 过渡有视觉伪影** | 🟡 中 | 从 `transvoxel-rs` 开始——已验证的实现；如果过渡单元仍有问题，降级为纯 MC (无 LOD) |
| **分帧加载赶不上玩家高速移动** | 🟡 中 | Phase 1 中玩家的移动速度限制为步行；Phase 3+ 引入预加载 |
| **SDF 操作过多导致 chunk 永远 dirty** | 🟢 低 | 0.5s 合并窗口确保每 chunk 最多 2 次重新 mesh/秒 |
| **噪声参数调不出"好看"的地形** | 🟡 中 | 手动调参 + LLM 辅助搜索参数空间；参考 Kosmos 的噪声组合方式 |
| **Rust ↔ Godot mesh 数据传输瓶颈** | 🟢 低 | 在 Phase 1 就验证——50KB-500KB/chunk 在现代硬件上 < 0.5ms |

## 六、LLM 辅助的噪声调参工作流

这是 "AI 最帮不上忙" 的部分——因为噪声参数的"好/坏"是主观的，只能用人眼判断。但 AI 可以加速调参循环：

```
1. 开发者在 Godot 中观察当前地形 → 觉得"山太尖了"
2. 告诉 LLM: "我的山太尖了，当前 parameter: mountain = RidgedPerlin(freq=1/1000, amp=100)。怎么让山更圆润？"
3. LLM: "降低 RidgedPerlin 的 gain 参数从 0.5 到 0.3，增加 fBm 的 octaves 从 4 到 6，使山体边缘有更多层次的中等起伏"
4. 开发者修改参数 → cargo run → Godot 重新生成 → 观察
5. 循环直到满意
```

**关键**：让 LLM 理解"尖""平坦""破碎"这些主观词对应的噪声参数含义——在提示词中预先定义这个映射。

## 七、与现有 Godot 体素方案的协作

在完全迁移到自建引擎之前，Voxel Tools 仍然有价值：

| 阶段 | 体素方案 |
|------|---------|
| Phase 1 初期（Week 1-2 的 Godot 侧） | Voxel Tools 用于**快速验证**：生成一个 VoxelTerrain → 确保 Godot 渲染管线正常 → 然后切换到自建 mesh |
| Phase 1-2 并行开发 | Voxel Tools 作为**参考实现**：当自建 mesh 有问题时，对比 Voxel Tools 的输出 |
| Phase 3+ | Voxel Tools 彻底移除——所有体素功能由自建引擎提供 |

---

> **本方案不是"从零发明"——是利用 `transvoxel-rs`（已有算法实现）+ `noise` crate（已有噪声库）+ LLM（代码生成加速器），构建一个完全适配 WoWorld 的独特需求的体素引擎。核心技术来自社区，组合方式为 WoWorld 定制。**
