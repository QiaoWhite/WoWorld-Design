## 一、远景需求 vs 013 方案——缺口分析

### 013 的当前设计

013 只考虑了 4 级 LOD（0.5m→1m→2m→4m）。在 LOD 3（4m/体素）下，远处的三角形密度仍然很高。以玩家视野 5km 为例：

```
视野 5km 半径 → 面积 78 km² → ≈ 78000 个 chunk (32m² each)
如果一半在视野中 → ~39000 个 chunk 需要"某种形式"的渲染
```

013 的活跃 chunk 上限是 256 个 LOADED + 1024 个 CACHED——差了 **38 倍**。

### 缺口清单

| 缺口 | 013 现状 | 远景需要 |
|------|---------|---------|
| LOD 级别数 | 4 级（×8 距离） | **7-8 级**（×128 距离） |
| 最远 LOD 精度 | 4m/体素 | **64-128m/体素**（仅地形轮廓） |
| 远景表示方式 | 仅 Transvoxel mesh | **混合**：中距 Transvoxel + 远距 heightfield + 极远距 billboard |
| 大气遮挡 | 未讨论 | **必需**：高度雾 + 距离雾 + 大气散射——既增强深度感又隐藏 LOD 过渡 |
| 内存上限 | 所有 LOADED chunk 在内存 | 远景 chunk 应该用 **流式纹理** 或 GPU 驻留——不占 CPU 内存 |

### 远景的可行性判断

**技术上完全可行**，但需要 LOD 体系的根本扩展：

```
LOD 0: 0.5m/voxel  →  0-50m     (013 方案: LOD 0)
LOD 1: 1m/voxel    →  50-100m   (013 方案: LOD 1)
LOD 2: 2m/voxel    →  100-200m  (013 方案: LOD 2)
LOD 3: 4m/voxel    →  200-400m  (013 方案: LOD 3)
LOD 4: 8m/voxel    →  400-800m  🆕 需要新增
LOD 5: 16m/voxel   →  800-1600m 🆕 需要新增
LOD 6: 32m/voxel   →  1.6-3.2km 🆕 Heightfield 替代
LOD 7: 64m/voxel   →  3.2-6.4km 🆕 Billboard/Impostor
LOD 8: 128m/voxel  →  6.4km+    🆕 纯大气散射 + 天空盒融合
```

对于 LOD 6+，不再运行完整的 Transvoxel——使用 **heightfield mesh**（一个 2D 高度图 + 低分辨率颜色的简化 mesh）。在 128m/体素的精度下，一个 64×64 顶点的 heightfield 就能覆盖 8km²——仅 ~8000 个三角形。

## 二、奇幻地形需求 vs 013 方案——缺口分析

### 013 的当前设计

013 的噪声叠加公式完全基于**模仿现实地形**——Perlin/Ridged/fBm/Worley 的组合。可以产生可信的山脉、丘陵、海岸线——但不能产生：

- **浮空岛**：密度在 y=500m 处为正，且不连接到地面
- **巨菇林**：数百个蘑菇形密度体散布在区域中
- **水晶峡谷**：几何形状的棱柱形密度体
- **魔法扭曲**：空间被拉伸、折叠、漩涡化

### 缺口

013 缺少一个**"奇幻密度提供器"**的概念——一个可以与基础地形层并列的密度源。

### 解决方案方向

在分层密度场中增加**奇幻特征层**：

```rust
struct FantasyFeatureLayer {
    /// 特征列表——每个特征是独立的密度提供器
    features: Vec<Box<dyn DensityProvider>>,
}

trait DensityProvider {
    /// 在世界坐标系中采样密度。只在自己覆盖的区域内返回非零值
    fn sample(&self, pos: Vec3) -> f32;
    /// 该特征影响的范围（用于空间查询——只评估覆盖当前 chunk 的特征）
    fn bounding_box(&self) -> Aabb;
}

// 示例实现
struct FloatingIsland {
    center: Vec3,
    radius: f32,
    height_variation: f32,
    noise_seed: u64,
}

impl DensityProvider for FloatingIsland {
    fn sample(&self, pos: Vec3) -> f32 {
        let dist_to_center = (pos.xz() - self.center.xz()).length();
        if dist_to_center > self.radius { return 0.0; }
        
        // 岛的形状：中心高、边缘薄的碟形
        let base_shape = self.radius - dist_to_center;
        // 底部：锥形衰减（岛不是悬在空中的方块——它的底部逐渐变细）
        let bottom_taper = if pos.y < self.center.y {
            (pos.y - (self.center.y - self.radius * 0.3)).max(0.0) / (self.radius * 0.3)
        } else { 1.0 };
        // 叠加上表面噪声（岛的顶部不是完全平坦的）
        let surface_noise = fbm_3d(pos.x / 20.0, pos.y / 20.0, pos.z / 20.0, self.noise_seed, 3) * self.height_variation;
        
        (base_shape + surface_noise) * bottom_taper
    }
}
```

**关键**：奇幻特征层不覆盖基础地形——它叠加在基础地形之上。浮空岛的基础地形下面是空气（密度为负），岛本身提供正的密度值。

## 三、高空生态群系需求 vs 013 方案——缺口分析

### 013 的当前设计

013 的生物群系分类基于**温度-降水**的二维空间——这是一个地表概念。温度由纬度 + 海拔决定——但这里的"海拔"隐含了"地表高度"的前提。

在数千米的高空——"温度"和"降水"这些概念不再适用。需要完全不同的群系分类逻辑。

### 缺口

- 013 没有定义**垂直生态分层**——地表、低空、中空、高空的群系是不同的
- 天空群系的决定因素不是温度/降水——而是**高度、魔法浓度、气流、光照强度**
- 013 的 chunk 是 32×32×32 的立方体——对于数千米的高度范围，需要**垂直堆叠**很多 chunk，但 013 的 chunk 管理假设了地表中心

### 解决方案方向

**垂直生态分层**：

```
高度范围        生态层          群系示例
─────────────────────────────────────────────
3000m+         极高空层        虚空裂隙、星辰浮岛、极光带
1500-3000m     高空层          云中森林、浮空岛链、风暴区
500-1500m      中空层          悬浮湿地、鹰巢崖壁、风力草原
100-500m       低空层          山巅、巨树冠层、飞行生物栖息地
0-100m         地表层          013 现有的温度-降水群系
-50-0m         浅地层         洞穴入口、地下湖泊
-200~-50m      深地层         洞穴网络、地下河流、矿脉
-2000~-200m    极深层         岩浆区、上古遗迹、虚空边界
```

天空群系的"气候"因素：

| 因素 | 来源 | 对群系的影响 |
|------|------|-----------|
| **高度** | 绝对 y 坐标 | 决定了是哪个生态层 |
| **魔法浓度** | 世界种子 + 高度（魔法在高空更浓） | 浮空岛的密度、魔法生物的出现 |
| **气流强度** | 基于地形 + 高度的风场模拟 | 决定了植被类型（风暴区 vs 平静区） |
| **光照** | 高度越高越强，但云层遮挡 | 天空森林的上层 vs 下层 |

## 四、高速移动需求 vs 013 方案——缺口分析

### 013 的当前设计

- Chunk 生成：最多 3 个/帧的 generating + 5 个/帧的 meshing
- 加载半径：固定 ~150m
- 优先级：距离 + NPC + 视野的加权

### 高速移动下的参数

```
步行速度:   5 m/s    → 穿越 150m 半径需要 30s  → 每帧需加载 0.3 个新 chunk
骑马速度:   15 m/s   → 穿越 150m 半径需要 10s  → 每帧需加载 1 个新 chunk
飞龙速度:   100 m/s  → 穿越 150m 半径需要 1.5s → 每帧需加载 6 个新 chunk
魔法飞行:   200 m/s  → 穿越 150m 半径需要 0.75s→ 每帧需加载 12 个新 chunk
```

013 的生成速度（3+5=8 个/帧）在步行和骑马时足够——但在飞龙和魔法飞行时**差了一个数量级**。

### 缺口

| 缺口 | 013 现状 | 高速移动需要 |
|------|---------|-----------|
| 生成吞吐量 | 8 个/帧 | **50-100 个/帧**（GPU 噪声 + 简化 mesh） |
| 加载半径 | 固定 150m | **动态**：速度 × 3 秒 = 飞龙时 ~300m |
| 预加载 | 无 | **基于速度向量的预测性预加载** |
| 速度降级策略 | 无 | **高速时允许更低 LOD**——先显示低质量再逐步细化 |
| 垂直加载 | 无 | **飞行时前后上下所有方向都需要加载** |

### 可行性判断

**高速加载在技术上是可行的**——但需要三个关键改变：

1. **噪声生成移到 GPU**：compute shader 生成 32³ 体素的噪声值——1 次 dispatch < 0.1ms。CPU 循环需要 ~1-2ms。GPU 加速后，噪声生成不再是瓶颈。

2. **高度压缩的 mesh 生成**：远 LOD 的 mesh 三角形极少——LOD 4 的 32×32×32 chunk 只有 ~100-500 个三角形（vs LOD 0 的 ~5000-10000 个）。远 LOD 的 mesh 生成在 CPU 上也极快。

3. **Heightfield fallback**：如果 chunk 的 Transvoxel mesh 还没生成好，先用 heightfield 近似。一个 32×32 顶点的 heightfield = 2048 个三角形——可以在 < 0.05ms 内生成。当 Transvoxel mesh 就绪后替换。

### 感知连续性——更重要的维度

玩家不会注意到 3km 外的 chunk 是 LOD 5 还是 LOD 6。但他们**会**注意到眼前的地形突然"跳出来"。

**加载策略**：不是"让所有 chunk 都准备好"——而是"确保玩家前进方向上的 chunk 优先准备好"。

```
飞行方向预测：
  velocity = current_velocity + 惯性
  predicted_path = player_pos + velocity * t (for t = 0.5, 1.0, 2.0, 5.0 seconds)
  
  沿预测路径的 chunk → 最高优先级
  预测路径两侧的 chunk → 中优先级
  玩家后方的 chunk → 低优先级（但保留在内存中一段距离，以备急转弯）
```

## 五、极高世界高度需求 vs 013 方案——缺口分析

### 013 的当前设计

013 的 chunk 是 32m³ 的立方体。世界高度从 y=-200m 到 y=+500m——共 ~700m 高度范围 = ~22 个 chunk 垂直堆叠。chunk 管理是 2D 网格（xz 平面）。

### 扩展到数千米高度

世界高度扩展到 y=-2000m 到 y=+4000m——~6000m 范围 = **~188 个 chunk 垂直堆叠**。

这意味着：
- Chunk 管理从 2D 网格变为**3D 稀疏网格**——大部分垂直列中只有少数 chunk 有实际内容（地表附近），高空中只有特殊区域（浮空岛）有 chunk
- 稀疏性是关键——6000m 高度范围的 99% 是空气。不需要为空气创建 chunk

### 解决方案：稀疏 3D Chunk 管理

```rust
struct SparseChunkGrid {
    // 不是 2D 的 `HashMap<(i32, i32), Chunk>`——
    // 而是 3D 的 `HashMap<(i32, i32, i32), Chunk>`
    chunks: HashMap<ChunkCoord3D, Chunk>,
    
    // 垂直列的快速查询：给定 (x, z)，返回"这个列中有哪些 chunk"
    vertical_columns: HashMap<(i32, i32), Vec<i32>>,  // (x, z) → [y1, y2, ...]
}
```

大多数 (x, z) 位置只有 1-5 个 chunk（地表附近）。只有浮空岛、巨树等特征才会在高空产生 chunk。

内存评估：100m × 100m 的区域 × 6000m 高度 → 只有 ~200 个活跃 chunk（vs 未优化的 2D 扩展中 ~6000 个）——因为 97% 是空气。

## 六、地面高速移动需求 vs 013 方案——缺口分析

### 与空中高速移动的差异

地面高速移动（~50-80 m/s）与空中高速有本质区别：

- **玩家贴近地表**：需要 LOD 0-2 的高精度 mesh——不能用 heightfield 糊弄。玩家距离地面只有几米，低精度 mesh 的视觉缺陷会被放大。
- **加载方向集中**：在空中飞行时，加载是 360° 全方位的。在地面高速奔跑时——玩家的视野和移动方向高度一致——加载压力集中在正前方 ±30° 的扇形区域。
- **地形碰撞**：地面移动需要精确的碰撞检测——玩家/NPC 撞上一块还没加载好碰撞体的石头 = 穿模/卡死。空中飞行没有这个问题。

### 缺口

| 缺口 | 013 现状 | 地面高速需要 |
|------|---------|-----------|
| 近景精度优先级 | 均匀分配 | **前方扇形区域优先**——LOD 0-1 的生成优先级远高于其他方向 |
| 碰撞体就绪保证 | mesh 和碰撞异步 | **碰撞必须先于视觉 mesh 就绪**——玩家不能穿模 |
| 地表植被加载 | 未讨论 | 高速奔跑时——树/石头等 scatter 物体也需要快速加载 |

### 解决方案方向

**碰撞优先管线**：

```
正常管线: 噪声 → 密度场 → mesh → 碰撞体
地面高速管线: 噪声 → 碰撞体（仅）→ mesh
                               ↑
                    碰撞体先让步于视觉——
                    玩家会看到一个"没有细节"的树桩（碰撞已就绪）
                    但不会穿模。0.2s 后 mesh 到位——树桩变成了完整的树。
```

## 七、天体周期与季节需求 vs 013 方案——缺口分析

### 013 的当前状态

013 **完全未涉及**天体系统、时间系统、季节系统。013 的 `WorldConfig` 没有时间相关字段。生物群系分类是静态的（仅基于温度-降水-高度）。

### 缺口

| 缺口 | 说明 |
|------|------|
| **天体渲染** | 太阳/月亮/星星的天空渲染——Godot 内置了 `Sky` 资源可以配置 procedural sky，但天体周期逻辑需要自己写 |
| **时间系统** | 游戏内的一天/一月/一年的时间推进。这个系统不存在于任何现有文档中 |
| **季节对群系的影响** | 同一个位置的群系在夏天和冬天应该看起来不同——这需要"季节化的群系参数" |
| **季节对区域的影响** | 热带可能只有雨季/旱季（两季），温带四���分明，寒带常年冰雪——这需要群系特定的季节规则 |
| **天气系统** | 雨/雪/暴风/晴天——与季节和群系的关系；天气的视觉表现（粒子、天空变化、地面湿润/积雪） |

### 解决方案方向

时间系统是完全独立于体素引擎的模拟层。在 Rust 模拟核心中新增 `TimeManager`：

```rust
struct TimeManager {
    /// 游戏内当前时间（自世界创建以来的秒数）
    total_seconds: f64,
    /// 一天的实际游戏时长（可配置——默认 24 分钟 = 1440 秒）
    seconds_per_day: f64,
    /// 一年的天数（可配置）
    days_per_year: u32,
    /// 当前日期
    current_day: u32,
}

impl TimeManager {
    fn time_of_day(&self) -> f32 { /* 0.0-1.0, 0=午夜 0.5=正午 */ }
    fn day_of_year(&self) -> u32 { /* 1-365 */ }
    fn season(&self, latitude: f32) -> Season { /* 根据纬度 + 日期计算 */ }
}
```

天体渲染在 Godot 侧——使用 `Sky` shader 中的太阳角度、月亮相位、星星亮度参数。这些参数由 Rust `TimeManager` 每帧计算并传给 Godot。

## 八、天气与环境互动需求 vs 013 方案——缺口分析

### 013 的当前状态

013 **完全未涉及**天气系统。生物群系是静态的——没有"正在下雨"的状态。

### 缺口

| 缺口 | 说明 |
|------|------|
| **天气类型** | 晴/阴/雨/雪/暴风/沙尘暴/魔法风暴——需要定义每种天气的参数和视觉效果 |
| **天气与群系的关系** | 沙漠不下雪、热带无冬季——群系限定了可能的天气范围 |
| **天气的环境影响** | 雨水浸湿地面（视觉变化）、积雪积累（微密度叠加）、暴风影响飞行速度 |
| **天气对 NPC 的影响** | NPC 雨天回家/穿厚衣、暴风时全部躲进建筑、高温天气减少户外活动 |
| **农作物与季节/天气** | 春耕秋收、霜冻毁坏作物、干旱减产——这需要农作物系统，超出了体素引擎的范围 |

### 解决方案方向

天气系统是模拟核心的一个独立子系统。在 Rust 侧新增 `WeatherManager`：

```rust
struct WeatherManager {
    /// 当前天气（每个区域可能有不同天气）
    regional_weather: HashMap<RegionId, WeatherState>,
    /// 天气变化的时间表
    weather_schedule: Vec<WeatherEvent>,
}

struct WeatherState {
    weather_type: WeatherType,  // Sunny/Rainy/Snowy/Stormy/...
    intensity: f32,             // 0.0-1.0
    temperature: f32,           // 当前实际温度（受季节+群系+天气影响）
    humidity: f32,              // 当前湿度
    wind_speed: f32,            // 风速（影响飞行和投射物）
    wind_direction: Vec3,       // 风向
}
```

天气的视觉表现（雨粒子、雪粒子、天空色彩变化）在 Godot 侧——使用 `ParticleSystem` + `Environment` 的动态调整。

环境互动（地面湿润、积雪覆盖）通过对体素密度场的**临时天气修改层**实现：

```rust
/// 天气对密度场的影响——临时的、可撤销的、不与存档混合的
struct WeatherDensityOverlay {
    snow_accumulation: HashMap<ChunkCoord3D, f32>,  // 积雪深度 (m)
    puddle_accumulation: HashMap<ChunkCoord3D, f32>, // 积水深度 (m)
}
```

## 九、天体自定义需求 vs 013 方案——缺口分析

### 需求

Modder 可以自定义天体：太阳的数量（可以是 0-3 个）、月亮的大小和颜色、星星的密度和颜色、是否有光环/极光、天体的运行轨道。

### 缺口

013 **完全未涉及**。这也超出了体素引擎的范围，属于渲染/天空系统的领域。

### 解决方案方向

在 Godot `Sky` shader 中暴露自定义参数：

```gdscript
# 天体配置（从 TOML Mod 文件加载）
var celestial_config = {
    suns = [
        { color = Color.YELLOW, size = 1.0, orbit_period = 1.0 },
        { color = Color.ORANGE_RED, size = 0.7, orbit_period = 2.5 },
    ],
    moons = [
        { color = Color.WHITE, size = 0.8, orbit_period = 28.0, phase_period = 28.0 },
        { color = Color.PURPLE, size = 0.3, orbit_period = 15.0 },
    ],
    stars = { density = 1.0, color_variation = 0.3 },
    has_aurora = true,
}
```

Rust 模拟核心中的 `TimeManager` 计算出每帧的天体位置（太阳角度、月亮相位）→ Godot 侧 `Sky` shader 使用这些参数渲染天体。

## 十、NPC 高速移动需求 vs 013 方案——缺口分析

### 与玩家高速移动的差异

玩家只有一个。NPC 可能有很多个同时在高速移动。

关键区别：
- **加载触发者多了 N 倍**：不是"1 个玩家触发加载"——是"1 个玩家 + N 个高速移动的 NPC 各自触发加载"。如果 20 个 NPC 同时在飞龙上飞向不同方向——加载需求从"玩家前方扇形"变为"N 个扇形区域的并集"。
- **NPC 的加载优先级**：NPC 看到的低精度 terrain 比玩家看到的低精度更可接受——NPC 不需要"沉浸感"。

### 缺口

| 缺口 | 说明 |
|------|------|
| **多触发源** | Chunk 优先级系统需要接受多个移动源——不只是玩家 |
| **NPC 视角不重要** | NPC 不需要 LOD 0——LOD 2-3 对 NPC 导航足够 |
| **预测性加载的扩展** | 需要为每个高速移动的 NPC 独立预测路径——但预测精度可以更低 |

### 解决方案方向

```rust
struct ChunkLoadingSources {
    /// 玩家——最高优先级加载源
    player: LoadingSource { position, velocity, priority_multiplier: 1.0 },
    /// 高速 NPC——较低优先级
    fast_npcs: Vec<LoadingSource>,
}

struct LoadingSource {
    position: Vec3,
    velocity: Vec3,
    priority_multiplier: f32,  // 玩家 1.0, 同伴 0.6, 无关 NPC 0.3
    min_lod: u8,              // 玩家 0, 同伴 2, 无关 NPC 3
}
```

NPC 的加载最低 LOD 高于玩家——这意味着 NPC 高速移动时，远方的 chunk 不需要以最高精度生成。只有玩家附近的 chunk 需要 LOD 0。

## 十一、013 方案的其他纰漏

除上述十一大需求外，013 还存在以下独立缺陷：

| 纰漏 | 严重度 | 说明 |
|------|--------|------|
| **Marching Cubes 的拓扑歧义** | 🟡 中 | 256 种配置中约 15 种存在拓扑歧义——可能产生孔洞。Transvoxel 论文有解决方案但 013 未提及 |
| **水面渲染** | 🟡 中 | 013 讨论了地形——但没有讨论水面。河流、湖泊、海洋需要特殊的渲染（透明/反射表面） |
| **植被渲染** | 🟡 中 | 生物群系决定了植被类型——但 013 没有讨论树/草如何渲染（GPU Instancing？） |
| **LOD 切换的 popping** | 🟡 中 | 即使 Transvoxel 过渡单元格消除接缝，LOD 切换仍可能有视觉 popping——需要 morphing 或 temporal blending |
| **初始世界生成时间**| 🟢 低 | 玩家按"新建游戏"后需要等待多久？如果 4km² 的世界需要 30s+ 来生成——需要进度条 |
| **时间系统的缺失** | 🟡 中 | 013 没有任何时间推进机制——静态世界。天体、季节、天气、农作物——全部依赖时间系统 |
| **天气视觉与体素的耦合** | 🟡 中 | 积雪、积水、地面湿润——对体素有物理影响（密度叠加），需要特殊处理 |

### 纰漏解决方案

- **MC 歧义**：使用 Transvoxel 论文中的"扩展查找表"——编码了歧义解决方案的 512 种配置（而非标准的 256 种）
- **水面**：不是体素的一部分——是在 y=0（海平面）处的一个半透明平面 + 河流的多段 mesh
- **植被**：每种生物群系的植被密度存储在 chunk 的 metadata 中。植被用 GPU Instancing 渲染——不在体素 mesh 中
- **LOD popping**：在两个 LOD 级别之间做 2-3 帧的 alpha blend（LOD N 淡出 + LOD N+1 淡入）
- **初始生成**：只生成玩家周围 500m 的区域（~2s）。其余在后台生成。玩家可以立即开始游戏
- **时间系统**：在 Rust 模拟核心中新增 `TimeManager` 单例——独立于体素引擎，所有子系统通过事件总线读取时间
- **天气-体素耦合**：新增 `WeatherDensityOverlay` 层——临时密度叠加，不与存档混合，天气变化时自动重新计算
