## 一、远景方案：8 级 LOD + 混合表示

### 1.1 扩展的 LOD 体系

```
LOD 0: 0.5m/voxel, 32³ voxels/chunk → ~5K-15K tris,  0-50m
LOD 1: 1m/voxel,   16³ voxels/chunk → ~2K-5K tris,   50-100m
LOD 2: 2m/voxel,   16³ voxels/chunk → ~1K-3K tris,   100-200m
LOD 3: 4m/voxel,   16³ voxels/chunk → ~500-1.5K tris, 200-400m
═══════════════════════ 013 方案止于此 ═══════════════════
LOD 4: 8m/voxel,   16³ voxels/chunk → ~200-800 tris,  400-800m
LOD 5: 16m/voxel,  16³ voxels/chunk → ~100-400 tris,  800-1600m
LOD 6: 32m/voxel,  Heightfield替代    → ~2K tris/64²,  1.6-3.2km
LOD 7: 64m/voxel,  Heightfield替代    → ~2K tris/64²,  3.2-6.4km
LOD 8: 128m/voxel, Billboard+Atmo     → ~100 tris,     6.4km+
```

### 1.2 Heightfield 替代（LOD 6-7）

对于 1.6km 以外的地形——玩家看到的是**山脊线和颜色**，不是地形细节。用 heightfield 替代 Transvoxel 完全足够。

```
Heightfield 生成（Rust 侧）:
  1. 对 64×64 的 xz 网格采样高度（用与基础地形层相同的噪声——保证一致性）
  2. 采样颜色（生物群系颜色映射）
  3. 生成高度图 mesh（64×64 顶点 = 7938 个三角形——极轻量）
  4. 可选：根据"复杂度"自适应网格密度（平坦区域 16×16 足够，山区 64×64）
  
传给 Godot: 与 Transvoxel mesh 相同的 ArrayMesh 接口——Godot 侧不需要区分
```

### 1.3 Billboard/Impostor（LOD 8）

6.4km 以外——只是天空背景的一部分。用**静态 impostor**：

- 预渲染一张从该方向看去的地形轮廓纹理（每 30° 一张，共 12 张）
- 在 Godot 中作为天空球的远景层
- 纹理在 chunk 更新时异步重新渲染（因为地形可能被修改）

### 1.4 大气散射与雾——隐藏 LOD 过渡的关键

```
高度雾：
  y < 0m: 浓雾（深海）
  y 0-100m: 地表轻雾（柔化远山轮廓）
  y 100-1000m: 稀薄大气（低空飞行时能看到较远）
  y 1000m+: 极稀薄（高空俯瞰——最远的 LOD 可见）

距离雾：
  0-1km: 无雾
  1-3km: 轻雾——隐藏 LOD 3→4 过渡
  3-6km: 中雾——隐藏 LOD 5→6 过渡
  6km+: 与天空盒颜色融合——隐藏 LOD 7→8 过渡

雾气颜色采样：
  从天空盒采样地平线颜色 → 远处地形融入天空色
  这比"均匀灰雾"自然得多——而且零额外成本（Godot Environment 内置）
```

## 二、奇幻地形方案：可插拔的密度提供器

### 2.1 体系设计

```rust
/// 密度提供器——任何可以返回密度值的东西
trait DensityProvider: Send + Sync {
    fn sample(&self, pos: Vec3, lod: u8) -> f32;
    fn bounds(&self) -> Aabb;
    fn provider_type(&self) -> DensityProviderType;
}

enum DensityProviderType {
    Procedural,   // 纯噪声生成——可在 GPU 上计算
    Geometric,    // 几何形状——CPU 计算，但可以缓存
    DataDriven,   // 从数据文件加载的密度场（Mod 支持）
}

/// 奇幻特征注册表
struct FantasyFeatureRegistry {
    // 按空间划分——快速查询哪些特征覆盖了当前 chunk
    spatial_index: RTree<FeatureHandle>,
    features: HashMap<FeatureId, Box<dyn DensityProvider>>,
}
```

### 2.2 预置奇幻特征类型

| 特征 | 密度函数 | 参数 |
|------|---------|------|
| **浮空岛** | 碟形 + 表面噪声 + 底部锥形 | center, radius, height, noise_seed |
| **巨菇** | 圆柱（茎）+ 半球（冠）+ 噪声（冠边缘） | center, stem_radius, cap_radius, height |
| **水晶簇** | 六方柱的 SDF 并集 + 随机旋转 | center, count, max_radius, height |
| **魔法漩涡** | 对输入坐标做 domain warping——基础地形的形状被旋转/拉伸 | center, radius, warp_strength |
| **浮空瀑布** | 从浮空岛边缘向下延伸到地表——细长的密度体 | top_center, bottom_center, width |
| **龙巢** | 悬崖上的凹陷（SDF 减法 + 噪声） | center, radius, depth |
| **石化森林** | 散布的圆柱体——不同高度和粗细 | center, count, radius_range, height_range |
| **虚空裂隙** | 密度场的"裂缝"——在三维空间中的片状密度降低 | center, orientation, length, width |

### 2.3 奇幻特征的世界生成规则

不是"随机放置"——奇幻特征的出现由**魔法浓度 + 群系 + 高度**决定：

```
浮空岛生成条件: 魔法浓度 > 0.7 AND 高度 > 500m AND 气流强度 > 0.5
巨菇林生成条件: 魔法浓度 > 0.4 AND 温带/热带森林群系
水晶峡谷: 魔法浓度 > 0.6 AND 山脉群系 AND 高度 > 300m
虚空裂隙: 魔法浓度 > 0.9 AND 高度 > 2000m
```

这些条件确保奇幻元素不是随机拼接——它们出现在"魔法浓度高的地方"——这本身是一个连贯的世界构建逻辑。

## 三、高空生态方案：垂直分层群系

### 3.1 群系分类的扩展

在 013 的"温度-降水"二维空间之外，增加**第三维度——高度**：

```rust
struct BiomeId(u16);

enum VerticalLayer {
    DeepUnderground,   // y < -200m
    ShallowUnderground, // -200m < y < -50m
    Surface,            // -50m < y < 100m    ← 013 的群系在此层
    LowSky,             // 100m < y < 500m

fn classify_biome(world_pos: Vec3, seed: u64) -> BiomeId {
    let layer = VerticalLayer::from_y(world_pos.y);

    }

```

### 3.2 天空群系的具体示例

| 群系 | 高度范围 | 特征 |
|------|---------|------|
| **云中森林** | 1500-2500m | 生长在浮空岛上的巨型树木、瀑布从岛边缘倾泻而下 |
| **风暴区** | 800-2000m | 永久的雷暴、强风、低可见度——危险但有独特资源 |
| **星辰浮岛** | 3000m+ | 小型浮空岛链、极光、低重力感、稀有魔法材料 |
| **天空草原** | 500-1200m | 在平坦的山顶或大型浮空岛上的草原——有独特的天空生物 |
| **虚空边界** | 4000m+ | 世界的"天花板"——在这里世界逐渐融入虚空。极高魔法浓度 |

### 3.3 垂直过渡

地表群系不突然跳转到天空群系——在山坡上有一个**过渡带**：

```
y=100m: 地表温带阔叶林
y=300m: 混合——仍然有阔叶树，但开始出现更耐寒的针叶树，雾气更浓
y=500m: 低空层——树木变矮，开始出现天空草原和岩壁特征
y=800m: 中空层——完全不同的生态系统
```

这个过渡不是硬编码的——它是**温度随海拔下降 + 魔法浓度随海拔上升**这两个连续函数自然产生的结果。

## 四、高速移动方案：GPU 加速 + 预测性加载 + 降级策略

### 4.1 GPU 噪声生成——突破 CPU 生成瓶颈

```
当前瓶颈: CPU 噪声生成 → 32³ 体素 × 5 层噪声 = 163840 次噪声采样/chunk
          → CPU 耗时 ~1-2ms/chunk

GPU 方案: compute shader 一次 dispatch = 32×32×32 = 32768 个线程
          → GPU 耗时 < 0.05ms/chunk

管线：
  1. CPU: 收到新 chunk 请求 → 确定种子 + 参数
  2. GPU: dispatch compute shader → 输出密度值 buffer
  3. GPU: readback 到 CPU（或保留在 GPU 上做 Marching Cubes）
  4. CPU: Transvoxel mesh 生成（使用已计算的密度值）
```

**关键**：步骤 3 的 GPU→CPU readback 有延迟（~1ms）。对于附近 chunk——这个延迟可以接受。对于远处 chunk——使用 heightfield 替代（更快）。

### 4.2 分速加载策略

不同的移动速度 → 不同的加载行为：

```rust
enum LoadingProfile {
    /// 步行/骑马——加载半径 150m，所有 LOD 完整生成
    Normal { radius: f32 },
    /// 飞龙/快速飞行——加载半径 400m，优先低 LOD → 高 LOD 逐步细化
    Fast { radius: f32, lod_boost: u8 },
    /// 魔法传送/极速——加载半径 800m，只加载 heightfield 和 impostor
    Extreme { radius: f32 },
}

impl LoadingProfile {
    fn from_player_speed(speed: f32) -> Self {
        match speed {
            s if s < 20.0 => Self::Normal { radius: 150.0 },
            s if s < 100.0 => Self::Fast { radius: 400.0, lod_boost: 2 },
            _ => Self::Extreme { radius: 800.0 },
        }
    }
    
    fn lod_offset(&self) -> u8 {
        // 速度越快 → 初始加载的 LOD 越高（先低精度 → 再细化）
        match self {
            Self::Normal { .. } => 0,
            Self::Fast { lod_boost, .. } => *lod_boost,
            Self::Extreme { .. } => 4,
        }
    }
}
```

### 4.3 预测性预加载

```rust
struct PredictiveLoader {
    /// 基于速度向量的预测路径
    predicted_path: Vec<(Vec3, f32)>,  // (world_pos, time_from_now)
}

impl PredictiveLoader {
    fn update(&mut self, player_pos: Vec3, player_velocity: Vec3) {
        self.predicted_path.clear();
        
        // 简单物理预测：匀速直线运动
        // 更精确的预测需要输入状态（是否在转向、是否在加速）
        for t in [0.5, 1.0, 2.0, 3.0, 5.0, 10.0] {
            let predicted_pos = player_pos + player_velocity * t;
            self.predicted_path.push((predicted_pos, t));
        }
    }
    
    fn get_loading_priority(&self, chunk_center: Vec3) -> f32 {
        let mut priority = 0.0;
        for (predicted_pos, time) in &self.predicted_path {
            let dist = (chunk_center - *predicted_pos).length();
            // 离预测路径越近、时间越紧迫 → 优先级越高
            priority += 1.0 / (1.0 + dist) * (10.0 / (1.0 + time));
        }
        priority
    }
}
```

### 4.4 降级策略：先显示低质量，再逐步细化

```
高速飞行时的时间线：

t=0: 玩家加速到飞行速度 → LoadingProfile 切换到 Fast
t=0.1s: 前方 400m 内的 chunk 被调度
t=0.2s: 最近的 chunk 完成了"noise generation + heightfield"（< 0.1ms/chunk on GPU）
        → 这些 chunk 以 LOD 6 (32m/体素 heightfield) 显示
        → 玩家看到远处的山脊轮廓——可以正常飞行
t=0.5s: 近处的 chunk 完成了 Transvoxel LOD 0-1
        → 替换 heightfield 为完整 mesh
t=1.0s: 中距离的 chunk 完成了 Transvoxel LOD 2-3
        → 玩家的感知：远山在逐渐变清晰——这是"自然的"
```

玩家永远**不会看到"空洞"**——最坏情况下看到低精度的近似，但地形总是在那里的。

### 4.5 感知测试

验证高速移动体验的测试场景：

```
测试 1: 步行探索 (5 m/s)
  → 从未看到未加载的 chunk（及格线）

测试 2: 骑马奔驰 (15 m/s)
  → 偶尔看到 LOD 5 的 heightfield——但 1s 内被高 LOD 替换（可接受）

测试 3: 飞龙俯冲 (100 m/s, 高度 500m)
  → 地面是 LOD 5-6 的 heightfield——但高度使得细节不可见（足够）
  → 近处的浮空岛是 LOD 0-1 的完整 mesh

测试 4: 魔法传送/急速 (瞬间位移 1km)
  → 1-2s 的"世界加载中"黑屏过渡——与传送动画配合（可接受的重负载场景）
```

## 五、极高世界高度方案：稀疏 3D Chunk 网格

### 5.1 数据结构

```rust
struct WorldChunkGrid {
    // 3D 稀疏存储
    chunks: HashMap<(i32, i32, i32), ChunkState>,
    
    // 快速查询"从 (x,z) 列中哪个 y 有 chunk"
    // 用于：从天空俯视 → 需要知道正下方最近的地表 chunk
    surface_columns: HashMap<(i32, i32), BTreeSet<i32>>,
    
    // 每个 (x,z) 列的地表高度缓存
    // 用于：快速"射线投射"——玩家脚下的地面在哪里
    surface_heights: LruCache<(i32, i32), f32>,
    
    // 世界高度配置
    height_config: WorldHeightConfig {
        min_y: f32,       // -2000m
        max_y: f32,       // +4000m
        sea_level: f32,   // 0m
        chunk_size: f32,  // 32m
    },
}

enum ChunkState {
    /// 纯空气——没有密度 > 0 的体素在这个位置
    Air,
    /// 需要生成——有特征（地形/浮空岛/建筑）在这个位置
    Pending,
    /// 已生成密度场但未 mesh
    Generated { density: Box<[f32; CHUNK_VOXELS]> },
    /// 完全就绪——有 mesh + 碰撞
    Loaded { mesh_data: ChunkMeshData },
}
```

### 5.2 稀疏性如何工作

```
世界初始化（给定种子）:
  1. 生成地表高度图（对所有 (x,z) 的 2D 噪声采样 → 知道每个位置的地表高度）
     → 这非常快——只采样一次 2D 噪声，不需要做 3D 采样
  2. 生成奇幻特征的 bounding box——知道浮空岛存在于哪些 (x, y, z) 区域
  3. 对于每个 (x,z) 列：标记地表附近的 ~5 个 chunk 为 Pending
  4. 对于浮空岛覆盖的 (x,z) 列：标记岛体覆盖的 ~10-20 个 chunk 为 Pending
  5. 其余所有 (x,y,z) → Air（不需要生成、不需要存储、不需要内存）
```

这避免了为 6000m 高度范围的 188 层 chunk 分配空间——99% 的 chunk 标记为 Air，占用 1 字节/chunk（一个枚举 Tag）。

## 六、地面高速移动方案：碰撞优先 + 扇形优先

### 6.1 与空中高速移动的加载差异

地面高速移动（~50-80 m/s）的核心挑战是：玩家贴近地表——需要近景的高精度 mesh 和准确的碰撞体。

**碰撞优先管线**：

```
正常: 噪声 → 密度场 → Transvoxel mesh → 碰撞体 → 就绪
地面高速: 噪声 → 密度场 → 快速碰撞体（粗粒度heightfield）→ 标记为"可通行"
              ↓（0.1-0.3s 后）
         Transvoxel mesh → 精确碰撞体 → 替换
```

在碰撞体就绪但 mesh 未就绪的短暂窗口内——玩家看到的是 LOD 2-3 的低精度 mesh（已经可以渲染），碰撞使用的是粗粒度 heightfield（碰撞在物理上正确，但可能不够精确——比如一块小石头暂时没有碰撞，玩家可以穿过。0.3s 后精确碰撞替换）。

### 6.2 扇形加载优先级

地面高速时——玩家的视野集中在前方 ±30° 扇形区域。后方和侧后方的 chunk 加载优先级大幅降低。

```rust
fn ground_speed_bias(chunk_center: Vec3, player_pos: Vec3, velocity: Vec3) -> f32 {
    let to_chunk = (chunk_center - player_pos).normalize();
    let forward = velocity.normalize();
    let dot = to_chunk.dot(forward);
    
    // 正前方 dot≈1.0 → 优先级 ×2.0
    // 侧面 dot≈0.0 → 优先级 ×0.5  
    // 后方 dot≈-1.0 → 优先级 ×0.1
    (dot + 1.0).clamp(0.1, 2.0)
}
```

## 七、天体周期系统方案

### 7.1 时间管理核心

在 Rust 模拟核心中新增 `TimeManager`——它是所有时间相关系统的单一真相来源。

```rust
struct TimeManager {
    /// 世界创建后的总秒数
    elapsed_seconds: f64,
    /// 一天的秒数（游戏时间）——可配置，默认 1440s = 24分钟
    seconds_per_day: f64,
    /// 一年的天数——可配置
    days_per_year: u32,
    /// 当前年份
    current_year: u32,
}

impl TimeManager {
    /// 每日的时间进度 0.0-1.0（0=午夜 0.25=日出 0.5=正午 0.75=日落）
    fn time_of_day(&self) -> f32 {
        (self.elapsed_seconds % self.seconds_per_day) as f32 / self.seconds_per_day as f32
    }
    
    /// 一年中的天数 1-365
    fn day_of_year(&self) -> u32 {
        (self.elapsed_seconds / self.seconds_per_day) as u32 % self.days_per_year + 1
    }
    
    /// 季节（基于纬度——赤道常年夏季，两极常年冬季，温带四季分明）
    fn season_at(&self, latitude: f32) -> Season {
        let day = self.day_of_year();
        let lat_offset = (latitude / 90.0) * 90.0; // 纬度偏移天数
        let effective_day = (day as f32 + lat_offset) % self.days_per_year as f32;
        
        let quarter = self.days_per_year as f32 / 4.0;
        match effective_day {
            d if d < quarter => Season::Spring,
            d if d < quarter * 2.0 => Season::Summer,
            d if d < quarter * 3.0 => Season::Autumn,
            _ => Season::Winter,
        }
    }
}

enum Season { Spring, Summer, Autumn, Winter }
```

### 7.2 部分区域只有 1-2 季

热带（纬度 < 15°）：常年夏季——或雨季/旱季两季轮替。
寒带（纬度 > 75°）：常年冬季——短暂的夏季窗口（几周）。
温带（15°-75°）：四季分明。

```rust
fn season_count_at(&self, latitude: f32) -> u8 {
    let abs_lat = latitude.abs();
    if abs_lat < 15.0 { 2 }       // 雨季/旱季
    else if abs_lat < 75.0 { 4 }  // 四季
    else { 1 }                     // 常年冬季
}
```

### 7.3 天体渲染（Godot 侧）

```gdscript
# Godot Sky shader 的天体参数——每帧从 Rust 更新

    var season = time_manager.season_at(player_latitude)

### 8.3 天气的视觉实现（Godot 侧）

| 天气效果 | 实现方式 |
|---------|---------|
| 雨 | `GPUParticles3D`——粒子从天空向地面降落。雨的密度和粒子速度由 `WeatherState.intensity` 控制 |
| 雪 | 同样的粒子系统——更慢的下落速度 + 更大的阻力 + 白色材质 |
| 风暴/沙尘 | 降低 Godot Environment 的 visibility 参数 + 风粒子 + 天空颜色变暗 |
| 地面湿润 | 在 chunk 渲染时应用一个临时的深色材质覆盖（湿度高的区域地面颜色偏暗） |
| 积雪 | `WeatherDensityOverlay` 在体素表面叠加 ~0.1-0.5m 的白色密度——经过 Transvoxel 后在地形表面产生可见的雪层 |

### 10.1 多源加载优先级

struct LoadingSource {
    /// 来源类型
    source_kind: LoadingSourceKind,
    /// 当前位置
    position: Vec3,
    /// 当前速度

}

    /// 主玩家——最高优先级，LOD 0 必需
    MainPlayer,
    /// 玩家队伍中的 NPC 同伴——高优先级
    PartyMember { npc_id: u64 },
    /// 无关的高速 NPC——低优先级，只需 LOD 2+
    IndependentNpc { npc_id: u64 },
}
```

### 10.2 合并加载请求

每个加载源产生自己的 chunk 需求，合并器去重并取"最紧急"的优先级：

```rust
fn merge_loading_requests(sources: &[LoadingSource]) -> Vec<ChunkRequest> {
    let mut requests: HashMap<ChunkCoord3D, ChunkRequest> = HashMap::new();
    
    for source in sources {
        let predicted_path = predict_path(source.position, source.velocity);
        for (chunk_coord, urgency) in chunk_along_path(&predicted_path) {
            let priority = urgency * source.priority_multiplier;
            let min_lod = source.min_lod;
            
            requests.entry(chunk_coord)
                .and_modify(|r| {
                    // 取最高优先级
                    r.priority = r.priority.max(priority);
                    // 取最低 LOD（最严格要求）
                    r.min_lod = r.min_lod.min(min_lod);
                })
                .or_insert(ChunkRequest { chunk_coord, priority, min_lod });
        }
    }
    
    requests.into_values().sorted_by_key(|r| -r.priority).collect()
}
```

### 10.3 性能保障

NPC 的高速移动加载不能影响玩家的体验。加载预算被严格划分：

- **玩家加载预算**：总预算的 70%——确保玩家的体验始终优先
- **同伴加载预算**：20%——玩家关心的 NPC
- **无关 NPC 加载预算**：10%——只保证最基本的 LOD 3+ 地形存在

struct RevisedDensityField {

    // 时间引用——用于季节化群系（014 新增）
    time_manager: Arc<TimeManager>,

impl RevisedDensityField {
    fn sample(&self, pos: Vec3) -> f32 {
        // 高度裁剪——超出世界范围的永远为空气
        if pos.y < self.height_config.min_y || pos.y > self.height_config.max_y {
            return -1.0;
        }
        
        // 各层叠加——按优先级从低到高
        let mut density = self.base_terrain.sample(pos, self.time_manager.season()); // 季节影响基础纹理
        density += self.geological.sample(pos);
        density += self.fantasy.sample(pos);
        density += self.cultural.sample(pos);
        density += self.npc_mods.sample(pos);
        density += self.player_mods.sample(pos);
        density += self.weather_overlay.sample(pos);  // 🆕 天气临时效果
        
        density
    }
}
