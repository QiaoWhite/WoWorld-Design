# 003-可行性评估与 013 方案整合

> 十一需求的可行性独立评估 + 与 013 四阶段计划的整合方式

---

## 一、逐项可行性评估

### 1.1 地平线远景（5-10km）

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **完全可行**。8 级 LOD + heightfield + impostor 是成熟技术——Minecraft 的 Far Plane、Horizon Zero Dawn 的地形 impostor 都经过验证 |
| 性能可行性 | ✅ LOD 6-8 的三角形数极低——5km 外所有 chunk 加起来可能不到 50K tris。Godot Forward+ 对此毫无压力 |
| 视觉可信度 | ✅ 大气散射 + 高度雾 + 距离雾——三个效应的组合可以无缝隐藏 LOD 过渡。这在 Godot 中只需配置 Environment 节点 |
| Solo 工作量 | 🟡 中。LOD 4-5 是 Transvoxel 的自然扩展（增加查找表行数）。LOD 6-7 的 heightfield 生成 ~200 行 Rust。LOD 8 的 impostor 是最简单的部分（12 张静态纹理） |
| 风险 | 🟢 低。如果时间不够，LOD 4-8 可以分 Phase 实现——Phase 2 只到 LOD 3，Phase 4 再扩展到 8 |

**结论：完全可行。最不紧急但最提升体验的功能。建议在 Phase 4 实现。**

### 1.2 奇幻地形

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **完全可行**。`DensityProvider` trait + 空间索引已经是一个标准的可扩展架构。浮空岛、巨菇、水晶的密度函数都是成熟的计算几何 |
| 视觉可信度 | ✅ 奇幻特征由"魔法浓度 + 群系 + 高度"控制——不是随机放置，而是有世界构建逻辑支撑 |
| 与涌现哲学的一致性 | ✅ 奇幻特征**不是脚本化的**——它们是由参数（魔法浓度、群系、高度）和规则（什么条件下出现什么特征）产生的。不同的种子 → 不同的魔法浓度分布 → 不同的奇幻特征分布 |
| Solo 工作量 | 🟢 低。每个特征类型的密度函数 ~50-100 行 Rust。LLM 可以生成 90%。注册表和空间索引是通用代码 |
| 风险 | 🟢 低。即使不做任何奇幻特征——013 的基础地形仍然可以产生美丽的自然景观 |

**结论：低成本高回报。建议在 Phase 2-3 之间作为"趣味性增强"插入。**

### 1.3 高空生态群系

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **可行**。垂直分层群系只是"在群系分类函数的输入中增加 y 坐标"——没有新的算法 |
| Chunk 管理 | 🟡 需要从 2D 改为 3D 稀疏——但稀疏性保证了不增加内存开销 |
| 视觉表现 | 🟡 天空群系需要专门的视觉资产（天空植被、浮空岛纹理、云效果）——这些资产是瓶颈，技术不是 |
| Solo 工作量 | 🟡 中。3D 稀疏 chunk 网格 ~500 行 Rust。垂直群系分类 ~300 行。其余是资产制作 |
| 风险 | 🟢 低。天空群系是可选的——如果资产不足，可以先只做 1-2 个天空群系原型 |

**结论：技术可行。资产是瓶颈。建议在 Phase 4 作为"天空扩展"实现，Phase 2-3 只实现到 500m 高度。**

### 1.4 高速移动

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **可行**。GPU 噪声生成 + 预测性预加载 + 降级策略——三重保障 |
| GPU 噪声的瓶颈 | 🟡 readback 延迟 ~1ms——如果 100 个 chunk 都需要 GPU 噪声 → 100×1ms = 100ms。**缓解**：GPU 噪声只用于附近 chunk（LOD 0-2）。远处 chunk 用 CPU heightfield 或预缓存的低精度噪声 |
| 预测性预加载 | 🟡 "玩家突然转向"——预测失效 → 前方 chunk 不在预加载队列 → 看到低 LOD。**缓解**：在转向时，丢弃旧的预测路径，立即加载新方向——可能看到 0.1-0.5s 的低 LOD |
| 降级策略的视觉影响 | ✅ 飞龙飞行时——从高空俯瞰，低 LOD heightfield 足够。地面细节不可见 |
| Solo 工作量 | 🔴 高。这是五个需求中最复杂的。GPU 噪声 compute shader ~200 行 WGSL + 200 行 Rust 集成。预测性加载 ~300 行。分速策略 ~200 行。总计 ~900 行——但调试和性能调优可能需要 2-4 周 |
| 风险 | 🟡 中。需要在 Phase 2 就做"高速移动的性能基准测试"——不要等到 Phase 4 才发现达不到 |

**结论：技术上可行但工作量最大。建议分两步：Phase 3 做分速策略 + 预测加载（CPU 噪声）→ Phase 4 加入 GPU 噪声加速。**

### 1.5 极高世界高度

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **完全可行**。3D 稀疏 chunk 网格——99% 的 chunk 是 Air（1 字节标记），没有内存浪费 |
| 与现有代码的兼容 | ✅ `LayeredDensityField.sample(pos)` 对任何 pos 返回密度——不需要知道"这个 pos 在哪个 chunk 里"。Chunk 管理对采样逻辑透明 |
| 视觉表现 | ✅ 从高空俯视地表——远处用 LOD 6-8 的 heightfield + impostor。浮空岛和天空群系在近处用完整 mesh |
| Solo 工作量 | 🟡 中。2D→3D chunk 坐标改变 ~200 行。垂直分层群系 ~300 行。其余与"高空生态"共享 |
| 风险 | 🟢 低。即使世界高度先限制在 -200m 到 +500m（013 默认），代码接口为未来扩展预留了空间 |

**结论：完全可行。建议 Phase 3 引入 3D chunk 网格，Phase 4 扩展到全高度范围。**

### 1.6 地面高速移动（50-80 m/s）

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **可行**。空中高速方案的大部分技术（预测加载、分速策略、降级质量）可以复用——关键是增加"碰撞优先管线" |
| 碰撞优先管线 | ✅ 噪声 → 粗碰撞体（<0.05ms）→ 低精度 mesh → 精确 mesh。在碰撞就绪前（~0.1-0.3s），使用相邻 chunk 的碰撞体延伸 |
| 碰撞-渲染时间差 | 🟡 短暂窗口内可能看到"没有视觉但碰撞存在"的地形特征。对沉浸感有 0.1-0.3s 的轻微影响 |
| Solo 工作量 | 🟡 中。碰撞优先管线 ~200 行 Rust。扇形加载优先级 ~100 行。与空中高速共享 80% 的代码 |
| 风险 | 🟢 低。如果碰撞优先管线达不到要求——降级为"先显示低 LOD mesh + 0.3s 后高 LOD"，玩家仍然可以正常移动 |

**结论：空中高速的技术方案可以直接支撑地面高速。Phase 3 同步实现。**

### 1.7 天体周期系统

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **完全可行**。`TimeManager` 是纯数学计算——太阳/月亮角度由时间参数驱动。Godot `Sky` shader 原生支持程序化天空 |
| 季节计算 | ✅ 基于纬度 + 太阳角度的季节判断——标准地理模型，实现简单 |
| 部分区域 1-2 季 | ✅ 热带纬度检查 → 两季轮替。寒带纬度 → 常年冬季。纯条件分支 |
| Solo 工作量 | 🟢 低。`TimeManager` ~150 行 Rust。天体位置计算 ~100 行。Godot shader 参数更新 ~50 行 |
| 风险 | 🟢 低。`TimeManager` 是独立模块——不影响任何现有系统 |

**结论：极低成本。建议 Phase 2 实现基础时间系统 + 太阳/月亮周期。Phase 4 加入月亮相位和星星自定义。**

### 1.8 季节与天气系统

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **可行**。天气状态机 + 群系约束规则——不是复杂算法 |
| 天气-体素耦合 | 🟡 积雪和积水需要 `WeatherDensityOverlay`——对体素 mesh 有实际影响。但这是临时层——不需要存档，天气变化时自动重新计算 |
| 天气对 NPC 的影响 | ✅ 天气通过事件总线通知 NPC 系统——NPC 系统独立决定响应方式。体素引擎不需要知道 |
| Solo 工作量 | 🟡 中。`WeatherManager` ~300 行。`WeatherDensityOverlay` ~150 行。天气粒子在 Godot 侧——Godot 内置 |
| 风险 | 🟢 低。与 `TimeManager` 同步实现——两个模块天然耦合 |

**结论：Phase 3 实现基础天气。Phase 4 加入天气-体素耦合（积雪/积水）。**

### 1.9 环境互动（农作物/生物/季节/天气）

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **可行**——但大部分超出了体素引擎的范围。属于 NPC 系统和独立子系统 |
| 体素引擎的责任 | 仅提供 `WeatherDensityOverlay`（积雪深度、地面湿润度）——供其他系统查询 |
| NPC 系统的责任 | 接收天气事件 → 修改行为权重（雨天减少户外活动） |
| 农作物系统 | 需要独立设计——不在 013/014 的范围内 |
| Solo 工作量 | 🟢 低（对体素引擎而言）。其他子系统的接口预留 ~100 行。实际实现属于各自的子系统 |
| 风险 | 🟢 低。这是接口设计问题——不是算法问题 |

**结论：体素引擎的责任边界清晰——提供天气影响的密度数据。其余由 NPC/农作物系统自行处理。Phase 4。（注：农作物系统需要独立的设计文档）**

### 1.10 天体自定义

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **完全可行**。TOML 配置文件 → Rust 解析 → Godot shader 参数。数据驱动的最纯粹应用 |
| Mod 友好度 | ✅ 这正是 TOML 数据驱动设计（011-002）的完美用例——Modder 修改一个 `.toml` 文件，天空就变了 |
| Solo 工作量 | 🟢 低。TOML schema 定义 + `CelestialConfig` 解析 ~100 行。Godot shader 参数映射 ~100 行 |
| 风险 | 🟢 低。如果 Godot `Sky` shader 不支持某些自定义需求——可以用自定义 full-screen shader 替代 |

**结论：Phase 2 做数据接口预留。Phase 4 实现完整自定义。**

### 1.11 NPC 高速移动

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ **可行**。多源加载系统是玩家加载系统的简单扩展——每个加载源有自己的优先级和最低 LOD |
| 对玩家加载的影响 | 🟡 多个高速 NPC → 加载需求增加。**缓解**：加载预算硬分割——玩家 70%、同伴 20%、无关 NPC 10% |
| NPC 的最低 LOD | ✅ NPC 不需要 LOD 0——LOD 2-3 对导航完全足够。这大幅降低了 NPC 加载的 GPU 压力 |
| Solo 工作量 | 🟡 中。多源合并加载 ~200 行。NPC 加载预算 ~100 行 |
| 风险 | 🟢 低。最坏情况——无关 NPC 的 LOD 降到 LOD 5。NPC"看到"低精度地形——但 NPC 不会抱怨 |

**结论：Phase 3 实现基础多源加载。Phase 4 扩展到全 NPC 高速移动。**

## 二、与 013 四阶段计划的整合

### 修订后的实施路线

```
Phase 1 (2-4 周): 最小可行管线——不变
  □ 单 chunk 球体密度场 → Godot 渲染
  □ 接口设计时为 3D chunk 坐标预留字段
  □ TimeManager 的 trait 接口预留（不实现）
  ✅ 里程碑：一个由 Rust 生成的体素 mesh 在 Godot 中渲染

Phase 2 (4-8 周): 噪声地形 + 基础 LOD + 时间天体 — 扩展
  原：4 级 LOD + 2D chunk
  新：🆕 LOD 0-3 的 Transvoxel 管线
      🆕 LOD 4-5 的 CPU heightfield fallback
      🆕 基础的大气雾设置（Godot Environment）
      🆕 奇幻特征的 DensityProvider trait + 第一个浮空岛原型
      🆕 TimeManager 基础（昼夜循环 + 太阳/月亮位置）
      🆕 CelestialConfig 数据接口（TOML schema 预留）
      🆕 3D 稀疏 chunk 坐标（为 Phase 4 预留）
      □ 世界高度限于 y=-200m 到 y=+500m
  ✅ 里程碑：昼夜交替 + 可探索的自然地形 + 第一个浮空岛

Phase 3 (4-8 周): 分层密度场 + 修改系统 + 高速 + 天气 — 扩展
  原：分层密度 + SDF 修改 + 存档
  新：🆕 分速 LoadProfile (Normal/Fast/Extreme + Ground/Air 模式)
      🆕 预测性预加载（基于速度向量）
      🆕 碰撞优先管线（地面高速的碰撞先于 mesh）
      🆕 地面高速的扇形加载优先级
      🆕 多源加载系统（玩家 + 同伴 + 无关 NPC 的加载预算分配）
      🆕 WeatherManager 基础（天气状态机 + 群系约束规则）
      🆕 天气粒子（雨/雪——Godot GPUParticles3D）
      🆕 GPU 噪声基础设施（compute shader 原型）
      □ 高速移动性能基准测试（模拟 100 m/s 空中 + 80 m/s 地面）
  ✅ 里程碑：飞龙速度下的世界加载跟上玩家 + NPC；天气系统运作

Phase 4 (长期): 完整管线 + 天体自定义 + 天空生态 — 扩展
  原：洞穴、LOD、生物群系、GPU 加速
  新：🆕 LOD 6-8 的 heightfield + impostor
      🆕 垂直生态分层（地表→低空→中空→高空→极高空）
      🆕 世界高度扩展到 y=-2000m 到 y=+4000m
      🆕 天空群系（云中森林、虚空边界、星辰浮岛）
      🆕 GPU 噪声生成的生产级实现
      🆕 完整的奇幻特征库（10-20 种类型）
      🆕 SeasonManager（四季 + 部分地区 1-2 季）
      🆕 WeatherDensityOverlay（积雪/积水——天气对体素的物理影响）
      🆕 天体完整自定义（Modder TOML → Godot sky shader）
      🆕 天气对 NPC 行为的影响接口
  ✅ 里程碑：从深海到虚空的完整世界——昼夜交替、四季轮转、天气变化、浮空岛上的日落
```

## 三、优先级矩阵

| 需求 | 对体验的增益 | 技术复杂度 | 建议 Phase | 理由 |
|------|-----------|----------|----------|------|
| **天体周期（昼夜）** | 🔴 核心 | 低 | Phase 2 | `TimeManager` ~250 行。无昼夜的世界没有"活着"的感觉 |
| **奇幻地形** | 🟡 亮眼 | 低 | Phase 2 | 浮空岛 = 100 行 Rust。立竿见影 |
| **高速移动（空中+地面）** | 🔴 核心 | 高 | Phase 3 | 飞龙+魔法飞行是核心体验——不能等 Phase 4 |
| **NPC 高速移动** | 🟡 重要 | 中 | Phase 3 | 与玩家高速共享代码——边际成本低 |
| **季节系统** | 🟡 沉浸感 | 中 | Phase 4 | 依赖 TimeManager——但群系参数+视觉资产需要更多准备 |
| **天气系统** | 🟡 沉浸感 | 中 | Phase 3-4 | 基础天气 Phase 3；体素耦合 Phase 4 |
| **环境互动（农作/生物）** | 🟢 锦上花 | 低（接口） | Phase 4 | 体素引擎仅提供接口——实际实现在各自子系统 |
| **天体自定义** | 🟢 锦上花 | 低 | Phase 4 | TOML 数据驱动——Modder 最爱 |
| **远景（5-10km）** | 🟡 沉浸感 | 中 | Phase 4 | 先跑起来——雾暂时遮住远处 |
| **高空生态** | 🟢 锦上花 | 中 | Phase 4 | 依赖天空资产 |
| **极高高度** | 🟢 锦上花 | 中 | Phase 4 | 与高空生态同步 |

## 四、风险总结

| 风险 | 等级 | 触发条件 | 缓解 |
|------|------|---------|------|
| GPU 噪声 readback 成为瓶颈 | 🟡 中 | Phase 3 性能测试时发现 > 50ms | 降级到 CPU heightfield + 限制 GPU 噪声只用于 LOD 0-1 |
| 3D 稀疏 chunk 的内存碎片 | 🟢 低 | 100K+ 活跃 chunk | HashMap 本身就处理稀疏——额外开销 < 10MB |
| LOD popping 在高空视角下更明显 | 🟡 中 | Phase 4 | 高空有大气散射作为天然遮罩——可能被自然隐藏 |
| 世界高度扩展导致噪声参数需要重调 | 🟢 低 | Phase 4 | 噪声参数是数据文件——不是硬编码 |
| 地面高速碰撞延迟导致穿模 | 🟡 中 | Phase 3 碰撞优先管线测试 | 0.3s 窗口内使用相邻 chunk 的碰撞体延伸——如果不够，增加 fallback 碰撞半径 |
| 天气-体素耦合增加 chunk 重新生成频率 | 🟢 低 | Phase 4 | 天气临时覆盖层是独立层——基础 mesh 不需要重新生成。Godot shader 处理颜色变化 |
| 多 NPC 高速移动耗尽加载预算 | 🟡 中 | Phase 3 多源测试时 > 50 个 NPC 同时高速 | 无关 NPC 的加载预算硬限制在 10%——如果不够，降级无关 NPC 到 LOD 5+ |

---

> **最终结论：十一个需求全部可行。核心变化：Phase 2 加入天体周期（成本极低效果极大），地面高速与空中高速在 Phase 3 合并实现（共享 80% 代码），季节/天气/天体自定义/高空生态在 Phase 4 作为完整的世界"活着感"打包上线。**
