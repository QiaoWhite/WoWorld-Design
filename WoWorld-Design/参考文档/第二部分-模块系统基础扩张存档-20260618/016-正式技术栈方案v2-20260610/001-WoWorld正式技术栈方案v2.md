## 一、设计哲学的技术映射

| 哲学原则 | 技术映射 |
|---------|---------|
| **只创造规则，不创造故事** | 参数驱动的涌现引擎——GOAP + 概率决策器。Mod 通过 TOML 调节倾向性浮点乘数，**不编写行为脚本**。没有 Rhai/Lua/Python 脚本引擎 |
| **创建另一个"人世"** | 100K NPC 的完整模拟——每个 NPC 有独立的心智、记忆、关系。分层模拟 (L1/L2/L3) 使这可行 |
| **每个存档都是新世界** | 种子确定性——地形、NPC、文化全部从种子参数化生成。存档仅保存与种子的差异 |
| **全球多元文化自然演化** | 文化种子参数 → 漂移/分化/融合 → 建筑/服饰/饮食的全链路影响。不由开发者预设模板 |
| **冒险与生活平等** | 所有玩法由同一模拟核心支撑——铁匠和冒险者在同一套规则下获得同等的系统深度 |
| **Mod 是涌现的延伸** | 数据驱动（TOML）+ 参数调节——Modder 创作的是"世界的新参数"，不是"NPC 的新脚本"。Modder 做的事情和开发者做的事情在本质上是相同的：调节参数让不同的涌现结果出现 |

### 事件总线协议

- **同步事件**（帧内）：`NpcActionCompleted`、`GoalAchieved`、`CollisionDetected`——必须在同一帧内处理
- **异步事件**（帧间）：`WeatherChangeEvent`、`SeasonChangeEvent`、`ModDataReloaded`——可以延迟到下一帧
- **FIFO 队列 + 优先级**：紧急事件（NPC 生存危机）插队到队列头部
- **所有子系统通过 trait 订阅**：`trait EventSubscriber { fn on_event(&mut self, event: &Event); }`

## 四、体素世界生成

### 4.1 分层密度场

```
最终密度 = 基础地形层(种子→噪声→生物群系)
         + 地质特征层(洞穴/悬崖/矿脉)
         + 奇幻特征层(DensityProvider trait 可插拔)
         + 文化覆盖层(建筑地基/道路/农田)
         + NPC修改层(挖掘/踩踏——批量提交合并重生成)
         + 玩家修改层(SDF 雕刻)
         + 天气临时层(积雪/积水——每chunk每30s最多一次重新mesh)
```

**NPC-Terrain 交互**：
```rust
// NPC 挖掘：SDF 操作先缓冲到队列，批量提交
npc_mod_queue.push(SdfOperation::SubtractSphere { center: dig_pos, radius: 0.5 });
// 每 0.5s 或每 50 个操作 → 批量提交到 chunk 的 ModificationBuffer
// 脏 chunk 重新 mesh 被节流到每 chunk 每 30s 最多一次
```

### 4.2 噪声基准参数

> **依据**：017 测试 Phase 1 确认——少于 2 层噪声或总海拔 <10m/500m 视距 → 地形不可感知为"平的"。

| 参数 | 值 | 说明 |
|------|-----|------|
| 低频骨架层频率 | 0.003-0.008 | 大尺度地形结构（山丘群/谷地/高地），500m 视距内 ≥1 个完整周期 |
| 高频细节层频率 | 0.03-0.06 | 小起伏（脚下的变化） |
| 低频骨架振幅 | 10-20m | 创造可感知的"高地"vs"谷地"差异 |
| 高频细节振幅 | 3-5m | 行走时的微地形变化 |
| 500m 视距内海拔总范围 | ≥10m | 确保地形"有结构"的最低阈值 |
| 噪声类型 | Simplex (默认) / Perlin (备选) | — |

**自动校验**：世界生成时检查 `max_elevation(500m_radius) - min_elevation(500m_radius) ≥ 10m`——不满足则重新采样种子。防止出现测试 #3（种子 0x0003：海拔范围仅 0.55m）的"死亡平板"。

### 4.3 资源分布策略

> **依据**：017 测试 Phase 2 确认——资源均匀分布 → NPC 行为均匀（无聊）。资源聚类 → 活动热点形成（涌现的基础设施）。

| 策略 | 说明 |
|------|------|
| **空间聚类** | 同类型资源集中在特定区域（如食物集中在东侧、水源在西侧）——形成"进食区"、"饮水区"等空间功能分区 |
| **资源日波动** | 每天资源位置和数量 ±20% 随机变化——作为人格放大器（压力下 NPC 行为差异更明显） |
| **初始稀缺** | Phase 3+ 推荐资源量 = 正常需求的 70-80%——适度稀缺驱动社交竞争，为人际事件涌现提供基础 |
| **不均匀分布** | 不同区域的资源密度差异 ≥2×——给 NPC 的日常路径创造有意义的空间结构 |

### 4.4 空间语义标签层

> **依据**：017 测试 Phase 1 确认——NPC 的空间记忆和聚落选址需要地形特征的语义标签（"谷地"、"高地"），而非仅海拔值。

```
地形特征识别器 (TerrainFeatureClassifier):
  输入: 局部高程图 (如 100m×100m 窗口)
  输出: 特征标签
    - 高地 (elevation > local_mean + 1.5σ)
    - 谷地 (elevation < local_mean - 1.5σ)
    - 山脊 (curvature + elevation 双条件)
    - 坡面 (gradient > 阈值)
    - 平地 (gradient < 阈值, elevation ≈ local_mean)
    - 洼地 (local minimum basin)
```

**应用**：NPC 记忆编码（"在谷地的水源边遇到了他"）→ 聚落选址评估（"建在河边的高地上"——防御+水源）→ 战斗地形策略（"占据高地"的战术优势）。Phase 4 开始实现，但数据采集从 Phase 1 地形生成时即可开始。

### 4.5 LOD 体系

```
LOD 0-3: Full Transvoxel (0.5m → 4m/voxel, 0-400m)
LOD 4-5: Transvoxel 低精度 (8m → 16m/voxel, 400m-1.6km)  
LOD 6-7: Heightfield 替代 (32m → 64m/voxel, 1.6km-6.4km)
LOD 8:   Billboard + 大气散射融合 (6.4km+)
```

### 4.3 奇幻特征

`DensityProvider` trait——预置浮空岛、巨菇林、水晶簇、魔法漩涡、浮空瀑布、龙巢、石化森林、虚空裂隙。由魔法浓度 × 群系 × 高度控制分布。

## 五、动画系统

### 5.1 管线

Rust CPU 侧批量计算骨骼矩阵（rayon 并行）→ `PackedByteArray` → Godot `TextureBuffer` → vertex shader 蒙皮。不创建 `AnimationTree` 节点。

### 5.2 Pose Database 资产管线

```
Blender / AI 动作生成
  ↓ 导出 GLTF
Rust `gltf` crate 解析
  ↓ 提取关键帧→PoseEntry（姿态标签+骨骼变换）
  ↓ 序列化为 .pose 二进制文件
Phase 1: 20-30 个基础姿态 (~200KB)
Phase 3: 100 个姿态 (~1MB)
Phase 4: 200 个姿态 (~2MB) — 加载到 GPU TextureBuffer
```

### 5.3 MultiMesh Per-Instance 数据

Godot 4.x 的 `MultiMesh.buffer` 默认每实例 1 个 `vec4`（4 个 float）。需要至少 6-8 个 float（animation_state_id, animation_time, emotion_pleasure, emotion_arousal, fatigue, holding_item_type）。**方案**：使用 `TextureBuffer` 作为扩展 per-instance 数据通道——vertex shader 中 `texelFetch()` 读取。

## 六、时间·天体·季节·天气

### 6.1 TimeManager

独立模块——游戏内时间推进。季节基于太阳赤纬模型（±23.44°）。热带两季，温带四季，寒带常年冬季。

**默认起始时间**：新建存档 = **清晨 05:50**（日出前 10 分钟）。使玩家的第一眼看到日出——"光本身在讲故事"。`/time HH:MM` 开发者命令支持测试跳转。

**昼夜→NPC 行为乘数**（Phase 2 即实现，非推迟至 Phase 5-6）：
- 夜晚（日落后）→"睡眠"概率权重 ×2.0，"户外行为"概率权重 ×0.5
- 白天（日出后）→正常权重
- 这是最简单的"时间→行为"映射——不需要完整的 WeatherManager。Phase 5-6 的天气/季节乘数在此基础上扩展。

### 6.2 天气

`WeatherManager` 状态机 + 群系约束。`WeatherChangeEvent` 通过事件总线广播——携带浮点乘数（如 `outdoor_activity_multiplier: 0.3`）。NPC 概率决策器读取乘数——**不硬编码"下雨→回家"**。

**天气视觉节流**：`WeatherDensityOverlay`（积雪/积水）每 chunk 每 30s 最多触发一次重新 mesh。视觉色彩变化（地面湿润）通过 shader 参数实现——不需要重新 mesh。

## 七、Mod 系统（无脚本引擎）

### 7.1 核心原则

**Modder 调节涌现参数——不编写行为。** 没有 Rhai、Lua、Python 或任何脚本引擎。Modder 做的和开发者做的本质相同：调整参数让不同的涌现结果出现。

| Mod 层级 | 内容 | 实现 |
|---------|------|------|
| Level 1 | 纹理/模型/音频替换 | Godot 资源覆盖 |
| Level 2 | NPC 模板/物品/建筑/天体配置 | TOML 新增 |
| Level 3 | 涌现倾向性乘数调节 | TOML 覆盖浮点乘数（如 `socialize_likelihood_at_noon: 1.5`） |
| Level 4 | 世界规则（文化参数/生成规则） | TOML 高级配置——**需新存档** |

### 7.2 新建存档向导

Level 4 Mod 需要新存档时——游戏引导玩家进入"新建存档向导"：选择启用的 Level 4 Mod → 配置世界种子 → 预览 Mod 效果摘要 → 创建。减少"Mod 不兼容"的困惑。

## 八、NPC 系统架构

### 8.1 模拟核心

无外部 ECS。SoA 布局 + Rust 模块化函数 + rayon 并行。

### 8.2 分层模拟

| | L1 | L2 | L3 |
|---|-----|-----|-----|
| 数量 | ≤1,000 | ≤10,000 | ~89,000 |
| 更新频率 | 每 0.2s | 每 1-2s | 每日/周 |
| 每帧处理量 | 200 个/帧 | 500 个/帧 | 批量统计 |
| 心智 | 情绪+GOAP+记忆+社交+导航 | 情绪衰减+简化状态机 | 命运骰子 |

### 8.3 记忆系统

双层 LMDB：事件事实库（去重）+ 主观记忆库（per-NPC 视角）。每 NPC 上限 2000 条。冷记忆自动摘要压缩——自然的遗忘和记忆错乱基础。"罗生门"效应——不同 NPC 对同一事件保留独立主观版本。

## 九、导航与寻路

### 9.1 三层导航架构

```
Layer 1: 认知优先 — NPC 在熟悉的区域走"记忆路线"（SpatialMemory 中的已验证路径）
         不需要实时寻路——直接沿记忆中的航点移动。O(1)
Layer 2: 局部烘焙 — NPC 在不熟悉的区域 → 对周围 100m 半径的 NavMesh 做局部 A*
         只在必要时触发——目标变化/迷路时。O(n log n)
Layer 3: 体素 A* — L3 NPC（宏观推演）→ 对 2D 可通行性网格做粗粒度 A*
         每日/周运行一次——不要求实时。O(n log n)，但 n 很小
```

### 9.2 高速移动时的导航

高速时（>30 m/s）自动切换到粗粒度导航图（2m 网格替代 0.5m）。精度损失在高速下不可感知——但寻路速度提升 4-16 倍。

## 十、对话系统（技术占位）

对话系统不在此技术方案的完整范围内（需要独立设计文档），但技术接口预留：

```
Rust 侧: DialogueManager
  - 根据 NPC 人格+情绪+关系+记忆 生成对话上下文
  - 关键词/主题匹配 → 选择预制对话模板
  - LLM 可选增强（异步 HTTP → 生成更自然的对话文本）

Godot 侧: DialogueUI (Control 节点)
  - 显示对话文本+选项
  - 情绪对选项的影响标注（愤怒时某些选项不可用）
```

## 十一、性能预算

### 11.1 60fps 帧预算（16.7ms）

| 类别 | 预算 | 说明 |
|------|------|------|
| **Rust 模拟核心** | ≤7.0ms | |

### 11.3 存档性能

| 操作 | 目标时间 | 方式 |
|------|---------|------|
| 增量存档（自动保存） | < 500ms | 仅序列化脏 chunk + 变更的 NPC 数据，后台线程 |
| 全量存档（手动保存） | < 5s | 完整 LMDB snapshot + 全量序列化，后台线程 |

### 11.4 玩家体感规范

> **依据**：017 测试 Phase 1 确认——移动没有"体感"时，玩家感觉自己"在操作镜头"而非"在世界中存在"。

| 参数 | 值 | 说明 |
|------|-----|------|
| 步行速度 | 6.0 m/s | 在空旷地形中不感觉"拖泥带水" |
| 跑步速度 | 10.0 m/s | 按住 Shift |
| 跳跃高度 | 1.5m | 基础跳跃 |
| 镜头 FOV（默认） | 75° | — |
| 镜头 FOV（跑步） | 80° | 跑步时 +5°，模拟"速度感" |
| 头部摆动振幅 | ±0.03m 垂直 | 模拟走路时头部上下微动 |
| 脚步声频率 | 步行 0.5s/步, 跑步 0.3s/步 | 通用音效，Phase 1 即实现 |
| 落地反馈 | 镜头下沉 0.05m + 通用落地音效 | — |

### 11.5 世界边界设计

> **依据**：017 测试 Phase 1 确认——不可见碰撞墙是体验最差的世界边界处理。

采用软边界方案：

```
距边界 100m: 雾浓度开始线性增加（从当前值 → 100%）
距边界 60m:  移动速度开始线性衰减（从 100% → 50%）
距边界 30m:  HUD 显示文字提示"前方的世界尚未生成……"
距边界 0m:   不可穿越的崎岖地形（噪声频率+振幅剧增的大型乱石地貌）
              + 可选：角色自语"这里太过险峻，无法前行"
```

**Phase 4 世界扩展后**：世界边界外的区域变为可生成 → 软边界自动向外移动。Phase 6 完整世界中——世界边界位于 16km，距玩家日常活动范围足够远 → 用自然海洋/山脉替代软边界。

## 十三、本地化

TOML 数据文件支持多语言字段。Godot 侧使用标准 `tr()` 函数。Rust 侧不处理本地化——只传递语言 key。

```toml
[identity]
name = { en = "Blacksmith", zh = "铁匠", ja = "鍛冶屋" }
```

## 十四、开发路线

> **依据**：017 测试 Phase 1-3 验证——NPC 能力应是开发主线，技术为支撑。Phase 重新划分基于 30 次测试的发现。

| Phase | 内容 | NPC 能力跃迁 | 预计时间 |
|-------|------|-------------|---------|
| **1** | 双层噪声地形 + 玩家移动 + 昼夜循环 + 软边界 | — | 3-5 周 |
| **2** | 1-5 NPC 需求驱动 + **昼夜感知** + **最小社交响应** + **需求表现层** | NPC 有目的、有节律、会"注意"他人 | 4-6 周 |
| **3** | 10-30 NPC + 大五人格 + 8 情绪 + 短记忆(30条) + **GOAP最小核心** + **资源波动** + F1碰撞战斗 | NPC 有性格、有情绪、会记忆、会战斗 | 6-8 周 |
| **4** | 50-100 NPC + 完整记忆(200条) + 关系网络 + GOAP完整 + 分层密度场 + 生物群系 + 村庄级聚落(S1) + 对话v1 | NPC 有"过去"和"关系" | 8-12 周 |
| **5** | 500-1000 NPC + 文化三层 + 团体 + 事件 + 经济 + 法律 + 建筑内部 + 道路 + 奇幻特征 + F2战斗(元素+风格) | NPC 有"社会" | 10-14 周 |
| **6** | 100K NPC(L1/L2/L3) + 完整情绪(30复合) + 记忆(2000条) + 全球文化 + 8级LOD + 天气+季节 + 全部职业 + 存档+Mod + F3完整战斗 | 完整"人世" | 12-16 周 |
| **7** | 性能优化 + LLM增强 + 内容丰富 + 本地化 + 打磨 | 世界越来越"像真的" | 持续 |

**Phase 4+ 的时间为预估值**——将在实际开发 Phase 1-3 完成后根据真实测试数据修正。

## 十五、风险矩阵

| 风险 | 等级 | 缓解 |
|------|------|------|
| `godot-rust` 停止维护 | 🟡 | 封装 GDExtension C API fallback——不依赖 crate |
| Rust 学习曲线 | 🟡 | LLM 按需辅助——Phase 1 GDScript 原型先跑起来 |
| 100K NPC 内存超预算 | 🟢 | SoA + LMDB 冷记忆——~1GB 在现代 16GB+ PC 可接受 |
| 地面高速碰撞不精准→穿模 | 🟡 | 三级 fallback——粗碰撞→相邻延伸→球形兜底 |
| Godot 5.0 GDExtension 不兼容 | 🟡 | 锁定 4.6 LTS——GDExtension 通过 trait 隔离 |
| MultiMesh per-instance 数据不足 | 🟡 | TextureBuffer 扩展通道——vertex shader texelFetch |
| LMDB 写入性能不达预期 | 🟢 | WAL 模式+批量事务——记忆编码是低频操作 |
| Pose Database 资产制作 | 🟡 | Phase 1 仅 20-30 姿态——AI 动作提取工具+Mod 社区远期贡献 |
