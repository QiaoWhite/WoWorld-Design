# WoWorld 设计文档补全规划

> **开发代号**: WoWorld (Wonder World)
> **引擎**: Godot 4.6.2
> **开发者**: 独立游戏开发者（Solo）
> **日期**: 2026-06-13

---

## 背景

当前完成 12 个 Phase 的设计产出——11 个模块 ~23,750 行开发规格。但全面审计发现：

- **5 个致命缺失模块**（被现有模块大量交叉引用，阻塞理念走通）：经济与社会、文化系统、载具系统、蓝图与建造、UI/UX
- **5 个高度缺失**：存档系统、政治实体、采矿深度、名声系统、玩家系统（仅 66 行）
- **7 个中度缺失**：音频、疾病/医疗、派系、节日/仪式、教程、冒险小队、世界机/Storyteller
- **12+ 个低度缺失**：成就、MOD、死亡深度、考古、收藏、竞技、宠物、无聊创新、水体互动、美食、火/光物理、地图迷雾
- **现有模块薄弱点**：魔法系统缺性能预算和魔法 AI、战斗系统缺远程武器、NPC v2.1 SkillCategory 待同步、NPC 缺完整感官系统、家具 v0.1

此外，`想法/` 中还有 20+ 个概念设计未被正式化为开发规格。

**目标**：在写任何一行代码之前，穷举所有必要模块的设计文档，走通跨模块理念，完成可行性验证。

## 总体规划概览

```
Phase 13: 社会运转层（经济·文化·法律·政治·名声）
    ↓
Phase 14: 世界交互层（载具·建造·采矿·农业·航海·地下城）
    ↓
Phase 15: 玩家体验层（UI/UX·玩家扩展·教程·死亡传承·日志）
    ↓
Phase 16: 深度涌现层（节日·娱乐·时尚·饮食·NPC生命周期扩展）
    ↓
Phase 17: 现有模块增强（NPC感官系统·魔法深扩·战斗远程·NPC v2.1·家具 v2.0·疾病医疗·音频）
    ↓
Phase 18: 技术支撑层（存档·MOD·无障碍·本地化·调试工具·难度系统）
    ↓
Phase 19: 全面跨模块理念走通验证
```

每个 Phase 内部结构：
1. **模块设计** — 产出开发规格文档
2. **跨模块接口建立** — 明确该模块与已有模块的数据/行为契约
3. **理念走通验证** — 设计若干个"若玩家做 X，系统如何响应"的场景，逐链验证
4. **CLAUDE.md 契约更新** — 新增接口契约

## Phase 13: 社会运转层

> 一个有人居住的世界必须有经济、文化、法律、政治和名声。这些系统相互独立又深度配合——文化影响法律，法律约束经济，经济驱动政治，政治塑造名声。

### 13.1 经济系统（全新模块 `经济与社会/` 或独立 `经济系统/`）

**产出文档**（预计 6-8 篇，~2,000-3,000 行）：
- `001-经济系统总纲.md` — 设计哲学（"受控动态经济"——有隐藏稳定器，非自由市场）、模块边界
- `002-货币与价格.md` — 铜银金三级货币（已在物品系统定义）、基础价格表、供需价格浮动（±15%）、区域价差
- `003-生产与消费.md` — 生产设施（农场/矿场/工坊）、消费需求（食物/工具/奢侈品）、劳动力市场
- `004-贸易与商队.md` — 商路生成算法、商队组成（商人+护卫+驮畜）、商队 AI、贸易专业化（矿业镇出口矿石进口食物）
- `005-财富与阶级.md` — NPC 财富累积/消耗、阶级分层（贫民/平民/中产/富人/贵族）、阶级流动
- `006-金融与地产.md` — 银行/借贷/利息、土地/建筑所有权、租赁、继承（与 NPC 生命周期联动）
- `007-经济事件与危机.md` — 通胀/通缩、饥荒、贸易路线中断、经济繁荣/萧条周期

**跨模块接口**：
| 接口 | 提供方 | 消费方 |
|------|--------|--------|
| `PriceQuery` trait | 经济系统 | 物品系统（交易价格）、NPC（购买决策） |
| `WealthLevel` enum | 经济系统 | 世界生成（NPC 初始财富）、NPC（社会地位判断） |
| `TradeRoute` 数据 | 经济系统 | 载具系统（商船/商队路线）、信息传播（商队携带信息） |
| `ProductionProfile` | 经济系统 | 世界生成（定居点产业配置） |
| `MarketEvent` | 经济系统 | 历史系统（经济事件入史） |

**理念走通验证**：
- [ ] 场景 A：矿业镇粮食短缺 → 粮价上涨 → 商队从农业镇运粮 → 价格回落。玩家可参与（运粮赚差价 / 囤积居奇 / 抢劫商队）
- [ ] 场景 B：NPC 铁匠收入下降 → 从"中产"跌入"平民" → 情绪变差 → 可能酗酒/犯罪/搬家。玩家可帮助（资助/雇佣/拜师）
- [ ] 场景 C：玩家大量买入某商品 → 当地价格短暂上涨 → 吸引商队运货过来 → 价格回落。玩家无法"垄断"（受控动态稳定器）

### 13.2 文化系统（全新模块 `文化系统/`）

**背景**：`想法/文化设计.md` 有较完整的概念草稿，`魔法/018-魔法与文化.md` 有 86 行轻量处理。53 个文件引用"文化"（327 次）——是引用最多的缺失模块。需要**独立模块**而非附属于魔法。

**产出文档**（预计 5-6 篇，~2,000-2,500 行）：
- `001-文化系统总纲.md` — 文化定义（"一个群体的共享行为规范、价值观、审美偏好、沟通方式"）、文化三维（地域/阶层/时代）、模块边界
- `002-文化参数与生成.md` — 文化参数场（类似群系参数场，连续非离散）、种子驱动文化生成、文化传播/融合/分裂
- `003-文化沟通规范.md` — `CommunicationNorms`（已在语言表达 010 定义：打断容忍度/视线规范/个人空间/直接性/沉默容忍/情绪表达/敬语/触觉规范）→ 文化系统正式 Owner
- `004-文化审美与物质.md` — 建筑风格偏好、服饰风格、色彩象征、饮食禁忌（与 13.1 经济联动）、音乐偏好
- `005-文化与技术.md` — 技术作为文化属性、技术树（文明级非玩家级）、技术扩散/停滞/倒退、学者 NPC

**跨模块接口**：
| 接口 | 提供方 | 消费方 |
|------|--------|--------|
| `CultureParams` 参数场 | 文化系统 | 世界生成（聚落文化归属）、NPC（行为规范）、语言表达（敬语选择）、物品（服饰风格标签） |
| `CommunicationNorms` | 文化系统（正式 Owner） | 语言表达（TurnMode 打断阈值/沉默处理）、NPC（社交距离） |
| `TechnologyLevel` | 文化系统 | 物品系统（可制造物品门槛）、建造系统（可建造结构） |
| `CultureAffinity` | 文化系统 | NPC（跨文化态度——好奇/轻视/敌视） |

**理念走通验证**：
- [ ] 场景 A：两个文化的 NPC 相遇 → 一方直接视对方眼睛（他文化=尊重），另一方低头（他文化=尊重）→ 双方初始关系-5%（文化误解）。玩家可充当"文化翻译"
- [ ] 场景 B：某城市文化参数"开放度高" → 新技术/新时尚先在此出现 → 沿贸易路线扩散 → 保守地区最后接受（或永远不接受）
- [ ] 场景 C：玩家穿北方文化服饰进入南方城市 → NPC 反应：好奇（"外地人！"）/ 轻视（"土包子"）/ 敌视（"敌人！"）——取决于两文化关系

### 13.3 法律与秩序系统（`经济与社会/` 或独立 `法律与秩序/`）

**产出文档**（预计 3-4 篇，~1,200-1,800 行）：

- `002-犯罪与侦查.md` — 犯罪类型（盗窃/抢劫/诈骗/伤害/谋杀/纵火/叛乱）、目击→报案→调查→证据→逮捕链条、玩家作为罪犯/目击者/治安官
- `003-审判与惩罚.md` — 审判形式（权威审判/陪审团/神判/决斗审判/魔法审判——文化差异）、惩罚类型（罚款/监禁/流放/肉刑/死刑）、监狱作为物理地点

| `WantedLevel` | 法律系统 | NPC（守卫追捕）、物品系统（ stolen 物品标记） |

**理念走通验证**：
- [ ] 场景 A：玩家在 A 城偷窃被目击 → 守卫追捕 → 逃到 B 城。B 城法律不同（轻罪）或与 A 城有引渡协议（重罪）。文化差异体现——有的文化偷窃剁手，有的罚款了事
- [ ] 场景 B：NPC 因贫困偷食物 → 被抓住 → 审判（陪审团同情→轻判 vs 贵族法官→重判）→ 留下犯罪记录 → 以后找工作更难 → 更贫困 → 更多犯罪（贫困陷阱涌现）

- [ ] 场景 B：国王年老无嗣 → 继承危机 → 三个派系各推候选人 → 玩家支持一方 → 内战或和平解决。结果改变整个区域的政治格局

## Phase 14: 世界交互层

> 玩家和 NPC 不只是"存在于世界中"——他们要移动（载具）、改变（建造）、开采（采矿）、种植（农业）、航行（航海）、探索（地下城）。

### 14.1 载具系统（全新模块 `载具系统/`）

**背景**：49 个文件引用"载具/vehicle/mount"（451 次）——引用最多的缺失概念。被战斗（骑乘战斗）、经济（商队/商船）、世界生成（交通网络）、NPC（船员/乘客角色）、天气（航行影响）等几乎所有模块依赖。

**产出文档**（预计 5-6 篇，~2,000-2,800 行）：
- `001-载具系统总纲.md` — 载具定义（移动参考系 + 可承载多 NPC/玩家 + 内部可自由行走）、载具类型（陆上/水上/空中/魔法）
- `002-移动参考系.md` — 载具本地坐标系、载具 NavMesh、载具上 NPC 行为（船员/乘客角色分化——已在 NPC 模块预留 `VehicleRole` 枚举）、载具物理基础
- `003-载具建造与维护.md` — 载具作为"大型可建造物"（与建造系统联动）、材料/工时/技能需求、载具耐久与修理
- `004-陆上载具.md` — 马车/货车/战车（动物牵引）、魔力火车（轨道限定，快速大量运输）、坐骑（单 NPC 骑乘——与动物系统联动）
- `005-水上载具.md` — 帆船（风力驱动——与天气联动）、划桨船（人力）、魔力船、船体/帆/炮模块化（类似武器装配框架）
- `006-空中载具.md` — 飞艇/魔法浮空船（稀有高端）、飞行坐骑（狮鹫/飞龙）

**跨模块接口**：
| 接口 | 提供方 | 消费方 |
|------|--------|--------|
| `VehicleId` / `VehicleState` | 载具系统 | NPC（VehicleRole）、战斗（载具战）、经济（商队货物） |
| `Mountable` trait | 载具系统 | 生命系统（动物实现——可骑乘标签） |
| `VehicleConstructionRecipe` | 载具系统 | 建造系统（蓝图扩展为载具） |
| `NavMeshLocal` | 载具系统 | NPC（载具上寻路） |

**理念走通验证**：
- [ ] 场景 A：玩家登上帆船 → 船离港 → 海上航行（天气影响风速/浪高）→ 玩家在甲板上自由行走 → NPC 船员（船长/舵手/瞭望手/炮手）各司其职 → 遭遇海盗 → 海战 → 登船肉搏
- [ ] 场景 B：玩家乘坐魔力火车从 A 城到 B 城 → 车上 NPC 有乘客（聊天/看书/看窗外）和乘务员（检票/卖食物）→ 途经风景变化（海岸→平原→山脉）
- [ ] 场景 C：NPC 团体集体建造一艘大船（码头→龙骨→肋骨→甲板→桅杆→完工）→ 历时数日-数月（游戏时间）→ 建成庆典 → 首航

### 14.2 蓝图与建造系统（全新模块 `蓝图与建造/`）

**背景**：45 个文件引用"建造/建筑/blueprint"（312 次）。三途径（游戏内编辑器/外部 TOML 导入/委托 NPC）+ 集体施工 + 建筑参数化——在技术栈方案 v3.0 中已定调，但没有设计规格。

## Phase 17: 现有模块增强

> 将已有但偏薄的模块补充到与地基模块同等的深度。**本 Phase 第一个任务——NPC 感官系统——是 NPC 从"行为模拟"走向"存在模拟"的关键桥梁**：NPC 不再直接读取世界数据库，而是通过模拟的感官来感知世界。

### 17.0 NPC 感官系统（`NPC活人感/` 新增核心文档 ★关键桥梁★）

**背景与哲学**：当前 NPC 的 `SensoryProvider` trait（已定义于 NPC v2.0）和 `PerceptionContext` 提供了感官外壳，但只实现了天象感知（`SkyPerception`）。NPC 的 GOAP 决策直接读取 `Vitals` 数值、直接查询 `ItemEntId` 位置、直接访问其他 NPC 的关系数据——这本质上是"上帝视角"。真正的"活人感"要求 NPC **只能通过模拟感官获取信息**：看到什么（视觉）、听到什么（听觉）、闻到什么（嗅觉）、触到什么（触觉）。信息不完整→决策不确定→行为涌现→故事生成。

**核心设计原则**：
1. **感官是信息瓶颈**：NPC 不能"查询数据库"——只能通过感官采样获取有限的、可能有误差的世界信息
2. **感官 = f（种族生理，当前状态，环境条件，个性注意力）**：精灵视力≠矮人视力，疲劳时感知下降，雾天视觉受限，高开放性 NPC 更注意细节
3. **感官→记忆→决策**：感官输入经过注意力过滤→编码为感知事件→写入记忆→驱动 GOAP 目标
4. **玩家与 NPC 感官平等**：NPC 感知玩家的方式 = 玩家感知 NPC 的方式 = 同一套感官模型（只是玩家感官通过 Godot 渲染+UI 呈现，NPC 感官通过 Rust 模拟）

**产出文档**（预计 3-4 篇，~2,500-3,500 行）：

#### 17.0.1 `NPC感官系统-总纲与统一模型.md`（~800-1,000 行）

**统一感官基类**：

```rust
/// 感官模态——NPC 感知世界的五种通道
pub enum SensoryModality {
    Visual,     // 视觉
    Auditory,   // 听觉
    Olfactory,  // 嗅觉
    Tactile,    // 触觉（含温度、疼痛、材质）
    Interoceptive, // 内感觉（饥饿/口渴/疲劳/疼痛——来自自身 Vitals）
}

/// 单条感官采样——一次"注意到什么"的原子记录
pub struct SensorySample {
    pub modality: SensoryModality,
    pub source_position: Vec3,       // 感知源的世界坐标（可能有误差）
    pub source_entity: Option<NpcId>, // 如果是实体
    pub intensity: f32,              // 0.0-1.0 感知强度
    pub confidence: f32,             // 0.0-1.0 确信度（受距离/遮挡/噪音影响）
    pub content: SensoryContent,     // 感知到的内容
    pub timestamp: GameTime,
}

/// 感官内容——"感知到了什么"的枚举
pub enum SensoryContent {
    // 视觉
    VisualEntity { approx_distance: f32, direction: Vec3, recognized: Option<NpcId> },
    VisualObject { item_category: ItemCategory, approx_position: Vec3 },
    VisualEvent { event_type: VisualEventType }, // 闪光/烟雾/血迹/足迹
    // 听觉
    AuditorySound { sound_type: SoundType, approx_distance: f32, direction_cone: f32 },
    AuditorySpeech { content_hash: u64, language: Option<LanguageId>, clarity: f32 },
    AuditoryFootstep { count: u8, direction: Vec3, gait_guess: Option<GaitType> },
    // 嗅觉
    OlfactoryScent { scent_type: ScentType, intensity: f32, direction_hint: Option<Vec3> },
    // 触觉
    TactileContact { contact_type: ContactType, body_part: BodyPart, intensity: f32 },
    TactileTemperature { ambient: f32, surface: Option<f32> },
    TactilePain { source: PainSource, body_part: BodyPart, severity: f32 },
}

/// NPC 感官配置——由种族生理决定的基础感知能力
pub struct SensoryConfig {
    // 视觉
    pub visual_range_m: f32,           // 基础视距（晴天·平原）
    pub visual_acuity: f32,           // 0.0-1.0 视觉敏锐度（影响识别距离和细节）
    pub field_of_view_deg: f32,       // 视野角度（人类~180°，兔子~360°）
    pub night_vision: f32,            // 0.0-1.0 夜视能力（0=纯昼行，1=纯夜行）
    pub color_vision: ColorVisionType, // 全色/二色/单色
    pub motion_sensitivity: f32,      // 0.0-1.0 对运动的敏感度

    // 听觉
    pub hearing_range_m: f32,         // 基础听觉半径（安静环境）
    pub hearing_acuity: f32,          // 0.0-1.0 听觉敏锐度
    pub frequency_range: (f32, f32),  // 可听频率范围（Hz）
    pub sound_localization: f32,      // 0.0-1.0 声源定位精度

    // 嗅觉
    pub olfactory_range_m: f32,       // 基础嗅觉半径
    pub olfactory_acuity: f32,        // 0.0-1.0 嗅觉灵敏度
    pub scent_discrimination: f32,    // 0.0-1.0 气味分辨能力

    // 触觉
    pub tactile_acuity: f32,          // 0.0-1.0 触觉敏感度
    pub temperature_sensitivity: f32, // 0.0-1.0 温度感知精度
    pub pain_threshold: f32,          // 0.0-1.0 痛觉阈值（高=耐痛）
}

/// 感官注意力——当前帧 NPC 的注意力分配
pub struct SensoryAttention {
    pub focus_point: Option<Vec3>,     // 当前注意力焦点（世界坐标）
    pub focus_entity: Option<NpcId>,   // 当前关注的实体
    pub visual_attention: f32,        // 0.0-1.0 分配给视觉的注意力
    pub auditory_attention: f32,      // 0.0-1.0 分配给听觉的注意力
    pub peripheral_awareness: f32,    // 0.0-1.0 余光/背景音的整体警觉
    pub distraction_level: f32,       // 0.0-1.0 当前分心程度（情绪/疲劳/多任务）
}
```

**感官处理流水线**（每帧）：
```
世界客观状态 (WorldSnapshot)
  │
  ├─→ [感官采样] ← SensoryConfig (种族能力) × 环境条件 (光线/雾/噪音) × NPC 当前状态 (疲劳/醉酒/情绪)
  │      │
  │      ▼
  │   原始感官信号 (raw intensity × distance_attenuation × environment_modifier)
  │      │
  ├─→ [注意力过滤] ← SensoryAttention (当前焦点) × BigFive (开放性→注意细节, 神经质→注意威胁)
  │      │
  │      ▼
  │   注意到的信号 (intensity > attention_threshold → SensorySample)
  │      │
  ├─→ [感知编码] ← confidence × prior_knowledge (已有记忆中的相关信息)
  │      │
  │      ▼
  │   感知事件 (SensorySample → 可能写入 MemoryStore)
  │      │
  ▼      ▼
GOAP 目标激活 ← 情绪偏移 ← MemoryStore
```

**环境条件对感官的修正**（统一公式）：
```
effective_range = base_range × environment_multiplier
effective_acuity = base_acuity × environment_multiplier × state_multiplier

environment_multiplier = f(light_level, fog_density, precipitation, ambient_noise, wind_direction)
state_multiplier = f(fatigue, intoxication, emotion_arousal, focus)
```

**跨感官整合**：
- 多感官确认：看到+听到→确信度高（单感官→可能误判）
- 感官冲突：听到脚步声但没看到人→"有人吗？"（警觉反应）vs 看到人影但没声音→"幻觉？"
- 感官补偿：盲 NPC 听觉/嗅觉敏锐度+50%（生理自适应）

#### 17.0.2 `NPC感官系统-视觉.md`（~700-900 行）

**视觉模型**：
- 视锥采样（foveated）：视野中心高分辨率（识别面孔/文字），边缘低分辨率（仅检测运动/大致形状）
- 距离衰减：`recognition_probability = 1.0 - (distance / effective_visual_range)²`
- 光线修正：白天=1.0，黄昏=0.5，夜晚（无光源）=0.05，满月=0.2，城市灯光=局部照明
- 遮挡检测：射线检测（简化——用 WorldSnapshot 中的空间索引近似，不做完整物理射线）

**视觉感知内容**：
| 距离 | 可感知的内容 |
|------|-------------|
| 0-5m | 面孔识别、微表情、手中物品细节、文字阅读 |
| 5-20m | 身份识别（认识的人）、大致动作、持有武器类型 |
| 20-100m | 人形/非人形、移动方向、大致体型服饰颜色 |
| 100-500m | 人影、大型物体、烟雾/火光 |
| 500m+ | 地标建筑轮廓、山脉、大型烟雾柱 |

**视线与社会认知**：
- 视线追踪：NPC 注意到"谁在看谁"——社交推断的基础（已在 02-性别与吸引力 中预留 `gaze_bias_toward`）
- 视线回避/凝视：文化规范 `CommunicationNorms.eye_contact_norm`
- 玩家隐身/伪装：对抗 NPC 视觉敏锐度+注意力（潜行系统感官对抗）

#### 17.0.3 `NPC感官系统-听觉.md`（~600-800 行）

**听觉模型**：
- 距离衰减：声音强度 ∝ 1/distance²（对数衰减 `db_attenuation = 20 × log10(distance/reference_distance)`）
- 环境噪音修正：市场（-30% 听觉范围）、酒馆（-20%）、森林夜晚（+10%——安静）、瀑布旁（-60%）
- 遮挡衰减：墙后-15dB，门后-25dB，楼层间-20dB

**听觉感知内容**：
| 声音类型 | 基础可闻距离 | 信息内容 |
|---------|------------|---------|
| 对话（正常音量） | 5-10m | 语言内容（清晰度随距离衰减） |
| 对话（大声） | 15-30m | 部分语言内容 |
| 悄悄话 | 0.5-1m | 仅偷听检测成功时（已在语言表达 009 定义） |
| 脚步声 | 10-30m | 大致方向、步态猜测（沉重=重甲/大体型，急促=奔跑） |
| 武器出鞘/碰撞 | 20-50m | "有人在战斗" |
| 锻造声 | 50-100m | "附近有铁匠铺" |
| 钟声/号角 | 200-500m | 警报/集合信号/时间报时 |
| 雷声 | 500-2000m | 风暴方向 |

**听觉与社会认知**：
- 偷听对话（已在语言表达 009 定义 `PrivateMode::Whisper/Conspiracy` + 偷听检测公式）
- 从语调推断情绪（愤怒/恐惧/兴奋/悲伤——非语言声音线索）
- 从脚步声认出熟人（高亲密关系+高听觉敏锐度→可能）
- 战斗听觉：听到武器碰撞声→推断战斗位置和规模

#### 17.0.4 `NPC感官系统-嗅觉与触觉.md`（~500-700 行）

**嗅觉模型**：
- 气味传播：风向+距离+湍流简化模型（不做 CFD 流体模拟——用概率扩散模型）
- 气味类型枚举 `ScentType`：
  - **生存相关**：Smoke（火灾→逃离）、Blood（战斗/受伤→警觉或吸引食腐者）、Rot/Decay（腐烂尸体→避开或吸引食腐动物）
  - **食物相关**：Cooking（烹饪→"附近有食物"）、FreshBread（面包房定位）、Spices（香料商人定位）
  - **生物相关**：RacialScent（种族体味——NPC 可通过气味判断附近有"异族"）、PredatorScent（捕食者气味→猎物 NPC 警觉）、FearScent（恐惧信息素→附近 NPC 情绪传染）
  - **环境相关**：SaltWater（→海岸方向）、WetEarth（→河边/雨后）、Flowers（→花田/春天）、PineForest（→针叶林）
  - **职业相关**：ForgeSmoke（→铁匠铺）、Tannery（→皮革坊——恶臭）、Alchemy（药水气味→炼金店）、TavernAle（→酒馆）
- 气味记忆：最强烈的记忆触发方式——"这个味道……让我想起了……"（普鲁斯特效应）
- 气味追踪：猎犬/狼 NPC 可凭借气味追踪目标（留下气味痕迹随时间衰减）

**触觉模型**：
- 环境触觉：温度感知（冷/热——与天气系统 `ground_temperature` 联动）、风速感知（与天气联动）、湿度感知（与天气联动）、地面材质（草地/石地/沙地/泥地——通过脚下触感）
- 社交触觉：握手/拥抱/拍肩/推挤——受 `CommunicationNorms.touch_norms` 和文化规范约束（北欧人=极少肢体接触，地中海人=频繁肢体接触）
- 战斗触觉：被击中→疼痛信号→`TactilePain`→情绪偏移（恐惧/愤怒）+ 部位伤害感知
- 疼痛系统：痛觉阈值（种族+个体差异）→疼痛信号强度→如果疼痛超过阈值→NPC 可能：退缩/求饶/狂暴（人格决定）

**感官衰减与异常**：
- 老年：视觉（老花/白内障概率）、听觉（高频听力损失）、嗅觉（衰退）——感官能力随年龄自然衰减
- 疾病：感冒→嗅觉丧失+听觉模糊、发烧→视觉模糊+触觉过敏
- 醉酒：视觉模糊+听觉范围缩小+平衡感丧失（触觉-前庭）
- 魔法增强/诅咒：夜视药水、鹰眼术（视觉范围×3）、沉默术（听觉全失）、痛苦诅咒（触觉恒痛）

**理念走通验证**：
- [ ] 场景 A：夜晚·雾天·森林中 → NPC 守卫视觉范围从 100m 缩至 15m → 玩家潜行通过（只要不出声）→ 但如果踩到树枝（听觉事件）→ NPC 警觉→转头朝向声源→前去查看
- [ ] 场景 B：NPC 铁匠在锻造 → 视觉焦点在铁砧 → 听觉被锻打声占据（-40% 听觉敏锐度）→ 玩家从背后走过不被发现 → 但如果玩家碰到工具架（触觉/听觉）→ NPC 可能注意到
- [ ] 场景 C：两个 NPC 在酒馆对话 → 第三个 NPC 从旁经过 → 嗅觉检测到其中一人是"异族"（RacialScent）→ 如果该 NPC 文化对外族敌视 → 可能产生厌恶情绪或挑衅行为
- [ ] 场景 D：玩家受伤流血 → 留下血味痕迹（ScentTrail）→ 食肉动物/怪物在 200m 外闻到 → 沿气味追踪玩家 → 如果玩家渡河（气味中断）→ 追踪失败（除非动物在河对岸重新嗅到）
- [ ] 场景 E：老年 NPC 视觉+听觉衰退 → 在战斗中处于劣势（无法准确感知敌人位置）→ 但可能发展出"智慧"补偿（更依赖战术和陷阱而非反应速度）
- [ ] 场景 F：NPC 闻到烟味（嗅觉→火灾可能）→ 如果 NPC 有孩子在家（关系驱动）→ 立即回家查看 → 如果只是邻居在烧烤（视觉确认）→ 消除警觉。如果确实是火灾→触发警报行为（喊叫/敲钟→听觉传播→群集响应）

## Phase 19: 全面跨模块理念走通验证

> 所有模块设计完成后，必须进行一次彻底的跨模块一致性验证——确保理念层面不存在矛盾、断裂或"先有鸡还是先有蛋"的循环依赖。

### 19.1 跨模块接口契约最终审计

**方法**：4-6 个并行审计代理，按模块群分组审计：
- 组 A：社会运转层（经济/文化/法律/政治/名声）× 地基模块（物品/技能/天气/语言）
- 组 B：世界交互层（载具/建造/采矿/农业/地下城）× 世界生成/生命/战斗
- 组 C：玩家体验层（UI/玩家/教程/死亡传承）× NPC/历史
- 组 D：深度涌现层（节日/娱乐/时尚/饮食/生命周期）× 所有已有模块
- 组 E：技术支撑层（存档/MOD/音频/无障碍）× 技术栈方案

**产出**：`CHG-021-全模块跨模块一致性审计.md`，包含所有发现的冲突、断裂和循环依赖。

### 19.3 性能预算汇总与验证

将所有模块的性能预算汇总到一个文档 `性能预算总览.md`，验证：
- 总帧预算 = 各模块预算之和 ≤ 16.7ms（60fps）
- 总 VRAM = 各模块 VRAM 之和 ≤ 6GB（含 1.8GB 余量）
- 总内存 = 各模块内存之和 ≤ 32GB（含余量）
- 标记所有"[TUNING]"参数——确保有调优空间

### 19.4 设计哲学一致性审计

对照核心哲学检查所有新模块：
1. **"故事生成器，非故事容器"** — 新模块是否定义了"规则"而非"内容"？节日=文化参数×历法时间×历史事件（涌现），不是预设"龙舟节"。
2. **"玩家 AI = NPC AI"** — 新模块的玩家接口是否与 NPC 接口共用同一套代码路径？
3. **"冒险与生活平等"** — 新模块是否同时服务战斗向玩家和生活向玩家？建造系统是否铁匠也能用？
4. **"全球多元文化"** — 新模块是否避免了预设单一文化假设？法律系统是否容纳了多种法律传统？
5. **"无限探索"** — 新模块是否能扩展到无限世界？（程序化生成，非手工内容）

## 关键决策记录

| 决策 | 理由 |
|------|------|
| NPC 必须通过模拟感官感知世界 | 不能"查询数据库"——只有模拟五感（视听嗅触+内感觉）获取不完整信息，才能产生真正的行为涌现 |
| 文化系统独立模块 | 53 文件·327 次引用——是缺失模块中被引用最多的。文化参数场与群系参数场同级 |
| 先设计后开发 | Phase 13-19 全部为纯设计文档工作——在理念走通前不写代码 |
| Phase 顺序按依赖层次 | 社会运转层→世界交互层→玩家体验层→深度涌现层→现有增强→技术支撑→验证 |
| 每个 Phase 含跨模块接口契约 | 延续 CHG-013 接口契约表传统 |
| 经济系统采用"受控动态"模型 | 有隐藏稳定器——非自由市场。防止玩家垄断和通货膨胀失控 |
