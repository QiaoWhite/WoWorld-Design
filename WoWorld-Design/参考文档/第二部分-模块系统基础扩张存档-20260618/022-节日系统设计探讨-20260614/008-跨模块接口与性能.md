## 一、FestivalQuery trait —— 所有模块的只读入口

```rust
/// 节日系统只读查询接口
/// 
/// 设计原则与 CultureQuery 一致：Send+Sync，零分配，实现方不透明
pub trait FestivalQuery: Send + Sync {
    // ━━━ 基础查询 ━━━
    
    /// 查询指定文化在指定日期的所有节日
    /// 返回：该日激活的 FestivalInstance ID 列表
    fn festivals_on_date(&self, culture_id: CultureId, day_of_year: u16) -> Vec<FestivalId>;
    
    /// 查询指定文化在接下来的 N 天内的所有节日
    fn upcoming_festivals(&self, culture_id: CultureId, days_ahead: u16) -> Vec<(FestivalId, u16)>; // (id, days_until)
    
    /// 获取节日的完整框架数据
    fn festival_framework(&self, festival_id: FestivalId) -> Option<&FestivalFramework>;
    
    /// 获取节日本年度的实例（如果已创建）
    fn current_instance(&self, festival_id: FestivalId) -> Option<&FestivalInstance>;
    
    // ━━━ 参与查询 ━━━
    
    /// 指定 NPC 是否计划参加该节日
    fn npc_attending(&self, npc_id: NpcId, festival_id: FestivalId) -> Option<bool>;
    
    /// 获取该节日所有计划参加的 NPC 列表
    fn attending_npcs(&self, festival_id: FestivalId) -> Vec<NpcId>;
    
    // ━━━ 经济查询 ━━━
    
    /// 该节日是否有经济消费激增效应
    fn has_economic_impact(&self, festival_id: FestivalId) -> bool;
    
    /// 获取该节日的经济冲击数据（消费激增、价格乘数）
    fn economic_impact(&self, festival_id: FestivalId) -> Option<&FestivalEconomicImpact>;
    
    // ━━━ 空间查询 ━━━
    
    /// 该节日的主要举办地点
    fn festival_location(&self, festival_id: FestivalId) -> Option<FestivalLocation>;
    
    // ━━━ 文本查询 ━━━
    
    /// 该节日的名称（指定文化的语言）
    fn festival_name(&self, festival_id: FestivalId, culture_id: CultureId) -> Option<String>;
    
    /// 该节日的简短描述（1-2句——用于 UI 和对话）
    fn festival_description(&self, festival_id: FestivalId, culture_id: CultureId) -> Option<String>;
}
```

## 二、FestivalMut trait —— 仅内部使用

```rust
/// 节日系统修改接口——pub(crate)：仅文化系统和历史系统可调用
pub(crate) trait FestivalMut: FestivalQuery {
    // ━━━ 创建 ━━━
    
    /// 从 FestivalFramework 创建新节日（种子生成用）
    fn create_festival(&mut self, framework: FestivalFramework) -> FestivalId;
    
    /// 从涌现检测创建节日（运行时）
    fn create_emergent_festival(
        &mut self, 
        name: GeographicName, 
        day: u16, 
        culture_id: CultureId,
        source: EmergentFestivalSource,
    ) -> FestivalId;
    
    /// 从历史事件创建纪念性节日
    fn create_commemorative_festival(
        &mut self,
        event_id: EventId,
        culture_id: CultureId,
    ) -> FestivalId;
    
    // ━━━ 更新 ━━━
    
    /// 推进节日的阶段（宣告→预备→庆祝→余波→结束）
    fn advance_phase(&mut self, festival_id: FestivalId);
    
    /// 更新节日实例的实时数据（参与人数、规模）
    fn update_instance(&mut self, festival_id: FestivalId, instance: FestivalInstance);
    
    // ━━━ 删除 ━━━
    
    /// 标记节日为已废止
    fn abolish_festival(&mut self, festival_id: FestivalId, reason: AbolitionReason);
    
    // ━━━ 年度重置 ━━━
    
    /// 每年新年——创建所有节日的新年度实例
    fn create_annual_instances(&mut self, year: u64);
}
```

## 三、各模块接口契约

| # | 接口 | 提供方 | 消费方 | 数据格式 | 说明 |
|---|------|--------|--------|---------|------|
| 1 | `FestivalQuery` trait | 节日系统（文化系统 008） | 全部模块 | trait (Send+Sync) | 所有模块获取节日数据的唯一只读入口 |
| 2 | `FestivalMut` trait | 节日系统 | 文化系统、历史系统 | pub(crate) trait | 节日修改的唯一入口 |
| 3 | `FestivalId` | 节日系统 | 全部模块 | u32 (Copy) | 节日全局标识——高 16bit=CultureId, 低 16bit=序号, MSB=1→运行时创建 |
| 4 | `FestivalFramework` | 节日系统 | 文化系统（生成）、信仰系统（宗教节日数据） | struct (~120B) | 节日的静态框架——种子生成或运行时创建 |
| 5 | `FestivalInstance` | 节日系统 | NPC、经济、历史 | struct (~80B) | 节日的年度实例——动态数据 |
| 6 | `FaithCalendarQuery` trait | 信仰系统（待设计） | 节日系统（宗教节日生成） | trait | ★ 信仰系统定义，节日系统消费——信仰系统提供神圣日期列表 |
| 7 | `legitimacy.ritual` 因子 | 权力系统 004 | 节日系统（提供 `last_ritual_at`） | f32 (0-0.15) | 统治者主持节日→合法性因子的 ritual 分量更新 |
| 8 | `closed_on_holidays` | 经济系统 002 | 节日系统（提供"今天是节日"查询） | bool | 商店在节日期间是否关闭 |
| 9 | `OutfitMode::Ceremonial` | 物品系统 004 | 节日系统（提供节日触发器） | enum variant | 节日期间 NPC 切换到 Ceremonial 服装 |
| 10 | `TurnMode::Ritual` | 语言表达 005 | 节日系统（提供 RitualStep 序列） | enum variant + struct | 仪式对话的脚本化序列 |
| 11 | `TopicCategory::Festival` | 语言表达 005（新增） | 节日系统（提供话题触发） | enum variant | NPC 对话中的节日话题 |

| 13 | `EventType::FestivalHeld` | 历史系统 002（新增） | 节日系统（提供节日事件记录） | enum variant | 节日作为历史事件类型 |
| 14 | `RecurringEventSeries` | 历史系统 002（新增） | 节日系统 | struct | 周期性事件的轻量表示 |
| 15 | `EmergentFestivalDetection` | 节日系统（内部） | 节日系统 | 算法 | 检测 NPC 自发聚集是否已形成事实节日 |

## 四、与气候/季节系统的接口

```rust
// 节日系统消费天气/季节数据——通过已有 WeatherQuery trait
impl FestivalSystem {
    fn daily_tick(&mut self, weather: &dyn WeatherQuery) {
        let today = weather.current_season().day_of_year;
        let season = weather.current_season().season;
        let weather_sample = weather.sample(/* ... */);
        
        // 1. 检查今天是哪些节日的开始/结束日
        for festival_id in self.active_festivals() {
            let framework = self.festival_framework(festival_id);
            
            // 节日开始？
            if today == framework.day_of_year {
                self.activate_festival(festival_id);
            }
            
            // 节日进行中——天气影响？
            if let Some(instance) = self.current_instance_mut(festival_id) {
                if weather_sample.is_severe() {
                    // 极端天气 → 户外活动取消 → 节日规模降低
                    instance.scale *= 0.5;
                }
            }
        }
        
        // 2. 涌现型节日检测——季节性的自然现象
        self.detect_emergent_festivals(season, weather);
    }
}
```

## 五、性能预算

### 5.1 存储

| 项目 | 规模 | 说明 |
|------|------|------|
| `FestivalFramework` per festival | ~120B | 每文化 2-7 个节日 × 100 文化 ≈ 500 节日 × 120B ≈ 60KB |
| `FestivalInstance` per year | ~80B | 每年每节日一个实例 ≈ 500 × 80B ≈ 40KB（覆盖后懒创建） |
| `FestivalEconomicImpact` | ~64B | 仅在节日期间存在，动态分配 |
| 文本片段（节日相关） | ~10KB | ~80 个新片段 × ~120B |
| **总计** | **< 200KB** | 加上历史记录和年度摘要 ≈ 500KB 上限 |

### 5.2 运行时 CPU

| 操作 | 频率 | 成本 |
|------|------|------|
| `festivals_on_date()` | ~1 次/帧（UI 或 NPC 决策） | ~500ns（FxHashMap 查找） |
| `npc_attending()` | ~10 次/帧（NPC 决策） | ~100ns each |
| `daily_tick`（节日激活+阶段推进） | 每游戏日 1 次 | ~50µs（检查 500 个节日——但大部分不在今天） |
| `advance_phase` | 每节日每阶段 1 次 | ~10µs（仅更新状态机） |
| 人群聚集调度 | 节日庆祝期每帧 | ~100µs（远距离 NPC→流场，近距离 NPC→个体寻路） |
| 涌现型节日检测 | 每游戏日 1 次 | ~200µs（四叉树查询 NPC 密度） |
| 年度实例创建 | 每游戏年 1 次 | ~100µs |
| **总帧预算** | — | **< 0.1ms/帧**（不及 16.7ms 的 0.6%） |

### 5.3 人群寻路优化

这是节日系统最大的性能挑战。优化策略：

1. **分帧错开**：节日庆祝期第一天——NPC 不在同一帧同时离开家。宣告期+预备期的 3-7 天内 NPC 的行动计划已经分散。
2. **群体流场**：对 L2 外的 NPC——使用预计算的流场（从居住区→节日场地的方向场），而非个体 A*。
3. **L3/L4 统计模型**：远距离 NPC 不需要个体路径——统计层面知道"有 X 个 NPC 在前往节日"。
4. **广场容量**：每个节日场地有最大容量——超出容量的 NPC 在周边街道分流。

## 六、模块位置

节日系统作为 **文化系统/008-节日系统.md**——因为：
- 节日从文化参数涌现
- 节日反过来通过 `CultureMut::apply_culture_shift()` 影响文化
- 节目的命名使用文化音系规则（与地名系统 007 共享底层）
- 节日的情绪和行为由 `CommunicationNorms` 和 `CulturalBeautyStandard` 调制

**不与魔法系统 010-仪式魔法合并**——社会仪式与魔法仪式是不同概念（详见 002-节日时间与结构 §4.2）。
