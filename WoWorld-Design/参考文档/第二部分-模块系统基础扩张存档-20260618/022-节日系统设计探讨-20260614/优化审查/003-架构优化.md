> **状态**: 6 个架构问题已识别，逐一给出重构方案

## 架构问题 1：`FaithCalendarQuery` trait 归属错误

**原文**（001 §2.2 + 008 §3）：节日系统定义了一个 `FaithCalendarQuery` trait，让信仰系统去实现。原文说"★ 信仰系统定义，节日系统消费"——但 008 §3 的接口表又说提供方是"信仰系统（待设计）"。

**问题**：节日系统不应该定义一个属于信仰系统的 trait。按照已有地基模块的解耦模式（参见技能系统的 `MentalAccess` ——技能系统定义 trait，NPC 实现），应该是：

| trait | 定义方 | 实现方 |
|-------|--------|--------|
| `MentalAccess` | 技能系统（地基模块——因为它是消费者） | NPC 模块 |
| `FaithCalendarQuery` | **节日系统**（消费者——需要信仰数据的模块） | 信仰系统（待设计） |

**修正**：维持原方案——节日系统定义 `FaithCalendarQuery` trait，信仰系统实现。这与 `MentalAccess` 模式一致。但修改文档表述，明确说明**消费者定义 trait** 是 WoWorld 的标准模式。

```rust
// 在节日系统中定义（作为消费者，节日系统有权定义自己需要什么数据）
pub trait FaithCalendarQuery: Send + Sync {
    fn holy_days(&self, faith_id: FaithId) -> Vec<HolyDay>;
    fn primary_deities(&self, faith_id: FaithId) -> Vec<DeityInfo>;
    fn has_fasting_period(&self, faith_id: FaithId, day: u16) -> Option<FastingRule>;
}
```

**与 CLAUDE.md 对齐**：如果将 `FaithCalendarQuery` 写入跨模块契约，提供方 = 信仰系统，消费方 = 节日系统。trait 定义在消费方——与 CHG-015（技能系统定义 MentalAccess）的模式一致。

## 架构问题 2：`FestivalInstance` 与 `HistoricalEvent` 数据重叠

**原文**（007 §6.1）：节日记录为历史事件。但 `FestivalInstance` 本身已经包含了 `year`、`scale`、`participant_count`——与 `HistoricalEvent` 的字段重叠。

**问题**：两种数据——应该存哪？如果存在两处，一致性如何保证？

**修正**：采用**关联**模式（类似地名系统中的 GeographicEntity → EntityNames 关系）：

```rust
/// FestivalInstance 是"当前年份的运行时状态"——轻量、短暂
/// HistoricalEvent 是"过去的记录"——永久、不可变
/// 
/// 每年节日结束后：
///   1. FestivalInstance 的摘要数据 → 写入轻量的 RecurringEventSeries 年度摘要
///   2. 如果当年规模 ≥ 0.8 或发生特殊事件 → 创建完整的 HistoricalEvent
///   3. FestivalInstance 本身可以在节后 30 天被清理（只保留 framework）
```

**分离存储**：
```
节日系统存储：
  FestivalFramework —— 永久（种子生成或运行时创建）
  FestivalInstance —— 暂时（仅当年，过期清理）
  
历史系统存储：
  HistoricalEvent —— 永久（仅重大节日的单次记录）
  RecurringEventSeries —— 永久（所有节日的轻量年度摘要）
```

两个系统通过 `FestivalId` 关联——不共享可变数据，避免一致性冲突。

## 架构问题 3：`TopicCategory::Festival` 对语言表达模块的侵入

**原文**（006 §4.1）：在语言表达 005 的 `TopicCategory` 枚举中新增 `Festival` 变体。

**问题**：语言表达是地基模块——它不应该知道节日系统。`TopicCategory` 是对话系统的核心枚举，修改它需要改动对话系统的核心逻辑。

**修正**：不使用枚举变体——使用话题的**运行时标记**：

```rust
// 语言表达模块提供泛化的"外部话题"挂载点
pub struct ConversationTopic {
    pub category: TopicCategory,
    /// 可选的子标签——由消费模块注入，语言表达模块不解释
    pub sub_tags: Vec<TopicSubTag>,
}

/// 子标签——开放集合，不由语言表达模块枚举
struct TopicSubTag {
    tag_type: u16,      // 由注册表分配——避免冲突
    payload: Vec<u8>,   // 消费模块自定义数据（bincode 或其他）
}

// 节日系统在启动时注册自己的子标签类型：
// registry.register_sub_tag::<FestivalTopicPayload>("festival", 0x0100);
// 语言表达模块不需要重新编译
```

**但这是过度工程化**。实际上 `TopicCategory` 已有 `Knowledge`、`Gossip`、`Phatic` 等通用类别——节日对话可以挂载在 `Knowledge` 下（"分享节日知识"）或 `Phatic` 下（"节日问候"）。不需要新增枚举变体。

**最终方案**：**不修改** `TopicCategory`。节日对话使用已有的 `Knowledge` + `Gossip` 类别——通过 `ConversationContext` 中的 `current_festival: Option<FestivalId>` 字段（依赖注入——语言表达模块已有的模式）让对话系统知道当前是节日上下文。

## 架构问题 4：`OutfitMode::Ceremonial` 的切换耦合

**原文**（003 §3.1）：节日庆祝期开始时，节日系统直接调用 NPC 的 outfit 切换。

**问题**：节日系统直接操作 NPC 的内部状态——违反了 NPC 模块的自治性。

**修正**：节日系统**不直接切换** NPC 的衣服。NPC 模块自己在决策循环中检测：

```rust
// NPC 模块的决策循环中（概率行为树）：
fn evaluate_outfit_choice(npc: &NpcData, context: &NpcContext) -> OutfitMode {
    // 查询是否有正在进行的节日且 NPC 正在参加
    if let Some(festival_id) = context.current_festival {
        if context.festival_query.npc_attending(npc.id, festival_id) == Some(true) {
            if context.festival_query.current_instance(festival_id)
                .map(|i| i.phase == FestivalPhase::Celebration)
                .unwrap_or(false) 
            {
                return OutfitMode::Ceremonial;
            }
        }
    }
    
    // 正常决策逻辑……
}
```

节日系统不推——NPC 模块拉。这与 CHG-013 确认的"History 主动拉取"接口模式一致。

## 架构问题 5：`GeographicName` 复用——节日命名

**原文**（001 §2.1）：节日命名使用"模板"自行生成名称。007-地名系统已经有一套完整的命名基础设施（音系规则、描述词库、命名模板）。

**问题**：节日名称也应该通过 `GeographicName` 体系生成——复用已有的音系规则和描述词库，而不是自行打造一套平行的命名系统。

**修正**：

```rust
/// 节日名称使用 GeographicName 基础设施
fn generate_festival_name(
    festival_type: &FestivalTypeTag,
    emotional_tone: &EmotionalTone,
    culture: &CultureCoreParams,
    lexicon: &CultureToponymyLexicon,  // ← 地名系统 007 的词库
    rng: &mut impl Rng,
) -> GeographicName {
    match festival_type {
        CalendarTag { anchor } => {
            // 使用与自然地理实体相同的命名模板——但 feature_word 是"节/祭/庆典"而非"山/河"
            GeographicName::from_template(
                NameTemplate::DescriptorFeature,
                lexicon.season_word(anchor.season()),
                lexicon.festival_suffix(emotional_tone),  // "节"/"祭"/"庆典"/"日"
                culture,
                rng,
            )
        }
        ReligiousTag { .. } => {
            GeographicName::from_template(
                NameTemplate::AssociativeFeature,
                lexicon.deity_name(deity_id),
                lexicon.festival_suffix(emotional_tone),
                culture,
                rng,
            )
        }
        // ...
    }
}
```

节日名称存入 `EntityNames` 体系（扩展现有的 GeographicEntity 覆盖范围——增加 `GeographicEntityType::Festival`）。不同文化对同一节日的不同称呼通过 exonym 机制处理。

## 架构问题 6：`FestivalQuery` trait 粒度过细

**原文**（008 §1）：`FestivalQuery` 有 12 个方法——全在一个 trait 中。

**问题**：高频方法（`festivals_on_date`、`npc_attending`）和低频方法（`festival_description`、`economic_impact`）混在同一 trait 中。消费者只需要日期查询却必须依赖整个 trait。

**修正**：对标 `CultureQuery` / `CultureQueryExt` 的分层模式：

```rust
/// 核心查询——高频调用，所有消费者都依赖
pub trait FestivalQuery: Send + Sync {
    fn festivals_on_date(&self, culture_id: CultureId, day: u16) -> &[FestivalId];
    fn festival_framework(&self, id: FestivalId) -> Option<&FestivalFramework>;
    fn current_instance(&self, id: FestivalId) -> Option<&FestivalInstance>;
}

/// 扩展查询——低频调用
pub trait FestivalQueryExt: FestivalQuery {
    fn upcoming_festivals(&self, culture_id: CultureId, days: u16) -> Vec<(FestivalId, u16)>;
    fn npc_attending(&self, npc_id: NpcId, festival_id: FestivalId) -> Option<bool>;
    fn economic_impact(&self, festival_id: FestivalId) -> Option<&FestivalEconomicImpact>;
    fn festival_location(&self, festival_id: FestivalId) -> Option<FestivalLocation>;
    fn festival_name(&self, id: FestivalId, culture_id: CultureId) -> Option<&GeographicName>;
    fn festival_description(&self, id: FestivalId, culture_id: CultureId) -> Option<&str>;
}
```

**另注意**：`festivals_on_date` 返回 `Vec<FestivalId>` 会产生堆分配。改为返回 `&[FestivalId]`（引用内部预先按日期分组的切片）。

## 重构后的完整接口设计

```rust
// ═══════════════════════════════════════
// 节日系统 —— 消费方只读接口
// ═══════════════════════════════════════

/// 高频核心查询——每帧调用
pub trait FestivalQuery: Send + Sync {
    /// 指定日期激活的节日（按文化分组预计算切片）
    fn festivals_on_date(&self, culture_id: CultureId, day: u16) -> &[FestivalId];
    
    fn festival_framework(&self, id: FestivalId) -> Option<&FestivalFramework>;
    
    fn current_instance(&self, id: FestivalId) -> Option<&FestivalInstance>;
    
    /// 节日名称（指定文化视角）
    fn festival_name(&self, id: FestivalId, culture_id: CultureId) -> Option<&GeographicName>;
}

/// 扩展查询——低频
pub trait FestivalQueryExt: FestivalQuery {
    fn upcoming_festivals(&self, culture_id: CultureId, days: u16) -> Vec<(FestivalId, u16)>;
    fn npc_attending(&self, npc_id: NpcId, festival_id: FestivalId) -> Option<bool>;
    fn attending_npcs(&self, festival_id: FestivalId) -> &[NpcId]; // 返回切片而非 Vec
    fn economic_impact(&self, festival_id: FestivalId) -> Option<&FestivalEconomicImpact>;
    fn festival_location(&self, festival_id: FestivalId) -> Option<&FestivalLocation>;
    fn weather_sensitivity(&self, festival_id: FestivalId) -> WeatherSensitivity;
}

/// 信仰系统需要为节日系统实现的接口
/// 消费者定义 trait——与 MentalAccess 模式一致
pub trait FaithCalendarQuery: Send + Sync {
    fn holy_days(&self, faith_id: FaithId) -> &[HolyDay];
    fn primary_deities(&self, faith_id: FaithId) -> &[DeityInfo];
    fn fasting_rules(&self, faith_id: FaithId, day: u16) -> Option<&FastingRule>;
}

/// 修改接口——pub(crate)
pub(crate) trait FestivalMut: FestivalQuery {
    fn create_festival(&mut self, framework: FestivalFramework) -> FestivalId;
    fn create_emergent_festival(&mut self, name: GeographicName, day: u16, 
                                 culture_id: CultureId, source: EmergentSource) -> FestivalId;
    fn create_commemorative_festival(&mut self, event_id: EventId, 
                                      culture_id: CultureId) -> Option<FestivalId>;
    fn advance_phase(&mut self, id: FestivalId);
    fn cancel_festival(&mut self, id: FestivalId, reason: CancellationReason);
    fn abort_festival(&mut self, id: FestivalId, reason: AbortReason);
    fn abolish_festival(&mut self, id: FestivalId, reason: AbolitionReason);
    fn create_annual_instances(&mut self, year: u64);
    fn update_instance_data(&mut self, id: FestivalId, participants: u32, scale: f32);
}
```
