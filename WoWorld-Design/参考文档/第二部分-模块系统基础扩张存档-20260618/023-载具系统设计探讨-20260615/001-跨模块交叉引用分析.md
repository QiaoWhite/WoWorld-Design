# 001-载具系统跨模块交叉引用分析

> **日期**: 2026-06-15
> **状态**: 草稿——讨论进行中
> **目的**: 在正式设计载具系统之前，全面盘点所有其他模块对载具的已有引用、假设和期望，以及潜在关联

---

## 核心原则（来自用户明确指示）

> **本模块作为专门讨论本模块内容的地方，其他模块的内容涉及到本模块相关方面的，应当以本模块的设计为权威**——而不是完全倒过来，让本模块内容完全屈服于其他模块里各不相同、有冲突有遗漏等有缺陷的仅仅本意只是提供参考的看法。

### 1. 技术栈方案 v3.0（`开发阶段/技术栈方案/001`）——最权威技术蓝图

**引用密度：🔴🔴🔴🔴🔴 极高**

#### 已有明确定义的内容：
| 概念 | 当前描述 | 约束力 |
|------|---------|--------|
| 载具动力 | 魔力驱动——不考虑风力和洋流的复杂物理 | 🔒 强约束（CHG-008 明确决策） |
| 载具类型 | 小帆船/渔船、大帆船（商船/战舰）、火车（客运/货运）、飞行船（远期） | 🔓 基线——载具系统可以扩展/细化 |
| 移动参考系 | 载具全局变换矩阵 T_V + 本地坐标 P_local → 世界坐标 | 🔒 技术方向确定 |
| 载具 NavMesh | 局部 NavMesh——载具本地空间寻路 | 🔒 技术方向确定 |
| 载具上的 NPC 行为 | 船员角色(4个GOAP目标) + 乘客角色(社交/观景/日常) | 🔓 基线——需详细规格化 |
| 建造者 | 小船→玩家+少数NPC；大船/火车→NPC团体 | 🔒 与建造系统接口 |
| 容量 | 小船2-5人、大帆船20-80人、火车50-200人、飞行船10-50人 | 🔓 基线——可调整 |
| 性能预算 | 载具逻辑≤0.3ms/帧，VRAM含在系统~1GB内 | 🔒 硬约束 |
| 旅行节奏调节 | 暴风雨/海盗/礁石——不阻止旅行，调节节奏和风险 | 🔒 设计哲学确定 |

#### 需要载具系统解决的开放问题：
- 载具的物理模型（速度/加速度/转向/碰撞）尚未定义
- 船员GOAP目标的详细规格未写
- 载具的建造流程与蓝图系统的接口未定义
- 魔力消耗/燃料的经济模型未定义
- 铁路轨道的铺设/维护由谁负责未定义

### 2. NPC 活人感模块（`开发阶段/NPC活人感模块/ver2.0`）

**引用密度：🔴🔴🔴 高**

#### 已定义的接口：
```rust
// NPC 数据结构中已有字段
pub vehicle_state: Option<VehicleState>,

pub struct VehicleState {
    pub vehicle_id: VehicleId,
    pub role: VehicleRole,       // Crew / Passenger
    pub position_local: Vec3,
    pub current_duty: Option<CrewDuty>,
}

pub enum VehicleRole {
    Crew { rank: CrewRank },
    Passenger,
}
```

#### NPC 行为引用：
- `ActionCategory::Vehicular` — 载具事件分类
- `VehicleCondition` — 6种条件（OnShipDeck/InTrainCabin/CrewOnDuty等）
- `VEHICLE_ACTION_MODIFIERS` — 载具对行动权重的修正表
- GOAP 安全网：`Goal::VehicleEmergencyRepair` — 载具紧急维修
- 权重公式中已包含 `self.vehicle_modifier(&action, &npc.vehicle_state)`

#### 需要与载具系统协调的问题：
- `CrewRank` 枚举需要载具系统定义
- `CrewDuty` 枚举需要载具系统定义
- `VehicleCondition` 的条件判断逻辑由谁负责？
- NPC 上/下载具的决策由谁驱动？

### 3. 战斗系统（`开发阶段/战斗/011`）

**引用密度：🔴🔴 中**

#### 关键定义：
- **"马不是载具——它有独立的意志、恐惧和脾气"** — 坐骑≠载具
- 坐骑走完整 Life 基类（HP/体力/恐惧/灵元素）
- 骑乘战斗有独立的三层模型（本能/节奏/战略）
- 骑手×坐骑配合度（独立于载具系统）
- 空战 = 飞行坐骑 + 空中对决

#### 载具与坐骑的边界问题：
- 战斗系统已经明确：**坐骑不属于载具系统**（坐骑是有生命的、有独立AI的）
- 那么"载具上的战斗"（如船上战斗、火车上的战斗）——规则由谁定义？
- 载具在战斗中的角色：是环境？是参战方？是可被攻击的目标？

### 4. 天气系统（`开发阶段/天气与季节系统/004`）

**引用密度：🔴🔴🔴 高**

#### 已有明确接口（消费方=载具）：
```
载具轮询频率: 每决策周期 (每载具每1-10s)

消费字段:
├─ regional_wind_speed + wind_direction → 帆船航速/航向修正
├─ is_extreme_weather + storm_front_distance → 航行危险评估
├─ visibility_m → 航行安全距离 / 触礁风险
├─ surface_state (Ice) → 内河/近海冰封→航线中断
└─ gust_factor → 突发横风→小型船倾覆风险

载具 AI (NPC在载具上):
├─ MaintainVehicle (暴风雨后维修)
├─ AssistPassengers (安抚/引导乘客)
└─ 船长决策: 绕行/靠岸/继续航行
```

#### 矛盾/需要协调的问题：
- 技术栈方案说"魔力驱动——不考虑风力"，但天气系统列出了 `wind→帆船航速`。**存在明显矛盾**。需要载具系统权威裁决
- 天气系统还细分了"魔力驱动船: 不受风向影响（仅风暴影响）"和"帆船: 航速 = f(风速, 风角, 船体设计)"——这是否意味着非魔力驱动的帆船存在？

### 5. 世界生成（`开发阶段/世界生成/`）

**引用密度：🔴🔴🔴 高**

#### 交通网络（006-交通网络）：
- 陆地 MST 道路（主干道/次级/小路）+ 海上航线（Coastal/CrossSea/Oceanic）
- `SeaRoute` 数据结构（waypoints/travel_time_days/grade/seasonality）
- 旅行者 NPC 密度（Merchant/Courier/Pilgrim/Wanderer/Adventurer/Bandit/Refugee/Patrol）
- `TransportNetwork` = `RoadNetwork` + `Vec<SeaRoute>`
- `find_route()` 查询函数——输入 TravelMode，输出 Route

#### 体素密度场（007-体素设计决策）：
- Layer 7: 文化覆盖（含道路/建筑地基）——道路刻入密度场
- 铁路轨道的体素表达尚未定义

#### 需要与载具系统协调的问题：
- 铁路轨道在世界生成 P7 如何生成？轨道是密度场的一部分还是独立实体？
- 港口设施的生成——何时、如何选择港口位置？
- 航线生成与载具实际航速的关系——`compute_sailing_time()` 假设了 `BASE_SAILING_SPEED`，但航速应由载具系统定义
- `TravelMode` 枚举应由载具系统权威定义

## 三、已发现的关键矛盾与待裁决问题

### 🔴 严重矛盾：
1. **风力 vs 魔力驱动**：技术栈方案说"魔力驱动——不考虑风力"，但天气系统有详细的"帆船航速= f(风速,风角)"。**需要载具系统权威裁决**

### 🟡 中等模糊：
2. **载具 vs 坐骑边界**：战斗系统说"马不是载具"，但"马车"呢？马是坐骑，车是载具——"马车"的归属？
3. **载具的实体身份**：是物品系统的 ItemEntId 还是独立的 VehicleId？
4. **铁路轨道的归属**：是世界生成的内容，还是需载具系统定义的数据结构？

### 🟢 已对齐但需确认：
5. NPC 的 `VehicleState` 数据结构需要载具系统填充细节
6. 天气系统的载具消费接口假设了"魔力驱动船和帆船同时存在"
7. 世界生成的 `SeaRoute` 依赖载具系统提供 `BASE_SAILING_SPEED`

## 四、载具系统需要定义的权威接口（以载具系统为 Owner）

基于以上分析，载具系统需要提供以下 trait/数据结构的权威定义：

1. **VehicleId** — 载具的全局唯一标识
2. **VehicleDef** — 载具类型的定义数据（对标 ItemDefId/ItemDef）
3. **VehicleState** — 运行时状态（包含 NPC 已用的 VehicleState）
4. **VehiclePhysics** — 移动/速度/加速度模型（明确魔力驱动与自然力的关系）
5. **VehicleQuery trait** — 其他模块获取载具数据的只读接口
6. **VehicleCrewRole** — 船员角色定义（替代 NPC 中的幽灵类型）
7. **VehicleConstruction** — 与建造系统/蓝图系统的接口
8. **VehicleEconomy** — 与经济系统的接口（定价/交易/运营成本）
