## 零、核心哲学

| 原则 | 含义 |
|------|------|
| **载具 = 无意志的移动平台** | 马不是载具（有独立AI），船/火车/马车是（无人格，NPC操控） |
| **载具是社会的延伸** | 载具上的 NPC 继续过日子——移动的微型聚落 |
| **只创造规则，不创造故事** | 航线选择、运费定价、船员雇佣——全由 NPC 心智涌现 |
| **日常全自动，灾害半自动** | 和战斗系统同款哲学——NPC AI = 玩家 AI = 同一套代码 |
| **载具系统只定义物理事实** | 不碰价格、不碰裁决、不碰建造执行——只定义载具是什么 |
| **记忆优先，契书可选** | 所有权在记忆中成立——契书是可选的辅助证据，不是前提条件 |

## 一、总纲与核心概念

### 1.1 载具的双重身份

| 身份 | 标识 | 用途 |
|------|------|------|
| **世界实体** | `VehicleId` | 物理存在、可操控、可受损、可停靠、NPC在其上活动 |
| **产权锚（可选）** | `ItemEntId` (Financial, 契书) | 可放入背包/箱子、可交易、可继承、可有签名/印章 |

### 1.2 核心数据结构

```
VehicleDef        → 载具类型定义（原型×文化涌现，世界生成时确定性计算）
VehicleState      → 运行时状态（位置/损伤/船员/货舱/所有权）
VehicleDamageState → 三通道损伤（hull/propulsion/steering，连续值 1.0→0.0）
VehicleNamingContext → 命名素材（从各模块注入，NPC心智驱动选择）
```

### 1.3 载具分类——从参数涌现，非枚举

- **域**: Land / Water / Air / Amphibious
- **规模**: Tiny(1-2人) / Small(2-8) / Medium(8-30) / Large(30-100) / Huge(100+)
- **推进**: MusclePowered / AnimalTowed / WindPowered / MagicEngine / Hybrid
- **路径约束**: requires_rail, min_water_depth_m, max_water_depth_m
- **能力层**: 槽位（必有） + 活络（可选——大型载具才有 NavMesh）

### 1.4 存储布局与性能

```
载具数量: ~3K-10K（远少于100K+ NPC）
存储: HashMap<VehicleId, VehicleState> + 均匀网格空间索引（格边长1km）
索引: ownership_index(HashMap<EntityId, Vec<VehicleId>>)
      deed_index(HashMap<ItemEntId, VehicleId>)
      type_index(HashMap<VehicleDefId, Vec<VehicleId>>)
总量: ~2-5MB
导航更新: 每载具每1-10s，全量 ~0.05ms（远在0.3ms预算内）
```
不需要 SoA 列式存储——NPC 用 SoA 是因为 100K × rayon 并行需要缓存友好的列布局

### 1.5 CrewRole vs ProfessionTag —— 两层身份

| | ProfessionTag（经济系统） | CrewRole（载具系统） |
|---|---|---|
| **是什么** | 持久身份——"我是铁匠" | 临时任务——"我在船上瞭望" |
| **持续性** | 永久（除非改行） | 仅在上岗轮班时 |
| **谁可以有** | 所有 NPC | 在载具上被分配了岗位的 NPC |
| **如何获得** | 技能累积→经济系统涌现 | 船长（NPC GOAP）或玩家指派 |
| **例子** | Blacksmith, Sailor, ShipCaptain, Navigator | Lookout, HelmOperator, MaintainEngineer |
| **反向涌现** | 长期执行 CrewRole → 累积技能 → 自然获得 ProfessionTag | — |

载具系统只定义 `CrewSlotDef`（岗位模板）——谁上岗是船长 GOAP 决策的结果

## 二、推进与动力

**核心决策**: 动力来源多样性（魔力/人力/畜力/风力/混合），日常航行不受自然力影响，仅灾害级天气施加硬约束。

### 2.1 五种动力类型

| 类型 | 消耗 | 灾害影响 |
|------|------|---------|
| MusclePowered | Stamina（走生命系统Vitals） | 无特殊影响 |
| AnimalTowed | 动物Stamina（走Life trait） | 动物受惊可能失控 |
| WindPowered | 无消耗（帆是免费午餐） | 风暴→桅杆折断风险 |
| MagicEngine | 魔力水晶(ItemEntId) 或 施法者注入 | 魔力风暴→引擎过载风险 |
| Hybrid | 主+辅动力组合 | 取决于组成 |

### 2.2 魔力引擎——走魔法系统 + 物品系统

```
MagicEngine = 载具上的大型魔法物品（对标直接附魔）
  ├─ 内置魔力池 (mana_pool_capacity_ke: f32)
  ├─ 充能: 魔力水晶(消耗品) / 施法者注入 / 港口法阵
  ├─ 符文槽: 卡槽附魔——可替换的效率/抗性符文
  └─ 接口: ManaConsumer trait——载具系统不碰魔法内部逻辑
```

### 2.3 灾害模型

- 日常: 航速 = base_speed × crew_efficiency × damage_penalty（与风向洋流无关——NPC想走什么路线就走什么路线）
- 灾害: WeatherQuery→is_extreme_weather → 速度骤降 + 对应通道损伤 + 船长L3决策
- 所有动力类型统一受灾害影响——差异仅在于损伤通道和响应方式

### 2.4 手动操控

- 仅玩家 + 仅小型载具（`size_class ≤ Small`）
- 对标 Minecraft 划船体验——WASD 直接输入
- NPC 和大型载具总是全自动

## 三、操控与导航（L1-L3 半自动）

### 3.1 三层模型

| 层 | 时间尺度 | 内容 | NPC | 玩家 |
|---|---------|------|-----|------|
| **L3 战略** | 30s-数小时 | 路线选择、灾害响应 | GOAP人格驱动 | 弹出选择菜单 |
| **L2 战术** | 1-10s | 航向保持、速度调节、避障 | 自动执行 | 小载具可选手动 |
| **L1 本能** | 被动 | 紧急修理、弃船 | 自动触发 | 自动触发 |

### 3.2 船员效率

```
crew_efficiency = Σ(slot.staffing × slot.skill_factor × slot.fatigue_penalty) / slot_count
skill_factor = 0.3 + 0.7 × min(skill_level/100, 1.0)
fatigue_penalty = 1.0 - 0.4 × vitals.fatigue
```

### 3.3 VehicleController trait

- NPC 和玩家实现同一 trait
- plan_route(destination, knowledge, network) → Route —— NPC用主观认知，非上帝视角
- respond_to_hazard(hazard, vehicle) → HazardResponse —— 人格驱动

## 四、损伤、维修与沉没

### 4.1 三通道损伤

```
hull:       1.0 → 0.0（进水/解体/沉没）
propulsion: 1.0 → 0.0（无法移动）
steering:   1.0 → 0.0（失控漂移）
```

损伤效果非线性：>0.7 正常 / 0.4-0.7 中度惩罚 / 0.1-0.4 严重 / <0.1 濒危

### 4.2 维修

| 模式 | 场所 | 恢复量 | 持久性 |
|------|------|--------|--------|
| 紧急修补 | 海上/野外 | ≤0.3 | 12-72h衰退 |
| 港口维修 | 船坞+船匠 | 全量恢复 | 永久 |

### 4.3 沉没与搁浅

- hull=0 → 沉没倒计时开始
- 浅水区域(<min_water_depth) → 搁浅而非沉没（hull可降至0但载具不会消失→变成可探索残骸）
- 深水区域 → 沉没 → VehicleStatus::Wrecked
- 残骸 → 灵元素印记(AetherImprint, 走History系统) + 历史事件
- 船员/乘客: GOAP触发AbandonShip/SavePassengers/SalvageCargo
- VehicleId保留但标记为destroyed——契书变成废纸

## 五、产权与所有权

### 5.1 所有权 = 记忆中成立，不依赖契书

| 路径 | 机制 | 证据 |
|------|------|------|
| 口头交易 | 买卖双方记忆更新 + PowerEdge更新 | 卖方承认=最强证据 |
| 契书交易 | 口头交易 + 签署契书(ItemEntId) | 契书+签名+见证人 |
| 事实夺取 | 新主单方面记忆 + PowerEdge(Emergent/Conquered) | 旧主记忆"被抢了" |
| 原始建造 | 建造完成→建造者即所有者 | 建造事实 |
| 继承 | SuccessionRule自动触发 | 权力系统继承规则 |

### 5.2 契书伪造与篡改——走已有系统

| 场景 | 操作 | 走哪个系统 | 后果 |
|------|------|-----------|------|
| **偷契书** | 扒窃→纸被偷走 | 物品系统（盗窃=从对方库存移入自己库存） | PowerEdge不变——有纸没船 |
| **篡改契书** | 改动buyer字段+伪造签名 | 技能系统(Linguistics+Deception) → ItemState被修改 | tamper_marks新增——检测者vs篡改者技能对抗 |
| **伪造契书** | 从头创建假契书 | 同上——LanguageExpression生成假签名 | 一张假纸。PowerEdge不存在→上法庭被识破 |
| **合法交易** | 双方签署+PowerEdge更新 | 权力系统Contract + 经济系统Wallet | 完整签名链+记忆一致 |

### 5.3 裁决——权力系统 Adjudicate 原子

- 载具系统维护 `VehicleOwnershipRecord`（所有权事件链 + 活跃主张）
- 裁决由权力系统的 Adjudicate 原子执行——考虑证据、记忆、证人、法律规范
- 载具系统不裁决

## 六、经济与货运

### 6.1 载具只定义物理事实

```
VehiclePhysicalProfile:
  cargo_capacity_kg, cargo_capacity_liters, passenger_capacity
  fuel_item, fuel_consumption_per_100km, wear_rate_per_100km
  min_crew, crew_salary_reference（参考——非公式，实际工资由雇佣谈判决定）
```

### 6.2 货运 = 移动容器

```
VehicleState.cargo_hold: ContainerId  → 对标物品系统 Container
装卸 = 载具货舱 ↔ 码头仓库/市场
定价 = 船长NPC心智（成本估算+利润目标+竞争）→ 对话→Contract
全部走已有系统——经济/物品/NPC/权力/语言表达
```

## 十、跨模块接口

### 10.1 VehicleQuery trait（只读——所有模块消费）

```
vehicle() / position() / damage_state() / motion_vector()
collision_bodies() / cover_surfaces()           ← 战斗热路径
vehicles_in_radius() / vehicles_at_port()        ← 空间查询
vehicles_owned_by() / is_crew_on_duty()          ← 所有权查询
available_transport()                             ← 航线查询
physical_profile() / available_cargo_capacity()  ← 经济查询
```

### 10.2 VehicleMut trait（pub(crate)——仅授权模块写入）

```
update_navigation() / apply_damage()
set_crew_slot() / load_cargo() / unload_cargo()
register_vehicle() / deregister_vehicle()
```

### 10.3 消费者矩阵

| 消费者 | 使用的方法 | 频率 |
|--------|----------|------|
| NPC GOAP | vehicles_at_port, available_transport, is_crew_on_duty | 1-10s |
| 战斗 | damage_state, motion_vector, collision_bodies, cover_surfaces | 每帧 |
| 天气 | position（灾害区域判断） | 1-10s |
| 经济 | physical_profile, available_cargo_capacity | NPC商业决策时 |
| 世界生成 | register_vehicle（P8.5） | 一次性 |
| 历史 | vehicle（全量快照→事件记录） | 事件发生时 |
| 权力 | vehicles_owned_by（产权争议） | 偶尔 |
| 渲染(Godot) | position（每帧共享内存直读） | 每帧 |

## 十二、载具相关技能归属

全部归入已有五分类——不新增 SkillCategory：

| 技能 | 归属 | 说明 |
|------|------|------|
| Navigation（导航） | Academic | 地图阅读、罗盘使用、天文导航 |
| Sailing（船艺） | Survival | 操舵、收帆、划桨——实践型 |
| Shipwright（造船） | Artisan → Construction | 对标铁匠/木匠——工程类 |
| NavalCombat（海战） | Combat | 舰载武器操作、海上战术 |
| Engineering（机械） | Artisan | 维护魔力引擎、修理机械部件 |

载具系统只在 `CrewSlotDef.required_skill` 中引用 SkillId。XP 曲线、成长、教学——全走技能系统
