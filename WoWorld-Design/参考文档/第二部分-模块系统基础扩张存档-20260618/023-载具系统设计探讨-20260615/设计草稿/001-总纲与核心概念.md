## 零、核心哲学

### 0.1 载具 = 无意志的移动平台

这是载具系统最根本的定义——**将载具与坐骑彻底分离**：

| | 载具（本模块管辖） | 坐骑（战斗系统+生命系统管辖） |
|---|---|---|
| **意志** | 无——由 NPC 操控 | 有——独立 AI（恐惧/脾气/疲劳） |
| **AI** | 无自身AI——船员 GOAP 驱动 | 走 Life 基类的完整 AI |
| **战斗身份** | 战场环境（碰撞体+掩体+运动向量） | 独立参战方（骑乘战斗 011） |
| **例子** | 船、火车、马车、飞行船 | 马、飞行坐骑、驮兽 |
| **边界** | 马车：车体=载具，马=坐骑。两者协作——载具定义拉力需求，马走 Life AI | — |

### 0.2 六条设计原则

| # | 原则 | 含义 |
|---|------|------|
| 1 | **载具是社会的延伸** | 载具上的 NPC 继续过日子——移动的微型聚落 |
| 2 | **只创造规则，不创造故事** | 航线选择、运费定价、船员雇佣——全由 NPC 心智涌现 |
| 3 | **日常全自动，灾害半自动** | NPC AI = 玩家 AI = 同一套代码，对标战斗系统 |
| 4 | **只定义物理事实** | 不碰价格、不碰裁决、不碰建造执行 |
| 5 | **记忆优先，契书可选** | 所有权在记忆中成立——契书是辅助证据，非前提 |
| 6 | **动力来源不约束航行** | 日常 NPC 想走什么路线就走什么路线——自然力仅灾害级起效 |

### 0.3 管辖边界——载具系统定义什么、不定义什么

```
载具系统定义:                    载具系统不定义:
├─ VehicleId (世界实体)           ├─ 运费公式 → NPC心智 + 经济系统
├─ VehicleDef (类型定义)          ├─ 所有权裁决 → 权力系统 Adjudicate
├─ VehicleState (运行时状态)      ├─ 建造流程 → 建造系统
├─ VehicleDamageState (损伤)     ├─ 船员雇佣合同 → 权力系统 Contract
├─ VehicleQuery trait (只读)     ├─ 载具上的战斗规则 → 战斗系统
├─ VehicleMut trait (写入)       ├─ 载具模型/渲染 → Godot
├─ CrewSlotDef (岗位模板)        ├─ 载具在物品栏中的图标 → 物品系统
├─ VehiclePhysicalProfile (物理) └─ 载具命名素材内容 → 各模块提供
└─ VehicleConstructionSpec (建造需求)
```

## 一、载具的双重身份

每艘船/每列火车/每辆马车都有两个身份——可同时存在，也可仅有一个：

| 身份 | 标识 | 何时创建 | 何时销毁 | 在哪里 |
|------|------|---------|---------|--------|
| **世界实体** | `VehicleId` | 建造完成时 | 沉没/彻底拆解 | 永远在物理世界中（停泊/航行） |
| **产权契书** | `ItemEntId` (Financial) | 建造完成时（可选）或首次正式交易时 | 载具沉没后变成废纸 | 可放背包/箱子/保险柜——一张纸 |

**关键约束**：
- 载具**永远不进背包**——物理载具在世界中，停泊在码头/车站/马厩
- 契书是**一张纸**——对标经济系统的欠条（ItemCategory::Financial）
- 契书**可选**——口头交易同样成立所有权（§五），契书只是辅助证据
- 魔法载具可以有 `sealable: true`——被封印后化为小物件放入物品栏（走附魔系统，非常规机制）

## 五、VehicleDamageState —— 三通道损伤

```rust
/// 载具损伤——三通道连续值模型
/// 对标战斗系统 012 的部位伤害（轻/中/重伤三档），但用连续值表达
/// 三个通道独立损伤、独立维修
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct VehicleDamageState {
    /// 船体完整性 1.0(完好) → 0.0(解体/沉没)
    pub hull: f32,

    /// 推进系统 1.0(完好) → 0.0(完全无法移动)
    pub propulsion: f32,

    /// 操控系统 1.0(完好) → 0.0(舵卡死/失控漂移)
    pub steering: f32,

    /// 紧急修补记录——临时修复随时间衰退
    pub emergency_patches: Vec<EmergencyPatch>,
}

/// 损伤通道——用于 apply_damage() 定位
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum DamageChannel {
    Hull,
    Propulsion,
    Steering,
}

impl VehicleDamageState {
    /// 损伤效果——软阈值，非线性
    /// >0.7 正常 / 0.4-0.7 中度 / 0.1-0.4 严重 / <0.1 濒危
    pub fn speed_penalty(&self) -> f32 {
        let hull_factor = soft_threshold(self.hull);
        let prop_factor = soft_threshold(self.propulsion);
        // 取两个通道的最差情况——船体破和引擎坏都影响速度
        hull_factor.min(prop_factor)
    }

    pub fn steering_penalty(&self) -> f32 {
        soft_threshold(self.steering)
    }
}

/// 软阈值函数——连续值 → 连续的效应
/// 输入 0-1，输出 0-1
/// 1.0→1.0, 0.7→0.95, 0.4→0.5, 0.1→0.1, 0.0→0.0
fn soft_threshold(value: f32) -> f32 {
    // 使用平滑阶梯函数——在 0.7 和 0.4 附近有加速下降
    // 但始终连续可导——无突变
    let severe = smoothstep(0.1, 0.4, value);
    let moderate = smoothstep(0.4, 0.7, value);
    0.1 * severe + 0.4 * moderate + 0.5 * value
}
```

## 六、CrewSlotDef —— 船员岗位模板

```rust
/// 船员岗位定义——VehicleDef 的一部分
/// 载具系统定义"这个载具需要什么岗位"——但不定义"谁上"
/// 谁上岗是船长（NPC GOAP）或玩家（手动指派）的决策
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CrewSlotDef {
    pub slot_id: CrewSlotId,

    /// 岗位标签——"舵手"、"瞭望员"、"司炉"
    /// 此标签由文化系统本地化——不同文化对同一岗位的称呼可能不同
    pub label_key: LocalizationKey,

    /// 需要什么技能（SkillId 由技能系统权威定义）
    pub required_skill: SkillId,
    /// 最低胜任水平（技能等级）
    pub min_proficiency: f32,

    /// 岗位职责——影响 NPC 的 GOAP 目标
    pub duties: Vec<CrewDuty>,

    /// 此岗位需要的人数——大船可能同时需要 3 个瞭望员
    pub count: u32,

    /// 是否有等级（船长有 rank，普通水手没有）
    pub has_rank: bool,
    /// 如果有等级——可授予的最高等级
    pub max_rank: Option<CrewRank>,

    /// 是否必需岗位——缺员时载具会有严重效率惩罚
    /// 舵手缺员 → 无法航行；瞭望缺员 → 视野缩减但能走
    pub critical: bool,
}

/// 船员岗位的具体职责——NPC 在岗时执行这些 GOAP 目标

    /// 操舵/保持航向
    Navigate,
    /// 瞭望——前方陆地/礁石/船只/海盗/天气变化
    Lookout,
    /// 维护——修理/排水/灭火/清洁/引擎保养
    Maintain,
    /// 服务乘客——餐饮/安抚/引导
    ServePassengers,
    /// 管理货物——装卸/固定/清点
    ManageCargo,
    /// 防御——战斗时操作舰载武器/抵御登船

    /// 操作特定设备——魔力引擎/起重机/锚/帆
    OperateEquipment,

## 七、CrewRole vs ProfessionTag —— 两层身份

这是载具系统与经济系统的关键接口。

```
┌──────────────────────────────────────────────────────────┐
│              ProfessionTag（经济系统 004）                 │
│                                                          │
│  持久身份——"我是谁"                                        │
│  ├─ Sailor, ShipCaptain, Navigator, Fisherman, ...       │
│  ├─ 决定 NPC 的默认技能 proficiency                        │
│  ├─ 永久——除非 NPC 改行（经济系统涌现）                      │
│  └─ 不影响 NPC 在船上的具体岗位                             │
│                                                          │
│  一个 Blacksmith 可以上船当临时水手——技能低但合法             │
│  一个 Sailor 不上船时仍然是 Sailor——在港口打零工             │
└────────────────────┬─────────────────────────────────────┘
                     │
        CrewSlotDef.required_skill 从 ProfessionTag 推导基线
        但 NPC 可以不持有相关 ProfessionTag 也能上岗
                     │
┌────────────────────▼─────────────────────────────────────┐

## 八、存储布局与性能

```rust
/// 载具注册表——主存储
/// 载具数量 ~3K-10K，远少于 100K+ NPC
/// 因此不需要 SoA 列式存储——直接用 HashMap
pub struct VehicleRegistry {
    /// 主存储——VehicleId → VehicleState
    /// 1万条目 HashMap 开销 ~200KB，可忽略
    vehicles: HashMap<VehicleId, VehicleState>,

    /// 空间索引——均匀网格（格边长 1km）
    /// 稀疏存储：只存有载具的格子
    spatial_grid: HashMap<GridCell, Vec<VehicleId>>,

    /// 所有者索引——"这个 NPC/Group 有哪些载具"
    ownership_index: HashMap<EntityId, Vec<VehicleId>>,

    /// 契书索引——ItemEntId → VehicleId
    deed_index: HashMap<ItemEntId, VehicleId>,

    /// 类型索引——VehicleDefId → VehicleId（世界生成用）
    type_index: HashMap<VehicleDefId, Vec<VehicleId>>,
}

/// 网格坐标——用 i32 避免浮点哈希问题
#[derive(Hash, Eq, PartialEq, Clone, Copy)]
struct GridCell(i32, i32);

impl GridCell {
    fn from_world_pos(pos: Vec2) -> Self {
        GridCell(
            (pos.x / 1000.0).floor() as i32,
            (pos.y / 1000.0).floor() as i32,
        )
    }
}
```

| 查询 | 路径 | 复杂度 | 1万载具下约 |
|------|------|--------|-----------|
| 按 ID 查 | `vehicles.get(id)` | O(1) | ~50ns |
| 某位置半径内 | `spatial_grid[cell]` + 相邻格 | O(船只密度) | ~1µs |
| 某 NPC 的载具 | `ownership_index[npc_id]` | O(1) | ~50ns |
| 所有商船 | `type_index[def_id]` | O(1) | ~100ns |
| 契书→载具 | `deed_index[item_ent_id]` | O(1) | ~50ns |

**性能预算**（技术栈方案 v3：载具逻辑 ≤0.3ms/帧）：
- 导航更新：~0.05ms（全量 10K 载具，每 1-10s 一次）
- 空间索引更新：仅跨越网格边界时——绝大多数帧为 0
- 战斗查询（damage_state/motion_vector）：仅参与战斗的少数载具，每帧 ~1µs
- **总计远在 0.3ms 预算内**
