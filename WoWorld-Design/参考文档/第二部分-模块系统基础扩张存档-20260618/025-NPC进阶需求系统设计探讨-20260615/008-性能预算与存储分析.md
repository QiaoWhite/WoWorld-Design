# 008 — 性能预算与存储分析

## 一、存储增量

### 1.1 NpcData 新增字段

```rust
pub struct NpcData {
    // ... 已有字段 ...

    // 进阶需求 (v2.0 新增)
    pub esteem_deficit: f32,                          // 4 bytes
    pub competence_frustration: f32,                   // 4 bytes
    pub competence_frustration_chronic_days: u16,      // 2 bytes
    // padding: 2 bytes (对齐到 4)
    // 总计: 12 bytes / NPC
}
```

如果 `u16` 浪费了对齐空间，可以合并：
```rust
    pub esteem_deficit: f32,                          // 4 bytes
    pub competence_frustration: f32,                   // 4 bytes
    pub competence_frustration_chronic_days: u16,      // 2 bytes (并入已有的某个 u16 字段？)
```

### 1.2 NeedSensitivity 新增字段

```rust
pub struct NeedSensitivity {
    // 已有 (6 × f32 = 24 bytes)
    pub hunger: f32,
    pub thirst: f32,
    pub fatigue: f32,
    pub element_balance: f32,
    pub libido: f32,
    pub social: f32,

    // 新增 (2 × f32 = 8 bytes)
    pub esteem: f32,
    pub competence: f32,
    // 总计: 32 bytes (不变的对齐)
}
```

### 1.3 总计

| 项目 | 每NPC增量 | 100K NPC | 1M NPC |
|------|----------|----------|--------|
| NpcData 新字段 | ~12 bytes | 1.2 MB | 12 MB |
| NeedSensitivity 扩展 | 8 bytes | 0.8 MB | 8 MB |
| **总计** | **~20 bytes** | **2.0 MB** | **20 MB** |

对比基线：
- 当前 NpcData ~2KB/NPC → 200MB/100K
- 增量 20 bytes / 2KB = **+1%**
- 在 Rust 模拟总内存 (~1.4GB) 中占比 < 0.15%

**结论**：存储成本可忽略。

## 二、热路径分析（每决策周期）

### 2.1 操作分解

每次概率决策器运行（L1 NPC，每 ~0.5-5s 一次）：

| 操作 | 类型 | 估算成本 | 备注 |
|------|------|---------|------|
| need_action_match: 9维 urgency | 9次查表 + 18次乘法 | ~30 ns | 原来7维~24ns，+6ns |
| intrinsic_motivation_weight | ≤6次 relevance 查表 + ≤6次乘法 | ~25 ns | 平均3个active goals |
| survival_suppression | 1次 max + 1次 exp | ~20 ns | exp ≈ 15ns on modern CPU |
| frustration_regression check | 1次 mean + 条件分支 | ~5 ns | 几乎总是 false（avg<0.6） |
| frustration_regression apply | 4次乘法（如果触发） | ~5 ns | 罕见触发 |
| emotion: apply_psychological_pull | 3次比较 + 最多6次乘加 | ~10 ns | 每情绪tick |

**总计增量**：~85 ns / 决策周期（含 exp）

### 2.2 在预算中的占比

| 预算项 | 预算 | 新增消耗 | 占比 |
|--------|------|---------|------|
| L1 决策周期 (概率引擎) | ~5 µs | +85 ns | +1.7% |
| 每帧总 CPU (Rust 侧) | ~8 ms | +~0.02 ms | +0.25% |
| 每帧总 CPU (含 Godot) | 16.7 ms | +~0.02 ms | +0.12% |

**结论**：热路径成本可忽略。sigmid 中的 `exp()` 是最贵的单个操作（~15ns），但每决策周期只调一次。

## 三、非热路径分析

### 3.1 每日更新（所有 NPC，包括 L2-L4）

competence_frustration 每游戏日更新一次（对所有有 SkillMastery aspiration 的 NPC）：

| 操作 | 频率 | 单次成本 | 100K NPC总成本/游戏日 |
|------|------|---------|---------------------|
| 查询 SkillQuery::get_level | 1次/aspiration | ~50 ns | 极少（仅L1-L2） |
| 计算 gap | ~3次浮点 | ~5 ns | — |
| 慢性累积更新 | u16 ±1 | ~2 ns | — |
| personality modulation | 4次乘法 | ~5 ns | — |

每游戏日 = 48 分钟真实时间。所以这是**极端非热路径**——平均每秒 0.003 次调用。

**结论**：每日更新的成本在统计噪声级别。

### 3.2 SelfNarrative 反思（每 7 游戏天）

SelfNarrative.reflect() 已存在——我们只增加：
- esteem 自我评估（扫描近期 achievement memories）
- competence_frustration 的 LifeTheme 转换检查

这些增加的操作都是线性扫描已有的 memory 列表——成本与已有的反思逻辑相当。

## 五、缓存友好性

### 5.1 NpcData 的缓存行布局

```
NpcData (~2KB):
  [...已有字段...]
  esteem_deficit: f32          ← 放在 social_deficit 附近（同属心理需求）
  competence_frustration: f32  ← 同上
  competence_frustration_chronic_days: u16
  [...其他字段...]
```

这些字段与 social_deficit 和 NeedSensitivity 一起被访问。将它们放在 NpcData 中 social_deficit 附近的布局位置，可以确保**同一次决策中需要的 3 个心理需求值都在同一个缓存行内**。

### 5.2 SoA 考虑

如果 NPC 系统使用 SoA（Structure of Arrays）存储，则：
```
esteem_deficits: Vec<f32>        ← 连续内存，SIMD 友好
competence_frustrations: Vec<f32>
chronic_days: Vec<u16>
```

这允许向量化 need_action_match 的批量计算——但 v1.0 不做 SoA（对齐已有 AoS 设计）。

## 六、与现有系统的性能对比

| 系统 | 每NPC内存 | 热路径 | 总预算占比 |
|------|----------|--------|-----------|
| 基本需求 (7维, CHG-027) | 48 bytes | ~60 ns | 0.49% |
| **进阶需求 (本模块)** | **20 bytes** | **~85 ns** | **~0.7%** |
| SelfNarrative (已有) | ~200 bytes | 每7天 | 摊还 ≈ 0 |
| 情绪引擎 (已有) | ~100 bytes | ~500 ns | ~4% |
| 概率决策器 (已有) | — | ~5 µs | ~40% |

**进阶需求的增量 < 1% 的决策预算**——显著低于基本需求，因为我们没有引入新的生物代谢 tick，大部分逻辑是条件触发。

## 七、优化预留

### 7.1 当前不需要但可考虑的优化

1. **relevance 查找表**：`[GoalType; 6] × [ActionType; ~50]` → 300 个 f32 的静态表 → 完全编译时求值
2. **commitment 缓存**：commitment 在 SelfNarrative 反思时才变化（7天周期）——可以缓存到 goal 结构体中而非每帧重算
3. **survival_suppression 近似**：exp() 可以用快速近似（如 Padé 逼近）——精确度损失 < 0.001，性能提升 ~5×

### 7.2 v1.0 决定

**不做优化**。当前成本已经足够低（<1% 预算），过早优化违反"先让它工作，再让它快"原则。
