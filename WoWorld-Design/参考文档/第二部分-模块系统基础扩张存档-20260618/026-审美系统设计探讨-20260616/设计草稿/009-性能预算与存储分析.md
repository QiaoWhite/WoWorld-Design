# 009 — 性能预算与存储分析

## 一、存储分析

### 1.1 审美模块自身存储：零

审美模块是无状态引擎——不存储任何持久数据。对标 `WeatherQuery::sample()` 模式。

### 1.2 NPC 侧新增存储：AestheticTaste

| 字段 | 大小 | 说明 |
|------|------|------|
| `AestheticTaste` | 32 bytes | Copy 类型，6×f32 + 3×f32 + u32 |
| 每 NPC 新增 | 32 bytes | — |
| 10,000 NPC | 320 KB | 可忽略 |
| 100,000 NPC（大型世界） | 3.2 MB | SoA 布局 → L2 友好 |

**对标**：ReligiousPracticeProfile（~40B/NPC）——同量级。

### 1.3 记忆系统新增：AestheticMemory

| 字段 | 大小 | 说明 |
|------|------|------|
| `AestheticMemory` | ~48 bytes | target_hash(8B) + AestheticJudgment(16B) + somatic_impact(4B) + depth(4B) + timestamp(8B) + location(12B) |
| 每 NPC 每天平均写入 | 3-5 条 | 仅 depth≥0.1 时写入 |
| 记忆上限 | 2000 条（已有） | 审美记忆与其他记忆共享上限 |

与已有的 2000 条 NPC 记忆上限共享——审美记忆不独立占用配额。

## 二、CPU 预算

### 2.1 热路径（每帧/每 tick 执行）

| 操作 | 频率 | 每操作成本 | 总量（10,000 NPC） |
|------|------|-----------|-------------------|
| `HasAestheticSignal::aesthetic_signal()` | 仅在 Attend 时 lazy 计算 | ~100ns/次（查表+插值） | — |
| `judge()` | 每 NPC 每 5s 对 1-3 个目标 | ~150 FP 指令/次 | 10000×2×150÷5000ms ≈ **600 FP/ms** ≈ **0.0006ms/帧** |
| `React` 原子 | 同 judge() | ~50 FP 指令/次 | 10000×2×50÷5000ms ≈ **0.0002ms/帧** |

**热路径总量 < 0.001ms/帧**。远低于天气系统的 0.25ms/帧预算。

### 2.2 冷路径（事件驱动）

| 操作 | 频率 | 每操作成本 | 说明 |
|------|------|-----------|------|
| `Articulate` 原子 | ~1-5 次 / NPC / 游戏日 | 模板 <1µs, 深度批评 <50µs | 极低频，不影响帧预算 |
| `Adopt` 原子 | ~1-3 次 / NPC / 游戏日 | ~200 FP 指令/次 | 社交事件驱动 |
| `Embellish` 原子 | ~0.01-0.1 次 / NPC / 游戏日（仅创作者） | ~500 FP 指令/次 | 仅在创作时触发 |
| `mature_taste()` | 每 NPC 每年 1 次 | ~500 FP 指令/次 | 对标文化 drift，可分摊到多帧 |
| `traumatic_taste_shift()` | 极罕见 | ~200 FP 指令/次 | 几乎从不触发 |

### 2.3 最差情况

节日场景：1000 NPC 同时观看 1 个表演，每 2 秒 judge() 一次：

```
1000 × 150 FP ÷ 2000ms = 75 FP/ms ≈ 0.075ms/帧
```

仍然远低于天气系统预算。

## 三、VRAM / 内存

审美模块不新增 VRAM 使用。不引入新纹理、新 mesh、新 shader。

---

## 四、可并行性

- `judge()` 纯函数 → 完全可并行化（rayon 友好）
- 每个 NPC 的审美判断彼此独立 → SoA 数据布局 → SIMD 友好
- `HasAestheticSignal` 各实现者独立 → 无锁

## 五、TUNING 参数

对标已有模块，以下参数应为 TOML 可调：

```toml
[aesthetic]
# Signal jitter 幅度
jitter_range = 0.02

# judge() 内部参数
[aesthetic.judge]
novelty_valence_sigma = 0.15       # novelty 倒 U 宽度
complexity_valence_sigma = 0.25    # complexity 倒 U 宽度
arousal_sigmoid_steepness = 8.0    # arousal sigmoid 陡度

# React 原子
[aesthetic.react]
base_emo_weight = 0.15             # 情绪 delta 基础权重
familiarity_valence_penalty = 0.6  # 熟悉度对 valence 的衰减系数
familiarity_floor = 0.2            # valence 衰减地板

# Articulate 原子
[aesthetic.articulate]
expression_threshold_silent = 0.2
expression_threshold_social = 0.4
expression_threshold_critique = 0.7
pressure_suppression_factor = 0.95

# Adopt 原子
[aesthetic.adopt]
max_dim_shift = 0.25               # 单次最大维度偏移百分比
min_encounters_for_style_adopt = 5 # 反复接触触发阈值
adopt_window_days = 30

# Taste 演化
[aesthetic.taste]
complexity_age_peak = 45           # complexity_tolerance 峰值年龄
complexity_age_decline_rate = 0.008 # 老年下降率
virtuosity_skill_gain_cap = 0.25   # 技能增长对 virtuosity 权重的上限
novelty_travel_gain_cap = 0.20     # 旅行对 novelty 权重的上限

# Embellish 原子
[aesthetic.embellish]
novelty_skill_requirement = 0.2    # novelty 对技能的依赖（低——创意≠技术）
virtuosity_skill_requirement = 0.95 # virtuosity 对技能的依赖（高）
expressiveness_skill_requirement = 0.3 # expressiveness 对技能的依赖（低——真诚>技术）
noise_scale_low_skill = 0.15       # 低技能时的噪声幅度
```

## 七、总结

审美模块的运行时成本极低：
- 热路径 <0.001ms/帧（1万NPC规模）
- 冷路径事件驱动，不在帧预算内
- 存储增量 32B/NPC（1万NPC = 320KB）
- 零 VRAM 增量
- 纯函数 judge() 完全可并行化

**架构简洁 + 性能安全 + 存储轻量。**
