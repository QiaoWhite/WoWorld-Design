## 一、总览

建筑生成是 P6 阶段。v3 核心变化：
- **WFC 扩展到两层**：城市级（功能区→建筑类型填充）+ 建筑级（房间类型→房间布局）
- **模板 = 预定义 WFC 约束集**——不是预制 3D 模型，不是跳过 WFC
- **模板与个性化同系统双入口**——同算法、同输出格式，仅约束来源不同
- **CultureCoreParams→建筑风格**的完整映射链
- 保留质量衰减模型

## 二、两层 WFC 架构

```
P5 产出: 城市结构 + 功能分区 + 重要建筑定位
         │
         ▼
  ┌─────────────────────────────────────┐
  │  城市级 WFC（新增）                   │
  │  · 瓦片 = 建筑功能类型                 │
  │  · 邻接 = 功能区之间的建筑类型约束      │
  │  · 输出 = 每个功能区内的建筑类型分布    │
  └─────────────────────────────────────┘
         │
         ▼
  ┌─────────────────────────────────────┐
  │  建筑级 WFC（保留+扩展）              │
  │  · 瓦片 = 房间功能类型                 │
  │  · 邻接 = 房间间的允许/禁止/必须相邻    │
  │  · 输出 = 每个建筑的房间布局平面图      │
  └─────────────────────────────────────┘
         │
         ▼
    内饰填充 + 外装修 → 完整建筑
```

### 2.1 城市级 WFC（新增）

在每个功能区内，用 WFC 决定建筑类型的分布。

```rust
/// 城市级 WFC 的瓦片定义
struct CityWfcTile {
    /// 建筑功能类型
    building_type: BuildingFunction,
    /// 建筑密度权重
    density_weight: f32,
    /// 最小/最大地块面积
    plot_size: (f32, f32),
    /// 是否必须临街
    require_street_access: bool,
    /// 是否偏好角落位置
    prefer_corner: bool,
    /// 对邻近建筑类型的约束
    adjacency_overrides: HashMap<BuildingFunction, AdjacencyRule>,
}

/// 城市级 WFC 运行
fn run_city_wfc(zone: &FunctionalZone, culture: &CultureCoreParams) -> Vec<BuildingLot> {
    let tiles = build_tile_set(zone.zone_type, culture);
    let adjacency = build_city_adjacency_matrix(zone.zone_type);
    let constraints = CityWfcConstraints {
        min_buildings: zone.expected_building_count(),
        max_buildings: zone.expected_building_count() * 1.2,
        street_grid: zone.street_grid.clone(),
    };

    wfc_collapse(&tiles, &adjacency, &constraints, &mut zone.rng())
}
```

**简单区域（如普通居住区）可以用模板**：预定义的建筑类型分布比例+随机打散。城市级 WFC 主要用于复杂区域（市中心、市场区、宗教区——建筑类型多样且约束强）。

### 2.2 建筑级 WFC（保留+扩展）

每个建筑的房间布局——这是 WFC 的核心应用。保留原方案的邻接规则矩阵。

```rust
/// 建筑级 WFC 的瓦片定义
struct RoomTile {
    function: RoomFunction,
    min_area: f32,              // 最小面积 m²
    max_area: f32,              // 最大面积 m²
    min_count: u32,             // 此类型最少出现数
    max_count: u32,             // 最多出现数
    allowed_adjacent: Vec<RoomFunction>,
    required_adjacent: Vec<RoomFunction>,  // ⭐ 必须相邻
    forbidden_adjacent: Vec<RoomFunction>, // ❌ 禁止相邻
    prefer_exterior: bool,      // 偏好靠外墙（采光）
    prefer_ground_floor: bool,  // 必须在一楼
    weight: f32,                // 生成权重
}
```

### 2.3 邻接规则矩阵（保留原方案——以民居为例）

行=A，列=B。✅=允许，❌=禁止，⭐=必须。

|  | 主厅 | 卧室 | 厨房 | 储物 | 走廊 | 书房 | 工坊 |
|--|------|------|------|------|------|------|------|
| **主厅** | — | ✅ | ✅ | ✅ | ⭐ | ✅ | ❌ |
| **卧室** | ✅ | — | ❌ | ✅ | ⭐ | ✅ | ❌ |
| **厨房** | ✅ | ❌ | — | ⭐ | ✅ | ❌ | ❌ |
| **储物** | ✅ | ✅ | ⭐ | — | ✅ | ❌ | ✅ |
| **走廊** | ⭐ | ⭐ | ✅ | ✅ | — | ✅ | ✅ |
| **书房** | ✅ | ✅ | ❌ | ❌ | ✅ | — | ❌ |
| **工坊** | ❌ | ❌ | ❌ | ✅ | ✅ | ❌ | — |

> 此为基础默认矩阵。不同文化和建筑功能有不同的邻接变体。例如北方寒冷文化要求厨房与主厅必须相邻（共用炉火取暖）。

## 三、模板 vs 个性化——同一系统，双入口

### 3.1 核心澄清

模板 ≠ 预制 3D 模型。模板 ≠ 跳过 WFC。

**模板 = 预定义的 WFC 约束集**。模板路径和个性化路径**跑同一个 WFC 算法**，输出格式完全相同。唯一区别：约束集从缓存加载还是动态生成。

```
模板路径:     查约束缓存 → WFC → 布局 → 文化覆盖 → 完成
个性化路径:   动态构建约束集 → WFC → 布局 → 文化覆盖 → 完成
```

### 3.2 模板约束集定义

```rust
/// 一个模板 = 一套完整的 WFC 约束参数
struct WfcTemplate {
    /// 模板 ID
    id: TemplateId,
    /// 适用的建筑功能
    building_function: BuildingFunction,
    /// 适用的文化风格（可选——None 表示跨文化通用）
    culture_style: Option<CultureStyleId>,
    /// 瓦片集定义
    tile_set: Vec<RoomTile>,
    /// 邻接规则矩阵
    adjacency_matrix: AdjacencyMatrix,
    /// 全局约束
    global_constraints: GlobalConstraints,
    /// 面积占比建议
    area_ratios: HashMap<RoomFunction, f32>,
}

struct GlobalConstraints {
    total_area_range: (f32, f32),   // m²
    floor_count_range: (u32, u32),
    min_total_rooms: u32,
    max_total_rooms: u32,
}
```

### 3.3 模板覆盖策略

| 建筑类型 | 使用模板 | 理由 |
|----------|---------|------|
| 普通民居 | ✅ 90% | 布局高度重复，模板高效 |
| 商铺 | ✅ 70% | 基本结构相似，内饰不同 |
| 铁匠铺 | ✅ 80% | 功能布局固定（锻炉+铁砧+淬火槽+储物） |
| 酒馆 | ✅ 50% | 结构相似但内饰差异大 |
| 神殿/教堂 | ❌ 0% | 每种文化+神祇组合需要独特设计 |
| 王宫/要塞 | ❌ 0% | 聚落唯一，需要独特性 |
| 学院 | ❌ 20% | 规模差异大 |

### 3.4 约束生成器（个性化路径）

当没有匹配的模板时，从文化参数+建筑功能动态推导约束集：

```rust
trait ConstraintGenerator {
    /// 从建筑功能推导基础瓦片集
    fn derive_tile_set(function: BuildingFunction) -> Vec<RoomTile>;

    /// 从文化参数推导文化特定的邻接偏好
    fn culture_adjacency_modifiers(culture: &CultureCoreParams) -> AdjacencyModifiers;

    /// 从气候+地域推导房间偏好
    fn climate_room_preferences(climate: &ClimateZone) -> RoomPreferences;

    /// 综合——生成最终约束集
    fn generate_constraints(
        function: BuildingFunction,
        culture: &CultureCoreParams,
        climate: &ClimateZone,
        plot_size: (f32, f32),
    ) -> WfcConstraints;
}
```

### 3.5 缓存机制

个性化路径产出的约束集，如果布局验证通过且评分高，可被缓存为"已验证模板"——下次同文化+同功能的建筑直接复用。

```rust
struct TemplateCache {
    // 已验证的约束集缓存
    verified: HashMap<TemplateKey, WfcTemplate>,
    // 缓存上限——LRU 淘汰
    max_entries: usize,
}

struct TemplateKey {
    building_function: BuildingFunction,
    culture_cluster: CultureClusterId,  // 相似文化的聚类
    climate_zone: ClimateZone,
}
```

由此：**手写 3-5 套种子模板启动，生成器慢慢填充模板库，达到覆盖率大部分。**长期来看，模板的大部分来自生成器而非手写。

## 四、CultureCoreParams→建筑风格映射

### 4.1 风格维度

从聚落CultureCoreParams派生的建筑视觉参数：

```rust
struct BuildingStyle {
    /// 屋顶样式
    roof: RoofStyle,
    /// 墙体材质
    wall_material: WallMaterial,
    /// 装饰风格
    decoration: DecorationLevel,
    /// 颜色调色板
    palette: ColorPalette,
}

enum RoofStyle {
    Flat,        // 平顶——干旱/沙漠文化
    Sloped,      // 坡顶——温带/多雨文化
    Dome,        // 穹顶——宗教/东方文化
    Spire,       // 尖顶——哥特/北方文化
    Curved,      // 曲线——东方精致
    Longhouse,   // 长屋——北欧
    Thatched,    // 茅草——贫困/草原
}

enum WallMaterial {
    LocalStone,   // 当地石材（颜色 = f(当地地质)）
    Timber,       // 木材——森林区
    Brick,        // 砖瓦——发达聚落
    Adobe,        // 土坯——沙漠/贫困
    WattleDaub,   // 编条抹泥——中世纪平民
    ImportedStone,// 进口石材——极富裕
    Ice,          // 冰雪——极地
    Hide,         // 兽皮——游牧
}
```

### 4.2 材质决定因素

1. **当地可获取的建材**（最主要）：聚落周边有什么石头/木材 → 建筑色调
2. **文化传统**：同材料不同加工（粗面 vs 磨光，原色 vs 涂色）
3. **经济实力**：富裕聚落可进口远方高级建材
4. **宗教因素**：白色大理石神殿、深色玄武岩葬礼建筑

## 五、房间生成（保留）

每个房间经过四阶段：

1. **功能与文化确认**——核心用具类型确定
2. **核心用具摆放**——定义房间用途的物品（占面积 10-30%）
3. **辅助用具填充**——用具密度 = 每 4m² 一个槽位，按优先级填充
4. **装修风格应用**——墙面/地面/天花板/照明

详见原 `04-房间生成规则.md`（逻辑框架保留，GDScript→Rust）。

## 六、WFC 回溯策略（保留原方案）

当 WFC 遇到矛盾时：

1. **轻度矛盾**（<10% 格点）：局部重试——放宽受影响区域约束
2. **中度矛盾**（10-30%）：调整权重——降低冲突类型权重，重新全局坍缩
3. **重度矛盾**（>30% 或重试3次失败）：回退到传统逐级生成管线（安全网）

## 七、建筑质量衰减模型（保留）

```
崭新 → 正常(1-5年) → 老旧(10-30年) → 破损(风暴/战争) → 危房(缺维护) → 废墟 → 清除/重建
```

有主人的建筑定期维护；废弃建筑随时间恶化。维护行为与 NPC 行为系统挂钩。
