## 一、NPC 系统性能 🔴 最大挑战

### 1.1 核心矛盾

> **100,000 NPC × 复杂心理模拟 × 60fps 帧预算 = 根本不可行（如果全模拟）。远期目标需依赖分层模拟+硬件迭代+算法革新逐步逼近。**

三层 LOD 模拟是唯一可行的路径，但当前设计仍有若干未解决的性能陷阱。

### 1.2 L1 NPC 帧预算细算

当前 `技术架构.md` §7.2 给出的预算是 ≤15ms/帧（含渲染）。重新验算：

| 子系统 | 当前预算 | 实际风险 | 分析 |
|--------|---------|---------|------|
| **感官射线** | ≤4ms（2条×2ms） | 🟡 风险中等 | 50 个 L1 × 2条视觉射线 ÷ 分摊到 2s 更新 = 50条/帧 × 0.04ms ≈ 2ms。合理。但**听觉是事件驱动的不应轮询**，当前设计已确认这一点。 |
| **情绪引擎** | ≤1ms | 🟢 低风险 | 纯数学运算。8种基本情绪+30种复合情绪的更新是线性代数，极快。 |
| **GOAP 规划器** | ≤1ms | 🔴 高风险 | **最危险的单点。** A* 搜索在最坏情况下（深度8×每步6可选动作=1.6M种可能）可在单次规划中消耗数百ms。必须严格限制搜索深度和分支因子。 |
| **概率决策** | ≤1ms | 🟢 低风险 | 加权随机采样——O(n)，n=可选行为数（通常<20）。 |
| **记忆检索** | ≤1ms | 🟡 风险中等 | 200条记忆排序/搜索。但可以维护热缓存（最近50条），O(50)是安全的。 |
| **记忆编码** | ≤1ms | 🟢 低风险 | 仅在事件发生时触发，非轮询。 |
| **NPC 移动+动画** | ≤8ms | 🔴 高风险 | **50个 L1 × 动画更新 + 寻路查询。** 骨骼动画的 CPU 开销被严重低估。 |

### 1.3 GOAP 规划器的性能陷阱

**问题**：GOAP（Goal-Oriented Action Planning）使用 A* 搜索从当前世界状态到目标状态的最优行动序列。搜索空间 = 可选动作数 ^ 搜索深度。

```
当前设计指标：
  - 搜索深度 ≤8
  - 每次规划目标变化时触发（30-120s/NPC）
  - 50 个 L1 NPC ÷ 120s 间隔 = 每 2.4 秒一次规划

但在紧急情况下：
  - 50 个 NPC 同时触发 GOAP（例如：同时被饥饿/口渴/战斗触发）
  - 50 × 最坏情况 100ms = 5000ms → 帧完全卡死
```

**应对策略**：

1. **严格裁剪动作空间**：GOAP 不是“所有可能动作”的搜索——只搜索与当前目标相关的动作子集。饥饿→只搜索获取食物的动作（≤4个），不是全部可选动作。
2. **规划分摊**：即使 50 个 NPC 同时触发，也分摊到多帧执行。使用规划请求队列，每帧最多处理 2-3 个规划请求。
3. **缓存规划结果**：相同（世界状态, 目标）的规划结果缓存一段时间。NPC 处于相似状态时直接复用。
4. **设置硬超时**：单次 GOAP 规划超过 2ms 直接中断，返回当前最佳部分规划或 fallback 行为。
5. **预计算常用规划**：对于高频率的生存行为（饿了→找食物→吃），预计算行动链，不每次重新规划。

### 1.4 记忆系统的扩展性问题

**当前设计**：每个 NPC 200 条事件记忆。

| 维度 | 计算 | 风险 |
|------|------|------|
| 存储 | 200条 × 10,000 NPC = 2,000,000 条记忆 | 🟡 每条约 200 字节 → ~400MB。需要压缩。 |
| 检索 | 加权排序（情绪一致性/时间/冲击力） | 🟡 每次检索 O(200 log 200)，可接受但需缓存 |
| 社交传播 | NPC 之间闲聊传播记忆 | 🔴 **最被低估的开销**——见下文 |

**社交传播（B5b）的隐藏成本**：

当前设计：NPC 之间闲聊时可以传播记忆，每次传播冲击力衰减 30%。

这听起来很便宜——但如果在城市广场中有 30 个 NPC 在 L1 范围内，每 2 分钟每个 NPC 闲聊一次，那就是 30 × 29 / 2 = 435 个潜在传播对。

**优化**：
- 社交传播必须限速：每个 NPC 每游戏小时最多传播 1-2 条记忆
- 使用"传播事件"而非"轮询检查"——当 NPC A 与 NPC B 发起社交互动时，触发一次传播，而非持续检查
- L2 NPC 不参与主动传播（仅被动接收）

### 1.5 关系系统的 O(n²) 陷阱

当前设计：每个 NPC 的 `relationships` 字典存储 `{npc_id: {affection, trust, familiarity, status}}`。

**问题**：如果采用完全连接图，10,000 个 NPC → 约 5,000 万条有向关系边。

**当前设计已部分避免了这个问题**——NPC 只存储"实际互动过的"NPC 的关系，而非全部。但需要明确：

- **关系初始化时**：世界生成阶段，NPC 只与家庭成员、同团体成员、近邻建立初始关系
- **关系惰性创建**：当 NPC A 首次与 NPC B 互动时，才创建关系条目
- **平均关系数预估**：家庭(5-15) + 同事(3-10) + 朋友(5-20) + 熟人(10-30) ≈ **30-75 条/NPC**
- **总关系数**：10,000 × 50 ≈ 500,000 条（远少于 5,000 万的完全图）

### 1.6 动画更新开销

**骨架动画 CPU 成本**（Godot Skeleton3D）：

| NPC 数量 | 骨骼数 | 动画更新/帧 | 预计 CPU |
|----------|--------|------------|---------|
| 1 个 | 30 骨骼 | 1次 IK + BlendSpace | 0.05-0.1ms |
| 50 个 L1 | 30 骨骼 | 50次 | **2.5-5ms** |
| 500 个 L2 | 简化（15骨骼或更少） | 简化动画 | **低估风险** |

**关键问题**：`技术架构.md` §7.2 将"NPC移动+动画"预算为 ≤8ms，但**如果所有 50 个 L1 NPC 都在播放完整骨骼动画，仅动画就可能吃掉 5ms**，留给移动寻路和碰撞的预算只剩 3ms——非常紧张。

**应对策略（Low-Poly 风格的优势）**：
- L1 NPC：完整骨骼动画，但骨骼数精简到 15-20 根（低多边形不需要精细手指动画）
- L2 NPC：**顶点动画（Morph Target/Shape Key）** 或无动画（仅位置移动+旋转）
- L3 NPC：无 3D 节点
- 利用 Godot 的 `AnimationTree` 的 `NodeBlendTree` 模式而非每帧全量更新

## 二、大量实体同屏渲染 🔴

### 2.1 城市场景的压力测试

假设玩家站在一个**城市中心广场**（最密集场景）：

| 可见实体 | 数量（估算） | 面数/个（低多边形） | 总面数 |
|----------|------------|-------------------|--------|
| NPC（L1，完整模型） | 30-50 | 500-1000 | 15,000-50,000 |
| NPC（L2，简化模型） | 100-200 | 200-400 | 20,000-80,000 |
| 建筑（可见的） | 20-50 栋 | 300-800 | 6,000-40,000 |
| 家具/装饰（视野内） | 100-500 件 | 50-200 | 5,000-100,000 |
| 植被（树木/灌木） | 50-200 | 100-300 | 5,000-60,000 |
| 地形（体素 Mesh） | 整个场景 | — | 20,000-80,000 |
| **总计** | | | **~70,000-410,000 面** |

**评估**：低多边形的优势在此体现——即使最密集场景，面数也远低于写实风格（写实游戏单个角色就可能 50,000+ 面）。30-40万面在 2024 年后的 GPU 上可以在 2-4ms 内完成几何处理。

**真正的瓶颈不是面数，而是 Draw Call**。

### 2.2 Draw Call 爆炸

每个独特的 **材质+网格** 组合 = 一个 Draw Call。

| 实体类型 | 材质变体数 | 独特网格数 | 最坏 Draw Call |
|----------|-----------|-----------|---------------|
| NPC（肤色×服装×文化） | 50+ | 3-5 | **50+** |
| 建筑（文化×功能×等级） | 30+ | 10-20 | 30+ |
| 家具（类型×文化×质量） | 100+ | 30-50 | 100+ |
| 植被 | 10-20 | 5-10 | 20 |

**最坏情况下**，仅建筑就有 30+ 个 Draw Call，全部加起来可能 150-200+。在集成显卡或老旧 GPU 上，200 个 Draw Call 可能吃掉 5-10ms 的 CPU 提交时间。

### 2.3 Draw Call 优化策略

#### GPU Instancing（MultiMeshInstance3D）

Godot 的 `MultiMeshInstance3D` 可以将**同一网格+材质**的实例批量渲染为单个 Draw Call：

| 适用对象 | Instancing 的收益 |
|----------|------------------|
| 同种树木/灌木 | 🔴 极高——数千棵同种树 → 1个 Draw Call |
| 同文化同类型的建筑模块 | 🟡 高——同一面墙的数百个实例 |
| NPC（相同模型+服装） | 🟡 高——但 NPC 服装差异大，Instancing 效果打折 |
| 家具（同类型） | 🟡 高——同一款椅子的所有实例 |
| 草地/小花 | 🔴 极高——万级实例 → 1个 Draw Call |

#### 纹理图集（Texture Atlas）

将多个物体的纹理合并到一张大纹理上，减少材质切换：

- **建筑纹理图集**：一面墙的所有文化变体纹理在一张图上
- **NPC 服装图集**：所有基础服装纹理在一张图上
- **道具图集**：所有小型家具/工具的纹理在一张图上

**Low-poly + 像素纹理的优势**：纹理本身分辨率低（如 128×128 或 256×256），一张 2048×2048 的图集可以容纳 64-256 个物体纹理。

#### 材质合并

减少独特材质数量：
- 所有同文化建筑共享一个材质（使用图集+UV偏移区分不同部分）
- "材质实例参数"而非独立材质——颜色/粗糙度通过参数传入，不创建新材质

### 2.4 LOD 策略细化

#### 建筑 LOD

| LOD 级别 | 距离 | 面数 | 实现 |
|----------|------|------|------|
| LOD0 | 0-50m | 完整（300-800面） | 完整模块化建筑 |
| LOD1 | 50-150m | 简化（100-200面） | 移除小装饰、简化屋顶 |
| LOD2 | 150-300m | 轮廓（50-100面） | 仅主体外形，无窗户细节 |
| LOD3 | >300m | 极简（<50面）或 Imposter | 2D 广告牌或纯色块 |

#### NPC LOD

NPC 的 LOD 与三层模拟自然结合：

| 模拟层级 | 3D 表现 | 面数 | 动画 |
|----------|---------|------|------|
| L1 | 完整模型 | 500-1000 | 完整骨骼动画 |
| L2 | 简化模型 | 150-300 | 简化动画或顶点动画 |
| L3 | 无 3D 节点 | 0 | 无 |

#### 植被 LOD

| LOD | 距离 | 实现 |
|-----|------|------|
| LOD0 | 0-30m | 完整 3D 树/灌木（MultiMesh） |
| LOD1 | 30-80m | 简化 3D（减少枝条面数）+ 十字面片混合 |
| LOD2 | 80-200m | 十字面片（Billboard） |
| LOD3 | >200m | 不渲染或合并到地形颜色 |

### 2.5 遮挡剔除

Godot 4.x 支持 **Occlusion Culling**（使用 `OccluderInstance3D`）：

- 大型建筑内部放置 Occluder（遮挡建筑后面的物体）
- 城墙/山体后方的物体被自动剔除
- 在城镇中，被前排建筑遮挡的后排建筑不渲染

**需要手动放置**关键遮挡体，Godot 不会自动生成。

## 三、实时加载与流式传输 🔴

### 3.1 区块加载的性能模型

```
玩家移动速度：步行 ~5m/s，奔跑 ~8m/s，骑马 ~15m/s

以 8m/s 奔跑为例：
  每秒跨越 8m ÷ 32m/chunk = 0.25 chunk/s
  加载半径 = 3 超区块 ≈ 96 chunks（前瞻）
  
  新进入加载范围的 chunks/秒 ≈ 0.25 chunks/s（单方向移动）
  如果玩家快速转向或传送 → 突发大量 chunk 加载
```

**chunk 加载管线**（当前设计）：

```
体素密度场生成（噪声计算）
  → SVO 构建
  → Transvoxel 等值面提取（生成 Mesh）
  → 碰撞体生成
  → 纹理/材质应用
  → Navigation 导航网格烘焙
  → MultiMesh 植被放置
  → 聚落/建筑生成（如果该 chunk 包含聚落）
```

**每个 chunk 的生成时间预估**：

| chunk 类型 | 预计耗时 | 说明 |
|-----------|---------|------|
| 纯自然地形（无聚落） | 50-200ms | 噪声+等值面提取+碰撞+植被 |
| 含村庄的 chunk | 200-500ms | 加建筑生成+NPC 初始化 |
| 含小镇的 chunk | 500-1500ms | 更多建筑+更多 NPC |
| 含城市部分的 chunk | 1000-3000ms | 大量建筑+内饰+大量 NPC |

**问题**：如果玩家骑马奔向一个城市，每秒跨越 0.5 chunk，而城市 chunk 需要 1-3 秒生成——**加载速度赶不上移动速度**。

### 3.2 异步加载策略

```
加载优先级（从高到低）：
  1. 体素地形 Mesh（玩家能看到的地面）——必须最先
  2. 地形碰撞（玩家不能掉下去）——紧随其后
  3. 大型建筑外壳（远处的地标）——高
  4. NPC 实例化（让世界活起来）——中
  5. 建筑内饰/家具 ——低（玩家未进入建筑时不需要）
  6. 小装饰/植被细节 ——最低
  7. Navigation 导航网格 ——延迟烘焙
```

**分帧加载时间表**：

```
Frame 1-3 (0-50ms):
  → 噪声计算 + 体素密度场（纯 CPU，后台线程）
  
Frame 4-6 (50-100ms):
  → Transvoxel 等值面提取（后台线程）
  → 地形 Mesh 上传到 GPU（主线程，<1ms）
  → 地形碰撞体生成（后台线程）
  
Frame 7-10 (100-167ms):
  → 碰撞体就位（玩家可以站在地形上）
  → 大型建筑外壳渲染（如果该 chunk 有建筑）
  
Frame 11-20 (167-333ms):
  → 建筑内饰填充
  → 植被 MultiMesh 放置
  
Frame 21+ (333ms+):
  → NPC 实例化
  → Navigation 烘焙（延迟到玩家静止时）
```

### 3.3 预加载策略

**预测性加载**：根据玩家移动方向和速度，提前加载前进方向上的 chunk。

```
预加载窗口：
  玩家前方 120° 扇形 × 2 个超区块深度 = 提前加载
  玩家后方 60° 扇形 × 1 个超区块 = 正常加载
  玩家侧方 = 惰性加载（仅地形，不填充 NPC）
```

**加载线程管理**：
- 使用 Godot 的 `WorkerThreadPool` 或自定义线程池
- 体素生成（噪声计算、等值面提取）完全在后台线程
- Mesh 上传、节点实例化必须在主线程（Godot 限制）
- 每帧为加载操作预留 ≤3ms 主线程时间

### 3.4 存档/读档性能

`技术架构.md` 目标：存档序列化 <500ms，存档大小 <3GB。

**体素存档**：使用自定义二进制格式，仅保存**被修改过的 chunk**（脏标记），未修改的 chunk 从种子重新生成。

```
未修改 chunk：不存档（0 字节）→ 读档时从种子重新生成
已修改 chunk：SVO 压缩 → 二进制写入（~50-200KB/chunk）
NPC 数据：JSON Lines 格式，增量写入
```

**最坏情况估算**：
- 如果玩家改造了 10,000 个 chunk（大面积地形改造）
- 10,000 × 200KB = 2GB（体素数据）
- + NPC 数据 ~200MB + 其他 ~100MB
- 总计：~2.3GB——仍在 3GB 目标内

## 四、物理与碰撞性能 🟡

### 4.1 碰撞体数量

| 碰撞体来源 | 数量估算（城市场景） | 类型 |
|-----------|-------------------|------|
| 地形碰撞 | 1 个大型 ConcavePolygonShape | Static |
| 建筑碰撞 | 20-50 个 BoxShape/ConvexPolygonShape | Static |
| NPC 碰撞（L1） | 30-50 个 CapsuleShape | CharacterBody |
| 家具碰撞 | 100-500 个 BoxShape | Static |
| 物品掉落 | 0-20 个 | RigidBody |

**优化**：
- 建筑使用简化的碰撞体（AABB 盒子），而非 Mesh 精确碰撞
- 家具**默认无碰撞**——NPC 通过寻路避开家具所在区域，而非物理碰撞
- 地形碰撞使用体素插件提供的优化碰撞 Mesh（Voxel Tools 内置）

### 4.2 物理引擎选择

已确认使用 **Godot Jolt Physics**（`技术架构.md` §六）。Jolt 在以下方面优于默认 Godot Physics：
- 大型静态碰撞体的处理效率更高
- CharacterBody 的移动/碰撞响应更稳定
- 对大量碰撞体的扩展性更好

### 4.3 寻路性能

**Navigation 网格烘焙**：
- 每个 chunk 的导航网格烘焙需要 **50-200ms**（`06-体素设计决策.md` §3.3）
- 地形修改后**不应立即重新烘焙**——延迟 1-2 秒，批量处理相邻 chunk
- 烘焙在后台线程执行

**多人同时寻路**：
- 50 个 L1 NPC 同时在城市中寻路 → 50 次 A* 查询
- A* 在导航网格上的复杂度：O(n log n)，n = 导航多边形数量
- 城市 chunk 的导航网格可能有 500-2000 个多边形

**优化**：
- 路径缓存：同一 OD（起点-终点）对的路径复用
- 路径平滑：A* 结果通常是锯齿状的，平滑步骤可以分摊
- 分层寻路：先找"从哪个区域到哪个区域"（粗粒度），再在区域内细粒度寻路

## 五、内存占用分析 🟡

### 5.1 内存预算分配

| 类别 | 预算 | 当前估算 | 风险 |
|------|------|---------|------|
| NPC 数据 | <200MB | ~120-180MB | 🟢 |
| 体素数据（活跃 chunk） | <500MB | ~200-400MB | 🟢 |
| 网格/纹理资产 | <1GB | ~500-800MB（低多边形优势） | 🟢 |
| 音频 | <100MB | ~50MB | 🟢 |
| Godot 引擎开销 | <200MB | ~100-150MB | 🟢 |
| **总计** | **<2GB** | **~1-1.5GB** | 🟢 远低于 3GB 目标 |

**Low-Poly 风格的关键优势**：
- 纹理分辨率低（128-512px vs 写实的 2048-4096px）
- 每张纹理：16-256KB（低多边形） vs 4-16MB（写实 4K PBR 多贴图）
- 网格面数低，顶点数据量小
- AI 生成的纹理可以严格控制在目标分辨率内

### 5.2 NPC 内存细化

当前 `技术架构.md` 估算每个 NPC 的 Dictionary 约 12KB。重新验算：

| 字段 | 大小估算 |
|------|---------|
| 基础身份（ID/姓名/种族/性别/年龄） | ~300B |
| 大五人格（5 × float） | ~120B |
| 生理状态（4 × float） | ~100B |
| 情绪（3轴 + 8基本 + 1复合） | ~300B |
| 记忆（200条 × ~150B/条，压缩后） | ~30KB |
| 关系（50条 × ~120B/条） | ~6KB |
| 技能（10-20项 × ~80B/项） | ~1.6KB |
| 社会/团体/文化 | ~2KB |
| **总计/NPC** | **~40KB**（含记忆） |

10,000 × 40KB = **400MB**——超出 200MB 预算 😟

**压缩策略**：
- 记忆使用紧凑二进制格式（而非 Dictionary）：200条 → ~10KB（而非 30KB）
- 关系数据共享引用（同一团体内的 NPC 共享关系模板）
- L3 NPC 不保留详细记忆（仅统计摘要），L3 占总 NPC 的 ~95%
  - L1+L2 (550) × 40KB = 22MB
  - L3 (9,450) × 5KB = 47MB
  - **总计约 70MB**——非常安全

### 5.3 关系矩阵的内存

关系数据是 NPC 内存中最大的变量：
- 如果采用双向存储（A 存储了对 B 的关系，B 也存储了对 A 的关系），浪费约 30%
- **建议**：关系单向存储（NPC 只存储自己对他人的态度），读取时如果对方没有对应条目则视为"无关系/陌生人"
- 从不使用的"关系模板"继承——同一家庭的成员共享基础关系模板

## 六、世界生成性能 🟢

### 6.1 初始世界生成时间

世界生成是一次性成本（发生在"创建新游戏"时），对帧率无影响但仍需控制在合理范围内。

| 阶段 | 预计耗时 | 说明 |
|------|---------|------|
| 噪声+地形 | 5-15s | 4km×4km 区域的多层噪声叠加 |
| 水文模拟 | 2-5s | 河流侵蚀+湖泊填注 |
| 生物群系+资源 | 3-8s | 气候矩阵+矿田生成 |
| 聚落选址 | 1-3s | 资源评分+选址 |
| 建筑生成 | 10-60s | **主要耗时——城市建筑内饰最多** |
| NPC 初始化 | 5-20s | 人口分配+属性生成+关系初始化 |
| 道路生成 | 1-5s | MST 算法 |
| 历史模拟 | 10-120s | **最大变量——取决于模拟年份数** |
| **总计** | **~40s-4min** | 可接受（一次性，有进度条） |

### 6.2 历史模拟优化

阶段九（历史模拟）快进 N 年——如果模拟 100 年 × 10,000 NPC，即使简化模型也可能极慢。

**建议**：
- 使用统计模型而非个体模拟：不追踪每个 NPC 在每一天的行为，而是按年度推演人口/经济/势力变化
- 预设 N 个"标准历史模板"（和平发展/战乱/瘟疫/繁荣），由种子选择组合
- 历史模拟的结果 = 聚落的兴衰痕迹 + 初始势力格局 + NPC 的初始关系网

## 七、音频性能 🟢

音频通常是性能分析中被忽视的部分，但在 10,000+ NPC 的世界中：

| 问题 | 风险 |
|------|------|
| 大量 3D 音频源同时播放 | 🟡 50 个 L1 NPC 各有一个脚步声 → 50 个 AudioStreamPlayer3D |
| 环境音混合 | 🟢 每个群系 1-3 个环境音源，总量可控 |

**优化**：
- 音频距离衰减：超出听觉范围（如 30m）自动静音
- 同质化声音合并：10 个铁匠同时打铁 ≠ 10 个独立的打铁音效，而是 1 个"铁匠铺环境音"（提高音量/密度）
- Godot 的 AudioServer 会自动管理活跃音频源数量，但明确设置 `max_polyphony` 更安全

## 八、Low-Poly 美术风格的具体性能指导

### 8.1 面数预算

| 资产类型 | 目标面数 | 说明 |
|----------|---------|------|
| 玩家角色 | 800-1200 | 最重要的模型，可稍高 |
| NPC（人类/精灵/矮人） | 500-800 | 批量出现，必须精简 |
| 大型生物（龙/巨魔） | 1000-2000 | 同屏数量少 |
| 建筑模块（墙） | 50-150 | 大量实例化 |
| 建筑模块（屋顶） | 100-200 | 同上 |
| 树木（近景） | 100-300 | MultiMesh 实例化 |
| 家具（大型：床/桌/柜） | 80-150 | 城市级别可能数百件 |
| 家具（小型：杯/碗/工具） | 20-60 | 同上 |
| 武器 | 100-200 | 手持/展示 |
| 植被（灌木/花草） | 10-50 | 海量实例，十字片优先 |

### 8.2 纹理规范

| 纹理类型 | 分辨率 | 格式 | 说明 |
|----------|--------|------|------|
| 角色纹理 | 256×256 或 512×512 | PNG/WebP | 像素风格，少量颜色 |
| 建筑纹理图集 | 1024×1024 或 2048×2048 | WebP | 一张图集涵盖一种文化的所有建筑纹理 |
| 道具/家具图集 | 1024×1024 | WebP | 涵盖所有常见家具 |
| 地形纹理 | 512×512 或 1024×1024 | WebP | 按群系分图集 |
| UI 图标 | 64×64 或 128×128 | PNG | 像素图标 |

**像素纹理的 VRAM 优势**：
- 一张 2048×2048 RGBA8 = 16MB VRAM
- 写实游戏：8-10 套 PBR（BaseColor+Normal+ARM）× 4K = 10 × 3 × 16MB = 480MB
- Low-Poly：10 张 2048 图集 × 16MB = 160MB——**节省 66% VRAM**

### 8.3 调色板统一

严格要求所有纹理使用**统一的调色板**（如 64 色或 128 色全局调色板）：
- 视觉风格统一
- 调色板索引纹理 → 进一步降低 VRAM 使用
- AI 生成资产的后处理可以映射到统一调色板

## 九、Rust 核心 + Godot 客户端的性能边界

> **v3.0 更新**：WoWorld 的模拟核心（NPC/世界生成/战斗AI/经济/魔法/GOAP）全部运行在 Rust 侧，通过 GDExtension 与 Godot 集成。Godot 仅负责渲染/UI/音频/输入/玩家物理。

### 9.1 Rust 模拟核心的性能边界

模拟计算全部在 Rust 侧执行，不受 GDScript 性能限制。Godot 4.6 LTS 的渲染线程与 Rust 的模拟线程并行——模拟不占用 Godot 的渲染帧预算。

| 适合 Godot (GDScript/C#) | 适合 Rust 核心 (GDExtension) |
|--------------------------|------------------------------|
| UI 逻辑 / HUD | 噪声函数计算（世界生成） |
| 音频播放管理 | Transvoxel 等值面提取 |
| 渲染/动画 | NPC GOAP 规划器 |
| 输入处理 | NPC 情绪/记忆/关系——全部模拟 |
| 场景树管理 | 战斗 AI（三层模型） |
| 存档界面 | 经济/魔法/天气/时间系统 |
| — | 体素数据压缩/解压缩 |

**关键优势**：GOAP 规划器、NPC 全模拟、战斗 AI 全部在 Rust 中运行——不占用 Godot 的渲染帧预算。Rust 侧使用 rayon 并行 + SoA 数据布局，充分利用多核 CPU。GOAP 最坏情况搜索在 Rust 中预计 <1ms（vs GDScript 的 5-20ms），无需 GDScript→C++ 迁移的渐进式策略。

### 9.2 自建 Transvoxel（无插件依赖）

### 9.3 Forward+ vs Mobile 渲染器

Godot 4.x 默认使用 Forward+ 渲染器：

| 特性 | 对 WoWorld 的影响 |
|------|------------------|
| 大量动态光照 | Forward+ 擅长，但 WoWorld 应限制动态光源数量（城市夜景每栋建筑不超 1-2 个动态光） |
| 阴影 | 每个动态光 = 一个阴影贴图 = VRAM。**限制阴影投射光源数量**（如最多 8 个） |
| 反射 | Low-Poly 风格不需要高质量反射探头 |

## 十一、亟待解决的性能问题总结（按优先级）

| # | 问题 | 严重度 | 建议行动 |
|---|------|--------|---------|
| 1 | **GOAP 规划器最坏情况爆炸** | 🔴 极危 | 设置硬超时(2ms)、裁剪动作空间、预计算常用规划、分摊到多帧 |
| 2 | **城市 chunk 加载赶不上移动速度** | 🔴 极危 | 设计分帧加载优先级、预测性预加载、异步后台生成 |
| 3 | **L1 NPC 骨骼动画 CPU 预算紧张** | 🔴 高 | 精简骨骼(15-20根)、L2用顶点动画、实测后调整预算 |
| 4 | **社交传播的 O(n²) 交互对** | 🔴 高 | 限速传播(每NPC每游戏小时1-2条)、事件驱动非轮询 |
| 5 | **建筑/家具的 Draw Call 爆炸** | 🟡 中高 | 纹理图集、GPU Instancing、材质合并 |
| 6 | **NPC 记忆存储超预算（200条×10000=2M条）** | 🟡 中高 | L3不存详细记忆、紧凑二进制格式、低冲击记忆压缩为摘要 |
| 7 | **关系矩阵存储（O(n²)风险）** | 🟡 中 | 惰性创建关系、单向存储、关系模板继承 |
| 8 | **Navigation 导航烘焙耗时** | 🟡 中 | 延迟烘焙、后台线程、分层寻路 |
| 9 | **GPU 动态光源过多** | 🟡 中 | 限制阴影投射光数量（≤8）、城市夜景用烘焙光照+少数动态光 |
| 10 | **GOAP 规划器最坏情况爆炸** | 🟡 中 | Rust A* 预计算常用规划+硬超时+分摊到多帧——已在 v3.0 Rust 架构中缓解 |
| 11 | **存档读档时间** | 🟢 低 | 增量存档、仅保存脏 chunk、后台加载 |
| 12 | **音频源过多** | 🟢 低 | 同质化声音合并、限制 max_polyphony |
