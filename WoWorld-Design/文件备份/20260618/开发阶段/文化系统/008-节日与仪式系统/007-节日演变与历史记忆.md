## 一、节日的产生——四种诞生路径

| 路径 | 触发 | 数据存储 | 示例 |
|------|------|---------|------|
| **种子生成** | P2.5 文化生成（世界生成时） | `FestivalFramework` 存入文化数据 | 日历节日、宗教节日 |
| **历史事件结晶** | 重大事件 → 年度纪念行为持续 3 年 → 系统检测并正式化 | 创建新 `FestivalFramework`（运行时） | 胜利纪念日、建国日 |
| **涌现检测** | NPC 自发聚集持续 3 年 → 系统检测模式 → 正式化 | 创建新 `FestivalFramework`（运行时） | 鲑鱼节、流星雨夜 |

## 二、节日的衰落

### 2.1 自然衰落

```
条件（满足任一即进入衰落检测）：
1. 连续 5 年参与率 < 20%（居民不再关心）
2. 连续 3 年参与率 < 10%

→ 节日标记为 Declining
→ 接下来的 5 年观察期：如果参与率持续低迷 → Declined → Defunct
→ 如果有某年反弹（如特殊事件后重新激发）→ 恢复为 Active
```

**自然衰落的原因**：
- 文化参数偏移——原本 `religiosity=0.8` 的文化逐渐世俗化到 0.3 → 宗教节日失去意义
- 人口下降——聚落人口减半 → 节日规模不如此前
- 经济衰退——连年饥荒 → "没心情过节"
- 文化重心转移——航海文化变为农耕 → 航海神祇的节日逐渐被忽视

### 2.2 强制废止

```
征服者政策 → FestivalPolicy::Ban → 节日被正式废除
↳ 如果被征服文化仍然在私人场合庆祝 → 地下节日（见 005-权力与空间 §4.2）
↳ 如果征服持续 > 100 年 → 地下节日也逐渐消失
↳ 如果被征服文化重新独立 → 节日可能被恢复（增加了"解放"的额外意义层）
```

## 三、节日的融合——文化接触的产物

### 3.1 融合类型

| 融合模式 | 条件 | 结果 |
|---------|------|------|
| **日期借用** | 两个文化的节日日期接近（±10天）且 openness ≥ 0.4 | 合并为单一日期——通常是优势文化的日期 |
| **内容混合** | 长期贸易/通婚 ≥ 50 年 | A 文化的节日中出现了 B 文化的活动（如 A 的秋收祭中加入了 B 的舞蹈） |
| **名称融合** | 混合区聚落持续 ≥ 100 年 | 节日名称变为混合——A 文化词根 + B 文化后缀 |
| **全新节日结晶** | 混合文化区 + 300 年 | 两个文化的节日完全融合为新的第三个节日（克里奥尔化——与地名系统的克里奥尔化机制一致） |

### 3.2 融合算法

```rust
fn detect_festival_fusion(
    culture_a: CultureId,
    culture_b: CultureId,
    contact_years: u32,
    festivals_a: &[FestivalFramework],
    festivals_b: &[FestivalFramework],
) -> Vec<FusionCandidate> {
    let mut candidates = Vec::new();
    
    for fa in festivals_a {
        for fb in festivals_b {
            // 日期接近
            let date_dist = (fa.day_of_year as i32 - fb.day_of_year as i32).abs();
            if date_dist > 15 { continue; } // 太远的日期——不太可能融合
            
            // 类型相似
            let type_similarity = festival_type_similarity(&fa.festival_type, &fb.festival_type);
            if type_similarity < 0.3 { continue; }
            
            // 融合概率 = 接触年数 × openness(A) × openness(B) × 类型相似度 × 日期接近度
            let prob = (contact_years as f32 / 100.0).clamp(0.0, 1.0)
                * openness_a * openness_b
                * type_similarity
                * (1.0 - date_dist as f32 / 15.0);
            
            if prob > 0.5 {
                candidates.push(FusionCandidate {
                    source_a: fa.id,
                    source_b: fb.id,
                    fusion_probability: prob,
                    suggested_date: (fa.day_of_year + fb.day_of_year) / 2,
                });
            }
        }
    }
    
    candidates
}
```

## 四、节日内容的代际漂移

即使没有外部干涉，节日内容也会随代际自然演变：

```
每年节日举行时：
  1. 核心仪式（名称、日期、主持者）保持稳定——作为文化的"锚"
  2. 外围元素（具体食物、流行舞蹈、装饰风格）缓慢变化
  
漂移速率：
  - 不确定性规避高 → 几乎不变（"自古以来就这样，不能改"）
  - 不确定性规避低 → 每 10 年可见明显变化
  - 开放度高 → 外来元素更快融入
  - 艺术性高 → 创新被鼓励——新舞蹈、新音乐

代际漂移跟踪：
  decade_active: // "每 10 年"的快照
    第 10 年：Food(FestivalBread, RoastedMeat), Dance(CircleDance), Music(Drum)
    第 20 年：Food(FestivalBread, RoastedMeat, ImportedSpice), Dance(CircleDance, PartnerDance), Music(Drum, Lute)
    第 50 年：Food(FestivalBread, RoastedMeat, ForeignDelicacy), Dance(PartnerDance, LineDance), Music(Lute, Harp)
    // 核心不变（FestivalBread），周边演化（新舞蹈取代旧舞蹈）
```

## 五、节日对文化的反向影响

节日不只是文化的输出——它反过来塑造文化：

```rust
/// 节日每年完成后对文化参数的反馈
fn festival_cultural_feedback(
    festival: &FestivalFramework,
    instance: &FestivalInstance,
    culture: &mut CultureCoreParams,
) {
    // 反馈强度取决于节日规模 × 参与率
    let feedback_strength = instance.scale * instance.participant_ratio() * 0.002;
    // 最大每年 0.002 —— 比代际漂移（σ=0.003）稍弱但持续累积
    
    match festival.emotional_tone {
        EmotionalTone::Ecstatic => {
            culture.indulgence += feedback_strength * 0.5; // 狂欢→放纵度上升
        }
        EmotionalTone::Solemn => {
            culture.religiosity += feedback_strength * 0.3; // 庄重仪式→宗教性上升
            culture.uncertainty_avoidance += feedback_strength * 0.2;
        }
        EmotionalTone::Competitive => {
            culture.competition_orientation += feedback_strength * 0.4;
            culture.militarism += feedback_strength * 0.2; // 竞技→军事温和上升
        }
        _ => {}
    }
    
    match festival.festival_type {
        FestivalType::Religious { .. } => {
            culture.religiosity += feedback_strength * 0.15;
        }
        FestivalType::Calendar { anchor: CalendarAnchor::AutumnEquinox, .. } => {
            culture.long_term_orientation += feedback_strength * 0.2; // 收获节→长远规划
        }
        _ => {}
    }
    
    // 钳制——节日反馈不能超过 ±0.05/年（节日不是文化的主要驱动力）
    culture.clamp_all();
}
```

**长期累积**：一个文化连续 100 年庆祝 Ecstatic 型狂欢节 → `indulgence` 累积偏移约 0.1——可观但非革命性。节日是文化的**维护机制**——保持已有倾向，而非改造文化。

## 六、节日作为历史事件

### 6.1 什么被记录为 HistoricalEvent

不是每次节日都写入历史——那样太多了。记录规则：

| 条件 | 记录内容 |
|------|---------|
| **首届节日**（节日被创建时） | 完整的 HistoricalEvent——"第一届收获节在 X 年举行" |
| **年度规模 ≥ 0.8**（特大型） | 轻量 HistoricalEvent——"X 年的收获节因大丰收而格外盛大" |
| **节日被中断/取消**（战争/瘟疫） | 完整的 HistoricalEvent——"X 年的收获节因战争被取消" |
| **节日中发生特殊事件**（竞技中有人被杀/神明显灵） | 完整的 HistoricalEvent——附带因果链 |
| **年度规模 ≥ 0.4**（正常型） | 不单独记录——合并到当年的"社会生活摘要"中 |
| **年度规模 < 0.4**（冷清型） | 不记录 |

### 6.2 周期性事件的表示

历史系统当前无周期性事件概念——需要新增：

```rust
/// 周期性事件——在历史系统的时间轴上表示每年/每季重复的事件
struct RecurringEventSeries {
    /// 系列 ID——关联所有年度实例
    series_id: EventSeriesId,
    /// 关联的 FestivalId（如果是节日）
    festival_id: Option<FestivalId>,
    /// 首次发生的年份
    first_occurrence_year: u64,
    /// 最新的发生年份
    latest_occurrence_year: u64,
    /// 跳过的年份（战争/瘟疫导致该年未庆祝）
    skipped_years: Vec<u64>,
    /// 每年实例的轻量记录（仅存规模——不存完整 HistoricalEvent）
    annual_summaries: BTreeMap<u64, f32>, // year → scale
}
```

### 7.2 节日作为时间锚点

NPC 用节日标记时间——这是自然语言的涌现特征：

```
"那件事发生在收获节前三天"  → 语言表达的时间描述模板：
  TimestampTemplate::RelativeToFestival {
      festival_id: FestivalId,
      relation: Before { days: 3 },
  }
```

这在语言表达系统的时间描述模板中新增一个变体——不需要节日系统的特殊逻辑。

## 八、纪念性节日的特殊记忆效应

纪念性节日（战争胜利纪念日、灾难纪念日）有特殊的历史记忆功能：

```
纪念性节日的核心功能 = 防止历史被遗忘

没有纪念性节日：
  历史事件 → 1-80 年（目击者记忆）→ 80-500 年（口传，逐渐扭曲）→ 500+ 年（仅剩书籍/遗迹）
  
有纪念性节日：
  历史事件 → 每年重演 → 即使 500 年后，文化仍记得该事件的基本轮廓
  （细节可能随时间扭曲——但核心叙事被节日维护）
```

这就是现实世界中节日的社会功能——**集体记忆的维护机制**。在 WoWorld 中，它通过每年重复的纪念性节日发生同一种文化仪式来自然实现。
