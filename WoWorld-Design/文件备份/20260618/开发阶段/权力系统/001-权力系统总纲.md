## 一、模块定义

### 1.1 一句话

**权力系统定义"谁对谁能做什么、凭什么、以及被管的人认不认"的底层规则。**

### 1.2 核心洞察

权力不是国家的专利——它是二人以上社会关系的普遍属性。

| 尺度 | 权力关系 | 权力原子组合 |
|------|---------|------------|
| 亲子 | 父亲对孩子的宵禁 | `Constrain(behavior=night_outside)` |
| 师徒 | 师傅对学徒的训练要求 | `Compel(behavior=training)` + `Sanction(punishment=extra_drill)` |
| 雇佣 | 雇主对雇员的劳动索取 | `Extract(resource=labor)` + `Sanction(reward=wage)` |
| 行会 | 会长对会员的准入控制 | `DefineMembership` + `Access(market_participation)` |
| 领主 | 领主对领民的税收 | `Extract(resource=tax)` + `Constrain(behavior=leave_territory)` |
| 王国 | 国王对全体国民的统治 | 全套 17 原子的最大组合 |
| 帝国 | 皇帝通过总督链的间接统治 | `Delegate` 链 + `ConferRank` + `Extract(tribute)` |

**每行的 holder 和 subject 在代码中是同一套类型，同一套逻辑。规模不同，结构相同。**

### 1.3 不是什么

- ❌ **不是政治系统**——"政治"只是权力的一个应用场景（Polity 层次）。权力覆盖亲子、雇佣、行会等所有层次
- ❌ **不是法律系统**——本模块定义 `PrescribeRule` 原子和规范层级。法律的系统性组织（法典/犯罪类型/侦查链/审判程序）由独立的**法律系统**模块负责
- ❌ **不是名声系统**——`legitimacy` = subject 对 holder 权力的认可度。`reputation` = 他者对某实体的综合评价（勇敢/诚实/残忍…）。两个独立维度，两个独立模块
- ❌ **不是经济权力的执行层**——经济市场的 PowerAtom（定价/许可/铸币）保留在经济系统。本模块定义普适权力，桥接层完成映射
- ❌ **不是 NPC 的关系系统**——NPC 的 social relationships（好感/信任/熟悉）已存在于 NPC 模块。权力关系叠加在社交关系之上——两个朋友之间的权力关系可以平等（`legitimacy≈1.0`）也可以不平等（一方支配另一方）

## 二、核心概念层级

```
┌─────────────────────────────────────────┐
│ 涌现层: Polity / DiplomaticRelation     │  ← 007 文档
├─────────────────────────────────────────┤
│ 补充层: Duty / Immunity / 规范层级      │  ← 005 文档
├─────────────────────────────────────────┤
│ 存储层: PowerTopology (分表 SoA + 索引) │  ← 003 文档
├─────────────────────────────────────────┤
│ 原子层: 17 个 UniversalPowerAtom        │  ← 002 文档
├─────────────────────────────────────────┤
│ 基础层: EntityId / PowerDomain / 时间   │  ← 消费已有模块
└─────────────────────────────────────────┘
```

### 逐层概要

**原子层（002）**：权力的最小不可分单元。17 个原子分 5 类——结构（操作拓扑图本身）、自指（自我约束）、关系（单向作用于 subject）、规范（创建通用规则）、裁决（处理争议）。

**存储层（003）**：`PowerTopology`——有向多重图。分表 SoA 存储每条权力边及其专属参数。4 重索引——出边/入边/空间（四叉树）/原子类型。边的完整生命周期管理：创建、行使、衰减、失效、删除（软）。

**补充层（005）**：三层补充机制。
- `Duty`——义务实体。权力边或 PrescribeRule 创建时自动生成对应的义务。违约 → 塌缩链（找有权制裁者 → 找有意愿者 → 执行或 legitimacy 下降）。
- `ImmunitySet`——实体对某些权力原子的免疫。不是原子，是边属性的反向约束。覆盖外交豁免、议会免责、贵族特权。
- 规范层级——多条 `PrescribeRule` 覆盖同一 subject/domain 时的优先规则（委托链深度 > 领域专属 > legitimacy）。

**涌现层（007）**：从拓扑图涌现的高层结构。
- `Polity`——满足 4 条涌现条件的连通分量。惰性标签，每年重算。弱惯性反馈（≤0.15 legitimacy 加成）。政府形式从边模式推断。
- `DiplomaticRelation`——Polity 间的关系。连续分数（-1~+1）→ 离散状态（Allied→War）。War 状态唯一有硬效果的外交状态。

**基础层**（消费已有模块）：`EntityId`（NPC/Player/Group）来自 NPC 系统。`GameTimestamp` 来自时间系统。`WorldPos` 来自世界生成。`PowerDomain` 为本模块新增。

## 三、17 个普适权力原子总览

> 完整定义和行使语义见 [[002-普适权力原子]]。

### 3.1 五分类

| 类别 | 原子 | 操作对象 | 说明 |
|------|------|---------|------|
| **结构原子** | `Constitute` | 拓扑图 | 创建/解散权力结构（团体→部门→国家→条约组织） |
| | `DefineMembership` | 拓扑图 | 决定谁属于/不属于某权力结构 |
| | `Delegate` | 拓扑图 | 将自己的部分权力转移给他人行使 |
| | `Relinquish` | 拓扑图 | 主动放弃自身某条权力边（无接收方） |
| | `Contract` | 拓扑图 | **唯一的双边原子**。创建一对对称受限边 + 相互依存契约 |
| **自指原子** | `Pledge` | 自身 | 自我约束——"我承诺做/不做 X"。婚姻誓言、就职宣誓、信托 |
| **关系原子** | `Constrain` | subject 行为 | 禁止/限制 subject 的某些行为（宵禁、出境禁令、武器禁令） |
| | `Compel` | subject 行为 | 强制 subject 做出特定行为（兵役、劳役、出庭） |
| | `Extract` | subject 资源 | 从 subject 获取资源（税、贡品、劳役产物、罚金） |
| | `Access` | subject 准入 | 控制物理/信息空间的准入（城门钥匙、档案权限） |
| | `ConferRank` | subject 地位 | 赋予/调整/撤销 subject 在群体内的等级（爵位、军衔、段位） |
| | `Represent` | subject 与外部 | 以 subject 名义与外部互动（签条约、声明、交涉） |
| **规范原子** | `PrescribeRule` | subject 群体 | 制定适用于群体的通用行为规范（法律、规章、戒律） |
| | `Derogate` | subject 个体 | 豁免特定 subject 对某规则的遵守（特赦、特权许可） |
| **裁决原子** | `Adjudicate` | 争议 | 对涉及 subject 的争议做有约束力的判定 |
| | `Sanction` | subject | 对 subject 施加奖励或惩罚 |
| | `Remit` | subject | 豁免/撤销已判的惩罚 |

### 3.2 类别间的关系

```
结构原子 → 创建/修改图的节点和边（基础设施）
    │
    └→ 关系原子 / 规范原子 / 裁决原子 → 通过已存在的边对 subject 施加效果（运行时）
            │
            └→ 裁决原子 ← 当关系/规范原子被违反时由裁决原子处理
```

- `Delegate`（结构）授予的边中可包含关系原子、规范原子、裁决原子
- `Constitute`（结构）创建的结构内部自动产生成员间的权力边
- `PrescribeRule`（规范）创建规则 → 规则违反 → `Adjudicate` → `Sanction`/`Remit`
- `Pledge`（自指）创建的自我约束 → 违反 → 由外部 guardian 或 legitimacy 损失来处理

## 四、PowerTopology 总览

> 完整存储和 API 见 [[003-权力拓扑图]]。

### 4.1 数据结构

```rust
/// 权力拓扑——本模块的核心数据结构
pub struct PowerTopology {
    // === 边存储（分表 SoA） ===
    holder_ids: Vec<EntityId>,          // 边的持有者
    subject_ids: Vec<EntityId>,         // 边的承受者
    atom_tags: Vec<AtomTypeTag>,        // u8，17 种原子类型
    legitimacies: Vec<f32>,             // 0-1 legitimacy
    enforcements: Vec<f32>,             // 0-1 实际执行力
    sources: Vec<PowerSource>,          // 8 条获取路径之一
    established_at: Vec<GameTimestamp>,
    valid_until: Vec<Option<GameTimestamp>>,
    last_exercised_at: Vec<Option<GameTimestamp>>,
    succession_rules: Vec<SuccessionRule>,
    active: Vec<bool>,                  // 软删除

    // === 分表：每种原子专属参数（列式存储） ===
    domains: Vec<PowerDomain>,          // 大多数原子通用

    // Extract 专属
    extract_amounts: Vec<Option<u64>>,
    extract_resources: Vec<Option<ResourceType>>,

    // Constrain 专属
    constrain_behaviors: Vec<Option<BehaviorType>>,

    // Compel 专属
    compel_behaviors: Vec<Option<BehaviorType>>,
    compel_duration: Vec<Option<GameDuration>>,

    // Access 专属
    access_targets: Vec<Option<AccessTarget>>,
    access_levels: Vec<Option<AccessLevel>>,

    // ConferRank 专属
    rank_titles: Vec<Option<RankTitle>>,
    rank_attached_powers: Vec<Option<Vec<EdgeId>>>,  // 随 rank 附赠的权力边

    // Represent 专属
    represent_binds_principal: Vec<Option<bool>>,
    represent_scopes: Vec<Option<RepresentScope>>,

    // PrescribeRule 专属
    rule_contents: Vec<Option<RuleContent>>,
    rule_sanctions: Vec<Option<SanctionSpec>>,

    // Delegate 专属
    delegate_atoms: Vec<Option<Vec<AtomTypeTag>>>,
    delegate_revocable: Vec<Option<bool>>,

    // Pledge 专属
    pledge_witnesses: Vec<Option<Vec<EntityId>>>,
    pledge_sanctions: Vec<Option<SanctionSpec>>,

    // === 索引 ===
    outgoing: HashMap<EntityId, Vec<EdgeId>>,
    incoming: HashMap<EntityId, Vec<EdgeId>>,
    spatial_index: QuadTree<WorldBounds, Vec<EdgeId>>,
    atom_type_index: [Vec<EdgeId>; 17],

    // === 涌现缓存 ===
    polities: Vec<Polity>,
    entity_to_polity: HashMap<EntityId, PolityId>,

    // === 统计 ===
    total_edges_created: usize,
    last_legitimacy_recalc: GameTimestamp,
    last_polity_recalc: GameTimestamp,
}
```

### 4.2 边标识

`EdgeId = u32`。支持~40 亿条边——远超 100K NPC 规模下估计的 ~500K 边。

### 4.3 空间索引

对于 `domain = PowerDomain::Territory(polygon)` 的边，使用四叉树索引。查询复杂度 O(log N + M)，N 为领土域数量（估计 200），M 为匹配的边数。

### 4.4 NPC 移动优化

每个 16m×16m Chunk 预计算并缓存适用的领土权力边。NPC 在 Chunk 内移动时零查询。仅在跨 Chunk 边界时触发一次四叉树查询。

## 七、核心设计原则（重申与扩展）

### 7.1 原子组合原则

任何"角色"（国王、族长、行会会长、父亲）都是 17 个原子的特定组合。系统不定义角色——系统只定义原子和边的条件。角色从原子组合中涌现。

```
"国王" ≈ Constitute(polity) + Constrain(domain=Territory) + Extract(tax) + Compel(conscription)
         + PrescribeRule(law) + Delegate(administration) + Represent(diplomacy)
         + ConferRank(nobility) + Adjudicate(supreme) + Sanction(death_penalty)
         + DefineMembership(citizenship) + Access(state_archives) ...
```
> 注：`Constrain(domain=Territory)` 即领土控制——006 解释为何不独立为原子。

换一个 holder，他可能拥有其中 7 个原子。另一个 holder 可能拥有 14 个。他们都被称为"国王"，但权力范围不同——**头衔是从原子组合中派生的标签，原子组合才是事实。**

### 7.2 信息不完备原则

Holder 的信息受限于他收到的汇报。汇报经过语言表达系统的 5 通道 + 5 失真算子传播 → 国王听到的"民情"和地面的"民情"之间可以有巨大差距。

这不是 bug——这是"昏君被奸臣蒙蔽"和"明君深入民间"的涌现机制。两个国王拥有完全相同的原子组合，但因为信息质量不同，治理结果天差地别。

### 7.3 玩家 = NPC 原则

`exercise_power(holder, subject, atom)` 不区分 holder 是 NPC 还是 Player。玩家的 8 条权力获取路径 = NPC 的 8 条获取路径。玩家行使权力的验证逻辑 = NPC 的验证逻辑。

### 7.4 涌现不锁死原则

Polity 是惰性快照——不锁定其内部权力边。但提供弱惯性反馈（legitimacy 加成 ≤0.15）。权力边变化 → 下次重算时 Polity 可能扩大/缩小/分裂/消失。不循环因果，不锁死。

## 九、性能预算（概要）

> 详见 [[008-跨模块接口与性能预算#二、性能预算总表|008 §二]]

| 子系统 | 每帧 CPU | 内存 |
|--------|---------|------|
| 权力行使 | <0.01ms | — |
| 领土查询 | <0.01ms | 四叉树 ~2MB |
| Legitimacy 重算 | <0.02ms（分摊） | 缓存 ~5MB |
| Duty 检查 | <0.001ms | Duty 实体 ~80MB |
| Polity 涌现重算 | <0.05ms（每游戏年） | Polity 缓存 ~5MB |
| PowerTopology 图存储 | — | ~148MB（500K 边 × 各列） |
| **合计** | **<0.1ms/帧** | ~**250MB**（优化后 ~180MB） |

占总 Rust 预算 7ms 的 ~1.4%，总内存预算 1.4GB 的 ~18%（优化后 ~13%）。权力操作是低频社会事件——非性能瓶颈。
