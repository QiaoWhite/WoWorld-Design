## 一、核心哲学：片段组合，非单体模板

### 1.1 为什么不用完整模板

WoWorld 的文本场景组合爆炸太大：
- 书籍类型 15 × 文化区域 20 × 历史时代 3 = 900 种
- 对话场景 30 × 个性类型 10 × 关系层级 5 = 1,500 种
- 如果每个组合写一个完整模板→ 2,400+ 模板→不可能手工维护

### 1.2 片段组合模型

```
不是: 一个模板 = 一段完整文字
而是: 片段(fragment) × 参数(params) × 滤镜(filter) → 最终文字
```

**组合爆炸的反转**：
```
完整模板: 场景数 × 文化数 × 个性类型 × 关系层级 × 情绪状态 = 指数爆炸
片段组合: 场景数 + 文化数 + 个性类型 + 关系层级 + 情绪状态 = 线性增长
```

5 种场景 + 3 种关系 + 2 种个性 = 10 个片段（不是 30 个完整模板）。维度继续增加时差距持续扩大。

### 1.3 数量估算

| 分类 | 片段数 | 说明 |
|------|--------|------|
| 书籍/铭文片段 | ~150 | 编年史/宗教/技术/诗歌/信/碑文/招牌……每种 10-20 片段 |
| 对话片段 | ~250 | 问候/告别/交易/信息/情感/请求/教学/天气/关系……每种 10-25 片段 |
| 咒语/卷轴片段 | ~30 | 魔法方程式文本编码的固定格式 |
| **总计** | **~430** | 全部 TOML 数据驱动 |
| 内存占用 | ~430 × 200B ≈ **86 KB** | 完全可忽略 |

## 二、TextFragment——片段积木

### 2.1 定义

```rust
/// 文本片段——可按条件组合的最小文字单元
pub struct TextFragment {
    pub id: u32,
    /// 激活条件——什么情况下使用这个片段
    pub conditions: FragmentCondition,
    /// 带插槽的文本
    /// "{speaker}说{pronoun}在{location}{verb}了{object}"
    pub text: String,
    /// 该片段需要的参数名列表
    pub required_params: Vec<String>,
    /// 片段的情感基调（用于 SocialField 的 absorb_utterance）
    pub emotional_tone: EmotionalTone,
    /// 输出预估长度（字符数，用于 UI 分页）
    pub estimated_chars: u32,
}

/// 片段激活条件——是函数，不是枚举穷举
pub enum FragmentCondition {
    /// 总是激活
    Always,
    /// 仅在特定文化参数范围内使用
    CultureParam { param: String, range: (f32, f32) },
    /// 仅在说话者某人格维度高于/低于阈值时
    PersonalityThreshold { trait_name: String, min: f32, max: f32 },
    /// 仅在关系信任度高于某值时（如"亲爱的"）
    MinTrust(f32),
    /// 仅在特定情绪下
    Emotion(MoodLabel),
    /// 仅在特定文体下
    Genre(TextGenre),
    /// 仅在一天中的特定时段
    TimeOfDay { from_hour: u8, to_hour: u8 },
    /// 仅在特定天气下
    Weather(WeatherType),
    /// 逻辑组合——所有子条件满足
    All(Vec<FragmentCondition>),
    /// 逻辑组合——任一子条件满足
    Any(Vec<FragmentCondition>),
    /// 逻辑取反
    Not(Box<FragmentCondition>),
}

pub enum TextGenre { Formal, Casual, Poetic, Technical, Religious, Military }
```

### 2.2 CompositeTemplate——片段拼装

```rust
/// 复合模板——片段的有序序列 + 组装策略
pub struct CompositeTemplate {
    pub id: u32,
    /// 候选片段组（按顺序排列）
    pub fragment_groups: Vec<FragmentGroup>,
    /// 片段间的连接策略
    pub join_strategy: JoinStrategy,
}

/// 一组可互替的片段——从中按条件选 0-1 个
pub struct FragmentGroup {
    /// 候选片段（按优先级排序）
    pub candidates: Vec<FragmentRef>,
    /// 选择策略
    pub selection: FragmentSelection,
}

pub enum FragmentSelection {
    /// 选第一个满足条件的
    FirstMatch,
    /// 从满足条件的片段中随机选一个
    RandomMatch,
    /// 选所有满足条件的（如列举）
    AllMatching,
}

pub enum JoinStrategy {
    /// 直接拼接（无分隔符）
    Concat,
    /// 用换行分隔
    Newline,
    /// 用指定分隔符
    Separator(String),
    /// 每个片段是一段——空行分隔
    Paragraphs,
}
```

## 三、TextGenerator——生成引擎

### 3.1 核心实现

```rust
pub struct TextGenerator {
    /// 所有片段——按 ID 索引
    fragments: HashMap<u32, TextFragment>,
    /// 所有模板——按 ID 索引
    templates: HashMap<u32, CompositeTemplate>,
}

impl TextGenerator {
    /// 完整生成流程
    pub fn generate(&self, content: ResolvedContent, ctx: &ReaderContext) -> ReadResult {
        // ① 查找模板
        let template = self.templates.get(&content.template_id)
            .expect("Template not found");

        // ② 筛选片段
        let selected_fragments: Vec<&TextFragment> = template.fragment_groups.iter()
            .filter_map(|group| self.select_fragment(group, &content, ctx))
            .collect();

        // ③ 填入参数 → 每个片段独立填参后拼接
        let raw_text = self.assemble(&selected_fragments, &content.params, &template.join_strategy);

        // ④ 计算可读性
        let language_proficiency = ctx.known_languages
            .get(&content.language)
            .map(|p| p.reading_multiplier())
            .unwrap_or(0.0);
        let aid_boost = ctx.reading_aids.iter()
            .map(|aid| aid.condition_boost())
            .sum::<f32>();
        let effective_condition = (content.condition * (language_proficiency + aid_boost)).min(1.0);

        // ⑤ 应用退化
        let degraded_text = self.apply_degradation(raw_text, effective_condition);

        // ⑥ 返回结果
        ReadResult {
            text: degraded_text,
            language: content.language,
            condition: effective_condition,
            metadata: content.metadata,
            reliability: content.reliability(),
        }
    }

    /// 从一组候选片段中按条件选择
    fn select_fragment<'a>(
        &self,
        group: &FragmentGroup,
        content: &ResolvedContent,
        ctx: &ReaderContext,
    ) -> Option<&'a TextFragment> {
        match group.selection {
            FragmentSelection::FirstMatch => {
                group.candidates.iter()
                    .filter_map(|r| self.fragments.get(&r.id))
                    .find(|f| f.conditions.evaluate(content, ctx))
            }
            FragmentSelection::RandomMatch => {
                let matching: Vec<_> = group.candidates.iter()
                    .filter_map(|r| self.fragments.get(&r.id))
                    .filter(|f| f.conditions.evaluate(content, ctx))
                    .collect();
                // 确定性随机——种子来自 content 的 hash，保证同内容同结果
                if matching.is_empty() { None }
                else { Some(matching[content.deterministic_index(matching.len())]) }
            }
            FragmentSelection::AllMatching => {
                // 返回所有匹配的——由 assemble 处理多个片段的连接
                unimplemented!("AllMatching used in list-type joins")
            }
        }
    }

    /// 文本退化——破损/语言不熟 → 字符概率替换为 '█'
    fn apply_degradation(&self, text: String, condition: f32) -> String {
        if condition >= 0.8 { return text; }  // 几乎完整——不退化

        let threshold = condition;  // 0-1
        text.chars()
            .map(|c| {
                if c.is_whitespace() || c.is_ascii_punctuation() { return c; }
                // 每个字符独立概率: P(可读) = condition
                // 退化是概率性的——同一本书每次打开可能看到不同的模糊文字
                if fast_rand() < threshold { c } else { '█' }
            })
            .collect()
    }
}
```

```rust
// TextGenerator 总是产出文本。三种编码的差异体现在模板选择:

// Encoding::Text → 标准模板填充

//          "中心为{node_shape}释放节点。法阵用途：{purpose_summary}。"

// Encoding::Audio → 转录模板
//   参数: { incantation_text, duration_seconds, caster_name, element, effect_summary }
//   模板: "咒语音频记录。施法者：{caster_name}。咒文转录：\"{incantation_text}\"。"

// Godot 层在收到 ReadResult 后检查 ContentMetadata.encoding:
//   if Geometric → 用同一个 ExpressionRef 调 Magic::render_formation() 画图
```

### 3.3 参数填充——条件替换

```rust
impl TextGenerator {
    fn fill_params(&self, template: &str, params: &HashMap<String, String>) -> String {
        let mut result = template.to_string();
        for (key, value) in params {
            // 支持条件替换: {key?if_true:if_false}
            // 如 "{casualties?死伤约{casualties}人:无伤亡}"
            if let Some(cond_result) = self.resolve_conditional(key, value, params) {
                result = result.replace(&format!("{{{key}}}"), &cond_result);
            } else {
                result = result.replace(&format!("{{{key}}}"), value);
            }
        }
        // 清理未填充的插槽——替换为通用占位符
        self.clean_unfilled_slots(result)
    }
}
```

## 四、滤镜管道

```rust
/// 文本滤镜——在 TextGenerator.generate() 的步骤③和④之间按顺序应用
pub trait TextFilter {
    /// 对文本进行后处理
    fn apply(&self, text: String, ctx: &FilterContext) -> String;
}

pub struct FilterContext {
    pub npc_personality: Option<NpcPersonality>,
    pub npc_mood: Option<Mood>,
    pub relationship_trust: Option<f32>,
    pub genre: TextGenre,
}

/// 个性滤镜——调整措辞风格
pub struct PersonalityFilter;
impl TextFilter for PersonalityFilter {
    fn apply(&self, text: String, ctx: &FilterContext) -> String {
        let personality = match &ctx.npc_personality { Some(p) => p, None => return text };
        let mut result = text;
        // 低友善度 NPC → 去掉感叹号、缩短句子
        if personality.agreeableness < 0.3 {
            result = result.replace('！', '。');
            result = result.replace("欢迎光临", "嗯。");
        }
        // 高外向性 NPC → 增加生动性、口语化
        if personality.extraversion > 0.7 {
            // 保持原样——高外向性的人说话本来就生动
        }
        // 低开放性 NPC → 减少抽象/比喻
        if personality.openness < 0.3 {
            // 过滤掉诗意的比喻片段——但这是片段选择阶段做的事（FragmentCondition）
        }
        result
    }
}

/// 关系滤镜——调整亲密度措辞
pub struct RelationshipFilter;
impl TextFilter for RelationshipFilter {
    fn apply(&self, text: String, ctx: &FilterContext) -> String {
        let trust = ctx.relationship_trust.unwrap_or(0.1);
        let mut result = text;
        if trust < 0.2 {
            // 陌生人——正式/疏离
            result = result.replace("你", "您");
        }
        if trust > 0.7 {
            // 亲近——可以用亲昵称呼
            // (由 FragmentCondition::MinTrust 在片段选择阶段处理)
        }
        result
    }
}

/// 滤镜链——按注册顺序依次应用
pub struct FilterChain { filters: Vec<Box<dyn TextFilter>> }
impl TextFilter for FilterChain {
    fn apply(&self, text: String, ctx: &FilterContext) -> String {
        self.filters.iter().fold(text, |t, f| f.apply(t, ctx))
    }
}
```

## 六、性能预算

| 操作 | 单次耗时 | 频率 | 说明 |
|------|---------|------|------|
| 模板查找 | ~10 ns | 交互 | HashMap u32 key |
| 片段筛选 | ~1 µs | 交互 | ~10-20 候选片段遍历 + 条件检查 |
| 参数填充 | ~1-10 µs | 交互 | 字符串替换（文本长度 100-2000 字符） |
| 退化处理 | ~1-5 µs | 交互 | 逐字符概率替换 |
| 滤镜管道 | ~1-5 µs | 交互 | 3-4 个简单文本替换 |
| **一次 generate() 总耗时** | **~10-50 µs** | 人控频率 | 不每帧 tick |
| 片段库加载 | ~10 ms | 启动时 | 一次性 TOML 解析 |
| 片段库内存 | ~86 KB | 常驻 | ~430 片段 × 200B |

**文本生成不在性能预算雷达上。** 真正的性能瓶颈是体素网格、路径寻找、物理模拟——不是字符串拼装。
