## 〇、生命周期 L1-L4 ↔ LODCoordinator ai_lod 0-4 映射 ★CHG-049

> **关联**: [[../../../../../Change/CHG-049-LOD架构全面深化-20260620|CHG-049 §十一]] · [[../../模块接头总览/00-全局基础设施/001-核心类型注册表|LodPrescription]]

### 0.1 显式映射

| 生命周期层级 | 距离 | ai_lod | advance_age 频率 | 行为模拟精度 |
|------------|------|--------|-----------------|-------------|
| L1 | 0-50m | ai_lod 0-1 | 每帧 (~0.00035d) | 全精度——逐NPC逐帧 |
| (L1 延伸) | 50-150m | ai_lod 1 | 每帧 | 轻量决策——advance_age 极廉价 (<0.1% CPU)，无需中间层 |
| L3 | 150m-5km | ai_lod 2-3 | 月末批 (delta≈30d) | 统计模板/聚合——批处理 |
| L4 | 5km+ | ai_lod 4 | 年末批 (delta≈120d) | 仅存在——统计计数器，无个体状态 |

### 0.2 核心不变量（不受此映射影响）

1. **统一时间流速**：所有层级 `Δtime` 完全相同。批处理 = 累积 delta 一次算完，非加速
2. **advance_age 零 LOD 耦合**：纯函数 `(entity, delta_days, rng) → (LifeEntity, Vec<LifeEvent>)`——不引用 LOD 概念
3. **LOD 重构弹性**：未来 LOD 层数/距离带变化 → 只改调度器调用频率，不改 advance_age

## 一、统一时间流速——核心约束

### 1.1 不可加速老化

所有 LOD 层共用**完全相同的时间流速**。L3/L4 的 NPC 不能加速老化。

| LOD | 时间流速 | advance_age 调用频率 | 当量 |
|-----|:-------:|---------------------|------|
| L1 | 1× | 每帧 (~0.00035 天) | 逐帧 |
| L3 | 1× | 每 30 游戏日（月末批处理） | 月 |
| L4 | 1× | 每 120 游戏日（年末批处理） | 年 |

30 天后——无论玩家在世界哪个角落——所有 NPC 都老了 30 天。批处理只是把 30 天的变化**一次算完**，不是让时间加速 30 倍。

### 1.2 为什么不能加速

如果 L3 NPC 以 4× 老化，当玩家从 A 村走到 B 城时，B 城的 NPC 已经老了 4 倍的量。"一别三月，故人已老三十年"——世界一致性破裂。

`advance_age()` 是纯函数——同输入 + 同 delta + 同种子 → 同输出。如果 delta 不同，输出不同。如果不同 LOD 用不同 delta，世界状态分裂。

## 二、advance_age 性能分析

### 2.1 单次调用成本

| 操作 | 次数 | 成本 |
|------|:---:|------|
| 阶段边界查表 | O(1)——预计算数组索引 | ~1 次浮点比较 |
| age 推进 | 1 | ~1 次浮点加法 |
| 阶段转换检测 | 最坏 6 次比较 | ~6 次整数比较 |
| 妊娠推进 | 仅若 pregnant | ~2 次浮点运算 |
| Gompertz 生存 | 仅若 age_pct≥0.7 | **1 次 exp()** |
| 事件构造 | 极罕见 | ~0（一生 ≤6+1 次） |

**单次典型成本**：~50-100 CPU 周期（无 exp），~200-300 周期（有 exp，仅老龄 NPC）。

### 2.2 分 LOD 负载

| LOD | NPC 数 | 调用频率 | 含 exp 的调用比例 | 年化单 NPC 调用次数 |
|-----|--------|---------|:--------------:|:--------------:|
| L1 | ~50 | 每帧 | 30%（老龄） | ~120×365=43800 |
| L3 | ~2000 | 月批 | 30% | 12 |
| L4 | ~50000 | 年批 | 30% | 1 |

**等效 per-frame 负载**：

```
L1: 50 NPC × 1次/帧 = 50 次/帧    (<0.1% CPU)
L3: 2000 NPC × 1次/30天/每1.5秒批 = 分摊后 <0.01% CPU
L4: 50000 NPC × 1次/120天/每6秒批 = 分摊后 <0.001% CPU
```

**性能瓶颈不在 advance_age 计算——在事件的副作用派发。**

## 三、关键优化

### 3.1 阶段边界预计算

```rust
// NPC 创建时一次性：
let boundaries = LifeStageBoundaries::from_race(&race);
// 6 个 f32 值 → 24 字节/NPC

// advance_age 中：
// 不需要每次从 age_pct 重算阶段
// 只需要比较 age_years 是否越过下一个 boundary
fn days_until_next_boundary(age: f32, boundaries: &[f32; 6]) -> f32 {
    for &b in boundaries {
        if b > age { return (b - age) * DAYS_PER_YEAR; }
    }
    f32::MAX  // 已达到最大寿命——仍需要周期性检查
}
```

### 3.2 衰老查表（L3/L4）

```rust
/// 每个物种共享一张 SenescenceTable
/// 大小：(age_pct 步长 0.01) × (constitution 步长 0.1) = ~80×11 = 880 f32
/// 单表 ~3.5KB。全物种 ~20 表 ≈ 70KB 总内存
pub struct SenescenceTable {
    survival_30d: [[f32; 11]; 80],  // [age_idx][constitution_idx]
}

// L3/L4 查表替代 exp():
fn senescence_survival_l3(age_idx: usize, constitution_idx: usize, table: &SenescenceTable) -> f32 {
    table.survival_30d[age_idx][constitution_idx]
}
```

### 3.3 L3/L4 批处理事件接口

```rust
/// 替代逐实体返回 Vec<LifeEvent>
/// BatchedLifeEvents 使用预分配大 Vec——减少分配次数
fn advance_ages_batched(
    entities: &[LifeEntity],
    delta_days: f32,
    rng: &mut Rng,
) -> (Vec<LifeEntity>, BatchedLifeEvents);

struct BatchedLifeEvents {
    deaths: Vec<(EntityId, DeathCauseId)>,
    births: Vec<BirthDueEvent>,
    stage_transitions: Vec<(EntityId, LifeStage, LifeStage)>,
    gestation_outcomes: Vec<(EntityId, GestationOutcome)>,
}
```

### 3.4 L4 不产生独立事件

L4 实体在年末批处理中——只更新状态，不产生 LifeEvent。事件在 L4 级别没有意义（没有人在附近感知、没有 GOAP 需要响应）。人口变化汇总到统计计数器。

## 四、L1↔L3↔L4 切换

### 4.1 L4→L3/L1 切换（玩家靠近）

```
L4 实体已通过年末批处理推进到当前时间
→ 状态是最新的（age_years 正确、is_dead 已确定）
→ 切换到 L1：从 LifeEntity 状态反推当前阶段、物理属性、InfantDependency
→ 不需要"模拟中间的每一帧"——确定性函数保证状态一致
```

### 4.2 L1→L3/L4 切换（玩家远离）

```
L1 实体正在被逐帧处理
→ 切换到 L3：将 stale_days（自上次 L3 批处理以来累积的天数）清零
→ 下次 L3 批处理时以累积天数推进
→ 不需要"冻结状态"——stale_days 机制确保不丢失时间
```

## 五、内存预算

### 5.1 运行时内存分配（100k NPC 规模）

| 数据 | 每个 NPC | 总量 | 占比 |
|------|:-------:|------|:---:|
| LifeEntity (基础) | ~512B | 51.2 MB | 非生命周期独有——所有 LifeEntity 均有 |
| AgeClock 相关（边界+种子） | ~56B | 5.6 MB | 新增 |
| InfantDependency | ~16B | 0.2 MB | 仅 Juvenile 阶段 NPC 持有 |
| PrenatalAccumulator | ~256B | <1 MB | 仅 Pregnant 标签持有者 |
| GestationState | ~128B | <0.5 MB | 仅受孕中的 NPC |
| DeathSummary | ~3KB | ~3-6 MB | 死人——随时间积累。冷归档到 LMDB |
| SenescenceTable（共享） | ~70KB | 70 KB | 全物种共享——不计入 per-NPC |

**新增内存总量（相对现有）**：~10-15 MB（100k NPC 规模）。在 1.4GB 预算中占比 <1%。

### 5.2 死亡压缩的效果

如果不压缩，每个死 NPC 保留完整 NpcData（~50KB）→ 100k NPC 中假设 30% 已死 → 1.5GB——超出预算。压缩后：30k 死 NPC × 3KB = 90MB 内存 + LMDB 冷存储。

## 六、LOD 重构弹性

### 6.1 当前 LOD 层数（4）是否会大改？

设计中考虑了 LOD 架构可能在未来被重构——从 4 层变更多层、或更改分层标准。

**advance_age 对 LOD 架构完全无感知**：

```rust
// advance_age 不引用任何 LOD 概念
// 它只接收 delta_days 和 entity 状态
// 谁调用、以什么频率调用、实体处于哪个 LOD 层——advance_age 完全不知
pub fn advance_age(entity: &LifeEntity, delta_days: f32, rng: &mut Rng) 
    -> (LifeEntity, Vec<LifeEvent>);
```

未来 LOD 重构：
- 只改变**调度器**（谁在什么时候以什么频率调用 advance_age）
- 不改变 advance_age 本身
- 不改变事件协议
- 不改变各模块对事件的响应

## 七、性能检查清单

- [x] Gompertz exp() 不是 per-frame per-NPC——仅老龄 + 仅 L1 逐帧
- [x] L3/L4 使用查表消除 exp()
- [x] 阶段边界预计算——O(1) 查表而非 per-call 百分比重算
- [x] L3/L4 批处理接口消除 per-entity Vec 分配
- [x] L4 不产生事件——仅统计计数器
- [x] InfantDependency L3/L4 退化为纯统计——不需要逐次喂奶模拟
- [x] 衰老查表全局共享（Arc<SenescenceTable>）——不计入 per-NPC 预算
- [x] 死亡三阶段压缩——内存从 ~50KB/NPC 降至 ~1-3KB 最终 ~128B
- [x] advance_age 对 LOD 架构零耦合——任意重构不影响

**整个生命周期系统在总帧预算中的占比**：<0.2%。
