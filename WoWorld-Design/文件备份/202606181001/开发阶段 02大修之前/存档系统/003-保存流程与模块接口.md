## 一、SaveableModule trait v2.0

### 1.1 完整定义

```rust
/// 模块与存档系统的唯一契约。
/// 存档系统只知道此 trait——不知道任何模块的具体类型。
///
/// 共 14 方法：4 必须覆盖 + 10 有合理默认值。
/// 最简单的模块（无持久化数据）：只需覆盖 4 个必须方法。
trait SaveableModule: Send + Sync {
    // ═══════════════════════════════════════════════════════
    // 必须覆盖（4 方法）
    // ═══════════════════════════════════════════════════════

    /// 本模块的唯一名称。用于 module_versions 映射和日志。
    /// 示例: "npc", "items", "history"
    /// 此名称是持久化身份——改名称 = 新身份 = 旧数据不被迁移。
    fn module_name(&self) -> &'static str;

    /// 模块当前数据格式版本号。
    /// 存档时将版本写入 Header.module_versions。
    /// 读档时和存档中的版本比对——决定是否需要迁移。
    /// 版本号必须线性递增（1, 2, 3, ...）。
    ///
    /// migrate(from_version, txn) 保证 from_version < current_version()——
    /// 模块内部负责链式执行所有中间版本。from_version 和 current_version()
    /// 之间的差可以 >1——只要每个中间版本的迁移函数存在。
    fn current_version(&self) -> u32;

    /// 本模块写入的 named_db 列表及键前缀。
    ///
    /// 返回 `&[(db_name, key_prefix), ...]`。空切片 = 无持久化数据。
    /// (db_name, key_prefix) 组合在所有注册模块中必须唯一——
    /// SaveSystem 在模块注册时验证，冲突 → panic（debug）/ 注册失败（release）。
    ///
    /// 示例:
    ///   简单模块: &[("faith", "faith/")]
    ///   多前缀模块: &[("npc", "npc/memory/"), ("npc", "npc/relationship/")]
    ///   无持久化: &[]
    ///
    /// 键前缀以 module_name() 开头是强约定——模块违反此约定 = 模块 bug。
    fn named_dbs(&self) -> &[(&str, &str)];

    // ═══════════════════════════════════════════════════════
    // 元数据（可选覆盖——2 方法）
    // ═══════════════════════════════════════════════════════

    /// 本模块是否为致命模块。
    /// 致命模块的 named_db 损坏 → 拒绝加载整个存档。
    /// terrain、npc、player 应返回 true。
    fn is_critical(&self) -> bool { false }

    /// 估算下次存档的脏数据序列化后字节数。
    ///
    /// ## 精度约束
    /// 应控制在 ±50% 以内。如果无法满足此精度，返回 None。
    /// SaveSystem 对返回 None 的模块使用上次存档的模块体积 × 1.5 作为启发式估算。
    ///
    /// ## 用途
    /// SaveSystem 在存档前汇总所有模块的估算值，若估算总量超过
    /// mapsize 剩余空间的 80% → 弹出磁盘空间预警。
    fn estimate_dirty_bytes(&self) -> Option<u64> { None }

    // ═══════════════════════════════════════════════════════
    // 保存（可选覆盖——5 方法）
    // ═══════════════════════════════════════════════════════

    /// 本模块是否有脏数据需要持久化。
    /// 在 tick 边界查询。调用者保证无并发可变借用。
    fn has_dirty(&self) -> bool { false }

    /// 序列化本模块的脏数据到内存快照。
    ///
    /// ## 调用约定
    /// - 在 tick 边界调用。调用者保证此时没有对模块的可变借用。
    /// - `&self` 不可变借用——序列化只读。
    /// - 不消耗脏标记——脏标记只在 confirm_snapshot_written() 成功后清除。
    ///
    /// ## 线程安全
    /// `snapshot_dirty(&self)` 在 tick 边界的主线程调用，但后台线程
    /// （如 history monthly_tick）可能同时运行。`Send + Sync` 约束保证
    /// `&self` 可安全跨线程共享——实现者使用内部同步（Mutex/RwLock）确保数据一致性。
    ///
    /// ## 实现约定
    /// - 各模块自行决定序列化格式（bincode、rkyv 等）。
    /// - 存档系统不解析返回的 bytes——只传递。
    /// - 返回的 NamedDbSnapshot 的 db_name 必须在 named_dbs() 声明中——
    ///   SaveSystem 在 Phase 1 后验证，不匹配 → 返回错误。
    fn snapshot_dirty(&self) -> Result<DirtySnapshot> {
        Ok(DirtySnapshot::empty())
    }

    /// 流式写入脏数据——直接写入 LMDB 写事务，不经过内存快照收集。
    ///
    /// ## 为什么需要此方法
    /// `snapshot_dirty()` 将所有脏数据收集为 `Vec<(Vec<u8>, Vec<u8>)>` 再写入——
    /// 对于大模块（物品 ~1.7GB、NPC ~520MB），增量脏数据也可能达到数十 MB。
    /// `write_dirty` 允许模块逐实体/逐 Chunk 直接写 LMDB，避免内存峰值。
    ///
    /// ## 默认实现
    /// 调用 snapshot_dirty() → 逐条写入 txn。小模块无需覆盖。
    ///
    /// ## 注意事项
    /// - 在单个 LMDB 写事务内调用。写入失败 → 事务回滚，脏标记保留。
    /// - 实现者负责 key_prefix 与 named_dbs() 声明一致。
    fn write_dirty(&self, txn: &mut RwTransaction) -> Result<()> {
        let snap = self.snapshot_dirty()?;
        for db_snap in &snap.db_snapshots {
            let db = txn.open_db(Some(&db_snap.db_name))?;
            for (key, value) in &db_snap.entries {
                txn.put(&db, key, value)?;
            }
            for key in &db_snap.deleted {
                txn.del(&db, key)?;
            }
        }
        Ok(())
    }

    /// 写入全部初始数据——Initial 存档专用。
    ///
    /// ## 与 write_dirty 的区别
    /// - `write_initial`: 枚举所有实体，全量写入。初始状态无"脏"概念。
    /// - `write_dirty`: 仅写入自上次存档以来变化的实体。
    ///
    /// ## 默认实现
    /// 委托给 snapshot_dirty()——对于初始状态，snapshot 返回全部数据。
    /// 大模块应覆盖此方法做流式写入（Initial 存档仅发生一次，但数据量可达 GB 级）。
    fn write_initial(&self, txn: &mut RwTransaction) -> Result<()> {
        let snap = self.snapshot_dirty()?;
        for db_snap in &snap.db_snapshots {
            let db = txn.open_db(Some(&db_snap.db_name))?;
            for (key, value) in &db_snap.entries {
                txn.put(&db, key, value)?;
            }
            for key in &db_snap.deleted {
                txn.del(&db, key)?;
            }
        }
        Ok(())
    }

    /// 确认快照已成功写入 LMDB。
    ///
    /// ## 调用约定
    /// - 仅在写入 + 提交 + rename 全部成功后调用。
    /// - 如果写入失败，此方法不会被调用——脏标记保留，下次存档重试。
    ///
    /// ## 实现约定
    /// - 使用 snapshot.snapshot_tick 和实体内部的 dirty_since 字段比对。
    /// - 只清除 dirty_since <= snapshot_tick 的实体。
    /// - dirty_since > snapshot_tick 的实体保留（快照后又被修改了）。
    fn confirm_snapshot_written(&mut self, snapshot: &DirtySnapshot) {}

    // ═══════════════════════════════════════════════════════
    // 读档（可选覆盖——3 方法）
    // ═══════════════════════════════════════════════════════

    /// 从存档加载本模块的数据到内存。
    ///
    /// ## 调用约定
    /// - 可以假设 self 是初始状态（Default 或刚构造）。
    /// - 不需要在 load 前做任何预初始化。
    /// - 本方法不解析跨模块引用——那些在运行时懒解析。
    ///
    /// ## LoadContext
    /// `ctx.txn` 提供当前读事务。`ctx.create_txn()` 创建新的读事务——
    /// 模块用于渐进/按需加载（如物品系统按 Chunk 按需加载）。
    /// 模块不需要持有 `&SaveSystem` 引用——LoadContext 提供创建读事务的完整能力。
    ///
    /// ## Schema 版本转换
    /// 如果本模块存储了其他模块定义的 Schema 数据（ConsumableEffect 模式），
    /// 应在 load() 中完成批量版本转换——纳入加载画面预算（≤5s），
    /// 避免运行时首次访问的性能尖刺。
    ///
    /// ## 实现约定
    /// - 模块自行决定加载多少数据（全量反序列化 vs 只读元数据+按需加载）。
    /// - 大模块（如物品系统 1.7GB）不必全量加载到内存。
    fn load(&mut self, ctx: &LoadContext) -> Result<()> { Ok(()) }

    /// 将模块重置为默认状态。
    ///
    /// ## 调用约定
    /// - 仅在 load() 返回 Err 且 is_critical() == false 时调用。
    /// - 调用后模块处于干净初始状态——可继续游戏（以默认/空数据运行）。
    /// - 调用者保证 reset_to_default() 和 load() 不在同一 borrow 范围内。
    fn reset_to_default(&mut self) {}

    // ═══════════════════════════════════════════════════════
    // 迁移与验证（可选覆盖——2 方法）
    // ═══════════════════════════════════════════════════════

    /// 版本迁移——从旧格式升级到当前格式。
    ///
    /// ## 调用约定
    /// - `from_version` 是存档中的版本号（从 Header.module_versions 读取）。
    /// - 只在读档时调用——且仅在 from_version < current_version() 时。
    /// - 在独立的读写事务中调用。
    /// - from_version 到 current_version() 的差可以 >1——
    ///   模块内部用 while 循环链式执行所有中间版本。
    ///
    /// ## 实现约定
    /// - 迁移是"磁盘→磁盘"操作——不修改 self（因此是 &self 而非 &mut self）。
    /// - 模块自行实现版本链：while current < self.current_version() { match current { ... } }
    /// - 迁移后不删除旧版本的数据（留给后续迁移可能仍需的原数据）。
    fn migrate(&self, from_version: u32, txn: &RwTransaction) -> Result<()> { Ok(()) }

    /// 迁移后验证——在迁移事务提交前执行。
    ///
    /// ## 约束
    /// - 必须在 <1s 内完成。超过此时间为模块 bug（SaveSystem 记录日志但不强制超时）。
    /// - 在迁移的写事务内调用——验证失败 → 事务回滚。
    ///
    /// ## 默认行为
    /// 返回空 Vec——无额外验证。
    /// 致命模块应覆盖此方法做基本完整性检查（named_db 条目数、关键字段非空等）。
    fn validate_after_migration(&self, txn: &RoTransaction) -> Result<Vec<MigrationWarning>> {
        Ok(Vec::new())
    }
}
```

### 1.2 方法总览

| # | 方法 | 调用时机 | &self / &mut self | 必须覆盖 |
|---|------|---------|-------------------|---------|
| 1 | `module_name()` | 存档/读档 | &self | ✅ |
| 2 | `current_version()` | 存档/读档 | &self | ✅ |
| 3 | `named_dbs()` | 注册时/存档 | &self | ✅ |
| 4 | `is_critical()` | 预检 | &self | — |
| 5 | `estimate_dirty_bytes()` | 存档前 | &self | — |
| 6 | `has_dirty()` | 存档前查询 | &self | — |
| 7 | `snapshot_dirty()` | 存档收集阶段 | &self | — |
| 8 | `write_dirty()` | 流式增量写入 | &self | — |
| 9 | `write_initial()` | Initial 存档 | &self | — |
| 10 | `confirm_snapshot_written()` | 存档确认阶段 | &mut self | — |
| 11 | `load()` | 读档 | &mut self | — |
| 12 | `reset_to_default()` | load 失败恢复 | &mut self | — |
| 13 | `migrate()` | 版本迁移 | &self | — |
| 14 | `validate_after_migration()` | 迁移后 | &self | — |

**14 方法：4 必须覆盖 + 10 有合理默认值。** 最简单的模块（无持久化数据）只需覆盖 `module_name()`、`current_version()`、`named_dbs()`（返回 `&[]`）——3 行代码。

### 1.3 Object Safety

所有 14 个方法返回 `Sized` 类型或 `()`。`SaveableModule` 是 object-safe 的——可作为 `&mut dyn SaveableModule` 使用。

## 二、LoadContext——渐进加载上下文

### 2.1 定义

```rust
/// 渐进加载上下文。
/// 模块在 load() 中接收此引用——不依赖 &SaveSystem（零耦合）。
struct LoadContext<'env> {
    /// 当前读事务。用于主加载流程。
    txn: RoTransaction<'env>,

    /// 创建新读事务的工厂。
    /// 模块调用 ctx.create_txn() 获取独立读事务做按需/渐进加载。
    /// 每次调用创建新的 LMDB 读事务——模块应复用而非频繁创建。
    create_txn: Box<dyn Fn() -> Result<RoTransaction<'env>> + 'env>,
}

impl<'env> LoadContext<'env> {
    /// 创建一个新的独立读事务。
    ///
    /// ## 用途
    /// 大模块（物品系统、NPC 系统）在 load() 中先读元数据/索引，
    /// 运行时按需调用 create_txn() 加载特定 Chunk 的数据。
    ///
    /// ## 性能
    /// 每次调用有 LMDB 事务创建开销（~µs 级）。模块应批量读取
    /// 而非逐实体创建事务。典型用法：每 Chunk/每区域一个事务。
    fn create_txn(&self) -> Result<RoTransaction<'env>> {
        (self.create_txn)()
    }
}
```

### 2.2 使用示例

```rust
impl SaveableModule for ItemsModule {
    fn load(&mut self, ctx: &LoadContext) -> Result<()> {
        // 1. 用当前事务加载物品元数据
        let meta_db = ctx.txn.open_db(Some("items"))?;
        self.load_metadata(&ctx.txn, &meta_db)?;

        // 2. 记录 LMDB 路径，运行时按 Chunk 渐进加载
        //    不在此处加载全部 1.7GB 物品数据
        self.lmdb_reader = Some(ctx.create_txn_factory());

        Ok(())
    }
}

// 运行时按需加载
impl ItemsModule {
    fn load_items_in_chunk(&self, chunk: ChunkCoord) -> Result<Vec<ItemState>> {
        let txn = self.lmdb_reader.as_ref()
            .ok_or(Error::NotLoaded)?
            .call()?;  // 调用 create_txn 工厂
        // 范围扫描此 Chunk 的物品...
    }
}
```

## 三、脏标记的时间戳保护（竞态安全）

### 3.1 问题

`snapshot_dirty(&self)` 是 `&self`——不消耗脏标记。但如果在 snapshot 返回后、LMDB 写入完成前的这段时间内，另一个游戏 tick 修改了同一批数据——如果 confirm 时简单清除所有脏标记，那些"快照后才变脏"的修改会丢失。

### 3.2 解决方案：`dirty_since` 时间戳

> **以下为参考实现。** 各模块自行决定如何维护 `dirty_since`——可以是实体字段、`BTreeMap<EntityId, u64>` 映射、`DirtyTracked<T>` wrapper，或任何其他方式。存档系统只通过 `DirtySnapshot.snapshot_tick` 提供时间戳给 `confirm_snapshot_written()`。模块内部实现完全自治。

每个持久化实体携带一个 `dirty_since: u64` 字段（参考示例）：

```rust
struct PersistEntity {
    data: EntityData,
    dirty_since: u64,  // 变脏时的游戏 tick。u64::MAX = 干净
}

impl PersistEntity {
    const CLEAN: u64 = u64::MAX;

    fn mark_dirty(&mut self, current_tick: u64) {
        self.dirty_since = current_tick;
    }

    fn is_clean(&self) -> bool {
        self.dirty_since == Self::CLEAN
    }
}
```

### 3.3 confirm 时比对

```rust
fn confirm_snapshot_written(&mut self, snapshot: &DirtySnapshot) {
    let tick = snapshot.snapshot_tick;
    for entity in self.all_dirty_tracked_mut() {
        if entity.dirty_since <= tick {
            entity.dirty_since = PersistEntity::CLEAN;
        }
        // dirty_since > snapshot_tick → 快照后被修改 → 保留脏标记
    }
}
```

### 3.4 场景验证

```
Tick 100（存档触发）:
  实体A dirty_since=80, 实体B dirty_since=90, 实体C dirty_since=95
  snapshot_dirty(&self) → 序列化 A,B,C → snapshot_tick=100
  → LMDB 写入中...

Tick 101-103（游戏继续）:
  实体B dirty_since=101（玩家又改了它）
  实体D dirty_since=102（新变脏）

LMDB 写入成功。
confirm_snapshot_written(snapshot, snapshot_tick=100):
  实体A dirty_since=80 ≤ 100 → 清除 ✓
  实体B dirty_since=101 > 100 → 保留 ✓（快照后又被修改了）
  实体C dirty_since=95 ≤ 100 → 清除 ✓
  实体D dirty_since=102 > 100 → 保留 ✓（快照后才变脏）

下次存档: B 和 D 会被包含。✓
```

## 四、DirtySnapshot 与 NamedDbSnapshot

```rust
/// 一个模块的脏数据快照
struct DirtySnapshot {
    /// 按 named_db 分组的写入数据
    db_snapshots: Vec<NamedDbSnapshot>,
    /// 快照创建时的游戏 tick——用于 confirm 时的时间戳比对
    snapshot_tick: u64,
}

/// 单个 named_db 的快照
struct NamedDbSnapshot {
    /// 目标 named_db 名称（如 "npc"）——必须在 named_dbs() 声明中
    db_name: String,
    /// (LMDB_key_bytes, serialized_value_bytes)
    entries: Vec<(Vec<u8>, Vec<u8>)>,
    /// 需要从 named_db 删除的键
    deleted: Vec<Vec<u8>>,
}
```

**存档系统不解析 entries 和 deleted 的内容。** 键和值都是不透明 bytes。

## 五、保存流程

### 5.1 Initial 存档（直写路径）

> **Initial 存档不走三阶段流程。** 原因：Initial 存档需要全量写入 ~3GB 数据，三阶段流程的 Phase 1 会将所有模块的序列化数据同时收集到内存中——超过 Rust 1.4GB 内存预算。

```rust
fn create_initial_save(
    &self,
    world_dir: &Path,
    modules: &[&dyn SaveableModule],  // &self——初始状态，无脏标记
    header: HeaderBuilder,
) -> Result<SaveResult> {
    // 1. 创建临时文件
    let tmp_path = world_dir.join("saves").join("initial.woworld.tmp");

    // 2. 打开 LMDB 环境 + 写事务
    let env = open_lmdb(&tmp_path, INITIAL_MAPSIZE)?;
    let txn = env.begin_rw_txn()?;

    // 3. 逐个模块直接写入——不经过内存收集
    for module in modules {
        if !module.named_dbs().is_empty() {
            module.write_initial(&mut txn)?;
        }
    }

    // 4. 写入 Header
    write_header(&txn, &header.build(modules))?;

    // 5. 提交 + rename + 验证
    txn.commit()?;
    env.sync()?;
    drop(env);
    fsync(&tmp_path)?;
    rename(&tmp_path, &final_path)?;

    // 6. 写入后验证（全 named_db 遍历）
    validate_save_file(&final_path)?;

    Ok(SaveResult::Written { path: final_path, size: file_size })
}
```

### 5.2 增量存档（三阶段流程）

```
┌─ Phase 1: 收集快照（主线程，tick 边界）───────────────────┐
│                                                             │
│  let header = HeaderBuilder::new(...).build(modules);       │
│                                                             │
│  let mut snapshots: Vec<(&str, DirtySnapshot)> = Vec::new();│
│  for module in modules.iter() {                             │
│      if module.has_dirty() {                                │
│          let snap = module.snapshot_dirty()?;               │
│          if snap.is_empty() {                               │
│              warn!("{}: 跳过空快照", module.module_name()); │
│              continue;  // 不写入、不 confirm，脏标记保留   │
│          }                                                  │
│          // 验证: snapshot 中每个 db_name 在 named_dbs() 中 │
│          verify_db_names(module, &snap)?;                   │
│          snapshots.push((module.module_name(), snap));      │
│      }                                                      │
│  }                                                          │
│                                                             │
│  // Manual 存档：即使无脏也强制写（玩家要求）                │
│  // Death Exit 存档：总是写入                               │
│  // Auto/Quick：无脏 → 返回 SaveResult::Skipped             │
│                                                             │
│  // 磁盘空间预警: Σ estimate_dirty_bytes() ≥ mapsize*0.8    │
│  //   → 返回 SaveWarning::LowDiskSpace                      │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Phase 2: 写入 LMDB（临时文件）────────────────────────────┐
│                                                             │
│  1. 创建 save_<uuid>.woworld.tmp                            │
│  2. 打开 LMDB env                                           │
│  3. 创建写事务                                              │
│  4. 对实现了 write_dirty 的大模块 → 调用 write_dirty(txn)  │
│     对使用默认实现的模块 → 从 snapshots 逐条写入             │
│  5. 写入 meta/header（HeaderBuilder 构造的 SaveHeader）     │
│  6. 提交事务                                                │
│  7. 关闭 LMDB env                                           │
│  8. fsync 临时文件                                          │
│  9. 原子重命名: .tmp → .woworld                             │
│                                                             │
│  任一失败 → 事务回滚 → .tmp 残留（启动清理）→ 旧存档完好     │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Phase 3: 确认清理（主线程）────────────────────────────────┐
│                                                             │
│  for each (module_name, dirty_snap) in snapshots:           │
│      module = find_module(module_name);                     │
│      module.confirm_snapshot_written(&dirty_snap);          │
│                                                             │
│  刷新运行时 LMDB 读句柄（关闭旧→打开新）                     │
│  更新 world_meta.json（latest_save_path, latest_save_tick） │
│  更新 session.lock 时间戳                                   │
│  返回 SaveResult::Written                                   │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 错误恢复保证

- Phase 1 中 `snapshot_dirty` 失败 → `?` 传播，dirty 标记保留。下次存档重试
- Phase 1 中 `named_dbs()` 验证失败 → 返回错误。模块声明与实际写入不一致——模块 bug
- Phase 2 中写入失败 → 事务回滚，dirty 标记保留。下次存档重试
- Phase 3 中 confirm 失败 → 模块内部错误。dirty 标记可能残留（下次存档重新序列化）——数据不丢失

### 6.2 防重入 + 死亡打断

```rust
fn save(&self, world_dir: &Path, modules: &mut [&mut dyn SaveableModule],
        header: HeaderBuilder) -> Result<SaveResult> {

    // 死亡存档：打断进行中的低优先级存档
    if header.save_kind == SaveKind::Exit && header.save_status == SaveStatus::PendingInheritance {
        // 死亡 Exit 存档：等待当前存档完成（不拒绝）
        // 如果是 Auto 正在进行 → 等待完成 → 立即执行死亡存档
        while self.save_in_progress.load(Ordering::Acquire) {
            std::thread::yield_now();
        }
    } else {
        // 常规存档：防重入
        if self.save_in_progress.swap(true, Ordering::AcqRel) {
            return Err(SaveError::AlreadySaving);
        }
    }

    let _guard = SaveGuard { save_system: self };
    // ... 实际保存逻辑
}
```

### 6.3 请求队列——调度（非替换）

```rust
/// 请求队列：保留所有待处理的存档意图。
/// Manual > Death Exit > Quick > Auto 优先级出队。
/// 处理完高优先级后检查低优先级——如果低优先级仍待处理 → 立即补执行。
struct SaveQueue {
    pending: Mutex<SaveQueueState>,
}

struct SaveQueueState {
    has_manual: bool,
    has_death_exit: bool,
    has_quick: bool,
    has_auto: bool,
}

impl SaveQueue {
    fn push(&self, kind: SaveKind) {
        let mut state = self.pending.lock().unwrap();
        match kind {
            SaveKind::Manual => state.has_manual = true,
            SaveKind::Exit => state.has_death_exit = true,  // 死亡 Exit
            SaveKind::Quick => state.has_quick = true,
            SaveKind::Auto => state.has_auto = true,
            SaveKind::Initial => {},  // Initial 不走队列
        }
    }

    fn pop(&self) -> Option<SaveKind> {
        let mut state = self.pending.lock().unwrap();
        if state.has_manual {
            state.has_manual = false;
            return Some(SaveKind::Manual);
        }
        if state.has_death_exit {
            state.has_death_exit = false;
            return Some(SaveKind::Exit);
        }
        if state.has_quick {
            state.has_quick = false;
            return Some(SaveKind::Quick);
        }
        if state.has_auto {
            state.has_auto = false;
            return Some(SaveKind::Auto);
        }
        None
    }

    /// 检查是否还有待处理请求（一次存档完成后调用）
    fn has_pending(&self) -> bool {
        let state = self.pending.lock().unwrap();
        state.has_manual || state.has_death_exit || state.has_quick || state.has_auto
    }
}
```

保存循环：`while let Some(kind) = queue.pop() { execute_save(kind); }`——所有排队存档依次执行。`pending_save_count` 更新让 UI 显示 "正在保存... (2/3)"。

### 8.2 键前缀冲突检测（注册时）

```rust
impl World {
    /// 验证所有已注册模块的 (named_db, key_prefix) 组合不重叠。
    /// Debug 构建在 new() 中自动调用。Release 构建在 create_initial_save 前调用一次。
    fn verify_key_prefixes(&self) -> Result<()> {
        let modules = self.saveable_modules();
        let mut seen: HashMap<(&str, &str), &str> = HashMap::new();
        for m in &modules {
            for &(db, prefix) in m.named_dbs() {
                if let Some(existing_module) = seen.insert((db, prefix), m.module_name()) {
                    return Err(SaveError::KeyPrefixConflict {
                        db: db.to_string(),
                        prefix: prefix.to_string(),
                        module_a: existing_module.to_string(),
                        module_b: m.module_name().to_string(),
                    });
                }
            }
        }
        Ok(())
    }
}
```

## 九、关键设计问题回应

### Q: 为什么 snapshot 是 &self 而不是 &mut self

`snapshot_dirty(&self)` 是 `&self`（不可变借用）。序列化只需要**读**数据——`&self` 语义正确。

如果 snapshot 是 `&mut self`，它消费了脏标记。但如果后续 LMDB 写入失败——脏标记已消费，数据丢失。`&self` + `confirm_snapshot_written(&mut self)` 两阶段分离了"读取"和"确认清除"，保证写入失败时脏标记保留。

### Q: 为什么没有 flush_realtime_writes()

10 分钟自动存档间隔内最多丢失 ~25KB 数据（2 个历史事件 + 10-50 条大日志条目）。这个数据量在单机游戏的容错标准之内。玩家可以缩短自动存档间隔到 1 分钟。单独开辟实时写入路径增加架构复杂度，收益不大。

### Q: 为什么没有 checksum

LMDB 自身有页级 CRC 和 MVCC copy-on-write 保护。应用层 SHA-256 校验 2GB 文件需要 2-3 秒——不划算。文件识别靠 `magic` 字节 + Header 验证已足够。写入后验证遍历全部 named_db 检查完整性——比 checksum 更轻量且覆盖更多错误模式。

### Q: ConsumableEffect 跨模块迁移如何解决

采用**惰性迁移**。Schema 所有者提供 `convert_from_version()` 函数签名，数据存储方在 `load()` 中做批量版本转换——纳入加载画面预算（≤5s）。运行时零开销。零模块间协调。

### Q: Initial 存档如何处理 3GB 数据

Initial 存档不走三阶段 Phase 1 内存收集。使用 `write_initial()` 逐模块直接流式写入 LMDB 临时文件——数据在序列化后立即写入磁盘，不保留在内存中。
