## 一、本模块是什么

### 1.1 定位

**家具与放置物品系统**是[[../001-物品系统总纲|物品系统]]的子模块——处理"物品从背包中被拿出来、放在世界中的某个表面上"的全部语义。

物品系统定义了**东西是什么**（ItemDefId/ItemEntId/ItemProperties）。本模块定义了**东西被摆放在世界中之后的行为**——从一把椅子到桌上的烛台，从壁炉到雕塑，从地毯到野外神龛。

### 1.2 核心问题

本模块回答这些问题：

- 一把椅子在背包里（ItemEntId=0xE001）——把它放在地上后，它现在是什么？  
  **答**：仍然是同一个 ItemEntId=0xE001，但多了一个 `PlacedState`（位置、方向、支撑面）。provenance 链不中断——History 仍然知道"这是铁匠老王打造的那把椅子"。

- NPC 怎么知道面前的椅子可以坐？  
  **答**：`EntityIndex::affordances_of(id)` 返回 `AffordanceSet::CAN_SIT`——位运算 < 5ns。NPC 不需要区分"建筑组件椅"和"家具椅"。

- 桌上有个盘子，盘子下有桌子，桌子下有地毯，地毯下有地板——地板块被 BREAK 摧毁会怎样？  
  **答**：`SurfaceDestroyed` 事件 → 整条表面链在同一帧内级联坠落（while 循环收束）。

- 穷村和富城的家具为什么不同？  
  **答**：Pass B 消费 NPC 的 wealth_level + profession + CultureFurnishProfile → 参数化家具族的参数落在参数空间的不同区域。

- 魔法火球砸到餐桌上——餐桌有"木"元素吗？  
  **答**：ItemEntId → material_instance(MaterialDefId: OakWood) → elemental_affinity[木]=0.85, [火]=0.1。IGNITE 自动计算 flammable → 着火。

## 二、设计哲学

### 2.1 统一放置实体连续谱——不区分"建筑组件"和"家具实体"

WoWorld 中一切在世界上占据空间、能被感知与交互的人造物，共享同一套空间系统（`EntityIndex` trait）。差异不在类型上，而在**属性梯度**上：

```
承重墙 ──── 门/窗 ──── 固定长椅 ──── 可移动桌椅 ──── 装饰花瓶 ──── 桌上餐具
  │           │           │              │              │           │
Structural  PassThrough  Attachment    FreePlacement  FreePlacement FreePlacement
  网格吸附    网格吸附     子网格吸附     无网格          无网格        无网格
  有连接面    有连接面     无连接面      无连接面        无连接面      无连接面
  参与结构    不参与       不参与        不参与          不参与        不参与
  Graspability: Immovable  Attached    Free           Free         Free
```

**关键**: 物理原子层（`GRASP`/`SIT`/`BREAK`/`IGNITE`/`PUSH`）对 EntityId 操作——原子不知道"这是建筑组件还是家具"。`Graspability` 告诉原子"这个东西能拿吗？需要多大的力？"——这已足够。

### 2.2 一切皆物品，一切皆可交互

世界上每一个被摆放的实体都绑定一个 `ItemEntId`（→ `EntityId`）。玩家能用剑砍它、NPC 能坐上去、魔法能作用于它、History 能追踪它的 provenance。

**不存在的二分**: 不存在"组件椅"和"物品椅"——只有一把椅子，它可被捡起来、搬走、砸碎、再放回地上。

### 2.3 参数化家族——不穷举枚举

~9 个家具族 × 平均 3 个 mesh 模板 × 连续参数（材料/文化/品质） × morph target = ~12,000+ 视觉变体。新增文化风格只需调整参数映射表（TOML），零 Rust 代码改动。

**这和武器装配框架是同一种哲学**——5 种柄 × 8 种刃 × 4 种护手 × 3 种配重 = 480 种变体，不需要手工建模 480 把剑。

### 2.4 双 Pass 生成 + 种子确定性

- **Pass A (P11.5a)**: 附着家具——壁炉、嵌墙架——消费建筑模块的 `ConnectionFace.usage_tag`
- **Pass B (P11.5b)**: 自由家具——消费 `CultureFurnishProfile` + NPC 属性 + 经济水平 + 本地材料
- **Pass C (P11.5c)**: 野外放置实体——神龛/摊位/路标——不属于任何建筑

所有家具**全量预生成**（种子确定性），写入 LMDB。NPC 从 day 1 就知道"我的床在 Region=卧室, 坐标=(12.3, 0, 5.7)"——不需要等玩家走近才第一次确定。**渐进的是渲染、物理 LOD 和 chunk 加载，不是家具的存在性。**

### 2.5 物品系统的基础设施定位——不定义玩法

物品系统总纲的原则完全适用于本模块：

| 本模块**定义** | 本模块**不定义** |
|--------------|---------------|
| 东西怎么被摆放到世界上（place/pick_up） | NPC 为什么选择这把椅子（GOAP/Aesthetic） |
| 摆放后的物理供给标记（AffordanceSet） | NPC 坐在椅子上之后的具体行为序列（NPC 行动系统） |
| 家具的参数空间和变体选择机制 | 文化为什么偏好某一种家具风格（文化系统） |
| 表面依赖链（桌子→盘子） | 魔法如何改变家具材质（魔法系统——通过 EntityIndex 回调） |

## 三、核心概念

### 3.1 PlacedItem — 物品在世界中的放置状态

一件物品（ItemEntId）可以是：
- **在库存中**（NPC/玩家的 Inventory）
- **被握持中**（GripHandle，由 `GRASP` 原子管理）
- **被摆放在世界中**（PlacedState，由 `PlacementStore` 管理）
- **掉落在地上**（DroppedItem——松散物理刚体，无放置语义）
- **被摧毁**（ItemEntId 保留——History 仍可引用——但物理实例不存在）

物品在库存和摆放之间切换时，ItemEntId **不变**。`place()` 把 ItemEntId 从 Inventory 移到 PlacementStore。`pick_up()` 把 ItemEntId 从 PlacementStore 移回 Inventory。

### 3.2 PlacementStore — 物品系统内部的空间存储

`PlacementStore` 是物品系统拥有的数据结构，存储所有被摆放物品的空间数据。它实现 `EntityIndex` trait（woworld_core）——物理原子层的 35 个原子通过 `&dyn EntityIndex` 查询它，不知道也不关心 PlacementStore 的存在。

**外部模块永远不直接碰 PlacementStore。** 它们只通过 `EntityIndex` trait 或 `SurfaceProvider` trait 消费。

### 3.3 表面链 (Surface Chain)

一件物品被放在一个表面上。这个表面可能是：

```
建筑组件(地板/墙面) → PlacedItem(桌子) → PlacedItem(盘子) → PlacedItem(食物)
  BuildingComponent       家具表面            家具表面          家具表面
```

`SurfaceId`（woworld_core）用 4 个变体覆盖所有表面类型：Building / Terrain / Feature（大树干/巨石）/ Furniture（已放置物品的顶面）。

当底层表面被摧毁时，`SurfaceDestroyed` 事件触发同一帧内的 `while` 循环——所有 `surface_ref` 匹配的物品级联坠落。

### 3.4 AffordanceSet — 物理供给位集

`AffordanceSet: u64` 是一个 64-bit 位集，存储物品的物理能力标记。NPC 行动系统用它快速过滤——"周围有什么东西可以坐？" = `entities.filter(|id| entity_index.affordances_of(id) & CAN_SIT != 0)`。位运算，< 5ns。

**语义标记（IS_SACRED、IS_CURSED）不放在 AffordanceSet 中**——它们走 `ItemProperties.tags`（哈希查表），因为它们是低频查询（观察/鉴定时），不应污染热路径。

### 3.5 家具族 (FurnitureFamily)

9 个参数化家具族，每个族 ~3 个 mesh 模板 + 连续参数 + morph target：

> 详细见 [[012-家具族参数化定义|012-参数化定义]]

| 族 | 核心参数 | 模板数 |
|---|---------|-------|
| 座椅族 | seat_height, backrest_angle, leg_style, armrest_bool... | 4 |
| 桌台族 | table_height, surface_area, leg_count, edge_profile... | 3 |
| 卧具族 | bed_length, headboard_style, mattress_thickness... | 3 |
| 柜族 | height, door_count, drawer_rows, leg_style... | 3 |
| 灯具族 | height, light_radius, fuel_type, shade_style... | 3 |
| 地毯族 | length, width, pattern_complexity, fringe_style... | 2 |
| 壁炉族 | width, height, mantel_style, hearth_style, flue_type... | 3 |
| 华盖族 | width, depth, pole_count, fabric_type, opacity... | 2 |
| 装饰族 | size, shape, attachment_type, ornament_density... | 2 |
| 雕塑族 | 底座(参数化) + 主体(标记化目录 ~66 mesh) | (特殊) |

### 3.6 雕塑的特殊处理

雕塑不能通过 morph target 覆盖——"站立武将"和"抽象球体"在拓扑上是完全不同的。因此雕塑采用**底座参数化 + 主体标记化目录**的两层架构：

> 详细见 [[012-家具族参数化定义|012 §十 雕塑族]]

底座走参数化族（同桌台族）。主体走预生成 mesh 目录（~66 个 AI 生成 mesh），通过 `SubjectTag` 标记选择——`FaithSpecific(war_god)` → 战神雕像候选。材质 × 底座 × 涂装 × 风化 = ~66,000 种视觉组合。

## 四、数据流全览

### 4.1 世界生成时的家具创建

```
P2.5: CultureCoreParams 生成
  → CultureFurnishProfile::derive(params, race, climate)
  → 存储于文化系统，通过 trait 查询

P6: 建筑结构生成
  → BuildingGenerator 产出 Building + ConnectionFace.usage_tag
  → 烟道组件（Chimney）在 WFC 中解析

P8: 人口投影与家族协同
  → NPC 的 profession + wealth_level + aesthetic_taste 确定

P11 (经济 Bootstrap): 
  → 供需聚合 → 订单簿 → 钱包 → EconomicRelation
  → NPC 经济水平已确定

P11.5a (Pass A — 附着家具):
  胶水代码遍历所有建筑的 Attachment 连接面
  → 检查 ConnectionFace.usage_tag
  → 壁炉口 → 分配 ItemEntId(壁炉) → Place(fireplace, chimney_opening)
  → 嵌墙书架口 → 分配 ItemEntId(书架) → Place(shelf, wall_attachment)
  → 全部种子确定性 ID

P11.5b (Pass B — 自由家具):
  遍历所有建筑的 Region
  → 输入: Region 功能 + NPC 属性 + CultureFurnishProfile + 本地材料
  → FurnishEnsemble::roll(room) → 掷一次骰子确定房间审美
  → 家具清单推导 → 按锚点→关系→表面链→散落分层放置
  → 七步验证每个放置
  → PlacementStore 写入

P11.5c (Pass C — 野外放置):
  市场摊位/路边神龛/矿井设备/猎人晾肉架
  → BuildingId=None 的 ComponentInstance（有 item_id）
  → 或独立的 PlacedEntry（PlacementStore）

P13 (终组):
  家具完整性校验——锚点物品（床）是否成功放置
  → 未放置 = 生成错误 → log warning
  → ✓ WorldState → LMDB
```

### 4.2 运行时的家具交互

```
NPC 进入房间:
  → OBSERVE → PerceptBatch (包含 EntityId 列表)
  → EntityIndex::affordances_of() 位过滤
  → GOAP 规划器匹配 ActionCandidate
  → 找到可坐的椅子 → 执行 POSTURE(Sit, chair_entity_id)

NPC 打铁:
  → 找到 WorkstationTag=Forge 的 EntityId
  → 验证周围有 Anvil → 验证燃料 → 开始 CRAFT(锻造) 复合原子

玩家/NPC 造了一把新椅子:
  → CRAFT 消耗材料(3×Oak Wood, MaterialDefId: OakWood)
  → 产出 ItemEntId(新椅子), material_instance=OakWood
  → FurnishParams 从 creator 的文化 Profile + 噪声 + 技能水平派生

玩家把椅子从背包放到地上:
  → validate_furniture_placement() 七步验证
  → PlacementStore::place() → 写入 PlacedEntry
  → 发布 WorldEvent::ItemPlaced

战斗中火球击中桌子:
  → IGNITE(EntityId) → MaterialProperties.combustibility → 着火
  → 桌子 health 归零 → BREAK → 碎片 DroppedItem
  → SurfaceDestroyed(桌面) → 桌面上的盘子级联坠落

玩家考古:
  → 远古精灵废墟 → PlacedEntry 存在但 quality=0.1, is_broken=true
  → History 查询 ItemEntId provenance
  → "这把残破的雕花木椅——曾是第三纪元精灵王庭的座椅。"
```

## 六、关键问题回应

### Q: "FurnitureChest"(0x44) 和 "FurnitureItem"(0x54) 是什么关系？

**A**: 0x44 废弃。所有家具统一在 FurnitureItem(0x54) 下，子类别 0x54_02 = 存储类（Chest/Cabinet/Wardrobe/Barrel/Bookshelf）。`FurnitureChest` 作为废弃标签保留（便于旧存档迁移），不再新建此类型的物品。

### Q: NPC 怎么知道灶台可以烹饪？

**A**: 不是通过 enum 匹配——是通过 `AffordanceSet::IS_WORKSTATION | IS_HEAT_SOURCE`（第一层，位运算）→ `workstation_tag_of(id)` 返回 u16（第二层，查表）→ 烹饪领域模块检查 `compatible_workstation_tags` 是否包含 `CookingStove` 的注册 tag。

### Q: 桌子推开后上面的盘子怎么办？

**A**: `SurfaceEvent::Moved` 发布 → PlacementStore 反向查找所有 `surface_ref` 匹配的 PlacedItem → 重新运行放置验证（至少 Step 2 坡度容差）。如果盘子不能满足新坡度 → 滑落 → 成为 DroppedItem。

### Q: 一块矿石在铁匠铺地上——是 PlacedItem 还是 DroppedItem？

**A**: 松散堆叠的材料（黏土堆/矿堆/木料堆）走 PlacedItem，`stackable=true`。掉落物（被丢弃/被炸飞）走 DroppedItem。区别在于：PlacedItem 有确定的表面锚定（它被"放在"那里），DroppedItem 是纯物理刚体。

## 七、性能核算

| 项 | 值 | 备注 |
|---|-----|------|
| PlacedEntry 热数据 | 32 bytes | Copy type, L1 cache 友好——一个 cache line 装两个 |
| PlacedCold | 32 bytes | 低频查询——不在渲染路径上 |
| 序列化 / 件 | 48 bytes | bincode packed |
| 全图 2M 物品 LMDB | 96 MB | 在 140MB 存档预算内 |
| LOD0 运行时 (玩家周围 ~10 chunks) | ~30 KB 热数据 | ~650 PlacedEntry |
| entities_in_radius(30m) | < 1μs | 3 chunks × ~65 items 遍历 |
| place() 全套验证 (含 A*) | < 150μs | 运行时——NPC 搬一件家具 |
| place() 生成时 (走廊预留法) | < 50μs | 世界生成——简化版无 A* |
| pick_up() + 子物品级联坠落 | < 500μs | while 循环快速收束 |
| affordances_of() | < 5ns | 位运算——无哈希、无分支 |
| Chunk 加载 (反序列化 ~65 items) | < 20μs | bincode |
| 雕塑目录查询 | < 100ns | HashMap<SubjectTag, Vec<MeshKey>> |
