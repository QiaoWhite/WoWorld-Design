## 一、模块定位与边界

### 1.1 历史系统是什么

WoWorld 历史系统是**世界级事件记录、因果生成、痕迹物化、玩家知识管理的统一架构**。它负责回答三个根本问题：

1. **世界过去发生了什么**——从种子确定性生成的世界历史因果网络，以及玩家进入后新事件的实时记录
2. **历史怎么呈现在世界中**——通过废墟/文物/书籍/碑文/灵元素印记/关系遗产等物理化管道，让玩家在正常游戏过程中感知历史
3. **玩家怎么管理所知**——大日志作为纯功能性外置大脑，全量记录玩家的见闻，支持搜索、关联、渐进验证、纠错

### 1.2 历史系统不是什么

- ❌ 不是预设剧情——开发者不写任何一条事件的具体叙事
- ❌ 不是可读时间线——玩家正常游戏中不能直接查看 event_log
- ❌ 不是任务系统——没有"历史任务"，没有必须发现的历史内容
- ❌ 不是 NPC 记忆系统——NPC 个体记忆归 NPC 活人感模块管理
- ❌ 不是口头传说系统——信息在 NPC 之间的口头传播归信息传播系统管理
- ❌ 不是世界生成的主体——历史模拟是消费世界生成产出的世界状态来运行

### 1.3 模块边界：历史系统拥有什么、不拥有什么

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
                 历史系统与其他模块的边界
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

历史系统拥有:
  ├── event_log（世界事件日志——世界的 ground_truth）
  ├── 事件因果链（趋势层→力量层→事件层的三层涌现模型）
  ├── 命运种子（基于世界状态的结构触发器——v1.1 激活）
  ├── 运行时事件触发器（冲击型+累积型+熵型）
  ├── 生命痕迹（墓碑/日记/刻字/遗物/契约/供品/秘藏）
  ├── 团体档案（家族/商会/佣兵团半形式化记录——v1.2+）
  ├── 书籍与著作（Work→PhysicalBook 两层模型）
  ├── 灵元素印记（AetherImprint——空间中的历史残留）
  ├── 文物痕迹（魔法物品的历史包装层——v1.1）
  ├── 关系遗产（世仇/联盟/债务/附庸的因果起源——v1.2+）
  ├── 历史嵌入世界管道（废墟/建筑老化/文物/书本/读树/关系遗产）
  ├── 大日志（玩家的外置大脑工具）
  └── 外部接口（与信息传播/植物/魔法/NPC 的 trait 接口）

历史系统不拥有：
  ├── NPC 个体记忆——归 NPC 活人感模块（但历史事件是记忆的素材来源）
  ├── NPC 之间如何口头传播信息——归信息传播系统（但历史提供传播起点的数据）
  ├── 建筑本身——归世界生成模块（但建筑的老化/废弃状态由历史事件修改）
  ├── 魔法物品的魔法属性——归魔法模块（但物品的历史包装层归历史）
  ├── 植物的灵元素亲和力和读树交互——归植物模块（但印记归历史）
  └── 玩家的物理库存——归物品系统（但大日志作为特殊物品存在）
```

### 1.4 与世界生成 008 的关系

`世界生成/008-NPC初始化与历史模拟.md` 中 §二"历史模拟（P9）"是历史系统的**前身**——定义了世界生成最后一阶段的历史快进模拟。历史系统本规格**基于并大幅扩展**那个基础设计。

具体变化：

| 世界生成 008 P9 | 历史系统 v1.0 |
|----------------|-------------|
| 历史模拟作为世界生成的子阶段 | 独立为完整模块，世界生成 P9 成为调用点 |
| 因果链未定义 | 完整的趋势→力量→事件三层涌现因果图 |
| event_log 概念存在但数据结构简单 | 完整的因果边图 + 参与者索引 + 痕迹生成 |
| 无命运种子 | 完整的结构触发器机制（v1.1 激活） |
| 无运行时事件记录 | 冲击型触发器（v1.0）+ 累积型（v1.1） |
| 历史事件类型 6 类简单表 | 完整的因果模板 + 力量驱动 + 趋势调制 |

## 二、核心概念与分层架构

### 2.1 两种历史：ground_truth vs 可感知历史

WoWorld 中"历史"有两个层面，绝不可混淆：

```
┌──────────────────────────────────────────────────────────┐
│                                                            │
│   event_log (ground_truth)                                │
│   ───────────────────────────                              │
│   系统后台记录。世界的完整真相。                             │
│   · 确定性因果链                                           │
│   · 所有事件的完整参数                                     │
│   · 玩家正常游戏中不可查看                                  │
│   · 只在控制台/解包存档时可访问                             │
│                                                            │
│   ──────────── "历史嵌入管道" 转译 ────────────             │
│   (每条管道都是一个不完美的窗口，看到的是碎片和变体)          │
│                                                            │
│   可感知历史 (Perceivable History)                          │
│   ──────────────────────────────────                       │
│   玩家通过正常游戏方式接触到的历史：                          │
│   ├── 废墟（进入废弃的城市 → 感觉到"这里出过事"）            │
│   ├── 文物（捡到一把古剑 → 追溯它的来历）                    │
│   ├── 书本（读编年史 → 得知某个事件的记载）                  │
│   ├── 读树（触碰老树 → 感受灵元素印记的意象碎片）             │
│   ├── 关系遗产（发现两个家族有世仇 → 探究原因）               │
│   └── NPC 对话（听铁匠提起往事 → 一个视角下的片段）           │
│                                                            │
└──────────────────────────────────────────────────────────┘
```

### 2.2 两层历史：预生成 vs 运行时

| | 预生成历史 | 运行时历史 |
|------|-----------|-----------|
| **时间范围** | 玩家进入前（0-2000 年，种子决定） | 玩家进入后（游戏运行时） |
| **生成方式** | 世界生成 P9 批量模拟，种子确定性 | 月 tick 增量触发 |
| **玩家能改变吗** | 不能——既定事实（像新生儿面对已有历史） | 能——玩家是事件的参与者/目击者/受害者 |
| **事件数量** | ~30K-40K | 每年 ~3-8 条（速率可控） |
| **因果来源** | 根事件→力量层→因果传播 | 冲击型触发器 + 累积型触发器 + 命运种子 |
| **如何呈现** | NPC 传说/书本/文物/废墟 | 玩家亲身经历 → 大日志自动记录 |

### 2.3 历史时间尺度

```
玩家视角下的时间尺度:
  
  新生历史 (< 1年)
    → 玩家亲身经历。大日志直接记录。
    → 其他 NPC 也在传播（走信息传播系统的新鲜新闻通道）
  
  在世记忆 (1-80年)
    → 老年 NPC 的第一手/第二手记忆。对话可获得。
    → 可能有活着的当事人。
  
  口头传统 (80-500年)
    → NPC 传说、民歌、俗语。开始变形。
    → 当事人的孙辈可能还记得一些片段。
  
  深层历史 (500年+)
    → 书本、遗迹、老树读树、碑文。
    → 已经没有在世记忆，只有物理残留。
  
  彻底遗失 (任意深度，随机)
    → is_fully_lost = true。没有任何痕迹抵达"现在"。
    → 系统知道，但没有任何 NPC 知道、没有遗迹、没有记录。
    → 这是历史感的重要来源：知道有东西永远不知道了。
```

### 2.4 历史事件的定义

**什么进入 event_log：**

```rust
/// 事件的世界级影响评分
fn impact_score(event: &WorldStateDelta) -> f32 {
    let population_affected = affected_population_ratio(event);   // 0.0-1.0
    let duration_depth = duration_impact(event);                  // 0.0-1.0
    let structural_change = structural_change_degree(event);      // 0.0-1.0
    
    population_affected * 0.4 + duration_depth * 0.25 + structural_change * 0.35
}

/// 阈值规则:
///   impact_score > 0.7  → 全球级事件，无条件记录
///   impact_score 0.4-0.7 → 区域级事件，固定阈值 + 同区域冷却
///   impact_score 0.2-0.4 → 不进 event_log，走 NPC 个体记忆
///   impact_score < 0.2   → 不入库
```

**什么不进 event_log：**
- 单个 NPC 的日常行为（吃了什么、和谁聊天）
- 小规模的人际关系变化（两人吵架）
- 非结构性的经济波动（一天的面包涨价）
- 绝大多数个体的死亡（除非死于重大事件——死于战争的士兵入 event_log 的人数统计，个人死亡不进 event_log 的单条条目）

## 三、历史与玩家的关系

### 3.1 玩家改变不了过去——但玩家创造未来

```
玩家进入世界:
  
  过去（已定）:          现在（玩家进入）:         未来（玩家参与）:
  ─────────────          ───────────────         ───────────────
  预生成的 30K 事件      世界处于一个特定状态      玩家的行为创造新事件
  因果链已固定            NPC 带着历史记忆活着      新事件写入 event_log
  废墟已经在那儿          书本已经在图书馆里        玩家可能成为关键参与者
  老树已经吸收了印记     关系遗产已经存在          玩家的行为被大日志记录
  
  玩家能做的:
  ├── 探索过去留下的痕迹——但不能改变过去
  ├── 用自己的眼睛和耳朵感知历史碎片——拼凑出自己的理解
  ├── 通过与 NPC 交谈、读书、读树、探废墟——主动追寻历史
  └── 自己的选择和行为——创造这个世界的新的历史事件
```

### 3.2 玩家不是上帝——信息获取与其他 NPC 平等

**关键原则：玩家获取历史信息的方式，和世界上任何一个 NPC 获取信息的方式——完全相同。**

- NPC 能读书 → 玩家也能读书（如果识字、如果找得到书）
- NPC 能和别人交谈获取信息 → 玩家也是
- NPC 能读树感知灵元素印记 → 玩家也是（如果有灵元素亲和力）
- NPC 不知道 event_log → 玩家也不知道 event_log

唯一的例外：**大日志。** NPC 的个体记忆有容量上限（2000 条）、有遗忘曲线、有记忆重构。但玩家的大日志是完美记忆——不会遗忘、不会失真。这是玩家作为"玩家"的唯一超能力：完美外置记忆。仅此而已。

## 四、event_log 总览

### 4.1 event_log 的定位

event_log 是历史系统的**核心数据库**。它是世界全部"重大发生过的事"的唯一真相源。

```rust
struct EventLog {
    /// 事件图——所有历史事件及其因果连接
    graph: HistoryGraph,
    
    /// 按空间位置索引（用于历史嵌入管道的查询）
    spatial_index: BTreeMap<SpatialKey, Vec<EventId>>,
    
    /// 按参与者索引（哪个 NPC/政权/团体参与了哪些事件）
    participant_index: HashMap<ParticipantId, Vec<EventId>>,
    
    /// 按时间索引
    temporal_index: BTreeMap<i32, Vec<EventId>>,   // timestamp → events
    
    /// 全局根事件列表（种子决定的因果链起点）
    root_events: Vec<EventId>,
    
    /// 命运种子（v1.0 占位播种，v1.1 激活触发检查）
    fate_seeds: Vec<FateSeed>,
    
    /// 运行时事件触发器配置
    trigger_configs: Vec<HistoryTrigger>,
    
    /// 元数据
    metadata: EventLogMetadata,
}

struct EventLogMetadata {
    seed: u64,
    pre_generated_until: i32,      // 预生成覆盖到哪个时间戳
    total_events: u32,
    generation_version: String,    // 生成此 event_log 的历史模块版本
}
```

### 4.2 event_log 的写入规则

```
写入权限:
  ├── 世界生成 P9 历史模拟（批量写入）—— pre_generated_until 之前的时间戳
  ├── 运行时事件触发器（增量写入）—— pre_generated_until 之后的时间戳
  └── 命运种子触发（增量写入）—— 进入游戏后的任何时间

写入约束:
  ├── 时间戳 < pre_generated_until 的事件不可被修改（过去已定）
  ├── 时间戳 > pre_generated_until 的事件可写入但不可删除
  ├── 同区域冲击型事件有冷却（同类事件 2-5 年不重复）
  └── 总事件数超过 50K 硬上限时——停止生成，不是覆盖（可通过种子调节根事件数来控制）

读取权限:
  ├── 历史系统内部: 完全读取
  ├── 其他游戏模块: 通过 trait 接口读取（不直接访问 graph）
  ├── 大日志: 不读取 event_log（只通过正常游戏信息渠道获取 → 传到大日志）
  └── 玩家控制台: 可读取（开发调试用，非正常游戏）
```

### 4.3 event_log 与信息传播系统的区别

一个常被混淆的点：

| | event_log | 信息传播系统 |
|------|-----------|------------|
| **存什么** | 事件的完整真相 | 事件信息在 NPC 之间的传播链路和变体 |
| **谁访问** | 系统后端 | NPC 社交/对话系统 |
| **有失真吗** | 无——它就是真相 | 有——每次传播可能变形（失真算子） |
| **事件来源** | 历史模块拥有 | 历史模块提供 `PropagableEventSource` 接口 |
| **传播追踪** | 不追踪——不知道谁知道了这件事 | 追踪——谁传给了谁，当前版本的内容是什么 |

## 五、历史系统的质量原则

### 5.1 涌现优先于预写

```
✅ 涌现: 领土扩张力(冰冠, 七河) 0.78 → 概率触发战争事件 → 事件参数由世界状态确定
✅ 涌现: NPC_A 死亡 → 家属概率性立碑 → 碑文内容来自 NPC_A 的真实属性
✅ 涌现: 两个家族 3 代冲突 → 世仇力被检测到 → 下次冲突概率提升

❌ 预写: "冰冠和七河之间有一场战争，原因是..."
❌ 预写: "王铁牛墓碑上刻着：爱妻悼念..."
❌ 预写: "铁炉峡谷的传说：从前有一条龙..."
```

### 5.2 历史不是灌输——是拼图

玩家在 WoWorld 中学习历史的方式不是一个"历史概述"界面。而是：
- 在废墟中看到一个破损的家族纹章
- 在 300 公里外的另一座城发现了同一个纹章
- 在两座城的编年史中读到不同的解释
- 自己把碎片拼成一条因果链

**系统提供碎片——玩家自己拼。这就是故事生成器的含义。**

### 5.3 完整但不可尽知

```
世界的全部真相:          30,000+ 事件，完整的因果图
玩家能直接感知到的:       ~500-2000 线索（取决于探索深度和时长）
玩家能追溯到的深渊:       很多事件没有任何痕迹留下
                     （is_fully_lost = true）
                     
也就是说:
  · 世界确实有厚重的、完整的、复杂的过去
  · 但没有任何人（包括开发者）能预测玩家会发现哪些碎片
  · 玩家可能永远无法知道某些真相——这是历史感的核心来源
  · 不是一切都要被"完成任务"或"集齐成就"穷尽
```

### 5.4 事件生成质量——模版基础上的涌现

因果关系的可能性由预定义的因果模板约束——但**哪个模板在何时触发、作用于哪个实体、产生多强的效果，全部由世界状态的数值运算决定**。模板定义了"战争可能导致边境变更"这个因果类型，但不会定义"具体哪两个政权、在哪块领土、什么时间、为什么原因"——这些都是涌现的。

```
模板:  TerritorialWar → BorderShift(p=0.8)
涌现:  冰冠王国对七河联盟的边境变更 —— 因为力量检测到河谷地区的领土扩张力 0.78
       触发时间: 力量累积超过阈值 0.75 的那一 tick
       变更领土: forces[冰冠→七河].over = 河谷地区
       结果:    由双方军事力量比 + 随机决定（胜/败/平/谈和）
```

## 六、与玩家进入世界的配合

### 6.1 新游戏创建流程（历史模块视角）

```
玩家点"新游戏"
  → 种子生成
  → 世界生成 P0-P8（地形/聚落/建筑/NPC 初始化）
  → 世界生成 P9: 调用历史系统的预生成引擎
    ├── 1. 种子→世界参数约束→选根事件（5-20 个）
    ├── 2. 初始化趋势层（从世界参数计算基础趋势值）
    ├── 3. 初始化力量层（从世界状态扫描）
    ├── 4. 根事件递归因果传播（tick 1 到 N 个月）
    ├── 5. 每个 tick: 趋势更新 → 力量检测 → 力量阈值检查 → 触发事件 → 因果传播
    ├── 6. 事件同步写入世界状态（边界变更/人口变化/建筑损毁）
    ├── 7. 为 intensity > 0.5 的事件生成 AetherImprint
    ├── 8. 为预生成的 NPC 死亡/情绪峰值生成 LifeTrace
    ├── 9. 为写作职业 NPC 产出书籍骨架
    └── 10. 播种命运种子（v1.0 占位，v1.1 激活）
  → 世界生成 P10（世界边界）
  → 玩家降生。初始时间 = 预生成的"现在"
  → 大日志初始化为空（玩家还没有任何见闻）
  → 运行时事件触发器开始运行（每月 tick）
```

### 6.2 玩家进入后的持续运转

```
每个游戏月 tick（独立后台线程）:
  ├── 趋势层更新（人口/资源/文化参数慢变）
  ├── 力量层检测（扫描世界状态的张力）
  ├── 冲击型触发器检查（世界状态增量 → impact_score → 阈值判定）
  ├── (> v1.1) 命运种子触发条件检查
  └── (> v1.1) 累积型触发器检查

每触发一个新事件:
  ├── 写入 event_log
  ├── 更新世界状态（边界/人口/建筑）
  ├── 写入 AetherImprint（如果 intensity > 0.5）
  └── 触发 LifeTrace 生成（如果涉及 NPC 死亡/情绪峰值）
```
