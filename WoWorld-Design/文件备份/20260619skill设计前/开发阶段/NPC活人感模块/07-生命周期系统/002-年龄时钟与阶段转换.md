## 一、AgeClock：纯函数设计

### 1.1 核心原则

AgeClock 是一个**无状态的纯函数**：
- 同输入 + 同 RNG 种子 → 同输出
- 不感知 LOD 层——L1 每帧调用和 L4 年批调用产生相同最终状态
- 不感知"加速"——接收 delta_days，不关心这个 delta 是"真实的"还是"加速的"（虽然我们不加速）
- 不感知调用者——不关心是谁在调

```rust
// —— woworld_life ——
pub struct AgeClock;

impl AgeClock {
    /// 推进实体的生物学年龄
    ///
    /// # 参数
    /// - entity: 当前生命实体状态
    /// - delta_days: 推进的天数（可任意大小——1 帧到 365 天均可）
    /// - rng: 确定性随机数生成器
    ///
    /// # 返回
    /// - 更新后的 LifeEntity
    /// - 在此期间发生的所有 LifeEvent（按时间顺序）
    ///
    /// # 纯函数保证
    /// advance_age(e, 30, rng) 的效果 =
    ///   advance_age(advance_age(e, 15, rng1), 15, rng2)
    ///   其中 rng1 和 rng2 是 rng 的两次连续 split
    pub fn advance_age(
        entity: &LifeEntity,
        delta_days: f32,
        rng: &mut impl RngCore,
    ) -> (LifeEntity, Vec<LifeEvent>);
}
```

### 1.2 内部算法

```rust
fn advance_age(entity: &LifeEntity, delta_days: f32, rng: &mut Rng) 
    -> (LifeEntity, Vec<LifeEvent>) 
{
    let mut events = Vec::new();
    let mut state = entity.clone();
    let mut remaining = delta_days;

    // 步骤 1: 推进年龄并检测阶段转换
    while remaining > 0.0 {
        // 计算到下一个阶段边界的剩余天数
        let next_boundary_days = days_until_next_boundary(&state);
        let step = remaining.min(next_boundary_days);
        remaining -= step;
        state.age_years += step / DAYS_PER_YEAR;

        // 检查是否跨过阶段边界
        if at_life_stage_boundary(&state) {
            let old_stage = state.life_stage;
            state.life_stage = LifeStage::from_age_pct(
                state.age_years / state.max_lifespan,
                &state.race.life_stage_distribution,
            );
            events.push(LifeEvent::StageTransition {
                entity: state.id,
                from: old_stage,
                to: state.life_stage,
                at_age: state.age_years,
            });
        }
    }

    // 步骤 2: 衰老死亡检查（Gompertz 闭式积分，详见 007）
    if state.age_years / state.max_lifespan >= 0.7 {
        let survival = senescence_survival(
            entity.age_years / entity.max_lifespan,
            delta_days / DAYS_PER_YEAR / entity.max_lifespan,
            entity.physical.constitution,
            entity.health_history,
        );
        if rng.gen::<f32>() > survival {
            let cause = sample_senescence_cause(
                entity.age_years / entity.max_lifespan, rng
            );
            events.push(LifeEvent::Death {
                entity: state.id,
                cause,
                location: state.position,
                spirit_remaining: state.vitals.spirit,
            });
            state.is_dead = true;
            state.death_cause = Some(cause);
        }
    }

    // 步骤 3: 推进妊娠（若有）
    if let Some(ref mut gestation) = state.gestation_state {
        advance_gestation(&mut state, &mut events, delta_days, rng);
    }

    // 步骤 4: L3/L4 婴儿存活（若有 InfantDependency）
    if let Some(ref infant_dep) = state.infant_dependency {
        infant_l3_l4_survival_check(&mut state, &mut events, delta_days, rng);
    }

    (state, events)
}
```

## 二、阶段边界预计算——性能优化

### 2.1 预计算边界的理由

LifeStage 的边界由 `RaceTraits.life_stage_distribution` 定义——一个 `[f32; 6]` 数组，表示 6 个阶段的百分比边界。NPC 创建时一次性计算为绝对年龄数组，之后 advance_age 只需要比较 `age_years` 是否越过这些预计算值。

```rust
/// NPC 创建时一次性计算
struct LifeStageBoundaries {
    /// 6 个边界的绝对年龄，按升序排列
    /// 例如人类 [1.5, 7.5, 17.5, 28.0, 49.0, 70.0]
    boundaries_absolute: [f32; 6],
}

impl LifeStageBoundaries {
    fn from_race(race: &RaceTraits) -> Self {
        let max = race.lifespan_range.1;  // max_lifespan
        let dist = &race.life_stage_distribution.boundaries;
        // dist 存储为 [0.10, 0.25, 0.40, 0.70, 0.85, 1.00] 格式
        // 即每个阶段结束时的百分比
        LifeStageBoundaries {
            boundaries_absolute: dist.map(|pct| pct * max),
        }
    }
    
    /// O(1) 查找当前阶段——最坏情况 6 次浮点比较
    fn current_stage(&self, age: f32) -> LifeStage {
        for (i, &boundary) in self.boundaries_absolute.iter().enumerate() {
            if age < boundary {
                return LifeStage::from_index(i);
            }
        }
        LifeStage::Elder  // 超过 max_lifespan
    }
}
```

### 2.2 days_until_next_boundary

```rust
fn days_until_next_boundary(entity: &LifeEntity) -> f32 {
    let boundaries = &entity.life_stage_boundaries;
    let age_days = entity.age_years * DAYS_PER_YEAR;
    
    // 找到第一个大于当前年龄的边界
    for &boundary_abs in &boundaries.boundaries_absolute {
        let boundary_days = boundary_abs * DAYS_PER_YEAR;
        if boundary_days > age_days {
            return boundary_days - age_days;
        }
    }
    
    // 超过所有边界——返回到下一个检查点的天数
    // 使用 max_lifespan * 0.05 作为检查间隔
    entity.max_lifespan * 0.05 * DAYS_PER_YEAR
}
```

## 三、LifeStageTransition 事件协议

### 3.1 事件定义

```rust
/// 生命阶段转换事件——纯中性事实
///
/// 此事件不携带：
/// - 情感建议（"这是值得庆祝的"）
/// - 仪式触发（"应该举办成人礼"）
/// - 能力解锁（"现在可以使用技能 X 了"）
///
/// 每个消费方独立决定此事件对自己的意义。
pub struct LifeStageTransition {
    pub entity_id: EntityId,
    pub from: LifeStage,
    pub to: LifeStage,
    pub at_age: f32,
    pub game_time: GameDay,
}
```

### 3.2 各模块的独立响应

| 模块 | Transition | 响应 |
|------|-----------|------|
| NPC 生命周期 | 任意 | 更新此 NPC 的行为门控参数（渐进过渡，非开关） |
| 模型/动画 | Juvenile→Adolescent | 骨骼缩放因子开始插值（从儿童比例向成人比例过渡） |
| 模型/动画 | MiddleAge→Elder | 脊椎前倾角 + 轻微缩水 |
| 认知 | Adolescent→YoungAdult | CognitiveStyle 从流动态开始稳定化 |
| 审美 | Adolescent→YoungAdult | AestheticTaste::derive_taste() 结晶化 |
| 信仰 | Adolescent→YoungAdult | ChildFaithProfile 中 motivation 从 Habitual→可转变 |
| 文化 | Adolescent→YoungAdult | 若文化有成年礼 RitualDef→调度执行 |
| 权力 | 任意 | 若实体持有头衔且阶段转换影响执政能力→合法性重算 |
| 历史 | 任意 | 可选：记录为 HistoricalEvent（重大人物的阶段转换有历史意义） |

### 3.3 过渡是渐进的，不是瞬间的

阶段转换事件发布了一个**离散的标识**，但各模块的响应是**渐进的**：

```
模型缩放不在一帧内从 0.6→1.0——在 Adolescent 的整个区间内线性插值
学习率不在 "Adolescent→YoungAdult" 瞬间降低——在 YoungAdult 区间内
  从 adolescent_learning_rate 平滑过渡到 adult_learning_rate
feritlity 不在 "YoungAdult" 瞬间开——从 Adolescent 晚期开始平滑上升
```

## 四、不同物种的阶段分布

### 4.1 RaceTraits 中的 LifeStageDistribution

```rust
/// 每个物种有自己独立的阶段百分比分布
/// 所有百分比以 max_lifespan 为分母
pub struct LifeStageDistribution {
    /// boundaries[i] = 第 i+1 阶段结束时的年龄占 max_lifespan 的比例
    /// boundaries[5] 应始终为 1.0（Elder 结束 = max_lifespan）
    pub boundaries: [f32; 6],
}
```

### 4.2 种族对比示例

| 阶段 | 人类 (max=70) | 精灵 (max=300) | 矮人 (max=120) |
|------|:---:|:---:|:---:|
| Juvenile 结束 | 0.10 (7岁) | 0.08 (24岁) | 0.09 (11岁) |
| Adolescent 结束 | 0.25 (18岁) | 0.22 (66岁) | 0.23 (28岁) |
| YoungAdult 结束 | 0.40 (28岁) | 0.50 (150岁) | 0.42 (50岁) |
| Adult 结束 | 0.70 (49岁) | 0.75 (225岁) | 0.70 (84岁) |
| MiddleAge 结束 | 0.85 (60岁) | 0.90 (270岁) | 0.85 (102岁) |
| Elder 结束 | 1.00 (70岁) | 1.00 (300岁) | 1.00 (120岁) |

**关键差异**：精灵的青年期被拉得很长（占总寿命的 42%——人类只有 15%），因为长寿物种的繁殖窗口更宽但更慢。矮人的青年期和人类接近——短寿物种在比例上更相似。

## 五、阶段边界与涌现行为的关系

### 5.1 阶段边界不是行为边界

`LifeStageTransition` 发布的是生物学事实——"此实体的身体现在进入了新的物理阶段"。这不等同于行为变化。

行为变化来自两个独立的来源：
1. **物理属性变化**（LifeStage 乘数）——影响 GOAP 动作评估的成本/成功率
2. **认知/社会参数的自然演化**——独立于 LifeStage 的连续过程

一个 Adolescent NPC 的物理属性是成人的 50-70%，所以 GOAP 评估"搬重石"的预期成本更高——自然选择更"轻"的动作。这不是"青少年被禁止搬重石"——是评估结果自然不利。

### 5.2 同阶段内个体差异极大

两个同为 Juvenile 的 NPC：
- A: 8 岁，高天赋种子，已经在铁匠铺学了两年，动作库上百条
- B: 7 岁，普通种子，从未离开家，动作库十来条

他们的 GOAP 产出完全不同——但都在同一个 LifeStage 里。LifeStage 只给了物理属性范围的约束（两人力量都在 0.15-0.3 区间），不决定行为。
