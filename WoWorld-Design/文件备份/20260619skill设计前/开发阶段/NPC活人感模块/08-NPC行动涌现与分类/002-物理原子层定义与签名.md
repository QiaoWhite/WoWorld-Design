## §一、设计原则

### 1.1 什么是物理原子

物理原子是 NPC 身体与世界交互的**最小不可再分单元**。每个原子只做纯物理计算——不包含领域知识（不知道"打铁"还是"打仗"）、不检查游戏规则（不关心"是否可以攻击这个目标"）、不触发剧情。

### 1.2 统一的原子执行管道

所有 35 个原子共享同一个执行模式:

```rust
fn execute_atom<A: PhysicalAtom>(
    atom: A,
    input: A::Input,
    agent: &AgentSnapshot,
    terrain: &dyn TerrainQuery,
    entities: &dyn EntityIndex,
    events: &dyn SpatialEventBus,
    visibility: &dyn VisibilityQuery,
) -> Result<A::Output, AtomError> {
    // 1. 参数噪声化: agent参数 × execution_noise(input.skill_proficiency)
    //    input.skill_proficiency 由调用方(领域模块)从技能系统获取后填入
    //    物理原子层不反向依赖技能系统——仅接收 f32 归一化值
    // 2. 物理计算: 材料属性对比 + 力学/热学/电学公式
    // 3. 副作用: 自动推送 AtomEvent 到 SpatialEventBus (声音+视觉+嗅觉)
    // 4. 返回结果
}

// skill_vector_compressed 编码方案 (AgentSnapshot中):
//   98个技能 → 按技能系统的5大类(Combat/Magic/Artisan/Academic/Survival)
//   每个大类取该类主导技能的level均值 → 5×f32=20 bytes → 压缩为5×u8
//   + 3个特殊标记bit(识字/金融素养/感知) → 总计 ~6 bytes嵌在 u64 中
//   解压: fn decompress_skill_group(compressed: u64, group: SkillGroup) -> f32
//         返回该组的近似 proficiency (0-1), 用于 GOAP 评估
//   精确技能等级: 领域模块在调用物理原子时自行从 SkillEntry 获取并传入 Input
```
```

### 1.3 所有原子的共同副作用

**每个物理原子执行时自动产生**:
- **声音足迹** (SoundFootprint): 音频模块轮询消费，无需原子主动通知
- **空间事件** (AtomEvent → SpatialEventBus): 其他NPC可通过感官系统感知
- **历史痕迹条件判断**: 重大物理后果触发 AetherImprint 生成检查

## §二、身体力学原子 (4)

### MOVE

```rust
/// 改变身体位置
enum LocomotionType { Walk, Run, Sprint, Crawl, Sneak, Climb, Swim, Glide, Fly }

struct MoveInput {
    locomotion_type: LocomotionType,
    target: Vec3,
    speed_multiplier: f32,   // 0-1, 从AgentSnapshot.mobility_factor连续调制
    gait_params: GaitParams, // 9个参数从BigFive×Emotion×Fatigue派生
}

struct MoveOutput {
    new_position: Vec3,
    stamina_consumed: f32,
    noise_level: f32,         // dB @ 1m, 影响被听觉检测的概率
    surface_contacts: Vec<SurfaceContact>, // 脚步声的物理基础
}
```

**关键**: MOVE 不关心"这是去铁匠铺还是逃跑"——它只做位移。`gait_params` 的 9 维参数（hip_sway, stride_length, arm_swing, bounce, forward_lean, rhythm_regularity, foot_drag, shoulder_stability, gaze_level）从 BigFive 人格 × Emotion × Fatigue 连续派生，确保每个 NPC 的步态独一无二。

### POSTURE

```rust
enum Pose { Stand, Crouch, Prone, Sit, Kneel, Lean(f32), Hang, Brace }
// KNEEL/BOW/PROSTRATE 是 POSTURE 的 pose 参数变体，不是独立原子。
// 区别在于文化修饰符——同样的 Kneel pose，在权力场景=臣服，在信仰场景=祈祷。

struct PostureInput { pose: Pose, duration_ms: u32, transition_blend_ms: u32 }
struct PostureOutput { success: bool, balance_risk: f32 }
```

**关键**: POSTURE 在物理层面是姿态改变。KNEEL/BOW/PROSTRATE 的区别是**文化语义层**的——物理上是同一个 POSTURE(Kneel) 原子，权力的"下跪"和信仰的"跪祷"共享相同的物理骨骼约束。文化的 `PostureMeaning` 映射表决定观察者如何解读。

### JUMP

```rust
struct JumpInput { direction: Vec3, force_multiplier: f32, landing_pose: Pose }
struct JumpOutput { trajectory: JumpTrajectory, landing_success: bool, fall_damage: f32 }
```

## §三、物体操作原子 (7)

### GRASP

```rust
enum HandConfig { OneHand, TwoHand, Tail, Mouth, Telekinetic }

struct GraspInput {
    entity: EntityId,
    hand_config: HandConfig,
    grip_strength: f32, // 从AgentSnapshot.strength × (1-health_penalty)连续派生
}

struct GripHandle { // 16 bytes Copy类型 (EntityId=8B + grip_point=1B + stability=4B + padding)
    entity_id: EntityId,
    grip_point: u8,      // 握持点索引
    stability: f32,       // 0-1，高=不滑脱
    // GripHandle 生命周期: 由 GRASP 创建 → 被后续原子(STRIKE/THROW/RELEASE等)消费
    // 自动失效条件: NPC死亡/昏迷/缴械/传送超距/被握持entity被BREAK摧毁
    //   失效时: EntityIndex发送EntityDestroyed→GripHandle标记为stale
    //   → 下次使用时GRASP返回AtomError::TargetDestroyed
}

struct GraspOutput {
    grip: Result<GripHandle, GraspError>,
    // GraspError: TooHeavy | TooSlippery | TooLarge | OutOfReach | Occupied | Refused
}
```

**关键**: GRASP 不区分"抓武器"和"抓工具"和"抓食物"。`GripHandle` 是后续 STRIKE/THROW/RELEASE/DIP 等原子的输入。`grip_strength` 从 AgentSnapshot 连续派生——3岁幼儿抓柱子=grip_strength=0.05, 任何物体 force_check 失败。

### RELEASE

```rust
enum ReleaseMode { Place(Vec3), Drop, Pass(EntityId) }

struct ReleaseInput { grip: GripHandle, release_mode: ReleaseMode }
struct ReleaseOutput { released_entity: EntityId, impact_force: f32 }
```

### STRIKE

```rust
struct StrikeInput {
    grip: GripHandle,
    target: EntityId,
    trajectory: StrikeTrajectory, // linear/arc/spline
    force: f32,                    // 0-1
    contact_preference: ContactType, // Point/Edge/Flat
}

struct StrikeOutput {
    hit: Option<HitResult>,
    self_damage: f32,             // 反震伤害,硬度差×力→施加者自身吸收
    sound_intensity: f32,         // dB
    spark_intensity: f32,         // 0-1, 硬度差→火花
    deformation_applied: f32,     // 目标变形量, MaterialProperties驱动
    fracture_risk_weapon: f32,    // 武器断裂风险
    fracture_risk_target: f32,    // 目标断裂风险
}
```

**关键**: STRIKE 是整个物理引擎的核心。不区分"战斗攻击"、"锻打铁锭"、"砸矿石"、"敲钉子"——同一个原子，不同的 MaterialProperties 对比结果产生完全不同的物理行为。锻造与战斗的涌现差异来自:
- **目标不同**: 热铁锭 vs 活体组织
- **MaterialProperties 对比不同**: 铁锤硬度 0.9 vs 熟铁 0.45（热软化后更低）→ 变形而非碎裂
- **领域规则不同**: 锻造模块在 STRIKE 循环中加入 OBSERVE("颜色对了吗"), 战斗模块在 STRIKE 后加入 CombatStyleParams 的下一个动作决策

### THROW

```rust
struct ThrowInput { grip: GripHandle, direction: Vec3, force: f32, spin: f32 }
struct ThrowOutput { flight_result: FlightResult, released_entity: EntityId }
```

### PUSH / PULL

```rust
struct PushInput { entity: EntityId, direction: Vec3, force: f32, contact_surface: BodyPart }
struct PushOutput { displacement: f32, self_strain: f32, surface_friction_used: f32 }

// 推开门=PUSH, 一脚踹开门=STRIKE。NPC选择哪个取决于力量评估×噪声。
```

### LIFT

```

## §五、组合与装配原子 (5)

### ATTACH

```rust
enum BondType {
    Rigid,                                  // 0 DOF 刚性 → 剑刃+剑柄
    Hinge { axis: Vec3, limits_deg: (f32, f32) }, // 1 DOF → 双节棍
    Chain { links: u16, link_length_mm: f32 },    // N铰链串行 → 连枷
    Flexible { stiffness: f32, taper_ratio: f32 },// 连续弯曲 → 鞭
    Rope,                                   // 纯拉力约束
    Elastic { spring_constant: f32 },        // 弹性连接
    Adhesive { bond_strength: f32 },         // 粘合 → 余料建造式组合
}

struct AttachInput {
    entity_a: GripHandle,
    entity_b: EntityId,
    attachment_point: Vec3,
    bond_type: BondType,
}

struct AttachOutput {
    compound: CompoundEntity,
    bond_strength: f32,    // MaterialProperties派生: min(a,b的对应强度)
}
```

**关键**: ATTACH 是王国之泪余料建造的通用化。不预设"什么可以粘什么"——只计算两个实体的附着面兼容性和连接强度。蘑菇+盾=弹跳盾（Elastic bond），石头+剑=锤剑（Rigid bond），火箭+盾=弹射起飞（Adhesive bond with low strength, 一次性）。

### DETACH / STACK / INSERT / WEDGE

详见完整签名。ATTACH 的逆操作、重力接触堆叠、精确插入、楔入。

## §六、破坏与改变原子 (3)

### BREAK

```rust
struct BreakInput { entity: EntityId, force: f32, contact_point: Vec3 }
struct BreakOutput {
    fractured: bool,
    fragments: Vec<FragmentDesc>,  // 碎片数量/大小/初始速度
    sound_intensity: f32,
}
```

**关键**: 碎裂判定 = force / MaterialProperties.toughness。玻璃 toughness=0.1 → 轻击即碎。铁 toughness=0.55 → 需要巨大力量。BREAK 不区分"战斗破坏武器"和"砸碎矿石"和"破门"。

### CUT / CRUSH

CUT 与 BREAK 的区别: CUT 受控（有锋利边缘）、沿平面、产生整齐断面。CRUSH 大面积压力，压强=力/面积驱动。

## §七、能量与元素原子 (7)

### IGNITE

```rust
struct IgniteInput { target: EntityId, ignition_source: EntityId, intensity: f32 }
struct IgniteOutput {
    ignited: bool,
    burn_intensity: f32,    // MaterialProperties.combustibility × intensity
    smoke_production: f32,

**关键**: 检查 `target.MaterialProperties.flammability × intensity > target.MaterialProperties.ignition_point`。旷野之息的"火把→干草→木屋→整片森林"连锁涌现完整路径: IGNITE(干草) → HEAT(周围空气,持续) → 相邻木屋.flammability × 热辐射 > 燃点 → 木屋也 IGNITE → 连锁。

## §九、统一的错误类型

```rust
enum AtomError {
    // 物理限制
    TooHeavy, TooLight, TooSlippery, TooLarge, TooSmall,
    OutOfReach, OutOfSight, InsufficientStrength,
    // 材料限制
    MaterialIncompatible, Melted, Frozen, Shattered,
    // 环境限制
    Blocked, NoSurface, Underwater, NotUnderwater,
    // 状态限制
    Occupied, Exhausted, Interrupted,
    // 实体生命周期
    TargetDestroyed,     // GripHandle指向的entity已被BREAK等摧毁
    GripInvalidated,     // 握持被外部强制解除(缴械/传送/昏迷)
}
```

~19 变体覆盖全部 35 个原子的所有可能失败模式。`GraspError` 是 `AtomError` 的子集别名:

```rust
type GraspError = AtomError;  // TooHeavy | TooSlippery | TooLarge | OutOfReach | Occupied
```

不需要每个原子定义自己的错误类型。

### 9.1 三层错误传播链

物理原子的错误如何传播到 GOAP 规划器:

```
L1 物理原子                     L2 领域复合                    L3 GOAP
═══════════                    ═══════════                    ════════
AtomError                      CompositeResult                ActionResult
    │                               │                              │
    ├─ TooHeavy ──────────→ CRAFT.step1 失败                      
    │                        ├─ 可重试(GripHandle仍有效): 重试≤3次
    │                        ├─ 不可重试: 返回 CompositeError     → GOAP收到Failed
    │                        └─ 替代路径: 换更轻的工具/加热后再试    → GOAP收到Partial
    │                                                                  │
    ├─ Exhausted ──────────→ CRAFT暂停                              
    │                        └─ 返回 CompositeError::Paused       → GOAP收到Deferred
    │                                                                  │
    ├─ Interrupted ────────→ 外部中断(战斗/天气)                     
    │                        └─ 保存状态, 稍后恢复                  → GOAP收到Interrupted
    │                                                                  │
    └─ 其他 ───────────────→ 直接失败                              
                             └─ 返回 CompositeError               → GOAP收到Failed
                                                                      │
                                                              GOAP 响应:
                                                              ├─ Failed → 标记目标失败, 重规划
                                                              ├─ Deferred → 降低优先级, 稍后重试
                                                              ├─ Partial → 接受部分结果, 继续
                                                              └─ Interrupted → 暂停, 等待恢复条件
```

**关键约定**: 
- L2 领域模块**必须**捕获所有 AtomError 并转换为 CompositeResult——物理原子错误不允许泄漏到 GOAP 层
- L3 ActionCandidate 的 execute_fn 将 CompositeResult 转换为 ActionResult
- GOAP 只消费 ActionResult——从不直接接触 AtomError
- 重试策略和替代路径由 L2 领域模块定义——不在物理原子层也不在 GOAP 层

## §十、设计反模式（明确不做的事）

**不做**: 为每个原子定义独立的错误类型 → 使用统一 AtomError（GraspError 是 AtomError 的子集别名）
**不做**: 在原子中检查游戏规则（"这个NPC可以攻击那个NPC吗"）→ 领域模块在调用原子前检查
**不做**: 原子中调用 GOAP 或修改 NPC 记忆 → 副作用通过 AtomEvent 发布, 其他模块订阅
**不做**: 原子关心自己在"战斗"还是"锻造" → 只读 MaterialProperties 做物理计算
**不做**: 原子操作持久化数据 → 直接读写 EntityId 对应的实体状态, 持久化由 LMDB 层处理
**不做**: CHANT 作为独立第一层物理原子 → CHANT 是领域复合原子（属魔法领域），由 POSTURE + WAIT + 音频系统发声管道组合。CAST 序列中的"CHANT"指的是 POSTURE(施法姿态) + WAIT(持续咏唱)

## §十一、并发与Mod扩展

### 11.1 并发原子执行

多个 NPC 的原子通过 **rayon 并行迭代器** 在 SoA 数组上批量执行。每个原子通过 `EntityIndex` 读写实体状态:

```rust
// SoA批量: 1000个NPC的MOVE原子在rayon线程池上并行
npcs.par_iter_mut().for_each(|npc| {
    let output = execute_atom(MOVE, npc.move_input, &npc.snapshot, /*...*/);
    npc.apply_move(output);
});
```

**写冲突处理**: 两个原子写同一Entity时的并发安全:
- 实体状态通过 `EntityIndex.update_transform()` 使用 atomic 版本号写入
- 读-修改-写竞争: 使用 optimistic locking ——写前检查版本号，冲突时自动重试
- 破坏性操作 (BREAK/DETACH): 使用 `EntityIndex` 内部互斥——同一实体同一帧最多一个破坏性操作
- 后果一致性: 所有并发的 STRIKE→同一目标 的伤害在帧末批量累加，避免读写交错

### 11.2 Mod 扩展

物理原子作为 Rust enum 提供 SoA 批量优化。Mod 新增自定义物理交互通过 `ModAtom` 扩展点:

```rust
enum PhysicalAtom {
    // 35 个核心原子 (SoA批量优化)
    Move(MoveInput), Grasp(GraspInput), Strike(StrikeInput), // ...
    
    // Mod 扩展 (trait对象, 无批量优化但足够用于稀有交互)
    ModAtom(u32, Box<dyn ModAtomExecutor>),
}
```

**约定**: Mod 原子使用独立 ID 空间 (0x8000_0000+)，按需注册到 `woworld_core`。核心原子的物理保证不被 Mod 覆盖。
