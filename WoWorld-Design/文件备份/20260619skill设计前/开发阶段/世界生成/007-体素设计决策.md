## 〇、根本原则（保留）

```
体素 = 世界的地形骨架
网格 = 地形之上的一切
```

**体素负责大地**——山怎么长、河怎么流、矿在哪儿、玩家能不能挖洞。
**网格负责万物**——树、房子（外壳）、家具、NPC、武器、装饰。

这不是 Minecraft——所有东西不是方块。这是连续表面体素地形 + 网格物体混合世界。

## 一、体素引擎规格（v3 重写）

### 1.1 技术方案

| 项 | v3 方案 |
|----|---------|
| **等值面提取** | Transvoxel（Rust 自建） |
| **LOD 体系** | Clipmap 8 级（LOD 0-7）+ Signed Heightfield（3km+） |
| **Chunk 结构** | 垂直稀疏——水平 32m×32m，垂直动态（仅地表附近存体素） |
| **体素单位** | 0.5m³/体素 |
| **密度场** | 分层密度场（9 层：海陆→地形→地质→水文→奇幻→文化→NPC修改→玩家修改→天气临时） |
| **存储** | LMDB 流式 Chunk 存储——种子确定性 + 增量（只存修改过的 Chunk） |
| **与 Godot 交互** | GDExtension → Godot ArrayMesh |
| **碰撞** | 从 Transvoxel mesh 生成碰撞体（Godot PhysicsServer3D）→ CHG-033: 仅玩家保留, 其余 Rust TerrainQuery |
| **导航** | 从体素地形烘焙 NavigationMesh（局部增量重烘焙） |

> ⚠️ **[CHG-033 架构变更 — 2026-06-18 审计更新]**: 地形碰撞检测已从 Godot PhysicsServer3D 碰撞体烘焙迁移至 Rust TerrainQuery 密度场查询。Transvoxel mesh 碰撞体仅用于玩家 CharacterBody3D。NPC/动物/载具使用 TerrainQuery::height_at() + is_walkable() 进行地形交互。导航网格烘焙→Rust EntityIndex 空间索引。

### 1.2 Clipmap LOD 体系（v3.0 §4.5）

| LOD 级 | 体素精度 | 覆盖距离 | 算法 | 细节 |
|--------|---------|---------|------|------|
| LOD 0-1 | 0.5-1m/voxel | 0-100m | Transvoxel 全精度 | 悬挑/石桥/拱门——完整 3D |
| LOD 2-3 | 2-4m/voxel | 100-500m | Transvoxel 中等 | 悬挑简化，主要特征保留 |
| LOD 4-5 | 8-16m/voxel | 500m-3km | Transvoxel 低精度 | 大型岩架保留，细小悬挑丢失 |
| LOD 6-7 | 32-64m | 3km-10km | Signed Heightfield + 法线烘焙 | 悬挑→法线的光影欺骗 |
| LOD 8 | — | 10km+ | Billboard + 大气散射 | 天际线/远山轮廓 |

### 1.3 垂直稀疏 Chunk

```
Chunk 水平尺寸: 32m × 32m（不变）
Chunk 垂直范围: 动态
  地表以上: 不存储体素（空气），仅标记植被/建筑位置
  地下浅层: 0-50m（洞穴/矿脉/地基）
  地下深层: 按需展开（靠近特殊地质特征时）

活跃 Chunk 垂直层数: ~100
  但仅覆盖 [地表-50m, 地表+5m]——大部分是稀疏的
```

### 1.4 密度场分层架构

```rust
/// 最终密度 = 多层叠加
struct DensityField {
    layers: Vec<Box<dyn DensityProvider>>,
}

trait DensityProvider: Send + Sync {
    /// 查询指定世界坐标的密度值
    fn density_at(&self, pos: Vec3) -> f32;
    /// 查询材质 ID
    fn material_at(&self, pos: Vec3) -> u8;
    /// 该层的优先级（修改层的优先级高于生成层）
    fn priority(&self) -> u8;
}

// 层从低到高:
// Layer 0: 全球海陆掩码
// Layer 1: 大陆骨架地形
// Layer 2: 区域地形
// Layer 3: 局部细节
// Layer 4: 地质特征（洞穴/矿脉）
// Layer 5: 水文（河流侵蚀）
// Layer 6: 奇幻特征（DensityProvider trait——可插拔）
// Layer 7: 文化覆盖（建筑地基/道路/农田）
// Layer 8: NPC批量修改
// Layer 9: 玩家SDF操作
// Layer 10: 天气临时（积雪/积水）
```

## 四、体素 ↔ 网格边界（保留原则）

### 明确作为网格（非体素）的事物

| 事物 | 实现方式 |
|------|---------|
| 建筑(墙/屋顶/地板) | 模块化网格 |
| 家具 | 网格道具 |
| 角色(NPC/玩家/动物) | 骨骼网格 |
| 植被 | MultiMeshInstance3D |
| 物品(武器/工具/食物) | 网格道具 |
| 水面效果 | 着色器平面——Gerstner 波 |
| 粒子效果 | GPU 粒子 |
| 体积云 | 2D impostor + 3D 体积密度场 |

### 建筑放置与体素

建筑放置时：
- 自带地基网格（类似 Valheim）——**不修改体素地形**
- 仅在"挖地基建地下室"时进行 SDF 体素操作

## 五、SDF 地形编辑（v3 Rust 接口）

```rust
/// 体素编辑操作
enum VoxelEditOp {
    /// 降低体素密度——挖掘
    Dig { center: Vec3, radius: f32, amount: f32 },
    /// 提高体素密度——填埋
    Fill { center: Vec3, radius: f32, material: u8, amount: f32 },
    /// 平整——将区域密度设为指定值
    Flatten { center: Vec3, radius: f32, target_elevation: f32 },
}

trait TerrainEditor: Send + Sync {
    /// 提交编辑操作（SDF 叠加到密度场）
    fn submit_edit(&mut self, op: VoxelEditOp, rng: &mut StdRng);

    /// 批量提交——减少重生成次数
    fn submit_batch(&mut self, ops: Vec<VoxelEditOp>, rng: &mut StdRng);

    /// 撤销最近编辑
    fn undo_last(&mut self) -> bool;

    /// 标记受影响 Chunk 需要重新提取 Mesh
    fn mark_dirty_chunks(&self) -> Vec<ChunkCoord>;
}
```

**编辑限制**：
- 单次操作限制在单个 Chunk 范围内（32m³）
- 修改后该 Chunk + 直接邻居重新生成等值面
- 重生成预算 ≤ 2ms/帧
- 延迟 1-2 秒批量处理导航网格更新

## 六、采矿系统

### 6.1 工具挖掘（保留+参数微调）

| 工具等级 vs 体素硬度 | 速度 | 产量 | 说明 |
|----------------------|------|------|------|
| 工具等级 ≥ 硬度 | 100% | 100% | 正常开采 |
| 工具等级 = 硬度-1 | 70% | 80% | 勉强开采，有损耗 |
| 工具等级 = 硬度-2 | 45% | 55% | 困难开采，损耗较大 |
| 工具等级 ≤ 硬度-3 | 20% | 25% | 极难开采，大部分浪费 |
| 工具类型不对应 | ×0.6 | ×0.5 | 如用镐挖沙 |

### 6.2 采矿方式（保留）

- **露天采矿**：矿体<20m深→阶梯式挖掘→永久地形改变
- **地下开采**：竖井/斜井→巷道网络→采矿行为循环→废弃矿井成探索点
- **水域采矿**：采砂船→河流/浅海→砂金/锡石

### 6.3 采矿速度设计原则（保留）

采矿不应当像 Minecraft 任意的即时快速开采。应缓慢、费时费力，体现工程性质。游戏可快进（睡眠/等待）。
