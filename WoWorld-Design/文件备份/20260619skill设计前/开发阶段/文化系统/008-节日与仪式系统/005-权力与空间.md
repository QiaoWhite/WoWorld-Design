## 一、节日的组织者

### 1.1 谁组织节日

节日组织者从以下来源决定（按优先级递减）：

| 来源 | 条件 | 出现概率 |
|------|------|---------|
| **统治者** | 政体存在且统治者的 `power_distance > 0.3` | 40%（大多数节日由权威组织） |
| **神殿/宗教机构** | `religiosity > 0.4` 且该节日是宗教节日 | 35% |
| **行会/商会** | 该节日以经济活动为主（集市、拍卖） | 10% |
| **自发群众** | 该节日是从涌现型升级而来 | 10% |
| **特定家族/氏族** | `individualism < 0.3`（集体文化中的家族传统） | 5% |

### 1.2 组织者角色

```rust
struct FestivalOrganizer {
    organizer_type: OrganizerType,
    /// 组织者的权力边（如果是统治者——关联到合法性计算）
    power_edge_id: Option<PowerEdgeId>,
    /// 组织者承担的节日成本（铜币）
    financial_burden: u64,
    /// 组织者从节日获得的利益
    benefits: Vec<OrganizerBenefit>,
}

enum OrganizerType {
    Ruler { power_edge_id: PowerEdgeId },
    ReligiousInstitution { faith_id: FaithId, temple_id: BuildingId },
    Guild { guild_id: GuildId },
    Grassroots { founding_npc_id: NpcId },
    Clan { clan_id: ClanId },
}

enum OrganizerBenefit {
    LegitimacyGain,       // 统治者：合法性增加
    InfluenceGain,        // 神殿/行会：在聚落中的影响力增加
    FinancialProfit,      // 商人：从节日消费中获利
    ReputationGain,       // 个人：名声+慷慨
    SocialCapital,        // 自发群众：社区凝聚力增强
}
```

## 二、节日作为权力展示

### 2.1 统治者的合法性获取

权力系统 004 的合法性公式有一个 `ritual 0.10` 因子——需要 `last_ritual_at` 时间戳。**节日是更新这个时间戳的机制**。

```rust
/// 统治者主持节日仪式 → 更新权力边的 last_ritual_at
fn on_ruler_celebrates_festival(
    ruler_edge: PowerEdgeId,
    festival: &FestivalInstance,
    power_topology: &mut PowerTopology,
) {
    let edge = power_topology.get_edge_mut(ruler_edge);
    
    // 更新最近仪式时间
    edge.last_ritual_at = Some(GameTime::now());
    
    // 仪式质量取决于节日规模
    let ritual_quality = festival.scale * festival.participant_ratio();
    
    // 合法性因子重新计算——ritual 分量为 0.03-0.15
    let ritual_legitimacy_boost = ritual_quality * 0.15;
    edge.cached_legitimacy = recompute_legitimacy(edge, ritual_legitimacy_boost);
}
```

**统治者选择性主持节日**：不是所有节日都由统治者主持。优先选择：
- 日历节日中最重大的（元旦/秋收）
- 宗教节日中主流信仰的（国教的最高圣日）
- 纪念性节日中与统治者直接相关的（登基周年、建国日）

### 2.2 节日的失败风险

如果统治者主持节日但节日因故失败（天灾、战争打断、参与率极低）——合法性反而下降：

```
节日失败 → ritual 分量为 0（甚至负数——"连节日都办不好"）
         → NPC 的 discontent 情绪触发
         → 如果连续 2 年节日失败 → 合法性 crisis 检测概率增加
```

## 三、节日作为义务

### 3.1 强制参加的权力机制

统治者通过 `PrescribeRule` 权力原子创建"节日参加义务"：

```rust
/// 创建节日参加义务
fn create_festival_obligation(
    ruler_edge: PowerEdgeId,
    festival_id: FestivalId,
    power_topology: &mut PowerTopology,
) -> DutyId {
    let duty = Duty {
        duty_type: DutyType::Obligation,
        description: format!("Must attend the annual {} festival", festival_name(festival_id)),
        scope: NormScope::Everyone, // 或特定群体
        penalty: Penalty {
            penalty_type: PenaltyType::Fine { amount: 50 }, // 铜币罚款
            // 或 SocialStigma, 或 WorkPunishment
        },
        exemption_conditions: vec![
            ExemptionCondition::Health,     // 生病可免
            ExemptionCondition::Distance,   // 离家>X公里可免
            ExemptionCondition::Immunity,   // 有 ImmunitySet 的可免
        ],
    };
    
    power_topology.add_duty(ruler_edge, duty)
}
```

**为什么不所有节日都强制**：强制参加消耗权力资源——每次强制都是一次"权力行使"。如果统治者对所有节日都强制 → 权力被稀释 → 单个强制违约的制裁力度降低 → 权力失效。

### 3.2 豁免

某些群体自动豁免节日强制参加——通过 `ImmunitySet`（权力 005）：
- 异教徒（不对国教节日负责）
- 必要工种轮值者（守夜人、医生）
- 在服丧期的 NPC
- 儿童（视文化）

## 四、征服后的节日变更

### 4.1 征服者的策略

```rust
enum ConquerorFestivalPolicy {
    /// 保留被征服文化的所有节日（宽松统治）
    Preserve,
    /// 禁止特定节日（对征服者文化有威胁的——如纪念前朝胜利的节日）
    Ban { banned_festival_ids: Vec<FestivalId> },
    /// 用征服者的节日替换
    Replace { 
        removed: Vec<FestivalId>,
        introduced: Vec<FestivalId>, 
    },
    /// 强制融合——被征服文化的节日在征服者的日期举行
    Syncretize { mapping: FxHashMap<FestivalId, u16> },
}
```

**策略选择因子**：

```
conqueror.militarism 高 + power_distance 高 → Replace（强制同化）
conqueror.religiosity 高 → Ban 异教节日，Replace 为国教节日
被征服文化的 uncertainty_avoidance 高 → 更强的文化抵抗
被征服文化的 openness 低 → 被禁止的节日仍然在地下偷偷举行
```

### 4.2 地下节日

被禁止的节日不会消失——它们可能转为"地下"：

```
官方：节日被废除
现实：NPC 在私人住宅中秘密举行（参与率 20-50% 的原始人数）
风险：被守卫发现 → 参与 NPC 受到制裁
态度：被征服文化的 NPC 同时产生 discontent（对统治者）和 solidarity（彼此间）
→ 这是起义的完美燃料
```

## 五、节日空间——在哪里举办

### 5.1 节日地点选择

```rust
enum FestivalLocation {
    /// 聚落内部——广场/神殿前/市场
    Settlement {
        settlement_id: SettlementId,
        preferred_zone: FunctionalZoneType,
    },
    /// 聚落外特定地点——圣山/圣泉/古树/遗迹
    External {
        geographic_entity_id: GeographicEntityId,
    },
    /// 沿路线——游行/朝圣
    Procession {
        start: GeographicEntityId,
        end: GeographicEntityId,
        waypoints: Vec<GeographicEntityId>,
    },
}
```

**地点选择逻辑**：
1. 宗教节日 → 神殿前广场（或信仰系统指定的圣地）
2. 竞技节日 → 聚落的竞技场（如果有）或城门外空地
3. 日历节日 → 市中心广场（或市场广场——"收获节在卖粮食的地方举行最合适"）
4. 涌现型节日 → 该自然现象的发生地（鲑鱼节在河边）

### 5.2 广场——当前世界生成的缺失

当前世界生成没有"广场"建筑类型。需要新增：

**方案 A（推荐）：给 ReligiousDistrict 和 MarketDistrict 增加"开放空间"子区域**

```rust
// 在 P5 聚落内部布局的功能分区中：
FunctionalZone {
    zone_type: ReligiousDistrict,
    // 新增字段：
    open_space_ratio: f32, // 0.15-0.30 —— 该区域中不建建筑、保留为开放空间的面积占比
}

// 该开放空间自动作为该聚落的节日场地
fn get_festival_location(settlement: &SettlementData) -> WorldPos {
    // 优先：神殿前的开放空间
    // 备选：市场广场
    // 最后：城门外空地
}
```

**不需要新增 BuildingFunction::Plaza**——开放空间不是一个"建筑"，而是一个"缺乏建筑的区域"。它由城市级 WFC 的约束生成（神殿正前方必须留空 20-30m）。

### 5.3 朝圣目的地

某些宗教节日绑定到特定地理实体而非聚落：

```rust
/// 朝圣目的地——由信仰系统定义
struct PilgrimageDestination {
    geographic_entity_id: GeographicEntityId, // 圣山/圣泉/神树
    associated_festival: FestivalId,
    /// 吸引半径——多远外的 NPC 会来朝圣
    draw_radius_km: f32,
    /// 每年参与的估计人数
    estimated_annual_pilgrims: u32,
}
```

朝圣 NPC 在交通网络中生成 `TravelerType::Pilgrim` 的旅行者。大型朝圣目的地会形成沿途的基础设施（朝圣客栈、路边神龛）。

### 5.4 临时节日空间

节日集市摊位是临时的——运行时创建，节后拆除：

```
庆祝期第1天：市场广场/街道两侧出现临时摊位
  → 使用简化 mesh（木架+布顶——不需要完整建筑模型）
  → 每个摊位关联一个临时 Storefront（经济系统）
  
余波期第2天：摊位拆除
  → 摊位 mesh 消失
  → 临时 Storefront 关闭
  → NPC 商人的库存回到常规商店或旅行商人背包
```
