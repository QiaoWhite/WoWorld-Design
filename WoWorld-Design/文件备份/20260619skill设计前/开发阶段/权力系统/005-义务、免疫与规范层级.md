## 一、三层补充机制

权力原子定义了"谁能做什么"。但完整的权力关系还需要三个补充层：
1. **Duty（义务）**：权力边的镜像——subject 负有做什么/不做什么
2. **ImmunitySet（免疫）**：subject 对某些权力原子的豁免——即使 holder 有该原子也无法对 immune subject 行使
3. **规范层级**：多条 `PrescribeRule` 冲突时谁优先

## 二、Duty —— 义务实体

### 2.1 定义

```rust
/// 单条义务——"谁对谁有什么样的义务"
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Duty {
    pub duty_id: DutyId,
    /// 负有义务的实体
    pub obligor: EntityId,
    /// 对谁负有义务
    pub obligee: EntityId,
    /// 义务内容
    pub content: DutyContent,
    /// 义务类型
    pub duty_type: DutyType,
    /// 触发时机
    pub trigger: DutyTrigger,
    /// 违约时规定的制裁
    pub prescribed_sanction: Option<SanctionSpec>,
    /// 此义务的来源边
    pub source_edge: EdgeId,
    /// 监督者（可选——谁来确保义务被履行）
    pub guardian: Option<EntityId>,
    /// 状态
    pub active: bool,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum DutyType {
    /// 作为义务：必须做某事
    Obligation,
    /// 不作为义务：禁止做某事
    Prohibition,
    /// 容忍义务：必须允许他人做某事
    Toleration,
    /// 赔偿/补救义务：因违约/侵权产生的后续义务
    Remediation,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum DutyTrigger {
    /// 持续性：一直有效，每次相关事件发生时检查
    Continuous,
    /// 定期：按固定时间间隔检查
    Periodic { interval: GameDuration, next_check: GameTimestamp },
    /// 条件触发性：当特定条件满足时检查
    Conditional { condition: DutyCondition },
    /// 混合：持续 + 定期汇报
    ContinuousWithReport { report_interval: GameDuration },
}
```

### 2.2 Duty 的自动生成

| 权力边的创建 | 自动生成的 Duty |
|------------|----------------|
| Extract(tax) 边 | **Obligation**：obligor = subject，向 holder 定期缴付税金 |
| Constrain(behavior) 边 | **Prohibition**：obligor = subject，禁止执行被约束行为 |
| Compel(service) 边 | **Obligation**：obligor = subject，按频率执行被强制的服务 |
| PrescribeRule(scope, content) 边 | **批量生成**：scope 内每个 subject 获得对应的 Obligation/Prohibition |
| Pledge(self, content) 边 | **Obligation**：obligor = holder（=subject），对自己负有义务 |
| Contract(obligations) 边 | **双向生成**：A 对 B 的义务 → B 成为该 Duty 的 guardian（监督权=作为 guardian 参与制裁塌缩链） |
| Adjudicate(ruling) 执行时 | **Remediation**：败诉方负有赔偿/补救义务——Adjudicate 的裁决结果自动生成 Duty |

### 2.3 违约处理——制裁塌缩链

**核心原则**：Duty 违约时，制裁**不是自动执行**——系统查找"谁有权制裁 + 谁有意愿制裁"。如果无人制裁 → legitimacy 下降。如果 legitimacy 降到临界 → 触发革命或权力结构重组。

```rust
/// 制裁塌缩链——从违约到结果
fn resolve_breach(
    duty: &Duty,
    breach: &DutyBreach,
    topology: &PowerTopology,
) -> BreachResolution {
    // === 阶段 1：找出所有对此 obligor 持有 Sanction 权力的 holder ===
    // 注：duty.domain() 是 Duty 的派生方法——内部通过 source_edge 从 PowerTopology 
    // 查询对应的 PowerEdge.domain。Duty 自身不存储 domain 字段以避免数据冗余。
    let potential_enforcers = topology.find_holders_with_atom_over(
        AtomTypeTag::Sanction,
        duty.obligor,
        &duty.domain(),
    );
    
    // 如果有指定的 guardian，加入候选
    if let Some(guardian) = duty.guardian {
        if !potential_enforcers.contains(&guardian) {
            potential_enforcers.push(guardian);
        }
    }
    
    // === 阶段 2：评估每个候选人的执行意愿 ===
    let willing: Vec<(&EntityId, f32)> = potential_enforcers.iter()
        .map(|h| {
            let willingness = compute_enforcement_willingness(*h, duty, breach);
            (*h, willingness)
        })
        .filter(|(_, w)| *w > WILLINGNESS_THRESHOLD)  // 0.15
        .sorted_by(|(_, w1), (_, w2)| w2.partial_cmp(w1).unwrap())
        .collect();
    
    // === 阶段 3：塌缩逻辑 ===
    match willing.len() {
        0 => {
            // 无人愿意制裁 → legitimacy 下降 → 边可能被视为"废纸"
            BreachResolution::Unenforced {
                legitimacy_loss: 0.15,
                reason: "no_willing_enforcer",
            }
        }
        1 => {
            // 唯一执行者 → 发起制裁
            let enforcer = *willing[0].0;
            BreachResolution::Enforced {
                enforcer,
                method: duty.prescribed_sanction
                    .unwrap_or_default_sanction(duty),
            }
        }
        _ => {
            // 多个执行者 → 最高优先级的执行
            let primary = willing.iter()
                .max_by_key(|(h, w)| {
                    // 委托链距离（近优先）× 意愿
                    let distance = topology.delegation_distance(**h, duty.obligor);
                    (w * 100.0 / (distance + 1) as f32) as i32
                })
                .map(|(h, _)| *h);
            BreachResolution::Enforced {
                enforcer: primary.unwrap(),
                method: duty.prescribed_sanction
                    .unwrap_or_default_sanction(duty),
            }
        }
    }
}

/// 计算 holder 对某违约的制裁意愿
fn compute_enforcement_willingness(
    holder: EntityId,
    duty: &Duty,
    breach: &DutyBreach,
) -> f32 {
    // 与 obligee 的关系强度 → 正向（为其出头）
    let relation_with_obligee = relationship_strength(holder, duty.obligee);
    
    // 与 obligor 的关系强度 → 负向（包庇动机）
    let relation_with_obligor = relationship_strength(holder, duty.obligor);
    
    // 尽责性 → 正向
    let conscientiousness = personality(holder).conscientiousness;
    
    // 对 duty.source_edge 的 legitimacy 感知 → 正向
    let rule_legitimacy = perceived_legitimacy(holder, duty.source().holder, duty.domain());
    
    // 恐惧 → 负向（怕 obligor 报复）
    let fear_of_obligor = fear_assessment(holder, duty.obligor);
    
    relation_with_obligee * 0.35
        + conscientiousness * 0.30
        + rule_legitimacy * 0.30
        - relation_with_obligor * 0.25
        - fear_of_obligor * 0.10
}
```

### 2.4 违约无人制裁的后果

当 `BreachResolution::Unenforced` 累积 → 权力结构合法性侵蚀：

```rust
/// 每游戏月检查——是否有"无人制裁的违约"积压
fn check_unenforced_breaches(&mut self) -> Vec<PowerEvent> {
    let mut events = Vec::new();
    
    for holder in self.all_holders() {
        let unenforced_count = self.count_unenforced_breaches(holder);
        if unenforced_count > UNENFORCED_THRESHOLD {  // [TUNING] 5
            // 降低所有关联权力边的 legitimacy
            for edge_id in self.outgoing_edges(holder) {
                let edge = self.get_edge_mut(edge_id);
                edge.legitimacy -= 0.05 * (unenforced_count as f32 / 5.0);
            }
            
            if unenforced_count > UNENFORCED_CRISIS_THRESHOLD {  // [TUNING] 15
                events.push(PowerEvent::LegitimacyCrisis {
                    holder,
                    unenforced_count,
                });
                // 触发革命检测
                self.flag_for_revolution_check(holder);
            }
        }
    }
    events
}
```

### 2.5 革命检测

```rust
/// 检测一个 holder 是否面临革命威胁
fn check_revolution_risk(
    topology: &PowerTopology,
    holder: EntityId,
) -> Option<RevolutionRisk> {
    // 找出所有对 holder 的 legitimacy < 0.15 的 subject
    let disaffected: Vec<EntityId> = topology.subjects_of(holder)
        .filter(|s| topology.perceived_legitimacy(*s, holder, PowerDomain::Universal) < 0.15)
        .collect();
    
    if disaffected.len() < REVOLUTION_MIN_DISAFFECTED {  // [TUNING] 需求的 subject 数量的某个最小百分比
        return None;
    }
    
    // 检查 disaffected 中是否有人有替代方案（另一个 holder 或自治结构）
    let alternatives = topology.find_alternative_holders(&disaffected, holder);
    
    Some(RevolutionRisk {
        holder,
        disaffected_count: disaffected.len(),
        total_subjects: topology.count_subjects(holder),
        alternative_leaders: alternatives,
        /// 革命概率 = f(不满者比例 × 替代方案存在 × 最近触发事件)
        probability: compute_revolution_probability(&disaffected, &alternatives),
    })
}
```

### 3.1 定义

```rust
pub struct ImmunitySet {

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Immunity {
    /// 免疫的权力原子类型
    pub atom_type: AtomTypeTag,

pub enum ImmunitySource {
    /// 法律授予（PrescribeRule 产生的豁免规则）
    Legal { rule_edge: EdgeId },
    /// 契约授予
    Contractual { contract_edge: EdgeId },
    /// 身份附带（如外交官身份 → 对东道国的 Sanction 免疫）
    StatusBased { status: String },
    /// 神授/魔法
    Divine { ritual: String },
    /// 习俗/传统（文化决定的默认免疫）
    Customary { culture_id: CultureId },
}
```

### 3.2 免疫检查

```rust
fn has_immunity(
    entity: EntityId,
    atom_type: AtomTypeTag,
    domain: &PowerDomain,
) -> bool {
    self.immunities.get(&entity)
        .map(|iset| {
            iset.immunities.iter().any(|imm| {
                imm.atom_type == atom_type
                    && (imm.domain.is_none() || imm.domain.as_ref() == Some(domain))
                    && (imm.valid_until.is_none() || current_time() <= imm.valid_until.unwrap())
            })
        })
        .unwrap_or(false)
}
```

### 3.3 典型免疫场景

| 场景 | 免疫 | 来源 |
|------|------|------|
| 外交使节在外国 | 对东道国的 `Constrain`(拘留) + `Sanction`(刑事) + `Extract`(征税) 免疫 | StatusBased |
| 议员在议会中发言 | 对 `Sanction`(诽谤罪) 免疫（议会言论免责） | Legal（宪法层 PrescribeRule） |
| 神职人员在神权国 | 对世俗法律的 `Adjudicate` 免疫——由宗教法庭专属管辖 | Customary |
| 贵族对平民法 | 对 `Extract`(平民税率) 免疫——适用贵族税率 | Legal |
| 年幼儿童 | 对 `Compel`(兵役)、`Sanction`(死刑) 免疫 | Customary |
| 魔法契约的保护方 | 对特定 `Compel` 的免疫——由 Contract 约定 | Contractual |

### 3.4 Immunity 与 Derogate 的区别

| | Immunity | Derogate |
|------|---------|---------|
| 谁控制 | subject 自带 | holder 授予 |
| 方向 | subject → holder（"你不能动我"） | holder → subject（"我不动你"） |
| 持续性 | 随身份/状态持续 | 可到期，可撤销 |
| 示例 | 外交豁免 | 国王特赦某人免服兵役 |

两者互补——Immunity 是 subject 的盾牌，Derogate 是 holder 的让步。

## 四、规范层级——PrescribeRule 冲突解决

### 4.1 问题

多个 holder 各自行使 `PrescribeRule`，但规则覆盖同一 subject 和同一行为域。例如：

- 国王的规则："偷窃者斩手"
- 城主的规则："偷窃者罚 2 倍铜币"（国王授权城主自行立法）
- 教会的规则："偷窃者忏悔 3 次即可"（教会对 member 有独立立法权）

**谁优先？**

### 4.2 三级优先规则（硬编码元规范）

```rust
/// 解析两条冲突的 PrescribeRule 的优先级
/// 返回：应该适用的规则边 ID
fn resolve_rule_conflict(
    rule_a: EdgeId,
    rule_b: EdgeId,
    subject: EntityId,
    topology: &PowerTopology,
) -> EdgeId {
    let edge_a = topology.get_edge(rule_a);
    let edge_b = topology.get_edge(rule_b);
    
    // === 第 1 级：委托链距离（离 subject 近的优先） ===
    let dist_a = delegation_distance(edge_a.holder, subject, topology);
    let dist_b = delegation_distance(edge_b.holder, subject, topology);
    
    if dist_a < dist_b { return rule_a; }
    if dist_b < dist_a { return rule_b; }
    
    // === 第 2 级：领域专属原则（domain 约束的优先于通用域） ===
    let a_specific = !matches!(edge_a.domain, PowerDomain::Universal);
    let b_specific = !matches!(edge_b.domain, PowerDomain::Universal);
    
    if a_specific && !b_specific { return rule_a; }
    if b_specific && !a_specific { return rule_b; }
    
    // === 第 3 级：legitimacy 比较 ===
    if edge_a.legitimacy > edge_b.legitimacy { return rule_a; }
    if edge_b.legitimacy > edge_a.legitimacy { return rule_b; }
    
    // === 平局：触发 Adjudicate 事件（由更高级 holder 或第三方裁决） ===
    // 这不是自动选择——而是产生一个待裁决的争议
    trigger_rule_conflict_adjudication(rule_a, rule_b, subject, topology);
    
    // 暂时两条规则都适用——等裁决确定后只有一条生效
    // 对 subject 来说：适用 stricter 的规则（保守原则）
    stricter_rule(rule_a, rule_b)
}
```

### 4.3 委托链距离计算

```rust
fn delegation_distance(
    holder: EntityId,
    subject: EntityId,
    topology: &PowerTopology,
) -> u32 {
    // 从 subject 沿入边查找 holder——距离 = 中间经过的 Delegate 跳数
    let mut visited = HashSet::new();
    let mut queue = VecDeque::new();
    
    // BFS 从 subject 开始，沿入边追溯
    for edge in topology.incoming_edges(subject) {
        if edge.atom_type == AtomTypeTag::Delegate && edge.subject == subject {
            queue.push_back((edge.holder, 1u32));
        }
    }
    
    while let Some((current, dist)) = queue.pop_front() {
        if current == holder { return dist; }
        if !visited.insert(current) { continue; }
        
        for edge in topology.incoming_edges(current) {
            if edge.atom_type == AtomTypeTag::Delegate {
                queue.push_back((edge.holder, dist + 1));
            }
        }
    }
    
    u32::MAX  // 不在委托链上
}
```

### 5.2 Contract 的生效条件

Contract 与其他原子的关键区别——它需要**双方同意**才能生效：

```rust
fn validate_contract(
    params: &ContractParams,
    topology: &PowerTopology,
) -> Result<(), PowerExerciseError> {
    // 1. 双方必须是不同的实体
    if params.party_a == params.party_b {
        return Err(PowerExerciseError::GeneralFailure("不能与自己签约".into()));
    }
    
    // 2. 双方都必须有能力签约（没有"禁止签约"的 Constrain 边）
    for party in [params.party_a, params.party_b] {
        if topology.is_constrained_from_contracting(party) {
            return Err(PowerExerciseError::BehaviorConstrained);
        }
    }
    
    // 3. 双方必须同意（在模拟中：双方都需要"同意"信号）
    //    玩家侧：UI 确认
    //    NPC 侧：NPC 决策引擎评估契约条款是否可接受
    //    （评估由 NPC 心智系统完成——不在权力系统内）
    
    Ok(())
}
```

## 六、Pledge 的自我约束机制

### 6.1 Pledge 的效果

```rust
fn execute_pledge(
    topology: &mut PowerTopology,
    holder: EntityId,
    params: &PledgeParams,
) -> Result<PowerExerciseResult, PowerExerciseError> {
    // 1. 创建一条 holder=self 的权力边
    let edge_id = topology.create_edge(PowerEdgeData {
        holder,
        subject: holder,     // self
        atom_type: AtomTypeTag::Pledge,
        domain: PowerDomain::Universal,
        source: PowerSource::Appointed { appointer: holder },
        succession_rule: SuccessionRule::ExtinguishWithHolder,
        ..Default::default()
    });
    
    // 2. 创建一条 Duty——holder 对自己负有此义务
    let duty_id = topology.create_duty(Duty {
        obligor: holder,
        obligee: params.pledged_to.unwrap_or(holder),  // 对谁承诺
        content: DutyContent::from(params.content.clone()),
        duty_type: DutyType::Obligation,
        trigger: DutyTrigger::Continuous,
        prescribed_sanction: params.sanction_on_breach.clone(),
        source_edge: edge_id,
        guardian: params.witnesses.first().copied(),   // 见证人作为 guardian
        active: true,
    });
    
    Ok(PowerExerciseResult::PledgeCreated { edge_id, duty_id })
}
```

### 6.2 Pledge 违约的独特性

Pledge 违约的制裁塌缩链与普通 Duty 违约不同：
- 若 `pledged_to = None`（对自己承诺）→ 无人可制裁 → 纯粹 legitimacy 损失
- 若 `pledged_to = Some(deity)`（对神承诺）→ 信仰系统介入（神可能"降下神罚"）
- 若 `pledged_to = Some(people)`（对人民承诺）→ guardian = 人民的代表 → 如果代表无力制裁 → 人民可能集体反抗

## 九、补充：default_sanction

当 Duty 没有显式指定 `prescribed_sanction` 时的默认制裁：

```rust
fn default_sanction(duty: &Duty) -> SanctionSpec {
    match duty.duty_type {
        DutyType::Obligation | DutyType::Remediation => {
            SanctionSpec { method: SanctionMethod::Fine, magnitude: 0.3 }
        }
        DutyType::Prohibition => {
            SanctionSpec { method: SanctionMethod::Fine, magnitude: 0.5 }
        }
        DutyType::Toleration => {
            SanctionSpec { method: SanctionMethod::Fine, magnitude: 0.2 }
        }
    }
}
```
