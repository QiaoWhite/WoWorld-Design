## 一、定位

本模块连接 Layer 1（文化概念空间）和 Layer 2（语言词汇）。

```
Layer 1: ConceptLocalId + CultureId → "这个概念是什么"
              ↓ 本模块
Layer 2: 特定语言中的词语 + 语法信息 → "这个概念在某种语言中怎么说"
```

**核心函数**: `f(ConceptLocalId, LanguageId) → LexicalForm`。不同概念类型的映射策略不同。

---

## 二、概念的四层分类 × 四种映射策略

| 层级 | 概念类型 | 数量 | 映射策略 | 存储 |
|------|---------|------|---------|------|
| **L1** | 原子概念 | ~250 | 预定义词库（TOML） | `core_concepts.toml` |
| **L2** | 复合概念 | 无限 | 组合函数（morphology规则） | `morphology.toml` + `special_forms.toml` |
| **L3** | 世界特定实体名 | 无限 | 参数化模板生成 | TextGenerator 模板 + 生成后缓存 |
| **L4** | 涌现概念 | 动态 | 描述性短语→社区固化→LMDB | `emergent_concepts` LMDB表 |

**重要**: L1 原子概念列表是**开发者与概念之间的约定**——不是全球客观真理。某些概念在某些文化中可能根本不存在（兽人文化没有"自由"）。原子概念的定义权在文化概念空间（002）。本模块只负责"当概念被识别出来后，在某种语言中怎么表达"。

## 三、核心数据结构

### 3.1 LexicalEntry —— 概念在一种语言中的全部词形

```rust
/// 一个概念在一种语言中的完整词汇条目
/// 存储在 TOML 中，运行时加载到 ConceptLexicon
pub struct LexicalEntry {
    /// 该概念在目标语言中的多种词形
    pub forms: Vec<WordForm>,
    /// 语法信息
    pub grammar: WordGrammar,
    /// 文化维度——默认值。可被文化概念空间（002）中的 CultureConceptDef 覆盖
    pub cultural: CulturalUsage,
}

/// 单个词形——附带使用条件
pub struct WordForm {
    pub word: String,
    pub register: Register,
    pub context: WordContext,
    pub frequency: f32,       // 0-1
    pub is_default: bool,
}

pub enum Register {
    Formal, Neutral, Casual, Intimate,
    Humble, Honorific, Poetic, Archaic,
}

pub struct WordContext {
    pub min_familiarity: Option<f32>,
    pub max_familiarity: Option<f32>,
    pub social_context: Option<SocialContext>,
}

impl WordContext {
    /// 检查此词形是否适用于给定的社交上下文
    pub fn matches(&self, ctx: Option<SocialContext>) -> bool {
        match (&self.social_context, ctx) {
            (None, _) => true,  // 无场景约束 → 总是适用
            (Some(expected), Some(actual)) => *expected == actual,
            (Some(_), None) => true,  // 有约束但调用方未指定场景 → 适用
        }
    }
}

pub enum SocialContext {
    Marketplace, Temple, Court, Battlefield,
    FamilyGathering, Funeral, Storytelling,
}

pub struct WordGrammar {
    pub part_of_speech: PartOfSpeech,
    pub grammatical_gender: Option<GrammaticalGender>,
    pub countability: Countability,
    /// 不规则形式——如英语 foot/feet、go/went
    pub irregular_forms: Vec<IrregularForm>,
}

/// 不规则形式——词干的特定语法形式变体
pub struct IrregularForm {
    pub form_type: IrregularFormType,  // Plural, PastTense, Comparative, ...
    pub surface_form: String,          // "feet", "went", "better"
}

pub enum IrregularFormType {
    Plural, PastTense, PastParticiple,
    Comparative, Superlative,
    ThirdPersonSingular, PresentParticiple,
}

pub enum PartOfSpeech {
    Noun, Verb, Adjective, Adverb, Preposition,
    Conjunction, Pronoun, Numeral, Classifier, Particle,
}

/// 概念的语义类别——用于歧义解析、组合优先级
/// 在 core_concepts.toml 中与词汇一同定义
pub enum ConceptCategory {
    Material,        // 铁/铜/木……——修饰成分
    Profession,      // 铁匠/裁缝/木匠……——核心成分（但本身可能由 Material+Agent 复合）
    Agent,           // 匠/师/工……——施事者后缀来源
    Place,           // 铺/店/坊……——场所后缀来源
    Tool,            // 锤/锯/针……
    Product,         // 剑/衣/桌……——产物
    Action,          // 打/切/缝……——动作
    Kinship,         // 父/子/妻……——亲属
    Entity,          // 通用实体——不属于以上任何类别
}

pub struct CulturalUsage {
    /// 禁忌等级 0-1（语言默认值——可被文化概念空间覆盖）
    pub taboo_level: f32,
    pub euphemism_for: Option<ConceptLocalId>,
    pub usage_note: Option<String>,
}

/// 文化对词汇使用的覆盖——存储在 cultures/{id}/concepts.toml 中
/// 当同一语言服务多个文化时使用（如中原通用语服务中原和北方边境）
pub struct CulturalOverride {
    pub concept_id: ConceptLocalId,
    pub culture_id: CultureId,
    pub taboo_level_override: Option<f32>,      // 覆盖默认禁忌等级
    pub preferred_register: Option<Register>,    // 该文化更偏好的语域
    pub disallowed_forms: Vec<String>,           // 该文化中禁止使用的词形
}
```

}

### 3.2 ConceptLexicon —— 运行时数据结构

```rust
/// 概念→词汇的完整索引
/// 启动时从 TOML 加载，存于 woworld_lang
pub struct ConceptLexicon {
    /// (ConceptLocalId, CultureId, LanguageId) → LexicalEntry
    /// CultureId 是键的一部分，因为 002 中 ConceptLocalId 可以在不同文化中重用
    entries: HashMap<(ConceptLocalId, CultureId, LanguageId), LexicalEntry>,
}

impl ConceptLexicon {
    /// 给定概念（附带其所属文化）和语言，选择最佳词形
    pub fn select_form(
        &self,
        concept_id: ConceptLocalId,
        culture_id: CultureId,
        language: LanguageId,
        register: Register,
        familiarity: f32,
        social_context: Option<SocialContext>,
    ) -> Option<&WordForm> {
        let entry = self.entries.get(&(concept_id, culture_id, language))?;

        // ① 匹配精确的 context + register
        // ② 回退到相同 register 的任意 context
        // ③ 回退到 Neutral register
        // ④ 回退到 is_default
        // ⑤ 回退到频率最高的 form

        entry.forms.iter()
            .find(|f| f.register == register && f.context.matches(social_context))
            .or_else(|| entry.forms.iter().find(|f| f.register == register))
            .or_else(|| entry.forms.iter().find(|f| f.register == Register::Neutral))
            .or_else(|| entry.forms.iter().find(|f| f.is_default))
            .or_else(|| entry.forms.iter()
                .max_by(|a, b| a.frequency.partial_cmp(&b.frequency).unwrap()))
    }
}
```

## 四、第一层：原子概念 → 预定义词库

### 4.1 什么是原子概念

人类社会中的**高频语义基元**——在大多数文化中存在对应概念（尽管切分方式不同）。数量有限但**不强制封闭**——Mod 可以在预留命名空间中添加。

### 4.2 原子概念与复合的边界

**判据**: 如果一个概念可以由两个更基础的概念通过 morphology 规则组合而成，且组合结果在至少 3 种模拟语言中是合法的自然表达——它就是复合概念，不是原子。

- ✅ 原子: `IRON`(铁), `SMITH`(工匠/打造者), `PLACE_SUFFIX`(场所)
- ❌ 非原子: `BLACKSMITH`(铁匠) = `compose_word([IRON, SMITH], zh_cn)` → "铁匠"
- ✅ 原子: `FATHER`, `MOTHER`, `SON`, `DAUGHTER`
- ❌ 非原子: 不存在于语言中的亲属细分

### 4.4 ConceptLocalId 分配规则

`ConceptLocalId(u16)` = 高 4 位命名空间 + 低 12 位序号。

- **0x0xxx**: 核心概念（开发者定义）。TOML 中 `id` 字段的第一位 hex = `0`。
- **0x1xxx-0xFxxx**: Mod 命名空间。每个 Mod 有自己的 4096 个槽位。
- 加载时：如果两个 TOML 文件中出现了相同 `id` → 后加载的 Mod 条目覆盖核心条目（Mod 优先）。启动日志中输出警告。

**映射**: TOML section 名（`SON`）→ `id` 字段（`0x0010`）→ `ConceptLocalId(0x0010)`。TOML section 名仅为人读标签——运行时使用 `id` 值。

## 五、第二层：复合概念 → morphology 规则 + special_forms 覆盖

### 5.1 concept_to_stem() —— 概念→词干的唯一入口

```rust
// woworld_lang/src/lexicon.rs

/// 从 LexicalEntry 中提取词干（用于复合构词）
/// 词干 = 该概念在 target 语言中的默认词形
pub fn concept_to_stem(
    concept_id: ConceptLocalId,
    culture_id: CultureId,       // ★ NE3/CD2修正: 概念所属文化——作为键的一部分
    language: LanguageId,
    lexicon: &ConceptLexicon,
) -> Option<String> {
    lexicon.entries.get(&(concept_id, culture_id, language))
        .and_then(|entry| {
            entry.forms.iter().find(|f| f.is_default)
                .or_else(|| entry.forms.first())
        })
        .map(|form| form.word.clone())
}
```

### 5.2 compose_word() —— 二元组合（修正版）

```rust
// woworld_lang/src/morphology.rs

/// 将两个原子概念组合为复合词
pub fn compose_word(
    modifier: ConceptLocalId,
    head: ConceptLocalId,
    culture_id: CultureId,
    language: LanguageId,
    lexicon: &ConceptLexicon,
    morphology: &MorphologyRegistry,
) -> String {
    let modifier_stem = concept_to_stem(modifier, culture_id, language, lexicon)
        .unwrap_or_else(|| "[?]".into());
    let head_stem = concept_to_stem(head, culture_id, language, lexicon)
        .unwrap_or_else(|| "[?]".into());

    let lang_config = morphology.language_config(language);
    let raw = match lang_config.compound_order {
        CompoundOrder::ModifierHead => format!("{}{}", modifier_stem, head_stem),
        CompoundOrder::HeadModifier => format!("{}{}{}", head_stem, lang_config.compound_separator, modifier_stem),
        CompoundOrder::HeadOnly => head_stem,
        CompoundOrder::Separate => format!("{} {}", modifier_stem, head_stem),
        CompoundOrder::PrepositionalPhrase => format!("{}{}{}", modifier_stem, lang_config.possessive_particle, head_stem),
    };

    // 检查 special_forms 覆盖（正向+反向）
    if let Some(special) = morphology.lookup_special_form(modifier, head, language) {
        return special;
    }

    raw
}

/// 将 3+ 概念的组合树拆分为二元组合序列。
///
/// 歧义解析规则: 检查 morphology.toml 中注册的后缀类概念。
/// 如果某个组件在 morphology 中被标记为 suffix（PLACE_SUFFIX/AGENTIVE_SUFFIX），
/// 它总是绑定到前面已组合的整个前缀——而非单独绑定到紧邻词。
///
/// 判据来源: morphology.toml 的 place_suffixes 和 agentive_suffixes 表。
/// compose_multi_word 不查询 lexicon 的语义类别——只查询 morphology。
pub fn compose_multi_word(
    components: &[ConceptLocalId],
    culture_id: CultureId,
    language: LanguageId,
    lexicon: &ConceptLexicon,
    morphology: &MorphologyRegistry,
) -> String {
    assert!(components.len() >= 2);

    let lang_config = morphology.language_config(language);

    // 寻找后缀组件的索引——检查 morphology 的注册表
    let suffix_idx = components.iter().position(|c| {
        morphology.is_registered_suffix(*c, language)
        // is_registered_suffix 检查:
        //   place_suffixes 表 + agentive_suffixes 表中是否包含此 ConceptLocalId
    });

    match suffix_idx {
        Some(idx) if idx > 0 => {
            // 分治: [前缀组...] + [后缀]
            let prefix: Vec<ConceptLocalId> = components[..idx].to_vec();
            let suffix = components[idx];

            let prefix_word = if prefix.len() == 1 {
                concept_to_stem(prefix[0], culture_id, language, lexicon)
                    .unwrap_or_else(|| "[?]".into())
            } else {
                compose_multi_word(&prefix, culture_id, language, lexicon, morphology)
            };

            let suffix_stem = concept_to_stem(suffix, culture_id, language, lexicon)
                .unwrap_or_else(|| "[?]".into());

            match lang_config.compound_order {
                CompoundOrder::ModifierHead => format!("{}{}", prefix_word, suffix_stem),
                CompoundOrder::Prefix => format!("{}{}", suffix_stem, prefix_word),
                CompoundOrder::Separate => format!("{} {}", prefix_word, suffix_stem),
                _ => format!("{}{}", prefix_word, suffix_stem),
            }
        }
        _ => {
            // ★ NE4修正: 递归应用 compose_word，而非简单空格拼接
            let mut result = compose_word(
                components[0], components[1],
                culture_id, language, lexicon, morphology
            );
            for i in 2..components.len() {
                // 每步都作为二元组合: (之前的产物, 新组件)
                result = compose_word_pair(
                    &result, components[i],
                    culture_id, language, lexicon, morphology
                );
            }
            result
        }
    }
}

/// 将一个已组合的字符串与一个新概念做二元组合
/// 用于 3+ 概念的递归组合路径
fn compose_word_pair(
    prefix: &str,
    next: ConceptLocalId,
    culture_id: CultureId,
    language: LanguageId,
    lexicon: &ConceptLexicon,
    morphology: &MorphologyRegistry,
) -> String {
    let next_stem = concept_to_stem(next, culture_id, language, lexicon)
        .unwrap_or_else(|| "[?]".into());
    let lang_config = morphology.language_config(language);

    match lang_config.compound_order {
        CompoundOrder::ModifierHead => format!("{}{}", prefix, next_stem),
        CompoundOrder::HeadModifier => format!("{}{}{}", next_stem, lang_config.compound_separator, prefix),
        _ => format!("{} {}", prefix, next_stem),
    }
}
```

**歧义解析示例**:
```
components = [IRON, SMITH, PLACE_SUFFIX]
suffix_idx = 2 (PLACE_SUFFIX)
prefix = [IRON, SMITH] → compose_multi_word → "铁匠"
suffix = "铺" (中原语中 PLACE_SUFFIX for blacksmith)
→ "铁匠铺" ✓

components = [IRON, SMITH, AGENTIVE_SUFFIX]
suffix_idx = 2
→ "铁匠" ✓ (不是 "铁匠人")

components = [GOLD, SMITH]
suffix_idx = None (SMITH 不是后缀类)
→ compose_word(GOLD, SMITH) → "金匠" ✓
```

### 5.3 special_forms —— 惯用形式覆盖

```rust
// MorphologyRegistry 中
pub fn lookup_special_form(
    modifier: ConceptLocalId,
    head: ConceptLocalId,
    language: LanguageId,
) -> Option<String> {
    // 正向: (modifier, head)
    if let Some(s) = self.special_forms.get(&(modifier, head, language)) {
        return Some(s.clone());
    }
    // ★ NG3修正: 反向: (head, modifier)
    if let Some(s) = self.special_forms.get(&(head, modifier, language)) {
        return Some(s.clone());
    }
    None
}
```

```toml
# 规则产出 "blacksmith shop" → 惯用 "smithy"

[[special_form]]
components = [0x0420, 0x0315]      # IRON + SMITH
language = "northern"
actual = "blacksmith"
# 规则产出 "iron smith" → 惯用 "blacksmith"（历史上 iron→black 的语义演变）

[[special_form]]
components = [0x0315, 0x0602]      # SMITH + PLACE_SUFFIX → "blacksmith shop"
language = "zh_cn"
actual = "铁匠铺"
# 中原语的规则产出和惯用形式一致——此条实际上不需要
# 列在此处作为记录和 Mod 参考

[[special_form]]
components = [0x0500, 0x0601]      # APOTHECARY (如果存在) + PLACE_SUFFIX
language = "northern"
actual = "apothecary"
# 规则产出 "apothecary shop" → 惯用 "apothecary"（同词）
```

### 5.4 回退策略与已知局限

```
compose_word 执行顺序:
  ① concept_to_stem(modifier) + concept_to_stem(head)
  ② morphology 规则组合 → raw_string
  ③ special_forms 检查 → 如有覆盖则返回 actual
  ④ 如果 concept_to_stem 返回 None → raw_string = "[?]"
  ⑤ 最终回退: 描述性短语——使用目标语言的语法规则生成
     (通过 TextGenerator 的 description 模板)
```

**已知局限**: `compose_multi_word` 的平面概念列表无法编码语义关系。
`[IRON, SMITH, TOOL]` 是"铁匠的工具""用铁做的匠人工具"还是"冶铁匠用的工具"？
答案取决于上下文——平面列表丢失了这种信息。

**缓解策略**:
- 后缀类概念（PLACE_SUFFIX/AGENTIVE_SUFFIX）通过 morphology 注册表定位 → 正确绑定
- 3+ 概念组合中不含后缀时 → 走递归 `compose_word_pair` 路径
- 结果不自然 → 在 `special_forms.toml` 中登记惯用形式
- 完全无匹配 → TextGenerator 的通用描述模板作为最终回退
- 如果某个复合模式在高频使用 → 加入涌现命名管道，固化后走 LMDB 词库

```rust
// woworld_lang/src/transliteration.rs

/// 将名字从一种文化的语音系统音译为另一种
/// 基于 TOML 定义的音素映射表
pub fn transliterate(
    name: &str,
    source_culture: CultureId,
    target_culture: CultureId,
    mapping: &TransliterationRegistry,
) -> String {
    // ① 如果 source 和 target 使用相同音素库存 → 直接返回原名字
    // ② 否则: 逐音素映射 + 目标语言音系约束（如汉语无辅音丛 → 在辅音间插入默认元音）
```

## 七、第四层：涌现概念 → 生命周期

### 7.1 四个阶段（修正版）

```
阶段 0: 新模式出现
  客观世界出现新的权力模式/经济关系 → PatternSignature 已存在
  classify_pattern() 返回空 → 当前文化概念空间中没有匹配概念

阶段 1: 匿名存在
  概念没有 ConceptLocalId——仅以 PatternSignature 作为标识
  NPC 只能用描述性短语表达——通过 TextGenerator 的 generic_description 模板
  每次描述可能不同——因为没有固化名称
  (ConceptLocalId 是固化后才分配的。匿名阶段用 PatternSignature 做键)

阶段 2: 涌现命名
  触发条件: 同一聚落中对该 pattern_sig 的提及次数 > EMERGENCE_FREQUENCY_THRESHOLD
  多个 NPC 各自尝试缩略或简化描述 → 产生候选名称
  候选竞争 → 最频繁出现的缩略形式胜出 → 记录到 LMDB emergent_concepts 表

阶段 3: 聚落固化
  使用该命名的 NPC 数量 > ADOPTION_THRESHOLD
  → 名称进入聚落级 emergent_concepts LMDB 表
  → 新 NPC 可以通过 concept_exposure_buffer 习得这个概念
  → 习得后 effective_concept_space() 中包含该概念

阶段 4: 跨代固化（仅在该文化持续使用超过 3 代时）
  → 写入该文化的 LMDB permanent_concepts 表
  → 新存档启动时从 LMDB 加载（而非 TOML）
  → 可被手动提升到 TOML 以用于后续开发
  → 但**永远不从 Rust 代码自动写入 TOML 文件**
```

### 7.2 实现

```rust
// woworld_lang/src/emergent_naming.rs

/// UtteranceHistory — 聚落级 Utterance 抽样历史
/// ★ 存储优化: 仅保留包含涌现概念引用（pattern_sig.is_some()）的话语。
/// 常规对话（已命名概念的 Utterance）不写入此历史。
/// 这使存储量从 ~4.5MB/聚落降至 <50KB/聚落（抽样率 ~1%）。
/// 
/// 存储在 LMDB 中。key: (settlement_id, day_index) → Vec<UtteranceLogEntry>
pub struct UtteranceHistory {
    entries: Vec<UtteranceLogEntry>,
    retention_days: u32,  // [TUNING] 90
    /// 写入过滤: 只有 pattern_sig.is_some() 的 Utterance 才记录
    sampled_only: bool,   // 始终为 true
}

pub struct UtteranceLogEntry {
    pub pattern_sig: PatternSignature,           // 提及的涌现概念模式（非 None——过滤保证）
    pub descriptive_phrase: Option<String>,
    pub language: LanguageId,
    pub speaker_id: EntityId,
    pub timestamp: GameInstant,
}

pub fn check_emergent_naming(
    pattern_sig: PatternSignature,
    settlement_id: SettlementId,
    culture_id: CultureId,
    utterance_history: &UtteranceHistory,
    language: LanguageId,
    morphology: &MorphologyRegistry,
) -> Option<EmergentName> {
    let mentions = utterance_history.count_mentions(pattern_sig, RETENTION_DAYS);  // [TUNING] 90

    if mentions < EMERGENCE_FREQUENCY_THRESHOLD { return None; }  // [TUNING] 50

    let phrases: Vec<String> = utterance_history
        .descriptive_phrases_for(pattern_sig)
        .collect();

    let candidates = morphology.abbreviate_phrases(&phrases, language);
    let best = most_frequent(&candidates);

    // NG2修正: adoption = 使用过与 best 相同或相似缩略的独立说话者数
    // "相似" = 编辑距离 ≤ 2 或 子串匹配
    let adoption = utterance_history.count_speakers_using_phrase(&best, 2);
    // count_speakers_using_phrase: 统计有多少个不同的 speaker_id
    // 在 RETENTION_DAYS 内使用了编辑距离 ≤ threshold 的短语

    if adoption >= ADOPTION_THRESHOLD {  // [TUNING] 10
        // 分配 ConceptLocalId
        let new_id = allocate_emergent_concept_id(culture_id);  // LMDB 中递增分配

        Some(EmergentName {
            pattern_sig,
            concept_id: new_id,
            name: best,
            settlement_id,
            culture_id,
            adoption_count: adoption,
            first_used_at: GameInstant::now(),
            stage: EmergentStage::SettlementConsolidated,
        })
    } else {
        None
    }
}
```

### 7.3 缩略规则——每种语言独立定义

```toml

# 从描述性短语的每个内容词中提取第一个字/词素
# "三家共同耕种的那块田" → 提取: "三""家""田" → 组合: "三家田"

```

```toml
# 从描述性短语中提取首字母或裁剪
```

### 6.5 数据加载——启动顺序

所有词汇/构词/语法数据在启动时一次性加载到 woworld_lang 的内存中：

```

2. languages/*/morphology.toml  → MorphologyRegistry
3. languages/*/grammar.toml     → GrammarRegistry
4. languages/*/special_forms.toml → 合并入 MorphologyRegistry
```

## 八、语法系统——与 morphology 的分工

### 8.1 morphology vs grammar 的边界

| morphology.toml 管 | grammar.toml 管 |
|-------------------|----------------|
| 构词: 复合规则、后缀、特殊形式 | 造句: 语序、一致关系、时体态 |
| 词形: 单数/复数(不规则)、比较级(不规则) | 短语级: 冠词、量词选择、介词语序 |
| 缩略规则 | 从句结构、疑问形式 |
| 数字格式 | 敬语系统的语法实现 |

**怎么配合**：TextGenerator 的片段库（~430条/语言）已经渲染了正确的语法——片段是由人用该语言书写的完整句子模板。morphology/grammar 系统只在以下情况介入：
1. 动态生成的复合词插入片段槽位时需要确认语法一致（如性别/数）
2. 涌现命名或描述性短语需要生成不在片段库中的自由文本

### 8.2 grammar.toml 概要

```toml
# languages/zh_cn/grammar.toml

[grammar]
word_order = "SVO"
morphology_type = "Isolating"
has_grammatical_gender = false
has_honorifics = true
writing_direction = "LeftToRight"
is_tonal = true

[temporal]
# 时态: 中原语通过时间词+体标记表达，无动词变位
past_marker = "了"       # 完成体
progressive_marker = "在"  # 进行体
future_marker = "会"     # 将来

[agreement]
# 汉语无一致关系
enabled = false

[classifiers]
# 量词系统
enabled = true
default_classifier = "个"
```

> **对齐**: 以上参数与 [[../语言表达/002-语言与文字系统|语言表达 002]] 中 `LanguageAttributes` 的定义一致。grammar.toml 是其详细扩展。

### 8.3 语法一致检查——复合词插入片段时

当复合词（通过 morphology 动态生成）被插入片段库模板的槽位时，需要确认该词的语法属性与模板中其他词一致：

```rust
// woworld_lang/src/grammar.rs

/// 检查动态生成的词与模板槽位的语法兼容性
/// 仅在语法性别/数的语言中需要（英语/汉语不需要）
pub fn agreement_check(
    word: &str,                        // 复合词
    word_entry: &LexicalEntry,         // 复合词核心成分的词条
    slot_gender: Option<GrammaticalGender>,  // 模板槽位要求的性别（None=不要求）
    slot_number: Option<GrammaticalNumber>,  // 模板槽位要求的数（None=不要求）
    language: LanguageId,
    grammar: &GrammarRegistry,
) -> AgreementResult {
    let lang = grammar.get(language);

    // 汉语、英语（语法性别=false）→ 直接通过
    if !lang.has_grammatical_gender {
        return AgreementResult::Compatible;
    }

    // 检查性别一致
    if let (Some(expected), Some(actual)) = (slot_gender, word_entry.grammar.grammatical_gender) {
        if expected != actual {
            return AgreementResult::GenderMismatch {
                expected,
                actual,
                suggestion: morphology_correct_gender(word, expected, language),
            };
        }
    }

    AgreementResult::Compatible
}
```

**注意**: 片段库中的模板是人写的——语法已经正确。此检查仅在**动态生成的复合词**插入片段槽位时触发——这是低频操作，性能可忽略。大部分情况下直接通过（汉语/英语无语法性别，不需要检查）。

### 9.3 跨文化渲染

```rust
pub fn render_name(
    name: &PersonalName,
    observer_culture: CultureId,
    familiarity: f32,
    lexicon: &ConceptLexicon,
) -> String {
    if observer_culture == name.name_culture {
        match familiarity {
            f if f > 0.7 => name.given_name.clone(),
            f if f > 0.3 => format!("{} {}",
                name.family_name.as_deref().unwrap_or(""), &name.given_name),
            _ => name.epithet.clone()
                .unwrap_or_else(|| format!("{} {}",
                    name.family_name.as_deref().unwrap_or(""), &name.given_name)),
        }
    } else {
        transliterate(&name.given_name, name.name_culture, observer_culture)
    }
}
```
