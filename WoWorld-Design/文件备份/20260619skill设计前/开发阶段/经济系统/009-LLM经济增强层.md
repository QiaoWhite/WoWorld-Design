## 一、核心定位——经济解说员，非经济模拟器

### 1.1 铁律

**LLM不参与任何经济计算。经济系统100%在Rust确定性规则层运转。**

```
经济规则层（Rust）         LLM增强层（可选）
───────────────────── vs ─────────────────────
价格从订单簿撮合中涌现      → 把价格变化包装成有因果、有人物、有情绪的叙事
NPC用BigFive+情绪+记忆做决策 → 让NPC的对话听起来更自然
交易记录写入数据库          → 让历史记载读起来像人类写的编年史
```

LLM不替换任何经济机制。LLM只改善信息的"包装"——把结构化数据翻译成人类可读的自然语言。

**类比**：经济系统是气象站（测量温度、湿度、气压），LLM是天气预报员（"明天会是个阴冷潮湿的日子，出门记得带伞"）。气象站的数据不变——预报员只是让它更可感。

### 1.2 LLM参与的条件

三个条件全部满足时LLM才介入：

| 条件 | 说明 |
|------|------|
| 玩家启用了LLM增强 | `LlmSceneConfig.master_switch = true`，且对应场景开关为 `true` |
| 经济事件需要叙事化描述 | 不是每笔交易都走LLM——只有跨越"有意义阈值"的事件 |
| LLM后端可用 | 本地/云端任一后端在线，且未超过速率限制 |

**LLM失败/关闭/不可用 → 无缝回退模板。** 模板管道始终存在且完整——LLM是装饰器，不是替代品。

### 1.3 有意义阈值——不是每笔交易都惊动LLM

```rust
/// 决定一个经济事件是否值得LLM叙事化
fn is_llm_worthy_event(event: &EconomicEvent) -> bool {
    match event.event_type {
        // 总是值得叙事化
        EconomicEventType::MarketCrash { .. } => true,
        EconomicEventType::Famine { .. } => true,
        EconomicEventType::MonopolyDetected { .. } => true,
        EconomicEventType::TradeRouteOpened { .. } => true,
        EconomicEventType::TradeRouteBlocked { .. } => true,

        // 超过阈值才值得
        EconomicEventType::PriceSpike { change_pct, .. } => change_pct > 25.0,
        EconomicEventType::PriceCrash { change_pct, .. } => change_pct > 25.0,
        EconomicEventType::NewTraderEmerged { market_impact, .. } => market_impact > 0.3,
        EconomicEventType::StorefrontClosed { historical_significance, .. } => {
            historical_significance > 0.5
        }

        // 日常波动不值得
        _ => false,
    }
}
```

经济系统的每笔日常交易（日均数千笔）不走LLM。只有跨越阈值的经济事件——价格飙升25%+、市场崩溃、饥荒、商路开通/阻断——才触发LLM叙事化。

## 二、五大增强方向

### 2.1 经济事件叙事化

**无LLM（模板输出）**：
> 「铁炉城」铁矿石价格上涨36%（50→68铜/块）。可能原因：北山矿难、战争流言。

**有LLM（叙事化）**：
> 老铁匠张铁心今天早上站在铁炉城中央市场的矿石堆前，摇了摇头。"我打了四十年铁，"他对旁边的学徒说，"从没见过铁矿石贵到这个地步。上个月还能用五十个铜板买一块好矿，现在要六十八了。"
>
> 北山的矿道塌了——这是集市上每个人都在重复的话。有人说是矿主偷工减料，有人说是地下的什么东西醒了。不管真相是什么，矿石少了，铁匠铺的炉子却还要烧。而在城门外五十里，战争的流言正像乌云一样聚拢——每个领主都在囤铁。老铁匠叹了口气，把最后三块矿石搬回了铺子里。他不知道明天还能不能买到。

LLM从以下结构化数据综合出叙事：

| 数据来源 | 注入内容 |
|---------|---------|
| 交易记录（TradeRecord） | 价格从50→68，交易量下降 |
| 事件因果链（EconomicEvent.cause_chain） | 矿难→供给减少，战争流言→需求增加 |
| 受影响NPC的记忆（MemoryStore） | 黑smith_zhang的known_prices中有旧价格记忆（锚定效应） |
| NPC当前情绪（EmotionState） | 黑smith_zhang.pleasure 下降，anger 轻微上升 |
| NPC关系数据 | 黑smith_zhang对矿主trust下降（如果他认为矿难是偷工减料造成的） |

**LLM不生成价格数据、不判断因果关系、不修改NPC状态。** LLM只把这些已有的结构化数据编织成自然语言叙事。

### 2.2 商人NPC对话增强

**无LLM（模板）**：
> 商人李：[选项1] 我想买东西。[选项2] 我想卖东西。[选项3] 有什么推荐的？

**有LLM（商人自然语言建议）**：
> 商人李四海擦了擦柜台，凑近了点："客官，这批铁矿石我劝你别买。南边青石镇刚开了个新矿，我在那边的朋友前天带信来说，第一批矿石下周就能到货——到了以后，这边的价格至少便宜两成。你要是能等，就等等。"
>
> （这句话基于真实数据：李四海的 known_prices 中有青石镇矿的消息，source_confidence 高于 0.7，运输时间估计 5-7 天。不是 LLM 编造的。）

LLM被注入的数据结构：

```json
{
  "npc": "merchant_li_sihai",
  "player_inquiry": "想买铁矿石",
  "current_market": {
    "item": "iron_ore",
    "price_copper": 68,
    "trend": "rising",
    "volatility": 0.23
  },
  "merchant_known_alternatives": [
    {
      "market": "greenspire_mine",
      "item": "iron_ore",
      "known_price": 45,
      "source_confidence": 0.75,
      "estimated_arrival_days": 6,
      "source_npc": "miner_wang_laojiu",
      "relationship": "friend"
    }
  ],
  "merchant_personality": {
    "agreeableness": 0.72,
    "extraversion": 0.65,
    "honesty_bias": 0.8
  },
  "relationship_with_player": {
    "trust": 0.55,
    "familiarity": 0.3,
    "past_trades": 4
  }
}
```

**关键保证**：商人给出的建议（"青石镇新矿→下周到货→价格至少便宜两成"）全部来自 `merchant_known_alternatives` 中的结构化数据。LLM负责措辞——信息真实性由Rust侧保证。

**人格驱动的对话风格差异**：

| 人格维度 | 对对话风格的影响（LLM接收的参数） |
|---------|------------------------------|
| 高agreeableness | 主动给出省钱建议，"我不忍心看你多花钱" |
| 低agreeableness | 可能不告知替代来源，"买不买随你" |
| 高extraversion | 话多，附带市场八卦 |
| 高conscientiousness | 严谨，"下周到货"→"按我朋友的信，快则五天慢则七天" |
| 高neuroticism | 焦虑语气，"这行情我晚上都睡不好" |
| 低trust（对玩家） | 保留关键信息，不推荐最优选择 |
| 高trust（对玩家） | 透露"内部消息"——包括地下市场的信息 |

### 2.3 经济谣言增强

**无LLM（模板——语言表达系统信息传播通道）**：
> "听说铁炉城的铁矿石卖到六十八铜了。"

**有LLM（谣言叙事增强）**：
> "诶，你听说了吗？铁炉城那边出大事了——老张家铁匠铺的铁矿石，一块要六十多个铜板！"说话的人压低了声音，眼睛左右扫了扫。"我表哥的连襟在那边做矿工，他说北山的矿道不是塌了——是被什么东西从上往下挖穿了。矿主把剩下的矿石全囤起来了，等再涨一波再卖。"他顿了顿，"要我说，现在谁手里有铁，谁就是大爷。"

LLM注入失真算子数据：

```json
{
  "payload": "iron_ore_price_68_copper_at_ironforge",
  "propagation_count": 3,
  "applied_distortions": [
    {"type": "ScaleDistortion", "magnitude": 1.2, "description": "价格被夸大了10%"},
    {"type": "DetailLoss", "magnitude": 0.4, "description": "矿难原因丢失"},
    {"type": "EmotionalAmplification", "magnitude": 0.7, "description": "恐慌情绪放大"},
    {"type": "AttributionShift", "magnitude": 0.5, "description": "矿主被暗示囤积居奇"},
    {"type": "Moralization", "magnitude": 0.6, "description": "附加了道德评价"}
  ],
  "rumor_stage": "Sustained",
  "nonverbal_cues": ["压低声音", "左右扫视", "停顿强调"]
}
```

传播次数越多，失真越严重。LLM根据失真算子的类型和幅度生成对应的叙事化扭曲——不是LLM自己添加扭曲，而是把已经由Rust侧计算好的失真参数视觉化。

### 2.4 玩家商业咨询

**无LLM**：玩家手动打开多个市场的订单簿，对比价格，自己计算套利空间。

**有LLM（NPC商业顾问）**：玩家可以找到有商业头脑的NPC，询问"现在做什么生意最赚钱？"NPC基于自己的known_prices + EconomicCognition + 与玩家的关系，用自然语言给出分析。

```json
  "cognition": {
    "financial_literacy": 0.85,
    "market_understanding": 0.9,
    "market_search_breadth": 4
  },
  "known_opportunities": [
    {
      "item": "silk",
      "buy_market": "southport_docks",
      "buy_price": 120,
      "sell_market": "ironforge_noble_district",
      "sell_price": 350,
      "transport_cost": 45,
      "risk_factors": ["bandits_on_route", "weather_season"]
    },
    {
      "item": "iron_ore",
      "buy_market": "greenspire_mine",
      "buy_price": 45,
      "sell_market": "ironforge_central",
      "sell_price": 68,
      "transport_cost": 10,

    }
  ],
  "relationship_with_player": {
    "trust": 0.7,

  },
  "personality": {
    "agreeableness": 0.8,
    "openness": 0.75
  }

```

LLM综合这些数据生成自然语言分析：

> 退休商人周老海摸了摸胡子："你想赚钱？那我告诉你两条路。第一条，现在跑丝绸——南港码头收一百二十铜，运到铁炉城贵族区能卖三百五十。但这条路不好走，中间有段路闹山贼，而且这个季节雨多，丝绸潮了就废了。"
>
> "第二条，铁矿——青石镇那边新矿，四十五铜一块，拉到铁炉城就是六十八。赚得少，但路近，安稳。"
>
> "要我选的话，"他往椅背上一靠，"我年轻时跑丝绸。现在这把老骨头嘛……铁矿吧。看你胆量了。"

**LLM不决定NPC推荐什么路线。** NPC的推荐倾向由Rust侧决定（risk_tolerance从人格派生 + relationship_trust决定是否诚实推荐）。LLM只负责措辞。

### 2.5 历史经济事件记录

**无LLM（模板——历史系统记录）**：
> 纪元1247年·秋，铁炉城饥荒。粮价上涨320%（8→34铜/公斤）。受影响NPC：12,847人。死亡：2,341人。

**有LLM（叙事化历史记载）**：
> 《铁炉城志·饥荒卷》
>
> 纪元一千二百四十七年的秋天，铁炉城记住的不是落叶的颜色，而是饥饿的形状。春天的雨来得太晚了——晚了整整四十天。夏天的太阳却来得太早，把刚冒芽的麦苗晒成了枯草。
>
> 市场里的粮价是从八铜开始涨的。先是十铜，然后十五。到了秋收前的最后一个月，一公斤粗麦卖到了三十四铜——四倍于正常年景。面包师傅关了铺子，不是因为不想做，是因为没有面粉。矿工把铁镐卖了买粮食。铁匠铺的炉子冷了——吃不饱的人打不动铁。
>
> 到第一场雪落下来的时候，两千三百四十一人没有等到下一年的春天。

LLM从以下数据综合出历史叙事：

| 数据来源 | 注入内容 |
|---------|---------|
| 价格历史（last_price时间序列） | 粮价从8→34的时间线 |
| 天气记录（WeatherSample历史） | 春雨迟到40天，夏旱 |
| 事件因果链 | 晚雨→减产→供给崩塌→价格螺旋 |
| 受影响NPC的集体数据 | 受影响人数、死亡人数、职业分布 |
| 多NPC的记忆摘录 | "面包师傅关了铺子"来自面包师NPC的记忆事件 |
| 传闻数据（信息传播通道） | 多个NPC记忆中的市场恐慌描述 |

## 三、LLM与确定性数据的交互

### 3.1 数据注入格式——EconomicNarrativePayload

LLM永远不直接读取经济数据库。经济系统生成 `EconomicNarrativePayload` 结构体，由LLM增强层的 prompt builder 转化为LLM prompt。

```rust
/// 经济叙事负载——从经济系统注入LLM的结构化数据
/// LLM不读取任何经济数据库——只消费此结构体
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EconomicNarrativePayload {
    /// 事件类型
    pub event_type: EconomicEventType,

    /// 核心价格数据
    pub item: ItemDefId,
    pub item_display_name: String,
    pub market: MarketId,
    pub market_display_name: String,

    /// 价格变化
    pub old_price_copper: u32,
    pub new_price_copper: u32,
    pub change_pct: f32,
    pub direction: PriceDirection,

    /// 检测到的原因（经济系统的事件因果链——非LLM推断）
    pub detected_causes: Vec<EconomicCause>,

    /// 受影响的关键NPC快照（非完整NpcData——仅叙事所需）
    pub affected_npcs: Vec<AffectedNpcSnapshot>,

    /// 叙事风格提示
    pub narrative_tone: NarrativeTone,

    /// 模板回退文本（LLM不可用时直接使用）
    pub fallback_text: String,

    /// 模板回退文本（详细版）
    pub fallback_text_detailed: String,
}

/// 受影响NPC的快照——仅包含叙事所需的最少数据
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct AffectedNpcSnapshot {
    pub npc_id: NpcId,
    pub display_name: String,
    pub occupation: String,
    /// NPC对此事件的主观情绪标签
    pub sentiment: SentimentLabel,
    /// 模板文本（无LLM时使用的版本）
    pub template_quote: String,
    /// NPC的已知价格历史（供LLM构建"我打了四十年铁…"风格的叙事）
    pub relevant_price_memory: Vec<PriceMemorySnippet>,
}

/// 价格记忆片段
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PriceMemorySnippet {
    pub item: String,
    pub price_copper: u32,
    pub days_ago: u32,
    pub memory_emotion_tag: String,
}

/// 经济原因——由经济系统的事件因果链推断
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum EconomicCause {
    SupplyShock { source: String, magnitude: f32 },
    DemandSurge { source: String, magnitude: f32 },
    TradeRouteBlocked { route: String, blockage_reason: String },
    TradeRouteOpened { route: String },
    WeatherImpact { weather_event: String, affected_crops: Vec<String> },
    PolicyChange { authority: String, policy: String },
    RumorsOrPanic { description: String, intensity: f32 },
    MonopolyAction { actor: String, action: String },
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum PriceDirection { Rising, Falling, Stable, Volatile }

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum SentimentLabel { Worried, Angry, Hopeful, Desperate, Resigned, Opportunistic, Confused, Calm }

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum NarrativeTone {
    MatterOfFact,   // 平实
    Dramatic,       // 戏剧化（重大事件）
    Folklore,       // 民间传说风格（谣言）
    Personal,       // 个人视角（商人对话）
    Historical,     // 历史编年体
}
```

### 3.2 LLM提示构建——经济场景专用

经济系统的LLM提示由语言表达模块的 `PromptLibrary` 构建。经济系统提供以下专用提示模板：

```
你是WoWorld中一个经济事件的叙事化引擎。

你正在将结构化的经济数据转化为自然语言叙事。请严格遵循以下约束：

[约束]
1. 不得编造任何经济事实——所有数字、原因、事件必须来自下方提供的数据
2. 不得修改价格、数量、时间等任何数值
3. 不得添加数据中不存在的人物、地点、事件
4. 你可以：
   - 为NPC添加符合其人格和情绪的合理对话和内心活动
   - 添加符合世界设定的环境描写
   - 用比喻和日常语言解释经济概念（如"供不应求"→"矿石少了，铁匠铺的炉子却还要烧"）
   - 以NPC的视角表达其主观感受
5. 叙事长度：{length_constraint}
6. 叙事风格：{narrative_tone}
7. 文化语境：这是一个剑与魔法的中世纪奇幻世界——避免现代经济学词汇
8. 受众：玩家——叙事应该让玩家感到这个世界是"活的"，NPC是"真实的人"

[数据]
{serialized_payload_json}
```

**关键**：LLM收到的数据已经是 JSON 字符串——LLM不调用任何经济系统API。这是"数据注入"而非"LLM查询"。

### 3.3 安全性——LLM不能修改的数据

```rust
/// LLM叙事输出的安全校验
/// 确保LLM没有"编造"经济事实
fn validate_llm_narrative(
    llm_output: &str,
    original_payload: &EconomicNarrativePayload,
) -> Result<(), NarrativeValidationError> {
    // 校验1: LLM输出不能包含不同于原始数据的数字
    let numbers_in_output: Vec<u32> = extract_numbers(llm_output);
    let numbers_in_payload: HashSet<u32> = extract_payload_numbers(original_payload);

    for n in &numbers_in_output {
        // 允许LLM使用近似表达（"六十多铜" vs 精确的68）
        if !numbers_in_payload.contains(n) && !is_approximate_match(*n, &numbers_in_payload) {
            return Err(NarrativeValidationError::HallucinatedNumber(*n));
        }
    }

    // 校验2: LLM不能引入payload中不存在的地名/NPC名
    // （允许提到玩家已知的上下文地名——如邻城——但需标记为"LLM推断"）

    // 校验3: 核心经济事实方向不得扭曲
    // 价格涨了→LLM不能写成跌了

    Ok(())
}
```

校验不通过 → 丢弃LLM输出 → 使用模板回退文本。校验开销 <50μs（字符串匹配），不影响性能。

## 四、回退机制

### 4.1 模板覆盖100%的经济事件类型

模板管道覆盖所有经济事件类型，LLM不可用时自动回退。模板不是简陋的占位符——精心设计的模板在无LLM时也能提供足够的叙事信息：

```
价格飙升模板：
"{market}的{item}价格上涨{change_pct}%（{old}→{new}铜/{unit}）。
 可能原因：{causes}。
 受影响的NPC：{affected_count}人。市场情绪：{dominant_sentiment}。"

饥荒模板：
"纪元{year}年·{season}，{region}发生饥荒。
 粮价上涨{change_pct}%（{old}→{new}铜/公斤）。
 起因：{weather_event}导致{impact}。
 受影响：{affected_population}人。死亡：{deaths}人。
 关键回忆：{npc_snippets}。"

商业建议模板：
"{merchant_name}建议{direction}：
 {item}——{reason}。
 买价{price_copper}铜/{unit}（{market_name}），
 运到{target_market}可卖{target_price}铜/{unit}。
 运输成本{transport_cost}铜。风险：{risks}。"
```

### 4.2 回退触发条件

| 条件 | 回退行为 |
|------|---------|
| `LlmSceneConfig.master_switch = false` | 所有经济场景走模板 |
| `LlmSceneConfig.economic_events = false` | 经济事件走模板 |
| `LlmSceneConfig.merchant_dialogue = false` | 商人对话走模板 |
| 所有LLM后端不可用 | 全局走模板 |
| 速率限制触发 | 本次请求走模板，3秒后可重试 |
| LLM返回超时（>2s） | 走模板，记录超时次数 |
| 安全校验不通过 | 走模板（丢弃LLM输出），记录校验失败 |
| LLM输出格式不可解析 | 走模板，记录格式错误 |

### 4.3 渐进回退——部分增强可用时

```rust
/// 经济叙事的分层回退策略
fn get_economic_narrative(
    payload: &EconomicNarrativePayload,
    llm_config: &LlmSceneConfig,
    orchestrator: &LlmOrchestrator,
) -> NarrativeOutput {
    // 第1层: 尝试LLM完整叙事
    if llm_config.economic_events && orchestrator.is_available("economic_narrative") {
        match orchestrator.generate_narrative(payload, NarrativeTone::Dramatic) {
            Ok(text) if validate_llm_narrative(&text, payload).is_ok() => {
                return NarrativeOutput::LlmEnhanced(text);
            }
            _ => { /* 回退到下一层 */ }
        }
    }

    // 第2层: LLM不可用——模板+（如果可用）NPC引语LLM润色
    let mut text = payload.fallback_text_detailed.clone();
    if llm_config.merchant_dialogue && orchestrator.is_available("npc_quote_polish") {
        // 仅润色NPC引语部分（更小的LLM请求）
        text = polish_npc_quotes_only(&text, payload, orchestrator);
    }

    NarrativeOutput::Template(text)
}
```

## 五、玩家可感知的LLM增强效果

### 5.1 体验对比

| 场景 | 无LLM | 有LLM |
|------|-------|-------|
| 价格飙升通知 | 系统通知："铁矿石价格+36%" | 叙事文本：老铁匠在市场摇头叹气，学徒问他为什么——一段小故事 |
| 询问商人 | 打开商店UI，手动比价 | 商人主动告诉你"别买，南边新矿下周到货"——省去货比三家的操作 |
| 传闻 | "听说XX涨价了" | 一段有语气、有失真、有人物的小剧场 |
| 饥荒记录 | 数据条目 | 可读的编年史——有人物、有因果、有悲剧 |
| 商业咨询 | 手动算套利空间 | 退休商人给你分析两条商路的利弊——像和一个有经验的NPC聊天 |

### 5.2 不改变的东西

LLM增强**不改变**以下任何东西：
- 价格本身（由订单簿撮合决定）
- NPC的消费决策（由NPC心智模型决定）
- 交易结果（由撮合引擎决定）
- 货币供应（由铸币/回收管道决定）
- 市场权力运作（由PowerAtom机制决定）
- 信息传播的失真算子（由语言表达系统的传播通道计算）

LLM只改变玩家**看到**这些信息的**文字形式**。世界运转逻辑完全不受影响。

## 六、架构与集成

### 6.1 依赖方向

```
经济系统（economic）           语言表达（lang_expression）
─────────────────────         ─────────────────────────
EconomicNarrativePayload  ──→  LlmOrchestrator::generate_narrative()
EconomicCause              ──→  PromptLibrary::build_economic_prompt()
AffectedNpcSnapshot        ──→  LlmDialogueEngine::enhance_merchant_line()
fallback_text              ──→  作为回退直接使用
```

经济系统不依赖LLM增强层——经济系统只生成 `EconomicNarrativePayload` 和 `fallback_text`。LLM增强层消费这些数据，产出叙事文本。经济系统永远不知道（也不关心）返回的文本是来自LLM还是模板。

### 6.2 调用流程

```
1. 经济事件发生（e.g. 价格飙升检测触发）
   ↓
2. 经济系统生成 EconomicNarrativePayload + fallback_text
   ↓
3. 经济系统调用 LangExpression::narrate_economic_event(payload)
   ↓
4. 语言表达模块检查 LlmSceneConfig
   ├─ LLM可用 → LlmOrchestrator → 返回叙事文本
   └─ LLM不可用 → 直接返回 fallback_text_detailed
   ↓
5. 叙事文本写入：
   - 大日志（历史系统）
   - NPC对话输出（如果玩家正在和市场NPC对话）
   - 玩家通知（HUD消息或日志条目）
   - 信息传播通道（作为PropagationPayload的叙事化包装）
```

### 6.3 新增LlmSceneConfig场景

经济系统需要语言表达模块在 `LlmSceneConfig` 中新增以下两个场景开关：

```rust
// 在 lang_expression/llm_enhancement/config.rs 中新增：
pub struct LlmSceneConfig {
    // ... 已有场景 ...

    // ── 经济场景（新增）──
    pub economic_events: bool,       // 经济事件叙事化（价格飙升/市场崩溃/饥荒/商路事件）
    pub merchant_dialogue: bool,     // 商人NPC对话增强（自然语言建议/商业咨询/谣言叙事）
}
```

这两个开关独立于其他场景——玩家可以只开启经济事件叙事化而关闭其他LLM增强，反之亦然。

## 七、性能与资源预算

### 7.1 LLM调用频率预估

| 场景 | 调用频率 | 说明 |
|------|---------|------|
| 经济事件叙事化 | ~1-5次/游戏日 | 仅跨越有意义阈值的事件 |
| 商人对话增强 | ~1-3次/玩家拜访商店 | 仅在玩家主动与商人对话且有话可说时 |
| 经济谣言增强 | ~1-2次/游戏日 | 仅在重大谣言的初始爆发阶段 |
| 玩家商业咨询 | 按需 | 玩家主动发起 |
| 历史经济记录 | ~0.5-2次/游戏年 | 年度/饥荒/重大事件的叙事化记录 |

**日均可控LLM调用 <20次**（经济场景），远低于对话系统的LLM调用（可数百次/玩家游戏日）。

### 7.2 响应时间预算

| 场景 | 目标延迟 | 超时阈值 |
|------|---------|---------|
| 经济事件叙事化 | <1s | 2s（超时→模板） |
| 商人对话增强 | <2s | 3s（NPC正在"思考"——2秒对对话节奏可接受） |
| 商业咨询 | <3s | 5s（深度分析需要更多token） |
| 经济谣言增强 | <1s | 2s |
| 历史经济记录 | <5s | 10s（大量数据的综合叙事，允许更长时间） |

### 7.3 Payload大小控制

| 场景 | 典型Payload大小 | 说明 |
|------|----------------|------|
| 价格飙升 | ~2-5KB | 单一物品+3-5个受影响NPC快照 |
| 饥荒叙事 | ~8-15KB | 多种物品+数十NPC快照+天气数据 |
| 商人对话 | ~1-3KB | 当前市场+1-3个替代来源+NPC人格摘要 |
| 商业咨询 | ~3-6KB | 多个套利机会+风险因素+路线信息 |

## 八、不做

| 条目 | 理由 |
|------|------|
| LLM参与价格决策 | 违反核心原则——价格从订单簿撮合涌现，LLM只做叙事包装 |
| LLM参与NPC消费决策 | NPC决策由BigFive+Emotion+Memory+Cognition驱动——这是确定性规则层的事 |
| LLM"发明"经济事件 | 经济事件由经济系统检测和生成——LLM不创造事件 |
| LLM直接读取经济数据库 | 通过EconomicNarrativePayload注入数据——LLM不调用任何经济系统API |
| LLM生成交易建议影响市场 | 商人给的建议是已有数据的自然语言化——不影响NPC的实际交易决策 |
| 每笔交易都走LLM | 仅跨越有意义阈值的事件——日均数千笔日常交易不走LLM |
| LLM回退时显示简陋占位符 | 模板管道覆盖100%事件类型，无LLM时也能提供充分信息 |
| 实时经济LLM仪表板 | 超出单人开发范围——留给未来的MOD或外部工具 |
