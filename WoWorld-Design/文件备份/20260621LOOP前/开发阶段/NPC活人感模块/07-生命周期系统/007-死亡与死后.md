## 一、死因分类：30 种 6 类

### 1.1 分类原则

按**死亡发生的生理路径**分类（单维度、互斥），替代现有跨损伤机制/环境条件/来源性质/特殊分类的混维分类。

| 类别 | 生理路径 | 死因 | 可成亡灵 |
|------|---------|------|:--:|
| **I. 创伤 Traumatic** | 身体结构被外力破坏 | Laceration 割裂 / BluntForce 钝击 / Fall 坠落 / Drowning 溺水 / Suffocation 窒息 / Burning 焚烧 / Exsanguination 失血 / Crush 碾压 | ✓ |
| **II. 匮乏 Deprivation** | 维持生命的资源断绝 | Starvation 饥饿 / Dehydration 脱水 / Asphyxiation 缺氧 / Hypothermia 失温 / Hyperthermia 过热 | ✓ |
| **III. 侵入 Invasion** | 外来物破坏体内平衡 | Infection 感染 / Poison 中毒 / Venom 注毒 / Parasite 寄生 / Disease 疾病 | ✓ |
| **IV. 奥术 Arcane** | 魔法/超自然因果 | SpellDirect 魔法直伤 / ManaOverload 魔力过载 / ElementalBacklash 元素反噬 / Curse 诅咒 / RitualFailure 仪式失败 | 依赖具体 |
| **V. 衰亡 Senescence** | 身体系统渐进失效 | OrganFailure 器官衰竭 / OldAge 老衰 / SpiritDecline 精魄衰退 | ✓ |
| **VI. 意志 Volition** | 有意图的外部裁决 | Execution 处决 / Sacrifice 献祭 / SelfWilled 自主离世 | 依赖具体 |

**IV 和 VI 的亡灵规则**（例外）：
- ManaOverload、RitualFailure → 精魄-血液界面塌缩 → **不可**成亡灵
- Sacrifice（献祭）→ 精魄转移到接收者 → **不可**成亡灵
- SelfWilled（自主离世）→ 取决于方法。自然手段可，魔法手段可能不可

### 1.2 DeathCause 的 Rust 架构——trait + 注册表

采用 CHG-014 的 `ItemDefId(u64)` 模式——不是 enum，是注册表：

```rust
pub type DeathCauseId = u32;

pub struct DeathCauseDef {
    pub id: DeathCauseId,
    pub category: DeathCauseCategory,
    pub can_become_undead: bool,
    pub spirit_dissipation: f32,       // 死亡瞬间精魄迸散比例 (0.0-1.0)
    pub is_violent: bool,              // 目击者创伤反应
    pub tombstone_text_key: String,    // 墓碑文本模板 key
}

pub trait DeathCauseRegistry: Send + Sync {
    fn get(&self, id: DeathCauseId) -> Option<&DeathCauseDef>;
    fn register(&mut self, def: DeathCauseDef) -> DeathCauseId;  // mod 用
}
```

**类别级规则**：I-V 类默认可成亡灵，IV/VI 有例外。不逐条枚举亡灵资格。

## 二、Gompertz 衰老死亡模型

### 2.1 为什么用 Gompertz

- "超过 max_lifespan 后概率死亡"太模糊——什么概率？逐日？逐年？
- "max_lifespan 就是确切死亡线"太硬——没有人能在 max 后多活哪怕一天？
- 需要：起点柔和（从 70% 寿命开始有极低风险）、加速上升（越老风险越高）、无上限（理论上可能超过 max）、闭式可积（支持批量计算）

Gompertz：`h(t) = a × exp(b × t)` → 危险率（hazard rate）随年龄指数增长。

### 2.2 参数化

```rust
/// 衰老生存概率——闭式积分
///
/// # 参数
/// - age_pct: 当前年龄占 max_lifespan 的比例 (0.0-1.5)
/// - delta_pct: 此时间段内 age_pct 的增量
/// - constitution: 体质 (0.0-1.0)
/// - health_history: 一生中重大伤病累积分 (0.0+)
///
/// # 返回
/// 此时间段内存活的概率 (0.0-1.0)
pub fn senescence_survival(
    age_pct: f32,
    delta_pct: f32,
    constitution: f32,
    health_history: f32,
) -> f32 {
    if age_pct < 0.7 { return 1.0; }  // 70% 寿命前无衰老风险
    
    let t = age_pct - 0.7;
    let dt = delta_pct.min(1.5 - t.max(0.0));
    if dt <= 0.0 { return 1.0; }
    
    let a = 0.001 * (1.5 - constitution) * (1.0 + health_history);
    let b = 6.0;
    
    // ∫ a×exp(b×s) ds = (a/b) × (exp(b×(t+dt)) - exp(b×t))
    let integral = (a / b) * ((b * (t + dt)).exp() - (b * t).exp());
    (-integral).exp().clamp(0.0, 1.0)
}
```

### 2.3 示例

| age/max | 月死亡概率 | 含义 |
|---------|:---------:|------|
| 0.70 | ~0.3% | 70 岁（人类 49 岁）——风险开始出现 |
| 0.85 | ~3% | 老年阶段开始——显著风险 |
| 0.95 | ~36% | 接近 max——每月超过 1/3 的概率 |
| 1.00 | ~71% | max 年龄——大多数人会在一年内死亡 |
| 1.10 | ~99.97% | 超过 max 10%——几乎必然死亡 |
| 1.20 | ~99.999% | 仍在数学上可能存活——"世界最老的人"涌现 |

**体质差的 NPC 整个曲线左移**——有人在 60% 寿命就开始面临显著风险。**健康生活 + 高体质 + SuperAging 的 NPC 整个曲线右移**——可能活到 120% 寿命。

### 2.4 衰亡死因抽样

```rust
fn sample_senescence_cause(age_pct: f32, rng: &mut Rng) -> DeathCauseId {
    if age_pct < 0.95 {
        // 更多人死于具体器官衰竭或疾病
        weighted_choice(rng, &[
            (DeathCauseId::ORGAN_FAILURE, 0.5),
            (DeathCauseId::DISEASE,      0.3),
            (DeathCauseId::OLD_AGE,      0.2),
        ])
    } else {
        // 超高龄——"无疾而终" / "老死" 比例更高
        weighted_choice(rng, &[
            (DeathCauseId::OLD_AGE,        0.7),
            (DeathCauseId::ORGAN_FAILURE,  0.2),
            (DeathCauseId::SPIRIT_DECLINE, 0.1),
        ])
    }
}
```

同一个数学模型下，"心力衰竭离世""安详地在睡梦中离去""精魄散尽，羽化而去"都是同一个终点的不同墓碑文本。

### 2.5 竞争风险合成

单个 NPC 可能同时面对多种死亡风险：

```rust
fn evaluate_all_death_risks(entity: &LifeEntity, world: &World) -> Option<DeathCauseId> {
    let mut risks: Vec<(DeathCauseId, f32)> = Vec::new();
    
    // 衰老
    if entity.age_pct() >= 0.7 {
        let p = 1.0 - senescence_survival(entity.age_pct(), ...);
        risks.push((DeathCauseId::OLD_AGE, p));
    }
    
    // 战斗（当前正在受伤）
    if let Some(combat_risk) = entity.combat_death_probability_this_tick() {
        for (cause, p) in combat_risk {
            risks.push((cause, p));
        }
    }
    
    // 疾病（若有活跃感染）
    if let Some(disease_risk) = entity.active_disease_death_probability() {
        risks.push(disease_risk);
    }
    
    // 环境（极端温度）
    if let Some(env_risk) = entity.environmental_death_probability() {
        risks.push(env_risk);
    }
    
    // 独立风险合成：P(存活) = ∏(1 - P_i)
    let total_p: f32 = risks.iter().map(|(_, p)| p).sum();
    if rand::random::<f32>() > total_p {
        return None;  // 本月存活
    }
    
    // 竞争抽样——风险高的更可能中选
    Some(weighted_choice_from(risks))
}
```

## 三、DeathEvent——中性事实

```rust
pub struct DeathEvent {
    pub entity_id: EntityId,
    pub cause: DeathCauseId,
    pub location: Vec3,
    pub time: GameDay,
    pub spirit_remaining: f32,   // 死亡瞬间精魄残留
    pub was_violent: bool,       // 目击者创伤反应
    pub body_intact: bool,       // 影响葬礼形式
}
```

**DeathEvent 不携带**：
- 情感建议（"此人值得哀悼"）
- 葬礼触发（"安排葬礼"→由信仰模块自己决定）
- 通知范围（"通知所有亲属"→亲属通过自己的感知和记忆发现）
- 亡灵生成（→由生命模块在信仰模块的辅助下判定）

## 五、死亡压缩——从全量到摘要

NPC 死亡后不保留完整的 NpcData（~50KB+）。三阶段压缩：

| 阶段 | 时间 | 操作 | 内存占用 |
|------|------|------|---------|
| **Phase 1: 保留** | T+0 | 标记 is_dead。完整 NpcData 保留在内存——其他 NPC 还在消化死讯、参加葬礼 | ~50KB |
| **Phase 2: 压缩** | T+7 天 | MemoryStore 大部分转 LifeTrace/ThoughtImprint。压缩为 DeathSummary | ~1-3KB |
| **Phase 3: 归档** | T+30 天 | 完整 NpcData→LMDB 冷存储。内存仅留 DeathSummary 引用 | ~128B（引用） |

```rust
struct DeathSummary {
    entity_id: EntityId,
    name: String,
    birth_day: GameDay,
    death_day: GameDay,
    death_cause: DeathCauseId,
    epitaph: String,               // 墓碑铭文
    key_achievements: [AchievementRef; 5],
    lingering_spirit: f32,         // 亡灵判定。降到 0 后彻底移除
    data_archive_key: Option<u64>, // LMDB 键
}
```

## 六、死后痕迹——各自模块拥有自己的痕迹

| 痕迹 | 所有者 | 内容 | 存续期 |
|------|--------|------|--------|
| **LifeTrace** | 历史模块 | 墓碑文本、传记摘要、出生-死亡日期、成就摘要、历史事件记录 | 永久 |
| **ThoughtImprint** | 认知模块 | ≤20 条高置信度 MentalModel 的冻结快照——读树技能提取 | 可被消费直到耗尽 |
| **遗嘱/遗产分配** | 权力模块 | 物品转移指令、头衔继承链 | 执行后归档 |
| **记忆（在他人脑中）** | NPC 活人感 | 其他 NPC 的 MemoryStore 中关于逝者的记忆条目 | 直到该 NPC 死亡或记忆超 2000 条被淘汰 |
| **尸体** | 生命 + 空间查询 | 物理存在→腐烂气味→骨架/灰尘 | 约 30 天 |
| **亡灵** | 生命 + 信仰 | 若条件满足→Undead 实体创建 | 直到精魄耗散至 0 |

### 6.1 墓碑——不是每个 NPC 都有

墓碑由历史模块决定是否生成：

```
立碑条件（涌现）：
- 当地文化中是否有立碑传统
- 逝者在该地的社会地位/荣誉权重
- 是否有活着的亲属愿意出资立碑
- 军事/战斗死亡可能集体立碑而非个人墓碑
```

**一个没有墓碑的 NPC 死了就是消失了——这是涌现，不是 bug。** 大多数 NPC 不会有墓碑。墓碑是稀有的、有价值的——玩家遇到一块墓碑时，它讲述一个值得被记住的人的故事。

### 6.2 亡灵生成——不是自动的，不是必然的

死因的 `can_become_undead` 是必要条件，不是充分条件。亡灵生成还需要：

- 精魄残留足够高（spirit_remaining > 0.3）
- 有绑定源（怨念/诅咒/亡灵法术/环境魔力/神谕/自愿/感染）——7 种已有 (生命 008)
- 周围有精魄可以吸收（维持亡灵存在的精魄来源）
- 不被信仰/文化立即"处理"掉（火化/圣水/祈祷）

**大多数满足这些条件的死亡仍然不会生成亡灵。** 亡灵是罕见的——这正是它们的冲击力所在。

## 七、死前行为——涌现，无临终系统

### 7.1 死前没有特殊行为模式

NPC 在死亡瞬间没有"切换到临终模式"。他们死前的行为和任何其他时刻的行为一样——由 GOAP + 情绪 + 处境驱动。

```
在睡梦中死亡 → 死前没有特殊行为。睡着了，再也没醒来。
在战斗中死亡 → 死前在战斗——和任何其他时刻的战斗一样。
在家人的注视中死亡 → 家人在场是因为他们选择在场，不是系统调度。
```

### 7.2 LifeTheme::Legacy 不是临终——是重启

LifeTheme::Legacy 在死亡前**很久**就触发了（age > 85% max）。NPC 在此主题下的行为——收徒、写书、立碑、反复讲同一批故事——不是"临终行为"，是"对有限剩余时间的认知驱动的长期规划"。

一个进入 Legacy 主题的 NPC 可能再活 10 年。这 10 年的行为模式与他之前 50 年的模式不同——因为他的认知世界的重心改变了。不是系统切换，是 SelfNarrative 的 7 天反思周期在评估"我还剩什么"之后自然调整了 Aspiration 分布。
