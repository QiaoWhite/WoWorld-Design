## 一、核心主张：玩家 = NPC + 控制层

### 1.1 本体论

```rust
// Player ≠ 独立实体类型。Player = Sapient + ControlMode。
// 玩家的 LifeEntity 和其他 NPC 完全相同。
// 同一套 Vitals、同一套 GOAP、同一套死亡规则。

// MindLayer 不变：
enum MindLayer {
    None, Minimal, Instinctive, Sapient, Divine, Player,  // Player 保留
}

// Player = SapientMind + ControlMode（新增字段）
struct SapientMind {
    // 所有 Sapient NPC 已有的字段...
    control_mode: Option<ControlMode>,  // Some = 可被玩家接管
}

enum ControlMode {
    Auto,                                 // 完全托管——GOAP 正常运转
    Manual { action_override: Action },   // 完全手操——GOAP 后台运行但不被执行
    DomainDelegated {                     // 域选择托管
        manual_domains: HashSet<ActionDomain>,
    },
}

enum ActionDomain {
    Movement,
    Combat,
    Speech,
    ItemUse,
    Interaction,
    // ...
}
```

### 1.2 关键性质

- **被控无感知**：被操控的角色不知道自己被操控。GOAP 在后台持续运转——产生"如果没被控制，会做什么"的假设动作流。这个差异可进入自我叙事记忆（"我今天做了不想做的事"——如果 NPC 的意愿和控制输入显著不同）
- **所有系统正常运行**：审美/认知/情感/关系不受 ControlMode 影响。玩家可以操控角色的移动，但角色的情感仍然来自他自己的情感引擎
- **同 PC 同死**：玩家操控的角色和任何其他 NPC 一样——同样的 DeathCause、同样的 Gompertz、同样的死亡后处理

## 二、双角色调度

### 2.1 两个可操控角色

```rust
struct PlayerAgent {
    active_1: EntityId,
    active_2: EntityId,
    current_control: EntityId,       // 当前活跃屏幕
    other_since: Option<GameTime>,   // 另一个角色上次被关注的时间
}
```

### 2.2 切换

- 同一时刻只操控一个角色
- 另一个角色在 Auto 模式下自主行事——和任何其他 NPC 一样
- 切换时：当前角色回到 Auto，目标角色切换到 Manual 或 DomainDelegated
- 切换瞬间显示目标角色的状态摘要（位置/当前活动/最近关键事件）——属于 UI 设计，不在本文档范围

### 2.3 时间跳过

当**两个角色都处于跳时间活动**（睡觉/冥想/昏迷/长途旅行）时：

```rust
fn next_player_event(active_1: EntityId, active_2: EntityId, world: &World) -> GameTime {
    let now = world.game_time();
    
    // 角色各自的预期醒时间
    let wake_1 = expected_wake_time(active_1, world);
    let wake_2 = expected_wake_time(active_2, world);
    let nearest_wake = wake_1.min(wake_2);
    
    // 可能的外部打断事件
    let external = world.next_significant_event_for(&[active_1, active_2]);
    // significant_event: 战斗开始/火灾/有人来敲角色 A 的门/婴儿哭了/...
    
    nearest_wake.min(external)
}
// 时间跳到 nearest_wake_or_event
// 跳过后，如果有外部事件→通知玩家
// 否则，切换到第一个醒的角色
```

## 三、死亡继承

### 3.1 继承顺序

```
角色死亡
  │
  ├─► 有一代以内血亲？
  │     ├─ 是 → 继承。玩家控制切换到该血亲。
  │     │        被继承的 NPC 保持自己的 Identity/Memory/Relationship。
  │     │        玩家现在用这个人的眼睛看世界。
  │     │
  │     └─ 否 → 继续
  │
  ├─► 两个角色 + 所有血亲都不可用？
  │     ├─ 是 → 弹出大日志选择
  │     │        显示候选人：年龄、生命阶段、职业、社会地位、位置、简要生平
  │     │        玩家选择一个或两个候选人继续游戏
  │     │
  │     └─ 用户拒绝 → 结算
  │                   存档保存
  │                   下次读取时仍弹出选择提示
```

### 3.2 一代以内血亲过滤

```rust
fn one_degree_blood_relatives(deceased: EntityId, world: &World) -> Vec<EntityId> {
    world.relationships_of(deceased)
        .filter(|r| matches!(r.kind,
            RelationshipKind::Parent | 
            RelationshipKind::Child | 
            RelationshipKind::Sibling))
        .filter(|r| r.is_blood_relation)
        .filter(|r| !r.target.is_dead)
        .map(|r| r.target)
        .collect()
}
```

### 3.3 继承后的世界

- 逝者的数据进入正常的死亡后管道（DeathSummary/ThoughtImprint/LifeTrace）
- 继承的 NPC 继续他的生活——只是现在玩家在操控他
- 被继承的 NPC 不丢失任何记忆或关系——玩家只是换了一个角色
- 如果玩家有两个活着的角色，另一个不受影响

### 3.4 玩家不扮演儿童的边界

如果候选继承人是婴幼儿，他的 action_repertoire 极小（≈["cry", "sleep", "eat"]），planning_horizon=1。玩家操控一个婴儿几乎不能做任何事——但这本身就是一种游戏内容。玩家可以在大日志中看到"2 岁婴儿"的选项并选择跳过它。

## 四、ControlMode 与 GOAP 的关系

### 4.1 Auto 模式

```rust
// GOAP 正常优先级排序 → 选择最优动作 → 执行
// 玩家观察到角色在做自己的事
```

### 4.2 Manual 模式

```rust
// 玩家输入覆盖 GOAP 的动作输出
// GOAP 照常在后台运行，产生"假设动作"
// 假设动作 和 实际动作 的差异度超过阈值时 →
//   记忆系统注册 "我做了一件不是我本心想做的事" 的感知
//   → 在下次自我叙事反思中产生认知张力
```

### 4.3 DomainDelegated 模式

```rust
// 例如：玩家接管 Movement，其他由 NPC 自主
manual_domains = {Movement}

// NPC 的 GOAP 决策引擎正常运转
// 产生类似 "我想去酒馆" → GOAP 规划路线 →
//   玩家正在控制 Movement → 玩家可能把他往反方向推
//   但同时 NPC 的 GOAP 产生 "肚子饿了" → 自己打开背包吃东西
```

## 五、与 NPC 生命周期模块其他系统的关系

| 生命周期系统 | 对玩家的影响 |
|------------|------------|
| AgeClock | 玩家角色同样老化。但玩家可能活不过自然寿命——冒险者生涯更可能死于战斗 |
| GestationState | 玩家角色可怀孕（如果是女性）。孕期身体变化同样适用 |
| InfantDependency | 如果玩家角色是母亲→GOAP 中出现 CareForInfant 目标。玩家可在手动模式下选择遵守或忽略 |
| DeathCause | 相同 30 种死因。玩家角色同样可能死于老衰/战斗/疾病/意外 |
| DeathSummary | 玩家角色的 LifeTrace 同样被记录。"上一个玩家的墓碑"成为世界景观 |
| LifeTheme | 玩家的自我叙事同样运转。玩家角色的 LifeTheme 变迁不受控制模式影响 |
