## 一、核心原则

**信仰系统只定义条件与修饰。所有执行走现有系统。**

| 信仰系统的角色 | 现有系统执行 |
|-------------|------------|
| 定义教众涌现条件 | Group 系统创建 Group 实体、管理 member/leader |
| 定义神职资格条件 | 权力系统创建 Divine 来源的 PowerEdge、管理 legitimacy |
| 定义宗教规范内容 | NPC CulturalNorm 系统注册、NormInternalization 自动生效 |
| 定义社交 modifiers | NPC 社交好感计算（同信加成/异教惩罚） |
| 定义神学争论触发条件 | 语言表达系统（ConversationTopic）→ 争论/辩论 |

## 二、宗教群体——教众/修会/教派

### 2.1 涌现条件

```rust
/// 信仰系统定义涌现条件。Group 系统执行创建。
/// 每游戏月在每个 settlement 检测一次。
fn congregation_emergence_check(
    faith_id: FaithId,
    settlement_npcs: &[(NpcId, &ReligiousPracticeProfile)],
    theology: &FaithTheology,
) -> Option<CongregationTemplate> {
    // 条件 1：足够多的同信仰者
    let believers: Vec<_> = settlement_npcs.iter()
        .filter(|(_, p)| p.participation.iter().any(|(f, w)| *f == faith_id && *w > 0.3))
        .collect();
    if believers.len() < 15 { return None; }  // 至少 15 人的临界质量
    
    // 条件 2：该信仰的 hierarchy_degree ≥ 0.2（需要一定组织度）
    // 如果 hierarchy_degree < 0.2 → 信仰在民间自然流动，不形成正式群体
    
    // 条件 3：如有足够深度理解者（depth > 0.5 ≥ 3 人）——群体更稳定涌现
    let scholars = believers.iter().filter(|(_, p)| p.theological_depth > 0.5).count();
    let stability = (scholars as f32 / believers.len() as f32).min(1.0);
    
    Some(CongregationTemplate {
        faith_id,
        group_type: if theology.hierarchy_degree > 0.5 {
            ReligiousGroupType::Congregation  // 正式教众
        } else {
            ReligiousGroupType::Circle        // 信仰圈——更松散的聚会
        },
        estimated_members: believers.len() as u32,
        stability,
    })
}
```

### 2.2 宗教群体类型

```rust
pub enum ReligiousGroupType {
    /// 教众（Congregation）——一个 settlement 内同信仰者的常规聚会
    /// 这是最常见的宗教群体类型
    Congregation,
    /// 信仰圈（Circle）——非正式聚会，无严格层级
    Circle,
    /// 修会（Order）——有特殊誓约/纪律的僧侣/修女群体
    /// 涌现条件：hierarchy_degree > 0.5 + 至少 3 个 depth > 0.7 的 NPC
    Order,
    /// 教派（Sect）——因神学分歧从主体分离的小众团体
    /// 涌现条件：folk_divergence > 0.3 + 有高 depth 的异见领袖
    Sect,
    /// 隐修团体（Hermitage）——远离聚落的隐修者小群
    /// 涌现条件：mysticism > 0.7 + nature_sacredness > 0.5
    Hermitage,
    /// 护教团体（MilitantOrder）——武装保卫信仰的团体
    /// 涌现条件：exclusivity > 0.6 + militarism > 0.5（文化参数）+ 感知到威胁
    MilitantOrder,
}
```

### 2.3 群体治理类型

信仰系统的宗教群体治理类型**叠加**在权力系统的 Group 治理之上：

```rust
/// 宗教群体特有的治理模式——与权力系统 Group 治理互补
pub enum ReligiousGovernance {
    /// 使徒传承——领袖指定继承者（对标教皇→主教链条）
    /// 映射到权力系统：Autocracy + SuccessionRule::DesignatedHeir
    ApostolicSuccession,
    /// 选举——教众投票选出神职
    /// 映射到权力系统：Democracy
    Elected,
    /// 魅力型——领袖因圣洁/神迹/启示自然涌现
    /// 映射到权力系统：Emergent 权力来源
    Charismatic,
    /// 长老议会——最资深者共同决策
    /// 映射到权力系统：CouncilOfElders
    Council,
}
```

**信仰系统提供治理类型的推荐**（从 FaithTheology 派生），Group 系统在创建宗教群体时消费。

## 三、神职体系

### 3.1 ClergyRole——神职角色

```rust
/// 信仰系统定义的通用神职角色类型。具体角色名称由 FaithDefinition 定制。
pub enum ClergyRole {
    /// 祭司/神父/伊玛目——主持仪式、解释教义、牧养信徒
    Priest,
    /// 高阶祭司/主教/大伊玛目——管辖多个祭司或一个区域
    HighPriest,
    /// 僧侣/修女——以出世修行为主，可能有独身誓约

    /// 萨满——通过出神/灵界旅程与灵沟通

    Ascetic,
    /// 神谕/占卜者——通过特定方法解读神意

    /// 护教者——以战斗保卫信仰（对标圣殿骑士/武士僧）
    Templar,
```

### 3.2 神职任命条件

```rust
/// 信仰系统提供——权力系统在创建 Divine PowerEdge 前验证资格。
fn can_become_clergy(
    agent: &dyn FaithAgent,
    faith_id: FaithId,
    role: &ClergyRole,
) -> bool {
    let profile = agent.practice_profile();
    
    // 1. 参与门槛
    let participation = profile.participation.iter()
        .find(|(fid, _)| *fid == faith_id)
        .map(|(_, w)| *w)
        .unwrap_or(0.0);
    if participation < 0.5 { return false; }
    
    // 2. 深度门槛（不同角色要求不同）
    let depth_required = match role {
        ClergyRole::Priest => 0.5,
        ClergyRole::HighPriest => 0.7,
        ClergyRole::Monk => 0.3,
        ClergyRole::Shaman => 0.4,
        ClergyRole::Prophet => 0.6,
        ClergyRole::Ascetic => 0.5,
        ClergyRole::Oracle => 0.5,
        ClergyRole::Templar => 0.3,
    };
    if profile.theological_depth < depth_required { return false; }
    
    // 3. 动机——不能是 SocialCompelled/Habitual（"我只是混饭吃的"不能做神职）
    if matches!(profile.motivation, ReligiousMotivation::SocialCompelled | ReligiousMotivation::Habitual) {
        return false;
    }
    
    // 4. theology 技能——依赖技能系统
    // 从技能系统查询 theology 技能等级 ≥ 对应阈值
    // （通过 SkillQuery trait 查询——信仰系统不导入技能系统类型）
    
    true
}
```

### 3.3 教会层级 → PowerTopology 映射

教会层级 = PowerTopology 中的 Divine 来源权力边链：

```
PowerTopology:
  HighPriest ──(Divine, Delegate)──→ Priest₁
  HighPriest ──(Divine, Delegate)──→ Priest₂
  HighPriest ──(Divine, PrescribeRule)──→ 信徒群体（Duty: 遵守教规）
  
  如有更高层级（ArchPriest → HighPriest → Priest）：
  通过权力系统的委托链递归表达。
```

信仰系统只提供：
- 层级结构定义（InstitutionStructure——见 002 §六）
- 任命资格验证（can_become_clergy()）
- 推荐的继承规则

权力系统执行实际的 PowerEdge 创建/更新/删除。

## 四、FaithNorm → CulturalNorm 注册

### 4.1 注册机制

信仰系统的禁忌和教义注册为 NPC 模块的 `CulturalNorm`：

```rust
/// 信仰系统在 settlement 初始化或新信仰传入时调用。
/// 将 FaithDefinition.taboos + doctrines 转换为 CulturalNorm 条目。
fn register_faith_norms(
    faith_id: FaithId,
    norms: &mut CulturalNormRegistry,  // NPC 模块的规范注册表
) {
    let def = faith_def(faith_id);
    
    for taboo in &def.taboos {
        let norm = CulturalNorm {
            id: NormId::new_faith_derived(faith_id, taboo),
            description: taboo.description.clone(),
            scope: NormScope::SpecificFaith(faith_id),  // ★ 新增 NormScope 变体
            severity_of_violation: taboo.severity,
            // ... 其他字段
        };
        norms.register(norm);
    }
    
    // 类似处理 doctrines 中的伦理教义（"不可杀人"→ CulturalNorm）
}
```

### 4.2 NormScope 新增变体

NPC 模块需要新增 `NormScope::SpecificFaith(FaithId)` 以支持信仰限定规范。这是对 NPC 模块的一处微调（详见 CHG-025）。

两个同文化但不同信仰的 NPC 通过此变体正确区分"你的禁忌不是我的禁忌"。

### 4.3 NormInternalization 的信仰维度

NPC 模块的 `NormInternalization`（cognitive_acceptance / emotional_alignment / behavioral_compliance）自动适用于信仰规范——信仰系统不需要额外机制。高 participation 的 NPC 自然会有更高的 behavioral_compliance。

## 五、信仰对社交的调制

### 5.1 同信仰亲和

```rust
/// 信仰系统提供。NPC 模块在社交初始化时调用。
fn faith_affinity_bonus(
    a: &ReligiousPracticeProfile,
    b: &ReligiousPracticeProfile,
) -> f32 {
    let mut bonus = 0.0;
    
    for (faith_id_a, weight_a) in &a.participation {
        if let Some((_, weight_b)) = b.participation.iter().find(|(fid, _)| *fid == *faith_id_a) {
            let shared = weight_a.min(*weight_b);  // 共同的投入程度
            let theology = faith_def(*faith_id_a).theology;
            // 排他性高的信仰 → 同信者之间更紧密
            let intensity = 0.15 + theology.exclusivity * 0.15;
            bonus += shared * intensity;
        }
    }
    
    bonus.clamp(0.0, 0.3)  // 最大 +0.3 初始好感加成
}
```

### 5.2 异信仰惩罚

```rust
fn interfaith_penalty(
    a: &ReligiousPracticeProfile,
    b: &ReligiousPracticeProfile,
) -> f32 {
    // 只在以下情况适用：
    // 1. 一方信仰的 exclusivity > 0.6
    // 2. 且 inter_faith_hostility(faith_a, faith_b) > 0.3
    // 惩罚范围：-0.05 ~ -0.2（远比同信加成弱——大多数人不主动敌视异信者）
    
    // 极少数反宗教 NPC 对所有宗教参与者有轻微负面感受
    if a.motivation == ReligiousMotivation::AntiReligious && !b.participation.is_empty() {
        return -0.05;
    }
    
    0.0  // 默认无惩罚
}
```

### 5.3 传教行为

```rust
/// NPC 是否考虑向目标 NPC 传教。
fn should_proselytize(
    self_profile: &ReligiousPracticeProfile,
    target_profile: &ReligiousPracticeProfile,
    theology: &FaithTheology,
) -> bool {
    // 条件：
    // 1. self 的某信仰 participation > 0.6
    // 2. 该信仰的 proselytizing 倾向 > 0.4
    // 3. 目标没有活跃参与该信仰
    // 4. self 的 motivation 不是 Habitual（"我习惯了去庙里——但我不觉得有必要拉别人去"）
    
    let top_faith = top_participation(self_profile);
    if top_faith.is_none() { return false; }
    let (faith_id, weight) = top_faith.unwrap();
    
    if *weight < 0.6 { return false; }
    if theology.proselytizing() < 0.4 { return false; }
    if target_has_active(target_profile, *faith_id, 0.3) { return false; }
    if matches!(self_profile.motivation, ReligiousMotivation::Habitual | ReligiousMotivation::SociallyCompelled) { return false; }
    
    true
}
```

传教行为通过 NPC 决策引擎的 `TravelReason::Religious`（已存在）触发——NPC 可能主动旅行到异教区域传教。

## 六、FaithAgent trait

```rust
/// 信仰系统定义的 trait——任何"有信仰的实体"都实现此 trait。
/// NPC 模块实现（NpcData impl FaithAgent）。
/// 玩家模块（未来）也实现此 trait。
/// 
/// 关键：信仰系统不区分调用者是 NPC 还是玩家。
/// 热路径（decision engine）读本地字段——零跨模块调用。
pub trait FaithAgent: Send + Sync {
    /// 实践画像——不可变引用（热路径）
    fn practice_profile(&self) -> &ReligiousPracticeProfile;
    
    /// 实践画像——可变引用（温路径——事件驱动）
    fn practice_profile_mut(&mut self) -> &mut ReligiousPracticeProfile;
    
    /// 精神需求——热路径（NPC 本地字段）
    fn spiritual_need(&self) -> f32;
    fn spiritual_need_mut(&mut self) -> &mut f32;
    
    /// 是否可以进行祈祷（当前不在战斗中、不在睡眠中、不在游泳中等）
    fn can_pray(&self) -> bool;
    
    /// 作为 NPC 引用——供 DivinePresence 调用（返回 None 表示玩家实体）
    fn as_npc_ref(&self) -> Option<&dyn NpcRef>;
}
```

**热路径说明**：
- `practice_profile()` 读 NPC 本地字段——零 trait 调用开销
- `spiritual_need()` 读 Physiology 本地字段——零 trait 调用开销
- NPC 决策引擎如需信仰修饰 → 直接读本地 `ReligiousPracticeProfile.participation`——不走 FaithQuery trait
