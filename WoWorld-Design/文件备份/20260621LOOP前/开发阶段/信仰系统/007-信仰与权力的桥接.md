## 一、核心原则

**信仰系统不直接操作 PowerTopology。所有桥接通过事件。**

```
信仰系统发出事件                    权力系统响应
─────────────────                   ────────────
DivineAuthorization {              → 创建 PowerEdge {
  npc: NpcId,                          holder: npc,
  faith_id: FaithId,                   source: Divine { ritual, deity },
  authorization_type: Oracle           atom: ClaimTerritory | PrescribeRule | ...
    | Ordination                     }
    | Prophecy
    | DivineRight
}
```

信仰系统发出的 `DivineAuthorization` 事件包含授权类型、候选人 NPC ID、关联的神祇。权力系统消费事件并创建对应 PowerEdge。

**为什么不用直接 API 调用**：信仰和权力是两个独立的社会子系统。信仰决定"这个人有神的授权"，权力决定"这个授权转换为哪一种权力 atom"。两者通过事件解耦——任一方的内部实现变化不影响另一方。

## 二、Divine 授权——神职任命

### 2.1 授权事件

```rust
/// 信仰系统发出。权力系统订阅。
#[derive(Clone)]
pub struct DivineAuthorizationEvent {
    /// 被授权者
    pub npc: NpcId,
    /// 关联的信仰
    pub faith_id: FaithId,
    /// 关联的神祇——（如该信仰为多神教，指定具体哪一位神授权）
    pub deity: Option<EntityId>,  // EntityId::Npc(神) 或 EntityId::Group(万神殿)
    /// 授权类型
    pub authorization_type: DivineAuthorizationType,
    /// 授权仪式——指向 RitualRegistry 中的 RitualDef
    pub ritual: String,
    /// 授权生效年份
    pub effective_year: i32,
}

pub enum DivineAuthorizationType {
    /// 神谕——通过解读神意（A 模式：神直接说话；B/C 模式：祭司解读征兆）
    Oracle,
    /// 叙阶——通过正式任命仪式成为神职
    Ordination,
    /// 预言——声称获得神的启示，通常伴随改革/新宗教创立
    Prophecy,
    /// 神权——通过"神的选择"获得统治权（如"君权神授"）
    DivineRight,
}
```

### 2.2 权力系统的消费

```rust
// 权力系统 004（权力获取与合法性）§2.6
// 消费 DivineAuthorizationEvent → 创建 PowerEdge

PowerEdgeData {
    holder: event.npc,                           // 获得授权的人
    subject: target_entity,                      // 目标实体（信徒群体/领土/行为域）
    atom_type: match event.authorization_type {
        Oracle => PrescribeRule,                 // 神谕 → 制定规则
        Ordination => ClaimTerritory | PrescribeRule, // 叙阶 → 管理教区+制定规则
        Prophecy => PrescribeRule,               // 预言 → 新的规则体系
        DivineRight => ClaimTerritory | PrescribeRule | Extract, // 君权神授 → 领土+法律+税收
    },
    source: PowerSource::Divine {
        ritual: event.ritual.clone(),
        deity: event.deity,
    },
    legitimacy: compute_initial_legitimacy(event, culture_query),
    succession_rule: match event.authorization_type {
        Ordination => SuccessionRule::DesignatedHeir,  // 使徒传承
        DivineRight => SuccessionRule::Primogeniture,   // 长子继承（君权）
        _ => SuccessionRule::Unspecified,
    },
    ..
}
```

### 2.3 初始 legitimacy 计算

```rust
/// Divine 来源的初始 legitimacy 取决于文化的 religiosity。
fn compute_divine_legitimacy(culture_religiosity: f32, auth_type: &DivineAuthorizationType) -> f32 {
    // 高 religiosity 文化 → Divine 授权高 legitimacy
    let base = match auth_type {
        DivineRight => 0.7,    // 君权神授——天然高 legitimacy
        Ordination => 0.6,     // 正式任命——较高
        Oracle => 0.5,         // 神谕——需要持续证明
        Prophecy => 0.3,       // 预言——最低——"你怎么证明神对你说话了？"
    };
    // religiosity 调制：0.3（世俗）→ 1.0（神权）
    let religiosity_mod = culture_religiosity * 0.7 + 0.3;
    (base * religiosity_mod).clamp(0.15, 0.95)
}
```

## 三、教会层级 → PowerTopology 委托链

### 3.1 层级表达

教会层级通过权力系统的委托链递归表达：

```
HighPriest (ArchPriest如适用)
    │
    ├──(Delegate)──→ Priest₁ ──(PrescribeRule)──→ 信徒群体A
    │                   └──(Adjudicate)──→ 信徒A₁（如需要异端审判）
    │
    ├──(Delegate)──→ Priest₂ ──(PrescribeRule)──→ 信徒群体B
    │
    └──(PrescribeRule)──→ 全体信徒（Duty: 遵守教规、什一税）

如果有更高层级（大主教 = ArchPriest）：
ArchPriest ──(Delegate)──→ HighPriest₁
ArchPriest ──(Delegate)──→ HighPriest₂
```

### 3.2 信仰系统的角色

- 提供层级结构定义（InstitutionStructure——002 §六）
- 提供任命资格验证（can_become_clergy()——004 §3.2）
- 提供继承规则推荐（ApostolicSuccession → SuccessionRule::DesignatedHeir）
- **不创建 PowerEdge**（权力系统负责）
- **不追踪谁对谁有权力**（PowerTopology 负责）

## 四、对神 Pledge 的违约处理

权力系统 005 §6.2 中，当 Pledge 的 `pledged_to = Some(deity)` 且发生违约时：

```rust
// 权力系统发出：PledgeBroken { pledger, pledged_to: Some(deity), ... }
// 信仰系统接收：
fn handle_pledge_broken_to_deity(event: &PledgeBrokenEvent, divine: &dyn DivinePresence) {
    match divine.mode() {
        DivineMode::A => {
            // 神可能亲自降下惩罚——DivinePresence::influence_world()
            // 根据神的性格+违约的严重程度决定响应
        }
        DivineMode::B | DivineMode::C => {
            // 只有社会后果：
            // 1. 如果有 witness（神职人员）→ witness 施加社会惩罚
            // 2. 如果无 witness → 纯粹 legitimacy 损失——破誓者的声誉下降
            // 3. 该信仰信徒对其 trust 下降
        }
    }
}
```

## 五、神职 Immunity

### 5.1 Divine 来源的免疫

权力系统 005 §3.1 已定义 `ImmunitySource::Divine { ritual: String }`。信仰系统不创建 ImmunitySet——它只提供"此神职应获得什么免疫"的推荐参数。

```rust
/// 信仰系统提供。权力系统在创建 Immunity 时消费。
fn recommended_clergy_immunities(role: &ClergyRole, theology: &FaithTheology) -> Vec<ImmunityRecommendation> {
    match role {
        ClergyRole::Priest | ClergyRole::HighPriest => vec![
            // 神权政体中——神职对世俗 Adjudicate 免疫
            // 由宗教法庭专属管辖
            ImmunityRecommendation {
                atom_type: AtomTypeTag::Adjudicate,
                scope: "世俗法律",
                source: ImmunitySource::Divine { ritual: "ordination".into() },
            },
        ],
        // ...
    }
}
```

### 5.2 与 Derogate 的区别

- **Immunity**：subject 自身属性——"你不能审判我"（神职自带）
- **Derogate**：holder 主动放弃——"我赦免你"（国王特赦某人不守斋戒）

信仰系统涉及的两者皆有：神职 Immunity 是常规免疫；特定仪式中祭司可授予 Derogate（如"忏悔后的赦免"）。

## 六、神权政体（Theocracy）

### 6.1 政体涌现

权力系统 007 已有 `GovernmentForm::Theocracy { religious_leader, deity }`。信仰系统不创建政体——它只在政体涌现检测时提供信仰侧数据：

- 领土内有多少比例的 NPC 归属同一信仰？
- 该信仰的 hierarchy_degree 是否 ≥ 0.5（足够组织化以形成政教合一）？
- 是否有高 depth 神职可获得 Divine 授权？

### 6.2 神权政体的特殊效果

当 Polity 被识别为 Theocracy：
- 经济系统 MarketRegulations：禁止买卖圣物（经济 005 已定义此规则）
- 权力系统：Divine 授权 legitimacy +0.15 加成
- NPC 信仰：该信仰的 participation 在政体领土内受到 +0.05 的 mild boost（社会强制）
- 战斗系统：对敌对信仰的 hostility 阈值小幅下降（更容易触发冲突）

**信仰系统只提供数据，不触发这些效果**——各消费方根据 `is_theocracy(polity_id)` 自行施加。

## 七、宗教法律（Canon Law）→ PrescribeRule

### 7.1 映射

宗教法律 = 神职通过 PowerSource::Divine 行使 PrescribeRule 原子：

```rust
PrescribeRuleParams {
    scope: RuleScope::BelieversOf(FaithId),  // 只对信徒有效
    content: RuleContent::ReligiousObligation {
        obligation: "安息日不可劳作",   // 从 FaithDefinition.taboos 派生
        expected_behavior: "休息与祈祷",
    },
    sanction_on_violation: SanctionSpec {
        // 根据 TabooConsequence 等级映射：
        // GuiltOnly → 无外部制裁
        // SocialDisapproval → 社区谴责
        // Excommunication → 逐出教会（Duty 撤销+社交排斥）
        // LegalPenalty → 仅当 theocracy 时——移交世俗法律执行
    },
}
```

### 7.2 与世俗法律的冲突

当宗教法律与世俗法律同时覆盖同一 subject 时，走**权力系统的规范层级**（权力 005 §四）：
- 委托链距离优先
- 领域专属优先（Domain::Behavior → 宗教法律 vs 世俗法律）
- legitimacy 高的优先
- 平局 → 触发 Adjudicate 事件，由上级裁决

信仰系统不需要参与冲突解决——权力系统的规范层级模型已经覆盖。
