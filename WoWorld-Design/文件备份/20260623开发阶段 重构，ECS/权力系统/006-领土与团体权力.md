## 一、领土不是新原子——是 PowerDomain 的特化

领土权力不需要新增原子。`ClaimTerritory` 不是一个独立原子——它是 `PowerDomain::Territory` + holder 对该领土内所有 entity 持有的一组原子集合的**组合效应**。

```rust
/// 领土域——权力的空间适用范围
PowerDomain::Territory {
    /// 领土的空间边界（多边形顶点集）
    bounds: WorldBounds,
    /// 进入此区域的实体是否自动受 holder 的权力约束
    /// auto_apply = true: 进入 → 自动创建临时的 subject 关系
    /// auto_apply = false: 进入 → 不自动约束（如声明无实际控制的领土）
    auto_apply: bool,
}
```

## 二、领土权力的工作机制

### 2.1 领土边创建

当一个 holder 对某地理区域声明管辖权时：

```rust
fn claim_territory(
    topology: &mut PowerTopology,
    holder: EntityId,
    bounds: WorldBounds,
    attached_powers: Vec<AtomTypeTag>,  // 随领土附带的权力原子
) -> Result<EdgeId, PowerError> {
    // 创建一条 domain=Territory 的边
    // holder 对 "territory" 这个抽象 entity（或直接对区域内所有人）
    let edge_id = topology.create_edge(PowerEdgeData {
        holder,
        subject: EntityId::World,  // 特殊 entity——世界本身
        atom_type: AtomTypeTag::Constrain,  // 领土的基底是 Constrain
        domain: PowerDomain::Territory {
            bounds: bounds.clone(),
            auto_apply: true,
        },
        source: PowerSource::Emergent {   // 初始声明可能以 Emergent 开始
            basis: EmergenceBasis { compliance_rate: 0.0, exercise_count: 0 },
        },
        legitimacy: 0.3,   // 初始低——需要时间建立事实控制
        enforcement: 0.1,
        succession_rule: SuccessionRule::Unspecified,
        ..Default::default()
    });
    
    // 为随领土附带的每个原子类型，记录"当 subject 在此领土内时，holder 对这些 subject 行使什么原子"
    for atom in attached_powers {
        topology.attach_territorial_power(edge_id, atom);
    }
    
    // 注册到空间索引
    topology.spatial_index.insert(bounds, edge_id);
    
    Ok(edge_id)
}
```

### 2.2 实体进入领土时

```rust
/// 当 entity 移动到 WorldPos 时调用（由 NPC 移动系统触发）
fn on_entity_enter_territory(
    topology: &PowerTopology,
    entity: EntityId,
    pos: WorldPos,
) -> Vec<PowerEvent> {
    let mut events = Vec::new();
    
    // 查询该位置适用的领土边
    let territorial_edges = topology.spatial_index.query_point(pos);
    
    for edge_id in territorial_edges {
        let edge = topology.get_edge(edge_id);
        if !edge.active { continue; }
        
        // 检查是否 auto_apply
        if let PowerDomain::Territory { auto_apply: true, .. } = edge.domain {
            // 检查 entity 是否已有此 holder 的对应入边
            if !topology.has_edge(edge.holder, entity, edge.atom_type) {
                // 为 entity 创建一条临时入边——subject = entity
                // 此入边是 territorial edge 的投影，不是独立边
                topology.create_territorial_projection(edge_id, entity);
                
                events.push(PowerEvent::EntityEnteredTerritory {
                    entity,
                    holder: edge.holder,
                    territory_edge: edge_id,
                });
            }
        }
    }
    events
}
```

### 2.3 NPC 移动性能优化

**核心优化——Chunk 缓存**：

```rust
/// 每个 16m×16m Chunk 缓存适用的领土边
struct ChunkPowerCache {
    /// LRU 缓存——最近使用的 1024 个 Chunk
    cached_chunks: LruCache<ChunkCoord, CachedTerritorialPowers>,
    /// 缓存失效标记——领土边变化时递增
    cache_generation: u64,
}

struct CachedTerritorialPowers {
    /// 该 Chunk 适用的领土边列表
    edge_ids: Vec<EdgeId>,
    /// 缓存生成号
    generation: u64,
}

impl ChunkPowerCache {
    fn get_territorial_powers(
        &mut self,
        pos: WorldPos,
        topology: &PowerTopology,
    ) -> Vec<EdgeId> {
        let chunk = ChunkCoord::from_world_pos(pos);
        
        // 1. 查缓存
        if let Some(cached) = self.cached_chunks.get(&chunk) {
            if cached.generation == self.cache_generation {
                return cached.edge_ids.clone();  // 缓存命中
            }
        }
        
        // 2. 缓存未命中 → 查四叉树
        let edges = topology.spatial_index.query_aabb(chunk.aabb());
        let result: Vec<EdgeId> = edges.iter()
            .filter(|&&e_id| {
                let edge = topology.get_edge(e_id);
                edge.active && edge.domain.contains(pos)
            })
            .copied()
            .collect();
        
        // 3. 写缓存
        self.cached_chunks.put(chunk, CachedTerritorialPowers {
            edge_ids: result.clone(),
            generation: self.cache_generation,
        });
        
        result
    }
    
    /// 领土边变化时调用
    fn invalidate_cache(&mut self) {
        self.cache_generation += 1;
        // LRU 中的旧条目在下次命中时被 generation mismatch 过滤
    }
}
```

**性能预估**：
- NPC 在 Chunk 内移动 → 零四叉树查询（每帧 ~5K NPC 移动，99% 同 Chunk）
- 跨 Chunk 边界 → 查 LRU（~1µs）→ 命中率 ~90% → 未命中才查四叉树（~5µs）
- 每帧 ~167 次边界跨越查询 → 总 <0.01ms/帧

## 三、领土争议

两个 holder 对同一区域有 `Territory` 域的权力边 → 领土重叠。

```rust
/// 检测领土争议
fn detect_territorial_disputes(
    topology: &PowerTopology,
    bounds: WorldBounds,
) -> Vec<TerritorialDispute> {
    let overlapping = topology.spatial_index.query_aabb(bounds);
    let mut disputes = Vec::new();
    
    // 找出所有有重叠领土域的 holder 对
    for i in 0..overlapping.len() {
        for j in i+1..overlapping.len() {
            let edge_a = topology.get_edge(overlapping[i]);
            let edge_b = topology.get_edge(overlapping[j]);
            
            if edge_a.holder != edge_b.holder {
                // 双方都对该区域有领土声明
                let overlap_area = edge_a.domain.territory_bounds()
                    .overlap_area(edge_b.domain.territory_bounds());
                
                if overlap_area > 0.0 {
                    disputes.push(TerritorialDispute {
                        holder_a: edge_a.holder,
                        holder_b: edge_b.holder,
                        contested_bounds: overlap_bounds(&edge_a, &edge_b),
                        severity: compute_dispute_severity(edge_a, edge_b, overlap_area),
                    });
                }
            }
        }
    }
    
    disputes
}

fn compute_dispute_severity(
    edge_a: &PowerEdge,
    edge_b: &PowerEdge,
    overlap_area: f32,
) -> f32 {
    // severity = f(重叠面积 × 双方 enforcement × 文化差异 × 是否有历史冲突)
    let enforcement_factor = (edge_a.enforcement + edge_b.enforcement) / 2.0;
    let claim_factor = overlap_area / (edge_a.domain.area() + edge_b.domain.area()) * 2.0;
    (enforcement_factor * claim_factor).clamp(0.0, 1.0)
}
```

领土争议被检测到 → 触发 `PowerEvent::TerritorialDispute` → 历史系统记录 → 语言表达系统传播 → 可能导致战争。

## 四、Group 作为 Holder

### 4.1 EntityId::Group

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum EntityId {
    Npc(NpcId),
    Player(PlayerId),
    /// 团体实体——可以作为 holder 或 subject
    Group(GroupId),
}
```

**关键设计**：Group 本身不能"思考"或"决定"——它是其内部治理结构的抽象。当 Group 行使权力时，系统查询 Group 内部**谁有权代表 Group**。

### 4.2 Group 的内部治理

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct GovernanceRules {
    /// 治理类型
    pub governance_type: GovernanceType,
    /// 决策阈值（投票通过所需的比例或票数）
    pub decision_threshold: f32,  // 0.5 = 简单多数, 0.67 = 绝对多数, 1.0 = 全票
    /// 否决权持有者
    pub veto_holders: Vec<EntityId>,
    /// 代表的委任规则
    pub representation_rule: RepresentationRule,
}

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub enum GovernanceType {
    /// 独裁——一人决定
    Autocracy { leader: EntityId },
    /// 寡头——少数人投票决定
    Oligarchy { council: Vec<EntityId> },
    /// 民主——全体成员投票
    Democracy { electorate: EntityId },  // electorate = this group itself
    /// 共识制——需要全体同意
    Consensus,
    /// 长老会——由最年长/最有经验的成员组成
    CouncilOfElders { member_count: u8 },
    /// 混合——不同事务由不同实体决定
    Hybrid { rules: Vec<(PowerDomain, GovernanceType)> },
}
```

### 4.3 Group 行使权力的流程

```rust
fn exercise_group_power(
    topology: &mut PowerTopology,
    group_id: GroupId,
    actor: EntityId,          // 谁在"替" group 行使权力（必须是 group 成员）
    subject: EntityId,
    atom_type: AtomTypeTag,
    params: AtomParams,
) -> Result<(PowerExerciseResult, Vec<PowerEvent>), PowerExerciseError> {
    // 1. 获取 group 的治理规则
    let governance = topology.get_governance(group_id)?;
    
    // 2. 检查 actor 是否有权代表 group 行使此原子
    let authorized = match &governance.governance_type {
        GovernanceType::Autocracy { leader } => actor == *leader,
        GovernanceType::Oligarchy { council } => {
            // 需要 council 投票通过
            let votes = simulate_council_vote(council, topology);
            votes.in_favor >= governance.decision_threshold
        }
        GovernanceType::Democracy { electorate } => {
            // 抽样投票（不模拟每个成员——用统计代表）
            let support = estimate_member_support(*electorate, actor, atom_type, &params, topology);
            support >= governance.decision_threshold
        }
        // ...
    };
    
    if !authorized {
        return Err(PowerExerciseError::NotAuthorized);
    }
    
    // 3. 以 group 的名义行使权力
    let group_entity = EntityId::Group(group_id);
    topology.exercise_power(group_entity, subject, atom_type, params)
}
```

### 4.4 Group 内部权力边的递归

Group 内部的治理结构本身就是一组权力边：

```
Group "家族"
  ├── 族长 持有 [Represent(家族对外) + DefineMembership(家族成员) + Adjudicate(内部纠纷)]
  ├── 成员 A 持有 []（服从族长 Constrain）
  └── 成员 B 持有 []（服从族长 Constrain）
```

这组内部权力边在 Constitute 原子创建 Group 时一并创建。当族长死亡 → 继承规则执行 → 新族长获得 Represent 边。

## 五、团体类型与涌现

| 团体类型 | 典型规模 | 治理规则 | 涌现条件 |
|---------|---------|---------|---------|
| 家庭/家族 | 2-50 | Autocracy（家长） | Constitute（婚姻+生育） |
| 行会 | 10-200 | Oligarchy（长老会） | 同职业 NPC 自发组成 |
| 门派/教团 | 5-500 | Autocracy（掌门/教主） | Constitute（师徒/传教） |
| 帮派 | 5-100 | Autocracy（老大） | Emergent（事实控制积累） |
| 公司/商会 | 3-50 | Oligarchy（股东） | Contract（合伙契约） |
| 政党 | 20-500 | Democracy | 围绕共同政治目标集合 |
| 部落 | 50-5000 | CouncilOfElders | 文化 + 血缘 + 共同领土 |
| 国家/王国 | 500-100K | 取决于政体 | Polity 涌现——见 007 |

**所有这些团体的权力关系都在同一张 PowerTopology 图中——只是规模不同。**
