## 一、设计起源与理念

### 1.1 问题："吃"的对立面

NPC 吃完东西后,身体需要处理摄入的物质。真实生理中这是排泄——但在 WoWorld 的创作语境中,战斗/冒险/社交的"上厕所"打断极其影响体验。

**替代设计**：元素平衡。基于魔法系统公理——"血元素 = 八元素的复合材料"。进食→吸收食物元素→合成血元素→多余元素堆积→需释放。

### 1.2 设计目标

1. **逻辑自洽**：在 WoWorld 魔法世界观中完全合理——血元素本身就是八元素复合物
2. **不打断冒险**：释放行为瞬间完成(几秒),有意识但无声无形;睡眠可被动释放
3. **周期长**：均衡饮食几乎不累积,偏科饮食 3-5 天才需要主动一次
4. **驱动涌现**：不同饮食选择→不同元素累积→不同释放频率→微小的文化/种族差异
5. **连接魔法**：法师角色对血元素平衡更敏感→饮食讲究→职业深度

## 二、血元素合成的数学

### 2.1 核心公式：瓶颈模型

```
给定：
  food_element_yield[i] for i in 0..7 (金木水火土风雷电, 八元素不包含血和灵)
  race_blood_ratio[i] for i in 0..7 (该种族血元素的八元素配比, Σratio = 1.0)
  portion: f32 (进食份量)

步骤 1: 计算每个元素用于合成血元素的需求量
  needed[i] = race_blood_ratio[i] × portion

步骤 2: 找瓶颈——最稀缺的元素决定了血元素合成效率的上限
  bottleneck = min(1.0, min_over_i(absorbed[i] / needed[i]))

步骤 3: 计算各元素去向
  used[i]    = needed[i] × bottleneck    // 实际用于合成血元素的
  surplus[i] = absorbed[i] - used[i]     // 正=过剩累积, 负=元素不足
  element_surplus[i] += max(0.0, surplus[i])

步骤 4: 血元素合成效率
  合成效率 = bottleneck
  直接关联 hunger 恢复——效率<1.0 → 同等食物量恢复的饱腹度下降
```

### 2.2 瓶颈模型的涌现意义

- 一个 NPC 吃肉吃了三天(火元素偏重,但血元素合成所需的全部元素都有)→ bottleneck≈1.0→ 饱腹恢复正常 → 但火元素过剩一直在累积
- 同一 NPC 改吃蔬菜(木/水元素),肉吃得少 → 火元素的 bottleneck 来了——血元素合成效率下降 → 同样的蔬菜需要更多才能吃饱 → 火元素过剩自然缓解
- **两个需求(饥饿和元素平衡)通过同一进食事件产生张力**——这不是设计的,是算出来的

### 2.3 日行元素的分层默认值

```
食物元素谱的三层回退：

Level 1: 手写值 → ItemProperties.consumable.element_profile
         例：龙肉 {fire: 0.8, earth: 0.1, ...}
             魔法蘑菇 {spirit_element_bias, ...}

Level 2: 群系默认 → biome_default_element(BiomeId)
         例：沙漠作物 {fire: 0.3, earth: 0.2, wind: 0.1, ...}
             海岛鱼类 {water: 0.5, wind: 0.1, ...}

Level 3: 全局默认 → [0.125; 8] (完全均衡)
         面包、普通水果——不产生元素过剩
```

**设计意图**：大多数食物不需要手写元素谱。只有特殊/异域/魔法食物才值得定义。日常 NPC 几乎不会产生元素过剩——睡眠被动释放足够。异文化/异域饮食的 NPC 更频繁地需要释放过剩元素→丰富的行为差异。

## 三、释放机制

### 3.1 主动释放 (InstantVent)

```
触发条件:
  - 安全环境 (非战斗/非危机)
  - urgency > 0.3 (至少一个元素过剩超过 30%)
  - 进入概率决策器作为低权重行为

执行:
  - 持续: 1-3 秒 (游戏时间)
  - 效果: element_surplus 全部归零
  - 感知: NPC 自身有意识,短暂停顿
  - 观察: 旁观 NPC 无感知事件,不产生记忆
  - 玩家: 如果盯着看→NPC 短暂停顿(和发呆难以区分);否则完全无感
  - 环境: v1.0 不改变环境元素浓度场
```

### 3.2 睡眠被动释放

```
触发条件: NPC 处于睡眠状态

释放率: 5% / 游戏日 (每游戏日 tick 时执行)
  即 element_surplus[i] *= (1.0 - 0.05 × dt_days)

设计意图:
  - 均衡饮食 NPC → 被动释放足够 → 永远不需要主动释放
  - 偏科饮食 NPC → 被动太慢 → 必须每 3-5 天主动释放
  - 时间跳跃后批量更新(绝对时间函数,非积分)
```

### 3.3 时间跳跃安全性

```
fn bulk_advance(dt_seconds: f32, was_sleeping: bool):
  if was_sleeping:
    days = dt_seconds / 86400.0
    for i in 0..7:
      surplus[i] *= (1.0 - 0.05 * days)
      surplus[i] = max(0.0, surplus[i])
  // 清醒时无被动释放→surplus 不变化
```

## 四、食物元素谱 → 物品系统注册

### 4.1 ConsumableEffect 结构体

```rust
/// Life 模块定义——消费模块定义 schema, 物品模块存储
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ConsumableEffect {
    /// 饱腹恢复量 (0.0-1.0), 吃掉该食物 hunger += satiation
    pub satiation: f32,

    /// 水分恢复量 (0.0-1.0), 吃掉该食物 thirst += moisture
    pub moisture: f32,

    /// 元素谱——None = 默认均衡 (三层回退)
    /// 仅金木水火土风雷电八元素——血和灵不参与消化-过剩循环
    pub element_profile: Option<[f32; 8]>,

    /// HP 恢复量 (绝对值, 如 5.0)
    pub vitality_restore: f32,

    /// 体力恢复量 (绝对值)
    pub stamina_restore: f32,

    /// 消化时长 (秒) —— 影响"多久后再吃"
    pub duration_seconds: f32,

    /// 体温瞬时调整 (热汤 +0.05, 冰水 -0.03)
    pub temp_adjust: f32,
}
```

### 4.2 物品模块的存储位置

```rust
// ItemProperties 新增字段
pub struct ItemProperties {
    // ... 已有字段 ...
    pub consumable: Option<ConsumableEffect>,  // None = 不可食用
}

// ItemRegistry 新增查询
impl ItemRegistry {
    /// 查询食物的消耗品效果
    /// None = 不可食用
    pub fn get_consumable(&self, id: ItemDefId) -> Option<&ConsumableEffect>;
}
```

### 4.3 对齐已有模式

物品模块已有 "消费模块定义 schema, 物品模块存储" 的模式：
- 魔法模块: `EnchantmentSchema` → ItemProperties 存储
- 装配模块: `AssemblySchema` → ItemProperties 存储
- ★ Life 模块: `ConsumableEffect` → ItemProperties 存储 (新增)

物品模块不从类型上理解 ConsumableEffect——只是提供存储和查询。

## 五、与魔法模块的关联

### 5.1 v1.0 的桥接点

```
┌── 元素平衡需求 ──────────────────────┐
│                                        │
│ 食物 → 吸收元素 → 血元素合成 → 过剩释放  │
│                          │             │
│                          ▼             │
│                    血元素质量          │
│                          │             │
│ 魔法模块 ──────────────── ▼ ────────── │
│                    灵元素作用于血元素   │
│                          │             │
│                          ▼             │
│                    提炼魔力 (Mana)     │
│                                        │
└────────────────────────────────────────┘
```

**v1.0 不做**：
- 元素过剩对法术施法的增益效果
- 释放过剩元素会改变环境元素浓度场
- 法师主动调节饮食以优化魔力质量——这是涌现行为,不是系统强制

**v1.0 保留的桥接点**：
- `blood_element_ratio` 种族参数在 Life 模块,魔法模块可通过 trait 查询(预留)
- 未来魔法模块可消费 `element_surplus` 做 "元素共振" 增益——那是魔法系统的功能

### 5.2 饮食→魔力质量的间接影响

虽然 v1.0 不做,但管道已经铺好：
- 食物元素谱→血元素过剩→血元素成分偏斜→(未来)魔力类型微偏
- 混元法师追求饮食均衡——涌现行为,不靠代码
- 火法师刻意多吃偏火食物——涌现行为,不靠代码

## 六、边缘情况

### 6.1 特殊饮食类型

Life 004 §七 已有 diet_type：

| diet_type | 是否产生元素过剩 | 处理 |
|-----------|:---:|------|
| Herbivore/Carnivore/Omnivore | ✅ | 正常走瓶颈模型 |
| FilterFeeder | ✅ | 进食频率高但量小 |
| Scavenger | ✅ | 正常 |
| Lithovore (食矿) | ✅ | 矿物元素谱极偏→频繁释放 |
| Manavore (吸魔) | ❌ | 无物理进食→无元素过剩 |
| Undead | ❌ | 无 Vitals |

处理方式：跟随已有 hunger/thirst 的 diet_type 分支。不吃东西的物种不产元素过剩。

### 6.2 种族血元素配比的种子生成

```
blood_element_ratio:
  从种子生成 0-1 之间的 8 个值, 归一化 Σ=1.0
  人类基准: 大致均匀, 微小随机偏移
  矮人: 偏向金/土 (高金属/矿物饮食, 身体中金土元素比例更高)
  精灵: 偏向木/风
  水生灵兽: 偏向水
  火元素生物: 偏向火

libido_type:
  人类/矮人/精灵: Continuous
  鹿/某些灵兽: Seasonal
  龙/某些神明: EventTriggered
```

### 6.3 NPC 死亡时

```
Life::on_death(vitals):
  element_surplus = [0.0; 8]  // 归零
  libido = 0.0                // 归零
  // 不需要释放——死亡即关机
```

## 七、玩家视角

- **玩家不需要管理元素平衡**：纯涌现系统
- **玩家进食**：和 NPC 走同一套代码路径。但玩家不会吃偏科食物(玩家饮食通常多样化)→几乎不累积
- **玩家观察**：可能偶尔注意到某个 NPC 暂停一下(释放过剩),但和发呆无异
- **法师玩家(未来)**：如果想优化魔力,可主动调节饮食→深度玩法(非 v1.0)

## 八、未解决的问题（待讨论）

1. 元素释放是否是"法术操作"？——Release 是魔法五基本操作之一。普通人的释放是无意识的、微弱到忽略的;法师的释放是自觉的、更大量的。是否需要在释放时做一个"是否为法师"的判定分支？
2. 元素过剩是否影响 NPC 的魔法能力？——v1.0 不做。但这可能是一个有趣的 beta 属性。
3. 是否需要 "赤字" 概念？——如果某个元素长期不足(吸收<需求),血元素合成持续低效。当前瓶颈模型自然覆盖了此情况。
