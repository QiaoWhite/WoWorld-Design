## 一、模块定位

### 1.1 信仰系统是什么

信仰系统管理 WoWorld 中一切与"超自然信念与崇拜实践"相关的数据与逻辑：

- **信仰定义**：FaithDefinition——教义、万神殿、禁忌、经典、神圣日
- **NPC 信仰状态**：ReligiousPracticeProfile——每个 NPC 参与哪些信仰实践、投入多深、动机是什么
- **宗教群体**：教众、修会、教派的涌现条件与神职体系
- **信仰动力学**：信仰的传播（NPC→NPC 接触传染）、演变（漂移/分裂/融合）、消亡
- **宗教日历**：HolyDay、FastingRule、朝圣——供节日系统消费

### 1.2 信仰系统不是什么

- **不是神明系统**——神明的存在模式（A/B/C）属于 [[../生命/013-神明与信仰|生命 010]]。信仰系统通过 `DivinePresence` trait 查询神明，对三种模式完全透明。
- **不是文化系统**——文化决定"宗教在社会中有多重要"（`religiosity`），信仰决定"信什么、怎么信"。见 [[../文化系统/001-文化系统总纲|文化 001]] §4.2。
- **不是权力系统**——教会层级通过 PowerTopology 表达，信仰系统只提供神职任命资格条件和教会层级结构定义。
- **不是节日系统**——信仰系统提供 HolyDay 数据（实现 `FaithCalendarQuery` trait），节日系统消费并生成 FestivalFramework。

### 1.3 核心设计原则

**原则 1：实践优先，神学后置**

NPC 没有"神学立场"。他们有参与实践——去不去庙、拜不拜神、捐不捐香火。大多数 NPC 的宗教实践是被动继承的社区习俗（"我们村都这样"），而非深思熟虑的神学选择。神学标签（"虔诚"/"民俗多神"/"不信者"）从实践画像中**派生**，不驱动实践。

**原则 2：有人的地方才有信仰**

信仰不是独立于 NPC 的空间现象。信仰数据以 NPC 的 `ReligiousPracticeProfile` 为存储单元，空间分布（"这座城主要信什么教"）从 NPC 人口聚合派生。信仰随 NPC 的移动、交谈、生育、死亡而传播、演变、消亡。

**原则 3：信仰塑造社会组织——通过现有系统**

信仰创建群体（教众/修会/教派）→ 走 Group 系统。信仰区分身份（神职/信徒/异端）→ 走 PowerTopology。信仰执行规范（禁忌/斋戒/着装）→ 走 CulturalNorm 系统。信仰系统**只定义条件与修饰**，执行全交给现有系统。

**原则 4：信仰系统 = 纯数据 + 纯逻辑**

信仰系统不拥有 NPC 数据。NPC 的热路径（决策/行为/情感）零跨模块调用——`ReligiousPracticeProfile` 是 NPC 本地字段（~40 bytes/NPC）。信仰系统通过 trait 暴露查询，通过事件触发其他模块的变更。

## 二、核心概念

### 2.1 FaithId——信仰标识符

```rust
/// FaithId = u32，扁平全局标识符。
/// 对标 CultureId（文化 001）。不编码类型/区域信息。
/// FaithGenealogy 独立存储谱系关系。
///
/// 注意：历史系统之前使用的 `ReligionId` 为同一概念——统一为 FaithId。
/// 历史文档中的 ReligionId → FaithId（接口层统一，不修改行文）。
#[derive(Copy, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct FaithId(pub u32);
```

每个 FaithId 对应一个具体的宗教实例。世界上不是"一个多神信仰"——而是可能有 3 个不同的多神教，各有万神殿、各有教义、各有 FaithId。

### 2.2 ReligiousPracticeProfile——NPC 的宗教实践画像

```rust
/// 每个 NPC 存储一个。信仰系统定义类型——NPC 模块存储。
/// 热路径零跨模块调用。约 40 bytes/NPC。
///
/// 核心理念：这不是"信仰立场"——这是"实践记录"。
/// 一个 NPC 可以同时参与多个信仰的实践（见神就拜）。
/// 神学标签（DerivedStance）从此派生，仅供 UI 使用。
#[derive(Clone, Serialize, Deserialize)]
pub struct ReligiousPracticeProfile {
    /// 参与哪些信仰社区的实践活动
    /// Key = FaithId, Value = 参与强度 (0.0-1.0)
    /// 多数 NPC：1-3 个信仰条目，各有不同参与度
    /// 条目为空 → 不参与任何宗教实践
    pub participation: Vec<(FaithId, f32)>,

    /// 神学理解深度
    /// 0.0 = 纯民俗（"我奶奶这么做的，我跟着做"）
    /// 1.0 = 神学家（"我能论证三位一体的神学基础"）
    /// 从 theology 技能 + intelligence + 受教育程度派生
    pub theological_depth: f32,

    /// 为什么参与——驱动因素
    /// 默认 = Habitual（大多数 NPC 的出厂状态）
    pub motivation: ReligiousMotivation,
}

/// 宗教参与动机——信仰系统定义，NPC 模块存储。
#[derive(Clone, Serialize, Deserialize)]
pub enum ReligiousMotivation {
    /// 社会习俗——"我们村都这样"（绝大多数 NPC 的默认状态）
    SocialCustom,
    /// 个人虔诚——"我真的信，我自己选的"（较少，通常随年龄增长或重大经历后出现）
    PersonalDevotion,
    /// 危机驱动——"神啊救救我！"然后参与激增（短期状态，危机过后可能回落到原动机）
    CrisisDriven { since: GameTimestamp, intensity: f32 },
    /// 身份认同——"这是我们民族/家族的标志"（高 faith_as_identity 的信仰的典型信徒）
    IdentityBased,
    /// 功利/实用——"拜这个保佑发财，拜那个保佑平安"（见神就拜的核心动机）
    Pragmatic,
    /// 灵性追求——"我在寻找真理"（稀有，NPC 中可能 <1%）
    SpiritualSeeker,
    /// 习惯——"从来没想过为什么，反正一直这么做"
    Habitual,
    /// 社会强制——"不去教会/寺庙会被说闲话/罚款"
    SociallyCompelled,
    /// 反宗教——主动拒绝所有宗教实践（罕见）
    AntiReligious,
}
```

### 2.3 FaithTheology——信仰的神学参数

```rust
/// 定义一个信仰的神学特征——连续参数，非离散类型。
/// "单神/多神/萨满/泛灵"等标签从此参数组合派生——仅用于 UI 分类。
/// 对标 CultureCoreParams 的 10 参数设计（文化 002）。
pub struct FaithTheology {
    /// 神祇/崇拜对象的数量
    /// 0.0 = 非神论（无崇拜对象——如某些佛教、儒家）
    /// 0.01-0.3 = 泛灵（无数小灵，无中心神祇）
    /// 0.5 = 二元论（善 vs 恶的永恒对立）
    /// 1.0 = 严格独一神（只有一位真神）
    /// 2.0+ = 多神（万神殿中的多位神祇）
    pub deity_count: f32,

    /// 祖先在信仰中的分量 (0.0-1.0)
    /// 0.0 = 祖先无角色，1.0 = 祖先崇拜为核心
    pub ancestor_importance: f32,

    /// 自然神圣性——泛灵倾向 (0.0-1.0)
    /// 0.0 = 自然纯粹是物质，1.0 = 万物皆有灵
    pub nature_sacredness: f32,

    /// 组织化程度 (0.0-1.0)
    /// 0.0 = 纯个人灵修，无组织层级
    /// 1.0 = 中央集权教阶（教皇→枢机→主教→祭司→信徒）
    pub hierarchy_degree: f32,

    /// 经典中心性 (0.0-1.0)
    /// 0.0 = 纯口传传统，1.0 = 文本正统（一切以经典为准）
    pub scripture_centrality: f32,

    /// 仪式正式度 (0.0-1.0)
    /// 0.0 = 自发即兴的个人表达，1.0 = 严格礼仪（必须精确到每个词每个动作）
    pub ritual_formality: f32,

    /// 排他性 (0.0-1.0)
    /// 0.0 = 多元包容（"条条大路通神/觉悟"），可与其他信仰共存
    /// 1.0 = 唯一真理（"只有我们能得救——其他都是假先知"）
    /// 当 exclusivity > 0.7 时，该信仰要求信徒只归属于它
    pub exclusivity: f32,

    /// 神秘主义倾向 (0.0-1.0)
    /// 0.0 = 理性/经院/律法主义——通过推理和文本理解神
    /// 1.0 = 神秘体验/直接启示——通过冥想/出神/灵界旅程接触超自然
    pub mysticism: f32,

    /// 正信 vs 正行 (0.0-1.0)
    /// 0.0 = 纯正行（Orthopraxy）——重要的是你做什么（正确的仪式/洁净规则/行为）
    /// 1.0 = 纯正信（Orthodoxy）——重要的是你信什么（正确的信念/教义）
    pub orthodoxy_vs_orthopraxy: f32,

    /// 信仰作为身份 (0.0-1.0)
    /// 0.0 = 信仰是个人选择——可以随意改变
    /// 1.0 = 信仰是与生俱来不可剥离的身份——"你生来就是X，永远都是"
    pub faith_as_identity: f32,
}
```

**FaithTheology 参数不是独立随机变量**（它们之间有自然相关性——但罕见的组合产生独特的、有记忆点的信仰）。生成时从世界种子产生，受 origin_culture 的 CultureCoreParams 温和偏移（上限 ±0.05）。

### 2.4 DerivedReligiosity——派生的虔诚度

旧设计使用 5 档 piety 标量（狂热/虔诚/普通/冷淡/无信仰）。实践优先模型**废除 piely 标量**，改为从 `ReligiousPracticeProfile` 实时派生的三维度：

```rust
/// 不从存储——从 ReligiousPracticeProfile 实时派生。
/// 不同消费方取用不同维度：
///   - 节日系统关心 behavioral_intensity → 决定参加概率
///   - 权力系统(Divine 授权)关心 theological_depth + motivational_autonomy
///   - NPC 社交关心 behavioral_intensity + 对特定信仰的 participation 权重
pub struct DerivedReligiosity {
    /// 行为投入量——所有 participation 权重的几何和，cap 1.0
    /// 高值 = 频繁参与宗教活动
    pub behavioral_intensity: f32,
    /// 神学理解深度——等同 profile.theological_depth
    pub theological_depth: f32,
    /// 动机自主性——0 = 完全被动(被迫/习惯)，1 = 完全自主(个人选择/灵性追求)
    pub motivational_autonomy: f32,
}
```

**UI 标签映射**（纯软映射，不进入任何系统逻辑）：

| 条件 | 标签 |
|------|------|
| behavioral_intensity > 0.7 + depth > 0.5 + autonomy > 0.7 | "虔诚" |
| behavioral_intensity > 0.5 + depth < 0.3 + 多信仰 | "民俗多神（见神就拜）" |
| behavioral_intensity > 0.5 + single faith | "信者" |
| behavioral_intensity > 0.05 | "冷淡" |
| behavioral_intensity ≈ 0 + motivation = AntiReligious | "反宗教" |
| 其他 | "不参与" |

## 三、模块边界

### 3.1 与生命模块（神明）的边界

```
生命模块 010（保留）                  信仰系统（本模块）
──────────────────                    ──────────────
DivineMind (MindType)                 FaithId
GodEntity (LifeEntity 子类)           FaithDefinition
DivinePresence trait                  FaithTheology（10 连续参数）
  ├─ RealGods (Mode A)               ReligiousPracticeProfile（NPC 侧存储）
  ├─ NoGods (Mode B)                 DerivedReligiosity（派生）
  └─ AmbiguousGods (Mode C)          宗教群体涌现
DivineDomain (War/Harvest/...)        信仰生命周期
ManifestationRule                     宗教日历（HolyDay/FastingRule/Pilgrimage）
PrayOutcome                           信仰⇄权力事件桥接
凡人登神路径                           魔法-宗教关系
                                      信仰-物品扩展
                                      葬仪/死亡习俗
```

**神明**留在生命模块——神在 A 模式下是 LifeEntity 的子类。三种存在模式是生命模块的核心设计决策。

**信仰**独立——处理 NPC 对超自然的认知与崇拜体系。信仰系统通过 `DivinePresence` trait 查询神明，**对三种模式完全透明**——NPC 的信仰行为（祈祷/朝圣/祭祀/节日/禁忌）在 A/B/C 三种模式下完全相同。

### 3.2 与文化系统的边界

| | 文化系统 | 信仰系统 |
|---|---------|---------|
| **拥有** | CultureCoreParams.religiosity（宗教渗透程度，0-1） | FaithId、FaithDefinition、FaithTheology |
| **不拥有** | 信什么神、教义、禁忌、神职 | 宗教在社会的渗透程度 |
| **提供** | religiosity 作为信仰分配加权因子 | MagicReligionRelation、FaithCalendarQuery |
| **反馈路径** | 宗教改革 → CulturalShift 事件修改文化参数 | — |

**双向桥接**：
- 文化 → 信仰：`religiosity` 影响"该文化生成几个信仰" + NPC 初始 piety 分布
- 信仰 → 文化：宗教改革发出 CulturalShift 事件 → 修改 `religiosity`/`uncertainty_avoidance`/`openness_to_outsiders`

## 四、信仰的生命周期（概要）

```
出生（P2.5）→ 扩散（NPC→NPC接触）→ 发展（漂移/民间分歧）
                                            ↓
                                        改变（分裂/融合/改革/强制改变）
                                            ↓
                                        消亡（信徒<临界值 → extinct）
                                            ↓
                                        复活（极少见——考古+学者+新信徒 → 宗教复兴）
```
