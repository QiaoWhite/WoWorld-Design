## 一、存档文件格式

### 1.1 物理格式

一个 `.woworld` 文件 = 一个 **LMDB 环境**，内含多个 **named_db**（逻辑数据库）。

**named_db 由各模块通过 `SaveableModule::named_dbs()` 动态声明**——而非硬编码列表。以下为典型模块的 named_db 清单示例（实际集合由注册时模块声明决定）：

```
save.woworld (LMDB Environment, mmap'd)
  │
  ├── named_db: "meta"           ← Header、版本信息（存档系统自身）
  ├── named_db: "terrain"        ← 地形 Chunk、群系、资源
  ├── named_db: "building"       ← Building、BuildingHistory
  ├── named_db: "npc"            ← NPC 记忆、关系、BigFive、认知、技能
  ├── named_db: "items"          ← ItemState、库存、家具放置
  ├── named_db: "life"           ← Vitals、BodyPartDamage、植物 T1-T3、动物
  ├── named_db: "history"        ← EventLog、痕迹、书籍、灵元素印记、关系遗产
  ├── named_db: "grand_journal"  ← 大日志（玩家的外置大脑）
  ├── named_db: "econ"           ← OrderBook、交易历史
  ├── named_db: "power"          ← PowerTopology、Duty
  ├── named_db: "faith"          ← ReligiousPracticeProfile
  ├── named_db: "culture"        ← 文化参数、节日、仪式
  ├── named_db: "skill"          ← SkillEntry
  ├── named_db: "vehicle"        ← 载具损伤
  └── named_db: "player"         ← 玩家角色、库存、设置
```

**每个 named_db 是 LMDB 内的一个独立 B+树。** 逻辑隔离——一个 named_db 的 bug 不污染另一个。物理共享——同一个文件、同一个写事务保证原子提交。

**键前缀由模块通过 `named_dbs()` 声明**（如 `&[("npc", "npc/memory/"), ("npc", "npc/relationship/")]`）。SaveSystem 在注册时验证 (db_name, key_prefix) 组合不重叠。详见 [[003-保存流程与模块接口#八、模块注册与调用（Rust 惯用解耦）|003 §八]]。

### 1.2 为什么是单文件 + named_db

| 维度 | 单文件 named_db | 多文件 LMDB |
|------|----------------|------------|
| 跨模块原子事务 | ✅ 单写事务覆盖所有 named_db | ❌ 无跨文件事务 |
| 备份/分享 | ✅ 一个文件 | ❌ 需要打包 |
| 模块隔离 | ⚠️ 逻辑隔离（named_db间键不冲突） | ✅ 物理隔离 |
| 损坏隔离 | ⚠️ 文件损坏 = 全完蛋 | ✅ 一个文件损坏不影响其他 |
| LMDB 管理 | ✅ 一个 env、一个 mapsize | ❌ 多个 env 各自调参 |

**选择理由**：跨模块原子性是压倒性的优势。损坏隔离靠备份（3 个自动轮转 + 无上限手动存档）弥补。

### 1.3 named_db 动态创建

存档时模块通过 `DirtySnapshot` 自声明要写入哪些 named_db。如果 named_db 不存在 → LMDB `create_db` 自动创建（幂等操作）：

```rust
let db = txn.create_db(Some(&db_snap.db_name), DatabaseFlags::empty())?;
```

## 二、存档 Header

### 2.1 完整结构

```rust
#[derive(Serialize, Deserialize)]
struct SaveHeader {
    // ──────── 文件识别 ────────
    magic: [u8; 8],                          // b"WOWSAVE\x00"
    format_name: String,                      // "woworld-save-v1"（人类可读，≤64 UTF-8 字节）
    global_version: u32,                      // 存档文件结构版本（机器判断用）

    // ──────── 唯一标识 ────────
    save_uuid: [u8; 16],                     // UUID v4，创建时生成，永不改变

    // ──────── 世界信息 ────────
    world_seed: u64,                          // 世界种子（容错恢复的根）
    world_name: String,                       // 玩家命名的世界名（≤128 UTF-8 字节）
    world_summary: WorldSummary,              // 世界初始生成统计（Initial存档填写，其他存档复制）
    current_snapshot: WorldSnapshot,          // 存档时世界统计快照

    // ──────── 存档信息 ────────
    save_kind: SaveKind,                      // Manual | Auto | Quick | Exit | Initial
    save_name: String,                        // 存档显示名（≤256 UTF-8 字节）
    save_status: SaveStatus,                  // Active | PendingInheritance
    timestamp_unix: i64,                      // Unix 时间戳（仅用于人类可读展示）
    game_tick: u64,                           // 存档时的游戏 tick（用于排序）

    // ──────── 当前角色 ────────
    current_character_name: String,           // ≤64 UTF-8 字节
    current_character_level: u32,

    // ──────── 版本兼容 ────────
    module_versions: BTreeMap<String, u32>,   // "npc"→3, "items"→2, ...
    mod_manifest: Vec<ModEntry>,              // 创建此存档时的活跃 Mod 清单
}

#[derive(Serialize, Deserialize)]
struct WorldSummary {
    world_bounds_km: (f32, f32),     // (宽, 高) 公里
    total_land_area_km2: f32,
    ocean_ratio: f32,                // 0.0-1.0
    major_biome_count: u32,
    initial_npc_count: u32,
    initial_settlement_count: u32,
}

#[derive(Serialize, Deserialize)]
struct WorldSnapshot {
    alive_npc_count: u32,
    building_count: u32,
    settlement_count: u32,
    game_days: u32,
    current_year: i32,
}

#[derive(Serialize, Deserialize)]
enum SaveKind { Manual, Auto, Quick, Exit, Initial }

#[derive(Serialize, Deserialize)]
enum SaveStatus { Active, PendingInheritance }

#[derive(Serialize, Deserialize)]
struct ModEntry {
    mod_id: String,                  // "dragon_weapons"
    mod_version: String,             // "1.2.0"
    mod_item_def_prefix: u8,         // 0x01
}
```

### 2.2 字段说明

| 字段 | 必填 | 用途 |
|------|------|------|
| `magic` | ✅ | 文件类型识别。`b"WOWSAVE\x00"`——快速识别"这是不是 WoWorld 存档" |
| `format_name` | ✅ | 人类可读的格式名。程序用 `global_version` 判断，人用此字段调试 |
| `global_version` | ✅ | 存档文件结构的版本号。改变 named_db 数量/命名/Header 字段 → 递增 |
| `save_uuid` | ✅ | UUID v4。存档创建时生成，永不改变。文件名可改，显示名可改，UUID 是唯一不变的身份 |
| `world_seed` | ✅ | 世界种子。容错恢复的根——种子可重现层（地形/群系/资源）损坏时可用种子重建 |
| `world_name` | ✅ | 玩家命名的世界名。`world_meta.json` 是权威源，Header 是快照缓存 |
| `world_summary` | ✅ | 世界初始生成统计。Initial 存档填写，其他存档复制——不随世界变化而改变 |
| `current_snapshot` | ✅ | 存档时的世界统计。用于存档列表预览（"这个存档有 85K NPC 存活"） |
| `save_kind` | ✅ | Manual/Auto/Quick/Exit/Initial。决定存档列表的分组和轮转行为 |
| `save_name` | ✅ | 存档显示名。手动存档由玩家命名；自动/快速/退出由系统生成时间戳 |
| `save_status` | ✅ | Active = 正常可玩。PendingInheritance = 所有角色死亡，加载时弹出继承界面 |
| `timestamp_unix` | ✅ | 用于人类可读的时间展示（"2026年6月21日 15:25"）。**不用于排序** |
| `game_tick` | ✅ | 存档时的游戏 tick。**用于排序**——系统时钟可能被修改，tick 是单调递增的 |
| `current_character` | ✅ | 存档时的玩家角色名和等级。存档列表 UI 展示用 |
| `module_versions` | ✅ | **每个模块的数据格式版本**。读档时和当前模块版本比对，决定是否迁移 |
| `mod_manifest` | ✅ | 存档时的 Mod 清单。加载时和当前 Mod 清单比对，检测增删 |
| 各字符串字段 | ✅ | **所有字符串字段有硬上限**（见各字段注释）。反序列化时强制检查——超长=损坏拒绝加载 |

### 2.3 反序列化安全

```rust
fn validate_header(h: &SaveHeader) -> Result<()> {
    // 1. magic 字节
    if &h.magic != b"WOWSAVE\x00" {
        return Err("不是 WoWorld 存档文件");
    }

    // 2. 字符串长度硬上限
    if h.format_name.len() > 64 { return Err("format_name 超长"); }
    if h.world_name.len() > 128 { return Err("world_name 超长"); }
    if h.save_name.len() > 256 { return Err("save_name 超长"); }
    if h.current_character_name.len() > 64 { return Err("角色名超长"); }

    // 3. 版本号合理性
    if h.global_version == 0 || h.global_version > 1000 {
        return Err("global_version 可疑");
    }
    for (name, ver) in &h.module_versions {
        if *ver == 0 || *ver > 1000 {
            return Err(format!("模块 {} 版本号可疑: {}", name, ver));
        }
    }

    // 4. Mod 条目合理性
    for entry in &h.mod_manifest {
        if entry.mod_id.len() > 128 || entry.mod_version.len() > 64 {
            return Err("Mod 条目超长");
        }
    }

    Ok(())
}
```

### 2.4 Header 的存储位置

Header 存在 LMDB 的 `"meta"` named_db 中，键为固定值 `meta/header`：

```
named_db: "meta"
  ├── "meta/header"  → bincode(SaveHeader)
  └── （未来可扩展更多元数据键）
```

### 2.5 校验和

**不做应用层 SHA-256 校验。** 理由：
- LMDB 自身有页级 CRC 和 MVCC copy-on-write，极端抗损坏
- 半截写入通过原子写入协议防止（临时文件 + rename）
- 文件识别靠 `magic` 字节 + `global_version` 已足够
- 2GB 存档 SHA-256 需要 2-3 秒额外时间——不划算
- 位翻转下 LMDB 的页头校验大概率让 `env.open()` 直接失败——不会静默加载错误世界

## 三、LMDB 键命名约定

### 3.1 格式

```
<module>/<entity_type>/<identifier>

示例:
  npc/memory/0000000000000001          → NPC #1 的记忆 blob
  npc/relationship/0000000000000001    → NPC #1 的关系图
  items/state/0000000000000002         → ItemEnt #2 的状态
  terrain/chunk/+0032/-0001            → Chunk (32, -1) 的地形数据
  history/event/0000000000000003       → Event #3
  player/character                    → 当前玩家的角色数据
  meta/header                         → 存档 Header
```

### 3.2 约定

- 键前缀以模块名开头——写入 `"npc"` named_db 的键必须以 `npc/` 开头
- 模块违反此约定 → 模块 bug。存档系统不验证键前缀
- 不同 named_db 之间的键可以重名——不冲突（LMDB 隔离）
- 同一 named_db 内的键由模块自行管理——存档系统只存 bytes，不解析内容

## 四、LMDB 配置

### 4.1 mapsize

LMDB 需要预分配最大文件大小（mmap 虚拟地址空间预留）。实际磁盘占用 = 写入量（稀疏文件）。

| 场景 | 预期写入量 | mapsize |
|------|----------|---------|
| 初始存档 | 500MB-1GB | 4GB |
| 100 小时 | 2-3GB | 4GB |
| 500 小时 | 4-5GB | 8GB（自动扩容） |

**扩容机制**：

```rust
fn write_with_growth(env: &mut LmdbEnvironment, data: &[u8]) -> Result<()> {
    loop {
        match try_write(env, data) {
            Ok(()) => return Ok(()),
            Err(LmdbError::MapFull) => {
                let current = env.mapsize()?;
                if current >= MAX_MAPSIZE {  // 16GB 硬上限
                    return Err(LmdbError::MapFull);
                }
                let new_size = current * 2;
                env.set_mapsize(new_size)?;
                // 重试写入
            }
            Err(e) => return Err(e),
        }
    }
}
```

初始 mapsize = 4GB。扩容翻倍（4GB→8GB→16GB）。硬上限 16GB。

### 4.2 运行时 LMDB 句柄

游戏运行期间 LMDB 保持打开（只读）。MVCC 下：

- 多个只读事务可同时存在
- 写事务和只读事务不互斥（读者看旧版本）
- 存档时创建写事务，提交后新读者看到新版本

**不需要关闭/重开环境切换读写模式。**

```
<用户数据目录>/
├── settings.json                    ← 全局游戏设置（last_played_world_uuid, auto_save_interval, language）
└── saves/                           ← 存档根目录
    ├── 冰冠_七河/                   ← 世界目录（特殊字符已过滤）
    │   ├── world_meta.json          ← 世界元数据（UUID·种子·名称·创建时间·最新存档）
    │   ├── session.lock             ← 运行时存在，正常退出删除
    │   └── saves/                   ← 存档子目录
    │       ├── initial.woworld
    │       ├── quicksave.woworld
    │       ├── exitsave.woworld
    │       ├── autosave_1.woworld
    │       ├── autosave_1.woworld.bak   ← 覆盖前自动备份
    │       ├── autosave_2.woworld
    │       ├── autosave_3.woworld
    │       └── manual_a1b2c3d4.woworld
    ├── 实验世界/
    │   └── ...
    └── 铁人模式/
        └── ...
```

### 5.2 文件名约定

| 存档类型 | 文件名 |
|---------|--------|
| Initial | `initial.woworld` |
| Quick | `quicksave.woworld` |
| Exit | `exitsave.woworld` |

## 六、原子写入协议

### 6.1 流程

```
1. 创建临时文件: save_<uuid>.woworld.tmp
2. 打开 LMDB 环境（临时文件）
3. 创建写事务
4. 写入所有 dirty_entities（所有模块的 DirtySnapshot）
5. 删除 deleted_entities 键
6. 写入/更新 meta/header
7. 提交事务
8. 关闭 LMDB 环境
9. fsync 临时文件
10. 原子重命名: save_<uuid>.woworld.tmp → save_<uuid>.woworld
```

### 6.2 崩溃场景分析

| 崩溃时机 | 后果 | 恢复 |
|---------|------|------|
| 写临时文件中 | `.tmp` 损坏，旧 `.woworld` 完好 | 启动时清理 `.tmp` |
| rename 前 | `.tmp` 完整但未 rename | 启动时清理 `.tmp`（旧存档完好） |
| rename 中 | POSIX rename 是原子的——要么旧文件在，要么新文件在 | 不存在"半新半旧" |

### 6.3 存档后刷新运行时读句柄

存档通过临时文件+rename写入新文件。rename后，游戏运行期间已打开的只读LMDB句柄仍指向旧inode（OS保证旧inode在最后一个引用关闭前不回收）。

**存档完成后必须刷新运行时读句柄**，否则后续 `create_read_txn()` 返回的读事务将看到存档前的旧数据：

```rust
// save() 成功后
fn refresh_read_env(&self, save_path: &Path) -> Result<()> {
    // 1. 关闭旧只读环境
    let mut env_guard = self.env.write().unwrap();
    *env_guard = None;
    // 旧 inode 引用计数减1。如果所有引用释放，OS回收旧 inode

    // 2. 打开新只读环境（指向rename后的新文件）
    let new_env = open_lmdb_read_only(save_path)?;
    *env_guard = Some(new_env);
    Ok(())
}
```

此刷新在 Phase 2（写入+rename）完成后、Phase 3（confirm清理）后执行。不影响当前运行中的读事务（它们的 inode 引用尚未释放）。

详细流程见 [[004-读档流程与崩溃恢复#三、运行时渐进加载|004 §三]]。

### 6.3 .tmp 清理

启动时扫描所有 `saves/` 目录 → 删除所有 `.tmp` 文件（残留）。

### 6.4 写入后健全性验证

rename 完成后 → 以只读打开刚写的存档 → 遍历**全部** named_db，每个至少读 1 条（开销 <1ms）→ 验证：
1. 所有 named_db 存在且非空（对预期非空的 db）或存在（对可为空的 db）
2. magic + global_version 合法
3. LMDB 页级统计：`ms_ps_pages > 0`、`ms_entries > 0`
4. 文件大小合理性：`file_size < mapsize` 且 `file_size > header_size`

全部通过 → 成功 ✓。任一失败 → 旧存档保留，损坏文件移至 `.corrupted`。
