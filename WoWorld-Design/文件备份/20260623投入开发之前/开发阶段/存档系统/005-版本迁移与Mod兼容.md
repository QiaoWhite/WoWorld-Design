## 一、版本管理模型

### 1.1 两级版本

| 级别 | 存储位置 | 含义 | 变更示例 |
|------|---------|------|---------|
| **全局版本** | `SaveHeader.global_version` | 存档**文件结构**的版本 | named_db 增减/改名、Header 字段增删 |
| **模块版本** | `SaveHeader.module_versions: BTreeMap<String, u32>` | 每个模块**数据格式**的版本 | MemoryEntry 新增字段、ItemState 新增枚举变体 |

**两个版本独立递增。** 改动 NPC 记忆结构 → 只影响 `module_versions["npc"]`，`global_version` 不变。

### 1.2 模块版本号规则

- 版本号**线性递增**：1 → 2 → 3 → 4 → ...
- `current_version()` 返回当前代码的版本号
- `migrate(from_version, txn)` 负责从 `from_version` 迁移到 `current_version()`——`from_version` 到 `current_version()` 的差可以 **>1**（如 v3 → v5）。模块内部用 `while` 循环链式执行所有中间版本。每个中间版本的迁移函数必须存在——缺失中间版本 = 模块 bug（代码审查问题，非运行时约束）
- `validate_after_migration()` 在迁移后、事务提交前执行——致命模块验证失败 → 事务回滚

### 1.3 版本链实现

```rust
impl SaveableModule for NpcModule {
    fn current_version(&self) -> u32 { 5 }

    fn migrate(&self, from_version: u32, txn: &RwTransaction) -> Result<()> {
        let mut current = from_version;
        while current < self.current_version() {
            match current {
                1 => { migrate_1_to_2(txn)?; current = 2; }
                2 => { migrate_2_to_3(txn)?; current = 3; }
                3 => { migrate_3_to_4(txn)?; current = 4; }
                4 => { migrate_4_to_5(txn)?; current = 5; }
                _ => return Err(MigrationError::UnknownVersion(current)),
            }
        }
        Ok(())
    }
}

fn migrate_1_to_2(txn: &RwTransaction) -> Result<()> {
    // 示例: v1 的 MemoryEntry 没有 emotion_tags，
    //       v2 新增了 emotion_tags: Vec<String>。
    // 迁移: 遍历所有记忆 → 读取 v1 格式 → 加空 emotion_tags → 写回 v2 格式
}
```

**迁移是幂等的吗？** 不是——迁移修改磁盘数据。但迁移只执行一次（from_version → current_version），不会重复执行。

## 二、迁移调度（读档时）

### 2.1 完整迁移流程

详见 [[004-读档流程与崩溃恢复#一、读档流程|004 §一·Phase 3]]。

关键步骤：
1. **迁移前自动备份**（`.bak`）
2. 关闭只读 LMDB → 以读写打开 → 迁移 → 提交 → 关闭 → 重新以只读打开
3. LMDB 打开三次的总开销 <200ms——版本迁移极少发生（仅游戏升级时），可接受
4. 迁移失败 → 事务回滚 + `.bak` 完好

### 2.2 模块分裂场景

如果 NPC 模块 v3 分裂为 `npc_identity` 和 `npc_memory` 两个模块：

```
存档 v3 (named_db "npc"):
  npc/identity/0001 → NpcIdentity (v3)
  npc/memory/0001  → NpcMemory (v3)

存档 v4 (named_db "npc_identity" + named_db "npc_memory"):
  npc_identity 模块.migrate(from=3, txn):
    从 "npc" db 读 npc/identity/* → 反序列化 v3 → 序列化 v4 → 写 "npc_identity" db
  npc_memory 模块.migrate(from=3, txn):
    从 "npc" db 读 npc/memory/*  → 反序列化 v3 → 序列化 v4 → 写 "npc_memory" db
```

**迁移后旧 `"npc"` named_db 不删除——只读保留。** 删除旧 db 不可逆（LMDB `drop_db` 永久删除），保守保留。

### 2.3 迁移失败

迁移失败 → 事务回滚 → 写入失败：
- 迁移直接在原存档文件上执行（在 LMDB 写事务内）。事务回滚后文件不变
- 迁移前已创建 `.bak` 备份作为额外的安全网
- 提示"版本迁移失败。原始存档的备份已保存为 .bak。请报告此问题。"

## 三、ConsumableEffect 模式的迁移约定

### 3.1 问题

WoWorld 中存在"Schema 由领域模块定义，数据由消费方模块存储"的模式：

| Schema 所有者 | 数据类型 | 存储方（named_db） |
|-------------|---------|------------------|
| 音频模块 | VoiceProfile | NPC (`"npc"`) |
| 音频模块 | AudioMaterial | Items (`"items"`) |
| Life 模块 | ConsumableEffect schema | Items (`"items"`) |

当 Schema 所有者升级版本 → 存储方 named_db 中的数据需要迁移。但数据在**另一个模块**的 named_db 里。

### 3.2 迁移约定——惰性迁移（唯一采用方案）

采用**惰性迁移**——不依赖迁移时跨模块协调：

1. **存储方**在 named_db 中存储 `(schema_version: u32, data: Vec<u8>)` 对——标注数据使用的 Schema 版本
2. **Schema 所有者**（如 Audio 模块）提供版本转换函数：
   ```rust
   trait SchemaProvider {
       fn current_schema_version(&self) -> u32;
       fn convert_from_version(&self, from_version: u32, data: &[u8]) -> Result<Vec<u8>>;
   }
   ```
3. **存储方**在自己的 `load()` 中做**批量转换**——读取所有数据 → 检测版本 → 调用 Schema 所有者的 `convert_from_version()` → 存入内存（已是新格式）
4. 批量转换在加载画面预算内完成（≤5s）。运行时零转换开销——无首次访问尖刺

**优势**：
- 零模块间迁移时协调——Schema 所有者和存储方不互相调用 `migrate()`
- 旧格式数据保留在 LMDB 中不变（只读）——只在内存中表现为新格式
- 下次存档时自动以新格式写回——自然渐进升级

**存档系统不强制此方案——但强烈推荐。** ConsumableEffect 模式涉及的模块（Audio→NPC/Items/Life/WorldGen、Life→Items）应统一使用惰性迁移。

### 3.3 原则

```
Schema 定义在 A，数据存储在 B:
  · A 负责定义 v1→v2 的转换逻辑（SchemaProvider trait）
  · B 存储 (schema_version, data_blob) 对
  · B 在 load() 中批量调用 A 的 convert_from_version() 进行转换
  · 转换在加载画面预算内完成——运行时零开销
```

## 四、Mod 兼容

### 4.1 Mod 操作分类

| 操作 | 对已有存档的影响 |
|------|----------------|
| **加 Mod** | ✅ 天然安全。新物品/蓝图从未生成过 → 无引用可悬空 |
| **删 Mod** | ⚠️ 悬空引用。存档中可能含有该 Mod 定义的物品/建筑 |
| **更新 Mod** | ⚠️ 取决于改动类型。属性修改 → 混合效果（旧实例保留旧属性）。破坏性改动 → 和删 Mod 相同 |

### 4.2 删 Mod 检测

存档 Header 的 `mod_manifest` 记录存档时的 Mod 清单。加载时和当前 Mod 清单比对：

```rust
fn detect_mod_changes(saved: &[ModEntry], current: &[ModEntry]) -> ModDelta {
    let saved_ids: HashSet<&str> = saved.iter().map(|m| m.mod_id.as_str()).collect();
    let current_ids: HashSet<&str> = current.iter().map(|m| m.mod_id.as_str()).collect();

    ModDelta {
        removed: saved_ids.difference(&current_ids).copied().collect(),
        added: current_ids.difference(&saved_ids).copied().collect(),
        version_changed: saved.iter()
            .filter_map(|s| {
                let c = current.iter().find(|c| c.mod_id == s.mod_id)?;
                if c.mod_version != s.mod_version { Some(s.mod_id.clone()) } else { None }
            })
            .collect(),
    }
}
```

### 4.3 降级策略

检测到 Mod 缺失 → 弹窗：

```
┌──────────────────────────────────────────────┐
│ ⚠ 缺少 Mod                                   │
│                                              │
│ 此存档依赖以下已卸载的 Mod：                    │
│ · 龙裔武器包 v1.2（53 个物品）                 │
│ · 北方建筑蓝图 v0.9（12 个建筑）               │
│                                              │
│ 选择处理方式：                                 │
│ ● 冻结——保留相关物品/建筑，但不可交易/修理       │
│ ○ 替换——用基础物品替换（此操作不可撤销）         │
│                                              │
│ [重新安装 Mod]  [冻结继续]  [替换并继续]        │
└──────────────────────────────────────────────┘
```

**冻结**：保留数据，标记为 `Frozen(mod_id)`。物品可在库存中查看，但不可交易、不可修理、不可通过配方创建。

**替换**：用游戏基础物品定义替换。该 Mod 的所有 ItemDefId → 基础 fallback。此操作不可逆。

**重新安装 Mod**：回到主菜单，安装 Mod 后重新加载存档 → 冻结的物品自动恢复。

### 4.4 冻结的实现

```rust
enum ItemProvenance {
    Vanilla,
    Modded { mod_id: String, mod_version: String },
}

struct ItemState {
    // ...
    provenance: ItemProvenance,
    frozen: bool,  // true = Mod 已卸载，物品冻结
}

// 冻结物品的限制
fn can_trade(item: &ItemState) -> bool { !item.frozen }
fn can_repair(item: &ItemState) -> bool { !item.frozen }
fn can_be_crafted(item_def: &ItemDef) -> bool {
    match &item_def.provenance {
        ItemProvenance::Vanilla => true,
        ItemProvenance::Modded { mod_id, .. } => {
            mod_manager.is_active(mod_id)  // Mod 存在 → 可制造
        }
    }
}
```

### 4.5 更新 Mod—属性变化

Mod 更新了物品属性（如铁矿石 `stack_size: 64` → `128`）→ 已有存档中的铁矿石实例：

- 已存在的堆叠不受影响（物品实例在存档中以"当前堆叠数"存储，不引用模板）
- 新生成的堆叠用新上限
- 修理公式变了 → 旧物品保留旧的 `max_durability`，但下次修理时用新公式计算新的最大耐久度——新旧混合效果

**存档系统不需要干预。** 属性更新是自然的"新旧混合"。

### 4.6 Mod 冲突预防

ItemDefId 分配规则：
- 官方物品：`0x00` 前缀（0—16,777,215）
- Mod 物品：`0x01—0xFF` 前缀（Mod 作者自行分配）

冲突了是 Mod 之间的事——存档系统不处理。

## 五、迁移的不可逆性

### 5.1 数据丢失是迁移的固有属性

如果 NPC 模块 v2→v3 删除了 `favorite_color` 字段 → 旧存档里这个字段的数据永久丢失。将来 v4 加回了 `favorite_color` → 无法恢复 v2 的旧值。

**原则**：迁移函数明确标注哪些字段被删除、哪些被修改。版本号注释记录迁移历史。

```rust
// 版本历史:
//   v1 (2026-06): 初始格式。字段: name, age, memories
//   v2 (2026-09): 新增 emotion_tags: Vec<String>。迁移: v1 的 emotion_tags 填空 Vec
//   v3 (2027-01): 删除 favorite_color (u32)。迁移: v2→v3 丢弃此字段
//   v4 (2027-06): 合并 hp/mp 为 Vitals 结构体。迁移: v3→v4 提取 hp+mp 构造 Vitals
//   v5 (当前): 当前格式
```

### 5.2 存档版本升级的单向性

存档迁移到新版本后 → 如果玩家想"降级"回旧版游戏 → 需要 `version_requirement` 标记：

```rust
fn min_compatible_version(&self) -> u32 { self.current_version() }
```

未来有可能支持"存档导出为旧格式"——但这不是 v1.0 的目标。

## 六、保留矛盾与设计决策

### 6.1 与原始世界生成设计的差异

世界生成模块 ([[../世界生成/README|世界生成 README]]、[[../世界生成/014-生成后校验与完整性保证|014 校验]]) 中 P13 包含 LMDB 写入，且声明"地形/群系/资源从种子复现不存储"。

当前存档系统设计：
- **P13 不再写 LMDB**——LMDB 写入权完全归存档系统
- **全量快照替代种子零存储**——所有模块数据全量存入初始存档。种子从"空间优化手段"重新定位为"容错恢复手段"

详见 [[000-跨模块依赖与接口全面清单|000 跨模块依赖]] 中世界生成模块的修正方向。

### 6.2 与历史模块独立 LMDB 的差异

历史模块 ([[../历史/006-外部接口与性能预算|历史 006]]) 设计了独立 LMDB 文件 `world_<seed>_history.lmdb`。当前设计改为存档内 named_db `"history"` + `"grand_journal"`。

选择理由：跨模块原子写入优先于独立文件带来的"损坏隔离"和"独立缓存"。详见 [[001-存档系统总纲|001 总纲]] 讨论记录。

### 6.3 Wear 状态机

技术栈方案 v3 声明 "wear 状态机 × 经济 × 审美（跨四模块）"。此声明来自早期设计草案——独立的 wear 状态机设计文档从未创建。各模块（物品耐久度、载具损伤、建筑老化、身体损伤）各自管理磨损数据。存档系统将每个磨损领域视为对应模块的持久化数据——不感知"统一磨损状态机"的存在。
