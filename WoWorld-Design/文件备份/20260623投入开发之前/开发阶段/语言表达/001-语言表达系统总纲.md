## 一、模块定位

### 1.1 为什么需要这个模块

WoWorld 的"文字/书写/语言/对话"当前分散在 10+ 个模块中，各自独立重复发明同一件事——"一段可表达的内容"：

| 碎片位置 | 当前做法 | 问题 |
|---------|---------|------|
| 历史 `Work.language` | `LanguageId` 被引用但**从未定义**——幽灵类型 | 语言系统不存在 |
| 历史 `Work`/`PhysicalBook`/`LifeTrace` | 完整的数据模型——但只服务历史模块自己 | 其他模块无法复用"书"的概念 |
| 魔法 方程式/法阵/咒语 | 纯概念描述——零数据结构 | 魔法系统统一中枢无代码表示 |
| 魔法 卷轴/魔导书 | 概念描述——没有文本编码模型 | "魔导书条目写什么"无格式定义 |
| 物品 `ItemEnt.inscription: Option<String>` | 裸字符串——无语言/作者/可读性 | 铭文没有语义 |
| NPC `CasualChat` | 裸枚举值——无主题/轮次/内容 | 对话系统无骨架 |
| 技能 `humanities.linguistics` | 只有技能名 | 语言学和语言使用零连接 |
| 世界生成 招牌/路牌/建筑铭文 | **零** | 完全空白 |
| 所有模块 "参数化文本" | 范式被所有人引用——但**无拼装函数** | 最后一步缺失 |

### 1.2 模块边界

**语言表达模块提供什么**：
- 语言系统——`LanguageId`/`ScriptId`/`LanguageFamily`/`Proficiency` 的类型定义和谱系模型
- 文本管道——从"参数化内容"到"玩家看到的文字"的完整生成链路
- 对话系统——多参与者（2→1000+）的会话机制
- 统一句柄 `ExpressionRef`——指向世界上任何"一段可表达的内容"的轻量指针

**语言表达模块不拥有什么**：
- 不拥有书籍的数据——历史模块保留 `Work`/`PhysicalBook` 所有权
- 不拥有魔法方程式的数据——魔法模块保留 `MagicEquation`/`Formation` 所有权
- 不拥有 NPC 的记忆/个性/关系——NPC 模块保留所有权
- 不生成"世界的故事"——故事由历史/魔法/NPC/世界生成模块共同产出

**语言表达模块是翻译官，不是作者。**

> ⚠️ **[CHG-024 所有权迁移]**: CommunicationNorms 定义权已移至文化系统。本模块为只读消费者。

### 1.3 与其他模块的关系

```
                        woworld_types
                      (纯类型，零依赖)
                     ╱               ╲
            lang_expression      各业务模块
           (行为实现)          (impl ContentResolver)
                │                    │
                └────────┬───────────┘
                         │
                  主循环/Godot 集成层
                  (唯一依赖聚合点)
```

## 二、三层 Crate 架构

### 2.1 woworld_types——纯数据类型层

```rust
// woworld_types/ —— 零行为，零业务依赖。被所有人依赖。

// language.rs
pub struct LanguageId(u32);           // 高16位=语系，低16位=语系内编号
pub struct ScriptId(u16);            // 文字系统——独立于语言
pub struct Proficiency(f32);         // 0.0=完全不懂 → 1.0=母语
pub struct LanguageFamily { ... }    // 谱系 DAG

// expression.rs
pub struct ExpressionRef {           // 8 字节 Copy handle
    carrier_type: u8,                // 载体类型标签
    local_id: [u8; 7],              // 模块内 ID
}
pub enum Encoding { Text, Geometric, Audio }
pub struct ContentMetadata { ... }
pub enum ReliabilityHint { Firsthand, Secondhand { hops }, Written { bias, gen }, Folklore { age }, Unknown }

// readable.rs
pub trait ContentResolver: Send + Sync { ... }
pub trait LocatableReadable: ContentResolver { ... }
pub struct ResolvedContent { ... }

// reader.rs
pub struct ReaderContext { ... }
pub enum ReadingAid { ComprehendLanguages { ... }, MagicalLens { ... }, TranslationField { ... } }
pub struct ReadResult { ... }

// carrier_type.rs —— 保留范围分配
// 0x01-0x0F: History
// 0x10-0x1F: Magic
// 0x20-0x2F: Items
// 0x30-0x3F: 预留

// params.rs —— 标准参数名常量 (~50个)
pub mod param_names {
    pub const YEAR: &str = "year";
    pub const RULER: &str = "ruler";
    pub const ITEM_NAME: &str = "item_name";
    // ... 等 ~50 个
}
```

### 2.2.1 修辞化渲染——新增依赖 (★ v1.1 CHG-057)

```rust
// lang_expression/text_generator.rs 新增——修辞装置渲染

/// 修辞装置类型
pub enum RhetoricalDevice {
    Simile,                   // 明喻——"淬火就像冬天的冷风吹过热铁"
    Metaphor,                 // 隐喻——"我们是守护人界的盾牌"
    AnalogicalExplanation,    // 类比解释——"锻造好比烹饪——火候对了才成"
}

/// 修辞标记——附加到 NpcResponse，供 Godot UI 区分常规 vs. 修辞文本
pub struct RhetoricalMarker {
    pub device: RhetoricalDevice,
    pub domain_a: String,          // 源域
    pub domain_b: String,          // 目标域
    pub confidence: f32,           // Creative Leap 置信度
}

/// 纯函数——从 NPC 认知参数计算出修辞能力
/// 无状态，不查数据库
pub fn rhetorical_ability(
    cognitive_style: &CognitiveStyle,     // from NPC cognitive
    wisdom_accumulation: f32,             // from MentalModel
    deliberation_depth: f32,              // from CognitiveTide (CHG-057)
) -> f32;

/// 消费 woworld_core::domain_similarity_raw() ——跨域结构匹配
/// UtteranceConcept.domain_sig: Option<u64> ——来自 NPC 认知系统
```

**关键依赖**:
- `woworld_core::domain_similarity_raw(u64, u64) → f32` — 从 UtteranceConcept 的 domain_sig 计算域相似度
- `UtteranceConcept.domain_sig: Option<u64>` — 由 NPC 认知系统在 utterance 成形时注入
- `CognitiveStyle` / `MentalModel.wisdom_accumulation` / `CognitiveTide.deliberation_depth` — 通过 DialogueContext 依赖注入

### 2.3 依赖方向——严格无循环

```
woworld_types ← lang_expression          (lang_expression 依赖 types)
woworld_types ← history / magic / items / npc  (业务模块依赖 types)
lang_expression → 不依赖任何业务模块！
主循环/集成层 → 依赖所有模块（正常的组装点）
```

## 三、ExpressionRef——统一的表达句柄

### 3.1 设计

```rust
/// 指向世界上任何"一段可表达的内容"的轻量指针
/// 8 字节，Copy，可存入 NPC 记忆、大日志、对话锚点
/// 范式对齐 ItemEntId——已有模块已理解和接受
#[derive(Copy, Clone, Hash, Eq, PartialEq)]
pub struct ExpressionRef {
    carrier_type: u8,   // 载体类型标签——决定由哪个 ContentResolver 处理
    local_id: [u8; 7],  // 模块内 ID——各模块自行定义含义
}
```

### 3.2 核心性质

- **不拥有数据**——ExpressionRef 只是指针。内容所有权在各模块
- **Copy 类型**——无所有权负担，可在任何 struct 中自由存储
- **可比较**——`Hash + Eq`，可用作 HashMap key
- **不反向依赖**——ExpressionRef 不包含任何模块特定的类型信息
- **8 字节**——栈上传递 ~1ns，不涉及堆分配

### 3.3 使用场景

| 场景 | 存储位置 | 用途 |
|------|---------|------|
| NPC 记忆引用 | `EventMemory.source_expression: Option<ExpressionRef>` | "我从这本书读到的" |
| 大日志条目 | `JournalEntry.source_expression: Option<ExpressionRef>` | "从碑文中读到" |
| 对话话题锚点 | `ConversationTopic` 中关联 | "聊聊上次那本编年史" |
| 世界生成放置 | `place_readable(location, ExpressionRef::from(book))` | 统一的可读物放置接口 |
| 物品铭文 | `ItemEnt.inscription: Option<ExpressionRef>` | 替换裸 `Option<String>` |

## 四、核心数据流

### 4.1 静态文本（书/碑文/卷轴）

```
玩家点击可读物
  │
  ▼
Godot → ExpressionRef
  │
  ▼
ExpressionRegistry::read(expr_ref, reader_ctx)
  │
  ├→ HashMap<u8, Box<dyn ContentResolver>>  → O(1) 分发
  │
  ▼
HistoryContentResolver::resolve(local_id, ctx)
  │  └→ LMDB 查询 → Work/PhysicalBook → ResolvedContent { template_id, params, encoding, condition, metadata }
  │
  ▼
TextGenerator::generate(content, ctx)
  │  ├→ 模板查找 (O(1) HashMap)
  │  ├→ 参数填充 (字符串替换)
  │  ├→ 可读性计算 (proficiency × condition × aids)
  │  ├→ 退化应用 (破损/语言不熟 → '█')
  │  └→ ReadResult { text, language, condition, metadata, reliability }
  │
  ▼
Godot UI → 显示文字
```

### 4.2 动态文本（NPC 对话）

```
NPC 行为树/GOAP 评估
  │
  ├→ 生成 DialogueIntent（5种驱动）
  │
  ▼
玩家交互 / NPC 走近
  │
  ▼
Conversation::start()
  │  └→ 构建 DialogueContext（调用者在外部从各模块收集数据）
  │
  ▼
TopicSelector::select_topics(npc, ctx)
  │  └→ Alias Method 加权采样 → Vec<ConversationTopic>
  │
  ▼
玩家选择话题（或语音/搜索框）
  │
  ▼
TopicResolver::resolve(topic, npc, ctx, generator)
  │  ├→ 查询 NPC 记忆/知识
  │  ├→ PhaticLayer 应酬片段插入
  │  ├→ PersonalityFilter 个性调整
  │  ├→ TextGenerator 产出文本
  │  └→ ★ 修辞化渲染 (CHG-057 v1.1)
  │       ├→ 检测: rhetorical_ability > 0.4 且 domain_similarity ∈ [0.1, 0.5]
  │       ├→ 装置: simile / metaphor / analogical explanation
  │       └→ 消费: UtteranceConcept.domain_sig via domain_similarity_raw()
  │
  ▼
NpcResponse { text, rhetorical_markers, emotion, player_options }
```

## 五、设计哲学

**语言表达模块不生成"世界的故事"——它提供统一的管道，让各模块生成的内容能被玩家读到、听到。**

- 历史模块贡献了"发生过什么"——程序化事件的参数
- NPC 模块贡献了"谁在说、和谁说"——记忆、个性、关系
- 世界生成贡献了"在哪里、叫什么"——地名、文化参数、语言分布
- 魔法模块贡献了"魔法的文本形态"——方程式的三种编码

语言表达模块是翻译官，不是作者。它不拥有内容——它让内容被感知。
