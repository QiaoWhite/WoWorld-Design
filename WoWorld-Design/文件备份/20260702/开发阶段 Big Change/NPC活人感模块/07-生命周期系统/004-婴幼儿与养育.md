## 一、InfantDependency：营养依赖状态机

### 1.1 核心概念

InfantDependency 描述的不是 NPC 的"能力"——是 NPC 的**营养获取方式**。它是生命模块拥有的生物学事实，存储在婴儿的 LifeEntity 上。

婴儿和成人使用同一套 GOAP、同一套 Vitals、同一套认知引擎。差异在于：
- 动作库只有先天反射（cry/suck/sleep/grasp）
- planning_horizon=1（只能一步规划："我饿了→哭"）
- 营养来源是母亲，不是自主觅食

```rust
pub enum InfantDependency {
    Nursing {
        mother_id: EntityId,
        last_fed: GameTime,
    },
    Weaning {
        mother_id: EntityId,
        independence: f32,   // 0.0 (完全依赖) → 1.0 (可独立进食)
    },
    Weaned,
}
```

### 1.2 状态转换

```
出生 ──→ Nursing ──→ Weaning ──→ Weaned
            ↑           │
            │ (罕见)    │
            └───────────┘
           (若母亲死亡且无替代，可能回到 Nursing，
            但更常见的是直接触发婴儿死亡风险)
```

**Nursing→Weaning**：当婴儿的 `age_years` 超过种族的 weaning_age（人类 ~2 岁），且 `independence` 积累到 > 0.3 时。independence 的增长来自婴儿的自主探索和母亲的渐进放手。

**Weaning→Weaned**：当 `independence >= 1.0`。约在 ~3-4 岁（人类）。

**母亲死亡**：如果在 Nursing 阶段母亲死亡或失踪 → 婴儿进入"孤儿路径"（见第四节）。

## 二、L1 模拟：母亲的 GOAP 驱动喂奶

### 2.1 母亲的目标

婴儿不需要自己处理喂奶——母亲（或替代照料者）的 GOAP 中包含 `CareForInfant` 目标。这个目标与母亲的所有其他目标（吃饭/工作/社交/休息）在同一个 GOAP 中平等竞争：

```
母亲 GOAP 世界状态：
  infant_hungry: bool  ← 从婴儿 Vitals.hunger > 0.5 派生
  infant_distance: f32 ← 从空间查询

GOAP 目标 "喂饱婴儿"：
  优先级 = infant_hunger_level × (1.0 - infant_distance / max_distance)
  → 婴儿越饿越近 → 优先级越高
  → 婴儿距离远且不饿 → 优先级低，母亲可以做别的

GOAP 目标 "回到婴儿身边"：
  优先级 = neuroticism × agreeableness × (infant_distance / safe_distance)
```

### 2.2 喂奶动作

```
母亲 GOAP 选择了 CareForInfant：
  → 导航到婴儿位置
  → 执行 NursingAction（耗时约 15-30 分钟游戏时间）
  → 婴儿 Vitals.hunger → 1.0（饱腹）
  → 婴儿 InfantDependency.last_fed → 当前时间
  → 母亲 Vitals.stamina 轻微消耗
```

### 2.3 母亲的张力——喂奶 vs 其他需求

如果母亲在喂奶之前自己的 hunger 已经很高 → 她可能先去找食物。婴儿在哭——但她实在太饿了。或者有敌人来袭——她必须带着婴儿逃跑还是先战斗？

这是 GOAP 的正常运转——没有"母婴系统"的特殊裁决。同一个 GOAP 引擎同时评估"喂奶""我自己饿了""有危险""这件工作再不完成就丢了"——产生最优动作。

## 三、L3/L4 模拟：统计近似

L3/L4 下的婴儿不模拟逐次喂奶事件。退化为纯统计：

```rust
fn infant_l3_l4_survival_check(
    infant: &mut LifeEntity,
    delta_days: f32,
    world: &World,
    rng: &mut Rng,
) {
    let dep = infant.infant_dependency.as_ref();
    let mother_id = dep.and_then(|d| d.mother_id());
    let mother = mother_id.and_then(|id| world.get_entity(id));
    let mother_alive = mother.map(|m| !m.is_dead).unwrap_or(false);
    let community_support = world.community_support_index(infant.position);
    
    let monthly_survival = match infant.infant_dependency {
        Some(InfantDependency::Nursing { .. }) if mother_alive && community_support > 0.5 => 0.998,
        Some(InfantDependency::Nursing { .. }) if mother_alive => 0.95,
        Some(InfantDependency::Nursing { .. }) if !mother_alive && community_support > 0.7 => 0.85,
        Some(InfantDependency::Nursing { .. }) if !mother_alive => 0.70,
        Some(InfantDependency::Weaning { independence, .. }) if mother_alive => 0.999,
        Some(InfantDependency::Weaning { .. }) if !mother_alive => 0.90 + community_support * 0.08,
        Some(InfantDependency::Weaned) | None => 1.0,
    };
    
    let survival = monthly_survival.powf(delta_days / 30.0);
    if rng.gen::<f32>() > survival {
        infant.die(DeathCause::Starvation);
    }
}
```

当玩家进入该区域（L4→L1 切换）时，婴儿已经是确定的生死状态。不需要回溯模拟"母亲是否喂过奶"。

## 四、孤儿——涌现机制

### 4.1 三条自然路径

母亲死亡后，婴儿的命运不由系统决定——由附近 NPC 的既有状态决定：

| 路径 | 触发条件 | 机制 |
|------|---------|------|
| **亲属接管** | 有血缘关系的 NPC 在感知范围内 | 关系引擎触发"亲属遇险→帮助"→ GOAP 添加 CareForInfant 目标。不需要"亲属抚养系统" |
| **社区接管** | 无亲属但聚落有收养文化 | 高 agreeableness + low existing_children 的 NPC 可能接管。这**不是**收养系统——是 GOAP 目标出现在非亲属 NPC 上，因为感知到了哭泣的婴儿 |
| **无人接管** | 无亲属、社区不收养、或孤儿未被发现 | 婴儿饥饿死亡。DeathCause::Starvation。这是涌现的悲剧 |

### 4.2 接管后的数据流

```
照料者 NPC 看到婴儿 → GOAP "CareForInfant" 目标出现 → 
  照料者接近婴儿 → 关系系统建立 Parent-Child(adoptive) Relationship →
  InfantDependency 中 mother_id 更新为照料者
```

**收养不需要"收养系统"。** 它是关系系统 + GOAP + InfantDependency 的自然产物。

### 4.3 孤儿死亡的涌现性质

同一个聚落、同样失去母亲的婴儿：
- 如果祖母健在且在附近 → 几乎一定会被祖母收养
- 如果聚落文化 adoption_accepted=true 且有多个高 agreeableness NPC → 大概率被社区接管
- 如果聚落在战争中、所有人都在求生 → 极大概率死亡
- 如果婴儿在野外、远离任何聚落 → 几乎必死

**没有"孤儿保护系统"。** 这就是涌现。

## 五、养育——哺乳到断奶的过渡

### 5.1 断奶过程

断奶不是一帧内的切换。`InfantDependency::Weaning.independence` 从 0 渐进到 1：

```
independence 增长来源：
- 婴儿自主进食的经历（自己拿起食物放进嘴里）
- 母亲主动减少喂奶频率
- 婴儿对固体食物的消化能力增长（生物学）

independence = f(尝试次数, 母亲退出意愿, 生物成熟度)
```

### 5.2 母亲何时主动断奶

没有"断奶系统"。母亲减少喂奶频率是因为：
- 婴儿已经吃了一部分固体食物 → infant_hungry 设为 true 的时机变晚
- 母亲自己的经济/工作需求 → GOAP 中工作目标的优先级上升
- 又怀孕了 → 新的妊娠会抑制哺乳（生物学事实）→ 奶水减少
- 文化规范——某些文化中有"断奶年龄"的传统，这个规范以 MentalModel 的形式存在于母亲脑中

### 5.3 不同物种的养育方式——留接口

```rust
/// 新生儿营养模式——当前只有母乳，未来可扩展
pub trait InfantNutrition: Send + Sync {
    fn method(&self) -> NutritionMethod;
    fn source_entity(&self) -> Option<EntityId>;
    fn dependency_duration_days(&self, race: &RaceTraits) -> f32;
    fn tick_l1(&self, infant: &mut LifeEntity, caregiver: &mut LifeEntity);
    fn l3_survival(&self, infant: &LifeEntity, caregiver_alive: bool, community_support: f32) -> f32;
}

pub enum NutritionMethod {
    Breastfeeding,     // 默认实现
    // 未来扩展：
    // EggBrooding,    // 孵蛋——母体在蛋旁边保温
    // BudFeeding,     // 出芽生殖——母体提供营养液
    // SporeMaturation,// 孢子成熟——在环境中吸收营养
    // MagicalNurturing,// 魔法养育——魔力灌注
}

/// 当前默认实现
pub struct Breastfeeding;
```

## 六、婴幼儿行为涌现

### 6.1 婴儿的 GOAP

婴儿的 GOAP 一直在运行：

```
# 新生儿
action_repertoire = [cry, suck, sleep, grasp]
planning_horizon = 1
mental_models = []  // 尚未形成任何模型

goal: reduce_hunger
→ plan: [cry]  (唯一可用的动作——引起母亲注意)

goal: reduce_fatigue
→ plan: [sleep]

# 6 个月
action_repertoire = [cry, suck, sleep, grasp, smile, babble, crawl]
planning_horizon = 1  // 仍是一步——"爬向母亲"还不是完整规划
mental_models = ["crying_brings_mother"(conf 0.7)]  // 第一条模型！

goal: reduce_hunger
→ plan: [cry]  // 仍然主要靠哭

# 1 岁
action_repertoire = [cry, suck, sleep, grasp, smile, say("妈妈"), walk_unsteady, point]
planning_horizon = 2  // 开始两步规划
mental_models = ["crying_brings_mother"(conf 0.9), "pointing_gets_what_I_want"(conf 0.5)]

goal: get_toy_on_high_shelf
→ plan: [point_at_toy, cry]  // 两步："指玩具" → 如果母亲没反应 → "哭"
```

### 6.2 婴儿的情感

婴儿有完整的情绪引擎。情感不是"幼儿版本"——是完整情感引擎在少量输入下的自然输出：

```
感知：母亲离开视野
情绪引擎输入：attachment_figure_absent + 未知原因
人格（即使由 mental_seed 派生，婴儿已有基础气质）：
  - neuroticism 高 → 剧烈不安→哭闹
  - neuroticism 低 → 轻微不安→可能继续睡觉
```

新生儿不是在运行"婴儿情感系统"——他们在运行和成年人完全相同的情感引擎，只是输入更简单、解释更基础。
