## 一、FaithQuery trait——完整签名

```rust
/// 信仰系统的只读查询接口——所有模块获取信仰数据的唯一入口。
/// 对标 CultureQuery（文化 006）。
/// 零分配，实现方不透明。高频方法通过 SoA 缓存 O(1)。
pub trait FaithQuery: Send + Sync {
    // ──── 基础查询 ────
    
    /// 获取信仰定义——不存在则 None
    fn faith_def(&self, id: FaithId) -> Option<&FaithDefinition>;
    
    /// 信仰名称（内名 + 外名）
    fn faith_name(&self, id: FaithId) -> &FaithName;
    
    /// 神学参数
    fn theology(&self, id: FaithId) -> &FaithTheology;
    
    /// 所有信仰 ID（包括已消亡）
    fn all_faith_ids(&self) -> &[FaithId];
    
    /// 当前活跃的信仰（未消亡）
    fn active_faiths(&self) -> Vec<FaithId>;
    
    // ──── 日历查询 ────
    
    /// 神圣日列表
    fn holy_days(&self, id: FaithId) -> &[HolyDay];
    
    /// 某日的斋戒规则
    fn fasting_rules(&self, id: FaithId, day: u16) -> Vec<&FastingRule>;
    
    /// 主要神祇列表
    fn primary_deities(&self, id: FaithId) -> &[DeityInfo];
    
    // ──── 空间查询 ────
    
    /// 查询某位置的主导信仰——从最近 settlement 的 snapshot 聚合 O(log n)
    fn dominant_faith_at(&self, pos: WorldPos) -> Option<FaithId>;
    
    /// 某聚落的信仰分布（按信徒数降序）
    fn faith_distribution(&self, settlement_id: SettlementId) -> &[(FaithId, f32)];
    
    /// 获取聚落信仰快照——供高频查询调用
    fn settlement_snapshot(&self, settlement_id: SettlementId) -> Option<&SettlementFaithSnapshot>;
    
    // ──── 教际关系 ────
    
    /// 信仰 A 对信仰 B 的宽容度 (0-1)
    fn tolerance_toward(&self, subject: FaithId, target: FaithId) -> f32;
    
    /// 两信仰之间的敌对度 (0-1)
    fn inter_faith_hostility(&self, a: FaithId, b: FaithId) -> f32;
    
    /// 两信仰之间的完整关系状态
    fn inter_faith_relation(&self, a: FaithId, b: FaithId) -> InterFaithRelation;
    
    // ──── 魔法关系 ────
    
    /// 信仰对魔法的态度
    fn magic_relation(&self, id: FaithId) -> MagicReligionRelation;
    
    // ──── 规范与禁忌 ────
    
    /// 禁忌列表
    fn taboos(&self, id: FaithId) -> &[Taboo];
    
    /// 核心教义
    fn doctrines(&self, id: FaithId) -> &[Doctrine];
    
    /// 神职角色定义
    fn clergy_roles(&self, id: FaithId) -> &[ClergyRole];
    
    // ──── 谱系 ────
    
    /// 分裂来源——None = 从种子独立生成
    fn parent_faith(&self, id: FaithId) -> Option<FaithId>;
    
    /// 分裂产生的子信仰
    fn child_faiths(&self, id: FaithId) -> &[FaithId];
    
    /// 融合的父信仰（如为融合产物）
    fn fusion_parents(&self, id: FaithId) -> Option<[FaithId; 2]>;
    
    /// 是否已消亡
    fn is_extinct(&self, id: FaithId) -> bool;
    
    // ──── 建筑参数 ────
    
    /// 神圣建筑参数——供世界生成建筑系统（P5-P6）
    fn sacred_architecture(&self, id: FaithId) -> &SacredArchitectureParams;
    
    /// 该信仰的所有圣地坐标
    fn sacred_sites(&self, id: FaithId) -> &[GeographicEntityId];
    
    /// 某聚落所需宗教建筑空间面积——供世界生成聚落布局（P4）
    fn religious_space_demand(&self, settlement_id: SettlementId) -> HashMap<FaithId, f32>;
    
    // ──── 神职 ────
    
    /// 候选人是否可以成为某信仰的某神职
    fn can_become_clergy(&self, agent: &dyn FaithAgent, faith_id: FaithId, role: &ClergyRole) -> bool;
    
    // ──── 派生 ────
    
    /// 从实践画像派生虔诚度三维度
    fn derived_religiosity(&self, agent: &dyn FaithAgent) -> DerivedReligiosity;
    
    /// 从实践画像派生 UI 立场标签
    fn derived_stance(&self, agent: &dyn FaithAgent) -> DerivedStance;
    
    // ──── 辅助 ────
    
    /// 信仰对战斗士气的加成——供战斗系统 008 调用
    fn faith_morale_bonus(&self, agent: &dyn FaithAgent, context: FaithCombatContext) -> f32;
    
    /// 子女信仰继承——供生命系统 012 在 NPC 出生时调用
    fn inherit_faith_profile(&self, parents: &[(ReligiousPracticeProfile, f32)],
                              community: &[(FaithId, f32)], rng: &mut StdRng) -> ReligiousPracticeProfile;
    
    /// 是否有某信仰的活跃参与（participation > threshold）
    fn has_active_participation(&self, agent: &dyn FaithAgent, faith_id: FaithId, threshold: f32) -> bool;
}
```

**FaithCombatContext**——信仰系统定义的轻量枚举（仅包含信仰相关的战斗上下文）：

```rust
pub enum FaithCombatContext {
    DefendingFaith，       // 保卫信仰——被异教徒攻击
    HolyWar，              // 圣战——主动进攻异教徒
    DefendingTemple，      // 保卫神殿
    DefendingAgainstReligiousAggressor， // 抵御宗教侵略者
    Crusading，            // 十字军——远征异教领土
    Generic，              // 无信仰相关的普通战斗
}

/// InterFaithRelation——运行时两信仰之间的实际关系状态。
/// 与 FaithDefinition.inter_faith_attitude（教义静态立场）不同：
///   - InterFaithAttitude = 教义定义的立场（"我们应该如何看待异教徒"）
///   - InterFaithRelation = 运行时实际状态（受历史事件影响——两个曾经和平共处的宗教可能因一次圣战变成敌对）
pub enum InterFaithRelation {
    Allied，        // 同盟——积极合作
    Friendly，      // 友好——互相尊重
    Neutral，       // 中立——互不关心
    Tense，         // 紧张——存在未解决的冲突
    Hostile，       // 敌对——互相敌视
    AtWar，         // 战争——公开冲突（圣战）
}
```

## 二、FaithMut trait

```rust
/// pub(crate)——信仰修改的唯一入口。仅世界生成管线和历史模拟引擎可调用。
/// 消费模块不可调用。
pub(crate) trait FaithMut: FaithQuery {
    /// 创建新信仰——P2.5 种子生成或 P9 分裂/融合
    fn create_faith(&mut self, def: FaithDefinition) -> FaithId;
    
    /// 注册信徒 participation 变化——NPC 模块在更新 profile 后调用
    fn register_participation_change(&mut self, npc_id: NpcId, faith_id: FaithId, delta: f32);
    
    /// 触发大分裂
    fn trigger_schism(&mut self, parent: FaithId, child_theology: FaithTheology, 
                       year: i32) -> Option<FaithId>;
    
    /// 触发融合
    fn trigger_fusion(&mut self, parents: [FaithId; 2], year: i32) -> Option<FaithId>;
    
    /// 触发宗教改革
    fn trigger_reformation(&mut self, original: FaithId, reformed_theology: FaithTheology,
                            year: i32) -> Option<FaithId>;
    
    /// 标记消亡
    fn mark_extinct(&mut self, faith_id: FaithId, cause: ExtinctionCause, year: i32);
    
    /// 注册 Divine 授权事件——发出事件到权力系统
    fn register_divine_authorization(&mut self, event: DivineAuthorizationEvent);
    
    /// 更新 theology 代际漂移
    fn apply_theology_drift(&mut self, faith_id: FaithId, rng: &mut StdRng);
    
    /// 发出文化偏移事件（宗教改革→文化变化）
    fn emit_cultural_shift(&mut self, faith_id: FaithId, params: Vec<(CultureParam, f32)>);
}
```

## 三、SettlementFaithSnapshot

```rust
/// 信仰系统的惰性缓存——每游戏日增量更新。
/// 不存储"区域信仰地图"——从 NPC 数据聚合。
pub struct SettlementFaithSnapshot {
    pub settlement_id: SettlementId,
    /// 信仰分布——按信徒数降序 [(faith_id, active_believer_count), ...]
    pub distribution: Vec<(FaithId, u32)>,
    /// 主导信仰
    pub dominant: Option<FaithId>,
    /// 信仰多样性——Herfindahl-Hirschman 指数的补数
    /// 0 = 所有人信同一个教（单一）, 1 = 无数信仰均匀分布（极多元）
    pub diversity_index: f32,
    /// 快照计算的游戏时间戳
    pub computed_at: GameTimestamp,
    /// 本日是否有变化（未变化则跳过重算）
    pub dirty: bool,
}
```

**更新策略**：
- 标记 dirty = true：当且仅当聚落内有 NPC 的 participation 发生变化
- 每游戏日 batch：只重算 dirty settlement 的快照
- 99% 的游戏日：零计算（NPC participation 极少每日变化）

## 四、跨模块桥接契约表

| # | 概念 | 权威 Owner | 消费方 | 关键约定 |
|---|------|-----------|--------|---------|
| 1 | `FaithId` | **信仰系统 001** | 全部模块 | u32 扁平全局——统一历史系统的 `ReligionId` |
| 2 | `FaithDefinition` | **信仰系统 002** | 世界生成/历史/节日/权力 | 种子驱动生成，~500B/faith |
| 3 | `FaithTheology` | **信仰系统 002** | 文化 005（技术派生）、UI | 10 连续参数——所有特征从此派生 |
| 4 | `ReligiousPracticeProfile` | **信仰系统 003**（定义）→ NPC 模块（存储） | NPC/信仰系统 | ~40B/NPC，热路径本地读取 |
| 5 | `FaithQuery` trait | **信仰系统 010** | 全部模块 | 30 个只读方法，零分配，O(1) 缓存 |
| 6 | `FaithCalendarQuery` trait | **文化系统 008**（定义）→ **信仰系统 006**（实现） | 节日系统 | 对标 MentalAccess 模式——消费方定义，提供方实现 |
| 7 | `FaithMut` trait | **信仰系统 010**（pub(crate)） | 世界生成管线、历史模拟引擎 | 消费模块不可调用 |
| 8 | `FaithAgent` trait | **信仰系统 004**（定义）→ NPC/Player（实现） | 信仰系统 | 信仰系统不区分调用者是 NPC 还是玩家 |
| 9 | `FaithCombatContext` | **信仰系统 010**（定义） | 战斗系统 008（TeamMorale） | 信仰系统定义自己的上下文枚举——战斗系统在调用 faith_morale_bonus() 时传入 |
| 10 | `DivineAuthorizationEvent` | **信仰系统 007**（发出） | 权力系统 004（消费） | 事件驱动——信仰不操作 PowerTopology |
| 11 | `MagicReligionRelation` | **信仰系统 008** | 文化 005（技术派生） | 覆盖 `religiosity × 0.5` 假设 |
| 12 | `ReligiousReproductionNorms` | **信仰系统 002**（从 Life 012 迁移所有权） | Life 012（繁衍——消费） | 所有权从生命模块迁至信仰系统 |
| 13 | `DivineRelicTemplate` | **信仰系统 009** | 世界生成 P3 | 神遗物的生成条件与属性范围 |
| 14 | `SacredArchitectureParams` | **信仰系统 010** | 世界生成 P5-P6（建筑） | 神殿建筑参数——信仰→WorldGen |
| 15 | `FuneralCustoms` | **信仰系统 006** | 生命 DeathCustom、历史墓碑 | 葬仪类型/陪葬品/对亡灵态度 |
| 16 | `SettlementFaithSnapshot` | **信仰系统 010**（惰性缓存） | 全部空间查询 | 从 NPC 聚合——不存储独立空间地图 |

## 六、性能预算

### 6.1 热路径——NPC 决策引擎（每帧）

| 操作 | 路径 | 预算 |
|------|------|------|
| 读 participation 权重 | NPC 本地字段——零跨模块调用 | < 0.001µs |
| 读 spiritual_need | Physiology 本地字段 | < 0.001µs |
| faith_modifier 计算 | 本地浮点乘加 | < 0.01µs |
| prayer 触发判断 | 本地字段 + HolyDay 缓存查表 | < 0.1µs |
| **合计** | — | **< 0.001ms/帧** |

对标：天气系统 `WeatherQuery::sample()` 轮询 Rust CPU ≤ 0.25ms/帧。

### 6.2 温路径——每游戏日 batch

| 操作 | 预算 |
|------|------|
| SettlementFaithSnapshot 增量更新 | 仅 dirty settlements（< 1% 日常），每 settlement < 0.1ms |
| spiritual_need 增长 | 每个 NPC 一次浮点乘，batch < 0.5ms（数万 NPC） |
| 信仰传染检测 | batch NPC 社交对处理，< 1ms（仅处理当天有社交的 NPC 对） |
| 教际关系 hostility 更新 | 惰性——仅在涉及冲突检测时计算，< 0.1ms |
| **合计** | **< 2ms/游戏日**（摊销至 ~200K 帧/日 ≈ < 0.001ms/帧） |

### 6.3 冷路径——事件驱动 / 每游戏年

| 操作 | 预算 |
|------|------|
| FaithTheology 代际漂移 | 200 信仰 × 10 参数 × 正态随机采样——< 0.1ms/年 |
| 消亡检测 | 遍历所有活跃信仰——< 0.5ms/年 |
| 分裂/融合检测 | 条件复杂——仅在满足前置条件时触发（罕见），< 1ms |
| FaithCalendarQuery 缓存重建 | 每游戏日懒惰——< 0.5ms |

### 6.4 内存

| 项 | 预算 |
|------|------|
| 信仰系统持久存储 | ~260KB |
| NPC 侧信仰数据 | ~44B × NPC数（数万 → ~数MB）——由 NPC 模块承担 |
| SettlementFaithSnapshot 缓存 | ~200B × 聚落数（数百~数千 → ~数十KB-数百KB） |
| **总 VRAM 增量** | 0（全部在 RAM） |
