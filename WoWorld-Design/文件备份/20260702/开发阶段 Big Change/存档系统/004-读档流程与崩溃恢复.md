## 一、读档流程

### 1.1 五阶段总览

```
玩家选择存档 → 点击"加载"
    │
    ▼
┌─ Phase 1: 文件校验（<100ms）─────────────────────────────┐
│  1. 检查文件存在                                          │
│  2. 检查 magic 字节 (b"WOWSAVE\x00")                     │
│  3. 以只读打开 LMDB 环境                                  │
│  4. 读 meta/header → 反序列化 SaveHeader                  │
│  5. 校验 Header（字符串上限、版本号合理性）                 │
│                                                            │
│  任一失败 → 拒绝加载 + 明确错误提示                         │
│  全部通过 → 进入 Phase 2                                   │
└────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Phase 2: 版本检查与预检（<50ms）────────────────────────┐
│                                                            │
│  preflight_check():                                        │
│    1. 读 Header.global_version → 可识别？                  │
│    2. 检查致命模块的 named_db 存在且非空                    │
│       (terrain, npc, player)                               │
│    3. 检查每个 module_version 合理（>0 且 <1000）           │
│    4. 比对 module_versions 和当前模块版本 → 是否需要迁移    │
│                                                            │
│  预检失败 → 拒绝加载。不覆盖任何内存数据                    │
│  预检通过 → 决定是否需要 Phase 3 迁移                       │
└────────────────────────────────────────────────────────────┘
    │
    ├─ 不需要迁移 → 直接进入 Phase 4 ──────────────────────┐
    │                                                        │
    └─ 需要迁移 → 进入 Phase 3 ─────────────────────────────┐
                                                              │
┌─ Phase 3: 版本迁移（如需要，<3s）────────────────────────┐
│                                                            │
│  1. 创建备份: save.woworld → save.woworld.bak              │
│  2. 关闭只读 LMDB env                                      │
│  3. 以读写打开 LMDB env                                    │
│  4. 创建写事务                                             │
│  5. for each module where 存档版本 < 当前版本:              │
│       module.migrate(from_version, &txn)?;                 │
│       // 迁移是链式的: while current < self.version() ... │
│       // from_version 到 current_version 差可以 >1         │
│  6. for each migrated module:                              │
│       module.validate_after_migration(&txn)?;              │
│       // 致命模块验证失败 → 事务回滚                       │
│  7. 更新 meta/header 的 module_versions 为迁移后的版本      │
│  8. 提交事务                                               │
│  9. 关闭读写 LMDB env                                      │
│  10. 以只读打开 LMDB env                                   │
│                                                            │
│  迁移失败 → 事务回滚 → .bak 可用 → 拒绝加载                │
│  迁移成功 → 进入 Phase 4                                    │
└────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Phase 4: 加载模块数据（<5s）────────────────────────────┐
│                                                            │
│  let load_ctx = LoadContext::new(&txn, txn_factory);      │
│                                                            │
│  // 所有模块按任意顺序加载——跨模块引用是运行时懒解析        │
│  for module in modules.iter_mut() {                        │
│      match module.load(&load_ctx) {                       │
│          Ok(()) => continue,                               │
│          Err(e) if module.is_critical() => {               │
│              // 致命模块加载失败 → 终止加载 → 返回主菜单    │
│              return Err(LoadError::CriticalModuleFailed(   │
│                  module.module_name(), e));                │
│          }                                                │
│          Err(e) => {                                       │
│              // 非致命模块加载失败 → 重置 + 警告 + 继续     │
│              module.reset_to_default();                    │
│              warnings.push(LoadWarning::ModuleLoadFailed { │
│                  module: module.module_name(),             │
│                  reason: e.to_string(),                    │
│              });                                           │
│          }                                                │
│      }                                                     │
│  }                                                          │
│                                                            │
│  // LoadContext 提供 txn_factory——模块可创建独立读事务     │
│  // 做渐进/按需加载（如物品系统按 Chunk 加载）              │
│  // Schema 版本转换在 load() 中批量完成（加载画面预算内）    │
│                                                            │
│  加载完成 → 进入 Phase 5                                    │
└────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Phase 5: 运行时初始化（<1s）─────────────────────────────┐
│                                                            │
│  1. 重建运行时缓存（PerceptualCache 空、GOAP 空—涌现行为）  │
│  2. 初始化渲染（Godot 加载玩家所在 Chunk）                  │
│  3. 创建 session.lock                                      │
│  4. SaveSystem 保持只读 LMDB 句柄（渐进加载用）             │
│  5. 启动模拟 tick                                          │
│  6. 玩家看到世界 → 可操作                                   │
└────────────────────────────────────────────────────────────┘
```

### 1.2 为什么不需要两阶段加载

最初设计过"基础层先加载 → 上层后加载"的分阶段顺序。但经过分析：
- 模块存储的跨模块引用全是 ID（整数）。NPC 存 `faith_id: 17`，不需要在加载时验证 FaithId 是否存在
- 跨模块引用在**运行时**懒解析：NPC 调 `faith_module.get_profile(17)` → 如果悬空引用 → 用默认 Profile
- `load()` 期间模块不需要查询其他模块的数据 → 可以按任意顺序加载

**单循环加载，无需拓扑排序，无需 `load_after()`。**

### 1.3 Preflight check

```rust
fn preflight_check(env: &LmdbEnvironment) -> Result<()> {
    let txn = env.begin_ro_txn()?;

    // 1. Header 可读且合法
    let header = read_header(&txn)?;
    validate_header(&header)?;

    // 2. 致命模块的 named_db 存在且非空
    for critical_db in &["terrain", "npc", "player"] {
        let db = match txn.open_db(Some(critical_db)) {
            Ok(db) => db,
            Err(_) => return Err(LoadError::MissingCriticalDb(critical_db)),
        };
        let mut cursor = txn.open_ro_cursor(db)?;
        if cursor.first().is_err() {
            return Err(LoadError::EmptyCriticalDb(critical_db));
        }
    }

    // 3. module_versions 各条目合理性
    for (name, ver) in &header.module_versions {
        if *ver == 0 || *ver > 1000 {
            return Err(LoadError::SuspiciousVersion(name.clone(), *ver));
        }
    }

    Ok(())
}
```

**预检在覆盖任何内存数据前执行。** 预检失败 → 不加载任何模块 → 旧游戏状态保留。

### 1.4 LoadResult

```rust
struct LoadResult {
    header: SaveHeader,
    warnings: Vec<LoadWarning>,
    migration_applied: bool,
    modules_migrated: Vec<String>,   // 哪些模块被迁移了
}

enum LoadWarning {
    /// 跨模块引用悬空（已用 fallback 填充）
    DanglingReference { entity: String, ref_id: u64, module: String },
    /// 部分非致命模块加载失败（跳过）
    ModuleLoadFailed { module: String, reason: String },
    /// Mod 缺失——数据已冻结
    ModMissing { mod_id: String, affected_entities: u32 },
}
```

UI 在加载后显示警告列表。

## 二、崩溃恢复

### 2.1 三件套总览

| 机制 | 防什么 | 原理 |
|------|--------|------|
| **临时文件 + 原子重命名** | 写入中途崩溃损坏存档 | 写 `.tmp` → 成功 → rename。rename 是原子的——要么旧在，要么新在 |
| **覆盖前自动备份** | 误覆盖后想回退 | 覆盖前旧文件 → `.bak`。玩家可手动恢复 |
| **session.lock** | 非正常退出后加载到崩溃前一刻 | 运行时存在，正常退出删除。残留 = 上次非正常退出 |

### 2.2 临时文件 + 原子重命名

详见 [[002-存档格式与文件架构#六、原子写入协议|002 §六 原子写入协议]]。

补充：rename 后做写入后健全性验证（读回 Header 检查 magic + global_version + 致命 named_db 非空）。验证失败 → 旧存档保留，损坏文件移至 `.corrupted`。

### 2.3 覆盖前自动备份

**所有覆盖操作均自动备份**——不仅 Auto 存档，Quick、Exit、Manual 覆盖同文件名时均适用：

```
覆盖 <存档名>.woworld:
  1. <存档名>.woworld 存在 → rename → <存档名>.woworld.bak
  2. 写新 <存档名>.woworld (临时文件 → rename)

如果新存档损坏:
  玩家在文件管理器里: 删 .woworld → 把 .bak 重命名回 .woworld → 恢复

备份深度 = 1（只保留上一个版本）。第二次覆盖同槽时 .bak 被替换。
玩家想保留更多版本应用手动存档。
```

### 2.4 session.lock

#### 创建与删除

```rust
// 游戏进入世界时
fn create_session_lock(world_dir: &Path) -> Result<()> {
    let lock_path = world_dir.join("session.lock");
    let content = chrono::Utc::now().to_rfc3339();
    std::fs::write(&lock_path, content)?;
    Ok(())
}

// 正常退出时
fn remove_session_lock(world_dir: &Path) {
    let lock_path = world_dir.join("session.lock");
    let _ = std::fs::remove_file(&lock_path);
    // 忽略删除失败——可能文件已不存在
}
```

#### 检测

```rust
fn check_session_lock(world_dir: &Path) -> Option<SessionLockStatus> {
    let lock_path = world_dir.join("session.lock");
    if !lock_path.exists() {
        return None;  // 正常退出，无残留
    }

    let content = std::fs::read_to_string(&lock_path).ok()?;
    let lock_time = chrono::DateTime::parse_from_rfc3339(&content).ok()?;
    let now = chrono::Utc::now();
    let age = now - lock_time;

    if age > chrono::Duration::hours(24) {
        return Some(SessionLockStatus::Stale { age_hours: age.num_hours() });
        // 超 24 小时 → 自动忽略（残留）
    }

    Some(SessionLockStatus::Active {
        locked_at: lock_time,
        age_minutes: age.num_minutes(),
    })
}
```

### 2.5 .tmp 残留清理

启动时：

```rust
fn cleanup_tmp_files(saves_root: &Path) -> Result<()> {
    for entry in walkdir::WalkDir::new(saves_root)
        .into_iter()
        .filter_map(|e| e.ok())
        .filter(|e| e.path().extension().map_or(false, |ext| ext == "tmp"))
    {
        let _ = std::fs::remove_file(entry.path());
        log::info!("清理残留临时文件: {:?}", entry.path());
    }
    Ok(())
}
```

## 三、运行时渐进加载

### 3.1 为什么需要

大模块（物品系统 ~1.7GB 数据）不能全量反序列化到内存（Rust 预算 1.4GB）。需要按需从 LMDB 读取。

### 3.2 MVCC 下的实现

游戏运行时 LMDB 保持只读打开。模块通过 SaveSystem 获取读事务：

```rust
impl SaveSystem {
    fn create_read_txn(&self) -> Result<RoTransaction> {
        let env = self.env.read().unwrap();
        let txn = env.as_ref()
            .ok_or(SaveError::EnvNotOpen)?
            .begin_ro_txn()?;
        Ok(txn)
    }
}
```

```rust
// 物品模块按需加载 Chunk 的物品数据
impl ItemModule {
    fn load_items_in_chunk(&self, save_sys: &SaveSystem, chunk: ChunkCoord) -> Result<Vec<ItemState>> {
        let txn = save_sys.create_read_txn()?;
        let db = txn.open_db(Some("items"))?;
        let prefix = format!("items/chunk/{:04}/{:04}", chunk.x, chunk.z);
        // 范围扫描: 读此 Chunk 的所有物品实例
        let items = scan_by_prefix(&txn, &db, &prefix)?;
        Ok(items)
    }
}
```

**MVCC 保证**：存档时写临时文件 → rename → 刷新读句柄（关闭旧 inode → 打开新文件）。运行中的读事务不受影响（OS 保留旧 inode 直到所有引用释放）。新创建的读事务看到最新存档数据。详见 [[002-存档格式与文件架构#六、原子写入协议|002 §六·6.3]]。

## 四、错误场景处理

### 4.1 全面故障矩阵

| 场景 | 检测 | 处理 |
|------|------|------|
| 存档文件不存在 | Phase 1 | 提示"存档文件不存在" |
| magic 不匹配 | Phase 1 | 提示"不是 WoWorld 存档文件" |
| Header 反序列化失败 | Phase 1 | 提示"存档文件可能已损坏" |
| 致命 named_db 缺失 | Phase 2 | 提示"存档缺少关键数据: terrain/npc/player" |
| 致命 named_db 为空 | Phase 2 | 同上 |
| module_version 可疑 | Phase 2 | 提示"版本信息异常: 模块 X 声明版本 0" |
| 全局 version 过高 | Phase 2 | 提示"存档由更新版本的游戏创建。请升级" |
| 迁移失败 | Phase 3 | 事务回滚 → 提示"版本迁移失败。备份已保存" → .bak 完好 |
| 非致命模块 load 失败 | Phase 4 | 跳过该模块 → 添加警告 → 继续加载 |
| 致命模块 load 失败 | Phase 4 | 终止加载 → 返回主菜单 |
| 磁盘满 | Phase 3/5 | 提示"磁盘空间不足" |
| 游戏加载中点击另一个存档 | 游戏内 | 先自动保存当前进度 → 再加载目标存档 |

### 4.2 部分降级加载

如果非致命模块的 named_db 损坏或缺失 → 不阻止加载：

- 信仰模块损坏 → 调用 `reset_to_default()` → 所有 NPC 用默认信仰 Profile（种子驱动）
- 文化模块损坏 → `reset_to_default()` → 用默认文化参数
- 技能模块损坏 → `reset_to_default()` → SkillEntry 全为 0（用进废退从零开始）
- 历史模块损坏 → `reset_to_default()` → 历史事件的运行时查询返回空（"世界失去了历史记录"）

**`reset_to_default()` 必须在 `load()` 失败后由 SaveSystem 调用**——确保模块回到干净初始状态，不会有半加载的脏数据残留。

核心规则：**致命模块（terrain/npc/player）必须全有全好。其他可降级。**

## 五、游戏中加载存档

玩家正在游戏中 → ESC → 选择"加载存档"。当前未保存的进度：

**自动保存后加载。** 不询问、不弹窗——直接先触发一次 Exit 存档，保存当前进度，然后加载目标存档。如果玩家不想保留这个自动存档——之后可以删掉。

这比"丢弃未保存进度"更友好，也比"弹窗询问"更流畅（减少一个操作步骤）。
