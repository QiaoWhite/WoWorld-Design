## 一、设计原则——薄层调度

施工调度系统只负责**任务编排**，不负责材料获取、不负责人员招募、不负责经济交易。它产出 `MaterialRequirementList`（声明式），施工负责人（NPC/玩家/公会）自行规划如何获取材料。

```rust
ConstructionTask 声明材料需求（"需要 12 Oak log + 4 Granite block"）
        │
        ▼
施工负责人的 EconomicBehavior 决定如何获取
   ┌────┼────┬──────────┐
   ▼    ▼    ▼          ▼
 市场  自采  库存取出   以物易物
```

**建筑模块不碰 Market trait。** 这是核心解耦决策。

## 二、ConstructionProject

```rust
/// 一个施工项目——建造或修改一栋建筑
pub struct ConstructionProject {
    pub id: ConstructionProjectId,
    pub target_building: Option<BuildingId>,  // None = 新建
    pub site: Option<BuildingSite>,           // 新建时提供
    pub operations: Vec<AddRemoveOp>,         // 要执行的组件操作序列
    pub phases: Vec<ConstructionPhase>,       // 按阶段编排
    pub material_requirements: MaterialRequirementList,
    pub workers: Vec<EntityId>,
    pub current_phase: usize,
    pub progress: f32,                        // 当前阶段 0-1
    pub state: ConstructionState,
    pub quality_modifier: f32,               // 施工质量影响——来自工人技能
}

pub enum ConstructionState {
    Active,
    Paused { reason: String },
    Cancelled,    // 产生半成品建筑
    Done,
}

/// 单个组件的操作
pub enum AddRemoveOp {
    Add(ComponentInstance, Vec<ConnectionEdge>),
    Remove(ComponentInstanceId),
    Replace(ComponentInstanceId, ComponentInstance),
    Modify(ComponentInstanceId, Box<dyn ComponentParams>),
}
```

### 2.1 阶段划分——逐层非全楼

```rust
struct ConstructionPhase {
    pub floor: i8,               // 此阶段操作的楼层
    pub phase_type: PhaseType,
    pub component_indices: Vec<usize>,  // 此阶段涉及的操作索引
    pub estimated_duration: GameDuration,
}

enum PhaseType {
    Foundation,     // 地基
    Frame,          // 框架（柱+梁+楼板）
    Enclosure,      // 围护（外墙+屋顶+门窗）
    Interior,       // 内部（内墙+楼梯+装饰）
}
```

真实建筑的施工顺序：先搭下层框架→下层围护→上层框架→上层围护→屋顶→从下层开始内部装修。Phase 划分从 Building 的组件图自动推导——最底层的组件最先施工，最高的屋顶最后。

```rust
fn auto_phase(project: &ConstructionProject) -> Vec<ConstructionPhase> {
    // 1. 按 floor 分组组件
    // 2. 每层内: Foundation(仅 floor 0) → Frame → Enclosure → Interior
    // 3. 层间交替: F0 Frame → F0 Enclosure → F1 Frame → F1 Enclosure → ...
    // 4. Roof 在所有围护完成后
    // 5. Interior 全部在最后（先住进去一层再装修上层）
}
```

## 三、ConstructionScheduler

```rust
pub struct ConstructionScheduler {
    active_projects: Vec<ConstructionProject>,
    /// 全局施工速度倍率（受季节/天气影响）
    global_speed: f32,
}

impl ConstructionScheduler {
    /// 每游戏 tick 调用一次
    pub fn tick(&mut self, delta: GameTime,
                modifiers: &[&dyn ConstructionModifier]) {
        let total_budget = Duration::from_millis(0.2);  // ★ 技术栈预算: ≤0.2ms

        for project in &mut self.active_projects {
            if project.state != ConstructionState::Active { continue; }

            let phase = &mut project.phases[project.current_phase];
            let speed = self.effective_speed(project, modifiers);
            project.progress += speed * delta.as_secs_f32() / phase.estimated_duration.as_secs_f32();

            if project.progress >= 1.0 {
                project.progress = 0.0;
                self.complete_phase(project);
            }

            if total_budget.elapsed() > total_budget { break; }
        }
    }

    fn effective_speed(&self, project: &ConstructionProject,
                       modifiers: &[&dyn ConstructionModifier]) -> f32 {
        let mut speed = self.global_speed;
        for m in modifiers {
            speed = m.modify_speed(speed);
        }
        // 工人数量 + 技能等级 → 加速
        speed *= (1.0 + (project.workers.len() as f32 - 1.0) * 0.3);
        speed
    }

    fn complete_phase(&mut self, project: &mut ConstructionProject) {
        // 1. 将此阶段的所有 AddRemoveOp 写入 Building
        // 2. 更新 Building 的空间索引
        // 3. 如果这不是最后阶段 → project.current_phase += 1
        // 4. 如果是最后阶段 → project.state = Done
        // 5. 触发 BuildingHistory 记录
        // 6. 如果中间阶段完成 → 建筑外观切换（渲染层监听 Building 变更）
    }
}
```

### 3.1 施工取消与半成品

玩家/NPC 取消施工 → `project.state = Cancelled`。已完成的组件操作保留在 Building 中。半成品建筑变成一种可见的世界状态——可以被后续继续建完、废弃风化、或被 NPC 占据改造。

### 3.2 新建与拆除

**从零建造**（NPC 新建建筑）：
```
NPC CONSTRUCT 原子 → ConstructionScheduler::new_building(tag, site, ctx, seed)
  → GeneratorRegistry.select(tag).generate(site, ctx, seed)
  → 产出 (Building, Vec<TerrainModificationRequest>)
  → building_to_ops(Building) // 拓扑排序: 地基→墙→梁→顶
  → ConstructionProject { ops, phases, ... }
```

**拆除**（整栋建筑逆序移除）：
```rust
fn demolish(&self, building_id: BuildingId) -> ConstructionProject {
    // 逆序拆: 屋顶→梁→墙→地基，从顶层到底层
    let mut ops = self.building_ops_reversed(building_id);
    ConstructionProject {
        target_building: Some(building_id),
        operations: ops,
        phases: reverse_auto_phase(&building),
        material_requirements: MaterialRequirementList::empty(), // 拆除不消耗材料，产出材料
        ...
    }
}
```
拆除的材料回收由 NPC EconomicBehavior 处理——`Remove` op 执行后材料自动进入拆除者的库存。

### 3.4 天气对施工速度的影响

`ConstructionScheduler` 不知道天气。调用方（NPC 行动系统）在调用 `tick()` 之前查询天气，直接将 `global_speed` 设为天气调整后的值。建筑模块零天气感知。

| 条件 | 户外阶段倍率 | 室内阶段倍率 | 说明 |
|------|------------|------------|------|
| 晴天 | ×1.0 | ×1.0 | |
| 小雨 | ×0.7 | ×0.9 | |
| 暴雨 | ×0.0 | ×0.7 | 户外停工 |
| 暴雪 | ×0.0 | ×0.5 | 材料运输中断 |
| 极寒 (< -20°C) | ×0.2 | ×0.5 | 材料冻结 |
| 大风 (> 8级) | ×0.0 (高处) | ×0.8 | 高空作业禁止 |

### 3.5 施工品质公式

NPC 技能等级 → 组件健康度初始值：

```rust
fn initial_component_health(
    skill_level: f32,       // 0-100
    material_quality: f32,  // 0-1
    quality_modifier: f32,  // 魔法等加成
) -> f32 {
    let base = 0.6;
    let skill_bonus = 0.3 * (skill_level / 100.0).min(1.0);
    let material_bonus = 0.1 * material_quality;
    (base + skill_bonus + material_bonus) * quality_modifier
}
// skill 0:   0.6
// skill 50:  0.75
// skill 100 + 好材料: 1.0
```

## 四、ConstructionModifier trait

魔法辅助建造的预留接口。建筑模块定义 trait，[[魔法]] 模块实现。

```rust
/// 建造过程中的魔法/特殊效果——由 [[魔法]] 模块实现
pub trait ConstructionModifier: Send + Sync {
    /// 识别此 modifier
    fn name(&self) -> &str;

    /// 修改单个任务的效果（可能跳过步骤、瞬间完成组件等）
    fn modify_task(&self, task: &mut AddRemoveOp) -> ModificationEffect;

    /// 修改施工速度的倍率
    fn modify_speed(&self, base_speed: f32) -> f32;

    /// 此 modifier 是否仍有效（魔法有时限）
    fn is_active(&self, current_time: GameTime) -> bool;
}

pub enum ModificationEffect {
    None,
    SkipTask,                   // 瞬间完成——不用工人动手
    ReduceMaterial { by: f32 }, // 材料消耗减少
    ImproveQuality { bonus: f32 }, // 施工品质提升
    MagicalReinforcement,       // 魔法加固——结构完整性 +0.1
}
```

**关键设计**：modifier 从外部传入（`tick()` 的参数），ConstructionScheduler 不持有 modifier 实例。魔法系统负责提供当前生效的 modifier 列表。

## 五、MaterialRequirementList — 只声明不获取

```rust
/// 施工材料需求——声明式，不指定获取方式
pub struct MaterialRequirementList {
    items: Vec<(MaterialRequirement, RequirementStrictness)>,
}

pub struct MaterialRequirement {
    pub material_class: MaterialClass,
    pub quantity: u32,
    pub unit: MaterialUnit,
}

pub enum RequirementStrictness {
    /// 硬性——必须此材质。如果市场没有 → 施工阻塞等待材料贸易
    Hard,
    /// 偏好——尽量此材质，没有时可用替代。替代容差 0.5
    Preferred { fallback_tolerance: f32 },
    /// 宽松——什么材质都行
    Flexible,
}

pub enum MaterialUnit {
    Log(u32),        // 原木根数
    Block(u32),      // 石块/砖块
    Plank(u32),      // 木板
    Sheet(f32),      // 板材面积 m²
    Volume(f32),     // 体积 m³
}
```

**施工负责人如何处理 MaterialRequirementList**：

```rust
// NPC 的 EconomicBehavior::fulfill_construction_needs()
//   1. 查自己的库存（Items 模块）
//   2. 查附近市场（Economy 模块 Market trait）
//   3. 自己采集（CONSTRUCT 原子的 HARVEST/MINE/FELL 子操作）
//   4. 找朋友借/交换（NPC 社交）
//   5. 如果以上都不够 → 施工延后，等待材料可用或贸易到达
```

建筑模块完全不参与以上任何一步。

## 六、NPC 放置宽容化

NPC 放置组件时不需要精确坐标寻路——ATTACH 操作有 0.25m（子网格）的 snap 容差：

```rust
fn execute_place(op: &AddRemoveOp, building: &mut Building) -> Result<(), PlaceError> {
    let mut component = op.component.clone();

    // snap 容差——位置微调到最近的合法位置
    let snapped_pos = snap_to_grid(component.hot.spatial.position, 0.25);
    component.hot.spatial.position = snapped_pos;

    // 验证放置不会使此组件与其他组件碰撞过深
    if let Some(penetration) = detect_penetration(&component, &building.components) {
        if penetration.depth < 0.1 {
            // 微调——容忍 < 0.1m 的穿透
            component.hot.spatial.position += penetration.push_out;
        }
        // 较大穿透 → 标记为放置异常但允许——涌现特征（"家具嵌墙"）
    }

    building.components.push(component);
    building.spatial_index.rebuild(&building.components);
    Ok(())
}
```

## 七、多人并发施工

两个 NPC 同时在同一区域施工 → 检测冲突：

```rust
impl ConstructionScheduler {
    fn would_conflict(
        a: &ConstructionProject,
        b: &ConstructionProject,
    ) -> bool {
        // 简单 AABB 重叠检测
        if let (Some(site_a), Some(site_b)) = (&a.site, &b.site) {
            if site_a.boundary.overlaps(&site_b.boundary) {
                // 检查是否修改相同的组件
                let ids_a: HashSet<_> = a.affected_component_ids().collect();
                let ids_b: HashSet<_> = b.affected_component_ids().collect();
                return !ids_a.is_disjoint(&ids_b);
            }
        }
        false
    }
}
```

冲突时 → 后来的项目被暂停，等待第一个完成。冲突检测 O(N)——N = 活动项目数（一般 < 10）。

## 八、Blueprint 施工与 WFC 施工的路径统一

| 路径 | 项目来源 | 操作序列 | 材料需求 |
|------|---------|---------|---------|
| 世界生成 WFC | `WfcBuildingSolver` 输出 | 直接写入 Building（跳过施工调度——世界初始态） | 无（世界初始） |
| 玩家 Blueprint | `BlueprintValidator` → `AddRemoveOp` 序列 | 走完整的 ConstructionScheduler | `MaterialRequirementList` 从 Blueprint 派生 |
| NPC 翻修 | `LocalIncrementalSolver` → `AddRemoveOp` 序列 | 走完整的 ConstructionScheduler | 从修改派生 |
| NPC 新建 | `WfcBuildingSolver`（运行时调用） → `AddRemoveOp` 序列 | 走完整的 ConstructionScheduler | 从 WFC 输出派生 |

世界生成时的建筑是唯一跳过 ConstructionScheduler 的路径——批量生成一次性完成，不需要逐 tick 模拟施工。
