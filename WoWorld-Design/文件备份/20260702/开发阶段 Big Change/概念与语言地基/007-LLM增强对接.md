## 一、LLM 在概念层操作——不在文本层

```
确定性路径（默认）:
  事件 → classify_pattern → Utterance { concepts } → TextGenerator → 文本 → TTS

LLM 增强路径（可选，19 场景开关）:
  事件 → classify_pattern → Utterance { concepts }
       → LLM 修改/丰富 concepts → Utterance { modified_concepts }
       → TextGenerator → 文本 → TTS
```

**关键原则**: LLM 永远不直接产出字符串。LLM 接收结构化概念列表，返回修改后的概念列表。TextGenerator 负责所有文本渲染——无论概念来自确定性管道还是 LLM。

## 二、安全边界

```
LLM 输入: Utterance { concepts: [...] } + NPC persona prompt
LLM 输出: Vec<UtteranceConcept> (JSON Schema 约束)

验证（在集成层）:
  ① 概念必须存在于说话者的有效概念空间中 → 不通过则回退
  ② 不该新增概念（除非 NPC 尚未掌握的）→ 严肃对待
  ③ 所有引用实体必须存在（EntityValidator）
  ④ 不违反 NPC 核心人格（PersonalityValidator）
  ⑤ 不当内容拦截（ContentFilter）
  ⑥ 全部 LLM 输出记录日志

失败 → 无缝回退到确定性概念 → TextGenerator
```

## 三、LLM 增强的概念管道

```rust
impl LlmOrchestrator {
    pub fn enhance_utterance(
        &self,
        utterance: &Utterance,
        npc: &NpcSnapshot,
        ctx: &DialogueContext,
    ) -> Utterance {
        if !self.is_active() { return utterance.clone(); }

        let system_prompt = PersonalityPromptBuilder::build(npc);
        let user_prompt = format_concept_list(&utterance.concepts);

        // 如果听者语言水平低 → prompt 中加入简化指令
        if ctx.listener_proficiency < 0.5 {
            user_prompt += &format!(
                "\nListener proficiency: {:.1}. Use simple concepts, avoid complex abstractions.",
                ctx.listener_proficiency
            );
        }

        match self.registry.call(&LlmRequest {
            system_prompt,
            user_prompt,
            response_format: ResponseFormat::Json(CONCEPT_SCHEMA),
            scene_tag: "dialogue",
            ..default()
        }) {
            Ok(response) => {
                let enriched = parse_concepts(&response.text);
                if validate_concepts(&enriched, npc.effective_concept_space()) {
                    Utterance { concepts: enriched, ..utterance.clone() }
                } else {
                    utterance.clone()  // 验证失败 → 回退
                }
            }
            Err(_) => utterance.clone(),  // LLM 不可用 → 回退
        }
    }
}
```

## 四、三种语言产出的 LLM 策略

| 产出类型 | LLM 操作 | 确定性回退 |
|---------|---------|-----------|
| 对话 | 丰富概念列表（更自然的语序、更丰富的修饰语） | 原始概念 → 模板渲染 |
| 书籍/铭文 | 扩展概念 + 风格注入 | 原始概念 → 模板文本 |
| 内心独白 | NPC 个性调制的概念重排 | 原始概念 → 简化模板 |

**全部三种情况的回退**: 原始 Utterance → TextGenerator。玩家看到的字符串永远一致的质量——只是 LLM 增强的版本可能更丰富、更自然。

## 五、LLM 增强与语音的关系

```
LLM 操作概念 → Utterance { enriched_concepts }
  → TextGenerator::render_utterance() → String
  → VoicePacket { text, ... }
  → TtsEngine::speak()
  → 音频输出

音频层不感知 LLM 的存在。
VoicePacket 中的 text 字段来自 TextGenerator——
TextGenerator 不知道也不关心这段文本是确定性概念还是 LLM 增强概念渲染的。
管道完全透明。
```
