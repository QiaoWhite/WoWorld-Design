# Task Plan: WoWorld 设计文档工作

## Goal
管理和推进 WoWorld 游戏的设计文档——在 `开发阶段/` 中产出正式开发规格，逐步将 `想法/` 中的概念设计成熟化、正式化。

## 仓库结构理解
- **`开发阶段/`** — 正式开发规格。经过讨论确定的、预备投入开发的设计。权威。
  - 技术栈方案/ · NPC活人感模块/ · 战斗/ · 魔法/ · 世界生成/ · 生命/ · 历史/
- **`想法/`** — 概念设计、脑暴、早期草稿。允许矛盾、模糊、未决。
- **`参考文档/`** — 参考性设计文档，编号归档。
- **`Change/`** — 设计变更追踪（CHG-XXX 格式）。

## Decisions Made
| Decision | Rationale |
|----------|-----------|
| 开发阶段 = 权威规格 | `想法/` 是草稿，`开发阶段/` 是正式结果 |
| 技术决策以 技术栈方案 v3.0 为准 | 所有模块技术方案的最高依据 |
| 变更用 CHG 文档追踪 | 结构化修改需要变更记录 + 交叉引用 |
| 文档深度 ~250-300 行/篇 | 匹配世界生成标准——全覆盖不缩水 |
| 植物不继承完整 Life | 极简 LifeBase——无 BodyState/Mind |
| 动物不拆水/陆/空文档 | 共享 12 维+InstinctiveMind——一份文档三个域 |
| 跨域 = 能力标签 | 主域(PrimaryDomain)+can_swim/can_fly——不是独立分类 |
| 历史系统 = 独立模块 | event_log + 因果链 + 痕迹 + 书籍 + 印记 + 大日志 |
| 口头传说归信息传播系统 | 历史模块只提供 PropagableEventSource 接口 |
| 大日志纯功能性 | 不升级、无温度叙事、?表示未知、纠错机制、双向链接 |
| **NPC 必须通过感官感知世界** | NPC 不能"查询数据库"——只能通过模拟五感（视听嗅触+内感觉）获取有限的、可能有误差的信息。信息不完整→决策不确定→行为涌现 |
| **先设计后开发** | Phase 13-19 全部为纯设计文档工作——在理念走通前不写代码 |
