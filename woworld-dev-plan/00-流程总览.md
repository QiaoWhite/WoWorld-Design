## 三、AI 标准作业流程（9 场景 SOP）

### 决策树：AI 启动时判断「我在哪个场景？」

```
新会话启动
    │
    ├─ 没有 Handoff 文件 或 上次是完整冲刺结束
    │   └─→ 场景 A「新会话启动」→ 场景 B「冲刺提案」
    │
    ├─ 有 Handoff 文件，且标记冲刺未完成
    │   └─→ 场景 D「中断恢复」
    │
    ├─ 用户带着反馈/纠偏出现
    │   └─→ 场景 F「用户纠偏」
    │
    └─ 用户明确说"阶段 N 完成了，进下一阶段"
        └─→ 场景 H「阶段过渡」
```

---

### 场景 A：新会话启动

**触发**: 新 Claude 实例启动。不知道当前状态。

**读取序列**（必须按顺序）:
1. `woworld-dev-plan/README.md` — 30 秒了解全局
2. 本文件（`00-流程总览.md`）— 知道六阶段 + 当前所处位置
3. `01-核心基础/README.md`（或当前阶段 README）— 知道阶段目标、出口标准
4. `01-核心基础/handoff/` 最新交接文档 — 知道精确 WIP 状态
5. `附录E-开发状态.md` — 知道模块完成度全景
6. `../开发文档/README.md`（如涉及 ECS 架构）— 了解 Component/System/Resource 全景

**执行**:
1. 运行 `cargo check --workspace`（验证代码编译状态）
2. 对比 Handoff 声称的状态 vs 实际代码（检查漂移）
3. 对比 Handoff vs 附录E（检查不一致）

**输出**:
```
## 当前状态摘要
- 阶段: Phase 1 — 核心基础
- 当前冲刺: [Sprint-NNN] / [无进行中冲刺]
- 已完成: [里程碑列表]
- 进行中: [里程碑 + 精确进度]
- 代码编译: ✅ / ❌
- 漂移检测: [无漂移 / 发现以下不一致...]
- 下一步建议: [提案下一冲刺 / 继续当前冲刺 / 修复漂移]
```

### 场景 B：冲刺提案

**触发**: 无进行中冲刺。需要提出下一个冲刺。

**前置**: 场景 A 已执行。

**读取序列**:
1. `附录D-模块依赖图.md` — 知道依赖约束
2. `附录E-开发状态.md` — 知道模块完成度
3. 当前阶段 `README.md` §4 子里程碑清单 — 知道哪些未完成
4. 最近 3 个 `devlogs/` 文件 — 知道最近的坑
5. `附录C-架构决策记录.md` — 知道历史决策约束
6. `git diff` 对 `WoWorld-Design/Happy Game/开发阶段/` 和 `开发文档/`（自上次冲刺以来）→ 有变更 → **触发场景 G**（三问法 + 四级别响应）→ 影响报告写入冲刺提案 `📋 依赖前提检查` 节

**输出**: 冲刺提案（写入 `sprint-proposals/sprint-NNN-描述-YYYYMMDD.md`）

**提案模板**（详见 [`附录F-冲刺模板.md`](附录F-冲刺模板.md)）:
```markdown
# Sprint-NNN: [简短标题]

## 📋 依赖前提检查
[列出本冲刺依赖的已完成项]

## 🎯 目标（≤3 个）
### 目标 1: [名称]
- 验收标准: [可客观判断的标准]
- 涉及模块: [模块名 + 设计文档路径]

### 目标 2: ...

## 🧪 研究事项
[本冲刺涉及的 🟡🟠🔴 级新问题 + 研究计划]

## 📊 决策矩阵（5 维加权）
| 候选 | 依赖链(×3) | 风险(×3) | 用户价值(×2) | 设计完备(×1) | 架构健康(×2) | 总分 |
|------|-----------|---------|-------------|------------|------------|------|
| ...  |           |         |             |            |            |      |

## ⚠️ 需用户裁决的事项
[设计文档矛盾？架构选型？性能取舍？]

## 📏 预估影响
- 文件数: ~N
- 冲刺数: 1
- 风险: [低/中/高]
```

**然后**: 提交用户审批。未审批不执行。

### 场景 C：冲刺执行

**触发**: 冲刺提案已获用户审批。

**前置**: 场景 A + B 已完成。

**执行循环**（冲刺内每个目标）:

```
对每个目标:
│
├─ 1. 识别: 本目标涉及 🟡🟠🔴 级新问题？
│   └─ 是 → 执行 🧪 研究协议（见 §C-研究）
│   └─ 否 → 直接进入编码
│
├─ 2. 编码前: 读设计文档 → 写理解声明（五层防御 层0）
│
├─ 3. 编码: 写代码
│
├─ 4. 编码后自检:
│   ├─ 层1: 反翻译并排对比（代码行为 vs 设计意图）
│   ├─ 层2: 约束检查清单勾销
│   ├─ 层3: 对抗性破坏提示（"找出 3 个可能搞错的地方"）
│   └─ 层4: 跨文档追溯矩阵
│
├─ 5. 机械门验证（粘贴真实输出）:
│   ├─ cargo build --workspace
│   ├─ cargo test --workspace
│   ├─ cargo clippy --workspace -- -D warnings
│   └─ cargo fmt --check
│
└─ 6. 更新冲刺提案中的进度
```

**冲刺结束**:
1. 🟢 轻量重构（本冲刺涉及文件：去重、删死代码、统一命名）
2. 全量机械门验证（含 `bash scripts/verify-consistency.sh` + `bash scripts/check-handoff-format.sh`，粘贴输出到 Handoff）
3. 写 DEVLOG（日记——发生了什么、学到什么）
4. 写 Handoff（状态快照——含 💾 恢复点 + 🔍 一致性验证）
5. 更新 `附录E-开发状态.md`（单一真相源——其他文件引用此状态，不复制）
6. 提交用户审核

#### §C-研究：🧪 研究先行协议

**触发条件**（满足任一即触发）:
- AI 从未实现过此类算法/技术
- 涉及性能关键路径（地形采样、寻路、网格生成、渲染管线）
- 涉及架构选型（自研 vs 第三方 crate）
- AI 的第一直觉是"从零写一个"

**四步法**:

```
1. 识别问题类别 → "这是一个 ____ 问题"
   例: "这是一个 GPU-driven LOD clipmap 问题，不是普通网格生成"

2. 搜索行业标准（必须使用 WebSearch，保留 URL）:
   - "[problem] best practice game dev 2025"
   - "[problem] Rust crate"
   - "how does [engine] solve [problem]"

3. 评估 2-3 个方案 → 决策矩阵:
   | 方案 | 业界采用 | 适配成本 | 性能 | 维护负担 |
   | A: [自造]    | 无 | 低 | ? | 高 |
   | B: [现成]    | 有 | 中 | 已知 | 低 |
   | C: [参考论文] | 有 | 中 | 已知 | 中 |

4. 记录决策:
   - 选择方案 X
   - 依据: [为什么]
   - 参考来源: [URL 列表]
   - 如果是 🔴 级 → 写入 ADR（附录C）
```

### 场景 D：中断恢复（冲刺做一半开新会话）

**触发**: Handoff 文件标记冲刺未完成。新会话需要继续。

**读取序列**:
1. `01-核心基础/handoff/` 最新 Handoff — 重点读「💾 恢复点」段
   - ⚠️ 如 Handoff 为旧格式（无「💾 恢复点」节）→ 回退到「代码状态」+「启动指令」节恢复状态
2. 当前冲刺提案文件

**验证**（不可跳过）:
1. `cargo check --workspace`（代码能编译吗？）
2. `git diff`（有未提交的改动吗？和 Handoff 说的一致吗？）

**判断**:
```
验证结果:
├─ 一切吻合 → 继续执行（场景 C 的对应步骤）
└─ 发现漂移:
    ├─ 小漂移（文件改动但目标不变）→ 更新 Handoff，继续
    └─ 大漂移（方向变了/代码被人类改了）→ 报告用户，请求指示
```

**Handoff 恢复点模板**（AI 写 Handoff 时必须包含此段）:
```markdown
## 💾 恢复点（下一会话 AI 必读）

- **当前冲刺**: Sprint-NNN
- **当前目标**: 第 X/3 个 — "[目标名]"
- **最后操作**: 修改了 `path/to/file.rs:142` — [做了什么]
- **机械门状态**: build ✅/❌ / test ✅/❌ (N passed) / clippy ✅/❌
- **未提交文件**: [git status 输出]
- **下一步**: [精确的下一步操作——命令或文件修改]
- **已知陷阱**:
  - ⚠️ [坑1]
  - ⚠️ [坑2]
- **待用户确认**: [如有]
```

### 场景 E：质量门失败

**触发**: 冲刺结束时 cargo test 失败 / clippy 有警告 / build 报错。

**诊断流程**:
```
0. 第一步：分诊（CONSTITUTION §6）
   ├─ 🔴 回归（之前可工作、现在坏了）→ 立刻修，跳过步骤 1-3
   ├─ 🟡 已知偏离（已在追踪中）→ 引用追踪 ID，不重复调查。标记 Handoff。
   ├─ 🟢 参数调优（数值不满意，非正确性错误）→ 附成本估算，用户决定。
   └─ 🔵 结构性依赖（根因依赖未来冲刺）→ 标注 Sprint-N，不现在修。

1. 分诊后需要修复 → 失败是本冲刺引入的吗？
   ├─ 是 → 继续诊断
   └─ 否（之前就坏了）→ 🔴 停止，报告用户。不修。

2. 失败原因明确吗？
   ├─ 明显（拼写错误、漏了分号）→ 立刻修，重新跑门
   └─ 不明确 → 最多修 2 次

3. 同一门连续 3 次不过 → 升级为用户裁决。
   计数规则: 同一冲刺内、同一门的失败次数。
   代码修改后重试 = 新一次。重跑不变代码 = 同一次。
   "冲刺目标 X 的质量门连续 3 次未通过。
    选项:
    A) 延到下一冲刺
    B) 缩小本冲刺范围（放弃目标 X）
    C) 继续投入（预计还需 N 次尝试）"
```

### 场景 F：用户纠偏

**触发**: 用户说"不对，我要的是 X 不是 Y"、"方向错了"、"这个不用做了"。

**流程**:
```
1. 立刻停止当前编码
2. 记录纠偏到 DEVLOG（标记 🏥 用户反馈）
3. 判断影响范围:
   🟢 只影响当前文件 → 调整继续
   🟡 影响本冲刺多个文件 → 更新冲刺提案
   🔴 影响已完成的里程碑 → 写入技术债务 + 下一冲刺提案中评估优先级
4. 向用户确认理解正确后再继续
```

---

### 场景 G：设计文档变更 → 执行手册更新

**触发**: 冲刺前例行检查发现 `WoWorld-Design/Happy Game/开发阶段/` 有修改。

**流程**:
```
1. git diff 开发阶段/ → 变更文件清单

2. 对每个变更执行「三问法」:
   问1: 变更了哪个模块？
   问2: 这个模块在哪个阶段/里程碑？
   问3: 该里程碑的验收标准是否仍然有效？

3. 按级别响应:
   🟢 参数微调（数值、范围）→ 不更新执行手册
   🟡 模块增补（新子模块、新文档）→ 追加里程碑验收项
   🟠 接口变更（trait 签名、数据结构）→ 追溯依赖链，更新下游验收标准
   🔴 架构变更（模块拆分/合并、层移动）→ 标记阶段 README §8 风险段

4. 输出: "设计变更影响报告"（放在冲刺提案的 📋 段）
5. 如果变更影响当前冲刺 → 暂停，请示用户
```

---

### 场景 H：阶段过渡

**触发**: Phase N 全部出口标准通过（含人类签字的 🎮 功能门）。

**流程**:
```
1. 🔴 阶段深度重构（硬门禁——不通过不进下一阶段）:
   □ 全库接口审计（所有 pub trait/struct 签名 vs CLAUDE-INTERFACES.md）
   □ 死代码扫描（cargo udeps 或手动审计）
   □ 依赖方向验证（woworld_core 不依赖 Godot 类型）
   □ 性能基线更新（至少 terrain height sample 延迟）
   □ 所有 TODO/FIXME 解决或登记
   □ 冲刺历史中重复出现的问题已根除

2. 归档本阶段:
   - 阶段 README 状态改为 ✅ 完成
   - Handoff 移入 handoff/archive/

3. 写阶段完成报告 → 存入阶段 devlogs/phase-N-completion.md:
   - 出口标准逐条验证结果
   - 未解决的技术债务清单
   - 下一阶段的已知风险

4. 更新本文件（00-流程总览.md）的「当前所处位置」

5. 更新 附录E-开发状态.md

6. 提案 Phase N+1 的第一个冲刺
```

---

### 场景 I：技术决策升级

**触发**: AI 遇到自己无权做的决策（架构选型 / 性能vs复杂度 / 设计文档矛盾）。

**流程**:
```
1. 停止编码，不猜测

2. 写升级报告（写入冲刺提案的「⚠️ 需用户裁决」段）:
   - 问题是什么
   - 为什么 AI 不能决定
   - 2-3 个可行方案 + 利弊
   - AI 推荐 + 理由
   - 如果用户不回应，建议的默认方案

3. 等待用户裁决

4. 裁决结果记入 ADR（如果是架构级 → 附录C）

5. 如果裁决与原设计文档冲突 → 触发场景 G
```

## 四、阶段切换决策树

```
Phase N 出口标准全部通过？
│
├─ 否 → 继续 Phase N
│   ├─ 技术门失败 → 修复后重检
│   ├─ 设计门失败 → 标记冲突，请示用户
│   └─ 功能门失败 → 回到对应子里程碑
│
└─ 是 → 进入场景 H「阶段过渡」
    │
    └─ 阶段深度重构通过？
        ├─ 是 → 进入 Phase N+1
        └─ 否 → 修复重构问题，重新跑门
```
