# 1H — 生命系统（ECS Phase 1）

> **状态**: 🔴 未开始
> **所属阶段**: Phase 1 — 核心基础
> **前置依赖**: 1A ✅（EntityId）+ 1J ✅（ECS 基础设施——hecs World + 核心 Component）

## 目标

实现 WoWorld 首个完整 ECS 模块——生命系统。以 ECS Component + System 模式实现"死亡→掉落→拾取→腐败→清理"完整链条，验证 ECS 架构在游戏逻辑层的可行性。

这是 ECS Phase 1（`开发文档/06-迁移映射/003-实现路线图.md` §阶段 1），也是后续 NPC/战斗/物品等 20+ System 的地基模板。

## 任务清单

### Step 1.1: 定义生命相关 Component

在 `woworld_ecs/src/components/` 下创建 `vitals.rs`：

- [ ] `Vitals` — hp: f32, max_hp: f32, stamina: f32, hunger: f32, thirst: f32, body_temp: f32, oxygen: f32
- [ ] `Corpse` — 零字段 tag Component（标记已死亡实体）
- [ ] `PendingLoot` — loot_table_id: u32（待处理的掉落）
- [ ] `LootResult` — item_ids: Vec<ItemDefId>（已确定的掉落物列表。⚠️ Vec 不内联——使用 Handle+Storage 模式：LootResultHandle + LootResultStorage Resource）
- [ ] `DecayingRemains` — decay_progress: f32, 0.0→1.0（腐败进度）
- [ ] `PendingDespawn` — frames_remaining: u32（延迟清理帧数）
- [ ] `DeathCause` — 枚举（30 种 6 类，见原设计文档）
- [ ] `RegenState` — hp_regen_rate: f32, stamina_regen_rate: f32（不随帧更新的基础再生率——用于 RegenSystem）

> ⚠️ 所有 Component 必须满足 ECS 铁律（纯数据、零方法、无堆数据内联、'static + Send + Sync）。定义完成后对照 `开发文档/00-ECS哲学与架构总纲/006-ECS铁律与陷阱.md` 逐条核验。

### Step 1.2: 实现生命 System

在 `woworld_ecs/src/systems/life/` 下创建（每个 System 一个文件）：

- [ ] `DeathWatchSystem` — 查询 `(Entity, &Vitals)`。hp <= 0 → 移除 Vitals, 插入 Corpse + PendingLoot + DeathCause
- [ ] `LootRollSystem` — 查询 `(Entity, &Corpse, &PendingLoot)`。查 LootTableRegistry Resource → 移除 PendingLoot, 插入 LootResultHandle
- [ ] `ItemSpawnSystem` — 查询 `(Entity, &Corpse, &LootResultHandle)`。查 LootResultStorage → 生成物品 Entity → 移除 LootResultHandle
- [ ] `CorpseDecaySystem` — 查询 `(Entity, &mut DecayingRemains)`。decay_progress += delta。>= 1.0 → 移除 Corpse, 插入 PendingDespawn{frames: 300}
- [ ] `RegenSystem` — 查询 `(Entity, &mut Vitals, &RegenState)`。每帧恢复 hp/stamina（上限 cap 在 Vitals 中）
- [ ] `CleanupSystem` — 查询 `(Entity, &mut PendingDespawn)`。frames_remaining -= 1。<= 0 → despawn Entity

> ⚠️ System 之间通过 Component 拆装通信，不设 before/after。顺序从组件变化中涌现：
> DeathWatch(装 Corpse+PendingLoot) → LootRoll(看到 Corpse+PendingLoot) → ItemSpawn(看到 Corpse+LootResult) → ...

### Step 1.3: 定义 Resource

- [ ] `LootTableRegistry` — HashMap<u32, LootTable>（哪个 loot_table_id 对应哪些可能掉落物）
- [ ] `LootResultStorage` — HashMap<EntityId, Vec<ItemDefId>>（Handle+Storage 模式——Component 只存 Handle，实际数据在 Resource 中）
- [ ] `SpeciesRegistry` — HashMap<SpeciesId, SpeciesVitals>（种族基础生命体征模板）

### Step 1.4: 集成测试

- [ ] "死亡→掉落→拾取" 完整链条测试：spawn Entity(Vitals{hp: 100}) → 外部设 hp=0 → 运行 6 个 System → 断言 Entity 被 despawn + 物品 Entity 被 spawn
- [ ] RegenSystem 测试：spawn Entity → 扣血 → 运行 N 帧 → 断言 hp 部分恢复
- [ ] CorpseDecaySystem 计时测试：spawn → 死亡 → 断言 decay_progress 递增

## 验收标准

### 🔧 技术

- [ ] `cargo build --workspace` 通过
- [ ] `cargo test --workspace` 现有 107 测试全绿 + 新增生命 System 测试 ≥ 6 个（每个 System 至少 1 个）
- [ ] `cargo clippy --workspace -- -D warnings` 零警告
- [ ] `bash scripts/verify-consistency.sh` 通过

### 📐 设计

- [ ] 所有 Component 通过 ECS 铁律 8 条逐条核验（对照 `开发文档/00-ECS哲学与架构总纲/006-ECS铁律与陷阱.md`）
- [ ] System 之间无直接函数调用——仅通过 Component 拆装 + Resource 读写通信
- [ ] System writes 集合无交集（同一 Component 不被两个 System 同时写入）
- [ ] 死亡→掉落→拾取 链条的一帧延迟是设计特性，不是 bug
- [ ] EntityId ↔ hecs::Entity 互转无损
- [ ] `hecs::World` 仅存在于 WorldDriver——未泄漏到 Godot 侧

### 🎮 功能

- [ ] Godot 中可通过 Rust `#[func]` 查询实体生命体征（查询 hecs World → 读 Vitals Component → 返回 PackedFloat32Array）
- [ ] 生命体征的增减逻辑正确（damage/heal/consume——通过 System 消费 Vitals Component 实现）
