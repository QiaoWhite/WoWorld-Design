# 开发日志 — 2026-06-30

## Sprint-018：缝隙消除攻势（15 次提交 · 全败）

**目标**：消除地形表面可见的透明缝隙（天空色，10-50m 间距，弯曲不规则）。

**起点**：Sprint-017 完成后，绿色伪影已修复，但 tile 边界透明缝隙 + 中距离 LOD 层次锯齿仍存在。

---

### 上午 — Sprint-018 启动：几何一致性修复

基于错误 RCA（认为缝隙来自相邻 tile 的 bottom_y 不一致 + GPU 光栅化裂纹），
实施 5 层修复：

1. **aligned_vertical()**: tile origin 锚定到 LOD+2 栅格，统一 bottom_y
2. **SH 边界重叠**: overlap=1，35² 顶点网格（33+2 重叠行）
3. **Transition half-thickness offset**: 过渡顶点沿面法线偏移 half_vs
4. **f64→f32 精度一致性**: 跳过（被 1-3 层消解）
5. **背面剔除恢复**: CullMode::DISABLED → BACK

**结果**: 缝隙仍在。且引入新 bug——地面偶尔闪透明（裙边法线被背面剔除裁剪）。
紧急回退 CullMode → DISABLED。

---

### 中午 — Sprint-018b：GPU 裂纹修复

经研究，发现缝隙可能是 GPU 光栅化多 MeshInstance3D 的亚像素裂纹。
实施：

1. **全局 bottom_y**: 替换 aligned_vertical，按 LOD 级别 poll() 预计算统一垂直范围
2. **Skirt geometry**: SH + Transvoxel tile 边缘生成向下裙边三角形

**结果**: 缝隙仍在。闪透明 bug 为此引入的回归。

---

### 下午 — Sprint-018c：MSAA + 顶点膨胀

深度搜索后发现 Godot 已知 Bug #16337——MSAA 导致多 MeshInstance3D 缝隙。
`project.godot` 中 `msaa_3d=2`。

1. **关闭 MSAA** → 启用 TAA + Debanding
2. **变强度顶点膨胀**: inflate_edges — 边界 1% 膨胀，中心 0%，二次衰减

**结果**: MSAA 关闭后"有所缓解但仍在"。顶点膨胀也在。
**关键发现**: 问用户缝隙间距——"十几米到几十米"——匹配 Transvoxel tile 尺寸。

---

### 傍晚 — 单 Mesh 合并

推断根因：Transvoxel 等值面对每个 tile 独立提取，边界三角形拓扑不一致。
**唯一正确方案：每 LOD 合并为一个 ArrayMesh。**

1. 重构 WorldDriver: 移除 600 节点对象池 → 8 个 LodLayer（每 LOD 1 个 MeshInstance3D）
2. `merge_meshes`: 顶点+索引拼接
3. 发现 bug: `completed.clear()` 导致合并 mesh 冻结 → 修复为 `retain()`

**结果**: 缝隙仍在。证明不是同 LOD tile 边界问题。

---

### 晚间 — 顶点焊接 + 平坦地形诊断

升级 merge_meshes 为空间哈希焊接：
- 5cm 容差焊接全部顶点 → 无效
- 边界感知焊接（仅 tile 边界 1m 内，20cm 容差） → 无效

**关键诊断**:
1. `test_adjacent_tiles_share_identical_edge_heights`: 相邻 tile 共享边 height_at() 完全一致
2. `test_flat_terrain_watertight`: 平坦密度场提取的等值面**完全水密**（无边流形错误）

平坦地形水密 → MC 算法正确。真实地形有裂缝 → 密度场梯度问题。

**尝试**: `half_band` 1.0→3.0（过渡带从 2m→6m） → 无效。
**尝试**: Camera3D near=1.0 far=20000（深度精度） → 无效。

---

### 深夜 — 投降

15 次提交，15+ 种修复，全部无效。132 tests 全绿，clippy 零警告。

**已排除的根因清单**:

| 类别 | 尝试 | 结果 |
|------|------|------|
| 后处理 | MSAA/TAA/Debanding 关闭 | 无效 |
| 几何 | bottom_y 统一/重叠/膨胀/裙子 | 无效 |
| 渲染 | AABB 膨胀/背面剔除/相机 near-far | 无效 |
| 架构 | 单 mesh 合并 + 顶点焊接 | 无效 |
| 算法 | half_band 梯度平滑 | 无效 |

**最终判断**: 裂缝大概率是 Godot 4.7 Forward+ 渲染器在超大世界坐标
（顶点达 10000m）下的固有限制——不是几何问题，不是材质问题，
是渲染管线层面的亚像素精度问题。

**下步建议**: RenderDoc 抓帧分析 / Godot Compatibility 渲染器测试 /
Floating origin 世界坐标缩减 / 最小化 Godot 场景复现。

### 教训

1. **RCA 必须测试驱动**: 5 层修复全部基于错误假设。如果能先写
   `test_flat_terrain_watertight` 确认 MC 正确性 + `test_adjacent_tiles`
   确认高度一致，就能跳过大部分错误尝试。
2. **"弯曲/不规则"才是关键信息**: 在问到这个之前，一直错误地假设
   缝隙是横平竖直的 tile 网格线。一个精准的问题节省半天调试。
3. **单 mesh 合并本身是正确的架构改进**: 即使没解决缝隙，
   从 560 draw calls → 8 的收益是实打实的。
4. **Godot 渲染调试需要可视化工具**: 纯代码推理无法定位渲染管线的
   亚像素问题，必须借助 RenderDoc/Godot Debug Draw 等工具。
