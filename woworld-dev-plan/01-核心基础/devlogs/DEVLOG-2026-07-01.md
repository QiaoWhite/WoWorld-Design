# 开发日志 — 2026-07-01

## Sprint-019：Floating Origin — 裂缝根因及修复

**承接**：Sprint-018 15 次提交全败后，交接文档诊断为 Godot Forward+ 渲染精度问题。

**成果**：经 6 次方案迭代（v1→v6），确认裂缝根因为 **GPU fp32 光栅化精度不足**（世界坐标达 10,000m 时精度仅 ~0.5mm），最终采用 **Vertex Shader Camera-Relative 方案**消除裂缝。

---

### 方案演进

| 版本 | 方法 | 结果 |
|------|------|------|
| v1 | WorldOrigin GDScript 节点 + `_process` 移动 | Sun 路径断裂 + 性能问题 |
| v2 | Rust `set_global_position(-snap)` 移动 WorldDriver | 裂缝消失 ✅ 但 AABB 错位 → 双重地形 |
| v3 | 顶点数据偏移（减 camera_pos，节点不动） | AABB 在原点 → frustum culling 错误 → 地形消失 |
| v4 | v2 + 本地 snap 追踪（消除 `get_global_position` FFI） | 同 v2 问题 |
| v5 | ShaderMaterial + `VERTEX -= camera_pos` | 双倍偏移错误（VIEW_MATRIX 已含 -camera） |
| **v6** | **ShaderMaterial + `mat3(VIEW_MATRIX) * (VERTEX - camera_pos)`** | **裂缝消失 ✅ 零节点移动 ✅ AABB 正确 ✅** |

### v6 最终方案细节

**Shader**（`terrain_chunk.rs` 中 `ShaderMaterial`）：
```glsl
shader_type spatial;
render_mode unshaded, cull_disabled;
uniform vec3 camera_pos = vec3(0.0);
void vertex() {
    vec3 rel = VERTEX - camera_pos;           // 相机相对坐标 < 500m
    vec3 view_pos = mat3(VIEW_MATRIX) * rel;  // 只旋转，跳过 VIEW 平移
    POSITION = PROJECTION_MATRIX * vec4(view_pos, 1.0);
}
void fragment() {
    ALBEDO = COLOR.rgb;
}
```

**关键数学**：`mat3(VIEW) * (V - cam) = R * V - R * cam`，与标准管线 `VIEW * V = R * V - R * cam` **完全等价**，但所有中间计算都是小值（高精度）。

**架构特征**：
- 场景树完全不变（无 WorldRoot，无 `set_global_position`）
- 顶点数据永不被修改（世界空间 AABB → frustum culling 正确）
- 每帧仅传一个 `camera_pos` uniform（`player.global_position`）
- `query_height(global_position)` → 物理查询正确

## 遗留问题

### 1. LOD 接缝（已知根因，待下个冲刺修复）

过渡单元已禁用（`transition_faces=0`），cross-LOD 边界有 ~1m 顶点偏差。当前被顶点焊接（20cm）+ 大气雾化掩盖，但不可见的地形放大后仍能看到。

**计划方案**：Skirt Geometry — 在每个 tile 边界生成向下延伸的垂直裙边几何。相邻 tile 的裙边在共享面上重叠，填满裂缝。这是 Unreal Engine / CDLOD / GPU Gems 的标准做法。

### 2. 性能卡顿

两个主要瓶颈：
- `estimate_ring_vertical()`：每帧 420 次 `height_at()` 噪声调用。修复：缓存结果，仅在玩家水平移动 >1 tile 后重算。
- `desired_keys()`：8 级 LOD 遍历每帧重算。修复：缓存 desired key set。

Dirty flag 已消除稳态 `merge_and_upload` 开销。
