## 第一问：地形修改编排层

### 调研

- **Veloren**（SPECS ECS·MMO 体素 RPG）：TerrainGrid 为 ECS Resource，BlockChange 批量积累，每帧 dispatch 1 chunk
- **bevy_voxel_world**：两层架构（procedural 函数 + persistent HashMap），set_voxel 自动 re-mesh
- **Ethertum**（Bevy）：Chunk 为 Entity，Surface Nets 提取
- 关键数据：32³ mesh 提取 0.1-1ms，密度场计算 5-20 倍

### 架构决策（ADR-007）

**内核不转 ECS**：地形数据保持 ECS Resource，不走 Archetype。
**编排层入 ECS**：修改请求→dirty→重 mesh→NPC 感知走 Component+System+Event。

### 五轮批判

| 轮次 | 谬误 | 遗漏 | 矛盾 | 关键发现 |
|------|------|------|------|---------|
| R1 | 3(RwLock→CoW 等) | 3 | 2 | +8邻居不适用Transvoxel |
| R2 | 2(Clipmap绕过栈等) | 4 | 0 | 2D/3D分裂是核心问题 |
| R3 | 1(VoxelChunk缺EditDensity) | 1 | 0 | VoxelChunk临时栈不含编辑层 |
| R4 | 1(Ocean绕过EditHeightfield) | 2 | 0 | Ocean有独立noise.sample_height |
| R5 | 1(density_at遗漏) | 2 | 0 | 密度叠加仅在栈内，手动查询遗漏 |

### 代码交付

| 文件 | 操作 | 行数 |

### 核心修复：2D/3D 分裂

原 `height_at()` 走裸噪声，`density_at()` 走 DensityStack——修改后 NPC 会浮空/沉地。修复：全部 9 个 TerrainQuery 方法通过 `actual_height_at()` 或手动密度叠加感知 EditTerrain。

# 第四场：RENDER-002 VoxelChunk LOD0 阴影修复

## 摘要

用户报告 LOD0 正方形区域光照异常：摄像机垂直向下时全黑，倾斜时出现锯齿几何亮斑，光照同时受太阳方向和摄像机角度影响。经 7+ 轮假设-验证循环（法线计算、绕序、PBR 参数、COLOR 传递、z-fighting、阴影投射/接收），最终定位根因为 DirectionalLight3D 阴影贴图中 VoxelChunk（MC 等值面）与 clipmap（高度场 displacement）深度值存在微小差异 → LOD 边界阴影贴图不连续 → PCF 滤波跨边界产生视角依赖的错误阴影。

**修复**：terrain + voxel 两个 shader 均添加 `render_mode shadows_disabled`（只投射阴影、不接收阴影）。地形是连续表面，不需要自阴影。

**总计**：3 文件修改（2 shader + 1 Rust），2 bug 条目更新，341 tests 全绿。

---

## 诊断过程

### 症状精确化

| 视角 | 表现 |
|------|------|
| 摄像机垂直向下 | LOD0 正方形**全黑** |
| 摄像机倾斜 | 正方形内出现**方正的、多条边的光亮区域**，边缘**锯齿状** |
| 高空俯视 | 异常**局限于** LOD0 正方形边界 |
| 低空飞行 | 异常**蔓延**到相邻区域，尤其屏幕边缘 |
| 太阳角度变化 | 区域内的光照/阴影随太阳**同步变化** |

### 排除路径（按时间顺序）

| 轮次 | 假设 | 测试方法 | 结果 | 结论 |
|------|------|---------|------|------|
| 1 | dFdx/dFdy 屏幕空间导数法线对小三角形不稳定 | 还原为 mesh 顶点法线 | 问题仍在 | ❌ 不是法线计算方式 |
| 2 | `winding_is_godot_front()` 极性与 3c3018c 法线方向修复不匹配 | 修改 `< 0.0` → `> 0.0` | **全透明**（全部三角形被剔除） | ❌ 不是绕序——且原始绕序正确 |
| 3 | PBR 参数 (ROUGHNESS/SPECULAR) 导致过暗 | 对比 terrain shader 相同参数 | terrain shader 正常工作 | ❌ 不是 PBR 参数 |
| 4 | vertex COLOR 未传入 shader → ALBEDO=(0,0,0) | 诊断日志 + `COLOR * 10.0` | 放大后可亮成白色 | ❌ COLOR 正确传入，但正常亮度下被压暗 |
| 5 | **着色诊断**: `ALBEDO = vec3(1,0,0)` 固定亮红 | 绕过所有顶点数据 | **亮红色**，但亮度仍随视角变化 | ✅ **问题在光照/阴影**，不在 mesh/颜色 |

| 7 | Godot 自动 NORMAL_MATRIX vs 手动变换 | 切换回 per-fragment dFdx/dFdy（与 terrain shader 对齐） | 问题仍在 | ❌ 排除所有法线计算路径 |
| 8 | **阴影投射**: VoxelChunk 自阴影 acne | `cast_shadows=OFF` | `全黑`消失，出现"正方形亮块" | ✅ **阴影是核心因素**——但未完全解决 |
| 9 | **z-fighting**: L1 clipmap(孔洞30m) 与 VoxelChunk(覆盖~40m) 重叠 | 扩大 L1 孔洞到 45m | 产生 LOD 缝隙，问题仍在 | ❌ 方向错误——回退 |
| 10 | **阴影接收**: VoxelChunk 接收 clipmap 投射的阴影 → 阴影贴图深度不匹配 | `shadows_disabled` (voxel shader) | LOD0 核心区域改善，低空边缘仍有 | ✅ 阴影接收是根因——但 clipmap 也在自阴影 |
| 11 | **决定性测试**: terrain + voxel 都禁用阴影 | `shadows_disabled` (both) | **所有异常消失** | ✅✅✅ **根因100%确认** |

### 根因机制

```
阴影贴图（DirectionalLight3D shadow map, 1024²）
├── VoxelChunk 区域 (0-40m): MC 等值面提取 → 深度 = MODEL_MATRIX * VERTEX
├── Clipmap L1 区域 (30-80m): 高度场 displacement → 深度 = VERTEX + node_pos, Y=heightmap
└── 边界 (30-40m): 两种深度路径产生 ~sub-mm 级差异
    └── PCF 滤波 (Percentage Closer Filtering) 跨边界采样
        └── 邻近纹素深度跳变 → 阴影判定错误
            └── 屏幕空间 PCF 采样模式视角依赖 → 异常随摄像机角度变化
```

PCF 采样模式由屏幕空间投影决定：
- 摄像机垂直向下：采样模式对称 → LOD0 区域一致变暗（`全黑`）
- 摄像机倾斜：采样模式不对称 → 边界处深/浅交替（`锯齿亮斑`）
- 低空飞行：地面覆盖更多屏幕像素 + 屏幕边缘极端透视 → 异常蔓延

## 修复

### 代码变更

### 设计原则

# 第五场：Clipmap 群系颜色修复——全地形只显示绿色

## 摘要

用户报告 bug：LOD0（VoxelChunk）正方形内和极远山中颜色正确（雪白、岩灰），但中间所有 Clipmap 地形（LOD 1-7）只有绿色。经代码追踪定位根因为 `actual_material_at()` 三个路径的执行顺序 bug——群系分类器路径是死代码，所有 Clipmap 地表材质回退到 `Grass`。修复：重排优先级（群系 → DensityStack）+ 消除 `sample_vertex` 冗余噪声调用 + 恢复水下 3 层材质分层。代码审查发现 3 个额外问题，全部在提交前修复。

**总计**：1 Rust 文件修改（`terrain.rs` ~60 行净变更），341 tests 全绿，clippy 零警告。

---

## 症状

| 区域 | 颜色 | 原因 |
|------|------|------|
| LOD0 正方形（VoxelChunk） | ✅ 正确（白/棕/灰等） | 自建 DensityStack，TerrainBaseDensity 携带 biome |
| Clipmap LOD 1-6（近~中距离） | ❌ 只有绿色 | → 下方根因分析 |
| Clipmap LOD 7（极远山） | ✅ 正确（灰/棕） | `material_from_height()` h>200→Gravel, h>500→Rock |

---

## 诊断过程

### 关键代码路径追踪

```
generate_heightmap_data()                          [clipmap.rs:195]
  └─ terrain.sample_vertex(x, z)                  [terrain.rs:178]
       └─ actual_material_at(pos)                  [terrain.rs:247]
            ├─ Path 1: EditDensity → None, skip
            ├─ Path 2: DensityStack::material_at() → ALWAYS Some(...)
            │    └─ fallback: layers.first() = TerrainBaseDensity (biome: None)
            │         └─ material_from_height() → most low terrain = Grass
            └─ Path 3: biome_classifier → ★ DEAD CODE
```

**根因**：`DensityStack::material_at()`（density.rs:71-81）的 fallback `self.layers.first().map(|l| l.material_at(pos))` 永远返回 `Some(...)`——栈永不为空。Path 2 永远匹配，Path 3（群系分类器）是死代码。

**为什么 VoxelChunk 正确？** `submit_voxel_job()` 创建自己的 DensityStack，TerrainBaseDensity 通过 `with_biomes()` 携带了群系引用。Clipmap 的 `generate_heightmap_data()` 使用 `self.terrain`（HeightfieldTerrain），其 DensityStack 中的 TerrainBaseDensity 没有 biome（`with_biomes()` 只设置了 `self.biome_classifier`，未重建 DensityStack）。

**为什么极远山正确？** `material_from_height()` 对 h>200 返回 Gravel、h>500 返回 Rock——这些恰好在色表中。

### 两个叠加问题

1. **`with_biomes()` 不更新 DensityStack**：只设置 `self.biome_classifier`，DensityStack 内的 TerrainBaseDensity 仍为 `biome: None`
2. **`DensityStack::material_at()` fallback 永不为 None**：Path 3 成为死代码

---

## 修复

### 主修复：`actual_material_at()` 重排优先级

**Before**: EditDensity → DensityStack（永远命中）→ ~~biome classifier（死代码）~~

**After**: EditDensity → 水下 3 层高度法 → 群系分类 → DensityStack（降级为地下/编辑回退） → 高度+坡度最终回退

```rust
// 新顺序（actual_material_at_with_height）：
// 1. EditDensity 材质覆盖（CoW 快照）
// 2. 水下：h<-100→Stone, -100≤h<-10→Gravel, -10≤h<0→Sand
// 3. 陆地：群系分类优先
// 4. DensityStack（地下/编辑区域回退）
// 5. 高度+坡度最终回退
```

### 附带修复 1：`sample_vertex` 消除冗余噪声调用

原 `sample_vertex` → `actual_height_at`（1 次 terrain_height）→ `actual_material_at`（再 1 次 terrain_height）= 2 次冗余。新增 `actual_material_at_with_height(x, z, h)` 私有辅助方法，复用预计算高度。Clipmap 256² 纹理每 LOD 节省 ~65k 次 Perlin 噪声评估。

### 附带修复 2：水下材质 3 层分层恢复

旧 `actual_material_at` 有 `h<-100→Stone | -100≤h<-10→Gravel | -10≤h<0→Sand` 三层。初版修复简化为 `h<-10→Sand | else→Water`，丢失了 Stone 和 Gravel。代码审查捕获此回归，已恢复。

### 附带修复 3：`with_biomes()` 同步重建 DensityStack

确保 DensityStack 内 TerrainBaseDensity 也携带 biome 引用——DensityStack::material_at() fallback 路径现在也能返回正确的群系材质。
