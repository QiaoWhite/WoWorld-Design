## VoxelChunk LOD 0 — 当前状态（⚠️ 未完成）

### 已实现的架构
- `voxel_chunk.rs`：VoxelChunk GodotClass（Node3D, 被动容器）
- `voxel_terrain.gdshader`：专用 spatial shader（顶点色直通, camera-relative）
- `init_voxel_grid()`：双 pass —— 先扫描 25 个 chunk 的地形高度, 计算统一 wy/vy, 再创建
- `submit_voxel_job()`：rayon 后台 `transvoxel_extract`, mpsc channel
- Harvest：每帧 max 1 upload, `upload_voxel_mesh()` 将 TerrainMeshData→ArrayMesh
- Grid 管理：drift-based destroy+create
- Material 延迟应用修复：`pending_material` 字段 + `ready()` 中应用

### 已诊断但未解决的核心问题

**问题**：VoxelChunk mesh 渲染为"彩色碎片"——chunk 内部有水平间隙(~1m), chunk 之间没有无缝连接, 也没有和 LOD1 clipmap 地形连接。

**已识别的因素**（各修复的边界和教训）：

| 尝试 | 根因判断 | 修复 | 结果 |
|------|---------|------|------|
| 法线方向 | 密度梯度指向固体内部 | gradient_from_stack 取反 | **无效**——法线不是主要问题 |
| 材质未设 | mesh_instance 在 ready()前为 None | pending_material 延迟应用 | ✅ 修复——chunks 现在有颜色 |
| 三角形绕序 | MC 标准 TRI_TABLE 绕序面向固体内部 | `cull_disabled` 临时解决 | ⚠️ 双面渲染不是正确方案 |
| y-origin 不一致 | 每个 chunk 独立计算 wy | 统一 y-origin | ❌ 用户判定"完全错误" |
| 顶点坐标系去 | 世界空间→本地空间双倍偏移 | 减去 chunk origin | ✅ 位置修正 |

### 对下一会话的建议

**根本问题方向**：VoxelChunk 碎片化的根因可能不在 Rust 侧（提取逻辑、world position、y-origin），而在**shader 侧或 Godot ArrayMesh 上传**的某个基础环节。建议从以下角度排查：

1. **用 Godot 内置 Cube/Sphere mesh 替换 VoxelChunk**：验证 VoxelChunk GodotClass 的 Node3D positioning 是否正确
2. **用静态 PB 数据替换 transvoxel_extract 输出**：直接构造已知正确的 TerrainMeshData（比如一个简单平面），验证 upload 管线
3. **检查 MODEL_MATRIX 在 voxel_terrain.gdshader 中的行为**：当 Node3D 的 position 不为原点时，`(MODEL_MATRIX * vec4(VERTEX, 1.0)).xyz` 是否真的产生了正确的世界位置
4. **比对 terrain.gdshader 和 voxel_terrain.gdshader**：terrain 网格使用 `VERTEX + node_pos` (node_pos 是 manual uniform)，voxel 使用 `MODEL_MATRIX`——两者可能产生不同的结果

**快速验证步骤**（建议在新会话中从第 1 步开始）：
1. 暂时禁用所有 VoxelChunk，恢复 LOD 0 clipmap
2. 创建一个最简单的测试 VoxelChunk：纯手工构造一个平面 mesh
3. 验证平面是否渲染在正确的位置和颜色
4. 逐步增加复杂度直到找到碎片化的触发点
