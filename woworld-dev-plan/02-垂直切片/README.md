> **旗子**: 「活着的村庄」——旁观 3-5 个 NPC 纯涌现地过完一天

## §1 阶段目标

**一句话（旗子·可判定）**：打开村庄场景、把时间调快，肉眼看着 **3-5 个 NPC** 各自**纯涌现**地过完一天——**醒来（昼夜节律调制）→ 饿/渴/困 → 走向最近的可采集食物/水/休息点 → 采集并进食使需求真实下降 → 附近有人就寒暄 → 否则漫游 → 夜里更倾向睡**；**存档退出、重载后村庄从中断处继续**。玩家全程不操作。

**为什么是「垂直切片」而不是「继续铺系统」**：Phase 1 做完后所有 Layer 1 系统就位，但它们是孤立的。垂直切片迫使我们将这些系统**纵向打通**——从 NPC 需求到世界反馈的完整闭环。这是风险最高的步骤：如果核心循环（涌现的日常）不成立，铺再多系统也没用。

**为什么叫"探针（proof-of-fun）"**：本切片**有意识地**在 Phase 1 未全部完成时启动（见 §2.1），目的是尽早验证"当前底层原子能否涌现出一个活着的村庄"。它是一次**诊断性探针**——市场冷清、行为单调都是**有效实验数据**（指向下一个真缺口），而不是失败。

**A1 纯涌现纪律（不可违反）**：只实现设计文档已规定的真实路径。**禁止引入任何"脚手架/占位驱动"代码**（如为让市场繁忙而造非涌现的生产驱动，或用假坐标当需求满足点占位）。涌现不出来的，如实呈现"空白/安静"，不得伪造。**宁可薄而真，不要满而假。** 依据用户裁决 A1（2026-07-12）。

## §3 子里程碑与依赖顺序

> ✅ 定稿 2026-07-12（含自审计修正：缩回 V0、插入 Vf 食物源、拆 V4a/V4b、接口纪律）。A1 纯涌现，执行顺序已锁（10 步）。

**执行顺序（1→10，箭头为依赖）**：

```
1 V0 舞台搭建 ─┐
2 V1 昼夜涌现 ─┼─→ 3 V4a 问候/情绪气泡  （依赖今天就绪 → 最早的"活起来"信号）
              │
4 Vf 食物源落地 ┘  （Poisson disc 可采集植被·P2.25 最小子集）
      │
      ▼
5 V2 牵引移动 → 6 V3a 代谢闭环(命门) → 7 V3b 市场接真 → 8 V4b 交易气泡
                                                            │
                                            ┌───────────────┘
                                            ▼
                                  9 V5 旁观工具 → 10 V6 快照存档
```

**顺序锁定理由**：
- V0（NPC 库存）+ V1（昼夜调制）是**无依赖地基**，先做。
- **V4a（问候/情绪气泡）提前到第 3 步**——依赖（邻近 `social.rs` / 情绪 / 需求）**今天就绪**，是"村庄活起来"最早的可见信号，不必等经济核心。
- **Vf（食物源落地）是自审计发现的缺失里程碑**——A1 下"食物来自采集"要求先有真实可采集植被（现 `query_harvestable` 恒返回空）。必须在 V2/V3a 之前。
- V2（走到食物点）需 Vf 的食物点存在；**V3a（采集→吃）是命门**，需 V0 库存 + Vf 植被 + V2 到达三者齐备。
- V3b 市场、V4b 交易气泡、V5 工具、V6 存档依次叠加。

## §5 出口标准

- 🔧 **技术门**: `cargo build` + `test` + `clippy -D warnings` + `fmt --check` 全部通过。
- 📐 **设计门**: 本切片涉及接口与 `CLAUDE-INTERFACES.md` 一致；**A1 纪律零违反**（无脚手架/假坐标/占位驱动代码）；**资源查询复用 `VegetationProvider`/`OceanProvider`，无平行抽象**。
- 🎮 **功能门（旗子）**: 人类打开村庄场景、加速时间，肉眼确认 3-5 个 NPC 完成 §1 描述的完整涌现日循环；存档→重载→从中断处继续。
- 🔬 **探针结论**: 如实记录"涌现出的村庄有多活"——哪些行为涌现成功、哪些冷清/单调，形成"下一个真缺口"清单（很可能指向 layer-3 经济驱动 / 技能 / AgentSnapshot）。这是探针的**首要产出**。
