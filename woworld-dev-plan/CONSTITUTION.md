## §1 核心哲学

1. **涌现优先**：玩家与 NPC 无本质区别——NPC 能正常运作，玩家只需"夺舍"NPC 再优化细节。许多内容从现有原子系统涌现，不轻易建立独立模块。
2. **编码不替设计做决定**：Claude 可以提建议，但不能自己填补设计空白。涉及设计决策的偏离必须用户裁决。
3. **贴文档写代码**：触及新模块时，第一步是逐字精读 `WoWorld-Design/Happy Game/开发阶段/<模块>/` 下的原设计文档。重大修改（偏离原设计意图的实现决策）必须提前与用户沟通，不可自作主张。
4. **设计→编码→反馈→修正循环**：不追求设计"冻结"再写代码。编码会反向暴露设计缺陷，这是健康的——按 §5 协议处理。
5. **冲刺自驱**：Claude 根据本宪法自主判断下一步该做什么，形成冲刺提案 → 用户审批 → 执行 → 审核 → 循环。
6. **长远眼光**：实现决策不以当前冲刺为唯一优化目标。每次编码选择（数据结构、算法、接口形状）必须考量：未来 25 个模块全部接入后，这个选择还成立吗？具体体现为——选 `u64` 而非 `u32`（未来可能有百万实体）；trait 签名考虑所有已知消费方需求；今天只实现一个模块，但接口设计预留了 25 模块的扩展点。
7. **计划反身性**：宪法和冲刺计划不是冻结文档。Claude 在冲刺执行中如果发现流程规则有改进空间（如检查步骤重复、协议从未触发、提案格式缺字段），应在冲刺提案的"需裁决问题"段或交接摘要中主动提出。小改进（不影响 §1-§11 主体规则）Claude 可自决并记录。结构性改进需用户审批。

## §2 优先级决策矩阵

### 五个维度

| 维度 | 含义 | 评分（1-5） |
|------|------|------------|
| **依赖链** | 其他任务是否阻塞在它上面？被多少东西依赖？ | 5 = 大量下游阻塞 |
| **风险验证** | 是否验证某个关键技术假设？不做的话后续可能是空中楼阁？ | 5 = 核心架构风险 |
| **用户价值** | 做出来用户/开发者能不能马上感知到进展？ | 5 = 立即可玩/可见 |
| **设计完备** | 对应的设计文档是否已充分冻结？ | 5 = 设计无歧义，可直接翻译为代码 |
| **长期架构健康** | 这个选择在 25 个模块全部接入后还成立吗？ | 5 = 长期最优，技术债最小 |

### 加权公式

```
候选任务总分 = D × w_d + R × w_r + V × w_v + C × w_c + H × w_h
```

### 默认权重

| 权重 | 值 | 含义 |
|------|-----|------|
| w_d（依赖链） | 3 | 最高优先级——依赖链断裂是阻塞性的 |
| w_r（风险验证） | 3 | 等同于依赖链——技术风险未验证的进度是假进度 |
| w_v（用户价值） | 2 | 重要但次于架构正确性 |
| w_c（设计完备） | 1 | 设计可补，不应成瓶颈——但太低触发 §7 🔴 级 |
| w_h（长期架构健康） | 2 | 短期爽但长期债的打低分——与依赖链同级 |

> **用户有权在任何时刻调整权重**。例如："这周只验证性能" → w_r=5, 其余=0。

### 使用方式

Claude 对候选项列表打分，得分最高的 3 个进入冲刺提案的"候选项对决"表。用户审视后可调整、否决、或指定候选。

## §4 质量门

### 三层检查

| 层 | 内容 | 执行者 | 通过标准 | 用户审？ |
|---|------|--------|---------|----------|
| **机械层** | `cargo build` + `cargo clippy -- -W clippy::all -W clippy::pedantic` + `cargo test` + `cargo fmt --check` | Claude | 全绿，零警告 | 否 |
| **契约层** | 实现的 trait/接口是否符合 `CLAUDE-INTERFACES.md` 契约 | Claude | 所有接口签名匹配，所有权归属正确 | 否 |
| **设计层** | 实现是否真的符合 `开发阶段/<模块>/` 设计文档的意图 | Claude 先自检（五层防御 SOP） | 逐项对照文档核验 | 是——你审查代码 + 运行行为 |

### 设计层自检 SOP（五层防御体系）

> 各层互相独立，层层拦截不同类型的 LLM 编码错误。详见 grill-me Q20（重）分析。

| 层 | 时机 | 内容 | 拦截错误类型 |
|---|------|------|------------|
| **层 0: 理解声明** | 编码前 | 输出结构化理解声明——本冲刺要创建的类型/算法/约束清单 + "我不确定的地方"及解读假设。用户可提前看一眼抓到大的理解偏差 | 语义漂移（类型 1-5） |
| **层 1: 反翻译并排对比** | 编码后 | 从代码提取中文行为描述，与原始设计文档逐条并排陈列（设计说 X \| 代码做 Y），差异比单读更容易识别 | 翻译错误（类型 6-10）、确认偏误（类型 15-16） |
| **层 2: 约束检查清单勾销** | 编码后 | 从设计文档提取具体的布尔判断约束清单（如"NPC 移动前必须调用 force_check"），逐条打勾。有一条没勾不 commit | 远距离约束遗漏（类型 3） |
| **层 3: 对抗性破坏提示** | 编码后 | 角色切换——"假设这段代码至少有一个严重的设计偏差。找出 3 个可能搞错的地方。"同样包含测试专项："假设测试漏测了至少一个关键边界条件，找出来。" | 确认偏误（类型 15）、知识边界同构（类型 18）、测试盲区 |
| **层 4: 跨文档追溯矩阵** | commit 前 | 对每次修改涉及的每个概念建立追溯链：定义者 → 契约 → 各消费方（代码+设计文档），标记一致/不一致 | 跨文档矛盾（类型 11-14）、接口演化不同步（类型 20-21） |

### 外部 API 预验证

> 防止 LLM 幻觉出不存在的 GDExtension/godot crate API 和 GDScript 内置函数（grill-me Q33 + CHG-064 审查）。

**Rust 侧**：编码前对每个要使用的非 `std` crate，列出计划调用的方法和文档引用。验证方式：用 `WebFetch` 查 docs.rs 或官方文档。

**GDScript 侧**（新增）：因为 LLM 训练数据中 GDScript 占比约为 Rust 的 1/1000，API 幻觉风险极高。写 `.gd` 文件前：
- [ ] 对每个使用的 GDScript 内置函数（`fmod`/`smoothstep`/`lerp`/`TAU` 等），用 `WebFetch` 查 Godot 4.7 官方文档确认存在性
- [ ] 对每个使用的 Godot 节点属性/方法（如 `DirectionalLight3D.position` vs `look_at()`、`ProceduralSkyMaterial` 属性列表），确认 API 签名
- [ ] 对每个 `_process`/`_ready`/`_input` 回调，确认在对应节点类型上的行为（如 plain `Node` 的 `_process` 是否默认运行）
- [ ] 冲刺提案的"外部 API 预验证清单"段必须列出本冲刺涉及的 GDScript API 及其文档验证状态

未验证的 API 不允许调用。这条不是建议——CHG-064 冲刺中 `double()`、`DirectionalLight3D` 光照方向、`fmod` 等多次 GDScript API 幻觉是直接后果。

### 错误处理哲学（三级策略）

> grill-me Q28。

| 级别 | 策略 | 适用 | Rust 实现 |
|------|------|------|----------|
| **Fatal** | `panic!` / `expect()` — 不可恢复 | 数据完整性错误（TOML 配方不可解析）、硬件不可用（GPU 缺失）、世界生成逻辑矛盾 | `expect("原因: ...")` 必须注释 |
| **Recoverable** | `Result<T, E>` + 降级策略 | 资源缺失（纹理/音效）、网络数据校验失败 | `thiserror` 定义错误类型 + `?` 传播 |
| **Expected** | 正常控制流，不用 `Result` | 寻路失败、NPC 行为冲突、碰撞穿透——正常游戏状态，不是"错误" | 返回 `Option` 或默认行为 |

### 分层测试策略

| 测试层 | 覆盖什么 | 谁来写 | 不过不 commit |
|--------|---------|--------|--------------|
| **单元测试** | 纯函数（公式、变换、序列化）、trait mock 实现 | Claude | ✅ |
| **集成测试** | 跨模块数据流（如"世界生成→内存→存档系统"） | Claude | ✅ |
| **行为快照测试** | 确定性场景输出（如"同一 seed 地形一致"） | Claude | ✅ |
| **LLM Rust 病测试** | 见附录A 专项检查清单 | Claude | ✅ |
| **交互测试** | Godot 实际运行、手感、视觉、涌现 | 用户 | 你的领地 |

> 测试必须至少包含一个与设计文档直接对照的用例——测试的输入/期望输出引用设计文档中的具体段落（grill-me 审计缺口 6）。

### §14 架构边界合规审计（每冲刺强制）

> **新增 2026-06-24**。根因：CHG-064 冲刺中昼夜渲染 100% 迁移到 GDScript，Rust WorldClock 太阳轨道部分成为僵尸代码，且无任何检查机制拦截此偏离。三层质量门（机械/契约/设计）全部通过但偏离未被发现。

#### 14.1 边界铁律

| 规则 | 内容 |
|------|------|
| **Rust 权威** | 所有游戏世界状态（时间、位置、属性、关系）的**唯一权威**在 Rust 侧。Godot 只读 Rust 状态，不独立推导 |
| **Godot 纯表现** | Godot 侧代码只能是：读 Rust 状态 → 设节点属性。**禁止在 GDScript 中计算游戏逻辑值**（时间进度、太阳位置、NPC 属性等） |
| **单权威原则** | 同一概念（如"现在几点"）只能有一个计算源头。如果 Rust 和 GDScript 都能算出同一数据，这是 bug——不是灵活性 |
| **GDScript 代码铁律** | 每个 `.gd` 文件的每一行必须属于以下三类之一：① 读 `Input`（键盘/鼠标）② 读 Rust `#[func]` 返回值 ③ 设 Godot 节点属性（position/color/material/shader param）。**禁止在 GDScript 中写数学公式（sin/cos/lerp/clamp）、状态机、条件分支基于游戏世界状态。** 渲染后处理效果（屏幕震动、颜色分级）是唯一例外——且必须在文件顶部标注 `## POST-PROCESSING: <说明>` |

#### 14.2 冲刺结束时的三层扫描

Claude 在冲刺结束前必须执行以下扫描，逐项打勾：

**层一：双权威检测**

扫描所有 GDScript 文件，检查是否存在与 Rust 类型/函数**同名或同义**的计算：

- [ ] GDScript 中无 `day_progress` / `time` / `clock` / `season` 等时间变量名
- [ ] GDScript 中无 `sin/cos` 用于计算太阳/天体位置（这些在 Rust `WorldTime` 中）
- [ ] GDScript 中无独立维护的颜色/数值查找表（色板、温度范围等——应在 Rust 或 TOML 中）
- [ ] GDScript 中无模拟参数常量（如 `SECONDS_PER_DAY`、`SUN_RADIUS`——应在 Rust `NoiseParams` 或 `WorldClock` 中）

**检出物处理**：每发现一项 = **架构偏离**。必须：
- 立即记录到冲刺交接摘要的"已知问题"段
- 标注严重级别（🟡 短期可接受但需修复 / 🔴 阻塞下一冲刺）
- 如果偏离是调试中不得已的临时方案 → 创建修复任务，排入下一冲刺的候选列表并标注 `H`（长期架构健康）维度

**层二：僵尸代码检测**

扫描所有 `#[func]` 方法和 `pub` 函数，检查是否被调用：

- [ ] Rust 侧所有 `#[func]` 方法在 GDScript 中有对应调用（`grep` 方法名跨 `.gd` 文件）
- [ ] Rust 侧所有 `pub` 类型/函数在 crate 内部或外部有消费方
- [ ] 无"为未来预留"的未调用方法（YAGNI——需要时再加）

**检出物处理**：每发现一项 = 要么删，要么标记 `#[allow(dead_code)]` 并注释说明**具体哪个未来任务会用到**（引用冲刺编号或 Issue 号）。不允许无注释的 dead_code 抑制。

**层三：边界穿越审计**

扫描本次冲刺新增/修改的所有 GDScript 文件：

- [ ] GDScript 文件的每一行属于以下三类之一：① 读 Rust `#[func]` 返回值 ② 设 Godot 节点属性（position/color/material）③ 输入处理（`Input.is_key_pressed`）
- [ ] GDScript 中无数学公式（`sin/cos/lerp/clamp` 用于游戏逻辑——渲染后处理如屏幕震动除外）
- [ ] GDScript 中无条件分支基于游戏状态（如 `if day_progress < 0.25`——这种判断应在 Rust 侧，GDScript 只应读到"当前色板 = X"）

**检出物处理**：每发现一项 = **架构偏离**，处理同层一。

#### 14.3 例外：原型豁免

以下情况可以申请豁免（记录到冲刺提案中，用户审批）：
- 冲刺的**明确目标**是"快速验证视觉效果的临时原型"（如"试试看昼夜循环长什么样"）
- 豁免必须标注**到期条件**（如"最晚 3 个冲刺内迁移到 Rust"）
- 豁免代码必须在文件顶部标注 `## ARCH-DEVIATION: <原因> <到期条件>`

#### 14.5 对抗性重新验证（"合理性"审查）

> CHG-064 第二轮审查触发。三个标记为 🟢 "合理"的偏离经过重新审视后——一个基于未验证假设（🔴）、一个有更优方案未实施（🟡）、一个留下死代码未清理（🟡）。根因：偏离的**制造者**和**审查者**是同一个 LLM——制造者构建的"合理性叙事"对审查者同样有说服力，因为它们共享同一个认知框架。

**规则**：冲刺结束时，对所有标记为 🟢 "合理"的偏离，执行对抗性重新验证——

对每一项，假设它是**不合理的**，然后寻找以下类型的反证：

| 反证类型 | 检查问题 | 证据来源 |
|---------|---------|---------|
| **未验证假设** | "合理性"理由中是否有任何未经测试的技术断言？ | 回溯当时的调试记录——是否真的验证了，还是推断的 |
| **残留死代码** | 偏离后是否留下了从未调用的结构体/函数/TOML 条目？ | `grep` 相关符号的调用点 |
| **更优方案未探索** | 在被认定为"唯一可行方案"之前，是否真的尝试过至少一个替代方案？ | 检查 git 历史——是否只有一个实现尝试 |
| **设计文档冲突** | 偏离方向是否与 `开发阶段/` 中任何设计文档的明确意图矛盾？ | 对照相关设计文档 |

**通过标准**：至少找到一条反证 → 重新分类为 🔴 或 🟡，进入偏离日志。未找到任何反证 → 🟢 评级成立，但必须在交接摘要中记录"对抗性验证通过 + 未发现反证的具体检查过程"。

**与 §4 层 3 的关系**：§4 层 3 是编码层面的对抗性自检（"假设这段代码有 bug，找出 3 个"）。§14.5 是架构层面的对抗性自检（"假设这个偏离是不合理的，找出反证"）。同一种方法论，不同粒度。

## §5 编码阻塞处理协议

当 Claude 贴文档写代码时遭遇以下情况，**立刻停止，不继续编码**：

| 情况 | 判定 | Claude 动作 | 用户角色 |
|------|------|------------|---------|
| **文档矛盾** | 设计文档 A 说 X，设计文档 B 说 Y，无法同时遵循 | 标记冲突位置 + 给出建议裁决 + **阻塞** | 等你裁决，记录到 CHG |
| **文档模糊** | 文档太笼统，无法翻译为代码（如"NPC 自行决定"但决定算法没写） | 给出 2-3 个合理实现方案 + 推荐一个 + **阻塞** | 等你选 |
| **设计不可行** | 写代码过程中发现设计的技术假设不成立或代价过高 | 报告不可行的技术原因 + 替代方案 + **阻塞** | 等你裁决 |
| **设计预警** | Claude 感知到设计决策可能导致严重问题（性能崩塌、游戏性破坏、与核心哲学冲突）——设计本身清晰可行，但后果明显不对 | 标记担忧点 + 推理链 + **阻塞**等用户判断 | 等你判断——这是预警，不是否决权 |

> 核心原则：**Claude 不替设计做决定。可以建议，不能填补空白。不可自行推翻设计。**

## §6 版本控制纪律

| 事件 | Git 动作 | 触发者 |
|------|---------|--------|
| 新任务开始 | 在当前工作树直接编码 | 用户指令 |
| 任务完成 | `git commit`（含验证结果附件），**不 push** | Claude（完成后） |
| Push 到远程 | `git push`（纯备份用途） | 仅用户要求时 |
| 审核通过 | 你合并到 master | 用户 |
| 审核不通过 | 追加修正 commit（不 force push） | Claude |
| Pull | 不需要（除非你明确要求） | — |

**铁律**：Claude 不自主合并到 master。节奏完全由用户手动控制。不按时间自动触发任何 git 操作。

### Commit 消息格式

```
<type>: <中文描述>
```
- type: `feat` / `fix` / `refactor` / `test` / `docs` / `chore`
- 示例: `feat: 轨A MC 体素提取` · `fix: 地形法线方向翻转` · `docs: CLAUDE.md 更新测试分布`
- 不强制模块括号或 Sprint 后缀——冲刺在 `sprint-proposals/` 中跟踪，不在 git 中

### Cargo.lock

`Cargo.lock` 纳入版本控制（保证可复现构建）。每次依赖变更后同步提交。

### 版本钉死策略（grill-me Q32）

| 依赖 | 策略 |
|------|------|
| `godot` (GDExtension API) | 钉死 `=0.5.x`，升级需用户审批 |
| Godot 可执行文件 | 路径固定 `tools/godot/Godot_v4.7-stable_win64.exe`，不自动升级 |
| `glam`/`rayon`/`hecs` 等 | 钉死小版本，安全补丁 Claude 可自决 |
| `lmdb-rkv` | 钉死，升级需用户审批 |
| Rust toolchain | `rust-toolchain.toml` 钉死 `1.80` |

Claude 的冲刺提案中如涉及依赖变更，必须单独列出并标注原因。`cargo update` 不是 Claude 的自主操作。

## §7 模块准入三级制

每个设计模块在当前 `DEVELOPMENT_STATUS.md` 中标记准入等级：

| 等级 | 标记 | 条件 | Claude 可做什么 | 限制 |
|------|------|------|----------------|------|
| **冻结** | 🔴 | 设计不完整、存在已知矛盾、或缺关键依赖模块 | 只能做设计补全（轨 B） | 禁止写实现代码 |
| **就绪** | 🟡 | 设计完整、接口契约已定、依赖模块至少 🟡 | 可以写实现 | 接口变更需你审批 |
| **稳定** | 🟢 | 上一阶段实现已通过你的审核 | 自由迭代优化 | 无需逐次审批 |

**等级变更**：Claude 可提议，用户最终裁定。等级维护在 `DEVELOPMENT_STATUS.md` 中。

### Crate 分裂规则（grill-me Q35）

| 触发条件 | 动作 |
|---------|------|
| `woworld_core` 内部某个子系统达到 ~500 行且有独立的外部依赖 | 拆为独立 crate，如 `woworld_worldgen` |
| 两个子系统之间可以完全编译不互相导入 | 可以拆，但不强制——复用优先 |
| 某个 crate 的编译时间超过 10 秒 | 考虑拆分 |
| GDExtension 桥接代码中出现了游戏逻辑 | 立刻抽出游戏逻辑到独立 crate |

**不触发拆分的条件**：纯粹为了"模块对应一个 crate"（设计模块和 crate 不需要 1:1）、过早引入复杂 crate 图。Claude 可提议拆分（在冲刺提案中说明理由），用户审批。Claude 不自主创建新 crate。

### 模块依赖图（`DEPENDENCY_GRAPH.md`）

由 CLAUDE-INTERFACES.md 契约推导，定义层 0→1→2→… 实现顺序。每往 `woworld_core` 新增类型时同步更新。Claude 在冲刺提案中必须标注"本冲刺的依赖前提已满足"。

## §9 失败与回滚协议

| 情况 | 处理 |
|------|------|
| 同一问题修 **3 次**仍不过自检 | **停止**，汇报"卡住了 + 三次尝试过程 + 我认为的根因"，等你介入 |
| 实现路径走不通 | **立刻停止**（不试第二次），汇报"不可行 + 替代方案"，等你裁决 |
| 发现设计矛盾 | 触发 §5 编码阻塞协议，**立刻停止** |

> 核心原则：**死磕是浪费——发现红灯立刻举手。不闷头反复修。**

### 回滚程序（grill-me 审计缺口）

当冲刺代码被用户判定为"架构方向错误"（非 bug 级可修），执行 `git revert` 回滚整个冲刺的 commit。回滚后：
1. Claude 分析方向错误根因
2. 写入交接摘要的"已知问题"段
3. 下一个冲刺重新提案，不重复错误方向

回滚不删除冲刺提案文件——保留为"此路不通"的历史记录。

## §10 交接摘要规范

每个冲刺结束（或对话窗口即将溢出时），Claude 产出交接摘要为 `.md` 文件，存入 `handoff/` 目录。

**命名**：`handoff-YYYYMMDD-NNN.md`（NNN 为递增序号）

### 交接激活协议（新窗口 Claude 启动流程）

> grill-me 审计缺口 1。"信任但验证"——交接摘要是线索，代码是事实。

新窗口 Claude 加载交接摘要后，在提出冲刺提案之前，必须执行**状态验证扫描**：
1. **读取实际代码**：加载交接摘要中列出的 WIP 文件，不依赖摘要描述
2. **对照确认**：逐项检查摘要中的"已完成目标"——代码是否存在、`cargo build` 是否通过
3. **对比状态文件**：摘要中的"下一步候选"与 `DEVELOPMENT_STATUS.md` 是否一致
4. **发现漂移**：如摘要描述与代码实际状态不一致 → 标记为"交接漂移"，先纠正再提案（同步 DEVELOPMENT_STATUS.md + 如有必要写入新的交接摘要）

### 内容模板：
```markdown
## 交接摘要 YYYY-MM-DD

### 冲刺目标回顾
- [已完成的冲刺目标列表]

### 当前 WIP 精确状态
- 正在进行的文件 + 进度百分比 + 下一步

### 下一步候选列表（含决策矩阵打分）
| 候选 | D(×3) | R(×3) | V(×2) | C(×1) | 总分 |
|------|-------|-------|-------|-------|------|
| ...  |       |       |       |       |      |

### 已知问题/阻塞项
- [待用户裁决的问题列表]

### 关键设计决策记录
- [本冲刺中用户做过的裁决，附日期]
```

**用途**：下一个对话窗口的入口上下文——Claude 加载此文件后即可继续，无需重读全部历史。

### 新会话启动指令

当上下文窗口逼近极限或冲刺自然结束时，Claude 在交接摘要末尾附加**下一会话启动指令**：

```
## 下一会话启动指令
1. 加载交接摘要: woworld-dev-plan/handoff/handoff-YYYYMMDD-NNN.md
2. 执行交接激活协议（本 §10）：读 WIP 代码 → 对照确认 → 对比状态文件 → 查漂移

```

如果用户手动结束会话（而非 Claude 发起），用户可自行参考最新交接摘要启动下一会话，无需 Claude 提醒。

## §13 Phase 生命周期

> Phase = `DEPENDENCY_GRAPH.md` 中一个依赖层的所有模块达到 🟢。Phase 是**回顾性**概念——不预先规划时间线，完成时自然标记。

### Sprint 与 Phase 的关系

```
Phase N (层 N-1) ─┬─ Sprint-X: 第一个模块
                  ├─ Sprint-Y: 第二个模块
                  └─ Sprint-Z: 层收尾 → 全部 🟢 → Phase 自然结束
                  ↓ 决策矩阵（§2）引导下一层的冲刺顺序
Phase N+1 (层 N) ─┬─ ...
```

### Phase 开始条件

- 上一 Phase 所有模块达到 🟢
- 对照 `DEPENDENCY_GRAPH.md` 确认依赖前提满足
- 无待裁决的阻塞项

Claude 按 §2 决策矩阵在 Phase 内自主排冲刺顺序——不需要额外的 Phase 计划。

### Phase 结束

Phase 结束时 Claude 在冲刺交接摘要中标注。不需要独立的 Phase 交接文档——`DEPENDENCY_GRAPH.md` 的层状态 + `DEVELOPMENT_STATUS.md` 的模块 🟢 已充分记录。
