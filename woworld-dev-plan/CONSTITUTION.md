## §1 核心哲学

1. **涌现优先** — 玩家与 NPC 无本质区别。内容从原子系统涌现，不轻易建立独立模块。
2. **编码不替设计做决定** — AI 可提建议，不可自行填补设计空白。偏离原设计意图必须用户裁决。
3. **贴文档写代码** — 触及新模块时，第一步逐字精读 `开发阶段/<模块>/` 原设计文档。
4. **长远眼光** — 每个编码选择考量：25 个模块全部接入后，这个选择还成立吗？
5. **计划反身性** — 流程本身可改进。AI 发现流程规则有改进空间时主动提出。
6. **ECS 数据行为分离** — Component = 纯数据（零方法），System = 纯函数。System 通过 Component 拆装通信，不设 before/after 顺序依赖。ECS 铁律见 [`开发文档/00-ECS哲学与架构总纲/006-ECS铁律与陷阱.md`](../开发文档/00-ECS哲学与架构总纲/006-ECS铁律与陷阱.md)。

## §2 Rust 是唯一权威

- **所有游戏世界状态**在 Rust 侧定义和维护
- Godot 是**只读表现层**：渲染、UI、音频、输入
- GDScript 代码只能是三种操作：读 Input、读 Rust `#[func]` 返回值、设 Godot 节点属性
- GDScript 中**禁止**数学公式（sin/cos/lerp/clamp）用于游戏逻辑
- GDScript 中**禁止**条件分支基于游戏状态
- 仅玩家保留 Godot `PhysicsServer3D`，其余全部 Rust 侧空间查询
- `hecs::World` 由 `WorldDriver` 管理——Godot 侧不持有 Entity 引用
- GDScript 禁止查询 `hecs::World`——只能通过 Rust `#[func]` 消费结果

## §3 冲刺审批（两级）

| 级别 | 内容 | 审批人 | 频率 |
|------|------|--------|------|
| **冲刺级** | 冲刺目标（≤3 条）、优先级、涉及模块 | **用户** | 每个冲刺前 |
| **任务级** | 冲刺内具体任务拆解与执行顺序 | **AI 自主** | 冲刺内自由调度 |

### 必须中断请求用户裁决

- 跨模块接口变更（修改 `CLAUDE-INTERFACES.md` 已有条目）
- 新增外部依赖（Cargo.toml 新 crate）
- 偏离原设计文档意图
- 🟠🔴 级架构决策（选型 / 性能vs复杂度 / 设计文档矛盾）
- ECS System 的 writes 集合与已有 System 冲突（并行安全违规）
- 设计文档变更（`开发阶段/` 或 `开发文档/` 有修改）——必须触发场景 G 影响评估

## §4 质量门不可跳过

| 层 | 内容 | 验证方式 | 签字 |
|---|------|---------|------|
| 🔧 **机械门** | build + test + clippy + fmt | **粘贴真实命令输出**到 Handoff | AI |
| 📐 **设计门** | trait 签名 vs CLAUDE-INTERFACES.md | 逐条对照表 + ECS 铁律合规检查 + 一致性脚本输出 | AI |

> **单一真相源原则**：`附录E-开发状态.md` 为模块状态的唯一权威源。其他文件（README/00-流程总览/1A-1J）只能**引用**状态，不能**复制**数字。复制 = 漂移风险。
| 🎮 **功能门** | Godot 场景行为正确 | 人类跑 checklist | **人类** |

> 机械门输出不可伪造——粘贴到 Handoff 中供人类抽查。功能门未签字 = 冲刺/阶段未完成。零例外。
> 详细检查清单见 [`附录G-质量检查清单.md`](附录G-质量检查清单.md)。

## §5 🧪 研究先行

遇到以下情况时，**必须先研究再编码**：

| 级别 | 触发 | 研究深度 |
|------|------|---------|
| 🟢 已知模式 | 常规数据结构、标准库用法 | 跳过 |
| 🟡 新算法/技术 | 未见过的算法、不熟悉的 crate | 搜索 2-3 个现有方案 |
| 🟠 性能关键路径 | 地形采样、寻路、网格生成、ECS 性能瓶颈（Archetype 迁移/查询优化） | 行业标准 + 现有 Rust 实现 |
| 🔴 架构级决策 | 自研 vs 第三方、数据流设计 | 深度调研 + ADR 记录 |

**研究四步法**：识别问题类别 → 搜索行业标准（WebSearch，保留 URL 引用）→ 2-3 方案决策矩阵 → 记录决策。

> 详细研究协议见 [`00-流程总览.md`](00-流程总览.md) 场景 C §2。

## §6 分诊协议

收到问题反馈时，**先分诊再行动**。

| 类别 | 判定 | 动作 |
|------|------|------|
| 🔴 **回归** | 之前可工作、现在坏了 | **立刻修** |
| 🟡 **已知偏离** | 已在追踪中 | 引用追踪 ID，不重复调查 |
| 🟢 **参数调优** | 数值/效果不满意 | 附成本估算，用户决定 |
| 🔵 **结构性依赖** | 根因依赖未来冲刺 | 标注 Sprint-N，不现在修 |

**回复格式**：
```
> **分诊**: [🔴🟡🟢🔵] — [一句话判定]
> **如果现在修**: 影响 ~N 文件, 预计 ~X-Y 分钟, 阻塞风险: [无 / 依赖 Sprint-N]
> **建议**: [现在修 / 记入已知问题 / 等待 Sprint-N 自然解决]
```

用户有权在任何分诊后说"不，现在修"——分诊是建议，不是否决。

## §7 偏差升级协议

```
同一目标连续 2 冲刺未完成 → 🟡 偏差：AI 自主调整时间线
同一目标连续 3 冲刺未完成 → 🔴 重评估：AI 分析 + 用户裁决（继续/降级/拆分）
同一里程碑 2 次重评估 → 🔴🔴 重规划：用户主导重新设计阶段
发现架构级不可行 → 立即停止，不等 3 次
```

| 层级 | 触发 | 范围 | 硬门禁 |

## §9 设计变更 → 执行手册更新

设计文档（`WoWorld-Design/Happy Game/开发阶段/` 或 `开发文档/`）变更时，**下一个冲刺启动前**必须执行影响评估：

1. `git diff 开发阶段/` → 变更文件清单
2. 三问法：变更了哪个模块？→ 在哪个阶段/里程碑？→ 验收标准是否仍有效？
3. 按级别响应（🟢参数微调不更新 / 🟡追加验收项 / 🟠追溯依赖链 / 🔴标记风险）
