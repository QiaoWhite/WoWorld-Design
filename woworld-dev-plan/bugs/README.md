# Bug 追踪知识库

> **定位**: LLM 优先的结构化 bug 知识库。目标：新会话遇到 bug 时，5 秒内判断"以前见过吗"。

## 准入标准

**前向判断**——修复完成后问自己：

> 如果这个 bug 在未来新会话中重现，LLM 能从现有文档（handoff / devlog / 代码注释）里快速定位解决方案吗？**如果不能 → 写入此文件夹。**

高信号特征（满足任一即强烈建议录入）：

| 特征 | 说明 |
|------|------|
| 跨会话复发风险 | 修过但可能以不同形式回来 |
| 长时排查（≥2h） | 根因发现过程曲折，值得保留 |
| 根因反直觉 | 症状和根因差距大，容易误诊 |
| 架构/平台限制 | 非代码 bug，但会持续影响 |

## 文件命名规则

`{MODULE缩写}-{NNN}-{kebab-症状关键词}.md`

例：`WG-001-clipmap-fp32-edge-cracks.md`

## 模板

新条目从 [[TEMPLATE]] 复制。字段说明见模板注释。

## INDEX.md 用法

- **LLM 调试第一步**：`grep "症状关键词" INDEX.md` → 命中则打开对应条目
- **新增 bug 后**：在 INDEX.md 索引表追加一行
- **状态变更后**：更新 INDEX.md 对应行的状态列

## 与现有文档的关系

| 文档 | 定位 | 与本文件夹关系 |
|------|------|---------------|
| **本文件夹** | 结构化 bug 知识库（权威源） | — |
| handoff/ | 会话交接，含实时 bug 修复记录 | handoff 是实时记录，bug 文件夹是事后结构化提取 |
| devlogs/ | 每日开发叙述 | devlog 是流水账，bug 文件夹是精炼结论 |
| 附录E-开发状态.md | 模块完成度追踪 | 附录E 的"已知问题"改为指向本文件夹的指针 |
| sprint-proposals/ | 冲刺提案，含已知问题表 | sprint 提案引用 bug ID，不重复写 bug 详情 |

## 创建与更新协议

| 时机 | 动作 |
|------|------|
| **修复完成后** | 判断准入标准 → 是则从 [[TEMPLATE]] 复制创建新文件 → 更新 INDEX.md |
| **复发时** | 编辑原 bug 文件，追加"复发记录"表格行 |
| **方案被推翻时** | 编辑原 bug 文件，更新"解决方案" + 更新 `last_verified` |
| **重构触及代码位置时** | 检查 bug 文件的"代码位置"是否过时 → 更新或标注 `⚠️方案过时` |

## 代码双向链接

在代码中对应位置标记 `// BUG:XX-NNN`，bug 文件中写 `## 代码位置` 列出函数签名 + 文件路径。两边可互相追溯。
