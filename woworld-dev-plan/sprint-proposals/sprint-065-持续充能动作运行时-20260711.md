## 🎯 目标（≤3 个）

### 目标 1: Continuous 动作运行时 + SustainDrain 消耗
- **验收标准**: `cargo test -p woworld_ecs` + `-p woworld_core` 通过；`action_controller_tick` 接受 `Continuous` 动作进入无限 Active；`action_system` 每帧按 `SustainDrain.rate_per_sec × dt` 消耗 `CVitals` 对应资源（Stamina/Mana）；`SustainPhase` 按 `overextend_threshold_secs`/`critical_threshold_secs` 迁移 `Normal → Overextended{penalty} → Critical{forced_release_in}`；Overextended 期间消耗按 `overextend_multiplier` 加速；资源耗尽或 Critical 到期 → 强制释放（发 `Completed`，`FinishReason::ForcedRelease`）；松键（`priority=RELEASE`）→ `ReleaseBehavior::Complete` 纯结束 / `Trigger{action_id}` 触发固定子动作。以下场景单测：正常松键结束、Stamina 耗尽强制结束、Overextended 加速消耗、Trigger 子动作注入。
- **涉及模块**: 角色控制器 006 §一/§二/§三 `WoWorld-Design/.../角色控制器/006-持续动作与充能动作.md`
- **涉及代码**: `woworld_core/src/action.rs`（`FinishReason::ForcedRelease` + 可能补 `ActiveAction.charge_elapsed`）；`woworld_ecs/src/systems/action/action_controller.rs`（解除 Discrete 门 + 释放分发）；`.../action/action_system.rs`（sustain drain + SustainPhase 迁移）

### 目标 2: Charge 动作运行时 + 充能阶梯 + 帧间子动作接续
- **验收标准**: `cargo test` 通过；`Charge` 动作进入无限 Active 并累计充能时长；松键时按 `elapsed` 命中最高满足的 `ChargeStage`（`threshold_ms` 降序匹配）→ 选定 `action_id` + `power_multiplier`；子动作经 **pending follow-up 机制**（新增 `CPendingFollowUp` 组件或 `ActionLifecycleEvent::FollowUp`——见 §裁决 Q3）在下一帧注入 `CActionRequestBuf`，`source=ChargedAction`；`OverchargeBehavior` 三分支：`AutoRelease`（到顶自动释放最高阶）/`Penalize{accuracy_loss_per_sec}`（继续充能，记精度损失）/`ForceCancel`（超时动作失败，发 `Interrupted`）。以下场景单测：quick/aimed/full 三阶按时长选择、overcharge AutoRelease、overcharge ForceCancel、子动作帧间注入闭环。
- **涉及模块**: 角色控制器 006 §一/§四/§五
- **涉及代码**: `woworld_core/src/action.rs`（`ActiveAction` 充能追踪字段 + follow-up 载体）；`woworld_ecs/.../action_controller.rs`（充能阶梯选择）；`.../action_system.rs`（overcharge 推进 + follow-up 注入）；新增 `CPendingFollowUp`（若 Q3=A）

### 目标 3: TOML 动作示例 + 廉价延后项折入（A3 + M4）+ 端到端集成

## 🔒 关键架构约束（沿用绞杀者模式）

- **绞杀者隔离**：现有 Discrete 动作路径不改语义——Continuous/Charge 为**新增分支**，Discrete 测试零回归是硬门。
- ActionController 仍是**纯函数**——不写 Vitals/不改 MovementState；sustain drain 的资源消耗在 `action_system`（ECS wrapper）内做，保持核心可单测。
- follow-up 子动作 `source=ChargedAction`——走与玩家/GOAP **完全相同**的仲裁路径（006 §六：无玩家/NPC 分支）。
- `CVitals` 写权单一化（Q2 定论后）——不得两系统同帧写 Vitals（铁律 #10）。
- 每个新 System 分支至少 1 个测试；Component 保持纯数据零方法。
