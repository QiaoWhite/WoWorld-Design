## 📐 实现设计

### 架构：三处精准修改，不改算法

**原则**：只修数据来源，不动撮合/EMA/订单簿。`order_creation_system` 的决策逻辑（surplus/deficit/bid/ask）全部保留——仅输入数据变真。

### 改动 1: `order_creation_system` — 真实 Needs → daily_need

```
旧: let daily_need = 1.0 + (entity_seed.wrapping_mul(17) % 3) as f32 * 0.2;
新: let hunger = world.get::<&Needs>(entity).map(|n| n.hunger).unwrap_or(0.0);
    let daily_need = 0.2 + hunger * 1.5;  // [0.2, 1.7] — 饱→低需求，饿→高需求
```

**设计依据**：`Needs.hunger` 0=满足→1=缺乏。V3a 中吃饱后 `hunger -= hunger_restore`。饥饿 NPC 的 daily_need 高→reserve_need 高→更可能产生买单。

### 改动 2: `order_creation_system` — 真实 InventoryRegistry 库存

```
旧: let holdings = registry.get_holdings(entity_id);  // EconomyRegistry.item_holdings
新: let holdings = inventory_registry.get_holdings(entity_id);  // InventoryRegistry (权威源)
```

移除首帧 `seed_npc_items` 调用——V3a 后 NPC 自己采集食物，初始种子已在 `inventory_init_system` 通过 `InventoryRegistry::seed_npc_items` 完成。

### 改动 3: Wallet 双账统一

**方案**: `order_creation_system` 从 registry 读钱包（唯一权威），ECS Wallet component 同步为只读镜像。

```
旧: let wallet_balance = wallet.total_copper();  // ECS component (可能过期)
新: let wallet_balance = registry.get_wallet(entity_id)
       .map(|w| w.total_copper()).unwrap_or_else(|| wallet.total_copper());
```

`market_matching_system` 之后 → 遍历成交参与者 → `cmd.insert_one(entity, Wallet::from(registry_snapshot))` 回写 ECS。

## 🚫 诚实边界

- **不碰撮合引擎**：`match_orders`/`match_all_markets`/`execute_trade` 零改动
- **不新增物品**：现有 TOML 物品池不变
- **不做生产驱动**：NPC 不会"制造"物品——Phase 3
- **不实现 `estimate_daily_consumption()` 完整版**：只读 Needs.hunger，不接入 profession/family_size
