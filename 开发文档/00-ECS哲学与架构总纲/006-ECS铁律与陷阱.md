# ECS 铁律与陷阱

> **关联**: [[001-ECS不是换了皮的POP]] · [[002-Component拆装机制]] · [[004-hecs存储与查询]]

---

本文档汇总了 ECS 架构中不可违背的 8 条规则，每一条都对应一个或多个已知的生产环境陷阱。

---

## 铁律 1: Component = 纯数据，零方法

```rust
// ✅ 正确
struct Vitals { hp: u32, max_hp: u32, stamina: f32 }

// ❌ 错误：Component 里有方法
impl Vitals {
    fn take_damage(&mut self, amount: u32) { self.hp = self.hp.saturating_sub(amount); }
}
```

**为什么**：
- 数据与行为分离是 ECS 的基石
- Component 内的方法破坏内存连续布局
- 无法清晰追踪"谁在何时修改了数据"
- 并行执行依赖对数据的确定性操作，方法引入不可预测的副作用

**防御**：写完 Component 后检查——`impl` 块只有 `impl Component for X {}`，没有其他 impl。

---

## 铁律 2: 大堆数据不进 Component 内联

```rust
// ❌ 错误：Vec/HashMap/String 破坏连续内存布局
struct MemoryStore { memories: Vec<Memory> }  // 2000条，堆分配

// ✅ 正确：轻量 Handle + 独立 Resource 存储
struct MemoryHandle { storage_key: u64, count: u16, oldest_tick: u64 }
```

**Component 允许的字段类型**：`u8/u16/u32/u64/i32/f32/f64/bool`、固定大小数组 `[T; N]`、`EntityId`。

**禁止的类型**：`Vec<T>`、`HashMap<K,V>`、`String`、`Box<T>`、`Arc<T>`、`&T`、`&str`。

**WoWorld 受影响类型及处理**：

| 原类型 | 数据量 | ECS 处理 |
|--------|--------|---------|
| MemoryStore | 2000条 | MemoryHandle Component + MemoryStorage Resource |
| SkillMap | HashMap | SkillHandle + SkillStorage Resource |
| Relationship | BTreeMap | RelationHandle + RelationStorage Resource |
| AgentSnapshot | 108B | ✅ 固定大小，直接作为 Component |

## 铁律 3: 拆装 vs 标签——决策指南

Component 拆装触发 Archetype 迁移。在 100 NPC 规模下不是瓶颈，但需知何时用拆装、何时用标签。

| 场景 | 机制 | 原因 |
|------|------|------|
| NPC 死亡（活→死） | ✅ 拆装 | 根本性状态变化，组件集完全不同 |
| 装备武器（无→有） | ✅ 拆装 | 低频事件（每分钟级别） |
| 战斗中/非战斗中 | ⚠️ 标签 | 高频切换（每秒级别） |
| 移动中/静止 | ⚠️ 标签 | 高频切换 |
| 白天/夜晚行为 | ⚠️ 标签 | 所有 NPC 同时切换 |

**标签（Tag Component）** = 零数据 Component（`struct InCombat;`）。切换标签也会触发 Archetype 迁移，但成本低于拆装多个数据 Component。

**对于极高频率的状态切换**（每帧可能变化）：用 Component 内的 `enum` 字段而非 Component 级别的增删。

## 铁律 4: 实体删除——标记 + 统一清理

```rust
// ❌ 错误：迭代中直接 despawn
for (e, health) in world.query::<&Health>().iter() {
    if health.hp == 0 { world.despawn(e).unwrap(); }  // 迭代器失效！
}

// ✅ 正确：标记 + 延迟清理
struct PendingDespawn;
fn death_watch(world: &hecs::World, cmd: &mut CommandBuffer) {
    for (e, health) in world.query::<&Health>().iter() {
        if health.hp == 0 { cmd.insert_one(e, PendingDespawn); }
    }
}
fn cleanup(world: &hecs::World, cmd: &mut CommandBuffer) {
    for (e, _) in world.query::<&PendingDespawn>().iter() { cmd.despawn(e); }
}
```

**`CommandBuffer` 的操作在迭代完成后才应用**——System 执行期间不修改 Archetype 结构。

## 铁律 5: 并行安全条件

Phase 1 中的 System 虽然平等，但 rayon 并行执行需要满足：

1. 两个 System **不能同时写**同一个 Component 类型
2. System A 写 + System B 读同一个 Component → **可以并行**（读旧值，下帧刷新）

```rust
struct SystemMeta {
    func: SystemFn,
    reads: &'static [TypeId],
    writes: &'static [TypeId],
}
// 注册时验证：任意两个 System 的 writes 集合不相交
```

**不满足条件的少数 System** → 标记为顺序执行（仍是例外，不是普遍规则）。

## 铁律 6: Component 必须 'static + Send + Sync

```rust
// hecs 的约束: pub trait Component: Send + Sync + 'static {}

// ❌ 错误：引用生命周期不足
struct NameComponent<'a> { name: &'a str }

// ✅ 正确：自有类型
struct NameComponent { name_hash: u64 }
```

**约束**：Component 不能包含任何引用（`&T`, `&str`, `&[T]`）。字符串用 `u64` hash 替代，原文存 Resource。

---

## 铁律 7: hecs 借用冲突会 panic

```rust
// ❌ 编译不过：同类型两次可变借用
world.query::<(&mut Position, &mut Position)>();

// ❌ 运行时 panic：两个 System 并行写同类型
// System A: query::<&mut Position>()
// System B: query::<&mut Position>()
// 如果在 rayon scope 中并行执行 → panic
```

**防御**：铁律 5 的 `SystemMeta` 注册时检查——开发期捕获，不会到运行时。

---

## 铁律 8: 不用 unwrap()

```rust
// ✅ query().iter() 已保证组件存在——不需要 unwrap
for (e, (needs, vitals)) in world.query::<(&Needs, &mut Vitals)>().iter() {
    vitals.hp = vitals.hp.saturating_sub(needs.hunger as u32);
}
// 确实可能失败的操作用 if let
let Some(vitals) = world.get_mut::<Vitals>(entity) else { continue; };
```

**原则**：`query().iter()` 过滤了 95% 的"组件不存在"情况。剩余 5% 用模式匹配。

## 测试策略

ECS System 是纯函数，天然适合单元测试：

```rust
#[test]
fn test_death_watch_kills_zero_hp() {
    let mut world = hecs::World::new();
    let mut cmd = CommandBuffer::new();
    let e = world.spawn((Vitals { hp: 0, max_hp: 100 }, Position::default()));
    
    death_watch_system(&world, &mut cmd);
    cmd.run_on(&mut world);
    
    assert!(world.get::<Vitals>(e).is_err());    // 拆了
    assert!(world.get::<Corpse>(e).is_ok());      // 装了
}
```

**约束**：每个 System 至少一个测试。System 通过组件状态耦合（不是调用耦合）——测试独立。

## 调试策略

数据与逻辑分离后追踪困难。补偿方案：

1. **SystemLog Resource**：每帧记录 System 名/实体数/组件变化数/耗时
2. **确定性回放**：固定 seed + 初始状态 → 逐帧录制 Component 快照
3. **Changed<T> 查询**：hecs 的变更检测——只处理被修改过的实体
